## 1. Spring 核心原理

### 1.1 IoC 与 DI

**IoC（控制反转）**：将对象的创建和依赖关系的管理从代码中移出，交给 Spring 容器负责。

**DI（依赖注入）**：IoC 的实现方式，容器在创建对象时自动注入其依赖的对象。

**Bean 的生命周期**：

1. 实例化（Instantiation）：通过反射调用构造器
2. 属性填充（Populate）：注入依赖（@Autowired）
3. 初始化前（BeanPostProcessor#postProcessBeforeInitialization）
4. 初始化（InitializingBean#afterPropertiesSet 或 @PostConstruct）
5. 初始化后（BeanPostProcessor#postProcessAfterInitialization）
6. 使用中
7. 销毁（DisposableBean#destroy 或 @PreDestroy）

**三级缓存解决循环依赖**：

- 一级缓存（singletonObjects）：完整的单例 Bean
- 二级缓存（earlySingletonObjects）：早期暴露的 Bean（未完成属性填充）
- 三级缓存（singletonFactories）：Bean 工厂，用于生成早期引用（支持 AOP 代理）

循环依赖解决流程：A 创建时放入三级缓存 → 注入 B → B 创建时注入 A → 从三级缓存获取 A 的早期引用 → B 完成创建 → A 完成属性填充。

**注意**：构造器注入的循环依赖无法解决（因为对象还未创建就需要依赖）。

### 1.2 AOP 原理

**AOP（面向切面编程）**：将横切关注点（日志、事务、权限）从业务逻辑中分离。

**两种代理方式**：

- **JDK 动态代理**：基于接口，通过 `Proxy.newProxyInstance()` 生成代理类，要求目标类实现接口
- **CGLIB 代理**：基于子类，通过字节码生成目标类的子类，不要求实现接口（Spring Boot 默认使用）

**五种通知类型**：@Before、@After、@AfterReturning、@AfterThrowing、@Around（最强大，可控制是否执行目标方法）

**@Transactional 失效场景**：

1. 方法非 public（代理无法拦截）
2. 同类内部调用（绕过代理，直接调用目标方法）
3. 异常被 catch 吞掉（事务感知不到异常）
4. 异常类型不匹配（默认只回滚 RuntimeException 和 Error）
5. 数据库不支持事务（如 MyISAM）

### 1.3 Spring Boot 自动配置

**核心注解 @SpringBootApplication** = @Configuration + @EnableAutoConfiguration + @ComponentScan

**自动配置原理**：

1. `@EnableAutoConfiguration` 通过 `@Import(AutoConfigurationImportSelector.class)` 触发
2. `AutoConfigurationImportSelector` 读取 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`（JDK21）或 `spring.factories`（旧版）
3. 加载所有 `AutoConfiguration` 类，通过 `@ConditionalOnClass`、`@ConditionalOnMissingBean` 等条件注解决定是否生效

## 2. Spring Cloud 微服务生态

### 2.1 核心组件对比

| 功能 | Netflix（旧） | Alibaba（推荐） | 说明 |
| --- | --- | --- | --- |
| 服务注册发现 | Eureka | Nacos | Nacos 同时支持 AP 和 CP |
| 配置中心 | Config + Bus | Nacos Config | Nacos 支持动态刷新 |
| 负载均衡 | Ribbon | LoadBalancer | Ribbon 已停维 |
| 服务调用 | Feign | OpenFeign | 声明式 HTTP 客户端 |
| 熔断降级 | Hystrix（停维） | Sentinel | Sentinel 功能更强大 |
| 网关 | Zuul | Spring Cloud Gateway | Gateway 基于 WebFlux 异步 |
| 链路追踪 | Sleuth + Zipkin | SkyWalking | SkyWalking 功能更全面 |

### 2.2 服务注册与发现

**Nacos 注册中心**：

- 支持 AP 模式（临时实例，默认）和 CP 模式（持久实例，基于 Raft）
- 客户端每5秒发送心跳，15秒未收到心跳标记不健康，30秒删除实例
- 支持推拉结合：服务变更时主动推送给客户端，客户端也定期拉取

**Eureka vs Nacos**：

- Eureka 是纯 AP，不保证数据一致性，适合对可用性要求高的场景
- Nacos 支持 AP/CP 切换，功能更丰富（配置中心、健康检查）

### 2.3 服务熔断与降级

**Sentinel 核心概念**：

- **资源**：需要保护的代码块（方法、接口）
- **规则**：流控规则（QPS/并发线程数）、熔断规则（慢调用比例/异常比例/异常数）、热点规则
- **熔断状态**：关闭（正常）→ 打开（熔断，快速失败）→ 半开（探测恢复）

**Sentinel vs Hystrix**：

| 特性 | Hystrix | Sentinel |
| --- | --- | --- |
| 隔离策略 | 线程池/信号量 | 信号量 |
| 熔断策略 | 异常比例 | 慢调用比例、异常比例、异常数 |
| 限流 | 有限支持 | QPS、并发线程数、热点参数 |
| 控制台 | Hystrix Dashboard | Sentinel Dashboard（实时规则推送） |
| 维护状态 | 停止维护 | 活跃维护 |

### 2.4 API 网关

**Spring Cloud Gateway 核心概念**：

- **Route（路由）**：网关的基本单元，包含 ID、目标 URI、断言、过滤器
- **Predicate（断言）**：匹配请求条件（路径、请求头、参数等）
- **Filter（过滤器）**：对请求/响应进行处理（鉴权、限流、日志）

**网关核心功能**：路由转发、负载均衡、认证鉴权、限流熔断、日志监控、协议转换。

**Gateway vs Nginx**：Gateway 是应用层网关，与 Spring 生态深度集成，支持动态路由；Nginx 是基础设施层网关，性能更高，适合静态路由和静态资源服务。

## 3. 消息队列

### 3.1 消息队列的作用

- **异步解耦**：生产者和消费者解耦，提升系统响应速度
- **流量削峰**：缓冲突发流量，保护下游系统
- **顺序保证**：同一分区/队列内消息有序
- **数据同步**：跨系统数据同步（如 binlog 同步）

### 3.2 主流消息队列对比

| 特性 | RocketMQ | Kafka | RabbitMQ |
| --- | --- | --- | --- |
| 吞吐量 | 十万级 | 百万级 | 万级 |
| 延迟 | 毫秒级 | 毫秒级 | 微秒级 |
| 消息可靠性 | 非常高 | 高 | 高 |
| 事务消息 | 支持 | 支持 | 不支持 |
| 延迟消息 | 支持（固定级别） | 不支持 | 支持（插件） |
| 消息堆积 | 亿级 | 亿级 | 万级 |
| 适用场景 | 金融、电商、业务消息 | 日志、大数据、流处理 | 企业集成、复杂路由 |

### 3.3 消息可靠性保证

**消息丢失的三个环节**：生产者发送 → MQ 存储 → 消费者消费

**生产者侧**：

- RocketMQ：同步发送 + 失败重试；事务消息保证本地事务与消息发送原子性
- Kafka：`acks=all`（所有 ISR 副本确认）+ 重试机制

**MQ 存储侧**：

- RocketMQ：同步刷盘（`SYNC_FLUSH`）保证消息不丢，但性能低；异步刷盘（默认）性能高但可能丢消息
- Kafka：多副本机制（`replication.factor≥3`）+ `min.insync.replicas≥2`

**消费者侧**：

- 手动 ACK：消息处理完成后再确认，避免处理失败但已 ACK 导致消息丢失
- RocketMQ：消费失败会重试（默认16次），最终进入死信队列

### 3.4 消息重复消费与幂等性

**为什么会重复消费**：网络抖动导致 ACK 未送达，MQ 重新投递消息。

**幂等性实现方案**：

1. **唯一消息ID**：消费前查询 Redis/DB 是否已处理该 ID，处理后记录
2. **数据库唯一索引**：利用唯一约束防止重复插入
3. **乐观锁版本号**：更新时检查版本号，版本不匹配则忽略
4. **状态机**：检查业务状态是否已流转，已流转则忽略

### 3.5 消息顺序性

**Kafka 顺序保证**：同一 Partition 内消息有序，通过指定相同的 Key 保证同一业务的消息进入同一 Partition。

**RocketMQ 顺序消息**：

- 全局顺序：只有一个 Queue，吞吐量低
- 局部顺序：同一业务 ID 的消息发送到同一 Queue（MessageQueueSelector）

**消费端顺序保证**：单线程消费，或使用 `MessageListenerOrderly`（RocketMQ）。

### 3.6 Kafka 核心原理

**核心概念**：Topic（主题）→ Partition（分区）→ Segment（段文件）→ Message

**高吞吐量原因**：

1. **顺序写磁盘**：消息追加写入，避免随机 IO
2. **零拷贝（sendfile）**：数据从磁盘直接传输到网络，跳过用户态
3. **批量压缩**：批量发送和压缩（gzip/snappy/lz4）
4. **Page Cache**：利用 OS 页缓存，读写都在内存中

**消费者组**：同一消费者组内，每个 Partition 只被一个消费者消费（负载均衡）；不同消费者组独立消费（广播）。

**Rebalance 触发条件**：消费者加入/离开、Topic 分区数变化、消费者心跳超时。Rebalance 期间所有消费者停止消费（STW），应尽量减少 Rebalance。

## 4. RPC 框架

### 4.1 RPC 原理

RPC（Remote Procedure Call）让调用远程服务像调用本地方法一样简单。核心流程：

1. 客户端调用本地 Stub（桩）
2. Stub 将方法名、参数序列化（编码）
3. 通过网络传输到服务端
4. 服务端反序列化，调用实际方法
5. 将结果序列化后返回客户端
6. 客户端反序列化得到结果

### 4.2 Dubbo 核心原理

**Dubbo 分层架构**：

- Service 层：业务代码
- Config 层：配置信息
- Proxy 层：服务代理（生成 Stub）
- Registry 层：服务注册发现
- Cluster 层：集群容错（Failover/Failfast/Failsafe）
- Monitor 层：监控统计
- Protocol 层：协议封装（Dubbo/HTTP/gRPC）
- Exchange 层：请求响应语义
- Transport 层：网络传输（Netty）
- Serialize 层：序列化（Hessian2/Protobuf/JSON）

**负载均衡策略**：随机（默认）、轮询、最少活跃调用数、一致性哈希。

**集群容错策略**：

- Failover（默认）：失败自动重试其他节点，适合读操作
- Failfast：快速失败，只调用一次，适合写操作（防止重复写）
- Failsafe：失败忽略，适合日志记录等非关键操作
- Failback：失败后定时重试，适合消息通知

### 4.3 序列化协议对比

| 协议 | 性能 | 跨语言 | 可读性 | 适用场景 |
| --- | --- | --- | --- | --- |
| JSON | 低 | 是 | 高 | HTTP API、配置 |
| Hessian2 | 中 | 部分 | 低 | Dubbo 默认 |
| Protobuf | 高 | 是 | 低 | gRPC、高性能场景 |
| Kryo | 高 | 否（Java） | 低 | Java 内部高性能 |
| Avro | 高 | 是 | 低 | Kafka、大数据 |

### 4.4 gRPC

gRPC 是 Google 开源的高性能 RPC 框架，基于 HTTP/2 和 Protobuf：

- **HTTP/2**：多路复用、头部压缩、服务端推送，解决 HTTP/1.1 的队头阻塞
- **Protobuf**：二进制序列化，比 JSON 小3-10倍，解析速度快5-10倍
- **四种通信模式**：Unary（一元）、Server Streaming、Client Streaming、Bidirectional Streaming
- **适用场景**：微服务内部通信、多语言混合架构、对性能要求极高的场景
