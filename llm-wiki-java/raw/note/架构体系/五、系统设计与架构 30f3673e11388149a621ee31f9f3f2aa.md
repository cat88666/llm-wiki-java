## 1. 高可用设计

### 1.1 高可用核心指标

**可用性（Availability）**：`A = MTBF / (MTBF + MTTR)`

- MTBF（平均无故障时间）：系统正常运行的平均时长
- MTTR（平均恢复时间）：故障发生到恢复的平均时长

| 可用性 | 年停机时间 | 俗称 |
| --- | --- | --- |
| 99% | 87.6 小时 | 2个9 |
| 99.9% | 8.76 小时 | 3个9 |
| 99.99% | 52.6 分钟 | 4个9 |
| 99.999% | 5.26 分钟 | 5个9 |

### 1.2 高可用设计手段

**冗余**：消除单点故障，所有关键组件部署多实例（主从、主主、集群）。

**故障转移**：

- 主动-被动（Active-Passive）：主节点故障后，被动节点接管
- 主动-主动（Active-Active）：所有节点同时提供服务，任一节点故障不影响整体

**限流**：保护系统不被超出容量的请求压垮，常用算法：令牌桶（允许突发）、漏桶（平滑输出）。

**熔断**：快速失败，防止故障扩散（雪崩效应）。熔断器状态：关闭→打开→半开。

**降级**：在资源不足时，关闭非核心功能，保障核心功能可用。降级策略：返回默认值、返回缓存数据、返回友好提示。

**超时与重试**：

- 所有远程调用必须设置超时（防止线程池耗尽）
- 重试需要幂等性保证，并使用指数退避（Exponential Backoff）+ 抖动（Jitter）防止惊群

**健康检查**：主动健康检查（定期探测）+ 被动健康检查（根据请求结果判断）。

### 1.3 异地多活

**同城双活**：两个数据中心在同一城市，延迟低（<1ms），可实现强一致性，主要应对机房级故障。

**两地三中心**：同城双活 + 异地灾备，异地仅做备份，不提供服务。

**异地多活**：多个地域的数据中心同时提供服务，用户就近访问。核心挑战是数据同步（跨地域延迟高，无法强一致）。

**异地多活数据分区策略**：

- 按用户 ID 分区：每个用户的数据只写入一个地域（主地域），避免跨地域写冲突
- 只读流量可跨地域（读本地副本），写流量路由到主地域

## 2. 高并发设计

### 2.1 高并发核心指标

- **QPS（Queries Per Second）**：每秒查询数
- **TPS（Transactions Per Second）**：每秒事务数
- **RT（Response Time）**：响应时间
- **并发用户数**：同时在线的用户数

**Little's Law**：并发用户数 = QPS × 平均响应时间

### 2.2 高并发架构设计

**读写分离**：主库处理写请求，从库处理读请求。注意主从延迟问题（写后立即读可能读到旧数据）。

**多级缓存**：

```
用户请求 → CDN（静态资源）→ Nginx 本地缓存 → Redis 分布式缓存 → 数据库
```

每一层缓存都能拦截大量请求，减少下层压力。

**异步化**：将耗时操作（发邮件、发短信、更新统计）通过消息队列异步处理，减少主链路响应时间。

**池化技术**：线程池、数据库连接池、HTTP 连接池，复用资源，减少创建销毁开销。

**无状态设计**：服务节点不保存用户状态（Session 外置到 Redis），支持水平扩展。

### 2.3 秒杀系统设计

**核心挑战**：瞬间高并发（QPS 可达平时百倍）、库存超卖、数据一致性。

**分层拦截策略**：

1. **前端**：按钮置灰防重复点击，随机丢弃请求（只放行10%）
2. **CDN/Nginx**：静态页面缓存，限流（`limit_req`）
3. **网关层**：用户维度限流（同一用户每秒最多1次请求）
4. **应用层**：本地缓存库存（减少 Redis 访问），队列排队
5. **Redis 层**：原子扣减库存（`DECR`），库存为0时直接返回失败
6. **数据库层**：异步写订单，乐观锁防超卖

**库存扣减方案**：

- Redis 原子操作：`DECR stock`，返回值 < 0 则超卖，回滚
- Lua 脚本：判断库存 + 扣减原子执行
- 数据库乐观锁：`UPDATE stock SET count=count-1 WHERE id=? AND count>0`

**防刷策略**：用户 Token 验证、IP 限流、验证码、黑名单机制。

## 3. DDD 领域驱动设计

### 3.1 核心概念

**战略设计**：

- **领域（Domain）**：业务问题空间，如电商领域
- **子域（Subdomain）**：核心子域（竞争优势，如推荐算法）、支撑子域（必要但非核心，如物流）、通用子域（通用能力，如认证）
- **限界上下文（Bounded Context）**：模型的边界，同一概念在不同上下文有不同含义（如"用户"在订单上下文和会员上下文含义不同）
- **上下文映射（Context Map）**：定义限界上下文之间的关系（防腐层、共享内核、客户-供应商等）

**战术设计**：

- **实体（Entity）**：有唯一标识，通过 ID 区分（如订单、用户）
- **值对象（Value Object）**：无唯一标识，通过属性值区分，不可变（如金额、地址）
- **聚合（Aggregate）**：一组相关实体和值对象的集合，有一个聚合根（Aggregate Root）
- **聚合根（Aggregate Root）**：聚合的入口，外部只能通过聚合根访问聚合内部对象
- **领域服务（Domain Service）**：不属于任何实体的领域逻辑
- **领域事件（Domain Event）**：领域内发生的重要事情（如订单已创建）
- **仓储（Repository）**：聚合的持久化接口（领域层定义接口，基础设施层实现）
- **应用服务（Application Service）**：协调领域对象完成用例，不包含业务逻辑

### 3.2 DDD 分层架构

```
┌─────────────────────────────────┐
│  用户接口层（User Interface）    │  Controller、DTO
├─────────────────────────────────┤
│  应用层（Application）          │  ApplicationService、Command/Query
├─────────────────────────────────┤
│  领域层（Domain）               │  Entity、ValueObject、DomainService
├─────────────────────────────────┤
│  基础设施层（Infrastructure）   │  Repository 实现、MQ、外部服务
└─────────────────────────────────┘
```

**CQRS（命令查询职责分离）**：将写操作（Command）和读操作（Query）分离，写操作走领域模型保证一致性，读操作走优化的查询模型（可以直接查 DB 或 ES）。

### 3.3 微服务与 DDD 的关系

一个限界上下文通常对应一个微服务（但不是绝对的，初期可以合并，后期再拆分）。DDD 提供了微服务拆分的理论依据：按业务边界拆分，而不是按技术层次拆分。

## 4. 设计模式

### 4.1 创建型模式

**单例模式**：确保类只有一个实例。推荐实现：枚举单例（线程安全、防反序列化破坏）或静态内部类（懒加载、线程安全）。

**工厂方法模式**：定义创建对象的接口，由子类决定实例化哪个类。Spring 的 `BeanFactory` 是典型应用。

**抽象工厂模式**：创建一系列相关对象（产品族）。如 JDBC 的 `Connection`、`Statement`、`ResultSet` 是一个产品族。

**建造者模式**：分步骤构建复杂对象。`StringBuilder`、Lombok 的 `@Builder`、MyBatis 的 `SqlSessionFactoryBuilder`。

**原型模式**：通过克隆创建对象。Java 的 `Cloneable` 接口，注意深拷贝和浅拷贝的区别。

### 4.2 结构型模式

**代理模式**：为对象提供代理以控制访问。Spring AOP（动态代理）、RPC Stub（静态代理）。

**装饰器模式**：动态添加职责，比继承更灵活。Java IO 流（`BufferedInputStream` 装饰 `FileInputStream`）。

**适配器模式**：将不兼容的接口转换为兼容的接口。Spring MVC 的 `HandlerAdapter`。

**门面模式（Facade）**：提供简化的接口，隐藏子系统复杂性。SLF4J 是日志框架的门面。

**组合模式**：将对象组合成树形结构，统一处理单个对象和组合对象。文件系统、菜单树。

### 4.3 行为型模式

**策略模式**：定义一系列算法，使其可以互换。支付方式选择、排序算法选择。消除大量 if-else。

**观察者模式**：定义对象间一对多的依赖，状态变化时通知所有观察者。Spring 的 `ApplicationEvent`、消息队列。

**模板方法模式**：定义算法骨架，子类实现具体步骤。Spring 的 `JdbcTemplate`、`RestTemplate`。

**责任链模式**：将请求沿处理链传递，每个处理者决定是否处理。Servlet Filter、Spring Interceptor、Netty Pipeline。

**命令模式**：将请求封装为对象，支持撤销、重做、队列。线程池的 `Runnable`/`Callable`。

**状态模式**：对象行为随状态变化而改变，消除大量条件判断。订单状态机、工作流引擎。

### 4.4 架构模式

**SOLID 原则**：

- S（单一职责）：一个类只有一个变化原因
- O（开闭原则）：对扩展开放，对修改关闭
- L（里氏替换）：子类可以替换父类
- I（接口隔离）：接口应该小而专一
- D（依赖倒置）：依赖抽象，不依赖具体实现

**事件驱动架构（EDA）**：通过事件解耦服务，生产者发布事件，消费者订阅事件，天然支持异步和扩展。

**CQRS + Event Sourcing**：不存储当前状态，而是存储所有事件序列，当前状态通过重放事件得到。优点：完整审计日志、支持时间旅行；缺点：查询复杂、存储量大。
