## 1. 分布式理论基础

### 1.1 CAP 定理

CAP 定理指出，分布式系统中以下三个特性最多只能同时满足两个：

- **C（Consistency，一致性）**：所有节点在同一时刻看到相同的数据
- **A（Availability，可用性）**：每个请求都能收到响应（不保证最新数据）
- **P（Partition Tolerance，分区容错性）**：网络分区时系统仍能继续运行

**核心结论**：在分布式系统中，P（分区容错性）是必须保证的（网络分区不可避免），因此只能在 C 和 A 之间取舍：

- **CP 系统**：牺牲可用性，保证一致性（如 ZooKeeper、HBase、etcd）
- **AP 系统**：牺牲强一致性，保证可用性（如 Cassandra、Eureka、DNS）

### 1.2 BASE 理论

BASE 是对 CAP 中 AP 方案的延伸，是互联网系统的实践指导：

- **BA（Basically Available，基本可用）**：允许损失部分可用性（如响应时间延长、降级服务）
- **S（Soft State，软状态）**：允许系统存在中间状态，不要求数据实时一致
- **E（Eventually Consistent，最终一致性）**：经过一段时间后，所有节点数据最终达到一致

**强一致性 vs 最终一致性**：强一致性要求写操作完成后所有读操作立即看到最新值；最终一致性只保证在没有新写入的情况下，最终所有节点数据相同。

### 1.3 Paxos 与 Raft

**Paxos**：分布式一致性的理论基础，但实现复杂，难以理解和工程化。

**Raft**（工程实践主流）：

- 将一致性问题分解为：**Leader 选举**、**日志复制**、**安全性**
- **Leader 选举**：节点分为 Leader、Follower、Candidate 三种角色。Follower 超时未收到心跳则发起选举，获得多数票（N/2+1）成为 Leader
- **日志复制**：客户端写请求发给 Leader，Leader 将日志复制给多数 Follower 后提交，返回客户端
- **任期（Term）**：单调递增的逻辑时钟，每次选举产生新任期，防止脑裂
- 应用：etcd、TiKV、CockroachDB

## 2. 分布式事务

### 2.1 为什么需要分布式事务

微服务架构下，一个业务操作跨越多个服务（如下单扣库存、扣余额、创建订单），每个服务有独立数据库，本地事务无法保证跨服务的原子性。

### 2.2 2PC（两阶段提交）

**流程**：

1. **Prepare 阶段**：协调者向所有参与者发送 Prepare 请求，参与者执行事务但不提交，返回 Yes/No
2. **Commit/Rollback 阶段**：所有参与者都返回 Yes 则发送 Commit，否则发送 Rollback

**缺点**：

- 同步阻塞：参与者在等待协调者指令时锁定资源
- 单点故障：协调者宕机导致参与者永久阻塞
- 数据不一致：Commit 阶段部分参与者收到 Commit 部分未收到

### 2.3 TCC（Try-Confirm-Cancel）

**三个阶段**：

- **Try**：预留资源（如冻结库存、冻结余额），检查业务可行性
- **Confirm**：确认执行（使用预留资源，真正扣减）
- **Cancel**：取消执行（释放预留资源，解冻）

**优点**：无需数据库层面的锁，性能好；业务侵入性强，需要实现三个接口。

**幂等性**：Confirm 和 Cancel 必须实现幂等，防止网络重试导致重复操作。

**悬挂问题**：Cancel 先于 Try 执行（网络延迟），需要在 Cancel 时检查 Try 是否已执行。

**空回滚**：Try 未执行就收到 Cancel，需要正常返回成功（不能报错）。

### 2.4 Saga 模式

将长事务拆分为一系列本地事务，每个本地事务有对应的补偿事务：

- 正向事务：T1 → T2 → T3 → ... → Tn
- 补偿事务：C1 ← C2 ← C3 ← ... ← Cn（某步失败则逆序执行补偿）

**优点**：无锁，高可用，适合长流程业务（如旅行预订）。

**缺点**：不保证隔离性，中间状态对外可见。

### 2.5 消息最终一致性（推荐方案）

**本地消息表**：

1. 业务操作和写消息表在同一本地事务中完成
2. 定时任务扫描消息表，将未发送的消息投递到 MQ
3. 消费者处理消息，处理成功后确认
4. 确认后删除或标记消息表记录

**RocketMQ 事务消息**：

1. 发送半消息（Half Message）到 MQ，MQ 不投递给消费者
2. 执行本地事务
3. 本地事务成功则发送 Commit，失败则发送 Rollback
4. MQ 定期回查事务状态（防止 Commit/Rollback 丢失）

**适用场景**：对实时性要求不高，允许最终一致的场景（如积分发放、通知发送）。

### 2.6 Seata 框架

Seata 提供四种事务模式：AT、TCC、Saga、XA。

**AT 模式（自动补偿，最常用）**：

- 基于 2PC 思想，但通过 undo log 自动生成补偿 SQL
- 一阶段：执行业务 SQL，同时记录 undo log，提交本地事务（不阻塞资源）
- 二阶段提交：删除 undo log；二阶段回滚：执行 undo log 中的补偿 SQL
- 通过全局锁解决写隔离问题，但读隔离需要 `SELECT FOR UPDATE`

## 3. 分布式锁

### 3.1 分布式锁的要求

- **互斥性**：同一时刻只有一个客户端持有锁
- **防死锁**：客户端崩溃后锁能自动释放（超时机制）
- **可重入**：同一客户端可以多次获取同一把锁
- **高可用**：锁服务本身需要高可用
- **高性能**：加锁解锁操作要快

### 3.2 Redis 分布式锁

**基本实现**：`SET key value NX EX seconds`（原子操作，NX=不存在才设置，EX=过期时间）

**释放锁**：必须验证 value 是否是自己的（防止误删他人的锁），用 Lua 脚本保证原子性：

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

**Redisson 看门狗**：持有锁的线程每隔 `lockWatchdogTimeout/3`（默认10秒）自动续期，防止业务未完成锁就过期。

**RedLock 争议**：Martin Kleppmann 认为时钟漂移、GC 停顿等问题会导致 RedLock 失效；Martin Fowler 建议在需要强一致的场景使用 ZooKeeper 或数据库锁。

### 3.3 ZooKeeper 分布式锁

**实现原理**：利用 ZooKeeper 的临时顺序节点：

1. 在锁节点下创建临时顺序节点（如 `/lock/node-0000000001`）
2. 获取所有子节点，如果自己是序号最小的节点，则获得锁
3. 否则监听前一个节点的删除事件，等待唤醒
4. 释放锁时删除自己的节点（或客户端断开连接时自动删除临时节点）

**优点**：天然支持公平锁、可重入、防死锁（临时节点）。

**缺点**：性能不如 Redis（ZooKeeper 写操作需要多数节点确认）。

### 3.4 数据库分布式锁

**乐观锁**：通过版本号（version）实现，更新时检查版本号是否匹配：

```sql
UPDATE table SET value=newVal, version=version+1 
WHERE id=? AND version=oldVersion
```

适合读多写少、冲突少的场景。

**悲观锁**：`SELECT ... FOR UPDATE`，适合写多、冲突多的场景，但性能差。

## 4. 分布式 ID

### 4.1 雪花算法（Snowflake）

64位 ID 结构：1位符号位（0）+ 41位时间戳（毫秒，约69年）+ 10位机器ID（最多1024台）+ 12位序列号（每毫秒最多4096个）。

**特点**：趋势递增、高性能（本地生成，无网络调用）、包含时间信息。

**问题**：时钟回拨会导致 ID 重复。解决方案：检测到时钟回拨时等待或抛出异常；使用逻辑时钟（百度 UidGenerator 方案）。

### 4.2 其他方案

| 方案 | 优点 | 缺点 |
| --- | --- | --- |
| 数据库自增 | 简单 | 性能瓶颈，单点故障 |
| 数据库号段 | 减少 DB 压力（批量取） | 重启丢失号段 |
| UUID | 简单，无中心 | 无序，不适合主键 |
| 雪花算法 | 高性能，趋势递增 | 依赖时钟，机器ID管理 |
| 美团 Leaf | 双缓冲号段，高可用 | 部署复杂 |

## 5. 分布式限流

### 5.1 限流算法

**固定窗口**：统计固定时间窗口内的请求数，超过阈值则拒绝。问题：窗口边界突刺（两个窗口交界处可能有2倍流量）。

**滑动窗口**：将时间窗口细分为多个小格，滑动统计，解决边界突刺问题。

**漏桶算法**：请求进入漏桶，以固定速率流出处理，超出容量则丢弃。强制平滑输出，无法应对突发流量。

**令牌桶算法**（推荐）：以固定速率向桶中放令牌，请求需要消耗令牌，桶满时令牌溢出。允许一定程度的突发流量（消耗积累的令牌）。Guava RateLimiter 实现了令牌桶。

### 5.2 分布式限流实现

**Redis + Lua 脚本**：利用 Redis 原子性实现分布式计数器，支持滑动窗口和令牌桶。

**Sentinel**：阿里开源，支持 QPS 限流、并发线程数限流、系统自适应保护，提供实时监控和规则动态推送。

**Nginx 限流**：`limit_req_zone` 和 `limit_req` 指令，在接入层实现限流，适合简单场景。
