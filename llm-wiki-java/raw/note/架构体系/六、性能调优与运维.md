## 1. JVM 性能调优

### 1.1 JVM 调优目标

调优的本质是在**吞吐量**、**延迟**、**内存占用**三者之间取得平衡：

- 吞吐量优先：减少 GC 频率，允许较长 STW（批处理、后台任务）
- 延迟优先：减少 STW 时间，允许更频繁的 GC（在线服务、实时系统）

### 1.2 常用 JVM 参数

**堆内存设置**：

```
-Xms4g              # 初始堆大小（建议与-Xmx相同，避免动态扩容）
-Xmx4g              # 最大堆大小
-Xmn2g              # 新生代大小（建议堆的1/3~1/2）
-XX:MetaspaceSize=256m      # 元空间初始大小
-XX:MaxMetaspaceSize=512m   # 元空间最大大小
```

**GC 选择**：

```
-XX:+UseG1GC                    # 使用 G1（JDK9+ 默认）
-XX:MaxGCPauseMillis=200        # G1 目标停顿时间（毫秒）
-XX:+UseZGC                     # 使用 ZGC（JDK15+ 正式，超低延迟）
-XX:+UseConcMarkSweepGC         # 使用 CMS（JDK9 废弃）
```

**GC 日志**：

```
-XX:+PrintGCDetails             # 打印 GC 详情
-XX:+PrintGCDateStamps          # 打印 GC 时间戳
-Xloggc:/path/to/gc.log         # GC 日志文件
-XX:+UseGCLogFileRotation       # 日志轮转
```

**OOM 时 Dump**：

```
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/path/to/dump.hprof
```

### 1.3 GC 调优实践

**GC 问题排查步骤**：

1. 查看 GC 日志，分析 GC 频率和停顿时间
2. 使用 `jstat -gcutil <pid> 1000` 实时监控 GC 情况
3. 分析 Eden/Survivor/Old 各区使用率和晋升速率
4. 根据问题类型调整参数

**常见 GC 问题**：

- **Full GC 频繁**：老年代空间不足（增大堆或优化对象生命周期）；元空间不足（增大 MetaspaceSize）
- **Young GC 频繁**：新生代太小（增大 -Xmn）；对象创建速率过高（优化代码减少对象创建）
- **GC 停顿时间长**：切换到 G1/ZGC；调整 `-XX:MaxGCPauseMillis`

### 1.4 内存泄漏排查

**常见内存泄漏场景**：

- 静态集合持有对象引用（如静态 Map 不断添加不删除）
- 未关闭的资源（Connection、Stream）
- ThreadLocal 未 remove（线程池场景）
- 监听器/回调未注销
- 内部类持有外部类引用

**排查工具**：

- `jmap -heap <pid>`：查看堆内存使用情况
- `jmap -histo <pid>`：查看对象实例数量（按类排序）
- `jmap -dump:format=b,file=heap.hprof <pid>`：导出堆快照
- **MAT（Memory Analyzer Tool）**：分析 heap dump，找到内存泄漏的 GC Root 引用链
- **Arthas**：线上诊断神器，不停机分析

### 1.5 CPU 问题排查

**CPU 100% 排查步骤**：

1. `top` 找到 CPU 占用高的进程 PID
2. `top -Hp <pid>` 找到 CPU 占用高的线程 TID
3. `printf '%x\n' <TID>` 将 TID 转为十六进制
4. `jstack <pid> | grep -A 30 '<十六进制TID>'` 找到对应线程堆栈
5. 分析堆栈，定位代码

**常见 CPU 高原因**：死循环、频繁 GC（GC 线程占用 CPU）、正则表达式回溯、序列化/反序列化。

## 2. 全链路监控

### 2.1 可观测性三大支柱

**Metrics（指标）**：数值型时序数据，如 QPS、RT、错误率、CPU 使用率。工具：Prometheus + Grafana。

**Tracing（链路追踪）**：记录请求在分布式系统中的完整调用链路，分析性能瓶颈。工具：SkyWalking、Jaeger、Zipkin。

**Logging（日志）**：记录系统运行事件，用于问题排查。工具：ELK（Elasticsearch + Logstash + Kibana）、EFK（Elasticsearch + Fluentd + Kibana）。

### 2.2 SkyWalking

**核心概念**：

- **Trace**：一次完整的请求链路，由多个 Span 组成
- **Span**：一次操作（如 HTTP 请求、DB 查询、MQ 消息），包含开始时间、结束时间、标签
- **Segment**：一个服务实例内的 Span 集合
- **TraceId**：全局唯一，贯穿整个链路

**SkyWalking 架构**：

- Agent（探针）：字节码增强，无侵入，自动采集链路数据
- OAP Server（Observability Analysis Platform）：接收、分析、存储数据
- Storage：Elasticsearch（推荐）/ MySQL / TiDB
- UI：可视化展示链路、拓扑图、指标

**关键指标**：P50/P95/P99 响应时间（百分位数）、错误率、吞吐量（CPM）。

### 2.3 Prometheus + Grafana

**Prometheus 数据模型**：时序数据，由 metric name + labels 唯一标识。

**四种 Metric 类型**：

- Counter：单调递增计数器（如请求总数）
- Gauge：可增可减的数值（如内存使用量）
- Histogram：分桶统计（如响应时间分布）
- Summary：百分位数统计

**告警规则**：通过 AlertManager 配置告警规则，支持邮件、钉钉、Slack 等通知渠道。

## 3. 容器化与云原生

### 3.1 Docker 核心原理

**容器 vs 虚拟机**：

- 虚拟机：硬件虚拟化，每个 VM 有独立 OS，资源隔离强，启动慢（分钟级），资源开销大
- 容器：OS 级虚拟化，共享宿主机内核，启动快（秒级），资源开销小，隔离性略弱

**Docker 核心技术**：

- **Namespace**：进程隔离（PID、Network、Mount、UTS、IPC、User）
- **Cgroups**：资源限制（CPU、内存、磁盘 IO）
- **UnionFS**：联合文件系统，镜像分层存储，层复用节省空间

**镜像优化**：

- 使用小基础镜像（alpine 约5MB vs ubuntu 约70MB）
- 多阶段构建（Multi-stage Build）：编译阶段和运行阶段分离
- 合并 RUN 指令减少层数
- .dockerignore 排除不必要文件

### 3.2 Kubernetes 核心概念

**核心资源对象**：

| 资源 | 说明 | 用途 |
| --- | --- | --- |
| Pod | 最小调度单元，包含一个或多个容器 | 运行容器 |
| Deployment | 管理无状态 Pod 的副本集 | 无状态服务部署、滚动更新 |
| StatefulSet | 管理有状态 Pod，保证稳定网络标识和存储 | 数据库、中间件 |
| DaemonSet | 每个节点运行一个 Pod | 日志采集、监控 Agent |
| Service | 为 Pod 提供稳定的访问入口（ClusterIP/NodePort/LoadBalancer） | 服务发现和负载均衡 |
| Ingress | HTTP/HTTPS 路由规则 | 外部流量入口 |
| ConfigMap | 非敏感配置 | 配置管理 |
| Secret | 敏感配置（密码、Token） | 密钥管理 |
| HPA | 水平自动扩缩容 | 根据 CPU/内存自动扩缩 |

**K8s 调度原理**：

1. 过滤（Filtering）：找出满足 Pod 要求的节点（资源、亲和性、污点等）
2. 打分（Scoring）：对候选节点打分（资源均衡、亲和性偏好等）
3. 选择得分最高的节点

**滚动更新策略**：

- `maxSurge`：更新过程中最多额外创建的 Pod 数量
- `maxUnavailable`：更新过程中最多不可用的 Pod 数量
- 蓝绿部署：同时维护两套环境，切换流量实现零停机
- 金丝雀发布：逐步将流量切换到新版本（如先切5%，观察无问题再全量）

### 3.3 CI/CD 流水线

**标准流水线**：代码提交 → 代码检查（SonarQube）→ 单元测试 → 构建镜像 → 推送镜像仓库 → 部署测试环境 → 集成测试 → 部署生产环境（审批）

**工具链**：

- 代码仓库：GitLab / GitHub
- CI/CD：Jenkins / GitLab CI / GitHub Actions / Argo CD
- 镜像仓库：Harbor / Docker Hub / ACR
- 配置管理：Helm（K8s 包管理器）/ Kustomize

## 4. 线上问题排查

### 4.1 排查思路

**黄金法则**：先止损，再排查。高流量场景下先限流/降级保住系统，再慢慢分析根因。

**排查工具箱**：

- `top` / `htop`：CPU、内存、负载
- `iostat -x 1`：磁盘 IO
- `netstat -an | grep ESTABLISHED | wc -l`：TCP 连接数
- `ss -s`：Socket 统计
- `tcpdump`：抓包分析
- `strace`：系统调用追踪
- `Arthas`：Java 线上诊断（watch、trace、monitor、jad）

### 4.2 Arthas 常用命令

```
dashboard     # 实时监控系统状态（CPU、内存、线程）
thread -n 5   # 显示 CPU 占用最高的5个线程
jad com.example.Service  # 反编译类（确认是否部署了最新代码）
watch com.example.Service method '{params,returnObj,throwExp}' -x 2  # 观察方法入参和返回值
trace com.example.Service method  # 追踪方法调用链路和耗时
monitor com.example.Service method -c 5  # 每5秒统计方法调用次数、成功率、RT
ognl "@com.example.Config@instance.timeout"  # 执行 OGNL 表达式（查看静态变量）
```
