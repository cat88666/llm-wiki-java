## 1. MySQL 深度

### 1.1 存储引擎

**InnoDB vs MyISAM**：

| 特性 | InnoDB | MyISAM |
| --- | --- | --- |
| 事务 | 支持 ACID | 不支持 |
| 外键 | 支持 | 不支持 |
| 行级锁 | 支持 | 仅表锁 |
| 崩溃恢复 | 支持（redo log） | 不支持 |
| 全文索引 | 5.6+ 支持 | 支持 |
| count(*) | 全表扫描 | 直接返回（存储行数） |

**InnoDB 为什么推荐自增主键**：自增主键保证插入时页面顺序写入，避免页分裂；UUID 等随机主键导致频繁页分裂，影响写性能。

### 1.2 索引原理

**B+ 树索引结构**：

- 所有数据存储在叶子节点，叶子节点通过双向链表连接（支持范围查询）
- 非叶子节点只存储键值，不存储数据，单个节点可存储更多键值（树高更低）
- 典型 B+ 树高度为 3，可存储约 2000 万条记录

**聚簇索引 vs 非聚簇索引**：

- 聚簇索引（主键索引）：叶子节点存储完整行数据，每张表只有一个
- 非聚簇索引（二级索引）：叶子节点存储主键值，查询需要**回表**（再次查聚簇索引）
- 覆盖索引：查询字段全部在索引中，无需回表

**索引失效场景**（面试高频）：

1. 最左前缀原则被破坏（联合索引必须从最左列开始）
2. 对索引列做函数运算或类型转换（如 `WHERE YEAR(create_time) = 2024`）
3. 使用 `!=`、`NOT IN`、`NOT LIKE`（无法利用索引定位）
4. `LIKE '%xxx'`（前缀模糊无法使用索引，`xxx%` 可以）
5. OR 条件中有非索引列
6. 数据量少时优化器选择全表扫描

**索引设计原则**：

- 区分度高的列优先建索引（性别不适合，用户ID适合）
- 频繁查询的列、WHERE/ORDER BY/GROUP BY 涉及的列
- 联合索引遵循最左前缀，将区分度高的列放左边
- 索引不是越多越好，影响写性能（每次写操作需维护索引）

### 1.3 事务与锁

**ACID 特性**：

- **原子性（Atomicity）**：undo log 保证，事务回滚时撤销所有操作
- **一致性（Consistency）**：由其他三个特性共同保证
- **隔离性（Isolation）**：锁机制 + MVCC 保证
- **持久性（Durability）**：redo log 保证，崩溃恢复

**四种隔离级别**：

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 实现方式 |
| --- | --- | --- | --- | --- |
| READ UNCOMMITTED | 可能 | 可能 | 可能 | 无锁 |
| READ COMMITTED | 不可能 | 可能 | 可能 | 每次读取最新快照 |
| REPEATABLE READ（默认） | 不可能 | 不可能 | 基本解决 | 事务开始时创建快照+间隙锁 |
| SERIALIZABLE | 不可能 | 不可能 | 不可能 | 串行执行 |

**MVCC（多版本并发控制）**：

- 每行记录有两个隐藏字段：`trx_id`（最近修改的事务ID）和 `roll_pointer`（指向 undo log 的指针）
- 读取时根据 ReadView 判断哪个版本可见：事务开始时创建 ReadView，包含当前活跃事务列表
- RC 隔离级别：每次 SELECT 创建新 ReadView（所以能读到已提交的新数据）
- RR 隔离级别：事务第一次 SELECT 时创建 ReadView，整个事务复用（所以不可重复读）

**InnoDB 锁类型**：

| 锁类型 | 说明 | 场景 |
| --- | --- | --- |
| 行锁（Record Lock） | 锁定单行记录 | 精确匹配索引 |
| 间隙锁（Gap Lock） | 锁定索引间隙，防止插入 | RR 级别防幻读 |
| 临键锁（Next-Key Lock） | 行锁+间隙锁，锁定左开右闭区间 | RR 级别默认 |
| 表锁 | 锁定整张表 | DDL 操作 |
| 意向锁 | 表级锁，表示即将加行锁 | 行锁与表锁兼容性检查 |

**死锁产生条件**：互斥、占有并等待、不可剥夺、循环等待。InnoDB 通过死锁检测（`innodb_deadlock_detect`）自动回滚代价小的事务。

### 1.4 redo log、undo log、binlog

**redo log（重做日志）**：

- InnoDB 特有，物理日志，记录数据页的物理修改
- WAL（Write-Ahead Logging）：先写日志再写磁盘，保证持久性
- 循环写入，固定大小，写满后覆盖（`checkpoint` 机制）
- 崩溃恢复时重放 redo log 恢复未持久化的数据

**undo log（回滚日志）**：

- 逻辑日志，记录数据修改前的值
- 保证原子性（事务回滚）和 MVCC（读取历史版本）
- 存储在系统表空间或独立 undo 表空间

**binlog（二进制日志）**：

- Server 层日志，所有存储引擎共享
- 记录所有 DDL 和 DML 操作（逻辑日志）
- 用于主从复制和数据恢复
- 三种格式：STATEMENT（SQL语句）、ROW（行变更，默认）、MIXED

**两阶段提交（2PC）**：保证 redo log 和 binlog 的一致性：

1. Prepare 阶段：写 redo log，状态为 prepare
2. Commit 阶段：写 binlog，然后将 redo log 状态改为 commit

### 1.5 SQL 优化

**EXPLAIN 关键字段**：

- `type`：访问类型，优劣顺序：`system > const > eq_ref > ref > range > index > ALL`
- `key`：实际使用的索引
- `rows`：预估扫描行数
- `Extra`：`Using index`（覆盖索引）、`Using filesort`（文件排序，需优化）、`Using temporary`（临时表，需优化）

**慢查询优化思路**：

1. EXPLAIN 分析执行计划，确认是否走索引
2. 检查索引是否合理（是否命中、是否需要联合索引）
3. 检查是否有回表（考虑覆盖索引）
4. 检查数据量（考虑分页、分表）
5. 检查 JOIN 是否合理（小表驱动大表）

## 2. Redis 核心原理

### 2.1 数据结构

**五种基本数据类型**：

| 类型 | 底层实现 | 典型场景 |
| --- | --- | --- |
| String | SDS（简单动态字符串） | 缓存、计数器、分布式锁 |
| List | quicklist（ziplist+双向链表） | 消息队列、最新列表 |
| Hash | ziplist（小）/ hashtable（大） | 对象存储、购物车 |
| Set | intset（纯整数）/ hashtable | 标签、共同好友、抽奖 |
| ZSet（Sorted Set） | ziplist（小）/ skiplist+hashtable | 排行榜、延迟队列 |

**特殊数据类型**：

- **Bitmap**：基于 String，按位操作，用于用户签到、在线状态
- **HyperLogLog**：基数统计，误差约0.81%，12KB 内存统计亿级 UV
- **Geo**：地理位置，底层 ZSet，支持附近的人、距离计算
- **Stream**：消息流，支持消费者组，类似 Kafka

**跳表（skiplist）原理**：

- 多层链表，每层是下层的子集，最底层包含所有元素
- 查找时从最高层开始，逐层向下，时间复杂度 O(log n)
- 相比红黑树：实现简单、范围查询更快、内存占用相近

### 2.2 持久化

**RDB（快照）**：

- 将内存数据以二进制快照形式保存到磁盘
- `BGSAVE`：fork 子进程，利用 COW（写时复制）机制，不阻塞主进程
- 优点：文件小、恢复快；缺点：可能丢失最后一次快照后的数据

**AOF（追加日志）**：

- 记录每条写命令，重启时重放恢复数据
- 三种 fsync 策略：`always`（每条命令，最安全）、`everysec`（每秒，默认，最多丢1秒）、`no`（OS 决定）
- AOF 重写：`BGREWRITEAOF`，将内存数据重新生成最小化 AOF 文件

**RDB vs AOF**：

| 特性 | RDB | AOF |
| --- | --- | --- |
| 文件大小 | 小（二进制压缩） | 大（文本命令） |
| 恢复速度 | 快 | 慢（需重放命令） |
| 数据安全性 | 低（可能丢失分钟级） | 高（最多丢1秒） |
| 性能影响 | fork 时有短暂影响 | fsync 有持续影响 |

**混合持久化（Redis 4.0+）**：AOF 重写时，先写 RDB 格式的全量数据，再追加增量 AOF 命令，兼顾两者优点。

### 2.3 高可用架构

**主从复制**：

- 全量复制：slave 第一次连接，master 执行 BGSAVE 生成 RDB，发送给 slave
- 增量复制：断线重连后，通过 replication offset 同步缺失的命令（Redis 2.8+）
- 复制延迟：异步复制，主从之间存在延迟，可能读到旧数据

**哨兵模式（Sentinel）**：

- 监控 master/slave 健康状态
- 自动故障转移：master 下线后，从 slave 中选举新 master
- 客户端通过哨兵获取 master 地址，实现透明切换
- 主观下线（SDOWN）：单个哨兵认为节点下线；客观下线（ODOWN）：多数哨兵认为节点下线

**Redis Cluster（集群）**：

- 数据分片：16384 个 slot，每个节点负责一部分 slot
- 哈希槽分配：`CRC16(key) % 16384`
- 无中心化架构，节点间通过 Gossip 协议通信
- 支持自动故障转移，每个 master 至少需要一个 slave
- 客户端重定向：访问错误节点时返回 MOVED/ASK 指令

### 2.4 缓存问题与解决方案

**缓存穿透**：查询不存在的数据，每次都打到数据库。

- 解决：布隆过滤器（Bloom Filter）拦截不存在的 key；缓存空值（设置短过期时间）

**缓存击穿**：热点 key 过期，大量请求同时打到数据库。

- 解决：互斥锁（只允许一个请求重建缓存）；逻辑过期（不设 TTL，异步更新）

**缓存雪崩**：大量 key 同时过期，或 Redis 宕机，请求全部打到数据库。

- 解决：过期时间加随机值；Redis 高可用（哨兵/集群）；限流降级；多级缓存

**缓存与数据库一致性**：

- **Cache-Aside（旁路缓存）**：读时先查缓存，未命中再查 DB 并写缓存；写时先更新 DB，再删除缓存（不是更新缓存）
- **为什么删缓存而不是更新缓存**：避免并发写时缓存与 DB 不一致；删除是幂等操作
- **延迟双删**：更新 DB 前删缓存，更新 DB 后延迟再删一次，减少不一致窗口
- **终极方案**：Canal 监听 MySQL binlog，异步更新缓存，保证最终一致性

### 2.5 Redis 分布式锁

**SETNX 实现**：`SET key value NX EX seconds`（原子操作）

**Redisson 分布式锁**：

- 基于 Lua 脚本保证原子性
- 看门狗机制：持有锁的线程每10秒自动续期（默认30秒过期）
- 支持可重入锁、读写锁、公平锁

**RedLock 算法**：向 N 个独立 Redis 节点申请锁，超过 N/2+1 个节点成功才算获锁成功，解决单点故障问题（但存在争议，Martin Kleppmann 指出时钟漂移等问题）。

## 3. 分库分表

### 3.1 为什么需要分库分表

- 单表数据量超过 1000 万行，查询性能明显下降
- 单库 QPS 瓶颈（通常 MySQL 单库 QPS 上限约 1-2 万）
- 存储容量瓶颈

**分表 vs 分库**：

- 分表：解决单表数据量过大的问题，仍在同一数据库
- 分库：解决数据库连接数和 QPS 瓶颈，数据分散到多个数据库实例

### 3.2 分片策略

**水平分片（Sharding）**：

- 范围分片（Range）：按 ID 范围分，如 0-100万在 shard1，实现简单但可能热点
- 哈希分片（Hash）：`shard = hash(shardKey) % shardCount`，数据均匀但扩容困难
- 一致性哈希：扩容时只迁移少量数据，但实现复杂

**垂直分片**：按业务模块拆分，如用户库、订单库、商品库。

### 3.3 分库分表的问题

**跨库 JOIN**：禁止跨库 JOIN，通过冗余字段、数据同步或应用层聚合解决。

**分布式事务**：跨库操作无法使用本地事务，需要分布式事务方案（Seata、TCC、消息最终一致性）。

**全局唯一 ID**：不能再用数据库自增 ID，需要分布式 ID 方案：

- **雪花算法（Snowflake）**：64位 = 1位符号 + 41位时间戳 + 10位机器ID + 12位序列号，每毫秒可生成 4096 个 ID，趋势递增
- 美团 Leaf、百度 UidGenerator 等改进方案

**分页查询**：跨分片分页需要在每个分片查询后合并排序，深分页性能差。优化：禁止跳页，只允许上下翻页；或使用游标分页。

**扩容迁移**：数据迁移是最大挑战，需要双写方案（同时写新旧分片）+ 数据校验 + 灰度切流。

### 3.4 主流分库分表框架

| 框架 | 类型 | 特点 |
| --- | --- | --- |
| ShardingSphere-JDBC | 客户端 | 无需部署代理，性能好，与应用耦合 |
| ShardingSphere-Proxy | 代理 | 透明代理，支持多语言，有额外网络开销 |
| MyCat | 代理 | 国内流行，功能全面 |
