### **1. 索引体系（Index System）**

**1.1 索引的本质：B+Tree 特点**

- 所有数据在叶子节点，天然有序
- 非叶子节点只存键，不存数据（减少 IO）
- 叶子节点链表结构 → 范围查询极快
- B+Tree 更适合磁盘顺序读写（页结构友好）
- B-tree 每层存储大量 data，导致树更高，IO 更多

**1.2 索引分类**

- **主键索引（聚簇索引）**
- **唯一索引（unique）**
- **普通索引（normal）**
- **联合索引（composite）:** 多字段组成索引、需最左前缀原则；
- **覆盖索引（covering index）：**通过索引值可以直接找到要查询字段值不需要通过主键值回表查询，那么就叫覆盖索引

**1.3 最左前缀原则：**联合索引 `(a, b, c)` 可使用的顺序：**第一个字段不能用范围，否则后面的字段失效**

**1.4 覆盖索引（Covering Index）：**查询字段全部包含在索引中 → 不回表 → 极快

```sql
select name, age from user where name = 'Tom';
索引：idx_name_age(name, age)
→ 直接在叶子节点取值，无需回表。
```

**1.5 索引下推优化ICP （Index Condition Pushdown）**

在 MySQL 5.6 之后：在索引遍历阶段就进行数据过滤，减少回表次数

```sql
where name like 'Li%' and age = 22 and position='manager'
idx_name_age_position)
ICP在索引层过滤 age、position，过滤掉大量不符合条件的行，减少回表次数。
**ICP失效：**
- 主键索引（聚簇索引本来就包含全行数据）
- 范围查询（结果集过大，MySQL 认为优化无意义）
```

**1.6 In / Or 是否走索引？**

- 表记录多 → 走索引
- 表记录少 → 全表扫描更快（不走索引）

**成本优化器自动判断**。

---

**1.7 强制索引（force index）:**强制索引 **未必更快,**因为回表次数可能非常大

```sql
select * from user force index(idx_a_b) where a > 10;
```

### **2. 查询优化（Slow Query Optimization）**

**2.1 慢查询的核心优化步骤**

1. 检查是否走索引
2. 索引选择是否最优
3. 是否出现回表过多
4. select * 是否查询过多字段
5. 排查是否需要分库分表
6. 服务器硬件瓶颈（CPU / IOPS）

**2.2 MySQL使用 Buffer Pool**大幅提升并发能力

- 直接更新磁盘：随机 IO 极慢
- 直接更新磁盘：并发能力下降
- **修改先写 Buffer Pool（内存）**
- **日志顺序写入 Redo log**
- 后台异步刷盘

### **3. 事务体系（Transaction System）**

**3.1 ACID**

- **Atomicity** 原子性：undo log
- **Consistency** 一致性：各种约束
- **Isolation** 隔离性：MVCC + 锁
- **Durability** 持久性：redo log

---

**3.2 四种隔离级别**

| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
| --- | --- | --- | --- |
| 读未提交 | ✔ | ✔ | ✔ |
| 读提交 | ❌ | ✔ | ✔ |
| 可重复读（MySQL默认） | ❌ | ❌ | ✔（由间隙锁解决） |
| 串行化 | ❌ | ❌ | ❌ |

### **4. MVCC（多版本并发控制）**

MVCC = **ReadView + Undo Log 版本链**

**4.1 Undo Log 版本链**

每次 update：

- 新值写入数据页
- 旧值进入 undo log
- roll_pointer 指向上一版本
- 形成一条“版本链”

**4.2 Read View**

关键字段：

- `min_trx_id`
- `max_trx_id`
- `活跃事务列表`

**4.3 MVCC 可见性规则（核心）**

从版本链头开始遍历：

1. **trx_id < min_trx_id** → ✔ 可见（已提交）
2. **trx_id > max_trx_id** → ✘ 不可见（未来）
3. **min ≤ trx_id ≤ max**
    - 在活跃列表中 → ✘ 不可见
    - 不在活跃列表 → ✔ 可见

### **5. 锁机制（Locks）**

**5.1 按粒度分类**

- **表锁**
- **行锁（Record Lock）**
- **间隙锁（Gap Lock）**
- **Next-Key Lock（记录锁 + 间隙锁）**

**5.2 按读写分类**

- **S 锁（共享锁）**
- **X 锁（排他锁）**

**5.3 行锁升级为表锁**

- 无索引条件查询
- 索引失效（函数、类型转换）
- 范围过大

**5.4 锁优化建议**

- 所有查询尽量使用索引
- 缩小检索范围
- 减少事务时间
- 降低隔离级别
- 修改语句尽量放事务末尾

### **6. 死锁分析**

检测到死锁后：

- MySQL 选择 **回滚“成本最小”的事务”**

优化建议：

- 统一访问顺序
- 小事务
- 使用 `select ... for update` 明确锁范围
- 避免大范围扫描

### **7. MySQL 架构优化**

**7.1 主从复制**

- 主库：写
- 从库：读
- 优点：提升读性能

**7.2 读写分离**

- N 个读库分担读压力
- 主库只写

**7.3 分库分表**

适用于：

- 数据量 > 5000 万
- 高并发热点表
- 超大宽表

### **8. Spring 事务传播机制（七种）**

| 传播行为 | 说明 |
| --- | --- |
| REQUIRED（默认） | 没有就建，有就加入 |
| REQUIRES_NEW | 无论如何都新建事务 |
| NESTED | 当前事务中再开启子事务 |
| SUPPORTS | 有事务就加入，没有就无事务 |
| NOT_SUPPORTED | 有事务就挂起，非事务执行 |
| NEVER | 有事务就报错 |
| MANDATORY | 必须要有事务，否则报错 |

### **9. MySQL CPU 飙升案例优化指南**

**主库 CPU 高 → 解决方案**

1. 归档历史数据
2. 优化 TOP10 慢 SQL
3. 大字段拆表
4. 读写分离
5. 减少回表

**从库 CPU 100% → 解决方案**

1. 限制导出数量
2. 限制导入数量
3. 单表查询替代 join
4. 减少不必要查询
5. 改写时间范围查询
