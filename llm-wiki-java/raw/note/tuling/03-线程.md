**二、📌 线程池**

**📌 线程池本质**

  0、池化技术提前保存大量资源，任务队列（BlockingQueue） + 工作线程（Worker） + 管理者（ThreadPoolExecutor）。

1. 线程池线程数量 ≤ **corePoolSize** 创建核心线程执行任务。
2. 线程池线程数量 > **corePoolSize** 新任务进入**阻塞队列**。
3. 当阻塞队列**满了&** 线程池线程数 ≤ **maximumPoolSize** 创建**非核心线程**执行任务。
4. 当阻塞队列**满了&** 线程池线程数 > **maximumPoolSize** 触发**拒绝策略handler**。
5. 当线程池线程数 > **corePoolSize** ，多的线程空闲时间超过**keepAliveTime**被回收。

**📌 常用线程池**

- **FixedThreadPool：**`LinkedBlockingQueue`**无界队列**OOM风险；
- **SingleThreadExecutor：**单线程顺序执行任务`LinkedBlockingQueue`**无界队列**OOM风险；
- **CachedThreadPool：**线程数无限制 ****，OOM风险；
- **ScheduledThreadPool：**线程数有限，**DelayedWorkQueue**支持定时和周期性任务，但定时任务异常可能导致任务悄悄停止执行。
- **线程池中 submit() 和 execute() 的区别：**`execute`提交Runnable无返回值，`submit(Callable<T> task)` / `submit(Runnable task)`返回`Future`对象结果

**📌 线程池配置**

**📌 线程池配置**

```
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    Runtime.getRuntime().availableProcessors(),    // 核心线程数 = CPU核心数
    Runtime.getRuntime().availableProcessors() * 2, // 最大线程数
    30L, TimeUnit.SECONDS,                         // 非核心线程空闲存活时间
    new LinkedBlockingQueue<>(10000),              // 有界队列避免OOM(预估峰值TPS × 最大容忍延迟时间(s) × 扩容系数(1.5~2))
    new NamedThreadFactory("business-pool"),       // 命名线程工厂
    new LogAndMetricsRejectPolicy()                // 监控+告警的拒绝策略
);

线程池CUP核心数公式：
1. CPU密集型任务（大量计算）：CPU核心数 + 1
2. IO密集型任务（大量磁盘/网络 IO）：CPU核心数 × 2+1

线程池队列已满：
1、无界队列（LinkedBlockingQueue默认构造）：不会触发创建非核心线程，不会触发拒绝策略，OOM风险；
2、有界队列（ArrayBlockingQueue，LinkedBlockingQueue）小于maximumPoolSize创建新线程，maximumPoolSize触发拒绝策略；
    
线程池四种拒绝策略：
1. AbortPolicy（默认）
    - 直接抛出 `RejectedExecutionException`，让调用方感知到异常。
2. CallerRunsPolicy
    - 由提交任务的线程自己执行该任务，既不抛异常也不丢弃。
3. DiscardPolicy
    -直接丢弃任务，不抛异常。
4. **DiscardOldestPolicy**：
    - 丢弃阻塞队列中最旧的任务，然后尝试重新提交当前任务。
```

**📌 线程池优雅停机配置**

```
@Component
public class GracefulShutdown implements DisposableBean {
    @Override
    public void destroy() throws Exception {
        // 1. 停止接收新任务，不再接收新任务，但会把队列中已提交的任务执行完毕。
        executor.shutdown();
        // 2. 等待正在执行的任务完成（第一阶段等待）
        if (!executor.awaitTermination(10, SECONDS)) {
            // 3. 发送中断信号（第二阶段：强制中断，并返回队列中尚未开始执行的任务列表）
            executor.shutdownNow();
            // 4. 最终等待
            if (!executor.awaitTermination(5, SECONDS)) {
                log.error("线程池未能正常关闭");
            }
        }
        // 5. 记录未完成任务到持久化存储
        savePendingTasks();
    }
}
```

### **三、📌 并发本质(JMM)**

**JMM三要素**

Java 并发来自 JMM 三大特性失效,JMM 通过工作内存 + 主内存模型、内存屏障和 happens-before 规则来重新构建跨线程的数据一致性语义。

Java并发不是锁问题是内存模型问题；锁、volatile、final 都是实现 happens-before 的不同手段。

**1. 原子性**

- 复合操作（i++、check-then-act、read-modify-write）不是原子的。要求：操作不可拆分不被中断
- JMM只保证基本读写是原子的（理论上 long/double 可能不是 64bit 原子，但现实现代 JVM 都是原子的）。

**2. 可见性**

- Java 使用工作内存(线程本地缓存)+主内存模型。线程不会直接读写主内存需要刷新和失效机制。
- 可见性：依靠“写刷新 + 读失效 + 内存屏障”

**3. 有序性:** 编译器优化、CPU重排会“改变执行顺序”， JMM提供**happens-before**来保证逻辑上的顺序。

**JMM三要素保证**

- **volatile：保证可见性与部分有序性,**写强制刷新到主内存，并禁止一定重排，读强制从主内存加载最新值。
- **synchronized / Lock：**CAS 保证原子性**,**锁释放：把工作内存的写入刷新到主内存，锁获取：让工作内存的缓存失效强制从主内存重新加载。
- **final：**构造函数结束前完成写 → 对其他线程只可见一次并且不可变。
- 有序性：线程启动/终止规则（start/ join），程序次序规则（单线程内不能重排影响语义）

**四、📌 锁体系**

**📌 Synchronized**

**Synchronized对象头锁升级流程：**无锁 → 偏向锁(**Lock**) → 轻量级锁(**自旋**) → 重量级锁(**锁膨胀**)

- **偏向锁：**Thread ID 写入对象头，单线程场景下无竞争，性能最优
- **轻量级锁：**CAS 竞争失败后，自旋尝试获取锁不立即阻塞
- **重量级锁：**自旋次数过多 → OS 监视器 → 阻塞 / 唤醒（park/unpark）

**📌 ReentrantLock**

ReentrantLock、公平锁、非公平锁、Semaphore、CountDownLatch 等同步器本质上都基于 AQS。AQS 背后是一个 CLH 队列 + CAS 修改的 volatile state + park/unpark 的阻塞唤醒机制。公平与非公平的区别发生在抢锁时是否允许插队。CAS 提供了所有无锁更新的基础，但存在 ABA 和自旋开销。整体上，AQS 统一了 Java 的所有同步器设计，让开发者只需关注 tryAcquire/tryRelease 之类的模板方法即可。

- **公平锁**：加锁前先检查队列是否有前驱节点 → 排队机制(AQS FIFO队列) → 遵循先来后到(释放锁后按队列顺序唤醒)。
- **非公平锁**：加锁时直接CAS抢占锁，不管队列里是否有人排队 → 性能更好。
- **可重入锁：**AQS  `state` 计数器实现，同一线程再次获得锁 → `state+1`，不会阻塞。
- **ReentrantLock `lock**()`：阻塞式获取锁，获取不到 AQS 队列等待`lockInterruptibly()` 支持中断。
- **ReentrantLock** `tryLock()`：立即返回 true/false，完全不阻塞。
- **ReentrantLock** `tryLock(timeout)`：支持超时等待，是分布式锁 / 解决死锁的关键手段。

**📌 ReentrantLock & Synchronized**

**ReentrantLock 给更多控制权；synchronized 更简单稳定。**

| 维度 | synchronized | ReentrantLock |
| --- | --- | --- |
| 实现层面 | JVM 关键字 | JDK AQS |
| 锁语义 | 自动管理 | 手动 lock/unlock |
| 公平性 | 固定非公平 | 可选公平/非公平 |
| 可中断 | 不可中断 | 可中断lockInterruptibly() |
| 尝试锁 | 不支持 | tryLock() |
| 多条件队列 | 不支持 | Condition 支持多队列 |
| 性能 | 轻锁竞争优 | 高竞争更优,**多条件队列** |

**📌 死锁**

- **死锁四条件：**资源互斥、占有等待不释放资源、已占有资源不可剥夺、循环等待资源
- **避免死锁：**锁按资源排序、tryLock超时、死锁检测ThreadMXBean.findDeadlockedThreads()，jstack / arthas dump
- 线程同步：多个线程按照顺序访问共享资源：Synchronized、ReentrantLock、Semaphore、

**📌  CAS & AQS**

**📌 CAS(Compare-And-Swap):**

乐观锁比较+条件更新: CAS(E,V,N):如果期望值(E)当前值(V)相等则更新为新值(N)，用比较交换构建无锁算法，有ABA长重试问题、只能对单变量操作局限。

- **无锁（lock-free）**：失败不阻塞只重试；
- **自旋**：失败后 loop retry；自旋开销大，适合低冲突场景，高冲突会自旋到死。
- **ABA 问题**：A → B → A，CAS 认为没变(AtomicStampedReference+内存屏障保证可见性(版本号解决))
- **实现方式：**CAS 是利用 CPU 的原子指令（如 x86 的 cmpxchg）实现不依赖Java 层。

**📌 AQS(AbstractQueuedSynchronizer):** 

CLH FIFO队列+CAS volatile state+park/unpark阻塞唤醒机制，统一独占锁 / 共享锁的实现，子类只需要实现 `tryAcquire/tryRelease。`

- 排队（每个节点表示一个等待的线程、状态、前驱/后继）
- 阻塞 / 唤醒（park/unpark）
- 安全的状态维护（CAS 修改 state）
- `ReentrantLock`、`CountDownLatch`、`Semaphore`、`ReentrantReadWriteLock`
- 两种模式：**独占模式**（ReentrantLock/ReentrantReadWriteLock写/FutureTask）**共享模式**（Semaphore/CountDownLatch/ReadWriteLock/ReentrantReadWriteLock读）
