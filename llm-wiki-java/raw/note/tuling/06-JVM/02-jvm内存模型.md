# 02-jvm内存模型

所有者: junk01

### **一、JVM 内存区域**

```
线程私有：
 ├─ 程序计数器PC
 ├─ 虚拟机栈（局部变量表/操作数栈/动态链接/返回地址）
 └─ 本地方法栈

线程共享：
 ├─ 堆（新生代 + 老年代）
 │     ├─ 新生代：Eden / S0 / S1
 │     └─ 老年代
 └─ 方法区（Metaspace）
        ├─ 类元数据
        ├─ 常量池
        └─ 静态变量
```

**1. 程序计数器 PC（私有）**

- 作用：当前线程锁执行字节码的行号，每条线程都有一个独立的程序计数器。记录当前线程执行的 **字节码指令地址**
- 特性：线程私有，每个线程都有独立 PC
- 执行 **Native 方法时 PC = 空**
- 唯一不会抛 OOM 的区域

**2. 虚拟机栈（私有）**

每次方法调用 → 创建一个**栈帧(**存储局部变量，操作数，动态链接，方法返回地址**) →**从调用到执行完毕→入栈和出栈

**（1）局部变量表：**

- 基本类型、对象引用、returnAddress
- long / double 占 **2 个 slot**
- 大小在编译期确定**不可动态扩容**

**（2）操作数栈（LIFO）**

- 存放操作数、运算中间结果、方法调用参数
- 编译期确定最大深度
- long/double 占 2 深度

**（3）动态链接（指向运行时常量池）**

- 为方法调用提供符号引用 → 直接引用的解析
- final/static 字段、静态方法 → 静态解析
- 普通方法调用 → 动态解析（invokevirtual）

**（4）方法返回地址**

- 正常执行 → 保存下一条指令的地址
- 异常退出 → 通过异常表处理

**异常情况**

- 栈帧深度超过 JVM 限制 → **StackOverflowError**
- 栈可扩展但内存不足 → **OutOfMemoryError**

**3. 本地方法栈（Native Stack，私有）**

- 作用：为 Native 方法提供栈帧
- 异常：也可能抛 StackOverflowError 和OutOfMemoryError

**4. 堆（Heap，共享）**

**“对象实例的唯一分配地”**

- JVM 启动时 → 创建**堆 - >** 存放对象实例
- 由 `Xms` / `Xmx` 设置大小
- 对象不足且无法扩容 → **OutOfMemoryError**

**4.1 堆的分代结构（默认）**

| 区域 | 比例 | 特点 |
| --- | --- | --- |
| **新生代（1/3）** | Eden(8/10)、S0(1/10)、S1(1/10) | 大量短命对象 |
| **老年代（2/3）** |  | 长期存活对象 |

**4.2 对象晋升**

- 对象年龄达到阈值（默认 15）进入老年代
- Survivor 不够 → **晋升失败**（直接进入老年代）

---

**5. 方法区（Metaspace，JDK 1.8+，共享）**

储存 类信息、类元数据、常量、静态变量、JIT 代码等

- JDK1.8 后替代永久代，使用 **本地内存**
- 字符串常量池从永久代（1.7）移入 **堆**

-XX:MetaspaceSize=18m
-XX:MaxMetaspaceSize=60m

### **二、GC 根与GC算法**

**1. 引用计数法（不用）**

- 实现简单，但无法处理 **循环引用**
- JVM 主流实现不采用

**2. 可达性分析（JVM 主流）**

从 GC Roots 出发找可达对象 -》不可达 → 垃圾

**GC Roots 包括：**

- 虚拟机栈中引用的对象（局部变量）
- 方法区中类的静态变量
- 常量池引用
- 本地方法栈的引用
- 正在运行的线程对象

---

### **三、GC算法**

**1. 分代收集理论：**新生代/老年代分别使用不同算法。

**2. 标记-复制（新生代）：**高效无碎片，浪费一半空间

流程：Eden + S0 → 将活对象复制到 S1 -》复制后空间整体清空

**3. 标记-清除（老年代）：**速度慢 + 产生碎片

阶段：标记活对象 -〉清除未标记对象

**4. 标记-整理（老年代）：**开销略高 + 无碎片

与标记-清除差别：活对象向一端移动 → 清理边界之外

---

### **四、Minor GC 时对象移动**

1. GC 扫描 GC Roots
2. 复制活对象到 S1（或老年代）
3. **记录“对象旧地址 → 新地址”的关系（复制指针）**
4. 在扫描过程中同时更新所有引用指向新地址

 **5、原区域对象不会立即清除**

- Survivor 旧区（S0）只是把指针重置到区头（认为空间可用）
- 原对象等新对象覆盖后自然消失
