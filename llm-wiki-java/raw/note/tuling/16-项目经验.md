- **IM架构项目**
    
    ### 一、IM 系统概述
    
    IM（Instant Messaging）系统，即即时通讯系统。
    
    - **简单 IM**：Netty + Socket 即可实现
    - **工业级 IM**：
        - 千万 / 亿级用户
        - 高并发、高可用、高可靠
        - 多端同步、离线消息、群聊、直播
    
    IM 的本质不是“聊天”，而是一个 **低延迟、高可靠的实时消息分发系统**。
    
    ---
    
    ### 二、IM 的典型应用场景
    
    - 即时通讯：微信 / QQ / 钉钉
    - 社区私信：微博 / 知乎
    - 直播互动：抖音 / 快手
    - 在线客服 / 电商
    - 物联网：智能家居控制
    - 出行 / 教育：位置共享、在线白板
    
    👉 **凡是“强实时交互”的系统，底层一定是 IM**。
    
    ---
    
    ### 三、IM 系统核心术语
    
    - 用户（User）
    - 消息（Message）：文本 / 图片 / 视频 / 文件
    - 会话（Session）：单聊会话
    - 群（Group）：群聊会话
    - 终端（Terminal）：Android / iOS / Web
    - 未读数（Unread Count）
    - 用户状态（在线 / 离线）
    - 关系链（好友 / 关注）
    - 单聊 / 群聊 / 客服
    - 信箱（Inbox / Outbox / Timeline）
    
    ---
    
    ### 四、消息可靠投递（不丢消息）
    
    1️⃣ 客户端重试 + 去重
    
    - 客户端为每条消息生成 **唯一 msgId**
    - 超时 / 失败自动重发
    - 服务端按 msgId 去重
    
    2️⃣ 服务端可靠存储
    
    - 使用 **RocketMQ / Kafka** 作为消息缓冲
    - 保证消息不因服务宕机丢失
    
    3️⃣ ACK 确认机制
    
    - 客户端收到消息后返回 ACK
    - 未 ACK 的消息可重推
    
    👉 **最终目标：At Least Once + 幂等消费**
    
    ---
    
    ### 五、离线消息高性能设计
    
    设计原则
    
    - **上线拉取，不推送**
    - **缓存抗并发，DB 兜底**
    
    实现方案
    
    - 离线消息存 Redis（集群）
    - 设置限制：
        - 最近 N 天
        - 最近 N 条（如 1000 条）
    
    冷数据处理
    
    - 客户端翻历史 → 查询 DB
    - 小概率操作，DB 可承受
    
    ---
    
    ### 六、海量历史消息存储方案
    
    表结构拆分
    
    1️⃣ 消息内容表（大表）
    
    ```
    im_msg_content(mid, content, type, create_time)
    ```
    
    - 按 **mid 分库分表**
    
    2️⃣ 用户信箱索引表（小表）
    
    ```
    im_user_msg_box(owner_uid, other_uid, mid, box_type)
    ```
    
    - 发件箱 / 收件箱各一条
    - 按 **owner_uid 分库分表**
    
    好处
    
    - 支持双方独立删除
    - 索引查询快
    - 避免大字段扫描
    
    ---
    
    ### 七、Redis 离线消息结构设计
    
    使用 ZSet
    
    ```
    Key: offline_msg_{uid}
    Score: mid
    Value: msg
    ```
    
    - 添加：`ZADD`
    - 查询：`ZREVRANGE`
    - 删除：`ZREMRANGEBYSCORE`
    
    👉 可按 **周 / 月分 Key** 防止单 Key 过大
    
    ---
    
    ### 八、群聊：读扩散 vs 写扩散
    
    读扩散（默认）
    
    - 群消息只存一份
    - 用户读同一份数据
    
    ✅ 写入快
    
    ❌ 难支持已读列表
    
    写扩散
    
    - 每个用户一条索引
    - 单独一份消息内容
    
    ✅ 精确未读 / 已读
    
    ❌ 万人群写入压力极大
    
    👉 **大群 / 直播：读扩散**
    
    👉 **小群 / 协作：写扩散**
    
    ---
    
    ### 九、未读数一致性（Lua + Redis）
    
    单聊未读数
    
    - 用户总未读数
    - 会话未读数
    
    👉 用 **Lua 脚本** 保证原子性
    
    群聊未读数
    
    ```
    HINCRBY msg:noreadcount:{gid} {uid} 1
    ```
    
    - 每人一个未读数
    - 客户端主维护，服务端用于多端同步
    
    ---
    
    ### 十、万人群聊系统难点
    
    1️⃣ 未读数爆炸
    
    - 1 条消息 → N 次 Redis 写
    - 解决方案：
        - 内存累计
        - **定时批量刷 Redis**
    
    2️⃣ 路由查询压力
    
    - 群消息需查用户在哪台 Netty
    - 大群 → Redis QPS 爆炸
    
    ---
    
    ### 十一、百万在线直播间设计
    
    核心优化思路
    
    - **绕过 Redis 路由查询**
    - 消息直接广播给所有 Netty 节点
    
    Netty 本地缓存
    
    - 每台 Netty 维护：
        - 本机连接的直播用户列表
    
    消息流程
    
    1. MQ 广播直播消息
    2. 每台 Netty 收到
    3. 只推送本机在线用户
    
    👉 极大降低 Redis 压力
    
    ---
    
    ### 十二、限流 & 熔断保障高可用
    
    为什么允许丢消息？
    
    - 直播间每秒几十条已是极限
    - 全量投递反而拖垮系统
    
    常见手段
    
    - 消息频率限流
    - 热点房间限流
    - 非核心消息直接丢弃
    
    👉 **系统稳定性优先于消息完整性**
    
    ---
    
    ### 十三、IM 架构设计总结
    
    - **单聊：强可靠 + ACK + 离线补偿**
    - **群聊：读扩散 + 批量优化**
    - **直播：广播 + 本地连接缓存**
    - **未读数：客户端为主，服务端兜底**
    - **高峰期：限流优先，体验兜底**

### **📌**  撮合系统

**撮合系统核心模块:**

**用户:** 终端用户委托报价与数量,生成订单发送至交易平台。

**网关:** 负责收集用户订单,并将其派发给撮合引擎。

**撮合引擎:** 交易系统中的核心部分,用于接收订单并根据业务逻辑实现订单   撮合同时生成交易记录,随后给予用户交易结果反馈。

**数据库:** 用来存放交易过程中的订单和交易记录,实现数据持久化。

**撮合分区：**不同类型金融交易将撮合模块划分为若干业务分区,每个分区独立进行撮合,彼此不受影响

**撮合交易核心算法：**

撮合引擎的核心业务模块就是撮合交易算法，撮合交易算法的任务一方面是完成对客户所下订单进行公平合理的排列和撮合功能,也要保证撮合算法的公平性、高效性以及扩展性等。

**算法1: 订单队列：**

撮合交易的重要组成部分就是买卖订单,通过对买卖订单进行撮合最后形成交易记录。所以对无法立刻完成撮合的订单,需要有买入队列和卖出队列保存订单。队列按照“价格优先、同价格下时间优先”的原则。买入队列按照委托价格从低到高的顺序, 卖出队列则按照委托价格从低到高的顺序排列

**算法2: 撮合顺序**

撮合引擎接收到新的买入订单,则会到卖出队列的头部查找是否存在符合价格规则的卖出订单,如果存在卖出价格小于或等于买入价格的订单,则从队列中取出此订单并撮合成一笔交易;

如果卖出队列为空或队列头部不满足价格关系,则将买入订单插入买入队列中,由于买入队列是按照价格与时间先后进行排序,所以新插入的订单会经过一次排序插入到买入队列的相应位置。

相同的,当撮合引擎接收到新的卖出订单,则会到买入队列的头部査找是否存在符合价格规则的买入订单,如果存在买入价格大于或等于卖出价格的订单,则从订单队列中取出此订单并撮合成一笔交易;如果买入队列为空或队列头部不满足价格关系,则将卖出订单插入到卖出队列中,由于卖出队列也是按照价格与时间先后进行排序的所以新插入的订单会经过一次排序插入到卖出队列的相应位置

**算法3:内存撮合**

当前的数据库撮合技术的性能低下的原因在于过多与数据库交互,使得I/O很多,系统整体处理速度同时受数据库事务逻辑约束。

内存撮合技术,通过最大程度去除与数据库的交互过程,将整个撮合逻辑放在内存中进行。因此比数据库撮合技术少了许多网络I/O交互时间,在性能上可以大幅提升撮合速度;内存撮合的弊端就是由于内存的易失性,当服务器出现故障停机时,所有的交易数据将会丢失,系统的可靠性以及一致性都相应大幅降低。因此本文在提高内存撮合技术可靠性的方面采用了多机热备份及分布式一致性技术作为补充,从而获得内存撮合技术的高性能以及数据库撮合技术的数据持久性。

**算法4:多机热备**

由于内存撮合技术在撮合引擎出现异常时的可靠性和一致性非常差,而金融交场系统因为其业务特性,对服务中断以及数据丢失的容忍度非常低,为了提高容错性,一般多采取的是多机热备技术。

本文采用多机热备份技术,将一组撮合引擎部署成互为备份的撮合引擎集群,并且在同一时间内只有一台撮合引擎提供服务。当其中运行这的一台撮合引擎出现故障无法继续正常工作 ,撮合引擎集群会迅速检测到这个故障,并选举出一个备份撮合引擎接管故障撮合引擎的任务从而保证整个撮合系统的正常运行

多机热备份技术的本质就足针对服务器临时故障所做的一种备份技术,本文迎过采用多机热备份技术,来避免长 间的撮合服务中断,保证撮合系统长期、可靠的服务。通过将多台撮合引擎进行热格份,从而保证在撮合引擎出现故障时,会在可以接受的时间内完成主机和备机之间的切换,由备份机提供无缝连续服务。
