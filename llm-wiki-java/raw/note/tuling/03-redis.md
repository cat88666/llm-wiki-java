### 01-Redis 数据结构

**String 字符串：**缓存字符串、JSON、分布式锁实现（setnx）、计数器、Session 共享、分布式 ID

**Hash 哈希（字典）：**适合存储对象（key-value 多属性）

**List 列表（链表）：**可作为队列/栈使用、用于消息流，如微信公众号、微博时间线

**Set 集合（字典+跳表）：**元素不重复；支持交集/并集/差集；场景：共同好友、点赞、好友推荐

**Sorted Set / ZSet 有序集合：**可设置 score 排序、用于排行榜、距离排序（Geo）

**Bitmap：**位图，适用于布隆过滤器、签到、活跃用户统计

**分布式锁：**setnx 实现基础加锁

**分布式 List：**双向链表 Lpush/Rpush，可用于新闻流

**GeoHash：**基于SortedSet 实现坐标存储、附近的人

**HyperLogLog：**不精确统计 UV、去重计数

### 02-Redis 线程模型

- 核心读写与网络 IO 单线程
- 但持久化、异步删除、复制等使用其他线程

**Redis 单线程很快**

- 全内存操作
- 无锁（避免线程切换消耗）
- 底层数据结构高效
- 使用 I/O 多路复用（epoll）

**Redis 单线程如何处理多并发？**

- 使用 epoll 事件驱动模型
- 事件队列 → 分派器 → 事件处理器

### 03-Redis 集群策略

1. 主从模式：主写从读；容易受单点影响、扩容困难

2. 哨兵模式（Sentinel）：自动主从切换、高可用；但仍不能解决容量瓶颈

3. Cluster 模式：多主多从；使用 Slot 16384 槽位分片；支持大容量扩展

### 04-Redis 分布式锁实现

**核心原理**

1. `SETNX` 保证互斥
2. 使用 Lua 脚本实现原子删除锁
3. 使用看门狗续期防止过期释放
4. 多节点红锁 RedLock 提升可靠性

**setnx + setex 的问题**

- 可能超时设置失败 → 死锁，解决：SET key value NX PX timeout
- value 存 uuid（线程唯一标识）
- 删除锁使用 Lua 保证原子性

**Redisson 解决的问题**

- 可重入锁
- 看门狗自动续期
- 原子性保证
- 分布式环境更安全的锁管理

### 05-Redis 主从复制原理

1. **建立连接，准备复制**
2. **全量同步：主节点发送 RDB 快照**
3. **主节点将复制期间的写命令写到 replication buffer**
4. **RDB 发送完成后，发送增量写命令**
5. **后续进入增量同步阶段**

### 06-Redis 持久化机制

**RDB（快照）**

**优点：**单文件 dump.rdb，方便备份、对性能影响小、启动恢复快

**缺点：**数据不够安全（间隔持久化）、fork 子进程耗时大

**AOF（日志）**

原理：将写命令以追加方式写入、可 rewrite 压缩文件、重启优先加载 AOF

优点：最安全（可丢失极少数据）、追加写不会损坏文件

缺点：文件更大、恢复慢

同步策略：everysec（默认）：最多丢 1 秒、always：最安全但最慢、no：交给系统决定

### 07-缓存问题

1. 缓存雪崩

- 大量 key 同时过期
- **解决：**过期时间随机化、集群高可用、热点预热、缓存标记、互斥锁

2. 缓存击穿

- 热点 key 失效导致瞬时打爆数据库
- **解决：**热点永不过期、加互斥锁、缓存续命

3. 缓存穿透

- 查不存在的数据
- **解决：**参数校验、缓存空值、布隆过滤器

---

### 07-分布式缓存寻址算法

Hash 取模：简单、扩容全部迁移（缺点）

一致性 Hash：只影响局部节点、虚拟节点解决分布不均

Hash Slot（Redis Cluster）：数据 → Slot → 节点；扩容只迁移 Slot

---

### 08-缓存过期策略

定时过期：内存最干净，CPU 占用高

惰性过期：访问时检查，可能堆积大量过期数据

定期过期（Redis 默认）：随机扫描，CPU 与内存平衡

分桶策略：类似时间轮，将相近过期 key 放一起扫描

### 09-Redis 优化与工程实践

1. 公共 Redis 迁移

- 找出商品线使用的所有 key
- 使用 redis-shake 工具迁移
- 全量刷新缓存（当天 + 明天）

2. 秒杀业务缓存与 DB 双写一致性

- 关键点：
    1. 统一缓存更新入口（避免分散写）
    2. 非敏感数据依靠过期触发更新
    3. 秒杀库存必须加分布式读写锁
    4. KeyExpirationEventMessageListener + binlog 实时更新
    5. MongoDB 记录库存变动日志
