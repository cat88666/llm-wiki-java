# 系统设计-电商系统

> 电商系统的主线是商品浏览、购物车、下单、库存、支付、履约和售后；高并发场景下，商品读链路用多级缓存，库存写链路用 Redis Lua/MQ 削峰/DB 乐观兜底，订单支付链路用状态机和幂等保证最终一致。

## 一、电商业务闭环：读多写少与交易强一致分层

```
商品浏览/搜索 → 加购物车 → 提交订单 → 锁定库存 → 支付
      → 支付成功扣减库存/履约 → 物流发货 → 售后退款/退货 → 对账
```

| 子系统 | 核心矛盾 | 默认选型 |
|--------|----------|----------|
| 商品详情 | 高读流量、低变更 | Caffeine + Redis + DB，热点预热 |
| 商品列表 | 多条件检索、排序 | ES 索引 + Canal/MQ 同步 |
| 购物车 | 用户维度读写频繁 | Redis Hash + DB 异步落盘 |
| 订单 | 幂等创建、状态正确 | 订单号唯一 + 状态机 |
| 库存 | 防超卖、防少卖 | Redis Lua 快速扣减 + DB 乐观兜底 |
| 支付 | 外部系统异步回调 | 待支付状态 + 回调幂等 + 定时反查 |

## 二、商品读链路：详情缓存、列表索引、热点保护

| 场景 | 推荐方案 | 代价 | 结论 |
|------|----------|------|------|
| 商品详情按 ID 查 | Caffeine + Redis + DB | 更新有短暂不一致 | 读多写少首选 |
| 商品列表复杂筛选 | ES | 同步延迟、索引维护 | 不要用 MySQL 扛复杂搜索 |
| 热门商品详情 | 热点预热 + 本地缓存 | 内存占用 | 大促前必做 |
| 缓存穿透 | BloomFilter/空值缓存 | 有误判/污染 | 高频不存在 key 必做 |
| 缓存击穿 | 互斥锁/逻辑过期 | 实现复杂 | 热点 key 必做 |

```
商品详情请求
  → Caffeine 本地缓存
  → Redis GET goods:{id}
  → DB SELECT by id
  → 回写 Redis + Caffeine
```

## 三、下单写链路：幂等订单、库存预扣、MQ 削峰

普通下单可以直接 DB 事务锁库存；秒杀/限量活动必须先削峰：

```
用户请求
  → 网关限流 + 活动资格校验
  → 秒杀 token/幂等 key 防重复
  → Redis Lua 原子预扣库存
  → MQ 投递 create_order
  → 消费者创建订单 + DB 乐观扣库存
  → 支付超时回补库存
```

| 方案 | 一致性 | 吞吐 | 适用场景 | 结论 |
|------|--------|------|----------|------|
| DB 乐观锁扣库存 | 强 | 热点行竞争大 | 普通低并发商品 | 简单优先 |
| Redis 预扣 + MQ | 最终一致 | 高 | 秒杀/抢购 | 高并发首选 |
| 队列串行化 | 强，逻辑简单 | 受单队列限制 | 库存极少、可排队 | 吞吐要求低时可选 |
| 库存分桶 | 降低热点冲突 | 汇总和回补复杂 | 极端热点单 SKU | 大促热点兜底 |

```lua
local stock = tonumber(redis.call('GET', KEYS[1]))
if stock == nil or stock <= 0 then
    return 0
end
if redis.call('SISMEMBER', KEYS[2], ARGV[1]) == 1 then
    return -1
end
redis.call('DECR', KEYS[1])
redis.call('SADD', KEYS[2], ARGV[1])
return 1
```

## 四、库存一致性：Redis 快路径，DB 流水做权威

| 问题 | 触发场景 | 处理 |
|------|----------|------|
| Redis 扣成功，订单创建失败 | 消费者异常/DB 超时 | 回补 Redis + 记录失败流水 |
| MQ 重复消费 | Broker 重投/消费者超时 | 订单唯一键 + 库存流水幂等号 |
| Redis 与 DB 不一致 | 回补失败/人工改库 | 定时对账，以 DB 库存流水修正 Redis |
| 少卖 | Redis 预扣后用户未支付 | 延迟消息超时关单并回补库存 |

库存权威模型：

```
可售库存 = 初始库存 - 已支付扣减 - 已锁定未支付 + 已取消回补

Redis 负责高并发快速判断，DB 库存流水负责最终对账。
```

```sql
CREATE UNIQUE INDEX uk_stock_event
ON stock_flow(event_id);
```

## 五、订单支付状态机：支付成功和超时关单互斥

```
INIT ──创建订单──▶ WAIT_PAY ──支付成功──▶ PAID ──履约──▶ DELIVERING
                         │
                         └─超时未支付──▶ CLOSED
PAID ──退款成功──▶ REFUNDED
```

```sql
-- 支付回调
UPDATE orders
SET status = 'PAID', paid_at = NOW(), version = version + 1
WHERE order_id = #{orderId}
  AND status = 'WAIT_PAY'
  AND version = #{version};

-- 超时关单
UPDATE orders
SET status = 'CLOSED', version = version + 1
WHERE order_id = #{orderId}
  AND status = 'WAIT_PAY'
  AND version = #{version};
```

反直觉：支付回调超时不等于支付失败。支付链路必须有 `PAYING/WAIT_CONFIRM` 或定时反查机制，不能因为本地 RPC 超时就直接关单。

## 六、Gacha/限量道具：电商库存模型在游戏里的变体

Gacha 限量道具本质上是"库存极少、QPS 极高、落盘必须可靠"的电商库存问题。

| 决策 | 推荐 | 原因 |
|------|------|------|
| 限量道具库存 | Redis Lua 原子扣减 | 单线程脚本天然防超卖 |
| 发放落盘 | 本地消息表 + MQ 补偿 | 防止扣库存成功但背包未写 |
| 普通道具 | 内存权重表 + 异步日志 | 无库存约束，不走重库存链路 |
| 保底机制 | Redis 计数 + 命中后清零 | 用户维度状态轻量 |

```java
@Transactional
public void grant(String orderId, String userId, String itemId) {
    backpackMapper.insertIgnore(orderId, userId, itemId);
    localMessageMapper.insert(orderId, "GACHA_GRANTED", "INIT");
}
```

## 七、电商系统决策树

```
是普通商品下单？
  └─ DB 乐观库存 + 订单唯一键 + 支付状态机

是秒杀/抢购？
  ├─ Redis Lua 预扣
  ├─ MQ 削峰创建订单
  ├─ DB 乐观库存兜底
  └─ 超时未支付回补库存

是极端热点单 SKU？
  ├─ 库存分桶
  ├─ 热点商品本地缓存
  └─ 活动前预热，活动后对账

是限量 Gacha 道具？
  ├─ Redis Lua 防超卖
  └─ 本地消息表防丢发
```

## 八、面试追问

### 1. 秒杀如何防超卖？

先用 Redis Lua 原子预扣库存承接高并发，再用 MQ 削峰异步创建订单，最后 DB `WHERE stock > 0` 乐观扣减做兜底。Redis 防大流量超卖，DB 防缓存异常后的最终超卖。

---

### 2. Redis 扣减成功但订单创建失败怎么办？

必须有库存流水和补偿。订单创建失败时回补 Redis；如果回补也失败，定时对账任务以 DB 订单和库存流水为准修正 Redis。不能只相信 Redis 的当前值。

---

### 3. 支付成功和超时关单同时发生怎么办？

订单状态机保证 `WAIT_PAY` 只能转到 `PAID` 或 `CLOSED` 一个终态，使用 `UPDATE ... WHERE status='WAIT_PAY' AND version=?`。如果关单先成功后收到支付成功，要触发退款；如果支付先成功，关单影响行数为 0。

---

### 4. 商品详情缓存和 DB 如何保持一致？

商品详情是读多写少，通常接受短暂最终一致。更新商品时先更新 DB，再删除 Redis 和广播删除本地缓存；热点商品可使用逻辑过期或延迟双删降低旧值窗口。

## 九、电商与 Redis、Kafka、幂等的关系

- [[概念-Redis]]：商品缓存、购物车 Hash、秒杀 Lua、库存分桶都依赖 Redis 的原子和高吞吐能力。
- [[机制-Kafka]]：订单创建、库存回补、商品索引同步适合用 MQ 解耦和削峰。
- [[概念-幂等设计]]：订单号、支付回调流水、库存事件 ID、Gacha 发放 orderId 都必须唯一。
- [[主题-三高架构]]：电商是高并发读、高一致写、高可用降级的组合场景。
