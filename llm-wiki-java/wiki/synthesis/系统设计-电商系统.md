# 系统设计-电商系统

> 高并发游戏电商的三类核心场景：秒杀（Redis Lua 漏斗削峰 + DB 乐观兜底）、Gacha 限量道具（零超卖 + 本地消息表落盘）、跨地域实时对战（gRPC + TCP_NODELAY + 状态机乐观锁）；选型结论取决于对一致性和吞吐的权衡。

## 一、场景概述与问题拆解

| 场景 | 核心矛盾 | 关键约束 | 结论 |
|------|---------|---------|------|
| 电商秒杀 | 瞬时高并发 vs 有限库存 | 不超卖、幂等、用户体验 | 多层漏斗削峰，DB 做最终防线 |
| Gacha 限量道具 | 10 万 QPS 下零超卖 | 性能 + 落盘可靠性 | Redis Lua 原子扣 + 本地消息表 |
| Gacha 普通道具 | 高并发写日志 | 无库存限制 | 无需特殊处理，异步写即可 |
| 实时对战 | 同一房间状态并发修改 | 顺序性 + 低延迟 | EventLoop 单线程路由 + 状态机 |
| 跨地域通信 | 跨 Region 延迟高 | P99 < 100ms | gRPC + TCP_NODELAY + GeoDNS |

## 二、电商秒杀设计

**多层漏斗削峰**：

```
用户请求
  ↓ 前端按钮置灰 + 用户级频控（一次只能参与一个活动）
  ↓ 网关令牌桶/漏桶限流
  ↓ 应用层秒杀 token 防重（用户+活动唯一 token）
  ↓ Redis Lua 原子预扣库存
  ↓ MQ 异步削峰
  ↓ DB 乐观扣减库存 + 创建订单
```

**Redis Lua 预扣减**：

```lua
-- 原子检查并扣减库存
local stock = redis.call('get', KEYS[1])
if tonumber(stock) <= 0 then
    return 0  -- 库存不足
end
redis.call('decr', KEYS[1])
return 1  -- 扣减成功
```

**DB 乐观兜底**：

```sql
UPDATE goods SET stock = stock - 1
WHERE id = #{id} AND stock > 0;
-- 影响行数 = 0 → 库存为 0，防止超卖
```

**库存分桶（热点行优化）**：

| 方案 | 优点 | 代价 | 结论 |
|------|------|------|------|
| Redis 预扣 + MQ | 吞吐高，保护 DB | 短暂不一致，需补偿 | 主流方案 |
| DB 乐观锁 | 强一致，简单 | 热点行竞争严重 | 并发低时可用 |
| 库存分桶 | 降低行锁冲突 N 倍 | 聚合查询复杂 | 极端热点时使用 |
| 队列串行化 | 不超卖，逻辑简单 | 吞吐受单队列限制 | 吞吐要求不高时可用 |

**全链路幂等**：

| 环节 | 幂等方案 |
|------|---------|
| 用户重复点击 | 前端置灰 + 秒杀 token |
| 网关重复请求 | 用户维度限流 |
| Redis 预扣 | 用户-活动唯一参与记录 |
| MQ 消费重复 | 消息 ID + DB 唯一键 |
| DB 创建订单 | `user_id + goods_id + activity_id` 唯一索引 |

## 三、Gacha 零超卖设计

### Redis Lua 原子扣减（核心方案）

```lua
-- deduct_stock.lua
-- KEYS[1] = 库存 key，ARGV[1] = 购买数量
local stock = tonumber(redis.call('GET', KEYS[1]))
if stock == nil then
    return -2  -- 道具不存在
end
if stock < tonumber(ARGV[1]) then
    return -1  -- 库存不足
end
return redis.call('DECRBY', KEYS[1], ARGV[1])  -- 返回扣减后剩余库存
```

```java
Long remaining = redisTemplate.execute(
    new DefaultRedisScript<>(luaScript, Long.class),
    Collections.singletonList("item:rare:" + itemId),
    String.valueOf(quantity));
if (remaining < 0) throw new SoldOutException("道具已售罄");
// 继续落盘流程
```

**Lua 原子性保证**：Redis 单线程执行脚本，整个脚本是不可分割的操作单元，中间不会有其他命令插入。

### 落盘流程（本地消息表，防丢单）

```java
// 步骤 1：同一事务写本地消息表
@Transactional
public void handleGachaSuccess(String userId, String itemId, String orderId) {
    inventoryService.addItem(userId, itemId);  // 写用户背包
    PendingGrant grant = new PendingGrant(userId, itemId, orderId, "PENDING");
    pendingGrantRepo.save(grant);              // 写本地消息表（同一事务）
}

// 步骤 2：触发 MQ 消息
mqTemplate.send("gacha.grant", new GachaGrantEvent(orderId));
// 更新本地消息表 status = SENT（MQ 成功后更新）

// 步骤 3：定时补偿（防止 MQ 投递失败）
@Scheduled(fixedDelay = 30_000)
public void retryPendingGrants() {
    pendingGrantRepo.findByStatus("PENDING", 100)
        .forEach(g -> mqTemplate.send("gacha.grant", g));
}
```

### 保底系统（Pity System）

```java
Long pullCount = redisTemplate.opsForValue()
    .increment("pity:" + userId + ":" + poolId);
if (pullCount >= PITY_THRESHOLD) {  // 如 90 连保底
    result = forcedRareItem();
    redisTemplate.delete("pity:" + userId + ":" + poolId);
} else {
    result = drawItem(poolId);
    if (result.isRare()) {
        redisTemplate.delete("pity:" + userId + ":" + poolId);
    }
}
```

## 四、对象池 + G1 GC 消除 GC 抖动

**问题**：Gacha 高峰期 QPS 激增，大量短命对象涌入 Young Gen，频繁 Minor GC，P99 出现抖动。

**对象池方案**：

```java
// Apache Commons Pool2 池化 DrawContext 对象
GenericObjectPool<DrawContext> contextPool = new GenericObjectPool<>(
    new DrawContextFactory(),
    new GenericObjectPoolConfig<DrawContext>() {{
        setMaxTotal(2000);
        setMinIdle(100);
        setMaxWaitMillis(100);  // 等待超时抛异常，保护系统
    }});

DrawContext ctx = contextPool.borrowObject();
try {
    return gacha(ctx, userId, poolId);
} finally {
    ctx.reset();               // 清除上次状态
    contextPool.returnObject(ctx);
}
```

**G1 关键参数**：

```bash
-XX:+UseG1GC
-XX:MaxGCPauseMillis=5            # 目标停顿 5ms（游戏场景延迟敏感）
-XX:G1HeapRegionSize=16m          # 大对象（DrawPool）不进 Humongous Region
-XX:InitiatingHeapOccupancyPercent=35  # 提前触发 Mixed GC，防 Old Gen 堆满
-XX:G1ReservePercent=20           # 预留 20% 防 Evacuation Failure
-Xms4g -Xmx4g                     # 固定堆大小，防动态扩展触发 FGC
```

## 五、实时对战房间状态机

```sql
-- 状态转换加乐观锁，防止并发重复推进
UPDATE game_room SET status = 'READY', version = version + 1
WHERE room_id = ? AND status = 'WAITING' AND version = ?;
-- 影响行数 = 0 → 另一线程已推进，忽略
```

**用户动作路由（保证同一房间操作串行）**：

```java
// 按 roomId 哈希路由到固定 EventLoop 线程
EventLoop el = eventLoopGroup.next(Math.abs(roomId.hashCode() % GROUP_SIZE));
el.execute(() -> processAction(roomId, userId, action));
// 同一 roomId 的操作始终由同一个 EventLoop 处理，天然串行
```

## 六、跨地域延迟优化

| 层级 | 策略 | 降低延迟 | 结论 |
|------|------|---------|------|
| 协议层 | REST → gRPC（HTTP/2 + Protobuf 二进制） | 20-30% | 服务间通信必选 |
| 连接层 | 长连接池 + `TCP_NODELAY=true`（关闭 Nagle）| 5-10% | 实时游戏必选 |
| 路由层 | GeoDNS / Anycast 就近接入 | 30-50% | 跨地域必选 |
| 数据层 | 非强一致数据 Region 内缓存（道具配置/公告）| 消除跨 Region 读 | 大幅降低延迟 |
| 传输层 | `SO_RCVBUF/SO_SNDBUF = 4MB`（高延迟大带宽链路）| 10-20% | 跨 Region 链路优化 |

**Netty TCP 参数（跨地域）**：

```java
ServerBootstrap bootstrap = new ServerBootstrap()
    .childOption(ChannelOption.TCP_NODELAY, true)        // 禁用 Nagle，减少小包延迟
    .childOption(ChannelOption.SO_KEEPALIVE, true)       // TCP keepalive 检测死连接
    .childOption(ChannelOption.SO_RCVBUF, 4 * 1024 * 1024)  // 4MB 接收缓冲
    .childOption(ChannelOption.SO_SNDBUF, 4 * 1024 * 1024)  // 4MB 发送缓冲
    .childOption(ChannelOption.WRITE_BUFFER_WATER_MARK,
        new WriteBufferWaterMark(512 * 1024, 1024 * 1024));  // 背压控制
```

**`TCP_NODELAY` 原理**：Nagle 算法缓积小包到 MSS 大小才发送，造成额外 ~200ms 延迟；游戏实时包小且延迟敏感，必须关闭 Nagle。

## 七、选型决策树

```
高峰 QPS > 1 万，有限量道具？
  → Yes → Redis Lua 原子扣减（必选）
  → 需要落盘可靠性？→ 本地消息表 + MQ 补偿（必选）
  → 有大量短命对象导致 GC 抖动？→ 对象池 + G1 调参（MaxGCPauseMillis=5ms）
  → 有实时对战房间状态？→ 状态机 + 乐观锁 + EventLoop 单线程路由
  → 跨地域服务调用？→ gRPC + TCP_NODELAY + GeoDNS + SO_RCVBUF 调优
  → 普通秒杀场景？→ Redis Lua 预扣 + MQ 削峰 + DB 乐观兜底
```

## 八、面试高频追问

**问**：Gacha 如何保证零超卖且不丢单？
**答**：两层保障：①Redis Lua 脚本原子扣减库存（Redis 单线程，脚本不可分割，天然防并发超卖）；②本地消息表落盘（扣减成功后同一事务写背包+消息表，MQ 发送后更新状态，定时任务扫 PENDING 状态补偿重发）。两层缺一不可：Lua 防超卖，本地消息表防丢单。

**问**：G1 参数如何针对游戏实时性场景调优？
**答**：核心目标是 MaxGCPauseMillis=5ms。关键参数：`G1HeapRegionSize=16m`（避免大对象进 Humongous Region 触发额外 GC）、`InitiatingHeapOccupancyPercent=35`（提前触发 Mixed GC，防 Old Gen 堆满导致 FGC）、`G1ReservePercent=20`（预留空间防 Evacuation Failure）、固定堆大小（`Xms=Xmx`，防动态扩展触发 FGC）。配合对象池减少短命对象分配，进一步减少 Young GC 频率。

**问**：实时对战中如何保证同一房间的操作顺序执行？
**答**：按 roomId 哈希路由到固定 EventLoop 线程（Netty NIO 的 EventLoop 是单线程的），同一 roomId 的所有操作都由同一个 EventLoop 处理，天然串行。不需要加锁。状态机 + 乐观锁（`UPDATE WHERE status=X AND version=Y`）作为额外保障，防止多节点并发推进同一状态。

**问**：TCP_NODELAY 关了 Nagle 会有什么副作用？
**答**：关闭 Nagle 后小包会立即发出，不再缓积。副作用：网络传输效率略下降（小包 header overhead 变大），对跨地域高吞吐数据传输（如文件传输）可能增加带宽消耗。游戏实时场景（包小且延迟敏感）关闭 Nagle 是必选；文件传输等吞吐优先场景保持 Nagle 开启。
