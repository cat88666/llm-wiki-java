# 订单超时关闭设计

电商/支付场景的高频考点：用户下单后 30 分钟未付款自动关闭。相同模式还有：超时自动收货、到期退款、到期锁定等。

---

## 方案对比

| 方案 | 时间精度 | 可靠性 | 分布式友好 | 推荐度 |
|------|---------|--------|----------|-------|
| 被动关闭（用户查询时判断）| — | ❌ 脏数据堆积 | ✅ | 学习用，不上线 |
| 定时任务扫表 | 低（取决于调度频率）| ✅ | 需防并发 | ✅ 业务量小时推荐 |
| JDK DelayQueue | 高 | ❌ 内存，重启丢失 | ❌ | 不推荐 |
| Netty 时间轮 | 高 | ❌ 内存，重启丢失 | ❌ | 不推荐 |
| Redis 过期监听 | 中 | ❌ 可能丢消息 | ✅ | 不推荐 |
| Redis ZSet 扫描 | 高 | 中（可能重复消费）| ✅ | 可用但需幂等 |
| RocketMQ 延迟消息 | 中（固定等级）| ✅ | ✅ | 大业务量时推荐 |
| RabbitMQ 死信队列 | 高 | ✅ | ✅ | 大业务量时推荐 |
| **Redisson 延迟队列** | **高** | **✅（持久化）** | **✅** | **分布式首选** |

---

## 推荐方案详解

### 方案 1：定时任务扫表（业务量小）

```java
@Scheduled(cron = "0 * * * * ?")  // 每分钟扫描
public void closeExpiredOrders() {
    List<Order> orders = orderMapper.findExpiredOrders(
        LocalDateTime.now().minusMinutes(30));
    for (Order order : orders) {
        orderService.close(order.getId());  // 需保证幂等
    }
}
```

**缺点**：
- 时间精度差（最多延迟 1 分钟）
- 集中扫表给 DB 带来 I/O 压力
- 分库分表时全表扫描更困难

**优化**：分片任务（不同节点处理不同 ID 范围）+ 多线程 + 分页防大查询

### 方案 2：Redisson 延迟队列（分布式首选）

Redisson 在 Redis 上实现了可靠的延迟队列，底层用 ZSet 存储（score=执行时间戳），定时轮询到期任务：

```java
// 订单创建时
RBlockingQueue<String> blockingQueue = redisson.getBlockingQueue("order:close:queue");
RDelayedQueue<String> delayedQueue = redisson.getDelayedQueue(blockingQueue);
delayedQueue.offer(orderId, 30, TimeUnit.MINUTES);  // 30分钟后放入队列

// 消费端（独立线程）
@PostConstruct
public void consumeExpiredOrders() {
    new Thread(() -> {
        while (true) {
            try {
                String orderId = blockingQueue.take();  // 阻塞等待
                orderService.closeIfUnpaid(orderId);   // 幂等关闭
            } catch (Exception e) { ... }
        }
    }).start();
}
```

**优势**：精确到秒、Redis 持久化（RDB/AOF）保证重启不丢任务、分布式多消费节点天然互斥。

### 方案 3：RocketMQ 延迟消息

```java
// 发送延迟消息（下单时）
Message msg = new Message("ORDER_CLOSE_TOPIC", orderId.getBytes());
msg.setDelayTimeLevel(16);  // Level 16 = 30分钟（RocketMQ固定等级：1s/5s/10s/...）
producer.send(msg);

// 消费端（关单逻辑）
consumer.registerMessageListener((msgs, ctx) -> {
    for (MessageExt msg : msgs) {
        orderService.closeIfUnpaid(new String(msg.getBody()));
    }
    return ConsumeConcurrentlyStatus.CONSUME_SUCCESS;
});
```

**缺点**：RocketMQ 开源版延迟时间只有固定 18 个等级（不支持任意延迟时间）；付费版支持任意时间。

### 为什么不推荐 MQ 实现（通用场景）

对于订单超时关闭，订单量大时 **MQ 中大量消息永远不会被消费**（订单提前付款后关闭消息就无效了），造成 MQ 积压、存储浪费。更好的做法是：

```
用户支付成功 → 取消延迟任务（Redisson cancelFromQueue 或消费时判断幂等跳过）
```

Redisson 可以通过 `delayedQueue.remove(orderId)` 在支付成功时主动取消延迟任务，MQ 方案无法轻易删除已投递的消息。

## 关闭逻辑必须幂等

无论哪种方案，`closeOrder` 都必须实现幂等，因为定时任务/消息队列都可能触发多次：

```java
@Transactional
public void closeIfUnpaid(String orderId) {
    Order order = orderMapper.selectForUpdate(orderId);  // 悲观锁
    if (order == null || order.getStatus() != OrderStatus.UNPAID) {
        return;  // 已付款/已关闭，直接返回
    }
    order.setStatus(OrderStatus.CLOSED);
    orderMapper.update(order);
    // 释放库存、优惠券等
}
```

## 选型决策树

```
业务量小 + 对时间精度要求不高（误差 < 1 分钟可接受）
  → 定时任务扫表

业务量大 + 已有 Redis + 需要精确到秒
  → Redisson 延迟队列

业务量大 + 已有 RocketMQ + 接受固定延迟等级
  → RocketMQ 延迟消息

业务量大 + 已有 RabbitMQ + 需要任意延迟时间
  → RabbitMQ 延迟消息插件（rabbitmq-delayed-message-exchange）
```
