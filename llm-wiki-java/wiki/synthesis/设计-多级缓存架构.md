# 多级缓存架构

> 在 Redis 之前再加一层 JVM 进程内本地缓存（Caffeine），核心收益是**消除网络往返**：热点数据在本地内存命中（纳秒级），Redis 和 DB 的压力指数级下降；代价是多层一致性维护复杂度显著上升。

## 问题驱动

**面试高频问题组合**：
- 本地缓存和 Redis 缓存如何搭配？两者的角色各是什么？
- 两级缓存如何保证一致性？有哪几种更新策略？
- 什么是双重检测防雪崩？和单级 Redis 分布式锁有什么区别？
- 本地缓存集群多节点如何同步失效？有哪些方案？

---

## 为什么需要多级缓存

单纯 Redis 的极限：
- 每次 Redis 访问 = 网络往返（通常 1-5ms）
- Redis 单实例 QPS 上限 ~10 万/s
- 高并发下连接池竞争成为新瓶颈

本地缓存的收益：

| 指标 | 本地缓存（Caffeine）| Redis |
|------|-------------------|-------|
| 延迟 | ~100ns（内存寻址）| 1-5ms（网络往返）|
| 吞吐 | 理论无上限（CPU bound）| 10 万 QPS/实例 |
| 故障影响 | Redis 宕机时继续提供旧数据 | 缓存层完全不可用 |
| 一致性维护 | 复杂（需跨节点同步失效）| 简单（所有节点共享）|

**典型适用场景**：商品详情（读多写少）、配置中心（变更频率低）、鉴权/权限数据（每请求必查）

## 两级缓存读路径

```
请求
 │
 ▼
L1：本地缓存（Caffeine）── 命中 → 直接返回（~100ns）
 │ Miss
 ▼
L2：Redis 分布式缓存   ── 命中 → 回填 L1 → 返回（1-5ms）
 │ Miss
 ▼
DB（MySQL 等）         ── 查询 → 回填 L2+L1 → 返回（10-100ms）
```

## 本地缓存选型：Caffeine

Caffeine（Spring Boot 2+ 默认 Cache 实现）核心优势：
- **W-TinyLFU 淘汰算法**：结合访问频率（LFU）和近期性（LRU），命中率比纯 LRU 高约 15-30%
- **无锁设计**：写操作异步批量处理，读路径完全无锁，并发性能极高

```java
Cache<String, Product> localCache = Caffeine.newBuilder()
    .maximumSize(10_000)              // 最多 1 万条（超出按 W-TinyLFU 淘汰）
    .expireAfterWrite(30, SECONDS)    // 写后 30s 强制过期（一致性兜底）
    .refreshAfterWrite(10, SECONDS)   // 10s 后异步刷新（防止集中过期雪崩）
    .recordStats()                    // 开启命中率统计
    .build();
```

`refreshAfterWrite` vs `expireAfterWrite` 的区别：
- `expireAfterWrite`：到期后**阻塞等待**重新加载，有短暂 Miss 风险
- `refreshAfterWrite`：到期后**异步刷新**，旧值继续返回直到新值加载完毕（推荐）

## 一致性策略

### 策略一：TTL 自然过期（最简单）

L1 短 TTL（30s～2min）+ L2 长 TTL（5min～1h），靠过期自然失效。

- ✓ 无需额外组件
- ✗ 不一致窗口 = L1 TTL，最长数分钟；不适合价格、库存等敏感数据

### 策略二：MQ 广播失效（事件驱动）

数据变更时主动通知所有节点删除 L1：

```
DB 更新
  ↓ 发 MQ 消息（topic: cache.invalidate, body: {type: product, id: 123}）
  ↓ 每个服务节点都订阅
  ↓ Consumer 收到消息
     → localCache.invalidate("product:123")
     → redis.del("product:123")
```

- ✓ 不一致窗口 ≈ MQ 延迟（通常 < 50ms）
- ✗ 节点数多时消费者数量线性增长；MQ 消息丢失需 TTL 兜底
- 适合变更不频繁但一致性要求较高的场景

### 策略三：Canal + binlog 异步失效

Canal 监听 MySQL binlog，自动触发缓存失效，与业务代码完全解耦：

```
MySQL 写入 → binlog → Canal → 解析变更事件 → 失效 L2(Redis) + 广播 MQ → 失效 L1
```

- ✓ 业务代码零侵入；天然覆盖所有写路径（含 DBA 直接操作 DB）
- ✗ binlog 延迟（通常 < 100ms）；Canal 本身是单点（需 HA 部署）

## 双重检测防雪崩

单级 Redis 加锁的问题：所有 Miss 的请求都要抢 Redis 分布式锁，锁竞争本身成为热点。

两级缓存的改进：先在 JVM 本地用 `synchronized`/`ReentrantLock` 挡掉同 JVM 内的重复穿透，只有第一个线程需要抢 Redis 分布式锁。

```java
public Product getProduct(String id) {
    // 1. L1 快速检查（无锁）
    Product p = localCache.getIfPresent(id);
    if (p != null) return p;

    // 2. 本地锁（同 JVM 内只有一个线程穿透到 Redis）
    synchronized (("lock:" + id).intern()) {
        p = localCache.getIfPresent(id);  // 二次检查（可能已被同 JVM 其他线程回填）
        if (p != null) return p;

        // 3. L2 检查
        p = redis.get(id);
        if (p != null) {
            localCache.put(id, p);
            return p;
        }

        // 4. 分布式锁（多节点间只有一个线程穿透到 DB）
        if (redisson.tryLock("dlock:" + id, 3, SECONDS)) {
            try {
                p = redis.get(id);       // 三次检查（等锁期间其他节点可能已回填）
                if (p == null) {
                    p = db.query(id);
                    redis.setex(id, 300, p);
                }
                localCache.put(id, p);
            } finally {
                redisson.unlock("dlock:" + id);
            }
        } else {
            Thread.sleep(50);
            p = redis.getOrDefault(id, EMPTY);  // 等锁超时后兜底
        }
    }
    return p;
}
```

**为什么需要三次检查**：每次拿到锁后都要重新检查，因为等锁期间其他线程可能已完成回填。

## 缓存预热

```
冷启动问题：服务重启 → L1 清空 → 瞬间大量 Miss → 穿透 Redis 甚至 DB

预热方案（按场景选择）：
  ① 启动时异步批量加载：Redis Pipeline 批量 GET 热点 key → 填充 Caffeine
  ② 渐进预热：启动后 N 分钟内限流（只放 10% 流量），缓存自然填充
  ③ 提前预热：上线前离线分析热点 key → 预写 Redis → 服务启动时拉取
  ④ refreshAfterWrite（Caffeine 内建）：到期异步刷新，避免同一时刻大量 key 集中失效
```

## 关键权衡

1. **一致性 vs 吞吐**：TTL 策略最简单但不一致窗口长；事件驱动一致性好但架构复杂——按业务容忍度选择（库存/价格用事件驱动；页面展示内容用 TTL）
2. **L1 容量 vs GC 压力**：本地缓存占 JVM 堆，容量过大增加 GC 压力；建议配合 `softValues()` 让 GC 在内存紧张时自动淘汰 L1
3. **多节点 L1 不一致窗口**：MQ 广播有 ms 级延迟，极端情况下 A 节点看到新值、B 节点看到旧值；可接受则用 MQ，不可接受则只用 Redis（放弃 L1）
4. **key 爆炸问题**：如果 key 基数极大（每用户唯一），L1 命中率趋近于零，预热无意义反而浪费堆内存
5. **双写顺序（Cache-Aside 正确姿势）**：先更新 DB → 再删缓存（L2+L1），不要先删缓存再更新 DB（并发读会将旧值回填）

## 与其他概念的关系

- **扩展 [[概念-缓存三大问题]]**：多级缓存主要缓解击穿（本地锁+分布式锁双层防护）和雪崩（L1 兜底 Redis 故障）；穿透问题仍需布隆过滤器
- **依赖 [[机制-Redis集群与高可用]]**：L2 Redis 的 Pub/Sub 可替代 MQ 做轻量级 L1 失效广播（但 Pub/Sub 无持久化，可靠性弱于 MQ）
- **依赖 [[机制-消息队列可靠性]]**：事件驱动失效依赖 MQ 可靠投递；MQ 消息丢失会导致 L1 长期持有旧值，必须用 TTL 兜底
- **配合 [[机制-Canal数据同步]]**：Canal 监听 binlog 驱动缓存失效，是业务零侵入的最优方案
- **属于 [[概念-分布式系统理论]] AP 模型**：多级缓存容忍短暂不一致换取高可用性，本质是 BASE 最终一致性的工程实现

## 应用边界

**适合多级缓存**：
- 读写比 > 10:1，热点数据集中
- 可容忍秒级～分钟级的不一致窗口
- 服务有多实例（本地缓存分摊 Redis 压力价值大）

**不适合**：
- 强一致性场景（余额、订单状态）：只用 Redis 或直接 DB
- key 基数极大（每请求唯一 key）：本地缓存命中率趋零
- 写多读少：缓存失效开销超过命中收益
