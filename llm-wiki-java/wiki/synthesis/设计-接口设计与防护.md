# 接口设计与防护

> 接口防护要按威胁分层选型：入口层拦截恶意流量，业务层保证幂等和状态正确，外部层隔离三方不稳定；默认组合是 **网关限流 + 签名/鉴权 + 幂等 Key + 超时熔断 + 审计日志**。

## 一、防护分层：入口挡流量，业务保正确，外部做隔离

| 层级 | 防什么 | 典型方案 | 结论 |
|------|--------|----------|------|
| 入口层 | 恶意请求、爬虫、突增流量 | 网关限流、鉴权、验证码、黑名单 | 先挡住无效流量 |
| 业务层 | 重复提交、并发状态错乱 | Token、幂等 Key、状态机、唯一索引 | 保证处理正确 |
| 外部层 | 三方超时、三方故障 | 超时、熔断、隔离线程池、异步化 | 不让外部拖垮自己 |

决策维度：

| 问题 | 选型影响 |
|------|----------|
| 是否需要前端协作 | 表单 Token 需要前端带回；幂等 Key 可纯后端 |
| 是否允许最终一致 | 三方调用可异步化；强同步只能超时降级 |
| 是否涉及资金/订单 | 必须有唯一幂等键和状态机 |
| 是否对外开放 | 必须有签名、防重放、AppKey 限流和审计 |

## 二、频率限制：固定窗口、滑动窗口、令牌桶怎么选

| 方案 | 优点 | 缺点 | 推荐场景 |
|------|------|------|----------|
| 固定窗口计数 | 实现简单 | 窗口边界可突刺 | 低风险接口 |
| 滑动窗口 ZSet | 精度高 | Redis 操作多 | 登录失败、短信频控 |
| 令牌桶 | 平滑限流，允许小突发 | 参数需调优 | 网关/API 级 QPS 控制 |
| 漏桶 | 输出最稳定 | 突发体验差 | 下游能力固定的接口 |

登录失败锁定示例：

```lua
local key = KEYS[1]
local lockKey = KEYS[2]
local now = tonumber(ARGV[1])
local windowMs = tonumber(ARGV[2])
local limit = tonumber(ARGV[3])
local lockSec = tonumber(ARGV[4])

redis.call('ZREMRANGEBYSCORE', key, 0, now - windowMs)
redis.call('ZADD', key, now, now)
redis.call('EXPIRE', key, math.ceil(windowMs / 1000))

local count = tonumber(redis.call('ZCARD', key))
if count >= limit then
    redis.call('SET', lockKey, 1, 'EX', lockSec)
    return -1
end
return limit - count
```

Lua 的价值是让清理窗口、计数、锁定成为一个原子操作；否则并发登录失败时可能计数不准。

## 三、防重复提交：Token 机制 vs 幂等 Key

| 方案 | 原理 | 优点 | 代价 | 结论 |
|------|------|------|------|------|
| 表单 Token | 页面加载生成 token，提交时一次性消费 | 防重复语义最强 | 需要前端配合 | 表单/支付确认页首选 |
| 幂等 Key | `userId + api + paramsHash` SETNX | 后端通用 | 参数必须稳定 | 通用 API 首选 |
| 分布式锁 | 同资源串行化 | 保护临界资源 | 不能识别重复语义 | 不作为防重首选 |

```java
String key = "idempotent:" + DigestUtils.md5Hex(userId + api + sortedParams);
Boolean ok = redisTemplate.opsForValue().setIfAbsent(key, "1", 30, TimeUnit.SECONDS);
if (!Boolean.TRUE.equals(ok)) {
    throw new DuplicateRequestException("重复请求");
}
```

反直觉：防重不是加锁。锁解决“同时处理会冲突”，幂等解决“同一个请求重放不能处理两次”。

## 四、开放接口：签名、防重放、限流、审计缺一不可

开放 API 必备字段：

| 字段 | 作用 |
|------|------|
| `appKey` | 标识调用方 |
| `timestamp` | 限制请求有效窗口 |
| `nonce` | 防重放 |
| `sign` | HMAC-SHA256 防篡改 |
| `requestId` | 排查和幂等 |

签名流程：

```
参数按 key 排序
  → 拼接 canonical string
  → HMAC-SHA256(secret, canonical + timestamp + nonce)
  → 服务端校验 timestamp 在 ±5min
  → Redis SETNX nonce:{appKey}:{nonce}
  → 按 appKey 限流
  → 写审计日志
```

| 防线 | 没有时的风险 |
|------|--------------|
| 签名 | 参数被篡改 |
| timestamp | 旧请求长期可用 |
| nonce | 请求可被重放 |
| AppKey 限流 | 单调用方压垮服务 |
| 审计 | 出问题无法追责和排障 |

## 五、三方接口隔离：超时、熔断、异步化

| 方案 | 适用 | 代价 | 结论 |
|------|------|------|------|
| 超时 + 降级 | 所有同步三方调用 | 需定义默认值/失败态 | 必选 |
| 熔断 | 三方错误率持续升高 | 半开探测和规则维护 | 不稳定三方必选 |
| 资源隔离 | 慢请求可能耗尽线程 | 线程池配置成本 | 核心系统必选 |
| 异步化 | 不要求实时返回 | 状态机/回调/轮询复杂 | 最彻底 |

```java
HttpClient client = HttpClient.newBuilder()
    .connectTimeout(Duration.ofMillis(500))
    .build();

HttpRequest req = HttpRequest.newBuilder()
    .timeout(Duration.ofMillis(2000))
    .uri(URI.create(thirdPartyUrl))
    .build();
```

熔断状态机：

```
CLOSED 正常
  → 错误率/慢调用率超过阈值
OPEN 快速失败
  → 等待窗口后放少量请求
HALF_OPEN
  ├─ 成功 → CLOSED
  └─ 失败 → OPEN
```

## 六、Token 安全：短有效期、轮转、黑名单、踢下线

| 防护 | 解决什么 | 代价 | 推荐 |
|------|----------|------|------|
| 短有效期 | 缩小被盗可用窗口 | 续期频繁 | 必选 |
| RefreshToken 轮转 | 防长期 refresh 被盗 | 服务端需存版本 | 高安全场景选 |
| 黑名单 | 注销后立即失效 | Redis 存储 | 主动登出必选 |
| 设备绑定 | 盗用后环境不匹配 | 换网误伤 | 风险操作可选 |
| 单点登录/踢下线 | 拉黑、挤下线 | 服务端有状态 | 管理后台/风控必选 |

```java
// 登录时保存当前有效 tokenId
redisTemplate.opsForValue().set("user_token:" + userId, tokenId, 7, TimeUnit.DAYS);

// 请求时校验
String current = redisTemplate.opsForValue().get("user_token:" + userId);
if (!tokenId.equals(current)) {
    throw new TokenInvalidException("Token 已失效");
}
```

## 七、接口防护决策树

```
接口是否对外开放？
  ├─ 是 → HMAC 签名 + timestamp + nonce + AppKey 限流 + 审计
  └─ 否 → 登录态/JWT + 权限校验

是否可能重复提交？
  ├─ 表单类 → Token 一次性消费
  ├─ API 类 → 幂等 Key SETNX + DB 唯一索引
  └─ 状态流转类 → 状态机 CAS

是否调用三方？
  ├─ 可异步 → 立即返回处理中 + MQ/任务补偿
  └─ 必须同步 → 超时 + 熔断 + 资源隔离
```

## 八、面试追问

### 1. 第三方支付本地超时但三方成功了怎么办？

网络超时不等于三方未处理。支付请求必须有唯一 `outTradeNo`，本地进入待确认状态，定时反查三方；如果三方成功则推进本地订单，如果失败再关闭或重试。不能因为本地超时直接标失败。

---

### 2. Token 被盗是否能伪造登录？

如果只校验 JWT 签名，Token 在有效期内被盗就能使用。降低风险的组合是短有效期、RefreshToken 轮转、黑名单、异常设备检测和必要时踢下线。做不到绝对防盗，只能缩小风险窗口。

---

### 3. 为什么防重复点击不推荐只用分布式锁？

分布式锁保护临界资源，防重保护请求语义。锁过期后重复请求仍可能进入，锁释放时机也不好定义；Token 一次性消费或幂等 Key 更能表达“同一请求只能成功一次”。

---

### 4. Sentinel 和 Hystrix 熔断怎么选？

新系统优先 Sentinel。Hystrix 已停止维护，偏线程隔离模型；Sentinel 规则更动态，支持慢调用比例、异常比例、热点参数等，和 Java 微服务生态更适合。
