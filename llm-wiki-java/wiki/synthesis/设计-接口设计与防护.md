# 接口设计与防护

> 接口防护三层模型：**入口层**（限流/鉴权/防重）→ **业务层**（幂等/超时/熔断）→ **外部层**（三方接口隔离）。每层有不同的防护手段。

---

## 频率限制：5分钟内最多登录3次

**两个子问题**：① 5分钟内计数；② 超限后锁定用户。

```
① 滑动窗口（Redis ZSet）
  Key: login_attempt:{userId}
  Value: ZADD key timestamp timestamp（以时间戳为 score 和 member）
  窗口清理: ZREMRANGEBYSCORE key 0 (now-300s)
  计数: ZCARD key

② 用户锁定（Redis String）
  Key: user_lock:{userId}
  SET user_lock:{userId} 1 EX {lockSeconds}  ← 超限时写入

③ Lua 脚本保证原子性（判断+写入在同一脚本执行）
```

**关键**：用 Lua 脚本将"判断次数 + 写入锁定"合并为原子操作，避免并发时多次触发锁定。

---

## 防重复点击（不依赖分布式锁）

**Token 机制**（前端幂等令牌）：
```
1. 用户打开页面时，后端生成唯一 Token，存入 Redis（key=token, value=1, TTL=60s）
2. 前端提交时携带 Token
3. 后端用 Lua 脚本原子执行：GET token → 存在则 DEL 并处理；不存在则拒绝
4. Token 一次性消费，天然防重
```

**幂等 Key（请求指纹）**：
```
幂等Key = hash(userId + 接口名 + 核心参数)
提交时: SET idempotent:{key} 1 NX EX 30  ← NX 保证只写一次
结果: 写成功→允许处理；写失败→重复请求，直接返回
```

**两种方案对比**：
- Token 机制：需要前后端协作，适合表单提交场景
- 幂等Key：纯后端，适合通用接口防重

## 防恶意刷流量：代码层面纵深防御

**限流只是第一道**，纵深防御体系：

```
第1层：网关限流（令牌桶/滑动窗口）— IP + 用户维度
第2层：接口鉴权（签名验证、JWT 有效期）
第3层：请求指纹（User-Agent + IP + 设备号黑名单）
第4层：业务幂等（防止相同请求重复处理）
第5层：人机验证（验证码/滑块，在敏感操作前）
第6层：账号维度黑名单（Redis SET 存黑名单用户 ID）
```

**对外提供接口注意事项**：
- 调用方鉴权：HMAC 签名（时间戳 + AppKey + 参数排序后签名）
- 防重放攻击：签名中含 timestamp + nonce，服务端校验时间窗口（±5min）
- 限流：按 AppKey 维度，防止一个调用方压垮服务
- 接口日志：记录调用方、入参、出参、耗时，便于审计和排障

## 三方接口超时：三种隔离方案

**核心原则**：不能让三方的不稳定拖垮自己的接口。

### 方案1：异步化（最彻底，但改造最大）
```
用户请求 → 我方收单（持久化到DB/MQ）→ 立即返回"处理中"
                    ↓ 异步
              调用三方接口
                    ↓ 三方回调/我方反查
              通知用户最终结果
```
- 适合：非实时场景（如批量导入、文件处理）
- 要求：有回调机制 or 主动轮询机制

### 方案2：超时 + 降级（同步调用的基础保障）
```java
// 必须设置超时，不能用默认无限等待
HttpClient client = HttpClient.newBuilder()
    .connectTimeout(Duration.ofMillis(500))
    .build();
HttpRequest req = HttpRequest.newBuilder()
    .timeout(Duration.ofMillis(2000))  // 读超时
    .build();

// 超时后降级处理
try {
    return client.send(req, ...);
} catch (HttpTimeoutException e) {
    return fallbackResponse();  // 返回缓存值/默认值
}
```

### 方案3：熔断（避免雪崩，Sentinel/Hystrix）
```
正常 → 错误率/超时率达到阈值 → 熔断（拒绝请求，快速失败）
   ↑                                       ↓（一段时间后）
关闭 ←← 半开（放少量流量试探）←←←←← 开路
```

**资源隔离（Bulkhead）**：为三方调用分配独立线程池，防止三方慢请求耗尽公共线程池。

## 第三方支付超时：到底成没成功？

**现象**：调用方显示超时/失败，但三方显示支付成功。

**根因**：网络超时 ≠ 三方未处理，响应可能在返回路上丢失。

**处理策略**：
```
1. 不要直接认为失败！先查单
2. 调用三方的"查询接口"确认真实状态
   - 已成功 → 更新本地状态为成功
   - 未成功 → 可重试或退款

3. 设计时引入"最终一致"对账机制
   定时任务扫描"待确认"状态订单 → 反查三方 → 补偿
```

**关键设计**：支付请求要有唯一的 `outTradeNo`（商户单号），三方幂等；重试时用相同 `outTradeNo`，三方会直接返回已有结果。

## Token 被窃取：如何防伪造登录

**JWT Token 被盗的影响取决于设计**：

| 防护手段 | 原理 | 代价 |
|---------|------|------|
| **短有效期** | Token 1~2h 过期，配合 RefreshToken 续期 | 用户体验略差 |
| **设备绑定** | Token 中绑定 UA + IP，验证时校验 | 用户换网络会失效 |
| **Token 轮转** | RefreshToken 使用后立即作废，换发新的 | 需要维护 Token 版本 |
| **黑名单** | 注销时将 Token 写入 Redis 黑名单（TTL=剩余有效期）| Redis 存储开销 |
| **登录踢下线** | 用 Redis 记录 userId→当前有效Token，新登录时旧 Token 失效 | 服务器存状态 |

**登录拉黑实现**（踢下线）：
```
Redis Key: user_token:{userId} = {当前有效tokenId}
登录时：SET user_token:{userId} {newTokenId} EX 7day
请求时：验证 JWT 签名 → 检查 Redis 中 tokenId 是否匹配
拉黑时：DEL user_token:{userId} 或写入黑名单
```
