# 接口设计与防护

> 接口防护三层模型：**入口层**（限流/鉴权/防重）→ **业务层**（幂等/超时/熔断）→ **外部层**（三方接口隔离）；每层防护手段不同，缺一层就有对应的漏洞。

## 一、场景概述与问题拆解

**接口防护的三类工程问题**：

| 问题类型 | 场景 | 核心约束 |
|---------|------|---------|
| 频率控制 | 登录失败锁定、短信验证码频控 | 时间窗口 + 原子计数 |
| 防重保护 | 表单重复提交、网络重试 | 幂等唯一性 |
| 外部隔离 | 三方接口超时、三方故障 | 不让外部不稳定拖垮自己 |

**决策关键维度**：是否需要前后端协作、是否允许最终一致、三方接口有无回调机制。

## 二、频率限制与用户锁定

**场景**：5 分钟内最多登录 3 次，超限锁定用户。

```java
// Redis ZSet 滑动窗口 + String 锁定，Lua 原子执行
String lua = """
    local key = KEYS[1]
    local lockKey = KEYS[2]
    local now = tonumber(ARGV[1])
    local windowMs = tonumber(ARGV[2])  -- 300000 = 5分钟
    local limit = tonumber(ARGV[3])     -- 3次
    local lockSec = tonumber(ARGV[4])   -- 锁定秒数

    -- 清除窗口外旧记录
    redis.call('ZREMRANGEBYSCORE', key, 0, now - windowMs)
    -- 加入当前时间戳
    redis.call('ZADD', key, now, now)
    redis.call('EXPIRE', key, windowMs / 1000)

    local count = tonumber(redis.call('ZCARD', key))
    if count >= limit then
        redis.call('SET', lockKey, 1, 'EX', lockSec)
        return -1  -- 超限，已锁定
    end
    return limit - count  -- 剩余次数
""";
```

**TTL 设计要点**：

| Key | 结构 | TTL |
|-----|------|-----|
| `login_attempt:{userId}` | ZSet | windowMs（= 5min） |
| `user_lock:{userId}` | String | 锁定时长（如 30min） |

**为什么用 Lua**：`ZREMRANGEBYSCORE` + `ZADD` + `ZCARD` + `SET` 四步操作必须原子执行，否则并发下可能触发多次锁定或计数不准。

## 三、防重复点击方案对比

| 方案 | 原理 | 适用场景 | 代价 | 结论 |
|------|------|---------|------|------|
| **Token 机制** | 页面加载时后端生成 Token，提交时一次性消费 | 表单提交、重要操作 | 需前后端协作 | 用户体验好，安全性高 |
| **幂等 Key** | 请求指纹（userId+接口+参数 hash），NX 写一次 | 通用接口防重 | 纯后端，参数需稳定 | 通用性强，不需前端改造 |

**Token 机制实现**：

```java
// 1. 生成 Token（页面加载时）
String token = UUID.randomUUID().toString();
redisTemplate.opsForValue().set("token:" + token, "1", 60, TimeUnit.SECONDS);

// 2. 提交时 Lua 原子消费（GET + DEL）
String lua = """
    if redis.call('GET', KEYS[1]) then
        redis.call('DEL', KEYS[1])
        return 1  -- 消费成功
    end
    return 0  -- 不存在或已消费
""";
Long result = redisTemplate.execute(script, List.of("token:" + token));
if (result == 0) throw new DuplicateRequestException("重复提交");
```

**幂等 Key 实现**：

```java
String idempotentKey = DigestUtils.md5Hex(userId + ":" + apiName + ":" + sortedParams);
Boolean success = redisTemplate.opsForValue()
    .setIfAbsent("idempotent:" + idempotentKey, "1", 30, TimeUnit.SECONDS);
if (!Boolean.TRUE.equals(success)) throw new DuplicateRequestException("重复请求");
```

## 四、防恶意刷流量纵深防御

**六层防御体系**：

| 层级 | 手段 | 结论 |
|------|------|------|
| 第 1 层：网关限流 | 令牌桶/滑动窗口，IP + 用户双维度 | 第一道门，拦截大量请求 |
| 第 2 层：接口鉴权 | 签名验证（HMAC）、JWT 有效期 | 无效 Token 在此拦截 |
| 第 3 层：请求指纹 | UA + IP + 设备号黑名单 | 封禁已知恶意设备 |
| 第 4 层：业务幂等 | 防止相同请求重复处理 | 即使绕过前三层也安全 |
| 第 5 层：人机验证 | 验证码/滑块（敏感操作前） | 脚本刷量的最终屏障 |
| 第 6 层：账号黑名单 | Redis SET 存黑名单 userId | 对已确认恶意账号直接拦截 |

**对外提供接口必须包含**：

```
① 调用方鉴权：HMAC-SHA256(timestamp + appKey + 参数排序后签名)
② 防重放：签名含 timestamp + nonce，校验时间窗口（±5min）+ Redis Nonce 去重
③ 限流：按 AppKey 维度限速（防止一个调用方压垮）
④ 审计日志：记录调用方、入参、出参、耗时（便于排障和合规）
```

## 五、三方接口隔离方案

**核心原则**：三方接口超时不能拖垮自己的接口（线程泄漏/资源耗尽）。

| 方案 | 原理 | 适用场景 | 代价 | 结论 |
|------|------|---------|------|------|
| **异步化** | 立即返回"处理中"，异步调三方 | 非实时场景（批量/文件） | 需要回调/轮询机制 | 最彻底，改造最大 |
| **超时 + 降级** | 硬设超时时间，超时返回缓存/默认值 | 同步调用的基础保障 | 需要定义合理的降级策略 | 必须实施 |
| **熔断（Sentinel）** | 错误率达阈值后快速失败，防止雪崩 | 三方长期不稳定 | 需要熔断状态管理 | 三方不稳定时首选 |

**超时 + 降级实现**：

```java
// 必须设置超时，不能用默认无限等待
HttpClient client = HttpClient.newBuilder()
    .connectTimeout(Duration.ofMillis(500))  // 连接超时
    .build();
HttpRequest req = HttpRequest.newBuilder()
    .timeout(Duration.ofMillis(2000))  // 读超时
    .uri(URI.create(thirdPartyUrl))
    .build();

try {
    return client.send(req, HttpResponse.BodyHandlers.ofString());
} catch (HttpTimeoutException e) {
    return fallbackResponse();  // 返回缓存值/默认值
} catch (Exception e) {
    circuitBreaker.recordFailure();  // 记录失败，触发熔断计数
    throw e;
}
```

**熔断状态机**：
```
正常 → 错误率/超时率 ≥ 阈值 → 熔断（拒绝，快速失败）
  ↑                                    ↓ 等待窗口
关闭 ←── 半开（放少量流量试探） ←────── 开路
```

**资源隔离（Bulkhead 舱壁模式）**：为三方调用配置独立线程池，防止慢请求耗尽公共线程池。

## 六、Token 安全防窃取

**JWT Token 被盗的防护手段**：

| 防护手段 | 原理 | 代价 | 结论 |
|---------|------|------|------|
| 短有效期 | Token 1-2h 过期，配合 RefreshToken 续期 | 用户体验略差 | 必选，基础防护 |
| 设备绑定 | Token 中绑定 UA + IP，请求时校验 | 用户换网络会失效 | 安全场景可选 |
| Token 轮转 | RefreshToken 使用后立即作废，换发新的 | 需维护 Token 版本 | 高安全场景必选 |
| 黑名单 | 注销时将 Token 写入 Redis 黑名单（TTL = 剩余有效期）| Redis 存储开销 | 主动注销场景必选 |
| 踢下线 | Redis 记录 userId → 当前有效 tokenId | 服务器存状态 | 单点登录/拉黑必选 |

**登录踢下线实现**：

```java
// 登录时：覆盖 Redis 中的有效 tokenId
String tokenId = UUID.randomUUID().toString();
redisTemplate.opsForValue().set("user_token:" + userId, tokenId, 7, TimeUnit.DAYS);

// 请求时：验证 tokenId 是否匹配
String currentTokenId = (String) redisTemplate.opsForValue().get("user_token:" + userId);
if (!tokenId.equals(currentTokenId)) throw new TokenInvalidException("Token 已失效");

// 拉黑/踢下线时：删除 Redis Key 或写黑名单
redisTemplate.delete("user_token:" + userId);
```

## 七、面试高频追问

**问**：第三方支付超时显示失败，但三方显示成功，是什么问题？
**答**：网络超时 ≠ 三方未处理，响应可能在返回路上丢失。处理策略：①不要直接认为失败；②调三方查询接口确认真实状态；③若已成功，更新本地状态；④设计"待确认"状态，定时任务扫描反查三方做补偿。关键设计：支付请求携带唯一 `outTradeNo`，三方幂等，重试返回历史结果。

**问**：Token 被盗能直接伪造登录吗？
**答**：如果只靠 Token 签名验证（无黑名单/无设备绑定），Token 被盗后在有效期内确实可以伪造。防护组合：短有效期（缩小被盗窗口）+ Token 轮转（RefreshToken 用一次即失效）+ 异常登录检测（异地/异设备告警）+ 必要时强制踢下线（Redis 黑名单）。无法做到完全安全，但可以将风险窗口压缩到分钟级。

**问**：为什么防重复点击不推荐只用分布式锁？
**答**：分布式锁的目的是"串行化并发"，适合保护临界资源；防重复点击的目的是"识别并拒绝重复请求"，Token 机制或幂等 Key 的语义更精确。分布式锁还有：①锁释放时机难控（持有锁的请求处理完才释放，前端无法感知）；②超时释放后第二个请求仍会处理（无法真正防重）。

**问**：Sentinel 熔断和 Hystrix 熔断有什么区别？
**答**：Sentinel 基于统计（错误数/比例/响应时间）触发熔断，规则可动态配置，无需重启；Hystrix 基于线程隔离（每个依赖独立线程池），信号量隔离也支持，但规则配置较静态，已停止维护。生产选 Sentinel：可在控制台实时调整规则，资源开销更小（线程池模式下 Hystrix 线程数多）。

## 八、与其他概念的关系

- 频率限制和防重底层依赖 [[概念-Redis]]：ZSet 滑动窗口、Lua 原子脚本、SETNX 幂等键
- 全链路防护属于 [[主题-三高架构]] 的"高可用"维度：接口防护是防止系统被流量打垮的核心手段
- 防重复提交是 [[概念-幂等设计]] 的应用场景：Token 一次性消费和幂等 Key 都是幂等实现的工程形式
- Token 安全和签名验证属于 [[概念-网络安全]] 范畴：防重放攻击、JWT 安全、HMAC 签名
