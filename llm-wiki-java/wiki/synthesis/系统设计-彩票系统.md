# 系统设计-彩票系统

> 彩票系统的核心不是抢占库存，而是 **截止前可投注、截止后不可变、开奖可验证、结算可幂等**；高并发投注用余额乐观冻结，开奖用状态机防重，百万注单结算用 Kafka 分批并行，资金正确性优先于极致吞吐。

## 一、投注到结算闭环：彩票不是秒杀

```
创建期号 → 开放投注 → 用户下注并冻结余额 → 截止收注
      → 开奖（可验证/第三方源） → 批量结算 → 发放奖金/解冻未中奖资金 → 对账
```

| 维度 | 秒杀 | 彩票 | 选型结论 |
|------|------|------|----------|
| 时序模型 | 谁快谁得，库存扣完即结束 | 先收注，截止后统一开奖 | 彩票不能照搬秒杀预扣库存模型 |
| 高并发核心 | 防超卖 | 防余额超扣、防重复下注 | 余额冻结必须原子 |
| 结果生成 | 用户抢占结果 | 平台/第三方开奖 | 开奖必须可验证、可审计 |
| 结算压力 | 下单时完成大部分结果 | 开奖后集中批量结算 | Kafka 分批并行更重要 |

核心不变量：

| 不变量 | 防线 |
|--------|------|
| 截止后不能新增有效注单 | 期号状态缓存 + DB 状态校验 |
| 同一 `betId` 不能重复下注 | Redis SETNX + DB 唯一索引 |
| 用户余额不能为负 | `available >= amount` 条件更新 |
| 同一期号不能重复开奖 | `UPDATE issue WHERE status='CLOSED'` |
| 同一注单不能重复结算 | `(bet_id, issue_no)` 结算流水唯一索引 |

## 二、投注写路径：幂等下注、期号校验、余额冻结

```
客户端提交 betId/userId/issueNo/content/amount
  → Redis SETNX bet:{betId} 防重复提交
  → 校验 issueNo 状态为 ACCEPTING
  → 钱包余额乐观冻结（available → frozen）
  → 写 bet_order（uk_bet_id）
  → 投递 bet.placed 事件
  → 返回投注成功
```

| 方案 | 一致性 | 性能 | 适用场景 | 结论 |
|------|--------|------|----------|------|
| 分布式锁锁 userId | 强，逻辑直观 | 同用户投注串行 | 冲突极高或资金极敏感 | 大多数彩票投注不首选 |
| DB 乐观锁冻结余额 | 强，事务内完成 | 并发好 | 单用户冲突不高 | 默认选 |
| TCC 冻结资金 | 跨钱包服务强约束 | 实现复杂 | 钱包独立服务且需跨服务一致 | 大型系统选 |

```sql
UPDATE wallet
SET available = available - #{amount},
    frozen = frozen + #{amount},
    version = version + 1
WHERE user_id = #{userId}
  AND available >= #{amount}
  AND version = #{version};
```

限号玩法用 Redis Lua 控制单号码销售上限，DB 仍保留最终结算依据：

```lua
local curr = tonumber(redis.call('INCRBY', KEYS[1], ARGV[1]))
if curr > tonumber(ARGV[2]) then
    redis.call('DECRBY', KEYS[1], ARGV[1])
    return 0
end
return 1
```

## 三、开奖状态机：截止、开奖、结算只能单向推进

```
ACCEPTING ──到截止时间──▶ CLOSED ──抢开奖锁──▶ DRAWING
    │                         │                       │
    └────禁止下注──────────────┘                       ▼
                         DRAWN ──结算任务完成──▶ SETTLED
```

| 当前状态 | 事件 | 下个状态 | 落地方式 |
|----------|------|----------|----------|
| ACCEPTING | 截止时间到 | CLOSED | 定时任务 `UPDATE ... WHERE status='ACCEPTING'` |
| CLOSED | 开奖任务抢占 | DRAWING | 状态 CAS，影响行数为 1 才开奖 |
| DRAWING | 结果生成成功 | DRAWN | 写开奖结果 + 存证 |
| DRAWN | 全量结算完成 | SETTLED | 结算进度表确认无未结算批次 |

```sql
UPDATE lottery_issue
SET status = 'DRAWING', updated_at = NOW()
WHERE issue_no = #{issueNo}
  AND status = 'CLOSED';
```

影响行数为 0 时必须直接退出，不能再生成结果；否则多节点并发开奖会造成后续结算分裂。

## 四、开奖公平性：Commit-Reveal vs 第三方数据源

| 方案 | 原理 | 优点 | 代价 | 推荐 |
|------|------|------|------|------|
| Commit-Reveal | 投注前公布 `Hash(R+Salt)`，截止后公布原文 | 用户可验证平台未改随机数 | 用户仍需信任随机源生成过程 | 平台自主开奖选 |
| 第三方开奖源 | 官方号码、区块哈希、监管源 | 平台不可操控 | 依赖外部源稳定性和延迟 | 监管/强公信力首选 |
| 纯 DB 随机 | 应用直接生成并写库 | 实现最简单 | 不可证明未篡改 | 不推荐 |

```
投注前：生成 R/Salt → 公布 Commit = SHA256(R + Salt)
截止后：公布 R/Salt → 用户验证 SHA256(R + Salt) == Commit
开奖后：结果写 DB + 上链/审计存证 + 推送监管
```

## 五、结算模型：固定赔率 vs 彩池分配

| 维度 | 固定赔率 | 彩池模型 | 结论 |
|------|----------|----------|------|
| 赔率确定 | 投注时锁定 | 开奖后按奖池分配 | 用户体验固定赔率更好 |
| 平台风险 | 赔率错误会亏损 | 抽水后分配，风险低 | 新系统优先彩池模型 |
| 结算复杂度 | 每注按赔率算 | 先汇总中奖注数再分配 | 彩池更适合批量结算 |
| 风控重点 | 限额、反套利 | 防机器人、投注上限 | 固定赔率风控压力更大 |

批量结算链路：

```
开奖事件
  → Master 按 issueNo 分页扫描注单（每批 1000）
  → Kafka 按 userId 分区投递，保证同用户钱包串行
  → Worker 比对中奖、计算奖金
  → 写 settlement_flow（uk_bet_issue）
  → 调钱包发奖/解冻
  → 更新 bet_order = SETTLED
```

```sql
CREATE UNIQUE INDEX uk_settlement_bet_issue
ON settlement_flow(bet_id, issue_no);
```

## 六、生产风险：重复结算、资金差异、截止漂移

| 风险 | 反模式 | 正确做法 |
|------|--------|----------|
| 重复开奖 | 多节点都生成结果 | issue 状态 CAS + 分布式任务单点调度 |
| 重复发奖 | 消费者重复消费无唯一键 | 结算流水唯一索引 + 消费幂等 |
| 余额超扣 | 先查余额再扣减 | `WHERE available >= amount` 原子更新 |
| 截止漂移 | 各节点本地时间不一致 | NTP + 统一服务端截止判断，DB 状态为准 |
| MQ 积压 | 开奖后一次性推百万消息 | 分批投递 + 结算进度表 + 可恢复扫描 |
| 开奖被篡改 | 只写业务 DB | 外部存证/监管推送，审计不可变 |

## 七、彩票系统决策树

```
是否有监管或强公信力要求？
  ├─ 是 → 第三方开奖源 + 外部存证
  └─ 否 → Commit-Reveal，用户可自行验证

赔率模型怎么选？
  ├─ 平台不想承担赔率风险 → 彩池模型
  ├─ 用户体验要求投注时确定收益 → 固定赔率 + 限额/反套利
  └─ 新系统冷启动 → 彩池模型更稳

结算量是否超过 10 万注/期？
  ├─ 是 → Kafka 分批 + Worker 并行 + 结算进度表
  └─ 否 → 单任务分页结算也可接受，但仍要唯一索引幂等
```

## 八、面试追问

### 1. 彩票和秒杀最大的设计差异是什么？

**时序模型不同**。秒杀是抢占有限库存，核心是 Redis Lua 防超卖；彩票是截止后统一开奖，核心是投注期资金冻结、开奖结果不可篡改、结算幂等。把彩票做成秒杀会忽略开奖后集中结算和公平性证明。

---

### 2. 如何保证开奖公平？

平台自主开奖用 Commit-Reveal：投注前公布哈希承诺，截止后公开原随机数和盐，用户可验证未篡改。监管场景用官方号码或区块哈希作为第三方源，结果写入不可变存储或上报监管。

---

### 3. 百万注单如何快速且幂等结算？

开奖后 Master 分页扫描本期注单，每批投递 Kafka，按 userId 分区保证同用户钱包变更串行；Worker 并行比对中奖并批量发奖。`settlement_flow(bet_id, issue_no)` 唯一索引拦截重复消费，结算进度表支持失败续跑。

---

### 4. 固定赔率如何防平台亏损？

固定赔率的核心风险在精算，不在技术。系统只能做限额、异常大额审核、反套利组合检测、单号销售上限；赔率模型错了，技术兜不住亏损。风险敏感的新系统优先彩池模型。

## 九、彩票与 Redis、Kafka、幂等的关系

- [[概念-Redis]]：投注幂等 SETNX、限号 Lua、期号状态缓存都依赖 Redis 原子能力。
- [[机制-Kafka]]：开奖后批量结算用 Kafka 分区并行，按 userId 分区保证钱包串行。
- [[概念-幂等设计]]：`betId`、`settlement_flow(bet_id, issue_no)`、MQ 消费幂等是彩票资金正确性的底线。
- [[机制-Seata]]：钱包独立服务时可借鉴 TCC，Try 冻结、Confirm 扣减、Cancel 解冻。
- [[系统设计-电商系统]]：秒杀的 Redis 预扣和 MQ 削峰可复用，但彩票主线是截止开奖和批量结算。
