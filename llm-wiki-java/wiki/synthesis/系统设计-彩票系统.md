# 系统设计-彩票系统

> 彩票系统与电商秒杀的核心差异：秒杀是"谁快谁得"，彩票是"先截止后开奖"——投注期内可以无限并发下注，截止后才开奖结算。这个差异决定了**开奖前截止 + 开奖结果不可篡改 + 结算幂等**是彩票系统的三条铁律。

## 问题驱动

- 高并发投注下如何防止号码超卖/余额超扣？
- 开奖公平性如何保证？开奖结果如何防篡改？
- 固定赔率 vs 资金池（Pool）模型各有什么代价？
- 大规模结算（百万注单同时开奖）怎么设计？
- 如何识别和处理异常投注（刷单/套利）？

---

## 整体架构

```
用户 App
  │
  ▼
API 网关（鉴权 + 签名 + 限流）
  │
  ├─ 投注服务（Betting Service）
  │     ├─ 幂等校验（betId 唯一）
  │     ├─ 余额冻结（钱包服务）
  │     ├─ 投注单写入（MySQL + MQ）
  │     └─ 缓存投注量统计（Redis）
  │
  ├─ 开奖服务（Draw Service）
  │     ├─ 截止投注（写入禁止）
  │     ├─ 随机数生成（VRF / 第三方源）
  │     └─ 开奖结果持久化（哈希锁定）
  │
  ├─ 结算服务（Settlement Service）
  │     ├─ Kafka 消费开奖事件
  │     ├─ 批量扫注单 → 比中奖号 → 计算奖金
  │     └─ 批量余额发放（钱包服务）
  │
  └─ 风控服务（Risk Control）
        ├─ 投注金额/频率异常检测
        ├─ 高赔率注单人工审核队列
        └─ 设备指纹 + IP 速率限制
```

## 一、投注系统设计

### 下单核心流程

```
客户端提交投注请求（betId, userId, issueNo, betContent, amount）
  │
  ├─ 1. 幂等校验：Redis SETNX betId（已存在则返回"重复投注"）
  │
  ├─ 2. 期号状态校验：issueNo 是否在"接受投注"状态（Redis 缓存）
  │
  ├─ 3. 余额冻结：调用钱包服务冻结 amount（乐观锁 CAS）
  │
  ├─ 4. 写入注单：INSERT INTO bet_order（唯一索引 betId 做兜底幂等）
  │
  ├─ 5. 发 MQ 事件：bet.placed topic（用于结算服务异步消费）
  │
  └─ 6. 返回投注成功
```

### 高并发下的号码并发控制

彩票与商品不同，多数玩法**号码无数量限制**（不存在超卖），并发难点在**余额冻结**：

```
同一用户并发下多注 → 多次扣余额 → 余额可能超扣

解法：乐观锁
UPDATE wallet SET frozen = frozen + #{amount}, available = available - #{amount},
       version = version + 1
WHERE user_id = #{userId} AND available >= #{amount} AND version = #{version}

失败（影响行数=0）→ 重试（最多 3 次），仍失败返回"余额不足"
```

**不用分布式锁的原因**：乐观锁性能更高，单用户余额操作热点程度远低于秒杀热点商品；分布式锁会将同一用户的并发投注串行化，体验差。

### 限号玩法（号码有上限）

部分玩法有"单号码限买"（如某号码最多卖 N 注），类似库存扣减：

```
Redis INCRBY lottery:issue:{issueNo}:num:{betNum} {qty}
  → 返回值 > maxLimit → DECRBY 回滚 → 返回"该号码已售罄"

高并发下用 Lua 脚本原子操作（参考秒杀系统 Redis 预扣减）
```

## 二、开奖系统设计

### 开奖流程状态机

```
ACCEPTING（接受投注）
    │ 到达截止时间 / 手动封盘
    ▼
CLOSED（已截止，不再接受投注）
    │ 触发开奖
    ▼
DRAWING（开奖中）
    │ 随机数生成完成
    ▼
DRAWN（已开奖）
    │ 发 MQ 触发结算
    ▼
SETTLED（已结算）
```

**关键**：状态机用 MySQL 乐观锁 + 状态校验，防止重复开奖：

```sql
UPDATE lottery_issue SET status = 'DRAWING', updated_at = NOW()
WHERE issue_no = ? AND status = 'CLOSED'
-- 影响行数 = 0 说明已被其他节点开奖，直接退出
```

### 可验证随机数（公平性保证）

**问题**：如果开奖随机数由服务器生成，用户无法验证是否被操纵。

**方案：VRF（Verifiable Random Function）或承诺-揭示方案**

```
方案A：承诺-揭示（Commit-Reveal）
  开奖前：服务器生成随机数 R，公布 Hash(R + Salt)（承诺）
  投注截止后：公布 R 和 Salt
  用户验证：Hash(R + Salt) == 公布的承诺值
  → 服务器无法在截止后修改 R（因为承诺已公布）

方案B：第三方开奖源
  使用公开第三方数据作为随机源：
  - 体彩/福彩官方开奖号
  - 指定股票收盘价后两位
  - 区块链特定区块 Hash
  → 开奖结果独立于平台，不可操控
```

### 开奖结果防篡改

```
开奖结果写入 MySQL 后，同步：
  1. 写入区块链 / 第三方存证（哈希上链）
  2. 发送签名消息到 MQ（结算服务验签后处理）
  3. 推送到监管系统（合规要求）
```

## 三、结算系统设计

### 固定赔率 vs 彩池模型

| 维度 | 固定赔率（Fixed Odds）| 彩池模型（Pari-mutuel）|
|------|---------------------|----------------------|
| 赔率确定时机 | 投注时锁定 | 开奖后按彩池分配 |
| 平台风险 | 高（赔率设置不当 → 亏损）| 低（平台抽水后余款分配）|
| 用户体验 | 好（下注时知道能赢多少）| 一般（开奖后才知道赔率）|
| 典型场景 | 体育博彩（让球/大小球）| 福彩双色球、赛马 |
| 高并发结算复杂度 | 高（每注单独计算）| 低（汇总后一次分配）|

**固定赔率结算**：

```
奖金 = 投注金额 × 赔率（赔率在投注时已锁定在 bet_order 中）
```

**彩池模型结算**：

```
彩池总额 = SUM(所有投注金额)
平台抽水 = 彩池总额 × 抽水比例（如 30%）
奖池 = 彩池总额 × (1 - 抽水比例)
中奖人均分 = 奖池 / 中奖注数
```

### 大规模批量结算

百万注单同时结算，**不能单线程循环**：

```
开奖事件（MQ）
    │
    ▼
结算 Master（单节点）
    ├─ 查询本期所有注单（分页，每批 1000）
    ├─ 投递到 Kafka 分区（按 userId 分区，保证同用户串行）
    │
    ▼
结算 Worker（多节点消费 Kafka）
    ├─ 消费一批注单
    ├─ 比对开奖号码，计算奖金
    ├─ 批量调用钱包服务发放奖金（批量 API，减少 RPC 次数）
    ├─ 写入结算流水（幂等：betId + issueNo 唯一索引）
    └─ 更新注单状态 SETTLED

幂等保障：结算流水唯一索引（betId, issueNo），重复消费安全
```

**结算性能估算**：
- 100 万注单 ÷ 100 Worker ÷ 批量 1000 = 10 批/Worker = 约 1-2 分钟完成

## 四、风控与防作弊

### 投注异常检测规则

| 异常类型 | 检测逻辑 | 处置 |
|---------|---------|------|
| 高频下注 | 同用户 1 分钟内 > N 笔 | 触发人机验证 / 临时限速 |
| 异常大额 | 单注 > 阈值 | 进入人工审核队列，延迟确认 |
| 多账户协同 | 同 IP/设备多账户下相同号码 | 标记风险，合并为一个用户处理 |
| 套利行为 | 同一玩法全号码覆盖（保赚）| 检测注单组合，触发反套利规则 |
| 内部人员 | 员工账号 | 禁止投注或结果公布后才能下注 |

### 风控实现要点

```
设备指纹：前端 SDK 采集设备信息（UA/屏幕/字体）→ 服务端聚合为 fingerprint
IP 速率：Redis Sliding Window（用户+IP 双维度）
大额审核：投注成功先入"待审核"状态，异步风控引擎判定，超时自动通过或拒绝
```

## 五、面试话术

**Q：彩票系统和秒杀系统最大的区别是什么？**

> "核心差异在于时序模型。秒杀是'先到先得，库存扣完为止'；彩票是'先截止收注，再统一开奖结算'。这决定了彩票系统的高并发压力不在于抢占，而在于开奖瞬间的批量结算和余额的并发扣减。秒杀用 Redis 预扣减防超卖，彩票则用乐观锁保证余额冻结的原子性，开奖后用 Kafka 分批驱动结算。"

**Q：如何保证开奖公平性？**

> "核心是让开奖结果不可被事后篡改。两种方案：一是承诺-揭示——开奖前公布随机数的哈希值，截止后才公布原始值，用户可以自行验证；二是使用独立第三方数据源（官方彩票号、区块链区块哈希），平台无法控制。同时结果要写入不可变存储或上链存证，防止数据库被改写。"

**Q：百万注单如何快速结算？**

> "用 Kafka 分批分区结算。开奖后 Master 节点将注单分批投递 Kafka，按 userId 分区保证同用户串行。多个 Worker 节点并行消费，每批 1000 条比对中奖、计算奖金、批量调钱包 API 发放，写结算流水用唯一索引保证幂等。百万注单 100 个 Worker 并行通常 1-2 分钟完成。"
