# 系统设计-彩票系统

> 彩票系统与秒杀的核心差异：秒杀是"谁快谁得"，彩票是"先截止后开奖"——投注期内并发下注，截止后统一开奖结算；选型结论的三条铁律：**开奖前截止 + 开奖结果不可篡改 + 结算幂等**。

## 一、场景概述与问题拆解

**彩票 vs 秒杀的时序模型差异**：

| 维度 | 秒杀 | 彩票 | 结论 |
|------|------|------|------|
| 时序模型 | 先到先得，库存扣完为止 | 先截止收注，再统一开奖 | 不同模型，不能套用同一设计 |
| 高并发核心问题 | 库存超卖 | 余额并发超扣 | 彩票重点在余额保护 |
| 防重手段 | Redis Lua 原子扣减 | betId 唯一索引 + 乐观锁 | 方案不同 |
| 结算时机 | 实时（扣减即结束） | 批量（开奖后集中结算） | 彩票结算量级更大 |

**核心工程问题**：
1. 高并发投注下如何防余额超扣？
2. 开奖公平性如何保证？结果如何防篡改？
3. 固定赔率 vs 彩池模型各有什么代价？
4. 百万注单同时开奖如何批量结算？

## 二、投注系统设计

### 下单核心流程

```
客户端提交投注（betId, userId, issueNo, betContent, amount）
  ↓
1. 幂等校验：Redis SETNX betId（已存在 → "重复投注"）
  ↓
2. 期号状态校验：issueNo 是否在"接受投注"状态（Redis 缓存）
  ↓
3. 余额冻结：乐观锁 CAS 扣可用余额（available -= amount, frozen += amount）
  ↓
4. 写入注单：INSERT INTO bet_order（唯一索引 betId 做兜底幂等）
  ↓
5. 发 MQ 事件：bet.placed topic（结算服务异步消费）
  ↓
6. 返回投注成功
```

### 余额并发安全（乐观锁方案）

**为什么不用分布式锁**：乐观锁性能更高，单用户余额热点程度远低于秒杀商品；分布式锁会将同用户并发投注串行化，体验差。

```sql
-- 乐观锁：余额充足时才扣减，版本号防并发
UPDATE wallet
SET frozen   = frozen   + #{amount},
    available = available - #{amount},
    version   = version + 1
WHERE user_id = #{userId}
  AND available >= #{amount}
  AND version = #{version};
-- 影响行数 = 0 → 重试（最多 3 次），仍失败返回"余额不足"
```

### 限号玩法（号码有上限）

```java
// 限号：Redis INCRBY 原子扣减，Lua 脚本保证原子性
String lua = """
    local curr = tonumber(redis.call('INCRBY', KEYS[1], ARGV[1]))
    if curr > tonumber(ARGV[2]) then
        redis.call('DECRBY', KEYS[1], ARGV[1])  -- 回滚
        return 0
    end
    return 1
""";
Long result = redisTemplate.execute(script,
    List.of("lottery:issue:" + issueNo + ":num:" + betNum),
    String.valueOf(qty), String.valueOf(maxLimit));
```

## 三、开奖系统设计

### 状态机防重复开奖

```
ACCEPTING（接受投注）→ CLOSED（已截止）→ DRAWING（开奖中）→ DRAWN（已开奖）→ SETTLED（已结算）
```

```sql
-- 乐观锁 + 状态校验，防止重复开奖
UPDATE lottery_issue SET status = 'DRAWING', updated_at = NOW()
WHERE issue_no = ? AND status = 'CLOSED';
-- 影响行数 = 0 说明已被其他节点开奖，直接退出
```

### 开奖公平性保证

| 方案 | 原理 | 可验证性 | 结论 |
|------|------|---------|------|
| **承诺-揭示（Commit-Reveal）** | 开奖前公布 Hash(R+Salt)，截止后公布 R | 用户可自行验证哈希 | 平台自主开奖且需公信力时选用 |
| **第三方开奖源** | 官方彩票号/区块链区块 Hash | 完全独立，不可操纵 | 监管要求下首选 |

**承诺-揭示实现**：

```
阶段1（投注前）：服务器生成随机数 R，公布 Commit = Hash(R + Salt)
阶段2（截止后）：公布 R 和 Salt
用户验证：Hash(R + Salt) == 之前公布的 Commit？→ 如一致则证明结果未被篡改
```

**开奖结果防篡改**：写 MySQL 后同步写入区块链存证 / 第三方审计 + 推送给监管系统。

## 四、结算方案对比

| 维度 | 固定赔率（Fixed Odds）| 彩池模型（Pari-mutuel）| 结论 |
|------|---------------------|----------------------|------|
| 赔率确定时机 | 投注时锁定 | 开奖后按彩池分配 | 固定赔率对用户体验更好 |
| 平台风险 | 高（赔率设错→亏损）| 低（抽水后余款分配）| 彩池模型平台零风险 |
| 典型场景 | 体育博彩（让球/大小球）| 福彩双色球、赛马 | 场景决定模型 |
| 结算复杂度 | 高（每注单独计算）| 低（汇总后一次分配）| 彩池更易实现批量结算 |

**彩池模型公式**：
```
彩池总额 = SUM(所有投注金额)
奖池     = 彩池总额 × (1 - 抽水比例)    -- 如抽水 30%
中奖人均 = 奖池 / 中奖注数
```

### 百万注单批量结算

```
开奖事件（MQ）
    ↓
结算 Master（单节点）
    ├─ 分页查询本期注单（每批 1000）
    └─ 投递到 Kafka 分区（按 userId 分区，保证同用户串行）
           ↓
    结算 Worker（多节点并行消费）
        ├─ 比对开奖号码，计算奖金
        ├─ 批量调用钱包 API 发放（减少 RPC 次数）
        ├─ 写结算流水（唯一索引：betId + issueNo 防重复）
        └─ 更新注单状态 SETTLED
```

**性能估算**：100 万注单 ÷ 100 Worker ÷ 批量 1000 = 10 批/Worker ≈ 1-2 分钟完成。

## 五、风控与防作弊

| 异常类型 | 检测逻辑 | 处置 | 结论 |
|---------|---------|------|------|
| 高频下注 | 同用户 1 分钟内 > N 笔 | 触发人机验证/临时限速 | 必实施 |
| 异常大额 | 单注 > 阈值 | 进入人工审核队列，延迟确认 | 必实施 |
| 多账户协同 | 同 IP/设备多账号下相同号码 | 标记风险，合并处理 | 中等优先 |
| 套利行为 | 同一玩法全号码覆盖（保赚）| 检测注单组合，触发反套利规则 | 中等优先 |
| 内部人员 | 员工账号 | 禁止投注或结果公布后才能下注 | 合规必选 |

**风控实现**：
```
设备指纹：前端 SDK 采集 UA/屏幕/字体 → 服务端聚合为 fingerprint
IP 速率：Redis 滑动窗口（用户+IP 双维度，Lua 原子执行）
大额审核：投注先入"待审核"，异步风控引擎判定，超时自动通过
```

## 六、生产风险与反模式

| 风险点 | 反模式 | 正确做法 |
|--------|--------|---------|
| 重复开奖 | 未加乐观锁，多节点并发开奖 | 状态机 + `UPDATE WHERE status=CLOSED` CAS 防重 |
| 结算重复入账 | 结算流水无唯一约束 | `(betId, issueNo)` 联合唯一索引，INSERT IGNORE |
| 余额超扣 | 余额校验和扣减非原子 | 乐观锁：`WHERE available >= amount AND version = ?` |
| 开奖被篡改 | 结果只写 DB，无外部存证 | 写 DB 后同步区块链存证或上报监管 |
| 超时导致截止时间不一致 | 各节点时钟不同步 | 使用 Redis 分布式时间 or NTP 同步，避免本地时钟判断 |

## 七、面试高频追问

**问**：彩票系统和秒杀系统最大的区别是什么？
**答**：时序模型不同。秒杀是"先到先得，库存扣完为止"，高并发核心是防超卖（Redis Lua 预扣）；彩票是"先截止收注，再统一开奖结算"，高并发压力不在于抢占，而在于开奖瞬间的批量结算和余额并发冻结。秒杀用 Redis 预扣减，彩票用乐观锁保证余额冻结原子性，开奖后用 Kafka 分批驱动结算。

**问**：如何保证开奖公平性？
**答**：两种方案：①承诺-揭示——开奖前公布随机数的哈希承诺，截止后才公布原始值，用户可自行验证哈希一致性；②使用独立第三方数据源（官方彩票号、区块链区块哈希），平台无法控制。同时结果要写入不可变存储或上链存证，防止 DB 被改写。有监管要求时选第三方数据源。

**问**：百万注单如何快速结算？
**答**：Kafka 分批并行驱动。开奖后 Master 节点将注单分批投递 Kafka（按 userId 分区保证同用户串行）；多 Worker 节点并行消费，每批 1000 条比对中奖、计算奖金、批量调钱包 API 发放；结算流水唯一索引（betId+issueNo）保证幂等。100 万注单 100 个 Worker 通常 1-2 分钟完成。

**问**：固定赔率系统如何防止平台亏损？
**答**：赔率设置时需要精算部门基于历史赔付率和期望利润率计算边际。系统层面的防控：①对异常大额单独进入审核（单注超阈值）；②检测全号覆盖套利行为（某玩家购买所有号码组合保赚）；③设置单号最高销售额上限（限号玩法）。风控是补充，核心还是赔率设计。
