# 账户钱包系统设计

> 平台内置钱包（如彩票/游戏/电商余额账户）与第三方支付渠道不同：它是**平台自己管的资金账本**，需要自己保证余额扣减的原子性、并发安全和资金不丢。核心约束与支付系统一致：**账不能错、钱不能少、流水不能丢**。

## 问题驱动

- 同一用户并发投注，如何保证余额不超扣？
- 冻结模型怎么设计？投注冻结 vs 实际扣减有什么区别？
- 流水表为什么要设计为不可变（append-only）？
- 余额扣减用行锁、乐观锁还是分布式锁，怎么选？
- 充值/提现如何与第三方渠道对账？

## 一、余额三态模型

```
账户总余额 = 可用余额 + 冻结余额

用户看到的"余额" = 可用余额（available）
实际到账/扣减操作的是 available
临时占用（下注/提现审核中）冻结在 frozen
```

| 字段 | 含义 | 变化时机 |
|------|------|---------|
| `available` | 可用余额，用户可感知 | 充值 +，提现/下注冻结 - |
| `frozen` | 冻结余额，已占用但未最终扣减 | 投注时 +，结算/取消时 - |
| `total` | 总余额 = available + frozen | 辅助校验，不单独维护（可计算得出）|

### 彩票场景完整流转

```
充值成功：  available += amount
投注下单：  available -= amount, frozen += amount
中奖结算：  frozen -= betAmount, available += winAmount
未中奖：    frozen -= betAmount（钱平台收走）
投注取消：  frozen -= amount, available += amount（退回）
提现申请：  available -= amount, frozen += amount
提现成功：  frozen -= amount（扣出平台）
提现失败：  frozen -= amount, available += amount（退回）
```

## 二、余额扣减并发安全

### 三种方案对比

| 方案 | 原理 | 适用场景 | 代价 |
|------|------|---------|------|
| **数据库行锁（悲观锁）** | `SELECT ... FOR UPDATE` 锁行，串行执行 | 低并发，强一致要求 | 锁等待，TPS 低（< 500）|
| **乐观锁（CAS）** | UPDATE 时带 version 条件，失败重试 | 中等并发，冲突率低 | 重试开销，冲突高时退化 |
| **Redis 分布式锁** | Redisson 锁住用户账户，串行处理 | 极高并发，Redis 可用 | 额外依赖 Redis，故障影响扩大 |

### 彩票场景推荐方案：乐观锁

彩票投注的余额操作特点：
- 同一用户的并发度通常不高（1~5 并发）
- 冲突概率低
- 响应时间要求高（用户等不起锁）

```sql
-- 乐观锁扣减余额
UPDATE wallet
SET available = available - #{amount},
    frozen    = frozen    + #{amount},
    version   = version   + 1,
    updated_at = NOW()
WHERE user_id = #{userId}
  AND available >= #{amount}     -- 余额充足校验
  AND version   = #{version}     -- 版本号校验（乐观锁）

-- 影响行数 = 0：version 被其他请求修改 → 重新 SELECT 后重试（最多 3 次）
-- 重试 3 次仍失败：降级为 SELECT FOR UPDATE（悲观锁兜底）
```

### 高并发场景（> 1000 TPS）补充方案

```
余额账户热点分片：
  将单个账户的可用余额拆分为 N 个分片（如 10 个）
  每次扣减随机选一个分片
  → 行锁竞争降低 10 倍

  聚合读（用户查余额）：SUM(shard_0..shard_9)
  
  注意：分片模型复杂度高，仅在 TPS > 5000/用户 时考虑
        彩票场景通常不需要（单用户并发不高）
```

## 三、流水表设计

### 核心原则：不可变（Append-Only）

每笔资金变动都写一条流水，**只 INSERT 不 UPDATE**：

```sql
CREATE TABLE wallet_transaction (
    id          BIGINT       NOT NULL AUTO_INCREMENT,
    txn_id      VARCHAR(64)  NOT NULL COMMENT '业务幂等号（betId/orderId/withdrawId）',
    user_id     BIGINT       NOT NULL,
    txn_type    VARCHAR(32)  NOT NULL COMMENT 'FREEZE/UNFREEZE/DEDUCT/CREDIT/WITHDRAW',
    amount      DECIMAL(18,4) NOT NULL COMMENT '正数=入账，负数=出账',
    balance_after DECIMAL(18,4) NOT NULL COMMENT '操作后的可用余额快照',
    biz_no      VARCHAR(64)  NOT NULL COMMENT '关联业务号（期号/订单号）',
    remark      VARCHAR(256),
    created_at  DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,

    UNIQUE KEY uk_txn_id (txn_id),       -- 幂等兜底，重复插入直接失败
    KEY idx_user_time (user_id, created_at)
) COMMENT '钱包流水表';
```

**关键设计点**：
- `balance_after`：记录操作后余额快照，支持**余额重算验证**（任意时刻余额 = 初始余额 + 所有流水 amount 求和）
- `txn_id` 唯一索引：业务幂等键，保证同一操作只落一条流水
- `amount` 带方向（正/负），对账时直接聚合

### 余额一致性校验

```sql
-- 定时任务 / 告警：验证余额快照与流水对账是否一致
SELECT user_id,
       MAX(balance_after) as snapshot_balance,  -- 最新流水的余额快照
       SUM(amount)        as calc_balance        -- 所有流水求和
FROM wallet_transaction
WHERE user_id = #{userId}
GROUP BY user_id
HAVING ABS(snapshot_balance - calc_balance) > 0.001  -- 不一致告警
```

## 四、充值链路设计

```
用户发起充值
    │
    ▼
平台生成充值订单（recharge_order，状态 PENDING）
    │
    ▼
跳转第三方支付（支付宝/微信/银行卡）
    │
    ├─ 同步回调（return_url）：仅做页面跳转，不更新账户（可伪造）
    │
    └─ 异步回调（notify_url）：第三方服务器主动推送
            │
            ├─ 验签（防伪造）
            ├─ 查询第三方订单状态（二次确认，防异步回调伪造）
            ├─ 幂等：SELECT recharge_order WHERE status='SUCCESS' → 已处理则跳过
            ├─ 更新充值订单 PENDING → SUCCESS
            ├─ 余额入账（available += amount，写流水）
            └─ 发 MQ 通知业务层（充值成功推送）
```

**充值幂等关键**：
```
第三方可能多次推送同一个 notify，必须做幂等：
  IF recharge_order.status = 'SUCCESS' → 直接返回 success（不重复入账）
  ELSE IF status = 'PENDING' → 处理入账，更新状态
```

## 五、提现链路设计

提现比充值复杂，因为钱是从平台出去的，**必须先验证再出款**：

```
用户申请提现
    │
    ▼
扣可用余额（available -= amount）+ 加冻结（frozen += amount）
写 PENDING 提现单
    │
    ▼
风控审核（金额 > 阈值 → 人工审核；小额自动通过）
    │
    ├─ 审核拒绝 → 退回余额（available += amount，frozen -= amount）
    │
    └─ 审核通过 → 调第三方出款 API
                     │
                     ├─ 出款成功：frozen -= amount（钱已离开平台）
                     │           写流水 WITHDRAW_SUCCESS
                     │
                     └─ 出款失败：frozen -= amount，available += amount（退回）
                                 写流水 WITHDRAW_FAIL，通知用户
```

**出款幂等**：调第三方出款 API 需携带平台侧唯一单号（withdraw_id），第三方重复收到同一单号直接返回历史结果。

## 六、对账设计

```
日终对账（T+1）：
  平台流水 SUM(amount)
  == 第三方账单 SUM(amount)
  差异分类：
    长款（平台多）→ 检查是否有未通知的充值
    短款（平台少）→ 检查是否有重复出款 / 未到账提现

实时监控：
  每笔充值/提现完成后，推 MQ 到对账服务
  与第三方实时查询接口对比（成功的支付单平台余额是否已更新）

兜底：
  定时任务扫描 PENDING 超时充值单（> 30 分钟）→ 主动查第三方状态
```

## 七、面试话术

**Q：余额扣减并发安全用什么方案，为什么？**

> "彩票/游戏场景同一用户并发度通常不高，优先用乐观锁：UPDATE 时带 version 字段，失败重试最多 3 次。好处是无锁等待、TPS 高；代价是高冲突时重试频繁。如果冲突率确实高（高并发游戏）才考虑分布式锁或余额分片。分布式锁会将并发串行化，体验差；余额分片复杂度高，通常不需要。"

**Q：为什么流水表要 append-only？**

> "核心是审计和校验。只 INSERT 不 UPDATE，每条流水记录一次资金变动的 amount 和操作后余额快照 balance_after。这样可以通过对所有流水求和来重算余额，随时验证账户数据一致性——任何时候 SUM(amount) 对不上 balance_after，就说明有数据问题。如果允许 UPDATE，这条审计链就断了。"

**Q：充值回调处理要注意什么？**

> "三点：一是只信异步回调不信同步回调（同步 return_url 可被伪造）；二是收到回调后必须再调一次第三方查询接口二次确认，防止回调报文被篡改；三是幂等处理，第三方会重复推送，用充值单状态或流水表唯一索引保证不重复入账。"
