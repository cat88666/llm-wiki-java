# 线上问题排查

> 以真实案例为基础的线上问题排查方法论：从现象 → 定位工具 → 根因 → 解决方案，按问题类型（CPU / 内存 / 数据库 / MQ / 基础设施）分类汇总。

---

## 诊断工具速查

| 工具 | 用途 | 关键命令 |
|------|------|---------|
| **Arthas** | 动态插桩，无侵入监控 | `thread -n 3 -i 1000`（CPU高的前N线程）<br>`trace <类> <方法>`（方法调用链耗时）<br>`watch <类> <方法> '{params,returnObj}'`（查看入出参）<br>`sc <类>`（查看运行时类信息/版本） |
| **top** | 进程/线程CPU | `top`（进程）；`top -Hp <pid>`（线程级） |
| **jstack** | 线程堆栈 / 死锁 | `jstack <pid> \| grep -A 30 'deadlock'` |
| **jmap** | 堆 Dump | `jmap -dump:format=b,file=heap.hprof <pid>`（建议 GC 前后各一次对比） |
| **MAT / VisualVM** | 堆 Dump 分析 | 查看大对象、GC Root 引用链 |
| **df / du** | 磁盘使用 | `df -h`（分区）；`du -h --max-depth=1 /var`（目录） |
| **netstat** | 网络连接 | `netstat -antlp`（查看异常外联 IP） |
| **ping / telnet / traceroute** | 网络可达性 | ping（ICMP 连通）；telnet（端口连通）；traceroute（路由路径） |
| **tcpdump / wireshark** | 抓包分析 | `tcpdump -i eth0 -w 1.cap`，用 wireshark 分析 |

**Arthas 字节码插桩原理**：在需要监控的方法入口/出口插入计时字节码（ASM/Javassist），运行期动态修改，无需改源码、无需重启。

## CPU 飙高排查

### 通用排查步骤

```
top                     → 定位 Java 进程 PID
top -Hp <pid>           → 定位 CPU 高的线程 TID
printf '%x\n' <TID>     → 十六进制转换
jstack <pid> | grep -A 30 '<hex_tid>'  → 找到代码行
```

或直接用 Arthas：`thread -n 3 -i 1000`（每隔1000ms采样3次，找持续高占用线程）

### 真实案例与根因

| 现象 | 根因 | 解决 |
|------|------|------|
| 压测 QPS 500 后 CPU 急剧升高 | **BeanValidator** 每次调用都 `Validation.buildDefaultValidatorFactory().getValidator()` 初始化（耗时操作每次重做） | Validator 提到类初始化时创建一次 |
| 压测 QPS 上升后 CPU 升高 | **TDDL Sequence** 每次 insert 重新 new 实例，本地 cache 失效，大量回 DB 取号；同时 DP 工具对 TDDL 日志做正则脱敏 | Sequence 实例改为单例复用 |
| logback AsyncAppender 线程持续占 CPU 50% | `queueSize=256` 太小频繁满队；`discardingThreshold=0` 不丢弃任何级别；`neverBlock=false` 满队阻塞等待 | 加大 queueSize，设 discardingThreshold 合理阈值，配置 neverBlock=true |

**死循环 vs 死锁对 CPU 的影响**（高频考点）：
- **死循环** → CPU **升高**（线程持续在用户态执行，不释放 CPU）
- **死锁** → CPU **不升高，甚至降低**（死锁线程进入 WAITING/BLOCKED 状态，让出 CPU；业务代码无法推进导致吞吐降低）

## Load 飙高排查

### 服务发布后 Load/CPU 短暂飙高后恢复

**根因**：JIT 预热期。JVM 启动初期所有请求走解释执行（慢），大量并发触发解释器持续工作 → CPU/Load 飙高。JIT 检测到热点代码后编译成机器码，性能恢复。

**解决方案**（按优先级）：

| 方案 | 说明 |
|------|------|
| **小流量切流** | 发布后先给 10% 流量，触发 JIT 优化后再放全量 |
| **缓存预热** | Spring 启动时在 `@PostConstruct` 中预加载热点缓存，避免缓存冷启动叠加 JIT 预热 |
| **JwarmUp（Dragonwell JDK）** | 记录上次运行的编译信息，下次启动时提前完成类加载和方法编译，跳过解释阶段 |
| **提前扩容** | 发布前多扩一批机器，保证存量服务器压力不增大 |
| **分更多批次** | 每批发布机器更少，减少单次冲击 |

## FullGC 与内存问题排查

### 通用排查步骤

```
1. 看 GC 监控/日志：确认是老年代还是 MetaSpace 触发
2. jmap 在 GC 前后各取一次 Heap Dump（对比可回收性）
3. MAT 分析：找 Retained Size 最大的对象 → 追溯 GC Root
4. 结合日志/监控时间线定位触发代码
```

### 真实案例与根因

| 现象 | 根因 | 解决 |
|------|------|------|
| MetaSpace 频繁 FullGC | **AviatorEvaluator** 未用缓存模式：每次 `compile(expr)` 生成新 ClassLoader + 匿名类 → MetaSpace 持续增长 | 改为 `compile(cacheKey, expr, true)`，相同表达式复用缓存 |
| 部分机器老年代被打满 FullGC | **数据倾斜**：分布式任务按用户ID前两位分段，22开头用户量远多于其他前缀，承接该段的机器处理对象量数倍于其他机器 | 按前3位分段，或改用主键 ID 分段（均匀分布） |
| 老年代大对象（2G+的 ArrayList）FullGC | 接口未对 `caseId` 做非空校验，前端 bug 未传参 → 后端全量查询60万条记录装入 List → 大对象直接进老年代 | 接口参数强制非空校验；后端防御性编程不信任入参 |
| 双十一 OOM（LinkedList 占73%堆） | 批量支付透支算法（分治+回溯）只考虑时间复杂度，空间复杂度 O(n·2^(n/2)) 在大数据量下爆内存 | 用简单非递归算法替代；**不炫技** |
| Excel 导出 OOM | Apache POI `XSSFWorkbook` 将全量数据加载进 JVM 堆 | 换用 `SXSSFWorkbook`（流式写入，数据写临时文件而非堆） |

## 慢 SQL 排查

### 通用排查步骤

```
1. 慢查询日志 / 监控系统找到慢 SQL
2. EXPLAIN 分析执行计划：重点看 type / key / rows / Extra
3. 结合业务数据量判断是索引缺失、索引区分度低还是回表问题
4. 建索引 / 改 SQL，再 EXPLAIN 验证
```

**EXPLAIN type 快速判断**：

| type | 含义 | 是否正常 |
|------|------|---------|
| `const/eq_ref` | 主键或唯一索引等值查找 | 最优 |
| `ref` | 普通索引等值查找 | 正常 |
| `range` | 索引范围扫描 | 可接受 |
| `index` | 扫描整棵索引树（无法用索引过滤）| 慢，通常是最左前缀未命中 |
| `ALL` | 全表扫描 | 一般需要优化 |

### 真实案例与根因

| 现象 | 根因 | 解决 |
|------|------|------|
| 反欺诈定时任务慢 SQL，`type=index` | 查询条件字段不是联合索引前导列，扫描整颗索引树 | 新建以查询条件字段为前导列的索引 |
| 回流数据后 RT 飙高，走索引仍慢 221ms | 索引未覆盖 `payer_id / payee_id`，命中索引后仍需回表过滤几万条数据（区分度不足） | 新建包含 payer_id/payee_id 的区分度更高的索引 |
| 大买家查询，走索引比全表扫描慢（优化器摇摆）| 索引不包含 `sum_payment`，每行都需回表，IO 成本等同全表扫描 | 将 sum_payment 加入联合索引，构成**覆盖索引**，消除回表 |
| group by 查询 Sort aborted | GROUP BY 字段未命中索引，走 filesort；数据量大时超时 | 建包含 WHERE 条件 + GROUP BY 字段的联合索引，让 filesort 走索引排序 |

## 数据库问题排查

### 死锁

**排查命令**：`SHOW ENGINE INNODB STATUS;`（查看死锁日志，获取两个事务各自持有的锁和等待的锁）

**经典根因**：同一事务中多条 UPDATE 使用不同索引（一条走主键、一条走二级索引），两个并发事务加锁顺序不同 → 循环等待。

**本案例细节**：前缀索引 `idx_seller_transNo(seller_id, fund_transfer_order_no(20))` 中，两条不同记录的 transNo 前20位相同 → 二级索引指向两个主键 → 两个事务各自持有一个主键锁并等待另一个 → 死锁。

**解决思路（优先级）**：
1. **统一加锁顺序**（代码层）：同一事务内操作多行时，所有事务以相同顺序加锁
2. **调整索引**：延长前缀索引长度，避免不同记录映射到同一索引值
3. **减小事务范围**：避免在一个事务中执行多条独立 UPDATE

### 数据库连接池耗尽

**根因**：热点行高并发 UPDATE → InnoDB 行锁排队 → 等待锁的事务持有连接不释放 → 连接池耗尽。

**解法**（乐观锁/CAS 不能解决 InnoDB 行锁问题）：

| 方案 | 说明 | 代价 |
|------|------|------|
| Redis 缓存热点数据 | 高频更新打到 Redis，异步落库 | 实现复杂 |
| 合并批量更新 | 10分钟汇总一次增量，一条 SQL 执行 | 数据实时性降低 |
| 热点数据拆分 | 分散到多行/多表，减少并发冲突 | 查询需合并 |
| 异步更新 | 削峰填谷 | 延迟 |

### 数据库 CPU 打满

**根因**：分布式任务性能优化后扫表加速，同一用户名下多条风险单**并发**修改同一条审核单 → 大量行锁等待。

**解法**：预合案（SQL 聚合 GROUP_CONCAT）→ 一次 UPDATE 批量提交，消除并发冲突；任务耗时从2小时降至10分钟。

## MQ 消费堆积排查

**RocketMQ 消费堆积经典根因**：

| 根因 | 现象 | 解法 |
|------|------|------|
| **订阅关系不一致** | ConsumerGroup 中部分机器订阅了 Topic A，部分订阅 Topic B，Broker 无法确定是否应消费 → 堆积 | 确保同一 ConsumerGroup 所有机器订阅关系完全一致 |
| Spring Cloud Stream + RocketMQ Native 同一 ConsumerId | 两个框架各自注册 MQConsumerInner，RocketMQ 1:1 映射强制覆盖 → Stream 的订阅关系被清除 | 修改其中一个框架的 consumerId，两套配置不共用 |
| 消费者处理耗时过长 | 线程数满，队列堆积 | 加消费线程数 / 优化消费逻辑 |

## 基础设施问题

### Java 进程挂了排查思路

```
先判断：假死 or 真挂？
假死（不响应但进程在）：
  → jstack 看线程状态：死锁 / 活锁 / IO 阻塞 / 长 GC 暂停
真挂（进程消失）：
  → 检查 OOM（/var/log/messages 或 JVM crash log）
  → 检查系统资源：CPU/磁盘/内存满
  → 检查宿主机/容器是否崩溃
  → 检查是否被 kill -9
```

### 磁盘满处理

```bash
df -h                           # 查看哪个分区满了
du -h --max-depth=1 /var        # 找占用大的目录
find /var/log -size +1G         # 找大文件

# 清理已采集的历史日志
rm -rf /app/logs/app.2026-01-*.log

# 若日志文件正在被应用持有，不能直接 rm，用截断替代
> /app/logs/app.log
```

**预防**：Prometheus + Grafana 监控磁盘，80% 时告警；配置 logrotate 自动轮转；日志盘与系统盘分离。

### 网络问题排查

```
ping <host>          → ICMP 连通性（防火墙可能屏蔽）
telnet <host> <port> → TCP 端口连通性
traceroute <host>    → 路由路径，定位哪跳延迟高
curl -w "%{time_total}" → HTTP 全链路耗时（含 DNS/TCP/TLS/首字节时间）
tcpdump + wireshark  → 抓包，分析协议级问题
```

### 服务器挖矿木马

```bash
top                  # kswapd0 进程 CPU 异常高（假冒系统进程）
netstat -antlp       # 发现异常外联 IP（挖矿矿池通常在境外）
ls -la /proc/<pid>/  # 找到进程实际文件路径
rm -rf <dir>         # 删除木马文件
kill <pid>           # 杀进程
crontab -l / -r      # 检查并清理定时任务
```

## 规律总结

1. **CPU 高不等于死锁**：死锁线程 WAITING，不消耗 CPU；死循环才导致 CPU 飙高
2. **索引命中 ≠ 查询快**：回表消耗大，区分度不足时走索引比全表扫描更慢（优化器摇摆）
3. **乐观锁不能避免 InnoDB 行锁等待**：CAS 解决的是业务层并发冲突，数据库 UPDATE 本身仍加行锁
4. **数据倾斜是隐性炸弹**：分布式任务/分库分表按非均匀字段分段，单机数据量可能是平均值的数十倍
5. **大查询入老年代**：未做入参校验的全量查询、未缓存的动态编译结果都会产生大对象直接进老年代
6. **发布时的 CPU/Load 高通常是 JIT 预热**：做小流量缓启动，而非无限扩容
7. **排查顺序**：现象 → 监控缩小范围 → 工具定位热点 → 代码层找根因 → 验证修复效果
