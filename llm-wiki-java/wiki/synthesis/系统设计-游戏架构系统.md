# 系统设计-游戏架构系统

> 一套基于 SOFA-Bolt + Netty + Dubbo 的实时多人在线游戏平台，5 个微服务分层协作：通信基础层（dx-net-bolt）→ 网关路由层（dx-net-gateway）→ 业务服务层（大厅 / 德州扑克 / 竞速大厅），支撑实时牌局、多玩家并发、保险算法等复杂游戏场景。

## 核心设计亮点

### 1. 用户维度线程隔离（dx-net-bolt）

**问题**：不同玩家的游戏请求若共享线程池，可能互相阻塞，导致某玩家的慢请求拖累所有人。

**方案**：`NetCommandExecutorDispatcher` 按 `userId` 创建独立的 `SingleThreadExecutor`，确保每个玩家的请求串行处理、互不干扰。

```java
// NetBoltServerUserProcessor.java
dispatcher.submitFor(userId, new DispatcherThread(...));
// 心跳独立线程池，不抢占业务队列
if (SystemConstant.BOLT_SERVER_HEART_BEAT.equals(requestTypeId)) {
    dispatcher = heartBeatExecutorDispatcher;
}
```

**关键权衡**：
- 优势：用户间完全隔离，天然串行，无需对单用户加锁
- 代价：用户数 × 线程数，线程开销随玩家规模线性增长，需配合线程池上限控制
- 线程数公式：`cpu核数 × 8`（可配置），队列深度 2000

### 2. 工厂模式服务路由（dx-net-gateway）

**问题**：网关需要将客户端请求分发到不同游戏服，硬编码 if-else 难以扩展。

**方案**：`DxServerManager` 实现 `InitializingBean`，自动扫描所有 `DxServerHandle` 实现类，构建 `ServiceTypeEnum.id → Handler` 的 Map，O(1) 路由。

```java
// DxServerManager.java (Spring 自动注入所有实现)
serverHandleMap = dxServerHandleList.stream()
    .collect(Collectors.toMap(k -> k.serviceType().getId(), Function.identity()));
// 路由：O(1)
DxServerHandle handle = serverHandleMap.get(serviceId);
```

新增游戏类型只需新增一个 `DxServerHandle` 实现类，无需修改网关核心代码（开闭原则）。

### 3. 协议智能转换与版本兼容（dx-net-gateway）

**问题**：同一接口 `HALL_MATCH_TABLE` 根据 `matchType` 参数要路由到不同后端（普通大厅 vs 竞速赛事）。

**方案**：`GatewayRequestProcess` 在路由前检查参数，动态修改 `requestTypeId` 并记录转换关系到 `RaceForwardCache`，响应时反向映射回客户端。

```java
// 同一协议根据参数动态路由到不同后端
if (requestTypeId == HallProto.HALL_MATCH_TABLE.getId()) {
    Integer matchType = reqJson.getInteger("matchType");
    if (matchType == 3) {  // 比赛类型 → 路由到竞速大厅
        requestTypeId = RaceProto.HALL_RACE_MATCH_TABLE.getId();
        serviceType = ServiceTypeEnum.RACE;
    }
}
```

### 4. 游戏状态机 + 分布式任务调度（dx-game-texas）

**问题**：德州扑克一手牌包含多个阶段（发手牌→下注→发公牌→结算），每个阶段有超时自动推进逻辑，且需支持多个并发牌桌。

**方案**：任务链模式，每个阶段封装为独立 `Task`，以 `(tableId, handCode, userId)` 为维度注册到 Redis 延迟队列，超时自动触发下一阶段。

```
SendHandCardTask (1.5s延迟)
  ↓ 所有玩家看牌后
UserBetCountdownTask (1.5s间隔，超时自动弃牌)
  ↓ 所有人行动后
SendPublicCardTask (发Flop/Turn/River)
  ↓
ProvisionalSettlementTask (自动结算)
  ↓
NextHandStartTask (开启下一手)
```

任务以 `eventKey = USER_BET_COUNT_DOWN_TABLE_{id}_UID_{uid}_HAND_{code}` 存储，支持：
- 多牌桌并发互不干扰
- 服务重启后状态恢复（Redis持久化）
- 超时自动推进（防止单个玩家导致游戏卡死）

### 5. 保险赔率实时计算（dx-game-texas）

**问题**：保险（Insurance）需要实时计算任意手牌组合的胜率，支持多玩家 all-in 场景，不能预建表（组合爆炸）。

**方案**：`CardInsureSmartOutsService`（1347行）基于组合数学实时枚举剩余牌，精确计算每张 Out 牌的改变胜率概率，动态生成赔率。

核心算法：
- 枚举所有剩余公牌排列（最多 C(47,2) = 1081 种）
- 对每种排列计算牌型强弱
- 统计每张 Out 牌出现后胜负变化
- 赔率 = 承担风险 / 期望收益（精确到小数点后2位）

**关键权衡**：
- 实时计算 vs 建表：实时计算灵活支持所有变体，但 CPU 密集；建表快速但组合空间大（52C2 × 52C2 = 数十亿）
- 最终选择实时计算 + 结果缓存（同一手牌阶段内不变）

### 6. 多级缓存架构（dx-game-hall）

**问题**：房间列表、牌桌状态、规则配置是高频读取的热数据，需要极低延迟。

**方案**：Guava LoadingCache（L1，进程内）+ Redis（L2，分布式）+ MySQL（L3，持久化）三层架构。

```java
// HallDataCache.java
LoadingCache<String, ConcurrentMap<Long, TableInfoRespDto>> hallRoomTableListCache =
    CacheBuilder.newBuilder()
        .expireAfterAccess(10, TimeUnit.HOURS)  // L1: 进程内，10小时TTL
        .build(new CacheLoader<>() {
            public ConcurrentMap<Long, TableInfoRespDto> load(String roomCode) {
                return hallRoomService.getHallRoomTableListByRoomCode(roomCode); // 从L2/L3加载
            }
        });
```

| 层级 | 存储 | 延迟 | 容量 | 特点 |
|------|------|------|------|------|
| L1 | Guava Cache（进程内） | 微秒级 | 受JVM堆限制 | 最快，但不共享 |
| L2 | Redis | 毫秒级 | 大 | 跨进程共享 |
| L3 | MySQL | 毫秒~秒 | 无限 | 持久，冷数据 |

### 7. Netty 高性能优化（dx-net-bolt）

```java
// NetNettyServer.java - 关键配置
bootstrap.option(ChannelOption.ALLOCATOR, PooledByteBufAllocator.DEFAULT) // 内存池
         .childOption(ChannelOption.TCP_NODELAY, true)    // 禁用Nagle，减少延迟
         .childOption(ChannelOption.SO_KEEPALIVE, true);  // TCP长连接保活
// IdleStateHandler: 5分钟无数据断开（清理僵尸连接）
pipeline.addLast(new IdleStateHandler(300, 0, 0));
// 双EventLoopGroup: Boss(接收连接) + Worker(处理请求)
EventLoopGroup bossGroup = new NioEventLoopGroup(1);
EventLoopGroup workerGroup = new NioEventLoopGroup(cpu * 2);
```

对应 [[机制-Netty]] 中的主从 Reactor 多线程模型，Boss 线程专门 accept，Worker 线程处理 I/O 读写，彻底分离。

## 关键权衡

1. **SOFA-Bolt vs Spring WebFlux 响应式网关**：Bolt 是 Dubbo 生态的通信层，与现有服务治理无缝集成；WebFlux 适合 REST 场景但与 Netty WebSocket 有额外适配成本

2. **用户级线程隔离 vs 全局线程池**：用户隔离更适合游戏场景（一个慢玩家不影响其他人），全局线程池利用率更高但有用户间干扰风险

3. **任务链 vs Actor 模型**：任务链实现简单，依托现有 Redis 调度；Actor 模型（如 Akka）更适合纯事件驱动但引入新的系统复杂度

4. **实时保险计算 vs 建表**：实时计算支持多种游戏变体，避免维护大量预计算数据；代价是 CPU 密集，通过手牌级别缓存缓解

- **L3 并发**：[[机制-线程池]] 中用户级 `SingleThreadExecutor` 实现天然串行；[[机制-CAS]] 驱动 `ConcurrentHashMap` 的无锁 TableManager

## 面试回答要点

### 项目一句话介绍
> "负责开发基于 SOFA-Bolt + Netty + Dubbo 的在线游戏平台，包含通信基础层、长连接网关、游戏大厅和德州扑克游戏服4个核心服务，支持实时多人游戏、保险赔率计算等复杂场景。"

### 技术亮点描述顺序（STAR法则）

**亮点1：用户级线程隔离**
- Situation: 游戏请求量大，不同玩家请求共享线程时互相干扰
- Task: 保证每个玩家的操作顺序、互不影响
- Action: 按 userId 创建独立 SingleThreadExecutor，心跳独立线程池
- Result: 彻底消除用户间竞争，不同玩家请求零互扰

**亮点2：游戏状态机分布式任务调度**
- Situation: 德州扑克多阶段游戏需要精确超时推进，且多桌并发
- Task: 设计可靠的分布式游戏流程引擎
- Action: 任务链 + Redis eventKey（含tableId/handCode/userId）唯一标识，服务重启可恢复
- Result: 支持任意数量并发牌桌，单玩家超时不影响其他桌

**亮点3：保险系统实时赔率计算**
- Situation: 德州扑克保险功能需要实时计算胜率，不能用预建表
- Task: 设计高效实时概率计算引擎
- Action: 组合数学枚举剩余牌排列（C(47,2)=1081种），统计Out牌改变胜率，手牌级缓存
- Result: 实时精确，支持多玩家all-in场景，单次计算 <10ms

**亮点4：可扩展的网关路由设计**
- Situation: 平台游戏类型持续增加，网关路由逻辑不能频繁修改
- Task: 设计开闭原则友好的路由系统
- Action: InitializingBean + List<DxServerHandle> 自动扫描，ServiceTypeEnum.id → Handler O(1)路由
- Result: 新增游戏类型只需增加Handler实现，零网关核心改动
