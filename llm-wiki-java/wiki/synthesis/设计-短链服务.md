# 短链服务设计

把长 URL 转成短 URL，核心考点：如何生成不重复的短码、如何高效存储和查找映射、301 vs 302 的取舍、防滥用机制。

---

## 核心问题拆解

设计一个短链服务需要考虑：
1. **短链生成算法**（唯一、短、不可穷举）
2. **存储方案**（Redis 缓存 + MySQL 持久化）
3. **跳转实现**（301 永久 vs 302 临时）
4. **高性能**（热点短链缓存、Redis 集群）
5. **防滥用**（内容扫描、频率限制）
6. **过期机制**
7. **同一长链是否生成相同短链**

---

## 短链生成：MurmurHash + Base62

直接对 URL 用 MD5/SHA 再截取会有碰撞风险；自增 ID 能被穷举；雪花算法太长。
业界常见方案：**MurmurHash + Base62 编码**。

**流程**：
```
1. 输入长 URL：https://www.example.com/very/long/path?query=xxx
2. MurmurHash3 计算 → 32 位整数哈希值（高性能、分布均匀）
3. Base62 编码（0-9 + A-Z + a-z，共 62 字符）→ 6-8 位短码
   例：1234567890 → "1LY7VK"（Base62）
4. 冲突处理：存储时检查是否已存在，冲突则在输入后追加盐值重新哈希
```

**为什么选 Base62 而非 Base64**：
- Base64 包含 `+` 和 `/`，在 URL 中需要额外转义（`%2B`/`%2F`）
- Base62 仅字母+数字，URL 安全，6 位 Base62 可表示 62^6 ≈ 568 亿种组合，够用

**不可穷举的保证**：MurmurHash 输出分布均匀，无明显规律，与纯自增 ID 相比难以按序枚举。

---

## 存储方案

**主存储**：MySQL（持久化，全量数据）
```sql
CREATE TABLE short_url (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  short_code VARCHAR(16) UNIQUE NOT NULL,  -- 唯一索引
  long_url TEXT NOT NULL,
  user_id BIGINT,
  expire_at DATETIME,
  created_at DATETIME
);
```

**缓存层**：Redis（热点访问加速）
```
key:   short:{code}        例：short:1LY7VK
value: https://original-url.com
TTL:   根据过期时间或热度设置
```

**查找流程**：
```
请求短链 → Redis 查找 → 命中直接跳转
                       ↓ miss
                    MySQL 查找 → 命中写回 Redis + 跳转
                              ↓ miss → 404
```

## 跳转：302 而非 301

| | 301 永久重定向 | 302 临时重定向 |
|--|--|--|
| 浏览器缓存 | 缓存，后续直接跳转（不经过服务器）| 不缓存，每次都经过服务器 |
| 访问统计 | ❌ 无法统计（绕过服务器）| ✅ 可统计 PV/UV/设备/来源 |
| 短链修改 | ❌ 已缓存的用户仍跳旧地址 | ✅ 随时可修改目标地址 |
| 性能 | 更高（少一次服务器请求）| 略低 |

**通常选 302**：短链的核心价值在于可追踪（转化统计）和可管理（随时修改/禁用）。
只有内部跳转且目标永不变的场景，才考虑 301 提升性能。

---

## 热点短链优化

热门短链（如营销活动链接）可能被百万次/秒访问：
1. **本地缓存**：应用层 Guava/Caffeine 缓存热门 code → URL 映射，减少 Redis 查询
2. **Redis 分片**：使用 hash tag 确保同一 key 的请求落同一节点，避免热 key 打垮单节点
3. **301 降级**：对确认不会修改的超热链接，临时切换为 301 让浏览器缓存，峰值过后切回 302

---

## 防滥用

短链服务面临的风险：钓鱼网站、恶意传播、接口刷量生成大量垃圾短链。

**防护措施**：
- **目标 URL 黑名单**：对接第三方恶意 URL 检测 API，拦截已知恶意域名
- **创建频率限制**：同 IP/用户限制每小时创建数量（令牌桶限流）
- **接口鉴权**：生成接口需 API Key 或登录态，匿名用户不能创建
- **短码随机性**：Base62 编码自带随机性，不使用自增 ID 防穷举扫描
- **用户绑定**：短链绑定创建者 ID，出问题可追溯、封禁

---

## 过期机制

- Redis TTL：对设置了过期时间的短链，Redis key 直接设 TTL（到期自动清理缓存）
- MySQL：`expire_at` 字段记录，查询时校验 `expire_at > NOW()`，到期返回 410 Gone
- 定时清理（可选）：定期删除 MySQL 中已过期数据，避免表膨胀

---

## 同一长链 → 相同短链（防重）

需要时：
1. MySQL 中 `long_url` 加唯一索引（先 `SELECT`，无结果再 `INSERT`）
2. 创建时加分布式锁（以 MD5(long_url) 为 key），防并发下重复创建
3. 两者同时使用：锁是第一道防线，唯一索引是兜底

不需要时（允许同一长链生成多个短链）：跳过去重逻辑，每次直接生成新短码。

---

## 架构总结

```
用户           短链服务                    存储
  │  生成短链    ├── MurmurHash+Base62生成code
  │────────────►├── Redis SETNX 检查冲突
  │             ├── 写 MySQL + 写 Redis
  │  访问短链    │
  │────────────►├── Redis 查 code→url（命中）→ 302 跳转
                ├── MySQL 查（miss）→ 302 跳转 + 回写 Redis
                └── 404/410（不存在/已过期）
```
