# 云原生选型

> IaaS/PaaS/SaaS/Serverless 的本质区别，以及 Java 后端开发者在面对云原生架构时的关键选型决策。

---

## 云服务交付模型

| 模型 | 全称 | 用户管理范围 | 典型产品 | 类比 |
|------|------|------------|---------|------|
| **IaaS** | 基础设施即服务 | OS + 运行时 + 应用 | 阿里云 ECS、AWS EC2 | 共享厨房（只有锅和灶，自备食材）|
| **PaaS** | 平台即服务 | 应用 + 数据 | 阿里云 EDAS、Heroku | 共享厨房 + 预制菜（只需会炒菜）|
| **SaaS** | 软件即服务 | 仅配置 | 飞书、Salesforce | 点外卖（直接吃，不用做）|
| **Serverless** | 函数即服务 | 仅业务逻辑 | 阿里云 FC、AWS Lambda | 自动按需启动的厨师 |

**核心差异**：从 IaaS 到 Serverless，用户管理的范围越来越小，灵活性降低，但运维成本也越来越低。

---

## 三种云部署模式

| 模式 | 特点 | 适用场景 |
|------|------|---------|
| **公有云** | 共享资源，按量付费，无限扩展 | 初创公司、弹性需求、面向公众应用 |
| **私有云** | 独享资源，高安全性，自主运维 | 政府、金融、医疗（数据合规要求高）|
| **混合云** | 核心数据私有云，峰值流量公有云（云爆发） | 大型企业：日常私有云，大促公有云扩容 |

**云爆发（Cloud Burst）**：私有云满负荷时自动溢出到公有云，用完即释放。解决了"为峰值买机器平时闲置"的问题。

## Serverless 核心特性与 Java 挑战

### 为什么云原生对启动速度要求极高？

```
Serverless 工作模式：
  无请求时 → 服务不运行（不占资源、不计费）
  有请求时 → 按需启动 → 处理请求 → 释放资源

→ 如果启动耗时 10s，用户等 10s 才收到响应 → 不可接受
→ 因此 Serverless 应用启动时间必须在 100ms 以内（理想）
```

### Java 的冷启动问题

Java 应用天然不适合 Serverless：
- JVM 启动 + 类加载 + JIT 预热：通常需要 **3-10 秒**
- Spring Boot 启动：额外 **1-5 秒**

**解决方案对比**：

| 方案 | 原理 | 启动时间 | 代价 |
|------|------|---------|------|
| GraalVM Native Image | AOT 编译为本地可执行文件，无 JVM 启动 | **< 100ms** | 编译时间长，反射/动态代理受限 |
| Spring Boot 3 + GraalVM | Spring 官方支持 AOT 模式 | ~50ms | 需要 Spring 6 + Java 17+，部分功能需手动注册 |
| Quarkus | 为 Kubernetes/Serverless 设计的 Java 框架 | ~100ms | 生态比 Spring 小 |
| 保持 JVM 热（Keep Warm）| 保持函数实例不销毁 | ~10ms | 有资源费用 |

## 无状态设计原则

**无状态**：每次请求独立处理，不依赖之前请求的内存状态。

| 有状态（需要避免）| 无状态（推荐）|
|----------------|------------|
| 本地 JVM 缓存（Guava Cache）| Redis 分布式缓存 |
| 本地文件存储 | OSS/S3 对象存储 |
| Session in JVM 内存 | JWT Token / Redis Session |
| 单机 WebSocket 连接 | 粘性路由 or 消息总线 |

无状态的好处：
1. **水平扩展**：任意加机器，无状态同步问题
2. **故障恢复**：实例挂掉不丢状态，换一台即可
3. **K8s 友好**：Pod 可随时销毁重建

## 选型决策树

```
需要部署 Java 后端服务
         │
是否有 DevOps 能力自运维基础设施？
    No  → 公有云 PaaS（EDAS/ACK）
    Yes → 继续
         │
是否有数据合规/安全要求（金融/政府）？
    Yes → 私有云或混合云
    No  → 公有云
         │
流量是否有明显峰谷（弹性需求强）？
    Yes → Serverless 或 K8s + HPA
    No  → 固定规格 ECS/ECS 弹性伸缩
         │
业务函数是否独立、无状态、执行时间短（< 15min）？
    Yes → Serverless 函数（阿里云 FC / AWS Lambda）
    No  → 容器化部署（K8s/Docker Compose）
```

## Java 后端云原生改造要点

| 改造项 | 传统做法 | 云原生做法 |
|--------|---------|----------|
| 配置管理 | 本地 application.yml | Nacos / Apollo / K8s ConfigMap |
| 日志 | 本地文件，人工收集 | 输出到 stdout，ELK/SLS 采集 |
| 健康检查 | 无 | /actuator/health（K8s liveness/readiness probe）|
| 链路追踪 | 无 | SkyWalking Agent 无侵入注入 TraceId |
| 优雅停机 | 强制 kill | server.shutdown=graceful + SIGTERM 处理 |
| 服务发现 | 配置文件写 IP | Nacos/K8s Service DNS 自动注册 |

## 面试话术

**Q：IaaS/PaaS/SaaS 有什么区别？**

> "从用户管理范围来说递减：IaaS 给你虚拟机，OS 以上自己搞；PaaS 给你运行平台，你写代码部署上去；SaaS 直接给你可用的软件。对 Java 后端开发，我们用的最多是 IaaS（ECS）和 PaaS（容器平台/函数计算）。"

**Q：Serverless 有什么缺点？**

> "最大的挑战是冷启动——Java 应用 JVM 启动慢，在 Serverless 下每次冷启动可能要 3-5 秒。解决方案有 GraalVM Native Image 做 AOT 编译，或者用 Keep Warm 策略保持实例活跃。另外，Serverless 要求应用完全无状态，本地缓存、本地文件都不能用，这对很多老系统是改造成本。"
