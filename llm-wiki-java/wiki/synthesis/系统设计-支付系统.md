# 支付系统

> 支付系统的核心约束：**资金不能错、不能多、不能少**（资损防控）；在分布式环境下，要求每一笔资金变动都有：幂等保护 + 分布式事务 + 多重对账兜底；技术复杂度的根源是"钱"的不可撤销性——代码 Bug 的代价是真实资金损失。

```

    + biz_type          │

                        │

                        │   ┌──────────────────────────────────┐
```

### 1.4 分布式事务：TCC 空回滚 + 悬挂

```
核心：每个 TCC 参与者维护 dist_tx(xid, branch_id, state) 表

──────────── 正常流程 ────────────

Try:     A.available-100, A.frozen+100
         B 创建 PENDING 入账流水
Confirm: A.frozen-100, 写 A OUT 流水
         B PENDING→SUCCESS, 写 B IN 流水
Cancel:  A.frozen-100, A.available+100（解冻）
         B 删除 PENDING 流水

──────────── 空回滚（Cancel 先于 Try 到达）────────────

TC 超时 ──▶ Cancel 先到达
                 │
                 ▼
        查 dist_tx 表：无 Try 记录
                 │
                 ▼
        写 state=CANCELLED，返回成功（不碰账户）
                 │
        ─────────────────────────
        （后续迟到的 Try 到达）
                 │
                 ▼
        查 dist_tx 表：state=CANCELLED ──▶ Try 直接返回失败

──────────── 悬挂（Try 在 Cancel 之后才到达）────────────

与空回滚处理相同：Try 看到 CANCELLED → 拒绝执行 → 资金不被锁死

──────────── 幂等要求 ────────────

Confirm / Cancel 可能被重试多次
  └──▶ 执行前查 dist_tx.state：
          CONFIRMED → 直接返回成功（已执行）
          CANCELLED → 直接返回成功（已执行）
```

### 1.5 双记账 vs 单记账

```
单记账（不可追溯）:

  UPDATE account SET balance = balance - 100 WHERE id = 'A'

  ✗ 100 元去了哪里？没有流水，发生差异无法定位

──────────────────────────────────────────────────────

双记账（完整溯源）:

                 付款方 A              收款方 B
  ┌────────────────────────┐  ┌────────────────────────┐
  │ flow: OUT 100          │  │ flow: IN  100          │
  │ balance_after: 900     │  │ balance_after: 1100    │
  └────────────────────────┘  └────────────────────────┘
                 ↓                        ↓
       ┌─────────────────────────────────────┐
       │  在途户（渠道处理中）               │
       │  pay_in_transit: +100 → 回调后冲销  │
       └─────────────────────────────────────┘

  会计恒等式：∑借（OUT）= ∑贷（IN），任意时刻成立
  余额校验：account.balance = 期初 + ∑IN - ∑OUT  （随时重算验证）

  余额三态防超扣：
  ┌───────────────┬──────────────┬──────────────┐
  │  操作          │ available    │ frozen       │
  ├───────────────┼──────────────┼──────────────┤
  │  发起支付      │     -100     │    +100      │
  │  支付成功      │      0       │    -100      │
  │  支付失败/取消 │    +100      │    -100      │
  └───────────────┴──────────────┴──────────────┘
  CHECK: available ≥ 0（DB 约束兜底，防透支）
```

## 二、统一下单与幂等

**幂等检查双重保障**：

```
商户发起支付（携带 outTradeNo）
    ↓
幂等检查（Redis SETNX + DB 唯一索引双重保障）
    ├── 已存在 → 查当前状态，直接返回（幂等返回）
    └── 不存在 → INSERT IGNORE 支付订单 → 继续流程
```

**支付订单状态机**：

```
INIT → PAYING（调用渠道中）→ SUCCESS（渠道回调成功）
                           └→ FAILED（渠道返回失败）
                           └→ TIMEOUT（超时关单）
SUCCESS → REFUNDING → REFUNDED
```

**状态流转原则**：单向不可逆，防止并发下状态回退覆盖。用乐观锁（version 字段）保证状态流转原子性。

**支付超时为什么不能直接返回失败**：渠道超时不等于交易失败。本地订单保持 `PAYING`，等异步回调或主动查单，查到成功才记账，查到失败再改失败。直接返回失败会导致用户重复支付（第一笔其实已扣款）。

## 三、渠道路由策略

| 路由维度 | 规则示例 | 结论 |
|---------|---------|------|
| 金额路由 | > 5 万走银联大额，< 1000 走微信小额快捷 | 按金额段匹配最优渠道 |
| 业务类型 | 跨行转账走大额支付，个人扫码走快捷支付 | 不同业务属性不同渠道 |
| 成功率兜底 | 微信失败率 > 5% → 自动降级到支付宝 | 熔断降级保成功率 |
| 成本路由 | 相同成功率下优先选手续费最低渠道 | 降低成本 |
| 商户配置 | 特定商户指定专属渠道（VIP 优先路由）| 定制化需求 |

路由策略存 Nacos 配置中心，支持热更新，无需重启。

## 四、分布式事务：TCC 落地

**跨行转账场景（A 向 B 转账 100 元）**：

```
Try 阶段：
  账户 A：检查余额 ≥ 100 → 冻结 100（balance -= 100, frozen += 100）
  账户 B：预创建 PENDING 入账流水

Confirm 阶段：
  账户 A：解冻并扣款（frozen -= 100，记 OUT 流水）
  账户 B：PENDING 流水改 SUCCESS（记 IN 流水）

Cancel 阶段（任一步骤异常）：
  账户 A：解冻（frozen -= 100, balance += 100）
  账户 B：删除 PENDING 流水
```

**空回滚处理**（Cancel 先于 Try 到达）：

```
Cancel 到达 → 查 A 账户无冻结记录 → 直接返回 Cancel 成功（空回滚，不做任何操作）
同时写入 distribute_transaction 表（state=CANCELLED），防止后续 Try 到达时执行
```

**悬挂处理**（Try 在 Cancel 之后才到达）：

```
Try 到达 → 先查 distribute_transaction 表：
  发现 state=CANCELLED → Try 返回失败（防止数据残留导致资金锁死）
```

**TCC vs 消息事务选型**：

| 维度 | TCC | 消息事务（本地消息表） | 结论 |
|------|-----|-------------------|------|
| 一致性 | 最终一致，但更接近同步 | 最终一致 | TCC 更强 |
| 适用场景 | 跨行转账余额实时扣减 | 积分奖励/通知 | 场景决定选型 |
| 代码侵入 | 高（实现 Try/Confirm/Cancel）| 低 | 消息事务更简单 |
| 性能 | 较好（无全局锁）| 好 | 消息事务性能略好 |

**结论**：核心资金链路（扣款、转账）用 TCC；周边操作（积分、通知、对账）用消息事务（最终一致性，成本更低）。

## 五、账务双记账

**会计恒等式**：资产 = 负债 + 所有者权益，每笔资金变动必须借贷平衡。

```
单记账（危险，无从追溯）：
  UPDATE account SET balance = balance - 100 WHERE id = 'A'
  → 100 元去了哪里？没有记录，无法对账

双记账（安全，完整溯源）：
  账户 A（付款方）流水：direction=OUT, amount=100, balance_after=900
  账户 B（收款方）流水：direction=IN,  amount=100, balance_after=1100
  平台中间账户：借贷抵消，净额为 0
```

**流水表核心设计**：

| 字段 | 说明 |
|------|------|
| `out_trade_no` | 业务单号（幂等 key）|
| `account_id` | 账户 ID |
| `direction` | IN / OUT |
| `amount` | 变动金额（正数）|
| `balance_after` | 变动后余额快照 |
| `biz_type` | 业务类型（PAY/REFUND/FREEZE）|

**流水只 INSERT 不 UPDATE**，账户余额 = ∑IN 流水 - ∑OUT 流水（可随时重算验证）。

**余额三态**：

| 操作 | 可用余额 | 冻结余额 | 结论 |
|------|----------|----------|------|
| 充值成功 | 增加 | 不变 | 第三方回调确认后入账 |
| 投注/下单 | 减少 | 增加 | 先冻结，避免超扣 |
| 扣款成功 | 不变 | 减少 | 冻结金额最终扣出 |
| 取消/失败 | 增加 | 减少 | 冻结归还可用 |
| 提现申请 | 减少 | 增加 | 审核/出款中先冻结 |
| 提现成功 | 不变 | 减少 | 钱离开平台 |

**分户账、总账与在途户**：

| 账户类型 | 职责 |
|---------|------|
| 分户账 | 用户维度余额，解决"单个用户的钱对不对" |
| 总账 | 平台科目汇总，解决"平台整本账平不平" |
| 在途户 | 渠道处理中资金，充值回调前挂 `pay_in_transit`，提现出款前挂 `withdraw_in_transit` |

没有在途户的后果：未确认资金当可用余额导致透支风险；对账时无法说清钱在哪个环节。

**分账**：主支付推进到 `SUCCESS` 后，异步生成分账指令，分账独立状态机（`INIT→PROCESSING→SUCCESS/FAIL`），失败进补偿队列。分账是平台后置处理，失败不能回滚主支付。

## 六、并发安全与余额扣减

| 方案 | 原理 | 适用场景 | 代价 | 结论 |
|------|------|---------|------|------|
| 乐观锁 | `UPDATE WHERE version=? AND available>=?` | 中等并发、冲突低 | 失败重试，高冲突退化 | 首选 |
| DB 行锁 | `SELECT ... FOR UPDATE` | 低并发、强一致 | 锁等待，吞吐低 | 备选 |
| 分布式锁 | Redis 锁串行处理 | 多实例并发保护 | 依赖 Redis，链路变长 | 辅助 |
| 余额分桶 | 热点账户拆成 N 个子余额 | 极端热点账户 | 查询/对账/清结算复杂 | 热点时使用 |

**乐观锁标准实现**：

```sql
UPDATE wallet
SET available = available - #{amount},
    frozen    = frozen    + #{amount},
    version   = version   + 1,
    updated_at = NOW()
WHERE user_id = #{userId}
  AND available >= #{amount}   -- 余额充足检查（防超扣）
  AND version = #{version};    -- 版本号防并发
-- 影响行数 = 0 → 重试（最多 3 次），仍失败返回"余额不足"或"繁忙"
```

**不能只依赖 Redis 锁**：Redis 锁不保证 DB 层的数据一致性，DB 条件更新（`available >= amount`）必须作为最终防线。

## 七、清结算与对账

**核心术语**：

- **对账**：核对双方记录，找长款/短款/漏单/重单
- **清分**：按角色拆分一笔交易（商户/平台/渠道/服务商各得多少）
- **清算**：机构间确认应收应付净额（渠道/银行侧）
- **结算**：钱真正打出去，商户到账

例：用户付 100，渠道 0.6，平台 1 → 清分得商户 98.4；清算确认净额；结算商户收到 98.4；对账全程核验。

**对账差异闭环**：

| 差异类型 | 处理方式 |
|---------|---------|
| 长款（内部有渠道无）| 怀疑误记成功，冻结差异单，查流水，必要时冲正 |
| 短款（渠道有内部无）| 回调丢失或本地事务失败，查单补单补记账 |
| 重单（同一笔业务扣两次）| 幂等拦截，已落地走退款/冲正 |
| 漏单（用户付了但订单未成功）| 扫 `PAYING` 状态订单，主动查单补偿 |

补单系统：扫超时中间态 → 调渠道查单 → 推进状态机 → 补记账/通知/分账，全程幂等。

**T+1 对账流程**：

```
每日 02:00 触发
  ① 拉取本系统昨日支付成功流水（内部流水文件）
  ② 拉取各渠道对账文件（微信/支付宝对账 API）
  ③ 双向比对（以 outTradeNo 为 join key）
     ├── 两边都有，金额一致 → 正常
     ├── 内部有，渠道无（长款）→ 发起渠道人工核查
     └── 渠道有，内部无（短款）→ 触发补单流程
  ④ 差异汇总报告 → 推送告警（企微/邮件）
```

**实时对账兜底**：每笔支付成功/失败后，同步发 MQ 消息给对账系统，实时感知差异。T+1 全量对账 + 实时对账构成双重兜底。

## 八、订单超时关闭方案对比

| 方案 | 时间精度 | 可靠性 | 分布式友好 | 结论 |
|------|----------|--------|------------|------|
| 被动关闭（查询时判断）| 低 | 差 | 好 | 只适合兜底，不适合主流程 |
| 定时任务扫表 | 低 | 好 | 需分片 | 小业务量可用 |
| JDK DelayQueue / 时间轮 | 高 | 差 | 差 | 进程重启丢任务，不推荐 |
| Redis 过期监听 | 中 | 差 | 好 | 通知可能丢失，不推荐 |
| Redis ZSet / Redisson 延迟队列 | 高 | 中高 | 好 | 已有 Redis 时优先选 |
| RocketMQ 延迟消息 | 中高 | 高 | 好 | 已有 MQ 且业务量大时选用 |

**关单逻辑必须幂等**：

```java
@Transactional
public void closeIfUnpaid(String orderId) {
    Order order = orderMapper.selectForUpdate(orderId);  // 行锁防并发
    if (order == null || order.getStatus() != OrderStatus.UNPAID) {
        return;  // 幂等：已关闭或已支付，直接返回
    }
    order.setStatus(OrderStatus.CLOSED);
    orderMapper.update(order);
    // 释放库存、优惠券、冻结金额等
}
```

**为什么不推荐无脑用 MQ 做超时关闭**：大量订单提前付款后，延迟消息仍会到期投递，消费时判断跳过，容易造成消息堆积。Redisson 延迟队列可在支付成功时主动移除延迟任务；MQ 方案则依赖消费端幂等和堆积处理能力。

## 九、资损防控三道防线

```
第一道：代码层
  幂等保护（Redis SETNX + DB 唯一索引双重防重复扣款）
  分布式锁（防并发超扣，Redis Redisson）
  TCC/消息事务（防中间状态资损）
  全链路加密（信封加密）：客户端 AES 密钥 → RSA 公钥加密得信封 → AES 加密请求体；服务端 RSA 私钥解信封得 AES 密钥。RSA 只保护 AES 密钥（比 AES 慢 100-1000 倍），AES 负责批量加密。

第二道：数据库层
  余额字段 UNSIGNED（防负数写入，数据库层兜底）
  唯一索引（防重复流水，txHash/out_trade_no）
  事务原子性（扣款+记流水同一事务，要么全成功要么全失败）

第三道：监控对账层
  账户余额异常波动实时告警（突降/突增超阈值）
  T+1 全量对账 + 实时差异告警（短款立即预警）
  人工审核通道（差异金额超限必须人工确认）
```

## 十、支付系统常见问题

### 1. 不丢单 + 不重单怎么保证？

双重幂等：`outTradeNo` 作为幂等键，Redis SETNX（快路）+ DB 唯一索引（慢路兜底）。

渠道超时后状态保持 `PAYING`（不改失败），依赖两条防丢单线：
- 渠道异步回调（webhook + 验签）
- 定时主动查单 Job（30s 扫一次 PAYING 订单）

两条线均幂等消费，回调结果以 payOrderId 唯一索引防重复入账。

---

### 2. 支付超时为什么不能直接返回失败？

渠道超时 ≠ 交易失败。若直接改 `FAILED`，渠道实际已扣款，用户看到失败后重试 → **重复扣款**。

正确做法：保持 `PAYING`，等回调或查单确认实际状态再流转。

---

### 3. 为什么用 TCC 不用 Saga？

| | TCC | Saga |
|---|---|---|
| 中间状态 | Try 阶段冻结，外部不可见 | 补偿前中间状态可被读到 |
| 适合 | 跨行转账（实时冻结，余额不能暂时不对）| 长流程、遗留系统改造 |
| 代价 | 实现三个接口，侵入高 | 只需正向 + 补偿，侵入低 |

资金场景无法容忍中间状态被读到，选 TCC。非核心链路（积分/通知）选 Saga 或消息事务。

---

### 4. 对账差异怎么处理？

| 差异类型 | 含义 | 处理 |
|---|---|---|
| 长款 | 平台有，渠道无 | 向渠道核查，确认无误后保留 |
| 短款 | 渠道有，平台无 | 主动查单确认后补录流水 |
| 重单 | 渠道重复扣款 | 退款一笔 |
| 漏单 | 平台遗漏入账 | 补单 |

差异超阈值 → 告警 → **人工审核**（金额操作不做自动补偿）。

---

### 5. 退款 / 撤销 / 冲正 / 预授权怎么区分？

| 操作 | 时机 | 本质 |
|---|---|---|
| 退款 | 支付成功后，T+N 日 | 新的逆向资金流 |
| 撤销 | 当日清算前 | 取消原交易，成本最低 |
| 冲正 | 系统异常纠偏 | 基于原流水的对称操作 |
| 预授权 | 支付前 | 先冻结额度，完单后扣减或释放 |

一句话：**退款是售后，撤销是取消，冲正是纠错，预授权是先占后扣**。

---

### 6. 分账失败会影响用户付款结果吗？

不会。主支付到达 `SUCCESS` 后，分账指令异步生成，走独立状态机。分账失败进补偿队列重试，不回滚主支付。支付成功是用户侧主交易，分账是平台后置处理。

---

### 7. 订单超时关闭：Redis ZSet 还是 MQ 延迟消息？

| | Redis ZSet / Redisson DelayedQueue | RocketMQ 延迟消息 |
|---|---|---|
| 优点 | 支付成功可主动 remove，不堆积 | 可靠性高，天然持久化 |
| 缺点 | Redis 故障影响任务 | 消费端必须幂等（收到超时消息时判断订单已支付则跳过）|
| 适用 | 订单量 < 百万/天 | 订单量大，已有 RocketMQ |

两种方案的关单逻辑**都必须幂等**。

---

### 8. 余额分桶有什么代价？

分桶解决热点行锁竞争（单账户 > 几十次/秒时），但引入：
- 查余额需 `SUM(sub_balance)`，多一次聚合
- 流水对账逻辑变复杂（需合并所有桶）
- 报表 SQL 复杂度上升

普通场景（< 10次/秒）优先乐观锁，不要过早分桶。

## 十一、与其他概念的关系

- TCC 分布式事务（空回滚/悬挂/反向补偿）参见 [[机制-Seata]]：Seata TCC 模式是支付跨服务转账的标准实现
- 支付系统是 [[概念-幂等设计]] 最严格的应用场景：下单/支付/回调/退款每个环节独立幂等，层次不同
- 余额冻结、分布式锁、延迟队列依赖 [[概念-Redis]]：Redisson 锁、ZSet 延迟队列、SETNX 幂等键
- 账户余额乐观锁和唯一索引依赖 [[概念-MySQL]]：行锁、UNSIGNED 约束、唯一索引防重复流水
- MQ 回调通知、异步对账消息依赖 [[机制-RabbitMQ]]（RocketMQ）：可靠投递保证至少一次语义
- 整体三高设计参见 [[主题-三高架构]]：分库分表（高并发）+ TCC（高一致）+ 对账告警（高可用）
