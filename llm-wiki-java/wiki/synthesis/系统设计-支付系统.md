# 支付系统

> 支付系统的核心约束：**资金不能错、不能多、不能少**（资损防控）；在分布式环境下，这要求对每一笔资金变动都有：幂等保护 + 分布式事务 + 多重对账兜底。技术复杂度的根本来源是"钱"的不可撤销性——代码 Bug 的代价是真实资金损失。

## 问题驱动

**面试高频问题组合**：
- 聚合支付架构怎么设计？多渠道路由如何实现？
- 支付链路的分布式事务怎么做？TCC 的空回滚和悬挂如何处理？
- 如何设计一个不丢单、不重单的支付系统？
- 账务系统为什么要用双记账？和单记账有什么区别？
- 如何做清结算对账？发现差异怎么处理？
- 如何防止并发场景下的资损？

## 统一下单与幂等

每笔支付必须携带**业务幂等号**（商户订单号），确保同一笔交易只处理一次：

```
商户发起支付（携带 outTradeNo）
    ↓
幂等检查（Redis SETNX + DB 唯一索引双重保障）
    ├── 已存在 → 查询当前状态，直接返回（幂等返回）
    └── 不存在 → INSERT 支付订单（INSERT IGNORE）→ 继续流程
```

**支付订单状态机**：
```
INIT → PAYING（调用渠道中）→ SUCCESS（渠道回调成功）
                           └→ FAILED（渠道返回失败）
                           └→ TIMEOUT（超时关单）
SUCCESS → REFUNDING → REFUNDED
```

状态流转必须是单向的（不可逆），防止并发下的状态回退覆盖。

## 渠道路由策略

| 路由维度 | 规则示例 |
|---------|---------|
| 金额路由 | > 5 万走银联大额，< 1000 走微信小额快捷 |
| 业务类型 | 跨行转账走大额支付系统，个人扫码走快捷支付 |
| 成功率兜底 | 微信失败率 > 5% → 自动降级到支付宝（熔断降级）|
| 成本路由 | 相同成功率下优先选手续费最低渠道 |
| 商户配置 | 特定商户指定专属渠道（VIP 商户优先路由）|

路由策略存储在配置中心（Nacos），支持热更新，无需重启。

## 分布式事务：TCC 在支付的落地

**跨行转账场景**（A 向 B 转账 100 元）：

```
Try 阶段：
  账户 A：检查余额 ≥ 100 → 冻结 100（balance -= 100, frozen += 100）
  账户 B：预创建 PENDING 入账流水

Confirm 阶段：
  账户 A：解冻并扣款（frozen -= 100，记 OUT 流水）
  账户 B：PENDING 流水改 SUCCESS（记 IN 流水）

Cancel 阶段（任一步骤异常）：
  账户 A：解冻（frozen -= 100, balance += 100）
  账户 B：删除 PENDING 流水
```

**空回滚**（Cancel 先于 Try 到达）：
```
Cancel 到达 → 查 A 账户无冻结记录 → 直接返回 Cancel 成功（空回滚，不做任何操作）
```

**悬挂**（Try 在 Cancel 之后才到达）：
```
Cancel 先执行完成 → 此后 Try 到达 → 检查是否有对应 Cancel 记录 → 有则 Try 返回失败（防止数据残留）
```

**实现要点**：TCC 需要额外的"防悬挂表"记录已执行的 Cancel，Try 执行前先查这张表。

## 账务双记账

**会计恒等式**：资产 = 负债 + 所有者权益，每笔资金变动必须借贷平衡，金额可追溯。

```
单记账（危险，无从追溯）：
  UPDATE account SET balance = balance - 100 WHERE id = 'A'
  → 100 元去了哪里？没有记录

双记账（安全，完整溯源）：
  账户 A（付款方）流水：direction=OUT, amount=100, balance_after=900
  账户 B（收款方）流水：direction=IN,  amount=100, balance_after=1100
  平台中间账户：借贷抵消，净额为 0
```

**流水表核心设计**：

| 字段 | 说明 |
|------|------|
| id | 雪花 ID（全局唯一）|
| out_trade_no | 业务单号（幂等 key）|
| account_id | 账户 ID |
| direction | IN / OUT |
| amount | 变动金额（正数）|
| balance_after | 变动后余额（快照）|
| biz_type | 业务类型（PAY/REFUND/FREEZE）|
| create_time | 精确到毫秒 |

流水只 INSERT 不 UPDATE，账户余额 = ∑IN 流水 - ∑OUT 流水（可随时重算验证）。

## 账户钱包子系统

平台钱包（彩票、游戏、电商余额）是平台自己管理的资金账本，核心约束是余额扣减原子、并发安全、流水可审计。

### 余额三态模型

```
账户总余额 = 可用余额 + 冻结余额

available：可用余额，用户可感知
frozen：已占用但未最终扣减的冻结金额
total：通常不单独维护，可由 available + frozen 计算
```

| 场景 | 可用余额 | 冻结余额 | 说明 |
|------|----------|----------|------|
| 充值成功 | 增加 | 不变 | 第三方异步回调确认后入账 |
| 投注/下单 | 减少 | 增加 | 先冻结，避免超扣 |
| 扣款成功 | 不变 | 减少 | 冻结金额最终扣出平台 |
| 取消/失败 | 增加 | 减少 | 冻结金额退回可用余额 |
| 提现申请 | 减少 | 增加 | 审核或出款中先冻结 |
| 提现成功 | 不变 | 减少 | 钱离开平台 |

### 余额扣减并发安全

| 方案 | 原理 | 适用场景 | 代价 |
|------|------|----------|------|
| 数据库行锁 | `SELECT ... FOR UPDATE` 串行更新账户行 | 低并发、强一致 | 锁等待，吞吐低 |
| 乐观锁 | `UPDATE ... WHERE version = ? AND available >= ?` | 中等并发、冲突低 | 失败重试，高冲突退化 |
| 分布式锁 | 按账户加 Redis 锁串行处理 | 多实例并发保护 | 依赖 Redis，可用性链路变长 |
| 余额分桶 | 热点账户拆成 N 个子余额 | 极端热点账户 | 查询、对账、清结算复杂 |

```sql
UPDATE wallet
SET available = available - #{amount},
    frozen    = frozen    + #{amount},
    version   = version   + 1,
    updated_at = NOW()
WHERE user_id = #{userId}
  AND available >= #{amount}
  AND version = #{version};
```

影响行数为 0 时重新读取余额后重试，重试仍失败再降级为行锁或返回繁忙。不要只依赖 Redis 锁保护余额，数据库条件更新必须作为最终防线。

### 充值与提现链路

充值只信第三方异步回调，不信同步跳转；收到回调后必须验签，并主动查询第三方订单状态做二次确认。重复回调用充值单状态和流水唯一索引保证幂等。

提现必须先冻结余额，再进入风控审核和第三方出款。审核拒绝或出款失败时解冻退回；出款成功时扣减冻结金额并写提现成功流水。调用第三方出款接口必须携带平台唯一单号，重复请求返回历史结果。

## 清结算与对账

**T+1 对账流程**：

```
每日 02:00 触发
  ↓
① 拉取本系统昨日支付成功流水（内部流水文件）
② 拉取各渠道对账文件（微信/支付宝对账 API）
③ 双向比对（以 outTradeNo 为 join key）
   ├── 两边都有，金额一致 → 正常
   ├── 内部有，渠道无（长款）→ 发起渠道人工核查
   └── 渠道有，内部无（短款）→ 触发补单流程
④ 差异汇总报告 → 推送告警（企微/邮件）
```

**实时对账兜底**：每笔支付成功/失败后，同步发 MQ 消息给对账系统，实现实时差异感知。T+1 全量对账 + 实时对账构成双重兜底。

## OTC 撮合引擎（区块链场外交易）

P2P 场外交易（BTC/ETH）的核心是**订单撮合**：

```
买单簿（价格从高到低，按价格优先/时间次之排序）：
  109 USDT: [Order1(10 BTC), Order2(5 BTC)]
  108 USDT: [Order3(20 BTC)]

卖单簿（价格从低到高）：
  110 USDT: [Order4(8 BTC)]
  111 USDT: [Order5(3 BTC)]

撮合规则：
  新买单 price ≥ 卖单簿最低价 → 立即成交（Taker 吃单）
  新卖单 price ≤ 买单簿最高价 → 立即成交
  无法匹配 → 加入订单簿等待（Maker 挂单）
```

**并发安全**：撮合核心设计为**单线程**（通过 Disruptor 高性能无锁队列接收订单），保证撮合结果全局有序，无需锁。

### 冷热钱包分离

```
热钱包（在线，< 10% 资金）
  ↕ 自动划转（触发阈值）
冷钱包（离线，> 90% 资金）
  → 硬件私钥/多签 → 人工操作

充值流程：用户充值 → 热钱包地址 → 确认 N 个区块 → 入账
提款流程：用户发起 → 热钱包余额检查 → 不足则触发补充告警 → 发起链上转账 → 异步确认入账
```

**对账**：链上交易通过 bitcoinj/web3j 订阅区块事件，与内部流水实时比对。

## 高并发支付链路优化

| 优化方向 | 方案 | 效果 |
|---------|------|------|
| DB 热点行锁 | 账户余额分桶（N 个子余额，扣款随机选桶，查询合计）| 锁竞争降低 N 倍 |
| 异步解耦 | 支付核心（扣款+记账）同步，通知/积分/对账异步 MQ | 同步链路深度从 8 步减到 3 步 |
| 分库分表 | Sharding-JDBC 按商户 ID 哈希分片 | 支撑 100W+ 商户，P99 < 200ms |
| 索引优化 | 商户单号唯一索引（幂等）、状态+时间复合索引（查单）| 慢 SQL 消除 |
| 连接池调优 | HikariCP maximumPoolSize 根据压测数据设定 | 消除连接等待 |

## 订单超时关闭

支付订单创建后未付款需要自动关闭，常见模式还包括超时自动收货、到期退款、到期锁定。关键不是“定时触发”，而是触发后关闭逻辑必须幂等。

| 方案 | 时间精度 | 可靠性 | 分布式友好 | 结论 |
|------|----------|--------|------------|------|
| 被动关闭 | 低 | 差 | 好 | 只适合兜底，不适合作主流程 |
| 定时任务扫表 | 低 | 好 | 需分片 | 小业务量可用 |
| JDK DelayQueue / 时间轮 | 高 | 差 | 差 | 进程重启丢任务，不推荐 |
| Redis 过期监听 | 中 | 差 | 好 | 通知可能丢，不推荐 |
| Redis ZSet / Redisson 延迟队列 | 高 | 中高 | 好 | 已有 Redis 时优先 |
| RocketMQ / RabbitMQ 延迟消息 | 中高 | 高 | 好 | 大业务量、已有 MQ 时可用 |

```java
@Transactional
public void closeIfUnpaid(String orderId) {
    Order order = orderMapper.selectForUpdate(orderId);
    if (order == null || order.getStatus() != OrderStatus.UNPAID) {
        return;
    }
    order.setStatus(OrderStatus.CLOSED);
    orderMapper.update(order);
    // 释放库存、优惠券、冻结金额等
}
```

不建议无脑用 MQ 做超时关闭：大量订单提前付款后，延迟消息仍会到期投递，只能消费时判断跳过，容易造成消息堆积。Redisson 延迟队列可在支付成功时主动移除延迟任务；MQ 方案则更依赖消费端幂等。

## 资损防控三道防线

```
第一道：代码层
  幂等保护（防重复扣款）
  分布式锁（防并发超卖）
  TCC/消息事务（防中间状态资损）

第二道：数据库层
  余额字段 UNSIGNED（防负数写入）
  唯一索引（防重复流水）
  事务原子性（扣款+记流水原子完成）

第三道：监控对账层
  账户余额异常波动实时告警（突降/突增超阈值）
  T+1 全量对账 + 实时差异告警
  人工审核通道（差异金额超限必须人工确认）
```

## 关键权衡

1. **TCC vs 消息事务**：TCC 适合强一致要求（跨行转账余额实时扣减）；消息事务适合最终一致（积分奖励、通知），成本更低
2. **同步链路深度 vs 一致性**：链路越短延迟越低，但过度异步化会让对账复杂度倍增；核心资金变动必须同步，周边操作异步
3. **实时对账 vs T+1 对账**：实时对账发现差异快但系统压力大；T+1 全量对账准确但滞后；生产环境两者结合
4. **余额分桶粒度**：分桶数越多并发越好，但查询合计的 SQL 越复杂，账期统计性能越差；根据热点程度选择 4-16 桶
5. **幂等粒度**：商户单号幂等（防重复下单）vs 渠道流水号幂等（防渠道重复回调）——支付系统两者都要，层次不同

## 与其他概念的关系

- **核心依赖 [[设计-分布式事务]]**：TCC 是支付跨行转账的标准方案；空回滚/悬挂是 TCC 生产落地必须解决的问题
- **核心依赖 [[概念-幂等设计]]**：支付系统是幂等设计最严格的应用场景，下单/支付/回调/退款每个环节独立幂等
- **依赖 [[概念-分库分表]]**：百万商户规模下支付流水表必须分库分表（按商户 ID 哈希分片）
- **依赖 [[机制-Redis分布式锁]]**：余额分桶更新、撮合引擎并发保护、支付状态 check-then-act 操作
- **依赖 [[机制-消息队列可靠性]]**：支付回调通知、异步对账消息、TCC 超时补偿均依赖 MQ 可靠投递
- **依赖 [[机制-InnoDB锁]]**：账户余额扣减最终依赖数据库条件更新、行锁或乐观锁兜底
