# 系统设计-短链服务

> 短链服务的核心决策不是"怎么把 URL 变短"，而是短码唯一性、跳转可统计性、热点访问承载和滥用治理之间的取舍；面向公网营销和增长场景，默认选 **MurmurHash + Base62 + 302 + MySQL 持久化 + Redis/Caffeine 多级缓存**。

## 一、短链业务闭环：创建、跳转、统计、治理

| 决策点 | 关键约束 | 推荐结论 |
|--------|----------|----------|
| 短码生成 | 短、唯一、不可连续穷举 | 公网短链选 MurmurHash + Base62，唯一索引兜底 |
| 是否同链唯一 | 节省存储 vs 允许多渠道统计 | 营销投放默认允许同一长链生成多短链 |
| 跳转语义 | 统计与可修改 vs 浏览器缓存 | 默认 302；目标永不变且不统计才用 301 |
| 主存储 | 权威数据、过期、审计 | MySQL 持久化，`short_code` 唯一 |
| 读加速 | 短链读多写少 | Redis 缓存 + 本地 Caffeine 热点缓存 |
| 防滥用 | 钓鱼、批量创建、枚举扫描 | URL 安全检测 + 鉴权 + Redis 滑动窗口 |
| 过期清理 | 存储膨胀、过期访问语义 | Redis TTL + MySQL `expire_at`，过期返回 410 |

读写链路分离：

```
创建链路：鉴权/频控 → URL 安全检测 → 生成 shortCode → INSERT 唯一索引 → 写 Redis

跳转链路：GET /{code}
  → Caffeine 本地缓存
  → Redis GET short:{code}
  → MySQL 按 short_code 查询
  → 过期校验/统计埋点
  → 302 Location: long_url
```

## 二、短码生成：MurmurHash + Base62 与碰撞重试

| 方案 | 短码长度 | 唯一性 | 安全性 | 结论 |
|------|----------|--------|--------|------|
| 自增 ID + Base62 | 4-6 位 | 天然唯一 | 连续可猜，易扫描 | 内部系统可用，公网不推荐 |
| 雪花 ID + Base62 | 10-11 位 | 天然唯一 | 较难猜 | 太长，不适合短链核心诉求 |
| MD5/SHA 截取 | 6-8 位 | 有碰撞 | 难猜 | 可用但计算和碰撞处理不如 MurmurHash 轻 |
| MurmurHash + Base62 | 6-8 位 | 需唯一索引兜底 | 分布均匀，不连续 | 公网短链首选 |

生成流程：

```
longUrl + salt
  → MurmurHash3 生成 32/64 bit hash
  → Base62 编码（0-9A-Za-z，URL 安全）
  → 截取 6-8 位 shortCode
  → INSERT short_url(short_code, long_url)
       ├─ 成功：返回短链
       └─ 唯一键冲突：salt++ 后重试，超过阈值改用更长码
```

```sql
CREATE TABLE short_url (
    id          BIGINT PRIMARY KEY AUTO_INCREMENT,
    short_code  VARCHAR(16) NOT NULL,
    long_url    TEXT NOT NULL,
    url_hash    CHAR(32) NOT NULL,
    user_id     BIGINT,
    status      TINYINT NOT NULL DEFAULT 1,
    expire_at   DATETIME,
    created_at  DATETIME NOT NULL,
    updated_at  DATETIME NOT NULL,
    UNIQUE KEY uk_short_code(short_code),
    KEY idx_user_created(user_id, created_at),
    KEY idx_expire_at(expire_at)
);
```

反直觉：6 位 Base62 有 `62^6 ≈ 568` 亿空间，不代表不需要查重。只要采用哈希截断，就必须让 DB 唯一索引做最终裁决。

## 三、存储与缓存：MySQL 做权威，Redis/Caffeine 加速跳转

| 层级 | 存什么 | 作用 | 失效策略 |
|------|--------|------|----------|
| MySQL | `short_code → long_url/status/expire_at` | 权威数据、审计、过期判断 | 永久保存或离线归档 |
| Redis | `short:{code} → long_url` | 跨实例缓存，承接大部分跳转 | TTL 对齐 `expire_at`，无过期可设 24h |
| Caffeine | 热点 code 本地映射 | 抗热 key，微秒级返回 | 1-5 分钟短 TTL，修改时广播失效 |

```java
public String resolve(String code) {
    return localCache.get(code, c -> {
        String url = redis.get("short:" + c);
        if (url != null) return url;

        ShortUrl row = shortUrlMapper.selectByCode(c);
        if (row == null || row.disabled()) throw new NotFoundException();
        if (row.expired()) throw new GoneException();

        redis.setex("short:" + c, row.ttlSecondsOrDefault(), row.longUrl());
        return row.longUrl();
    });
}
```

缓存一致性结论：短链目标通常写少读多，修改/禁用时采用 Cache-Aside，**先删 Redis 和本地缓存，再更新 DB 或更新后广播失效都可以接受短窗口不一致**；强一致成本高，不适合普通短链跳转链路。

## 四、跳转语义：默认 302，极端热点才临时 301

| 维度 | 301 永久重定向 | 302 临时重定向 | 推荐 |
|------|----------------|----------------|------|
| 访问统计 | 浏览器后续绕过服务端，统计丢失 | 每次经过服务端，可统计 PV/UV | 需要统计选 302 |
| 目标修改 | 浏览器可能长期缓存旧目标 | 可随时切目标 | 需要运营管理选 302 |
| 性能 | 最好，浏览器/CDN 可缓存 | 多一次服务端访问 | 超热点降级选 301 |
| 风险 | 错误目标难纠正 | 服务端压力更高 | 公网短链默认 302 |

> 面试口径：短链通常选 302，因为短链平台的价值在统计、风控和可运营；301 只适合目标永不变、无需统计、且访问量大到服务端必须卸载的内部或活动场景。

## 五、热点短链：本地缓存、Redis 分片与降级策略

| 热点级别 | 触发条件 | 方案 | 结论 |
|----------|----------|------|------|
| 普通热点 | 单 code 万级 QPS | Caffeine + Redis | 默认方案 |
| 极端热点 | 单 code 十万级 QPS | 本地缓存预热 + Redis Cluster | 避免 Redis 热 key |
| 活动洪峰 | 单 code 百万级 QPS | CDN/浏览器 301 临时降级 | 牺牲实时统计保可用 |

热点处理链路：

```
热点识别（访问计数/Redis hotkey）
  → 推送热点 code 到各应用本地缓存
  → 缩短统计链路：异步批量上报 PV/UV
  → 仍扛不住：活动期切 301 或 CDN 边缘跳转
  → 峰值结束：恢复 302，重新纳入精确统计
```

## 六、防滥用与过期：黑名单、频控、追溯、410

| 风险 | 触发方式 | 防护 | 结论 |
|------|----------|------|------|
| 钓鱼/恶意 URL | 创建恶意目标 | 第三方安全检测 + 域名黑名单 | 必选 |
| 批量刷创建 | 脚本调用创建接口 | 用户/IP/AppKey 滑动窗口 | 必选 |
| 短码枚举 | 扫描 `/aaaaaa` | 非连续短码 + 404 频控 | 公网必须防 |
| 过期仍访问 | 活动结束后访问 | Redis TTL + MySQL `expire_at` | 返回 410 Gone |
| 违规追责 | 用户创建恶意短链 | 绑定 `user_id/app_key` + 审计日志 | 必选 |

```java
// 创建频控：同一 userId 每分钟最多 60 次
ZADD rate:short:create:{userId} now requestId
ZREMRANGEBYSCORE rate:short:create:{userId} 0 now-60000
ZCARD rate:short:create:{userId}
```

## 七、短链选型决策树

```
是否需要 PV/UV/来源统计？
  ├─ 是 → 302
  └─ 否 → 目标永不变且访问极高？
           ├─ 是 → 301/CDN 跳转
           └─ 否 → 302，保留运营控制能力

短码是否公网可访问？
  ├─ 是 → MurmurHash + Base62 + 唯一索引 + 非连续码
  └─ 否 → 自增 ID + Base62 可接受

是否要求同一长链返回同一短链？
  ├─ 是 → `url_hash` 唯一 + 分布式锁防并发重复创建
  └─ 否 → 每次生成新 code，便于渠道归因
```

## 八、面试追问

### 1. 短码冲突怎么处理？

**DB 唯一索引是最终兜底**。生成短码后直接插入，成功即返回；冲突则追加 salt 重新 hash，重试 3 次仍冲突就提升码长。不要先查再插作为唯一保障，因为并发下两个请求可能同时查不到。

---

### 2. 为什么不用雪花算法？

雪花 ID 的优势是全局唯一，但 Base62 后通常 10-11 位，长度偏长；而且 ID 有时间趋势，攻击者更容易推测生成节奏。短链更看重短、分布均匀、不可连续枚举，所以公网场景选 MurmurHash + Base62。

---

### 3. Redis 和 MySQL 怎么保持一致？

短链属于写少读多，允许短暂不一致。创建时写 DB 成功后写 Redis；修改/禁用时删除缓存并更新 DB，再通过消息广播清本地缓存。真正不能错的是禁用恶意短链，此时可以在跳转链路对高风险 code 回源校验状态。

---

### 4. 如何防止短链被用于钓鱼？

创建前做 URL 安全检测和域名黑名单，创建接口必须鉴权并记录创建者，访问异常时能按 userId/appKey 批量封禁。检测有滞后性，所以追溯和快速下线能力比单次检测更关键。

## 九、短链与缓存、幂等、防穿透的关系

- [[概念-Redis]]：Redis String 保存 `code → url`，ZSet 实现创建频控，热点 code 通过本地缓存分担 Redis 压力。
- [[概念-BitMap]]：可用 Bloom Filter 思路提前判断 shortCode 是否存在，拦截不存在 code 的 DB 穿透。
- [[概念-幂等设计]]：同一长链唯一时，`url_hash` 唯一索引 + 分布式锁属于幂等创建；跳转链路本身应无副作用，统计异步幂等聚合。
