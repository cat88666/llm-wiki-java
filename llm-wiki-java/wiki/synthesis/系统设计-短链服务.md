# 系统设计-短链服务

> 短链服务的核心工程决策：短码生成算法（唯一性、不可穷举）、跳转语义（302 vs 301 影响统计能力）、同一长链是否生成相同短链（业务需求决定）；选型结论取决于对访问统计和 CDN 友好度的权衡。

## 一、场景概述与问题拆解

**七个核心设计问题**：

| 问题 | 关键约束 | 结论方向 |
|------|---------|---------|
| 短码生成算法 | 唯一、短、不可穷举 | MurmurHash + Base62 |
| 存储方案 | 持久化 + 热点加速 | MySQL 持久化 + Redis 缓存 |
| 跳转语义 | 统计 vs 性能 | 302（可统计）优先 |
| 热点优化 | 百万次/秒访问 | 本地 Caffeine + Redis 多级缓存 |
| 防滥用 | 钓鱼/刷量 | 黑名单 + 频率限制 + 鉴权 |
| 过期机制 | 存储膨胀 | Redis TTL + MySQL expire_at |
| 同一长链防重 | 节省存储 vs 实现复杂 | 按业务需求选择 |

## 二、短链生成算法对比

| 算法 | 短码长度 | 唯一性 | 可穷举 | 结论 |
|------|---------|--------|--------|------|
| MD5/SHA 截取 | 6-8 位 | 有碰撞风险 | 难以穷举 | 碰撞处理复杂 |
| 自增 ID | 4-6 位 | 唯一 | **容易穷举**（1,2,3...）| 不推荐（安全性差） |
| 雪花算法 | 18+ 位 | 唯一 | 难以穷举 | 太长，不适合短链 |
| **MurmurHash + Base62** | 6 位 | 碰撞极低 | 难以穷举 | **推荐** |

**MurmurHash + Base62 流程**：

```
1. 输入长 URL：https://www.example.com/very/long/path?query=xxx
2. MurmurHash3 计算 → 32 位整数哈希值（高性能、分布均匀）
3. Base62 编码（0-9 + A-Z + a-z，62 字符）→ 6 位短码
   例：1234567890 → "1LY7VK"
4. 冲突处理：DB 检查是否已存在，冲突则在 URL 后追加盐值重新哈希（最多 3 次）
```

**为什么选 Base62 而非 Base64**：Base64 包含 `+` 和 `/`，URL 中需要额外转义（`%2B`/`%2F`）；Base62 仅字母+数字，天然 URL 安全。6 位 Base62 可表示 62^6 ≈ 568 亿种组合，足够使用。

**碰撞概率估算**：568 亿个槽位，1 亿个短链，碰撞概率约 0.17%（生日问题估算）——这是设计上允许有冲突处理逻辑的原因。

## 三、存储方案

**主存储（MySQL）**：

```sql
CREATE TABLE short_url (
    id         BIGINT PRIMARY KEY AUTO_INCREMENT,
    short_code VARCHAR(16) UNIQUE NOT NULL,  -- 唯一索引，防重复
    long_url   TEXT NOT NULL,
    user_id    BIGINT,
    expire_at  DATETIME,
    created_at DATETIME
);
```

**缓存层（Redis）**：

```
key:   short:{code}        例：short:1LY7VK
value: https://original-url.com
TTL:   按过期时间设置，无过期时间则 24h（热度衰减）
```

**查找流程**：

```
请求 /1LY7VK
  ↓ Redis GET short:1LY7VK
  ├─ 命中 → 302 跳转（µs 级）
  └─ miss
       ↓ MySQL SELECT WHERE short_code = '1LY7VK'
       ├─ 命中 → 写回 Redis + 302 跳转
       └─ miss → 404（不存在）或 410 Gone（已过期）
```

## 四、跳转语义：301 vs 302

| 维度 | 301 永久重定向 | 302 临时重定向 | 结论 |
|------|-------------|-------------|------|
| 浏览器缓存 | 缓存，后续直接跳转（不经服务器）| 不缓存，每次经过服务器 | 302 可统计 |
| 访问统计 | ❌ 无法统计（绕过服务器）| ✅ 可统计 PV/UV/设备/来源 | 302 胜 |
| 目标可修改 | ❌ 已缓存用户仍跳旧地址 | ✅ 随时可修改目标地址 | 302 胜 |
| 性能 | 更高（少一次服务器请求）| 略低（每次经服务器）| 301 胜 |

**结论**：通常选 302。短链的核心价值在于可追踪（转化统计）和可管理（随时修改/禁用）。只有内部跳转且目标永不变的场景才考虑 301 提升性能。

## 五、热点短链与性能优化

**场景**：营销活动短链可能被百万次/秒访问，Redis 单节点可能成为热 key 瓶颈。

**三层优化策略**：

| 层级 | 方案 | 效果 | 代价 |
|------|------|------|------|
| 本地缓存（L1）| Guava/Caffeine 缓存热门 code → URL 映射 | 微秒级，减少 Redis 查询 | 内存占用，更新延迟 |
| Redis 分片 | hash tag 确保同 key 落同节点 | 避免热 key 打垮单节点 | 配置复杂度 |
| 301 降级 | 超热链接临时切换 301，峰值过后切回 | 彻底卸载服务器压力 | 失去统计能力 |

**本地缓存实现**：

```java
// Caffeine 本地缓存（L1）：最大 1 万条，5 分钟过期
Cache<String, String> localCache = Caffeine.newBuilder()
    .maximumSize(10_000)
    .expireAfterWrite(5, TimeUnit.MINUTES)
    .build();

public String resolve(String shortCode) {
    return localCache.get(shortCode, code -> {
        String url = redisTemplate.opsForValue().get("short:" + code);
        if (url == null) url = db.findByShortCode(code);
        return url;
    });
}
```

## 六、防滥用与过期机制

**防滥用措施**：

| 手段 | 实现 | 结论 |
|------|------|------|
| 目标 URL 黑名单 | 对接第三方恶意 URL 检测 API | 必选，拦截钓鱼链接 |
| 创建频率限制 | Redis ZSet 滑动窗口（同 IP/用户限速）| 必选，防接口刷量 |
| 接口鉴权 | API Key 或登录态，匿名不能创建 | 推荐，追溯恶意用户 |
| 短码随机性 | Base62 自带随机性，非自增 ID | 内置，防穷举扫描 |

**过期机制**：

```java
// 创建短链时设置 Redis TTL（与 expire_at 一致）
if (expireAt != null) {
    long ttl = Duration.between(LocalDateTime.now(), expireAt).getSeconds();
    redisTemplate.opsForValue().set("short:" + code, longUrl, ttl, TimeUnit.SECONDS);
}

// 访问时校验 MySQL 中的 expire_at
ShortUrl url = db.findByShortCode(code);
if (url.getExpireAt() != null && url.getExpireAt().isBefore(LocalDateTime.now())) {
    return ResponseEntity.status(410).build();  // 410 Gone
}
```

**同一长链生成相同短链（防重）**：
- 需要时：MySQL `long_url` 加唯一索引 → 先查后插 → 分布式锁（MD5(long_url) 为 key）防并发重复创建
- 不需要时（允许同一长链生成多个短链）：跳过去重逻辑，每次直接生成新短码

## 七、选型决策树

```
需要访问统计（PV/UV/来源）？
  → Yes → 302（必选）
  → No → 内部跳转且目标永不变 → 301（减少服务器压力）

短码生成策略？
  → 安全性要求：MurmurHash + Base62（难以穷举）
  → 允许同链重复：每次生成新短码（无去重逻辑）
  → 同链唯一：MySQL 唯一索引 + 分布式锁防并发

热点优化？
  → QPS > 10 万/s：本地 Caffeine + Redis 双级缓存
  → 营销活动极热：临时 301 降级（峰值过后切回 302）
```

## 八、面试高频追问

**问**：短链码冲突怎么处理？
**答**：冲突时在原 URL 后追加盐值（如 `_1`、`_2`）重新计算 MurmurHash + Base62，最多重试 3 次。DB 设 short_code 唯一索引作为最终兜底（`INSERT IGNORE` + 查询已有记录）。6 位 Base62 的 568 亿种组合下，1 亿个短链的冲突概率约 0.17%，实际碰撞非常低频。

**问**：为什么不直接用雪花算法生成短链？
**答**：雪花算法生成的 ID 是 64 位整数，Base62 编码后约 11 位，太长，失去了"短链"的意义（通常要求 6-8 位）。而且雪花 ID 自增有序，虽不容易穷举，但攻击者可以推测出生成规律。MurmurHash 输出分布更均匀、无规律。

**问**：如何防止短链被用于钓鱼攻击？
**答**：三层防护：①创建时调第三方恶意 URL 检测 API（如 Google Safe Browsing），拦截已知恶意域名；②接口鉴权（API Key 或登录态），匿名创建需要通过额外验证；③用户绑定，每个短链记录创建者 ID，发现问题可追溯并批量删除同用户的短链。检测是滞后的，关键是能追溯和快速响应。

**问**：Redis 缓存和 MySQL 如何保持一致？
**答**：短链是"写少读多"场景：创建时同时写 Redis 和 MySQL；修改/删除时先删 Redis（Cache-Aside），再更新 MySQL。因为短链内容基本不变（只有目标 URL 可能修改），不一致窗口短，可以接受短暂的旧值被访问（旧目标 URL 跳转）。过期时间让 Redis 自动淘汰旧缓存，无需额外一致性协议。

## 九、与其他概念的关系

- 访问热点防护使用 [[概念-BitMap]]：可用 BF 提前判断 short_code 是否存在，减少不存在 key 的 DB 穿透查询
- 存储和缓存查询依赖 [[概念-Redis]]：String 存 code→url 映射，ZSet 做频率限制滑动窗口
- 同一长链防重属于 [[概念-幂等设计]]：分布式锁 + DB 唯一索引是幂等创建的标准模式
