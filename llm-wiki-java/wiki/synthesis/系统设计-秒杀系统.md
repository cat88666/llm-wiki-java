# 系统设计-秒杀系统

> 秒杀的核心矛盾：**瞬时极高并发 vs 有限商品库存**。解决策略是多层漏斗过滤（前端→网关→缓存→消息队列→数据库），逐层削减并发量，让压力永远不到达最脆弱的数据库层。

## 问题驱动

**面试高频问题组合**：
- 如何防止超卖？Redis 和数据库各自的方案？
- Redis 预扣减后数据库库存如何保证一致性？
- 热点行更新的瓶颈在哪？如何解决？
- 除了分布式锁，还有哪些防超卖方案？

## 整体架构

```
用户请求
  ↓ [前端层] 按钮置灰 + 请求限速（同一用户）
  ↓ [网关层] 令牌桶/漏桶限流，超出直接拒绝
  ↓ [应用层] Token 防重（唯一令牌，防刷单）
  ↓ [缓存层] Redis 原子扣减库存（Lua 脚本）
  ↓ [消息层] 扣减成功 → 发送 MQ 消息（削峰）
  ↓ [数据库层] Consumer 消费消息 → 异步扣减 DB 库存 → 创建订单
```

## 核心机制

### Redis 库存预扣减（防超卖第一道）

```lua
-- Lua 脚本保证原子性（见 [[机制-Redis分布式锁]]）
local stock = redis.call('get', KEYS[1])
if tonumber(stock) <= 0 then
    return 0  -- 库存不足，直接拒绝
end
redis.call('decr', KEYS[1])
return 1  -- 扣减成功
```

**为何用 Lua 脚本**：Redis 单线程，Lua 脚本整体原子执行，避免 GET-判断-DECR 之间的竞争条件。

### 数据库层防超卖（兜底第二道）

```sql
-- 乐观锁：版本号/剩余库存作为条件
UPDATE goods SET stock = stock - 1
WHERE id = #{id} AND stock > 0;
-- 影响行数为 0 → 超卖被拒绝
```

- 不用 `stock - 1 = #{old_stock}` 版本号方式，用 `stock > 0` 条件更简洁且允许多次并发更新
- 比悲观锁（`SELECT FOR UPDATE`）并发度更高，失败时回滚重试

### 热点行问题

热点行（单行频繁更新）瓶颈：InnoDB **行锁**串行化，高并发下大量线程等待同一行锁，TPS 极低。

**库存拆分方案**：
```
原来：goods(id=1, stock=1000)  ← 单行热点

拆分后：
goods_stock(goods_id=1, bucket=0, stock=100)
goods_stock(goods_id=1, bucket=1, stock=100)
...
goods_stock(goods_id=1, bucket=9, stock=100)

扣减时：随机选一个 bucket → 并发度提升 10 倍
```

### MQ 异步削峰（见 [[机制-消息队列可靠性]]）

Redis 扣减成功后不立即操作 DB，而是写入 MQ：
- 好处：DB 写压力从瞬时峰值变为均匀的消费速率
- 注意：消费者必须做幂等（Redis 扣减成功 + MQ 消费各一次，DB 只写一次）

### 幂等设计（全链路防重）

| 环节 | 幂等方案 |
|------|---------|
| 用户重复点击 | 前端按钮置灰 + Token 令牌（每次秒杀发放唯一 token，使用后失效）|
| 网关层重复请求 | 令牌桶限流 + 用户维度频次控制 |
| MQ 消费重复 | 消息 ID + 数据库唯一键（一锁二判三更新）|
| DB 操作重复 | 订单表唯一索引（user_id + goods_id + activity_id）|

## 关键权衡

1. **Redis 预扣减 vs 直接操作 DB**：Redis 单机 QPS 10万+，DB 热点行 QPS 百级；必须在 Redis 层拦截 99%+ 请求
2. **MQ 异步 vs 同步写 DB**：异步提高吞吐但引入短暂不一致（扣 Redis 成功但订单尚未创建）；需要定时对账
3. **Lua 原子操作 vs 分布式锁**：Lua 脚本无锁，性能更好；分布式锁（SETNX）适合需要执行多步操作的场景
4. **库存拆分 vs 队列串行化**：库存拆分并发度高但实现复杂（合并查询总量、防止某 bucket 耗尽）；串行队列实现简单但吞吐低

## 工业实践补充：大厂如何在数据库层抗热点

> 教科书更常讲 `Redis 预扣 + MQ 异步落盘`，大厂还会直接在数据库层解决热点行竞争，减少 Redis 与 DB 不一致问题。

### 为什么不全靠 Redis

| 教科书方案 | 大厂实际痛点 |
|-----------|------------|
| Redis 预扣 + MQ 落盘 | Redis 宕机 / 网络抖动时要处理一致性补偿 |
| 纯 Redis 扣减 | 事务模型弱，强一致成本高 |
| 数据库乐观锁 | 热点行并发上来后 CPU 和锁等待迅速恶化 |

结论：**MySQL 最可靠，但热点行必须专项优化。**

### 阿里方案：Inventory Hint

核心是在热点 `UPDATE` 上加 Hint，让 MySQL 内核对同一热点行请求做分组优化：

```sql
UPDATE /*+ COMMIT_ON_SUCCESS ROLLBACK_ON_FAIL TARGET_AFFECT_ROW(1) */
  inventory SET quantity = quantity - 1
WHERE sku_id = ? AND quantity > 0;
```

三类收益：
- **减少行锁等待**：Leader 获取锁，同组 Follower 复用结果，不再逐个排队
- **减少索引遍历**：Leader 定位热点行后写入 Row Cache，Follower 直接复用
- **减少提交次数**：一组请求集中组提交，而不是每条事务各提交一次

适合场景：
- 强一致库存
- 已有自研数据库能力或云厂商提供内核增强

### 小红书方案：合并秒杀

核心思想是 **Leader-Follower 合并提交**：

```
Leader 获取热点行
  ↓
收集一小段时间窗口内的 Follower 请求
  ↓
在内存 / 全局缓存中完成多次扣减
  ↓
一次写回 InnoDB
  ↓
统一唤醒 Follower 返回结果
```

这个方案本质上把“很多次热点 update”压缩成“少量批次提交”，显著降低行锁竞争和事务提交开销。

### 秒杀方案选型

| 场景 | 更合适的方案 |
|------|-------------|
| 并发较低、实现成本优先 | 普通 MySQL + 乐观锁 |
| 中高并发、可接受短暂不一致 | Redis 预扣 + MQ 异步 |
| 电商强一致库存、数据库能力强 | Inventory Hint / 合并秒杀 |
| 超高并发大促 | 分桶库存 + 近端缓存 + SQL 合并组合方案 |

## 跨层依赖分析

- **L3 并发**：[[机制-线程池]] 控制消费者线程数量，防止 DB 被打垮
- **L5 存储**：[[机制-InnoDB锁]] 中行锁是热点行瓶颈的根因；[[概念-缓存三大问题]] 中缓存击穿问题在秒杀场景的特化
- **L5 Redis**：[[机制-Redis分布式锁]] 的 Lua 原子脚本是核心工具；[[概念-Redis数据类型与底层结构]] 的 String DECR 操作是预扣减的实现
- **L7 MQ**：[[机制-消息队列可靠性]] 的消费者幂等方案复用于订单创建

## 面试回答要点

1. **先说漏斗模型**（前端→网关→Redis→MQ→DB），体现系统思维
2. **再说 Redis Lua 原子扣减**，解释为何不用分布式锁
3. **提出 MQ 异步削峰**，说明与数据库一致性的处理（对账）
4. **热点行问题主动提出**，展示深度（库存拆分方案）
5. **幂等性**是加分项，每层都要有幂等保障
