## 二、项目深挖 A — 支付中台 & Seata TCC（15 min）

---

🧑‍💼 **面试官：** 简历上写「Seata TCC 解决跨行转账事务一致性，异常链路零资金损失」，这句话说起来很漂亮。我想听你讲一下你具体做了什么，遇到了什么难点。

👨‍💻 **参考回答要点：**
1. **为什么选 TCC 而不是 Saga 或 AT 模式？**
   > AT 模式依赖 undo log，对聚合支付这类操作第三方渠道的接口无法回滚；Saga 代码侵入大、补偿顺序难以保证；TCC 把"预留资源"和"真正扣款"分开，try 阶段只冻结不扣，confirm/cancel 确保原子性，与支付场景最贴合。

2. **try 阶段怎么设计的？**
   > 在账户表增加 `frozen_amount` 字段，try 阶段从 `available_balance` 扣减并加到 `frozen_amount`；confirm 阶段将 `frozen_amount` 清零并计入出账；cancel 阶段将 `frozen_amount` 还回 `available_balance`。

3. **幂等性怎么保证？**
   > Seata 的 xid + branchId 本身是唯一标识，但我们额外在业务表加了 `tx_id` 唯一索引，confirm/cancel 前先查，防止 Seata 框架重试时二次执行补偿逻辑导致超扣。

4. **悬挂（Hanging）和空回滚怎么处理的？**
   > 悬挂：try 未到达，cancel 先来。我们在事务记录表里插入"cancel 标记"，后续 try 到来发现标记直接返回失败不执行。空回滚：try 未执行 cancel 被调用，cancel 查不到冻结记录直接返回成功（不能抛异常，否则 Seata 会一直重试）。

📝 **考察点**：是否理解 TCC 三种模式的本质区别；幂等和悬挂是 TCC 的经典陷阱，必须答出。

---

🧑‍💼 **面试官：** 你们的 try 阶段，`available_balance` 和 `frozen_amount` 的更新是一条 SQL 还是两条？为什么？

👨‍💻 **参考回答：**
> 一条 SQL：`UPDATE account SET available_balance = available_balance - ?, frozen_amount = frozen_amount + ? WHERE id = ? AND available_balance >= ?`。原子更新 + 余额校验在同一行，避免两条 SQL 之间的并发窗口；同时 `WHERE available_balance >= ?` 是乐观锁检查，失败直接返回，不需要显式加行锁。

📝 **考察点**：候选人是否理解 SQL 原子性的边界；是否能用最小代价替代悲观锁。

---

🧑‍💼 **面试官：** 这条 SQL 的 `available_balance >= ?` 条件，和你们业务代码里的余额校验有什么关系？不会查了一次余额又校验一次吗？

👨‍💻 **参考回答：**
> 业务层的校验是"快速失败"，在读阶段就能拦截大部分余额不足的请求，避免走到数据库层。但业务层读和 SQL 写之间存在并发窗口（其他线程可能同时扣减），所以 SQL 层的 `WHERE available_balance >= ?` 是"终极兜底"。两层校验分工不同：业务层减压，SQL 层保安全。

📝 **考察点**：是否理解并发场景下"读写不原子"的问题；是否明白双重校验的必要性而非冗余。

---

🧑‍💼 **面试官：** 如果 TCC 的 confirm 阶段失败了怎么办？比如 confirm 执行到一半服务挂了。

👨‍💻 **参考回答：**
> Seata TC（事务协调器）会无限重试 confirm，直到成功或人工介入。所以 confirm 方法必须满足两个条件：
> 1. **幂等**：即使调用多次，结果一样。我们用 `tx_id` 唯一索引保证。
> 2. **最终一定能成功**：confirm 不能包含可能永久失败的逻辑（比如调用第三方接口），否则会陷入死循环。如果涉及第三方，try 阶段就已经预扣了第三方资源。
>
> 如果确实 confirm 永久失败（极端情况），触发监控告警，人工介入处理。我们有一个对账补偿系统，定时扫描"try 成功但 confirm/cancel 未完成"的事务，超时告警。

📝 **考察点**：confirm 重试 + 幂等是 TCC 的核心保障，候选人是否理解 confirm 不能放"不确定性操作"。

---

🧑‍💼 **面试官：** Sharding-JDBC 分库分表，你们的分片键怎么选的？跨分片事务怎么处理？

👨‍💻 **参考回答要点：**
1. **分片键选 `merchant_id`**，因为账本查询、流水查询 90% 以上都是按商户维度，分片后单分片数据量均衡且查询不跨分片。
2. **跨分片事务**：Seata AT 支持 Sharding-JDBC 事务集成，但我们核心转账是 TCC，所以 xid 是手动传播的，Seata 协调器负责跨分片 branch 的 confirm/cancel。
3. **跨分片聚合查询**：比如"今日全平台交易总额"，走定时任务预聚合到汇总表，避免全分片扫描。

📝 **考察点**：分片键选择的业务依据；是否理解 Sharding-JDBC + Seata 的集成方式。

---

🧑‍💼 **面试官：** `merchant_id` 做分片键，如果某个大商户交易量特别大，导致数据倾斜怎么办？

👨‍💻 **参考回答：**
> 确实存在这个风险。我们的处理：
> 1. **前期评估**：上线前统计商户交易分布，确认前 10 商户的占比不超过总量 30%，可以接受。
> 2. **热点商户独立分片**：如果某商户增长到占 20% 以上流水，为它分配独立库，路由规则里硬编码"商户 A → 独立库"。
> 3. **二级分片**：极端情况下对热点商户内部按 `order_id` 再做一层 hash 分表，但这会增加聚合查询复杂度。
> 实际运营中没有走到第 3 步，第 2 步足够。

📝 **考察点**：是否有数据倾斜的实战意识；解法是否务实。

---

🧑‍💼 **面试官：** 「盒盾」安全系统的全链路加密，接口的加解密方案是什么？

👨‍💻 **参考回答：**
> 采用"信封加密"方案：
> 1. **请求方向**：客户端生成随机 AES 密钥 → 用服务端 RSA 公钥加密 AES 密钥（得到"信封"）→ 用 AES 密钥加密请求体 → 将信封 + 密文一起发送。
> 2. **服务端方向**：用 RSA 私钥解密信封得到 AES 密钥 → 用 AES 密钥解密请求体。
> 3. **响应方向**：服务端用同一个 AES 密钥加密响应体返回（或客户端下一次请求生成新密钥）。
>
> 为什么不直接用 RSA 加密全部数据？因为 RSA 只能加密比密钥短的数据（2048 位密钥最多 245 字节），且性能差（比 AES 慢 100-1000 倍）。AES 负责批量加密，RSA 只保护 AES 密钥。
>
> 敏感数据脱敏用注解实现：`@Desensitize(type = PHONE)` 在序列化时自动将手机号中间 4 位替换为 `****`，日志打印也会生效。

📝 **考察点**：是否理解 RSA + AES 混合加密的必要性（性能 + 长度限制）；脱敏方案是否可维护。

## 三、项目深挖 B — JVM 0 FGC 调优（12 min）

---

🧑‍💼 **面试官：** 你说把游戏平台的 JVM 从「频繁 Full GC」调到「0 Full GC」，能详细说说你的调优过程吗？

👨‍💻 **参考回答结构（STAR 法）：**

**Situation：**  
> 游戏逻辑服每天高峰期 Full GC 3-5 次，每次 STW 1-3 秒，直接导致玩家掉线投诉，告警不断。

**Task：**  
> 目标：业务高峰期不允许任何 Full GC；JVM 参数要支持横向扩展（容器化部署）。

**Action（关键步骤）：**

1. **MAT 分析 Heap Dump，定位大对象**  
   > 发现游戏房间对象（RoomContext）每局创建一个，峰值在 Old Gen 堆积 2GB+，触发 CMS 阶段的 concurrent mode failure，退化为 Serial GC。

2. **对象池改造**  
   > 用 Apache Commons Pool2 实现 `RoomContext` 池化：房间创建时 borrow，结算完成后 return + reset，不再频繁 new/GC。峰值 Old Gen 使用率从 85% 降到 25%。

3. **切换 G1 并调参**  
   ```
   -XX:+UseG1GC
   -Xms4g -Xmx4g        # 堆固定，防止动态扩展触发 FGC
   -XX:MaxGCPauseMillis=50
   -XX:G1HeapRegionSize=16m
   -XX:G1ReservePercent=20
   -XX:InitiatingHeapOccupancyPercent=35  # 提前触发 Mixed GC
   ```

4. **String 常量池控制**  
   > 游戏道具名、技能描述 intern 了大量 String，永久代（Metaspace）也在膨胀。设 `-XX:MaxMetaspaceSize=512m` 并修复业务代码，减少不必要的 intern。

**Result：**  
> 上线两周观察：0 Full GC，Minor GC 频率每 2 分钟一次，每次 <30ms，GC 抖动彻底消失。

📝 **考察点**：必须能说出 MAT 分析 → 找到根因 → 对象池 → G1 调参 的完整链路；只说"换 G1"不够。

---

🧑‍💼 **面试官：** G1 的 `InitiatingHeapOccupancyPercent` 调低有什么代价？

👨‍💻 **参考回答：**
> IHOP 调低（默认 45%，我们用 35%）意味着更早触发 Concurrent Marking 和 Mixed GC。代价是 Mixed GC 频率增加，CPU 开销稍高；收益是 Old Gen 不会堆得太满，避免 Evacuation Failure 触发 Full GC。对于我们的场景——GC 停顿比 CPU 多用几个百分点更重要——这个权衡是值得的。

📝 **考察点**：是否真正理解 G1 的触发机制，而不是背参数。

---

🧑‍💼 **面试官：** 你对象池化之后，池子的大小怎么定的？如果峰值超过池子容量怎么办？

👨‍💻 **参考回答：**
> 池大小的确定：
> 1. **基线**：统计业务高峰期同时在线房间数，取 P99 值（比如 200 个房间同时在线），池大小设为 250（留 25% 余量）。
> 2. **动态扩展**：Commons Pool2 支持 `maxTotal` + `maxIdle` + `minIdle` 三级配置。`minIdle = 50`（保底），`maxTotal = 300`（硬上限），`maxIdle = 200`（常态回收多余对象）。
> 3. **超过 maxTotal**：配置 `BlockWhenExhausted = true` + `MaxWaitMillis = 3000`，超时则拒绝创建房间并返回"服务器繁忙"。比起直接 new 对象导致后续 Full GC 全服受影响，拒绝单个请求更合理。
> 4. **监控**：暴露 `numActive` / `numIdle` 到 Prometheus，配合 Grafana 告警。

📝 **考察点**：对象池参数不是拍脑袋的，要有数据支撑 + 兜底策略。

---

🧑‍💼 **面试官：** 你们结算事务 P99 从 800ms 降到 100ms，用了 RocksDB。RocksDB 在这里承担的是什么角色？

👨‍💻 **参考回答：**
> 结算流程：玩家下线/房间结束 → 批量结算写 DB。原来是同步写 MySQL，P99 高是因为高峰期大量请求竞争行锁。  
> 改造方案：  
> 1. 结算事件先写入本地 RocksDB（LSM 写入极快，几 μs 级）；  
> 2. 异步线程从 RocksDB 读取并批量提交到 MySQL（pipeline 合并降低行锁竞争）；  
> 3. RocksDB 作为"写前日志 + 兜底存储"，即使进程 crash，重启后 replay 保证不丢结算数据。  
> 这样对用户感知的写延迟从等 MySQL 行锁变成写本地 RocksDB，P99 自然骤降。

📝 **考察点**：RocksDB 不是万能的，候选人需要说清楚它解决的是"写缓冲 + 可靠性兜底"，不是替代 MySQL。

---

🧑‍💼 **面试官：** RocksDB 的异步线程挂了，或者合批写 MySQL 失败了，怎么办？

👨‍💻 **参考回答：**
> 1. **异步线程挂了**：我们用 `ScheduledExecutorService` + 守护线程，线程池的 `afterExecute` 里捕获异常并重新提交任务，同时告警。RocksDB 的数据不会丢（持久化在磁盘）。
> 2. **MySQL 写失败**：每条结算记录在 RocksDB 里有状态标记（pending/committed/failed）。写 MySQL 失败标记 failed，下一轮重试。重试超过 N 次后告警并跳过，人工介入。
> 3. **数据最终校验**：每日跑对账任务，比对 RocksDB 里的结算记录和 MySQL 里的到账记录，不一致则报警。

📝 **考察点**：异步架构必须考虑失败场景。能说出"告警 + 重试 + 对账"三层才完整。

---

🧑‍💼 **面试官：** 你为什么用 RocksDB 而不是直接用 MQ（比如 RocketMQ）来做异步？

👨‍💻 **参考回答：**
> 两个原因：
> 1. **延迟**：RocksDB 是本地存储，写入延迟 μs 级；MQ 是网络调用，最快也是 ms 级。游戏场景对结算响应时间敏感，不能多一跳网络。
> 2. **部署简单**：逻辑服是容器化部署，每个实例挂载独立的 RocksDB 目录。不需要额外维护 MQ 集群。
> 
> 但 MQ 也不是完全不用——我们在 RocksDB 之上，对某些需要跨服务消费的结算事件（如排行榜更新），会二次投递到 RocketMQ。架构是"本地快写 RocksDB → 批量刷 MySQL → 按需投递 MQ"。

📝 **考察点**：技术选型的理由是否明确；是否能清晰阐述本地存储 vs 远程消息队列的取舍。

## 四、项目深挖 C — CoinsOTC 区块链资金安全（10 min）

---

🧑‍💼 **面试官：** CoinsOTC 里你们的冷热钱包是怎么设计的？

👨‍💻 **参考回答：**
> **热钱包**：在线私钥，用于日常小额充提；私钥用 AES 加密存 DB，每次使用时解密，操作完立即清除内存（`Arrays.fill(keyBytes, (byte) 0)`）。  
> **冷钱包**：离线设备（air-gapped），只存储大额资产；每日或触发阈值时，由人工审批流将热钱包余额批量划转到冷钱包地址。  
> **划转逻辑**：热→热转账用 bitcoinj 构建原始交易，广播到链上；热→冷转账生成未签名 PSBT，导出到离线设备签名后再导入广播。

---

🧑‍💼 **面试官：** 热钱包私钥用 AES 加密，那 AES 密钥存在哪？如果 AES 密钥也存 DB 不就等于没加密？

👨‍💻 **参考回答：**
> AES 密钥不在数据库。我们分三层管理：
> 1. **AES 数据密钥（DEK）**：加密私钥用的密钥，每个钱包一个，加密后存 DB。
> 2. **AES 密钥加密密钥（KEK）**：用来加密 DEK 的密钥，存在环境变量中，由运维人员在部署时注入，不写入代码或配置文件。
> 3. **生产环境最终方案**：KEK 由 KMS（Key Management Service，如 AWS KMS/阿里云 KMS）管理，应用启动时通过 IAM 角色获取，KEK 从不落盘。
>
> 这样即使数据库被拖库，拿到的是加密后的 DEK + 加密后的私钥，没有 KEK 无法解密。

📝 **考察点**：密钥管理是安全架构的核心。能说出 DEK/KEK 分层 + KMS 是加分项。

---

🧑‍💼 **面试官：** 你用 Redisson 分布式锁保障「资金原子性操作」，如果 Redis 主节点宕机，分布式锁失效怎么办？

👨‍💻 **参考回答：**
> 这是 Redisson 标准锁的经典问题：主从复制是异步的，主节点宕机时锁信息可能未同步到从节点。  
> 我们的处理：  
> 1. **RedLock（多节点）**：对资金操作使用 RedLock，向 3 个独立 Redis 节点加锁，majority 成功才算成功，单节点宕机不影响锁的正确性。  
> 2. **DB 层幂等兜底**：链上转账有 txHash，我们在 DB 建唯一索引 `(txHash, direction)`，即使锁失效导致二次提交，DB 层唯一约束也能拦截重复入账。  
> 两层保障，锁是第一道，DB 是最终防线。

📝 **考察点**：只说 Redisson 不够，要能说出 RedLock 原理和局限，以及为什么业务层还要有幂等设计。

---

🧑‍💼 **面试官：** Martin Kleppmann 写过一篇文章批评 RedLock，你知道他的观点吗？你怎么看？

👨‍💻 **参考回答：**
> Kleppmann 的核心观点是：RedLock 依赖系统时钟的准确性。如果某个 Redis 节点发生时钟跳变（NTP 校准或 VM 暂停），锁的 TTL 可能提前过期，另一个客户端误以为锁已释放而加锁成功，造成两个客户端同时持锁。
>
> 他提出的替代方案是 **Fencing Token**：锁服务每次发锁附带一个单调递增的 token，存储层只接受比上次更大的 token 的写入。这样即使两个客户端同时持锁，存储层也能拒绝过期的写入。
>
> 我怎么看：在我们的场景里，RedLock + DB 唯一索引 其实就是变相的 Fencing——DB 唯一约束扮演了最终裁决者的角色。不能只靠锁，底层存储必须有能力拒绝重复操作。

📝 **考察点**：P7/P8 要有对技术方案的批判性思考能力。能说出 Kleppmann 的观点和 Fencing Token 是显著加分。

🧑‍💼 **面试官：** 充提款的异步对账是怎么做的？

👨‍💻 **参考回答：**
> 充值：监听区块链事件（bitcoinj 的 `WalletEventListener` / web3j 的 `Filter`），收到充值确认后异步入账。  
> 对账流程：  
> 1. 每 5 分钟扫描"待确认"充值订单；  
> 2. 用 txHash 查链上确认数，满足阈值（BTC ≥ 6 confirmations）则修改订单状态 → 增加用户余额；  
> 3. 差异报警：如果链上有入账但系统无记录，触发人工复核告警。  
> 提款同理：广播后记录 txHash，轮询确认数，超时未确认自动重广播（fee bump）。

🧑‍💼 **面试官：** BTC 为什么要等 6 个确认？以太坊呢？

👨‍💻 **参考回答：**
> BTC 6 个确认约 60 分钟。比特币用 PoW 共识，存在分叉可能——如果攻击者控制超过 50% 算力可以重写最近几个区块。6 个确认后被逆转的概率极低（< 0.1%），这是行业通用安全标准。大额交易可以更高（12 或更多）。
>
> 以太坊 PoS 合并（Merge）后，用 finality 机制——32 个 slot 组成一个 epoch（~6.4 分钟），两个 epoch 后 finalized（~12.8 分钟）。Finalized 后不可逆。我们一般等 64 个 block confirmation 或等 finalized 状态。

📝 **考察点**：区块链岗位加分题，不强求精确，但如果说"因为网上都说 6 个"就减分了。

## 五、项目深挖 D — Netty 自研网关 & 游戏架构（8 min）

🧑‍💼 **面试官：** 你在腾娱自研了一个 Netty 网关，为什么不直接用 Spring Cloud Gateway？

👨‍💻 **参考回答：**
> 游戏场景和 HTTP 微服务场景有本质区别：
> 1. **协议**：游戏客户端用的是自定义二进制协议（TCP 长连接），不是 HTTP。Spring Cloud Gateway 是 HTTP 网关（基于 WebFlux），不能直接处理自定义协议。
> 2. **连接模型**：游戏需要维持百万级长连接，每个连接有状态（玩家登录态、房间归属）。HTTP 网关是无状态的。
> 3. **消息格式**：我们用的是类 TTLV 协议（Type-Tag-Length-Value），需要自定义编解码器，Netty 的 Pipeline 机制天然适合。
> 4. **性能**：直接操作 ByteBuf，零拷贝 + 内存池化，比 HTTP 序列化/反序列化少很多开销。

📝 **考察点**：是否理解游戏长连接和 HTTP 短连接的本质差异；技术选型是否有业务依据。

---

🧑‍💼 **面试官：** 你们自研网关的"批量解码"是什么意思？为什么能提升吞吐量 200%？

👨‍💻 **参考回答：**
> 普通解码器是一次 channelRead 只处理一个消息。我们的优化：
> 1. **累积解码**：继承 `ByteToMessageDecoder`，一次 `decode()` 调用里循环解码 ByteBuf 中所有完整消息（直到 `readableBytes < headerLength`），输出一个 `List<GameMessage>`。
> 2. **批量分发**：`channelRead0` 收到 List 后，不是逐条处理，而是按消息类型分组后批量提交给业务线程池。
> 3. **减少线程切换**：原来 10 条消息 = 10 次 EventLoop → 业务线程池的 task 提交。现在 1 次提交 10 条，线程上下文切换减少 10 倍。
>
> 200% 的提升主要来自批量分发减少的线程切换和 EventLoop 的 `task queue` 竞争。

📝 **考察点**：Netty 性能优化不是只调参数，而是要理解事件循环和线程模型。

---

🧑‍💼 **面试官：** 你们的"优雅下线"具体怎么实现的？

👨‍💻 **参考回答：**
> 分三步：
> 1. **注册中心摘除**：先从 Nacos 注销该网关实例，Gateway 路由层不再将新连接分配给这台机器。
> 2. **拒绝新连接、保留旧连接**：在 Netty 的 `ServerBootstrap` 上调用 `channel().close()` 停止 accept 新连接，但已建立的 Channel 继续保持。
> 3. **等待旧连接排空**：设定一个超时时间（如 30 秒），等待所有 Channel 上的消息处理完毕。超时后强制关闭剩余连接（发一个"服务器维护"消息给客户端，客户端自动重连到其他节点）。
>
> 这个流程通过 JVM shutdown hook 触发（`Runtime.getRuntime().addShutdownHook`），K8s 的 `preStop` 也会先调用这个逻辑。

📝 **考察点**：优雅停机是生产级系统的标配，面试官看你是否有完整的生产意识。

---

🧑‍💼 **面试官：** 你们游戏中用户级线程隔离是怎么做的？

👨‍💻 **参考回答：**
> 每个玩家的消息处理绑定到一个固定的 `SingleThreadExecutor`，通过 `userId % N` 映射到线程数组中的一个。这样同一玩家的所有请求串行执行，不需要加锁；不同玩家并行执行，充分利用多核。
>
> 心跳等系统级消息用独立线程池处理，不占用业务线程，避免大量心跳阻塞业务消息。
>
> 这个模型的关键权衡是：N 的大小。太小（如 4），并发度不够；太大（如 1024），线程切换开销高。我们根据 CPU 核数设为 `Runtime.availableProcessors() * 2`。

📝 **考察点**：用户级隔离是游戏后端的经典模式，是否理解其解决的并发安全问题。

### 6.1 Java 并发（5 min）

🧑‍💼 **面试官：** AQS 的核心设计思想是什么？ReentrantLock 的公平锁和非公平锁在 AQS 层面有什么区别？

👨‍💻 **参考回答：**
> AQS 核心是一个 `volatile int state` + CLH 变种的等待队列。获取资源用 CAS 修改 state；失败则封装成 Node 入队，然后自旋/park。释放资源 CAS 还原 state 并 unpark 队头节点。  
> **公平锁**：`tryAcquire` 先调 `hasQueuedPredecessors()`，如果队列里有更早的等待者就直接入队；  
> **非公平锁**：`tryAcquire` 直接 CAS 抢锁，不看队列，抢到就用，抢不到再入队。  
> 非公平锁吞吐高（少一次队列判断，且刚释放锁的线程可能立即重获）；公平锁不会饥饿但上下文切换多。

📝 **考察点**：不能只说"维护一个等待队列"，要能说出 state 的语义和公/非公平的代码级差异。

---

🧑‍💼 **面试官：** `volatile` 能替代 `synchronized` 吗？什么时候可以，什么时候不行？

👨‍💻 **参考回答：**
> **可以**替代的场景：状态标志位（`boolean running`），只有一个线程写、多个线程读。volatile 保证可见性和有序性，代价比 synchronized 小。  
> **不能**替代：复合操作，如 `count++`（读-改-写三步，volatile 保证每步可见但不保证原子性整体）；需要独占访问某个代码块。这时要用 `AtomicInteger` 或 `synchronized`。

---

🧑‍💼 **面试官：** `synchronized` 和 `ReentrantLock` 你怎么选？

👨‍💻 **参考回答：**
> 默认用 `synchronized`——JVM 会做偏向锁/轻量级锁优化，无竞争时性能极好，且 JVM 保证释放（不怕忘记 unlock）。
> 选 `ReentrantLock` 的场景：
> 1. **需要 tryLock 超时**：避免死等（分布式场景常见）
> 2. **需要公平锁**：synchronized 只有非公平
> 3. **需要多个 Condition**：synchronized 只有一个 wait/notify 队列
> 4. **需要可中断获取**：`lockInterruptibly()`
> 5. **JDK21 虚拟线程**：synchronized 会 pin 载体线程，ReentrantLock 不会，虚拟线程场景应优先用 ReentrantLock

📝 **考察点**：不要只说"功能更强"，要说出具体场景。提到虚拟线程 pinning 问题是加分项。

---

🧑‍💼 **面试官：** 线程池的核心参数怎么设置？你有没有遇到过线程池配置不当导致的生产事故？

👨‍💻 **参考回答：**
> 7 个参数，关键 4 个：corePoolSize、maximumPoolSize、workQueue、rejectedExecutionHandler。
>
> **设置原则**：
> - CPU 密集型：core = CPU核数 + 1（多 1 是留给偶尔的 I/O）
> - I/O 密集型：core = CPU核数 * 2 或 CPU核数 / (1 - 阻塞比例)
> - 但实际中要压测调整，公式只是起点
>
> **踩过的坑**：
> 曾经一个支付回调服务用了 `Executors.newCachedThreadPool()`，高峰期瞬间创建了 2000+ 线程，内存飙升 OOM。改为固定大小 + 有界队列 + CallerRunsPolicy 后解决。教训：**永远不要用 Executors 工厂方法**，必须自己 new ThreadPoolExecutor，明确每个参数。

📝 **考察点**：公式只是基础，能讲出实际踩坑经历才有说服力。

### 6.2 MySQL（4 min）

🧑‍💼 **面试官：** MySQL InnoDB 的 MVCC 解决了什么问题？它为什么不能完全消除幻读？

👨‍💻 **参考回答：**
> MVCC 通过 undo log 版本链 + ReadView 实现"读不阻写"：SELECT 读历史快照，不加锁，写操作只锁当前行，解决了读写互斥问题。  
> **为什么不能完全消除幻读**：  
> - 普通 SELECT（快照读）不会幻读：每次都读同一个 ReadView 的快照；  
> - 当你在同一事务中先做快照读，再做 UPDATE 或加锁 SELECT（当前读），后者会读到新插入的行，这时再快照读可能也看得到了（因为 ReadView 里 trx_id 刷新）。  
> 真正消除幻读需要 Next-Key Lock（Gap Lock + Record Lock），把"将来可能插入的位置"锁住。

📝 **考察点**：能区分快照读和当前读；能说出 Gap Lock 是 InnoDB 消除幻读的实际机制。

🧑‍💼 **面试官：** 你们账本表是分库分表的，分片键是 `merchant_id`。如果要查某个时间段所有商户的交易汇总，你怎么做？

👨‍💻 **参考回答：**

> 1. **汇总宽表**：定时任务（每小时/每天）按商户聚合写入独立的汇总表（不分片），平台级查询走汇总表；  
> 2. **OLAP 分流**：明细数据实时同步到 ClickHouse，大宽表全量分析走 ClickHouse，不压 MySQL；  
> 3. **按时间二级分片**：如果确实需要时间维度查询，考虑复合分片键 `(merchant_id, date_partition)`，但这会增加路由复杂度，需要权衡。

🧑‍💼 **面试官：** 一条慢 SQL，你的排查流程是什么？

👨‍💻 **参考回答：**
> 1. **定位**：开启慢查询日志（`slow_query_log = ON`，`long_query_time = 1`），或从 Skywalking/APM 中找到慢 SQL。
> 2. **分析**：`EXPLAIN` 看执行计划——关注 `type`（ALL 全表扫描最差，ref/range 可接受）、`key`（是否命中索引）、`rows`（扫描行数）、`Extra`（Using filesort / Using temporary 是警告信号）。

### 6.3 Redis（4 min）

🧑‍💼 **面试官：** Redis 为什么快？单线程为什么能扛高并发？

👨‍💻 **参考回答：**
> 5 个原因：
> 1. **纯内存操作**：数据在内存，读写 ns 级
> 2. **单线程避免上下文切换和锁竞争**：命令串行执行，无需加锁
> 3. **IO 多路复用（epoll）**：一个线程监听大量连接，有数据才处理
> 4. **高效数据结构**：SDS、ziplist、skiplist 等都是为性能设计的
> 5. **Redis 6.0+ 多线程 I/O**：网络读写用多线程，命令执行仍然单线程
>
> 单线程能扛高并发的关键是**瓶颈不在 CPU 而在网络 I/O**。Redis 命令执行极快（μs 级），CPU 根本不是瓶颈。多路复用让一个线程高效处理数万连接的网络事件。

🧑‍💼 **面试官：** 缓存穿透、击穿、雪崩分别是什么？你在项目中遇到过哪个？

👨‍💻 **参考回答：**
> | 问题 | 定义 | 解法 |
> |------|------|------|
> | **穿透** | 查询不存在的数据，缓存和 DB 都没有 | 布隆过滤器 / 空值缓存（TTL 短） |
> | **击穿** | 热点 key 过期瞬间，大量请求打到 DB | 互斥锁（setnx 抢锁查 DB 回填）/ 永不过期 + 异步刷新 |
> | **雪崩** | 大量 key 同时过期 / Redis 宕机 | TTL 加随机值打散 / 限流降级 / 多级缓存 |
>
> **实际遇到的**：游戏平台每日 0 点刷新活动配置，所有活动缓存 TTL 统一设为 24h，0 点同时过期导致雪崩。解法：TTL = 24h + random(0, 3600)，再加一层本地 Caffeine 缓存兜底。

📝 **考察点**：三者定义清晰 + 有实战经历比纯背书更有说服力。

🧑‍💼 **面试官：** Redis 和 MySQL 的数据一致性怎么保证？你们用的哪种方案？

👨‍💻 **参考回答：**
> 核心原则是**先更新 DB，再删缓存**（Cache Aside Pattern）。不用"先删缓存再更新 DB"——并发场景下会导致脏数据。
>
> 我们的方案分两层：
> 1. **写路径**：更新 DB 后，删除 Redis 缓存（不是更新缓存，因为更新缓存在并发下可能覆盖新值）。下次读时 miss 再回填。
> 2. **兜底**：Canal 监听 binlog，异步删除缓存（防止写路径删除失败时缓存残留）。
> 3. **强一致场景**（如余额）：不走缓存，直接读 DB。缓存只用于"读多写少、容忍短暂不一致"的场景。
>
> 延迟双删方案（先删缓存→更新DB→sleep→再删缓存）理论上可以，但 sleep 时间难以确定，我们没采用。

📝 **考察点**：Cache Aside 是标准答案；能说出"不是所有数据都适合缓存"是加分。

### 6.4 Spring（3 min）

🧑‍💼 **面试官：** Spring 的 @Transactional 失效的场景有哪些？你踩过哪个？

👨‍💻 **参考回答：**
> 常见失效场景：
> 1. **方法非 public**：Spring AOP 默认只代理 public 方法
> 2. **自调用（this 调用）**：`this.methodB()` 不走代理，事务不生效
> 3. **异常被 catch 吞了**：Spring 看不到异常，不触发回滚
> 4. **rollbackFor 不匹配**：默认只回滚 RuntimeException，checked exception 不回滚
> 5. **多线程**：事务绑定在 ThreadLocal 上，新线程拿不到事务上下文
> 6. **传播机制 REQUIRES_NEW**：内层事务和外层事务独立，内层回滚不影响外层
>
> **踩过的坑**：支付回调里，先更新订单状态（事务 A），再调用一个内部方法发送通知（`this.sendNotification()`），通知方法有自己的 @Transactional(REQUIRES_NEW)。但因为是 `this` 调用，实际上 REQUIRES_NEW 没生效，通知失败时把外层事务也回滚了——订单状态没更新，钱已经扣了。
> 改法：注入自身 `@Lazy private OrderService self;`，用 `self.sendNotification()` 走代理。

📝 **考察点**：能说出 3 个以上场景 + 有踩坑经历。

🧑‍💼 **面试官：** Spring Bean 的循环依赖是怎么解决的？为什么需要三级缓存？

👨‍💻 **参考回答：**
> Spring 用三级缓存解决 setter 注入的循环依赖：
> - **一级缓存**（singletonObjects）：完整 Bean
> - **二级缓存**（earlySingletonObjects）：提前暴露的半成品 Bean（已实例化，未属性注入）
> - **三级缓存**（singletonFactories）：ObjectFactory，调用时决定返回原始对象还是代理对象
>
> 为什么不能只用二级？因为 AOP。如果 A 需要被代理，在循环依赖中 B 拿到的 A 必须是代理后的 A，不能是原始对象。三级缓存的 ObjectFactory 可以在第一次被获取时判断是否需要创建代理，返回正确的对象。
>
> 注意：构造器注入的循环依赖无法自动解决（因为实例化阶段就需要对方），要用 `@Lazy` 延迟加载。SpringBoot 2.6 默认禁止了循环依赖。

📝 **考察点**：三级缓存的核心是解决 AOP 代理对象的问题，不只是"提前暴露"。

### 6.5 分布式（4 min）

🧑‍💼 **面试官：** Nacos 同时支持 AP 和 CP 模式，注册中心应该选哪个？配置中心呢？

👨‍💻 **参考回答：**
> **注册中心选 AP**：服务注册/发现可以容忍短暂的数据不一致（某个节点还没感知到新实例），但不能因为网络分区导致整个发现不可用。宁可读到稍旧的服务列表，也要保证发现服务本身不挂。  
> **配置中心选 CP**：配置（如数据库密码、限流阈值）要求强一致，所有节点读到的必须是同一版本；如果读到的是旧配置可能导致服务行为异常。  
> Nacos 的实现：AP 模式用自研的 Distro 协议；CP 模式用 Raft。Nacos 2.x 默认注册中心 AP，配置中心默认 CP。

🧑‍💼 **面试官：** RocketMQ 和 Kafka 你们都用过，选型依据是什么？

👨‍💻 **参考回答：**
> | 场景 | 选 RocketMQ | 选 Kafka |
> |------|------------|---------|
> | 支付/金融事务消息 | ✅ 事务消息原生支持，精确一次语义 | — |
> | 游戏日志大吞吐 | — | ✅ 吞吐极高，顺序写盘 |
> | 消息延迟队列 | ✅ 定时消息原生支持 | 需自实现 |
> | 消费失败重试/死信队列 | ✅ 内置 | 需业务自己处理 |
>
> 我们支付中台用 RocketMQ（事务消息保证转账通知可靠投递）；游戏日志和行为埋点用 Kafka（吞吐优先）。

🧑‍💼 **面试官：** 如果 RocketMQ 消费者消费速度跟不上生产者，消息堆积了怎么处理？

👨‍💻 **参考回答：**
> 分紧急处理和根因治理两步：
> 
> **紧急处理**：
> 1. **扩消费者**：增加 Consumer 实例数量（但不能超过队列数，否则多余的消费者空闲）
> 2. **如果队列数不够**：临时新建一个 Topic，写一个"转发消费者"把积压消息转到新 Topic（有更多队列），再扩消费者消费新 Topic
> 3. **降级非核心逻辑**：如果消费逻辑中有非核心操作（如发通知邮件），临时跳过，只处理核心业务
> 
> **根因治理**：
> 1. **消费端优化**：检查是否有慢 SQL、远程调用阻塞、不必要的同步操作
> 2. **并行消费**：`consumeMessageBatchMaxSize` 调大，一次拉多条批量处理
> 3. **监控预警**：对消费延迟设告警阈值（如延迟 > 5 分钟就告警），不要等到堆积严重了才发现

📝 **考察点**：是否有处理生产事故的经验；方案是否分"止血"和"治本"两层。

🧑‍💼 **面试官：** 分布式事务除了 TCC，你知道本地消息表方案吗？和 TCC 比有什么优劣？

👨‍💻 **参考回答：**
> **本地消息表**：
> 1. 在发起方 DB 中，业务操作和写消息表在同一个本地事务里完成
> 2. 定时任务扫描消息表，将"待发送"的消息投递到 MQ
> 3. 消费方收到后处理业务，处理成功则回调或标记
> 4. 消费方做幂等（唯一索引）
>
> **对比**：
> | | TCC | 本地消息表 |
> |---|-----|---------|
> | 一致性 | 准实时（秒级） | 最终一致（可能分钟级） |
> | 代码侵入 | 高（需要写 try/confirm/cancel） | 低（只需要消息表 + 定时任务） |
> | 性能 | 高（不依赖轮询） | 定时任务有延迟 |
> | 适用场景 | 资金类（强实时性） | 通知类/异步类（容忍延迟） |
>
> 我们支付转账用 TCC，交易完成后的通知、积分、报表等下游用本地消息表。

## 七、系统设计 — 高并发抢购（Gacha）（10 min）

---

🧑‍💼 **面试官：** 游戏 Gacha 系统，日活 500 万，高峰期 10 万 QPS 的抽奖请求，要求抽奖结果实时返回，且不能超卖（某稀有道具限量 100 个）。你来设计一下。

👨‍💻 **参考设计思路：**

```
Client → API Gateway → Gacha Service → Redis Lua Script → DB 异步落盘
                                    ↓ 限量道具扣减
                                 RocketMQ → 库存核销服务 → MySQL
```

**关键设计点：**

1. **库存预热**：服务启动时将限量道具数量加载进 Redis（`SET item:rare:stock 100`）。

2. **Redis Lua 原子扣减**：
   ```lua
   local stock = redis.call('GET', KEYS[1])
   if tonumber(stock) <= 0 then return -1 end
   return redis.call('DECR', KEYS[1])
   ```
   Lua 脚本在 Redis 单线程执行，天然原子，无超卖。返回 -1 直接告诉用户"已售罄"。

3. **结果异步落盘**：抽奖成功后发 MQ 消息，消费者异步写 MySQL，避免高峰期 DB 成为瓶颈。

4. **概率算法**：在 Gacha Service 内存中维护权重表（不走 DB），随机数命中区间判断结果，无网络 IO，微秒级完成。

5. **限流保护**：Gateway 层对单用户做令牌桶（每秒最多 N 次），防刷；全局 Sentinel 流控保护下游。

6. **对象池**：抽奖请求对象（含随机数生成器、权重表）池化，减少 GC 压力（你们简历就有这个实践）。

📝 **考察点**：Lua 原子扣减是核心；要能说出异步落盘的必要性；对象池和 GC 结合自己简历加分。

---

🧑‍💼 **面试官：** Redis 宕机了，Lua 脚本里已扣减成功但 MQ 消息还没发，怎么处理？

👨‍💻 **参考回答：**
> 这是"本地事务 + 消息"的经典问题。方案：  
> 1. **本地消息表**：抽奖成功后，在同一个 DB 事务里既写抽奖记录，也写一条"待发送 MQ"记录（`pending_mq`）；  
> 2. 定时任务扫描 `pending_mq` 表，将未发送的消息补发到 MQ；  
> 3. 消费者做幂等（`order_id` 唯一索引），确保重复消费不重复发奖。  
> Redis 库存是"快速通道"，DB 才是"最终凭证"。Redis 挂了库存数据会丢失，重启后从 DB 聚合已发放数量重新加载库存即可。

---

🧑‍💼 **面试官：** 10 万 QPS 打到一个 Redis key 上，会不会有热点问题？怎么解决？

👨‍💻 **参考回答：**
> 确实是热点 key 问题。10 万 QPS 单 key 可能打满单个 Redis 节点的 CPU。
>
> 解法——**库存分桶**：
> 1. 把 100 个库存拆成 10 个 key：`item:rare:stock:0` ~ `item:rare:stock:9`，每个 key 10 个库存
> 2. 请求到来时 `userId % 10` 决定扣哪个 key，分散到不同 slot（如果用 Cluster 会分散到不同节点）
> 3. 某个桶扣完了（返回 -1），再尝试下一个桶（round-robin 或随机），直到全部桶都扣完才返回"售罄"
>
> 进阶优化：如果用 Redis Cluster，可以用 hash tag 把不同桶分到不同节点（`{stock}:0` ~ `{stock}:9` 或不加 tag 让它们自然分散）。
>
> 还可以加一层本地 Caffeine 缓存做"售罄标记"：某个桶返回 -1 后，本地标记 10 秒内不再请求该桶，减少无效的 Redis 调用。

📝 **考察点**：热点 key 是高并发场景的核心问题。分桶方案是标准解法，能说出"本地售罄标记"是加分。

---

🧑‍💼 **面试官：** 抽奖结果的概率如何保证公平？如果有用户反馈"抽了 1000 次都没中"怎么验证？

👨‍💻 **参考回答：**
> **概率算法**：权重表 + 区间映射。假设稀有道具概率 1%，普通道具 99%，则 [0, 0.01) 中奖，[0.01, 1.0) 不中奖。用 `ThreadLocalRandom.current().nextDouble()` 生成随机数，落在哪个区间就是什么结果。
>
> **保底机制**（防止"非酋"体验太差）：维护每个用户的累计未中奖次数，达到阈值（如 200 次）必定中奖（软保底）。这个计数器存在 Redis 中。
>
> **公平验证**：
> 1. 每次抽奖记录（用户ID、随机数种子、结果）写入日志
> 2. 运营后台可以回放某个用户的全部抽奖记录，验证每次随机数和概率是否匹配
> 3. 定期统计整体中奖率，确认和配置的概率一致（卡方检验）
>
> 这个设计在日本游戏行业是法规要求（「ガチャ規制」），必须能举证概率公正。

📝 **考察点**：概率公平性在游戏行业是法律问题，不只是技术问题。能说出保底机制和可审计性是加分。

🧑‍💼 **面试官：** 如果这个 Gacha 系统要做全球部署（日本、美国、欧洲），你的架构怎么调整？

👨‍💻 **参考回答：**

## 八、算法 & 编码（5 min）

---

🧑‍💼 **面试官：** 给你一个场景题：实现一个简单的 LRU 缓存，要求 get 和 put 都是 O(1)。说说你的思路和关键数据结构。

👨‍💻 **参考回答：**
> **数据结构**：`HashMap<K, Node<K,V>>` + 双向链表。
> - HashMap 保证 O(1) 查找
> - 双向链表保证 O(1) 移动/删除（维护访问顺序）
> - 最近使用的放链表头部，最久未使用的在尾部
>
> **操作**：
> - `get(key)`：HashMap 查到 → 移到链表头部 → 返回值；查不到 → 返回 -1
> - `put(key, value)`：已存在 → 更新值 + 移到头部；不存在 → 新建 Node + 插入头部 + HashMap 放入。如果超容量 → 删除链表尾部节点 + HashMap 删除对应 key
>
> **关键细节**：用 dummy head 和 dummy tail 简化边界判断。

```java
class LRUCache {
    private Map<Integer, Node> map = new HashMap<>();
    private Node head = new Node(), tail = new Node();
    private int capacity;

    public LRUCache(int capacity) {
        this.capacity = capacity;
        head.next = tail;
        tail.prev = head;
    }

    public int get(int key) {
        Node node = map.get(key);
        if (node == null) return -1;
        moveToHead(node);
        return node.value;
    }

    public void put(int key, int value) {
        Node node = map.get(key);
        if (node != null) {
            node.value = value;
            moveToHead(node);
        } else {
            node = new Node(key, value);
            map.put(key, node);
            addToHead(node);
            if (map.size() > capacity) {
                Node removed = tail.prev;
                removeNode(removed);
                map.remove(removed.key);
            }
        }
    }
}
```

📝 **考察点**：LRU 是高频题，5 分钟内能说清思路 + 写出核心逻辑即可。

---

🧑‍💼 **面试官：** 如果要让这个 LRU 线程安全，你怎么做？

👨‍💻 **参考回答：**
> 三种方案递进：
> 1. **简单粗暴**：`Collections.synchronizedMap` 包装或 get/put 方法加 `synchronized`。缺点是读写互斥，并发差。
> 2. **读写锁**：`ReentrantReadWriteLock`。get 加读锁（允许并发读），put 加写锁。但 get 也会修改链表顺序（moveToHead），所以 get 其实也需要写锁——这时和方案 1 没区别。
> 3. **分段锁 + 近似 LRU**：像 Caffeine 的做法，用 ConcurrentHashMap + Ring Buffer 记录访问事件，异步在单线程里维护淘汰顺序。高并发场景首选 Caffeine 而不是自己实现。

📝 **考察点**：能意识到"get 也修改链表"导致读写锁退化；知道 Caffeine 是生产级方案。

## 附 C：高频追问陷阱题集

> 以下是面试官常用的"深水区"追问。如果第一轮回答太顺利，面试官通常会用这些问题测试深度。

### C.1 分布式事务追问

🧑‍💼 **如果 Seata TC 本身挂了怎么办？**
> TC 本身也是集群部署的（高可用模式用 DB 或 Redis 存储事务记录），单节点挂了其他节点接管。但如果 TC 全挂了，所有全局事务都会超时，try 锁定的资源需要等 TC 恢复后 cancel。这是 Seata 的已知局限——TC 是单点风险。生产环境一定要做 TC 高可用。

🧑‍💼 **你们有没有考虑过去掉 Seata，直接用可靠消息方案？**
> 考虑过。可靠消息（如 RocketMQ 事务消息）可以替代部分 TCC 场景，但它是"最终一致"的，资金场景需要"准实时一致"。如果转账走 MQ，用户可能在几秒内看到"余额还没变"——虽然最终会一致，但体验和客服投诉都不可接受。所以核心支付链路保留 TCC，非核心链路（通知/积分/报表）走 MQ。

### C.2 JVM 追问

🧑‍💼 **你有没有考虑过 ZGC？为什么选 G1 不选 ZGC？**
> 当时（2019-2023）ZGC 还在实验阶段（JDK 11 是 Experimental，JDK 15 才 Production-Ready），我们用的 JDK 11，不敢在生产用。如果现在重新选型（JDK 17+），会考虑 ZGC——它的 STW 控制在 1ms 以内，不需要调 IHOP 这些参数。但 ZGC 的内存占用比 G1 大（着色指针需要额外空间），4G 堆可能不够，要 8G+。

🧑‍💼 **对象池化后，对象的状态重置如果漏了一个字段怎么办？**
> 这是对象池的经典坑。我们的解法：
> 1. `RoomContext` 实现 `Resettable` 接口，`reset()` 方法必须覆盖所有字段
> 2. 写了一个单元测试：通过反射遍历所有字段，检查 `reset()` 后每个字段是否等于默认值
> 3. 每次新增字段时 CI 会跑这个测试，漏了就会挂

### C.3 Redis 追问

🧑‍💼 **Redis 的 bigkey 问题你遇到过吗？怎么处理？**
> 遇到过。一个排行榜 ZSet 有 200 万成员，删除时 `DEL` 导致 Redis 主线程阻塞 2 秒。
> 解法：用 `UNLINK`（Redis 4.0+）异步删除；写入时拆分为多个 key（`rank:0`~`rank:9`），每个 key 20 万成员。
> 监控：定期用 `redis-cli --bigkeys` 扫描或 `MEMORY USAGE key` 检查。

🧑‍💼 **Redis Cluster 的 hash slot 迁移过程中，客户端怎么处理？**
> 迁移期间客户端可能收到 `MOVED`（key 已完全迁移到新节点）或 `ASK`（key 正在迁移中，需要临时问新节点）。
> - `MOVED`：客户端更新本地 slot 路由表，以后直接请求新节点
> - `ASK`：客户端先向新节点发 `ASKING` 命令，再发实际请求（一次性的，不更新路由表）
> Redisson/Lettuce 等客户端库已内置了 MOVED/ASK 的自动处理。

### C.4 架构层追问

🧑‍💼 **你怎么评估一个系统的性能瓶颈在哪？**
> 分层排查：
> 1. **接入层**：Nginx access log → 请求耗时分布，看是否在接入层就慢
> 2. **应用层**：Skywalking/Arthas trace → 方法级耗时，找到最慢的方法
> 3. **数据层**：慢查询日志 + Redis slowlog → 存储层是否瓶颈
> 4. **基础设施**：top/vmstat/iostat → CPU/内存/磁盘 IO/网络是否饱和
>
> 通常 80% 的性能问题出在数据层（慢 SQL、缓存未命中、锁竞争），15% 在应用层（不合理的串行调用、大对象序列化），5% 在基础设施。

🧑‍💼 **如果让你从 0 到 1 搭建一个系统，你最先做什么？**
> 1. **理解业务**：和产品对齐核心用例（不是全部用例），识别 MVP 范围
> 2. **技术选型**：根据团队能力和业务场景选基础设施（DB/缓存/MQ/框架），不追新
> 3. **定义接口**：先定义对外 API contract（OpenAPI），前后端/上下游可以并行开发
> 4. **基础设施**：CI/CD pipeline + 日志 + 监控 + 告警，这是第一天就要有的，不是上线前补
> 5. **最后写业务代码**

📝 **考察点**：P8 候选人不应该第一反应是"写代码"，而是"理解业务 + 搭基础设施"。
