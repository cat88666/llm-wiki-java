# P8 高频边界问题

> P7 能答出标准答案；P8 面试官追问的是标准答案的边界、你的方案是否最优、以及极端场景下怎么处理。本页从 55 个真实面经中提取高频追问模式，帮你提前准备边界回答。

---

## 一、系统边界与职责归属

### 追问模式

> "你做了很多反查上游的逻辑来保证一致性，这种高耦合方案真的好吗？"
> "为什么不让下游提供一个原子性的接口，而是让你们自己处理两步操作？"

这类问题考察**架构边界清晰度**：业务逻辑应该放在哪一层、哪个服务。

### 回答框架

1. **识别问题**：你做了上游本不该做的事，说明领域边界划分不清
2. **分析根因**：下游接口设计不合理（两个接口 → 应该是一个原子接口）
3. **更好方案**：让下游提供 `解冻并转账` 原子接口；上游无需感知下游实现细节
4. **权衡**：推动下游改造有成本，过渡期可保留 ACL（防腐层）做兼容

**核心答法**：

> "这个方案确实有高耦合问题，根因是下游接口设计不够原子化。更好的方案是推动下游将解冻+转账合并为一个幂等接口，这样上游只需调一次，也符合领域职责边界——下游的内部状态管理不应该暴露给上游。现阶段无法推动改造时，可以在基础设施层加防腐层做隔离，但这只是过渡方案。"

## 二、造轮子必须回答的五个问题

> "为什么不用 RabbitMQ，而要用 Redis 自己造个消息队列？"
> "你自己写的配置中心，和 Nacos 相比差在哪里？"

每次提到"我们自研了 XX"，必须能回答：

| 问题 | 参考答法 |
|------|---------|
| 为什么造？ | 现有轮子哪里不满足（依赖复杂/功能过重/延迟不符合要求）|
| 造之前评估过哪些方案？ | 列举 2-3 个竞品及放弃原因 |
| 你造的轮子优点是什么？ | 轻量/特定场景性能更好/运维简单 |
| 你造的轮子缺点是什么？ | 缺少 XX 特性（如 Redis Stream 没有消费组重平衡）|
| 如果要支持 YY 特性，你怎么做？ | 展示你真的理解了底层 |

**Redis Stream 被追问点**：
- 底层：Listpack（小数据）或 linked list of entries
- 消费组：`XREADGROUP` + `XACK` 标记已消费，`XPENDING` 查看未确认
- 没有重平衡机制（vs Kafka 的 Consumer Group Rebalance）

## 三、库存三态设计与一致性

### 库存三态模型

```
库存总量 = 可售库存 + 冻结库存 + 已扣减库存
```

| 状态 | TCC 阶段 | 操作 |
|------|---------|------|
| 可售库存 | 初始态 | 用户可抢购的数量 |
| 冻结库存 | Try 阶段 | 从可售转移到冻结（占位但未扣减）|
| 已扣减 | Confirm 阶段 | 从冻结真正扣减 |
| 回滚释放 | Cancel 阶段 | 冻结转回可售 |

### Redis 与 DB 库存一致性

> "Redis 扣减成功，但订单创建失败，库存怎么处理？"
> "Redis 和数据库的库存如何保证不出现永久不一致？"

**旁路验证方案**：

```
正常流程：
  Redis 原子扣减（Lua） → 创建订单 → DB 异步扣减 → 写库存流水

旁路验证：
  库存流水表（每次扣减/回补都记录）
  定时任务：对比 Redis 库存 vs DB 库存流水聚合值
  不一致时：以 DB 流水为准，修正 Redis

库存流水设计：
  key: event_type + 幂等号(order_id)
  value: 扣减数量 + 方向(+/-)
  保证 Lua 脚本写流水和扣库存原子操作
```

**Confirm 之后 Cancel 的处理**：
- TCC 事务表记录当前阶段（`TRIED/CONFIRMED/CANCELLED`）
- Cancel 前先查事务表：若已 Confirmed → 执行**反向补偿**（不是空操作）
- 反向补偿不等于回滚：向库存服务发"加库存"指令而非"解冻"指令

## 四、雪花算法边界追问

### 标准答案之外的追问

| 追问 | 边界回答 |
|------|---------|
| 没有时钟回拨还会重复吗？| **会**：同一毫秒内序列号耗尽（4096/ms），如果超过 4096 个并发，会等待下一毫秒；极端情况实现有 bug 可能重复 |
| Hutool 的 workerid 上限？| 最大 31（5 bit），超过 32 台机器需自定义 workerId 生成策略 |
| 超过 32 台机器怎么办？ | 重新分配 bit 布局（减少序列号位，增加机器位）；或用 ZK/Redis 分配 workeId（Leaf 的做法）|
| 时钟回拨解法有哪些？ | ①抛异常（强一致，不可用时间短）②等待到追上旧时间③向 ZK 申请新 workerId（Leaf Snowflake）|
| 机器码用 Redis 自增有什么问题？| Redis 重启后从 0 开始，可能和已有机器码冲突 → 需要 Redis 持久化或加起始偏移 |

## 五、MQ 深追问题

### 异步刷盘真的能保证不丢吗？

> "RocketMQ 用异步刷盘，异步刷盘还能保证消息不丢吗？"

**回答框架**：

| 层级 | 保证手段 | 异步刷盘的风险 |
|------|---------|--------------|
| 生产者 → Broker | acks 确认 | 消息已到 Broker 内存但未落盘，Broker 宕机丢失 |
| Broker → 磁盘 | 同步/异步刷盘 | **异步刷盘**：Broker 崩溃可能丢失缓冲区消息 |
| Broker → Consumer | 手动 ACK | Consumer 消费失败会重试，但需幂等 |

**结论**：异步刷盘**不能**保证 100% 不丢。如果需要不丢，必须用同步刷盘（`flushDiskType=SYNC_FLUSH`）+ 主从同步复制（`brokerRole=SYNC_MASTER`）。

### 如果让你设计一个消息队列，你怎么做？

**必须覆盖的 5 个维度**：

```
1. 存储模型：顺序写磁盘（Commitlog）+ 稀疏索引（减少随机 IO）
2. 可靠性：WAL + 同步/异步刷盘可选；acks 机制（0/1/-1）
3. 消费语义：At-least-once（默认）vs Exactly-once（幂等+事务）
4. 推拉模式：
   - 拉模式：消费者主动拉取，控制消费速率，Kafka/RocketMQ 选择
   - 推模式：Broker 推给消费者，实时性高，但消费者压力大
   - RocketMQ：长轮询（Broker hold 请求 30s，有消息立即返回）模拟推
5. 顺序消息：单分区内有序；跨分区无序；全局有序只能单分区（吞吐瓶颈）
```

## 六、设计追问模式

### OPS/QPS 突增 N 倍怎么办？

> "如果你的系统 OPS 突然提升 10 倍你会怎么设计？"

**分层处理框架**：

```
1. 流量来源分析（是正常增长还是攻击/爬虫？）
   → 正常：扩容；异常：限流+封禁

2. 接入层：
   - CDN 静态资源缓存，减少回源
   - Nginx 限流（limit_req_zone）
   - 熔断降级（Sentinel）

3. 应用层：
   - 水平扩容（无状态服务直接加机器）
   - 异步化（下单成功即返回，真正处理异步）
   - 本地缓存（Caffeine，避免缓存穿透到 Redis）

4. 存储层：
   - 读写分离（扩展读副本）
   - 分库分表（如果 TPS 超单库上限）
   - Redis 集群扩容

5. 兜底：
   - 限流保最核心链路可用
   - 降级非核心功能（推荐、积分）
```

### 10 万 OPS 会员系统怎么设计？

**关键决策点**：

```
读多写少 → 缓存优先（Redis 抗读，MySQL 持久化）
热点 → 本地 Caffeine 二级缓存，减少 Redis 热点 key
会员等级变更 → 低频写，直接 DB + 异步更新缓存
查询 → 会员信息强 hash（userId % 分桶），避免热点
容量 → 10万 OPS × 1ms 响应 = 100 QPS × 1000ms = 单机 Redis 轻松扛住
扩容边界 → Redis Cluster（3主3从）+ 应用无状态横向扩容
```

## 七、方案质疑话术

P8 面试官会主动质疑你的方案，典型话术：

| 面试官追问 | 正确回应姿势 |
|----------|------------|
| "这不是过度设计吗？" | 描述当时的业务量级和未来预期，说明复杂度匹配问题复杂度 |
| "你有没有考虑过更简单的方案？" | 主动说出 Plan B（更简单方案），再解释为什么选了当前方案 |
| "如果重新设计，你会怎么改？" | 说出当前方案的缺点，提出改进思路，展示持续反思能力 |
| "你在设计架构时除了技术还考虑什么？" | 团队能力/研发成本/机器资源/运维复杂度/业务演进速度 |

## 八、技术选型追问模式

> "你为什么用 ShardingJDBC 而不是 MyCat？"
> "你自己实现 TCC 而不用 Seata 是什么原因？"

**选型必须能说清楚的四个维度**：

```
1. 功能匹配度：你的场景需要哪些特性，候选方案是否满足
2. 性能/资源成本：引入了多少额外延迟，占用多少资源
3. 团队熟悉度/社区活跃度：维护成本
4. 权衡取舍：你选的方案缺少哪些竞品的特性，可以接受吗？
```

**ShardingJDBC vs MyCat 经典对比**：

| 维度 | ShardingJDBC | MyCat |
|------|-------------|-------|
| 接入方式 | JDBC 驱动，无需独立进程 | 独立中间件，改连接地址 |
| 性能 | 无额外网络跳转 | 多一次网络转发 |
| 运维 | 随应用发布，简单 | 独立运维，有单点风险 |
| 功能 | 分库分表/读写分离/数据加密 | 功能更全（如全局表广播）|
| 适用 | Java 应用，中小规模 | 多语言/遗留系统，大规模 |

## 九、线上问题定位追问

### Free 显示内存充足但频繁 FullGC

> "机器上 free 内存还有很多，但是有频繁 FullGC，可能是什么原因？"

| 原因 | 排查手段 |
|------|---------|
| JVM 堆上限配置太小（-Xmx 设置小）| `jinfo -flags pid` 查看 JVM 参数 |
| 元空间（Metaspace）频繁扩容 | GC 日志中看 `Metadata GC Threshold` |
| 堆外内存占用（DirectByteBuffer 等）| NMT（Native Memory Tracking）追踪 |
| 大量软引用在 FullGC 时被清理 | heap dump 分析软引用持有量 |
