# 德州扑克核心算法

> 德州扑克引擎的核心决策是精度和性能的取舍：牌型评估要 O(1) 比较，胜率计算按阶段选择查表/枚举/模拟，保险算法用方程组分配不同 Outs 投保额，底池结算必须确定性处理边池和平局余数。

## 一、算法边界：牌型、胜率、保险、底池四件事分开做

| 层级 | 解决的问题 | 性能/正确性约束 | 推荐 |
|------|------------|----------------|------|
| 牌型评估 | 两手牌谁大 | 高频调用，不能有 IO | 预编码 + 整数比较 |
| 胜率/Outs | 谁能翻盘，概率多少 | 阶段不同组合空间不同 | 查表/枚举/模拟分阶段 |
| 保险 | 多个 Outs 如何分摊投保额 | 赔率不同、收益约束 | 方程组求解 |
| 底池结算 | 多人 all-in 怎么分钱 | 必须确定性、可回放 | 按投入阶梯切边池 |

三种变体的影响：

| 变体 | 关键差异 | 对算法影响 |
|------|----------|------------|
| 标准德州 | 2 手牌 + 5 公牌任取 5 | 基准评估器 |
| 短牌 | 移除 2-5，葫芦 > 同花 | 牌型等级和最小顺改变 |
| 奥马哈 | 4 手牌，必须恰好 2 手牌 + 3 公牌 | 每次评估 60 种组合取最大 |

## 二、牌型评估：多项式编码让比较退化为整数比较

工厂 + 策略：

```java
PokerHandEvaluator evaluator = PokerHandEvaluatorFactory.getEvaluator(gameType);
int score = evaluator.evaluate(holeCards, boardCards);
```

多项式编码：

```
同花/高牌：c1*20^4 + c2*20^3 + c3*20^2 + c4*20 + c5
两对：     highPair*400 + lowPair*20 + kicker
一对：     pair*8000 + k1*400 + k2*20 + k3
三条：     trips*400 + k1*20 + k2
四条：     quads*100 + kicker
葫芦：     trips*100 + pair
```

| 方案 | 比较复杂度 | 优点 | 结论 |
|------|------------|------|------|
| 多字段逐项比较 | O(k) | 直观 | 分支多，容易错 |
| 字符串排序比较 | O(k) | 调试方便 | 性能差 |
| 整数编码比较 | O(1) | 快、可缓存、可排序 | 引擎首选 |

奥马哈必须恰好 2+3：

```java
int best = 0;
for (int[] h : combinations(holeCards, 2)) {
    for (int[] b : combinations(boardCards, 3)) {
        best = Math.max(best, evaluateFiveCards(h, b));
    }
}
```

反直觉：奥马哈不是“4 张手牌任选参与”，必须恰好 2 张手牌和 3 张公牌，否则大量牌型会算错。

## 三、胜率与 Outs：Pre-Flop 查表，Flop 枚举，Turn 模拟

| 阶段 | 公牌数 | 剩余枚举空间 | 推荐策略 | 结论 |
|------|--------|--------------|----------|------|
| Pre-Flop | 0 | 组合空间大但模式固定 | 预计算查表 | 性能最优 |
| Flop | 3 | `C(47,2)=1081` | 完整枚举 | 精度最优 |
| Turn | 4 | `C(46,1)=46` | 精确枚举或蒙特卡洛 | 看实时性要求 |
| River | 5 | 0 | 直接比牌 | 精确 |

Outs 定义：当前落后方在后续公共牌中能反超或平分的牌集合。

```
计算流程：
  当前赢家/输家牌力
  → 枚举未发出的牌
  → 重新评估牌力
  → 结果由输转赢：反超 Outs
  → 结果变平：平分 Outs
```

## 四、智能保险：等利方案与多 Outs 方程组

保险目标：领先玩家花钱购买某些 Outs 的赔付，使“爆牌”和“不爆牌”的收益更接近。

| 方案 | 目标 | 推荐场景 |
|------|------|----------|
| 等利方案 | 爆牌和不爆牌收益相同 | 保守玩家 |
| 1/4 底池 | 覆盖部分波动 | 中性 |
| 1/8 底池 | 少量对冲 | 激进 |
| 1/16 底池 | 最低保护 | 极激进 |

等利推导：

```
设 T = 投保额，r = 赔率
爆牌后收益 = 新控制金额 + T*r - T
不爆牌收益 = 原控制金额 - T
令两者相等，可解出 T
```

多 Outs 方程：

```
每个 Outs i 投保额 Ai = (Hi + T) / (ri + 1)
约束：ΣAi = T

T = [Σ Hi/(ri+1)] / [1 - Σ 1/(ri+1)]

若 Ai < 0：清零该 Outs，重新计算，最多迭代 20 次
```

## 五、边池结算：按投入阶梯切池，按资格分配

```
A all-in 100，B 投 200，C 投 200

主池：100 * 3 = 300，参与 A/B/C
边池：100 * 2 = 200，参与 B/C
```

算法步骤：

```
1. 收集所有玩家有效投入，按金额升序去重
2. 每个金额阶梯生成一个 pot
3. pot 金额 = (当前阶梯 - 上一阶梯) * 仍有资格玩家数
4. pot 参与者 = 投入达到当前阶梯且未弃牌玩家
5. 每个 pot 独立比牌并分配
6. 平局余数按固定座位顺序分配，保证确定性
```

| 风险 | 错误做法 | 正确做法 |
|------|----------|----------|
| all-in 玩家赢不该赢的钱 | 所有人争总池 | 每个边池有参与资格 |
| 平局余数不确定 | 随机分配 | 固定座位/行动顺序 |
| 抽水重复 | 每个边池独立乱抽 | 按配置对有效盈利或底池统一计算 |

## 七、面试追问

### 1. 如何快速比较两手牌大小？

把牌型等级和同牌型内的关键牌编码成整数，比较时先比牌型等级，再比编码值。这样两手牌比较退化为整数比较，O(1)，适合游戏房间内高频调用。

---

### 2. Pre-Flop 为什么查表，Flop 为什么枚举？

Pre-Flop 没有公牌，手牌组合模式固定，可以预计算缓存；Flop 后只剩 Turn/River 两张未知牌，`C(47,2)=1081`，完整枚举成本可控且结果精确。

---

### 3. 多个 Outs 的保险金额怎么分？

不同 Outs 赔率不同，不能平均分。用方程组约束总投保额等于各 Outs 投保额之和，并让目标收益满足方案要求；如果某个 Outs 算出负投保额，就置零后重新求解。

---

### 4. 多人 all-in 怎么分边池？

按玩家投入金额升序切阶梯，每个阶梯形成一个池，只有投入达到该阶梯且未弃牌的玩家参与竞争。每个边池独立比牌，平局余数按固定座位顺序分配，保证可回放。
