# 德州扑克核心算法

> 一套支持标准德州/短牌/奥马哈三变体的游戏引擎，四层核心算法：**牌型评估**（多项式编码 O(1) 比较）→ **胜率/Outs 计算**（分阶段策略）→ **智能保险**（方程组求解赔率）→ **底池结算**（边池 + 抽水分配）；选型结论取决于计算精度要求和枚举空间大小。

## 一、场景概述与问题拆解

**四层算法的工程问题边界**：

| 层级 | 工程问题 | 核心约束 |
|------|---------|---------|
| 牌型评估 | 如何快速比较两手牌大小？ | 无 DB IO，微秒级完成 |
| 胜率/Outs | 如何实时计算落后玩家的翻盘概率？ | Pre-Flop 枚举空间太大，需要分阶段策略 |
| 智能保险 | 多个 Outs 同时投保时如何分配投保额？ | 赔率不同，需要方程组求解 |
| 底池结算 | 多人 all-in 时如何正确切分边池？ | 参与资格不同，余数处理必须确定性 |

**三变体对核心算法的影响**：标准德州 → 短牌（移除 2-5，葫芦>同花）→ 奥马哈（必须用恰好 2 张手牌 + 3 张公牌，C(4,2)×C(5,3)=60 种组合）。

## 二、牌型评估系统

### 工厂模式：按变体分配评估器

```java
// 工厂根据 gameType 分配评估器，新增变体只需新增实现类
PokerHandEvaluatorFactory.getEvaluator(gameType)
  // isTexas   → NormalPokerHandEvaluator（标准德州）
  // isShortDeck → ShotPokerHandEvaluator（短牌，葫芦>同花）
  // isOmaha   → AmhNormalPokerHandEvaluator（奥马哈，强制 2+3）
```

### 核心比较算法：多项式系数编码

**设计目的**：把 5 张牌的强弱映射到一个整数，O(1) 比较，避免多字段比较的复杂性。

```
同花（5张高牌编码）：
  result = c1×20⁴ + c2×20³ + c3×20² + c4×20 + c5
  （c1~c5 为从高到低排序的 5 张牌点数）

两对：result = pair1×400 + pair2×20 + kicker
一对：result = pair×8000 + k1×400 + k2×20 + k3
三条：result = three×400 + k1×20 + k2
四条：result = four×100 + kicker
葫芦：result = three×100 + pair
```

所有牌型映射到同一整数空间后直接 `>` 比较，两手牌比较 = 一次整数大小比较。

### 牌型层级

| 等级 | 牌型 | 短牌变化 |
|------|------|---------|
| 10 | 皇家同花顺 | 同 |
| 9 | 同花顺 | 同 |
| 8 | 四条 | 同 |
| **7/6 互换** | **葫芦/同花** | **短牌中葫芦 > 同花**（同花更难凑） |
| 5 | 顺子（最小：标准 A2345，短牌 A6789）| 不同最小顺 |
| 4 | 三条 | 同 |
| 3 | 两对 | 同 |
| 2 | 一对 | 同 |
| 1 | 高牌 | 同 |

### 奥马哈的特殊处理

```java
// 必须用恰好 2 张手牌 + 3 张公牌，穷举所有有效组合取最优
// C(4,2) = 6 种手牌组合 × C(5,3) = 10 种公牌组合 = 60 种
int bestScore = 0;
for (int[] holeCombo : combinations(4, 2)) {  // C(4,2) = 6
    for (int[] boardCombo : combinations(5, 3)) {  // C(5,3) = 10
        int[] fiveCards = merge(holeCombo, boardCombo);
        bestScore = Math.max(bestScore, evaluator.evaluate(fiveCards));
    }
}
```

## 三、胜率与 Outs 计算

### 分阶段策略（精度 vs 性能权衡）

| 发牌阶段 | 公牌数 | 策略 | 原因 | 结论 |
|---------|--------|------|------|------|
| Pre-Flop | 0 | **查表法**（预建 2 万+组合） | 组合固定，提前缓存 | 性能最优 |
| Flop | 3 | **完整枚举** C(47,2)=1081 种 | 空间可接受，精度高 | 精度最优 |
| Turn | 4 | **蒙特卡洛模拟** | 精度要求较低，快速估算 | 近似可接受 |
| River | 5 | **精确比牌**（公牌已固定）| O(n)，最简单 | 精确 |

### Outs 定义与计算

**Outs**：落后玩家翻出后能反超领先玩家的那些牌。

```
逻辑分支（findTwoHandCardOuts）：
  1. baseHand 有更高对子 → 返回 hand2 能凑出更高对子的 outs
  2. hand2 有更高对子 → 返回 baseHand 能凑出更高对子的 outs
  3. 都是高牌 → 返回 hand2 中比 baseHand 最高牌大的所有点数

Outs 类型：
  outsType=1（反超）：落后者能翻盘的牌
  outsType=2（平分）：多人同时获益导致平分
```

## 四、智能保险：赔率方程组求解

### 整体流程

```
findCanInsureUser() →

  getPredictCardDto()  — 构建各玩家当前最优牌力

  buildInsurePotDetailMap() — 处理背回逻辑
  checkAvailable()     — 检查方案可用性（4 个约束条件）
```

### 四种投保方案

| 方案 | 类型 | 投保逻辑 | 结论 |
|------|------|---------|------|
| 方案 0 | `WAITING_PROFIT` | 等利：投保后爆牌与不爆牌收益相同 | 保守型，消除运气成分 |
| 方案 1 | `ONE_FOURTH_POT` | 期望值使 1/4 底池损失平摊 | 中性 |
| 方案 2 | `ONE_EIGHTH_POT` | 期望值使 1/8 底池损失平摊 | 激进型 |
| 方案 3 | `ONE_SIXTEENTH_POT` | 期望值使 1/16 底池损失平摊 | 最激进 |

### 方案 0：等利方案推导

```
目标：投保后无论 Outs 中还是没中，收益相同

设 T = 总投保额，ri = 第 i 个 Outs 的赔率
  投保后爆牌收益 = 新控制金额 + T×ri - T = 新控制金额 + T(ri-1)
  不投保收益     = 原控制金额

令两者相等：
  T = (原控制金额 - 新控制金额) / (ri + 1)
```

### 方案 1-3：方程组迭代求解

**问题**：多个 Outs 赔率不同，如何在总投保额 T 约束下使各 Outs 收益达到目标 Hi？

```
建立方程组：
  每个 Outs i 的投保额 Ai = (Hi + T) / (ri + 1)
  ΣAi = T（约束）

消元求解 T：
  T = [Σ Hi/(ri+1)] / [1 - Σ 1/(ri+1)]

反推各 Ai = (Hi + T) / (ri + 1)

迭代修正（最多 20 次）：
  若存在 Ai < 0 → 将负值清零，重新计算 T
  直到所有 Ai ≥ 0 或迭代次数达上限
```

### 方案可用性检查

```
条件1: 赢利额 ≥ 期望收益值（不能亏钱）
条件2: 单池投保总额 ≤ 上限
       Turn 圈：上限 = 自己投入 × 0.25
       River 圈：上限 = 控制金额 - 上轮投保额
条件3: 控制金额 ≥ 10 × 大盲（底池充足才允许投保）
条件4: 方案投保额 > 0
```

## 五、底池结算与抽水

### 边池生成

```
示例：A all-in 投 100，B 投 200，C 投 200

主池 = min(100,200,200) × 3 = 300    参与：A, B, C
边池1 = (200-100) × 2 = 200          参与：B, C

若 B 在 all-in 150 时（B 合计投入 150）：
  主池 = 100 × 3 = 300               参与：A, B, C
  边池1 = 50 × 2 = 100               参与：B, C
  边池2 = 50 × 1 = 50                参与：C（超出 B all-in 的部分 C 独吞）
```

**平局余数处理**：`totalPot / winners.size()` 的余数分给**位置最靠前**（最先行动）的赢家，确保确定性。

### 两种抽水模式

| 模式 | 计算基础 | 适用场景 |
|------|---------|---------|
| **POT_SCALE** 按底池比例 | `应抽 = 总有效底池 × 抽水比例%` | 标准抽水 |
| **PROFIT_SCALE** 按盈利比例 | `应抽 = 总盈利 × 抽水比例%` | 只对赢的钱抽水 |

**玩家最终净收益**：`赢得金额(已扣水) - 玩家投入 + 玩家贡献的抽水返还`

## 七、面试高频追问

**问**：德州扑克中如何快速比较两手牌大小？
**答**：用多项式系数编码，把 5 张牌的强弱映射到一个整数。同花是 `c1×20⁴+c2×20³+...+c5`，两对是 `pair1×400+pair2×20+kicker`。比较两手牌只需一次整数大小比较，O(1)，无需重新排列或多字段比较。

**问**：保险赔率怎么计算？多个 Outs 时如何处理？
**答**：建立方程组：每个 Outs i 的投保额 Ai = (目标收益Hi + 总投保额T) / (赔率ri + 1)，且 ΣAi = T。消元得到 T 的解析式，反推各 Ai。若有负值（赔率过低）则清零后迭代修正，最多 20 次收敛。

**问**：边池如何分配？余数怎么处理？
**答**：多人 all-in 时按各人 all-in 金额切分主池和边池。每个边池只有当时有资格竞争的玩家才能参与。结算从主池开始逐级比牌。余数（整除后的尾差）统一归入位置最靠前的赢家，保证分配的确定性。

**问**：如何支持多个游戏变体？
**答**：工厂模式 + 策略模式：PokerHandEvaluatorFactory 根据 gameType 分配不同评估器；抽象基类封装公共逻辑，子类覆盖差异点（短牌的葫芦>同花、奥马哈的强制 2+3 选牌逻辑）。新增变体只需新增工厂条件和对应 Calculator 实现，不修改核心比较流程。

**问**：Pre-Flop 为什么能用查表法，Flop 为什么要枚举？
**答**：Pre-Flop 只有手牌组合（无公牌），两人的 2+2 张手牌组合数固定有限（约 2 万），可预建缓存。Flop 有 3 张公牌，剩余 47 张牌中选 2 张发 Turn+River，C(47,2)=1081 种，枚举可接受，精度更高。查表在 Pre-Flop 阶段性能最优，枚举在 Flop 精度最优。
