# P7架构思维

> P7 架构面试考的不是会不会用 Redis、MQ、微服务，而是能否把方案放进约束里讲清楚：为什么选 A、不选 B，代价是什么，如何补偿；默认表达范式是 **场景约束 → 候选方案 → 取舍结论 → 落地边界**。

## 一、架构表达：从“用了什么”升级到“为什么这么选”

| 常见低阶表达 | P7 应答方式 |
|--------------|-------------|
| 我们用了 Redis 做缓存 | 读多写少且允许最终一致，所以用 Redis；代价是短暂不一致，用 TTL/删缓存/消息刷新兜底 |
| 我们拆了微服务 | 单体出现独立扩容和团队协作瓶颈后拆；代价是分布式事务和治理成本 |
| 我们用了 MQ | 主链路不需要同步等待，所以用 MQ 解耦削峰；代价是最终一致和消费幂等 |

标准句式：

> 在 A 和 B 之间，我们选 A，因为当前核心约束是 X；A 的代价是 Y，我们用 Z 兜底。B 在 M 上更好，但对当前阶段收益不够覆盖复杂度。

架构目标天然冲突：

| 矛盾 | 典型冲突 | 取舍结论 |
|------|----------|----------|
| 可扩展性 vs 简洁性 | 抽象越多越难理解 | 没有明确扩展点前不抽象 |
| 高性能 vs 高可用 | 副本、隔离、确认都会增加延迟 | 核心链路按 SLA 取舍 |
| 安全性 vs 易用性 | 鉴权、审批、审计增加摩擦 | 资金/权限链路安全优先 |
| 快速交付 vs 代码质量 | 赶进度产生债务 | 债务必须显式化 |

## 二、好架构判断：适配阶段，而不是堆技术

| 特质 | 合格标准 | 失分信号 |
|------|----------|----------|
| 可理解性 | 新人 1-2 天能理解主链路 | 只能靠老人讲，文档缺失 |
| 可扩展性 | 新增能力主要加模块，不大改主流程 | 一个需求改多个服务 |
| 可维护性 | 线上问题 30 分钟内能定位方向 | 日志无 traceId、监控缺失 |
| 复杂度匹配 | 技术复杂度匹配业务规模 | 小系统上来微服务/DDD/中台全套 |
| 服务业务 | 架构支撑迭代速度 | 架构升级阻塞业务交付 |

结论：好架构不是技术栈先进，而是在当前业务规模、团队能力、稳定性要求下，复杂度和收益匹配。

## 三、微服务拆分：业务边界清晰后再拆

```
是否出现独立扩容、独立发布、团队协作瓶颈？
  ├─ 否 → 单体内模块化，先保留简单性
  └─ 是
      ├─ 业务边界是否清晰？
      │   ├─ 否 → 先梳理领域边界，避免拆出分布式泥球
      │   └─ 是 → 按团队边界和数据边界拆
      └─ 是否能承担治理成本？
          ├─ 否 → 延后拆分
          └─ 是 → 注册发现、配置、限流、链路追踪先补齐
```

| 拆分原则 | 工程含义 | 结论 |
|----------|----------|------|
| 职责单一 | 服务围绕一类业务能力 | 用户、交易、营销不要混在一起 |
| 单向依赖 | 禁止 A 调 B，B 又调 A | 循环依赖说明边界错了 |
| 数据自治 | 服务拥有自己的核心数据 | 共享库会让拆分失效 |
| 康威定律 | 服务边界匹配团队边界 | 两团队共管一服务会冲突 |
| 流量隔离 | 秒杀/大促与日常链路分离 | 高峰流量不能拖垮常规订单 |

## 四、技术选型：功能、成熟度、成本、团队匹配

| 维度 | 问题 | 结论导向 |
|------|------|----------|
| 功能满足度 | 当前和未来 1-2 年需求是否覆盖 | 不为遥远需求引入重组件 |
| 成熟度 | 生产验证、社区、故障案例是否充分 | 核心链路优先成熟技术 |
| 综合成本 | 学习、开发、迁移、运维成本 | 运维成本常被低估 |
| 团队匹配 | 团队是否熟悉、能否排障 | 孤岛技术会放大人员风险 |

示例：

| 选型 | 候选 | 推荐 |
|------|------|------|
| 消息队列 | Kafka/RocketMQ/RabbitMQ | 日志流选 Kafka，交易消息选 RocketMQ，简单异步可 RabbitMQ |
| 分库分表 | ShardingSphere/MyCat/自研 | Java 新项目优先 ShardingSphere-JDBC |
| 缓存同步 | TTL/删缓存/Canal | 普通业务删缓存，强依赖读新值才考虑 Canal |

## 五、架构演进：初始设计定骨架，演进解决新瓶颈

```
初始设计：模块边界、接口契约、存储选型、可观测性
  → 业务增长：流量、数据量、团队规模变化
  → 发现瓶颈：慢 SQL、热点、发布耦合、故障定位困难
  → 渐进演进：缓存、异步、拆服务、分库分表、治理补齐
```

技术债管理：

| 做法 | 适用场景 | 结论 |
|------|----------|------|
| 显式化 | 短期妥协、历史遗留 | TODO + 工单 + owner |
| 排优先级 | 多个债务并存 | 高频变更、高故障、高成本优先 |
| 渐进式改造 | 线上系统 | 双写、灰度、旁路验证 |
| 大重构 | 边界极少 | 风险高，默认不推荐 |

## 六、系统设计答题：先业务架构，再技术架构

项目介绍优先级：

```
1. 业务架构：核心模块和主流程
2. 技术架构：服务分层、通信方式、中间件
3. 数据架构：核心表、分库分表、缓存索引
4. 部署架构：机器规模、容器化、多活
```

亿级商品存储选型：

| 场景 | 方案 | 结论 |
|------|------|------|
| 商品详情按 ID 查 | MySQL 分库分表 + Redis | 主流可靠 |
| 商品列表复杂筛选 | ES + Canal/MQ 同步 | 不用 MySQL 扛复杂搜索 |
| 商品热点详情 | 本地缓存 + Redis | 大促前预热 |
| 免分片诉求 | TiDB/OceanBase | 运维成本高，中小团队慎用 |

单元化架构：

```
每个单元 = 完整业务链路 + 一部分数据分片
优势：单元内闭环、延迟低、支持异地多活
代价：全链路改造极高；一个核心节点未单元化，整条链路仍会回中心
结论：只适合超大规模公司，不是普通业务默认答案
```

## 七、面试追问

### 1. 架构是设计出来的还是演进出来的？

两者缺一不可。初始设计确定模块边界、接口契约和存储选型；演进解决业务增长后的新瓶颈。P7 的关键是预留演进空间，但不为三年后的假设过度设计。

---

### 2. 什么情况下不该拆微服务？

业务边界不清、团队规模不够、单体还没有扩展瓶颈时不该拆。拆分本质是用分布式复杂度换独立扩容和独立交付，没有瓶颈时这笔交易不划算。

---

### 3. 技术债怎么管理？

先显式化，再排序，再渐进还。高频改动、高故障、高排障成本的债务优先；避免一次性大重构，线上系统更适合灰度、双写和旁路验证。

---

### 4. 什么是没有银弹？

没有任何单一技术能消除软件本质复杂性。微服务解决单体扩展问题，但引入服务治理、分布式事务、链路追踪；DDD 改善模型表达，但增加建模和团队认知成本。

---

### 5. 线上系统性能瓶颈怎么分层定位？

按接入层、应用层、数据层、基础设施排查：Nginx access log 看整体耗时，链路追踪看方法和 RPC，慢 SQL/Redis slowlog 看数据层，`top/vmstat/iostat` 看机器资源。经验上数据层问题最多，但必须用指标证明。

## 八、P7思维与分布式、SpringCloud 的关系

- [[概念-分布式理论]]：CAP 是一致性、可用性取舍的理论基础，P7 要能讲业务为什么选 AP/CP。
- [[机制-SpringCloud]]：微服务拆分后的注册发现、配置中心、熔断限流是治理成本的一部分。
- [[机制-Spring]]：IoC 和依赖倒置支撑模块解耦，是单体模块化和服务化前的基础能力。
- [[概念-分布式场景]]：分布式锁、ID、Session、事务是 P7 架构选型中最常见的落地点。
