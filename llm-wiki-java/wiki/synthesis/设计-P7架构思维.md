# P7架构思维

> P7 架构面试考核权衡表达能力：不说"我用了 X"，而说"在 A 和 B 之间选了 X，代价是 Y，弥补手段是 Z"；选型结论必须有场景约束，没有绝对最优解。

## 一、架构权衡框架

**架构目标天然互相矛盾**：

| 矛盾对 | 典型冲突 | 结论 |
|--------|---------|------|
| 可扩展性 ↔ 简洁性 | 每层抽象都是复杂度 | 抽象前先问"真的需要扩展吗" |
| 高性能 ↔ 高可用 | 副本/冗余增加延迟 | 业务对延迟的容忍度决定取舍 |
| 安全性 ↔ 易用性 | 鉴权增加开发摩擦 | 核心链路安全优先，内部工具可简化 |
| 快速交付 ↔ 代码质量 | 产生技术债 | 技术债必须显式管理，不能任其积累 |
| 统一技术栈 ↔ 最优工具 | 强制一致 vs 因地制宜 | 团队规模小时统一优先，规模大再分化 |

**P7 面试必备表达范式**：

> 错误示范："我们用了 Redis 做缓存。"
>
> 正确示范："在一致性和性能之间，我们选择了 Redis 缓存，因为业务允许最终一致，读多写少场景命中率 > 95%，代价是缓存与 DB 短暂不一致，我们通过 Canal + MQ 异步刷新来弥补。"

**没有银弹（Brooks 定律）**：软件本质复杂性（Essential Complexity）无法被任何单一技术消除。微服务带来分布式复杂性，DDD 带来高建模成本，云原生带来运维复杂性。本质复杂性只是转移，不会消失。遇到"彻底解决 XX 问题"的方案，先问代价是什么。

## 二、好架构判断矩阵

| 特质 | 合格标准 | 失分信号 |
|------|---------|---------|
| **可理解性** | 新人 2 天能上手，模块职责清晰 | 需要口口相传，无文档 |
| **可扩展性** | 加功能不需要大改 | 改一处动全身 |
| **可维护性** | 故障 30 分钟内定位 | 无监控，日志无 traceId |
| **避免过度设计** | 只解决当前问题 | 为未来三年过度设计 |
| **服务业务** | 能跟上需求迭代 | 架构升级比业务需求慢 |

**结论**：好架构没有统一标准，只有"适合当前业务阶段和团队能力"的架构。

## 三、微服务拆分决策树

```
当前服务是否已出现扩展瓶颈或维护困难？
  → No → 先做单体内模块化，不急于拆分
  → Yes →
      服务间是否存在循环依赖？
        → Yes → 先梳理边界，拆分会固化循环依赖（更难解决）
        → No → 按以下原则拆：
            ① 单向依赖：下层不依赖上层，禁止循环
            ② 康威定律：服务边界 = 团队边界
            ③ 保障隔离：秒杀/在线 vs 日常/离线分开
            ④ 公共能力中台化：短信/权限/消息独立为基础服务
```

**微服务拆分原则对比**：

| 原则 | 说明 | 结论 |
|------|------|------|
| 职责单一 | 一个服务聚焦一件事 | 用户≠交易，拆后可独立扩容 |
| 单向依赖 | 禁止 A→B→A 循环 | 循环依赖说明边界划分不合理 |
| 康威定律 | 服务边界对齐团队边界 | 两团队维护一服务必然冲突 |
| 系统隔离 | 高峰与常规流量分离 | 秒杀流量不能打垮日常订单 |

**拆分时机**：业务边界清晰 + 单体已遇扩展瓶颈 + 团队规模能独立维护，三者同时满足。

## 四、技术选型四维评估

| 维度 | 评估问题 | 结论导向 |
|------|---------|---------|
| **功能满足度** | 当前 + 未来 1-2 年需求能覆盖吗？ | 过度强大 = 引入不必要复杂度 |
| **成熟度** | 大厂生产验证过？社区活跃？| 新技术踩坑成本极高 |
| **综合成本** | 学习/使用/迁移/运维成本 | 运维成本常被严重低估 |
| **技术栈匹配** | 和团队现有体系兼容吗？ | 孤岛技术 = 人才稀缺风险 |

**选型表达模板**：

> "在 A 和 B 之间选了 A。A 的优点是 [X]，代价是 [Y]；B 在 [Z] 上更优，但对我们当前阶段 [Z] 不是核心约束，引入 B 的学习成本 / 运维复杂度 / 技术栈兼容问题超过了收益。"

**反例**：全公司用 Kafka，为追求新颖引入 Pulsar → 团队无经验 + 运维成本翻倍。

## 五、架构演进与技术债

**初始设计 vs 演进的关系**：

```
初始设计（骨架）─────→ 业务增长 ─────→ 演进
  模块划分              发现瓶颈         拆分/合并服务
  通信方式              发现债务         引入新中间件
  存储选型              需求变化         技术债重构
  接口契约
```

P7 面试话术："我们当初设计时考虑了 X，随业务发展发现 Y 的问题，做了 Z 演进，核心权衡是在 A 和 B 之间选择了 B，代价是..."

**技术债管理**：

| 处理方式 | 适用场景 | 结论 |
|---------|---------|------|
| 显式化（TODO + 工单） | 已知短期妥协 | 立即做，不让债务隐形 |
| 按优先级还债 | 高频路径的债 | 改动频率 × 严重度 排序 |
| 规范预防新债 | 架构评审/测试门禁 | 防新债比还旧债更重要 |
| 大重构 | 几乎不推荐 | 风险极高，优先渐进式改造 |

## 六、系统设计答题框架

**介绍项目架构的四个维度（按优先级）**：

```
1. 业务架构（首选）：核心模块划分 + 核心业务流程图
2. 技术架构（首选）：技术栈 + 分层设计 + 服务通信 + 中间件
3. 部署架构（非主动问不提）：容器化/K8s + 机器规模
4. 数据架构（非主动问不提）：存储选型 + 分库分表 + 数仓
```

**亿级商品存储方案对比**：

| 方案 | 适用场景 | 代价 | 结论 |
|------|---------|------|------|
| MySQL + 分库分表 | 主流方案，商品 ID hash 分片 | 改造成本中等 | 优先选，生产验证充分 |
| TiDB/OceanBase | 兼容 MySQL 协议，免分片改造 | 运维成本高 | 中小公司慎用 |
| Redis 热点缓存 | 命中率 > 95% 时 DB 压力可接受 | 需防穿透/击穿 | 必须配套缓存策略 |

商品列表（复杂过滤）→ ES 建索引 + Canal 同步；商品详情（ID 精确查）→ 分库分表 + Redis。

**单元化架构（面试了解即可）**：

```
每个单元 = 完整业务链路 + 分配数据分片，单元内封闭执行，不跨单元
优势：低网络延迟、天然异地多活、弹性扩容
代价（高频考点）：
  1. 全链路改造成本极高（一个节点未单元化则整链路走中心）
  2. 用户漫游（跨单元）路由复杂
  3. 仅超大规模系统（阿里/蚂蚁）采用
```

**MVC vs 三层（一句话区分）**：MVC 是**表示层内部**分工（Model/View/Controller）；三层是**整个系统**纵向切割（表示/业务/数据）。两者不互斥，常组合使用。

## 七、面试高频追问

**问**：架构是设计出来的还是演进出来的？
**答**：两者缺一不可。初始设计确定系统骨架（模块划分、接口契约、存储选型），演进响应业务变化（拆分服务、引入中间件）。好架构师在初始设计预留演进空间，但不为未来三年过度设计。

**问**：什么是没有银弹？举个实际例子。
**答**：Brooks 定律：软件本质复杂性无法被单一技术消除。例如，微服务解决了单体扩展瓶颈，但引入了分布式事务、服务治理、链路追踪等新复杂性，本质复杂性只是从单体转移到了网络，没有消失。

**问**：什么情况下不应该拆微服务？
**答**：①业务边界不清晰（拆了仍会耦合，且更难修改）；②团队规模不够独立维护（每个服务需要独立部署和运维）；③单体 QPS 远未达到瓶颈（过早优化的代价是分布式复杂度）。拆分本质是"用分布式复杂性换单体扩展瓶颈"，无瓶颈时这笔交换不合算。

**问**：技术债怎么管理？
**答**：三步：显式化（TODO + 工单，不让债务隐形）→ 优先级排序（高频路径债务优先还）→ 规范预防（架构评审、测试覆盖率门禁）。避免"大重构"式一次性还债，渐进改造风险更低。

**问**：单元化架构最难的点是什么？
**答**：全链路改造成本。有一个节点（如老支付核心）未完成单元化，该节点之后所有链路都必须走中心单元，打破了单元封闭前提。这是单元化只有超大规模公司才采用的根本原因。

## 八、与其他概念的关系

- 架构权衡的理论基础来自 [[概念-分布式理论]]：CAP 定理是"一致性 vs 可用性"权衡的形式化模型
- 微服务拆分后的服务治理依赖 [[机制-SpringCloud]]：注册发现、配置中心、熔断降级是拆分后的标配
- Spring IoC 容器是架构解耦的实现基础，参见 [[机制-Spring]]：依赖倒置（DIP）落地依赖 IoC
- 分布式场景工程选型见 [[概念-分布式场景]]：分布式锁、Session、ID 生成的具体方案
