# 项目难点表达

> 项目难点不是罗列“用了 Redis/MQ/线程池”，而是把真实问题讲成可验证的工程决策：背景有规模，方案有对比，动作有边界，结果有数字；默认结构是 **问题背景 → 候选方案 → 我负责的实现 → 量化结果 → 缺陷和追问**。

## 一、亮点筛选：优先讲真实生产问题和可量化结果

| 亮点类型 | 可讲场景 | 不建议讲法 |
|----------|----------|------------|
| 性能优化 | 10s 降到 1s、P99 降低、吞吐提升 | “我用了线程池” |
| 并发正确性 | 支付与关单竞争、库存超卖、重复提交 | “加了分布式锁” |
| 可靠性一致性 | MQ 失败补偿、跨服务最终一致 | “用了 MQ 解耦” |
| 数据结构应用 | BitSet 降内存、ZSet 排行榜 | “Redis 很快” |
| 工程效率 | AOP 去重复、事件解耦、任务分片 | “代码更优雅” |

好亮点必须同时满足：

| 标准 | 说明 |
|------|------|
| 有业务背景 | 用户量、QPS、数据量、故障影响 |
| 有方案对比 | 为什么不是更简单方案 |
| 有个人动作 | 你负责设计/实现/压测/上线哪部分 |
| 有量化结果 | 耗时、吞吐、内存、故障率、任务量 |
| 有边界缺陷 | 方案在什么情况下不适用 |

## 二、性能优化：串行改并发、多级缓存、对象分配优化

### 2.1 CompletableFuture 并发编排：外部 I/O 串行改并发

> 我负责优化主动还款链路，原来要串行暂停 50 笔在途扣款单，每笔调用外部资金系统约 200ms，总耗时 10s 以上；我用 CompletableFuture + 自定义线程池并发编排，把耗时降到约 1s，代价是异常聚合和线程池隔离要额外设计。

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| 串行调用 | 简单 | 50 笔耗时线性叠加 | 不满足 RT |
| parallelStream | 写法短 | 共用 ForkJoinPool，不可隔离 | 不推荐 |
| CompletableFuture + 自定义线程池 | 并发可控、隔离外部 I/O | 代码复杂 | 推荐 |

```java
ExecutorService executor = Executors.newFixedThreadPool(32);
List<CompletableFuture<Result>> futures = orders.stream()
    .map(order -> CompletableFuture.supplyAsync(
        () -> paymentClient.pause(order.getId()), executor)
        .orTimeout(800, TimeUnit.MILLISECONDS))
    .toList();

CompletableFuture.allOf(futures.toArray(new CompletableFuture[0])).join();
```

追问边界：线程池大小按外部接口 RT、QPS、下游限流一起评估，不能无限加线程；外部失败要聚合结果，决定整体失败还是部分补偿。

### 2.2 多级缓存：读多写少接口降 P99

| 层级 | 作用 | 风险 |
|------|------|------|
| Caffeine | 本地微秒级读取 | 多实例不一致 |
| Redis | 分布式缓存 | 热 key、穿透、击穿 |
| DB | 权威数据 | 慢查询和连接池压力 |

口径要点：讲清命中率、P99 变化和一致性窗口；不要只说“加了 Redis”。

## 三、并发正确性：状态机、乐观锁、幂等 Key

### 3.1 支付成功和超时关单竞争

> 我们遇到过支付回调和超时关单同时到达的问题。我用订单状态机 + 乐观锁把 `WAIT_PAY` 到 `PAID/CLOSED` 做成互斥转换，CAS 失败后读取最新状态并触发退款或忽略，避免订单既支付又关闭。

```sql
UPDATE orders
SET status = #{newStatus}, version = version + 1
WHERE order_id = #{orderId}
  AND status = 'WAIT_PAY'
  AND version = #{version};
```

| 方案 | 适用 | 结论 |
|------|------|------|
| 悲观锁 | 冲突高、事务短 | 支付关单冲突低，不首选 |
| 乐观锁 | 冲突低、状态互斥 | 默认选择 |
| 分布式锁 | 跨进程互斥 | 不能替代 DB 状态条件 |

### 3.2 防重复提交：Token 与幂等 Key

| 场景 | 推荐方案 | 表达重点 |
|------|----------|----------|
| 表单提交 | Token 一次性消费 | 前后端协作、GET+DEL 原子 |
| API 重试 | 幂等 Key | 参数归一化、TTL、DB 唯一兜底 |
| 资金流水 | 业务唯一号 | 不依赖 Redis TTL，必须落 DB |

## 四、可靠性一致性：本地消息表、补偿任务、对账

### 4.1 本地消息表：业务写入和消息发送解耦

> 订单创建后要通知积分系统，直接发 MQ 存在“订单写成功但消息发送失败”。我把订单和本地消息表放在同一 DB 事务里，事务提交后异步投递 MQ，失败由定时任务补偿；消费者按业务幂等键去重，保证最终一致。

```java
@Transactional
public void createOrder(Order order) {
    orderMapper.insert(order);
    localMessageMapper.insert(new LocalMessage(order.getId(), "ORDER_CREATED", "INIT"));
}
```

```sql
SELECT *
FROM local_message
WHERE state = 'INIT'
  AND created_at < NOW() - INTERVAL 5 MINUTE
LIMIT 100;
```

| 方案 | 优点 | 代价 | 结论 |
|------|------|------|------|
| 直接发 MQ | 简单 | 事务和消息不原子 | 不推荐 |
| MQ 事务消息 | 语义强 | 依赖中间件能力 | RocketMQ 可选 |
| 本地消息表 | 通用、可控 | 需要补偿任务 | 通用首选 |

### 4.2 对账：异步链路必须有最终裁决

凡是用了 MQ、缓存、三方接口，都要准备对账口径：以哪个系统为准、差异怎么修复、多久扫描一次、是否有人工介入。

## 五、数据结构亮点：BitSet、ZSet、布隆过滤器

### 5.1 BitSet 商品预约：空间压缩

| 方案 | 100 万用户内存 | 查询 | 结论 |
|------|----------------|------|------|
| Redis Set | 约几十 MB | O(1) | 内存大 |
| Redis BitSet | 约 125 KB | O(1) | userId 连续时首选 |
| BloomFilter | MB 级 | O(1)，有误判 | 适合存在性判断，不适合精确预约 |

```java
redisTemplate.opsForValue().setBit("reserve:" + itemId, userId, true);
Boolean reserved = redisTemplate.opsForValue().getBit("reserve:" + itemId, userId);
```

边界：userId 极大且稀疏时 BitSet 会浪费空间，要按 userId 分桶或改用 RoaringBitmap。

### 5.2 ZSet 排行榜：同分排序要提前设计

```java
double rankScore = score * 10000000000D + (MAX_TS - timestamp);
redisTemplate.opsForZSet().add("rank:" + gameId, userId, rankScore);
```

表达重点：Redis ZSet 解决实时 TopN，但历史榜单、分页深翻和审计仍要落 DB 或离线存储。

## 六、工程效率：注解 AOP、事件解耦、分布式任务

### 6.1 自定义注解 + AOP：统一 Facade 防腐

> 多个 Facade 接口都有参数校验、日志、异常转换和耗时统计，我用 `@Facade` 注解 + AOP 统一处理，新接口只加注解即可覆盖通用逻辑，减少重复代码并统一错误口径。

| 能力 | AOP 处理 | 价值 |
|------|----------|------|
| 参数校验 | Hibernate Validator | 统一入参错误 |
| 日志 | 入参/出参/耗时/traceId | 排障统一 |
| 异常 | 业务异常和系统异常转换 | 对外口径一致 |
| 监控 | 统计耗时和失败率 | 可观测 |

边界：AOP 不适合隐藏复杂业务逻辑，只适合横切关注点；否则会让代码执行路径不透明。

### 6.2 XXL-JOB 分片扫表：大任务拆小

> 分库分表后超时关单扫表量大，我用 XXL-JOB 分片广播让多实例并行处理，并用游标分页避免深分页；每个分片只处理自己的数据范围，任务耗时随实例数近似线性下降。

| 做法 | 目的 |
|------|------|
| 分片广播 | 多实例并行 |
| 游标分页 | 避免 offset 深分页 |
| 状态 CAS | 防重复关单 |
| 失败重试 | 保证最终处理 |

## 七、STAR 表达模板：一句话先给结论，再展开取舍

```
S：什么业务背景，规模是多少？
T：你负责解决什么目标？
A：候选方案有哪些，为什么选当前方案，具体怎么落地？
R：结果如何量化，副作用和后续优化是什么？
```

可直接套用：

> 我负责优化 X 链路，当时规模是 A，问题是 B。我们评估了方案 1/2/3，最终选方案 2，因为它在当前约束下能平衡 C 和 D。我的实现包括 E/F/G，上线后指标从 M 降到 N。这个方案的代价是 P，后续可以通过 Q 优化。

反例：

| 低质量表达 | 问题 |
|------------|------|
| 我用了 Redis 做缓存 | 没有背景、没有指标、没有一致性代价 |
| 我用了线程池提升性能 | 没有线程池参数、下游限流、异常处理 |
| 我用 MQ 保证一致性 | MQ 不是一致性，需要幂等和补偿 |

## 八、面试追问

### 1. 如何证明这是难点而不是普通开发？

给规模、故障代价和取舍。比如“50 个外部 I/O 串行导致 10s 超时”比“用了 CompletableFuture”更像难点；“支付和关单并发导致资损风险”比“加了乐观锁”更有说服力。

---

### 2. 面试官问“为什么不用更简单方案”怎么办？

先承认简单方案的优点，再说明它在哪个约束下不满足。例如 parallelStream 写法简单，但线程池不可控，会影响其他 ForkJoinPool 任务；所以选自定义线程池。

---

### 3. 如何讲方案缺陷？

缺陷要讲边界和补偿，不要自曝无方案。例如 BitSet 在 userId 稀疏时浪费内存，所以按桶拆分；本地缓存有多实例不一致，所以 TTL 短并配合消息失效。

---

### 4. 亮点没有量化指标怎么办？

优先补压测、日志统计或上线前后对比。没有数字时至少给数量级和观测方式；完全没有指标的亮点，面试说服力很弱。
