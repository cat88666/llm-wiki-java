## 一、亮点挖掘维度

| 维度 | 典型场景 | 结论（选型方向） |
|------|---------|----------------|
| 性能优化 | 串行接口改并发、多级缓存 | CompletableFuture、Caffeine + Redis |
| 并发与幂等 | 状态竞争、重复提交 | 状态机 + 乐观锁、Token/幂等 Key |
| 可靠性/一致性 | 跨服务数据一致 | 本地消息表、最终一致 |
| 数据结构应用 | 空间/时间优化 | BitSet 预约、ZSet 排行榜 |
| 工程可维护性 | 减少重复代码 | 自定义注解 + AOP |

**STAR 框架的三个必要条件**：
1. 有具体数字（50笔→1s、20万→1千、内存降400倍）
2. 说明为什么选这个方案而不是其他（选型对比）
3. 提到方案的缺陷和改进方向（成熟度信号）

## 二、性能优化类亮点

### CompletableFuture 并发编排

**背景**：用户主动还款时并行暂停 50 笔在途扣款单，串行调用外部资金系统耗时 10s+。

**技术选型对比**：

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| 串行调用 | 实现简单 | 50 笔串行 = 10s+ | 不可接受 |
| parallelStream | 写法简单 | 共用 ForkJoinPool，无法控制线程数 | 不推荐 |
| CompletableFuture + 自定义线程池 | 可控线程数，I/O 密集型优化 | 代码复杂度略增 | **选此方案** |

```java
// I/O 密集型线程池：核心线程数 = CPU × 2
ExecutorService executor = Executors.newFixedThreadPool(Runtime.getRuntime().availableProcessors() * 2);

List<CompletableFuture<Void>> futures = orders.stream()
    .map(order -> CompletableFuture.runAsync(
        () -> paymentService.pauseOrder(order.getId()), executor))
    .collect(Collectors.toList());

// 任一失败则抛异常，不推进主流程
CompletableFuture.allOf(futures.toArray(new CompletableFuture[0])).join();
```

**结果**：50 笔暂停耗时从 10s → 1s，降低 90%。

**追问应对**：
- 异常处理？→ `exceptionally` 捕获单任务异常，`handle` 统一聚合结果
- 为什么不用 parallelStream？→ 无法控制线程池，CPU 密集型任务会被饿死

## 三、并发与幂等类亮点

### 状态机 + 乐观锁解决并发关单

**背景**：订单 10:00 创建，10:30 超时关单任务和支付成功回调同时到达。

**核心设计**：

```java
// 状态机：CONFIRM → PAID 或 CLOSED 互斥终态，业务层拦截非法流转
// 乐观锁：CAS 保证只有一个线程写成功
int rows = orderMapper.update(null, new LambdaUpdateWrapper<Order>()
    .eq(Order::getId, orderId)
    .eq(Order::getStatus, OrderStatus.CONFIRM)  // 旧状态
    .eq(Order::getLockVersion, order.getLockVersion())
    .set(Order::getStatus, newStatus)
    .set(Order::getLockVersion, order.getLockVersion() + 1));

if (rows == 0) {
    // CAS 失败：另一线程已完成状态转换，查询最新状态
    Order latest = orderMapper.selectById(orderId);
    if (latest.getStatus() == OrderStatus.CLOSED) {
        // 关单先成功，支付回调需触发退款
        refundService.refundIfPaid(orderId);
    }
}
```

**结论**：乐观锁比悲观锁更适合低频并发（支付/关单冲突概率低），不拉长事务持有时间。退款路径必须有明确触发条件（不能靠人工处理）。

### Redis BitSet 商品预约

**背景**：热门商品需要预约才能购买，需记录哪些用户预约过。

```java
// Key = reserve:{商品id}，Bit 位 = userId
String key = "reserve:" + itemId;
redisTemplate.opsForValue().setBit(key, userId, true);   // 预约
boolean reserved = redisTemplate.opsForValue().getBit(key, userId);  // 查询
Long count = redisTemplate.execute((RedisCallback<Long>) conn ->
    conn.bitCount(key.getBytes()));  // 统计总预约数
```

| 方案 | 100万用户内存 | 查询复杂度 | 结论 |
|------|------------|----------|------|
| Redis Set | ~50 MB | O(1) | 内存占用大 |
| **Redis BitSet** | ~125 KB | O(1) | 内存降 400 倍，首选 |
| Bloom Filter | ~1.2 MB | O(1) | 有误判率且不可删除，不适合 |

**追问**：userId 很大（如 10 亿级）怎么办？→ BitSet 预分配到 10 亿位约 120MB，可改用 Hash 分桶（`userId / bucket_size` → 子 key）。

### Redis ZSet 秒级排行榜

```java
// 新增/更新分数
redisTemplate.opsForZSet().add("rank:" + gameId, userId, score);
// 加分（增量）
redisTemplate.opsForZSet().incrementScore("rank:" + gameId, userId, delta);
// 取 Top10
Set<ZSetOperations.TypedTuple<String>> top10 = redisTemplate.opsForZSet()
    .reverseRangeWithScores("rank:" + gameId, 0, 9);
```

**同分排序技巧**：`score = 积分 × 10000 + (10000 - 注册时间戳末4位)`，用低位区分同积分用户，注册早的排在前面。

## 四、可靠性与一致性类亮点

### 本地消息表保证最终一致

**背景**：订单创建后需通知积分系统，MQ 发送可能失败导致不一致。

```java
// 关键：写订单 + 写本地消息表必须在同一事务内
@Transactional
public void createOrder(Order order) {
    orderMapper.insert(order);
    // 同一事务写本地消息表
    LocalMessage msg = new LocalMessage(order.getId(), "ORDER_CREATED", "INIT");
    localMessageMapper.insert(msg);
}

// 事务提交后（在 @Transactional 外）异步发 MQ
@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
public void afterOrderCreated(OrderCreatedEvent event) {
    try {
        mqTemplate.send("order.created", event);
        localMessageMapper.updateStatus(event.getOrderId(), "SENT");
    } catch (Exception e) {
        // 发送失败由定时任务补偿
        log.warn("MQ send failed, will retry", e);
    }
}
```

```sql
-- 定时补偿：扫描 INIT 状态超过 N 分钟的消息
SELECT * FROM local_message WHERE state = 'INIT' AND gmt_create < NOW() - INTERVAL 5 MINUTE;
```

**追问**：为什么不用 MQ 事务消息？→ RabbitMQ 不支持事务消息；本地消息表更通用，实现复杂度可控。

### TTL 解决线程池上下文丢失

```java
// 问题：子线程无法获取父线程的 ThreadLocal（traceId、userId）
// 解决：TransmittableThreadLocal + TtlExecutorService 包装

private static final TransmittableThreadLocal<String> traceIdHolder =
    new TransmittableThreadLocal<>();

// 包装线程池（自动 capture/replay 父线程上下文）
ExecutorService ttlExecutor = TtlExecutors.getTtlExecutorService(
    Executors.newFixedThreadPool(10));
```

| 方案 | 父子线程传递 | 线程池复用 | 结论 |
|------|------------|---------|------|
| ThreadLocal | 不支持 | N/A | 只在同线程有效 |
| InheritableThreadLocal | 仅 new Thread | 复用时串数据 | 线程池场景错误 |
| **TransmittableThreadLocal** | 支持 | 正确隔离 | **线程池场景必选** |

## 五、工程效率类亮点

### 自定义注解 + AOP

**背景**：多个 Facade 方法需要统一做参数校验、日志记录、异常转换，手写重复逻辑。

```java
@Aspect
@Component
public class FacadeAspect {
    @Around("@annotation(facade)")
    public Object around(ProceedingJoinPoint pjp, Facade facade) throws Throwable {
        // 1. 参数校验（HibernateValidator）
        Object[] args = pjp.getArgs();
        for (Object arg : args) {
            Set<ConstraintViolation<Object>> violations = validator.validate(arg);
            if (!violations.isEmpty()) {
                return BaseResponse.fail(violations.iterator().next().getMessage());
            }
        }
        // 2. 记录入参 + 耗时
        long start = System.currentTimeMillis();
        log.info("Facade入参: method={}, args={}", pjp.getSignature(), Arrays.toString(args));
        try {
            Object result = pjp.proceed();
            log.info("Facade出参: method={}, cost={}ms, result={}", 
                pjp.getSignature(), System.currentTimeMillis() - start, result);
            return result;
        } catch (Exception e) {
            // 3. 异常统一转换
            log.error("Facade异常: method={}", pjp.getSignature(), e);
            return BaseResponse.fail("系统内部错误");
        }
    }
}
```

**结果**：新增接口只需加 `@Facade` 注解，自动覆盖参数校验 + 日志 + 异常处理 + 耗时监控，删除数百行重复代码。

### Spring Event 同步转异步

**背景**：借据生成后触发案件合并，同步耦合重；定时批量扫表 20 万条打满数据库连接池。

```java
// 生成借据时发布事件（try-catch 包裹，失败不影响主流程）
@Transactional
public void createLoan(Loan loan) {
    loanMapper.insert(loan);
    try {
        applicationEventPublisher.publishEvent(new CaseStartFinishEvent(loan.getId()));
    } catch (Exception e) {
        log.warn("Event publish failed, will be compensated by scheduled task", e);
    }
}

// 监听器异步处理（@Async + 分布式锁防并发）
@EventListener
@Async("caseHandlerExecutor")
public void onCaseStartFinish(CaseStartFinishEvent event) {
    RLock lock = redisson.getLock("case:merge:" + event.getLoanId());
    if (lock.tryLock(0, 30, TimeUnit.SECONDS)) {
        try {
            caseMergeService.merge(event.getLoanId());
        } finally {
            lock.unlock();
        }
    }
}
```

**结果**：定时任务只做补偿（扫漏网之鱼），日常处理量从 20 万 → 1 千条，连接池压力消除。

## 六、大数据处理类亮点

### XXL-JOB 分片 + 分库分表扫表

**背景**：分库分表后需扫描所有订单做超时关单，单机扫表慢且量大。

```java
// 分片广播：每个实例获得 shardIndex / shardTotal
@XxlJob("closeExpiredOrders")
public void closeExpiredOrders() {
    ShardingUtil.ShardingVO sharding = ShardingUtil.getShardingVo();
    int shardIndex = sharding.getIndex();
    int shardTotal = sharding.getTotal();
    
    // 按 userId 逆序余数分片（逆序字段支持前缀 LIKE 走索引）
    List<Order> orders = orderMapper.findExpiredBySharding(
        shardIndex, shardTotal, LocalDateTime.now());
    
    // 生产者消费者：扫表线程 → BlockingQueue → 消费者线程池
    BlockingQueue<Order> queue = new LinkedBlockingQueue<>(1000);
    executor.submit(() -> orders.forEach(o -> { try { queue.put(o); } catch (Exception e) {} }));
    for (int i = 0; i < CONSUMER_THREADS; i++) {
        executor.submit(() -> { while (true) {
            Order o = queue.poll(100, TimeUnit.MILLISECONDS);
            if (o != null) closeOrder(o);
        }});
    }
}
```

**结果**：5 台实例并行，扫表速度提升 5 倍，任务堆积消除。

**追问**：为什么用逆序 userId 分片？→ 按 userId 尾号分片需要 `LIKE "%xx"`（后缀 LIKE 不能走 B+树），冗余逆序列后用 `LIKE "xx%"`（前缀 LIKE 走索引）。

## 七、STAR 表达框架

```
S（Situation）：我负责的系统遇到了什么问题？（量化规模：50万DAU、1000QPS）
T（Task）：   我的具体职责是什么？（不是"团队做了"，是"我负责做了"）
A（Action）： 我做了哪些技术选型和实现？（说为什么选这个方案）
R（Result）： 最终产生了多少可量化的 impact？（数字，数字，数字）
```

**好回答的三个信号**：
1. 有具体数字（10s→1s、20万→1千、内存降400倍）
2. 说明选型对比（为什么不用方案 B）
3. 提到方案的缺陷（BitSet 大 userId 场景的内存问题）

**亮点选择原则**：

| 优先级 | 场景 | 说明 |
|--------|------|------|
| **高** | 解决了生产真实问题 | 有明确的 before/after |
| **高** | 方案有选型对比 | 展示深度思考 |
| **中** | 用了常见技术 | 但必须能说清楚为什么 |
| **低** | 只是"用了 Redis" | 没有场景约束和代价分析 |

## 八、与其他概念的关系

- CompletableFuture 并发编排的线程池选择依赖 [[概念-并发]]：I/O 密集型和 CPU 密集型线程数公式不同
- XXL-JOB 分片扫表依赖 [[概念-分布式任务]]：分片广播路由策略和 ShardingUtil 用法
- Token 防重复提交和滑动窗口限流属于 [[设计-接口设计与防护]] 的防护层实现
- 本地消息表是 [[概念-幂等设计]] 的工程实现：写消息表和业务操作同一事务，确保原子性
