# 项目难点表达

> 把项目经历转化为面试竞争力：用"背景→技术选型→你做了什么→量化结果"的 STAR 框架，将通用业务场景包装为有深度的技术亮点。

## 亮点挖掘维度

面试中可用的项目亮点来自以下五个方向：

| 维度 | 典型例子 |
|------|--------|
| 性能优化 | 多级缓存、CompletableFuture 并发编排、批量插入 |
| 并发与幂等 | 分布式锁、状态机+乐观锁、Token 防重 |
| 可靠性/一致性 | 本地消息表、最终一致、状态机终态控制 |
| 数据结构创新应用 | BitSet 预约、ZSet 排行榜、雪花算法 ID |
| 代码可维护性 | 自定义注解+AOP、Spring Event 解耦 |

## 高频亮点模板库

### 1. CompletableFuture 并发编排提升接口性能

**背景**：用户主动还款时需并行暂停多笔在途扣款单，串行调用外部资金系统耗时 10s+。

**技术选型**：CompletableFuture（Java 8）+ 自定义线程池（避免 ForkJoinPool 陷阱）。

**你做了什么**：
- 将 50 笔订单的暂停请求封装为 CompletableFuture 任务列表
- 用 `allOf(...).join()` 等待全部完成，任一失败则不推进主流程
- 线程池独立配置（I/O 密集型，核心线程数 = CPU × 2）

**结果**：50 笔暂停耗时从 **10s → 1s**，降低 90%。

**追问应对**：
- 为什么不用 parallelStream？→ 无法控制线程池，共用 ForkJoinPool 会饿死 CPU 密集型任务
- 异常怎么处理？→ `exceptionally` / `handle` 捕获单任务异常，统一聚合结果

### 2. 基于状态机 + 乐观锁解决支付/关单并发

**背景**：订单在 10:00 下单、30 分钟超时，10:30 支付成功回调与自动关单定时任务并发到达。

**你做了什么**：
1. **状态机**：CONFIRM → PAID 或 CLOSED 是互斥终态，状态机在业务层面拦截非法流转
2. **乐观锁**：`UPDATE order SET status=?, lock_version=? WHERE id=? AND lock_version=?`，CAS 保证只有一个线程写成功
3. **逆向流程**：若关单先成功、支付后失败 → 检测 `ORDER_ALREADY_CLOSED` → 触发原路退款

**结果**：消灭重复支付，退款路径有明确的触发条件，业务闭环完整。

**追问应对**：
- 为什么不用悲观锁？→ 支付低频并发，乐观锁重试代价低；悲观锁会拉长事务持有时间
- 退款怎么保证幂等？→ 退款单加唯一约束，同一支付单只能有一条退款记录

### 3. Redis BitSet 实现高效商品预约

**背景**：热门商品（类似茅台）需要预约才能购买，需记录哪些用户预约过，要求查询快、存储省。

**技术选型**：Redis BitSet（`SETBIT`/`GETBIT`）+ MySQL 持久化兜底。

**你做了什么**：
- Key = `reserve:{商品id}`，Bit 位 = userId（整数映射）
- 预约：`SETBIT key userId 1`；查询：`GETBIT key userId`；统计：`BITCOUNT key`
- 100 万用户预约状态 ≈ 125 KB（vs 普通 Set 约 50 MB）

**结果**：内存降低 **400 倍**，查询 O(1)，Redis 宕机时 fallback 到 MySQL 预约表。

**追问应对**：
- userId 很大（如 10 亿级）怎么办？→ BitSet 会预分配 offset 到 10 亿位，约 120 MB，需评估内存；可改用 Hash 分桶
- 为什么不用 Bloom Filter？→ BF 有误判率且不可删除；BitSet 精确且支持删除

### 4. Redis ZSet 实现秒级排行榜

**背景**：游戏平台需要分钟/秒级实时榜单，数据库 `ORDER BY` 性能差。

**你做了什么**：
- Key = `rank:{gameId}`，Member = userId，Score = 积分
- `ZADD`（新增）→ `ZINCRBY`（加分）→ `ZREVRANGEWITHSCORES`（取 TopN）
- 积分相同按注册时间排序：`score = 积分 + (1 - 注册时间戳/1e13)`，利用小数部分区分

**结果**：O(log N) 插入，O(log N + K) 查询 TopK，单 Redis 节点可支撑百万级用户实时榜。

### 5. 本地消息表保证最终一致性

**背景**：订单创建成功后需通知积分系统加积分，MQ 发送可能失败导致不一致。

**你做了什么**：
1. 在**同一本地事务**中写订单 + 写本地消息表（state=INIT）
2. 事务提交后（在 `@Transactional` 外）同步发 MQ，成功则更新消息 state=SENT
3. 定时任务扫 INIT 状态超过 N 分钟的消息补偿重发，下游消费幂等

**核心设计**：上游不回滚（消息写入成功），下游指数退避重试直至成功。

```sql
CREATE TABLE local_message (
  id          BIGINT PRIMARY KEY,
  message_type VARCHAR(64),
  biz_type    VARCHAR(64),
  identifier  VARCHAR(128) UNIQUE,  -- 幂等键
  content     TEXT,
  state       VARCHAR(16),          -- INIT / SENT / CONSUMED
  retry_count INT DEFAULT 0,
  gmt_create  DATETIME,
  gmt_modified DATETIME
);
```

**追问**：为什么不用 MQ 事务消息？→ 并非所有 MQ 都支持（RabbitMQ 不支持），本地消息表更通用

### 6. XXL-JOB 分片 + 生产者消费者扫分库分表

**背景**：分库分表后需扫描所有订单做超时关单，单机扫表慢且量大。

**你做了什么**：
1. XXL-JOB 路由策略选**分片广播**，每个实例获得 `shardIndex` / `shardTotal`
2. 按 `userId 逆序 % shardTotal == shardIndex` 路由数据（逆序字段支持 `LIKE "xx%"` 走索引）
3. 每个实例内部用 生产者线程扫表 → BlockingQueue → 消费者线程池 并行处理

**结果**：5 台实例并行，扫表速度提升 **5 倍**，任务堆积消除。

**追问**：LIKE "userId%" 走索引，但 userId 尾号分片为什么不能直接用 LIKE "%xx"？→ 前缀 LIKE 能走 B+树，后缀 LIKE 需全表扫描。解法：冗余一个逆序 userId 列

### 7. 自定义注解 + AOP 减少冗余代码

**背景**：多个 Facade 方法需要统一做参数校验、日志记录、异常转换，手写重复逻辑多。

**你做了什么**：
- `@OpLog(bizType="ORDER", spel="#order.id")`：切面用 SpEL 动态取对象唯一标识记日志
- `@Facade`：切面统一做 HibernateValidator 参数校验，校验失败直接返回 BaseResponse
- 一个 Facade 切面同时覆盖：入参出参日志 + 参数校验 + 异常统一转换 + 耗时监控

**结果**：删除数百行重复代码，新增接口只需加注解，不遗漏任何横切逻辑。

### 8. 滑动窗口限流 + API 密钥防恶意调用

**背景**：对外 API 需要防盗用和防流量暴增。

**技术方案**：
- **认证**：为每个调用方生成唯一 API Key，HTTP Header 携带，Filter 层 Redis 校验
- **限流**：Redis ZSet 滑动窗口，`ZADD` + `ZREMRANGEBYSCORE` + `ZCARD` 原子执行（Lua 脚本）

```
// Lua 伪代码：5 分钟内最多 100 次
ZADD key now now
ZREMRANGEBYSCORE key 0 (now - 5min)
count = ZCARD key
if count > 100: reject
```

### 9. TTL（TransmittableThreadLocal）解决线程池上下文丢失

**背景**：请求进来生成 traceId 存入 ThreadLocal，但提交到线程池后子线程无法获取。

**技术选型**：阿里开源 `TransmittableThreadLocal`（TTL），继承自 ThreadLocal，支持线程池场景下的父子线程值传递。

**你做了什么**：
- 将 ThreadLocal 替换为 `TransmittableThreadLocal`
- 用 `TtlExecutors.getTtlExecutorService(executor)` 包装线程池
- 提交任务时 TTL 自动 capture 父线程上下文，任务执行完自动 replay/restore

**对比**：
| | ThreadLocal | InheritableThreadLocal | TransmittableThreadLocal |
|---|---|---|---|
| 父子线程传递 | 不支持 | 仅手动 new Thread | 线程池也支持 |
| 线程池复用隔离 | N/A | 会串数据 | 正确隔离 |

### 10. Spring Event 同步转异步解耦定时任务

**背景**：借据生成时需触发案件合并，若同步调用则主流程耦合重；若定时批量扫表则数据库连接池被打满（20万条/次 → 改造后 1000 条/次）。

**你做了什么**：
- 借据生成 publish `CaseStartFinishEvent`（try-catch 包裹，失败不影响主流程）
- 监听器 `@EventListener` + `@Async` 异步处理案件合并，加分布式锁防并发重复
- 定时任务只做**补偿**（扫漏网之鱼），日常处理量从 20 万 → 1 千条

## STAR 表达框架

```
S（Situation）：我负责的模块/系统遇到了什么问题？
T（Task）：我的具体职责是什么？
A（Action）：我做了哪些技术选型和实现？
R（Result）：最终产生了多少可量化的 impact？
```

**好回答的特征**：
1. 有具体的数字（50笔→1s、20万→1千、内存降400倍）
2. 说明为什么选这个方案，而不是其他方案（选型对比）
3. 提到方案的缺陷和改进方向（成熟度信号）
