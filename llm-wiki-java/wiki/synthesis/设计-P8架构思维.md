## 一、边界追问：先承认代价，再给演进方案

面试官常见质疑：

> “你做了很多反查上游的逻辑，这种高耦合好吗？”
>
> “为什么不是下游提供一个原子接口，而是你们自己拼两步？”

回答矩阵：

| 步骤 | 内容 | 结论 |
|------|------|------|
| 承认问题 | 当前方案确实有耦合 | 不硬辩 |
| 找根因 | 下游接口不够原子，暴露内部状态 | 边界设计问题 |
| 给理想方案 | 下游提供 `解冻并转账` 幂等原子接口 | 领域职责回归下游 |
| 给过渡方案 | ACL 防腐层隔离反查和补偿 | 控制耦合扩散 |

P8 口径：短期方案可以不完美，但要讲清楚为什么当时这么做、风险在哪里、下一步如何演进。

## 二、自研组件：五问框架决定能不能防住追问

| 问题 | 必须回答 |
|------|----------|
| 为什么要自研？ | 现有组件哪个约束不满足 |
| 对比过什么？ | 至少 2 个成熟方案及放弃原因 |
| 自研优势是什么？ | 轻量、延迟、可控、适配内部场景 |
| 自研缺陷是什么？ | 可靠性、生态、运维、边界能力 |
| 如果业务扩大怎么办？ | 演进到成熟组件或补齐哪些能力 |

Redis Stream 追问边界：

| 能力 | Redis Stream | Kafka | 结论 |
|------|--------------|-------|------|
| 消费组 | 支持 XREADGROUP/XACK | 成熟 Rebalance | Stream 轻量但扩缩容弱 |
| 消息堆积 | 内存/磁盘受 Redis 限制 | 分区日志适合大堆积 | 大规模堆积选 Kafka |
| 可靠性 | 依赖 Redis 持久化 | 副本和日志模型成熟 | 核心交易消息慎用 Stream |

## 三、库存一致性：Redis 快路径不是最终真相

库存三态：

```
总库存 = 可售库存 + 冻结库存 + 已扣减库存
```

| 阶段 | 状态变化 | 关键风险 |
|------|----------|----------|
| Try | 可售 → 冻结 | 防超卖，必须幂等 |
| Confirm | 冻结 → 已扣减 | 成功后不能再当普通 Cancel |
| Cancel | 冻结 → 可售 | 空回滚/悬挂要处理 |

Redis 与 DB 不一致的兜底：

```
Redis Lua 扣减 → 创建订单 → 写库存流水 → DB 扣减
              ↘ 失败补偿：回补 Redis + 失败流水

定时对账：DB 库存流水聚合值 vs Redis 当前值
  → Redis 不一致时，以 DB 流水修正
```

P8 结论：Redis 是高并发快路径，DB 流水才是库存最终事实。只讲 Lua 防超卖还不够，必须讲少卖、回补失败和旁路对账。

## 四、分布式组件边界：雪花、MQ、TCC 的极端情况

雪花算法：

| 追问 | 边界回答 |
|------|----------|
| 没时钟回拨会重复吗 | 实现正确时不会；同毫秒序列耗尽会等待下一毫秒，bug 才会重复 |
| workerId 不够怎么办 | 调整 bit 分配或用 ZK/Redis/Leaf 动态分配 |
| 时钟回拨怎么办 | 等待追平、拒绝服务、换 workerId，强一致场景宁可不可用 |
| Redis 分配 workerId 有风险吗 | Redis 重启和持久化策略不当会重复，必须持久化和租约校验 |

MQ 刷盘：

| 目标 | 配置 | 代价 |
|------|------|------|
| 高吞吐，允许少量丢 | 异步刷盘 | Broker 崩溃可能丢内存消息 |
| 交易消息尽量不丢 | `SYNC_FLUSH` + `SYNC_MASTER` | 延迟上升，吞吐下降 |
| 消费不丢 | 手动 ACK + 重试 + DLQ | 消费端必须幂等 |

TCC 陷阱：

```
Cancel 到达时：
  ├─ 无 Try 记录 → 空回滚，写 CANCELLED 防悬挂
  ├─ TRIED → 正常 Cancel，冻结回可售
  └─ CONFIRMED → 不能普通 Cancel，要做反向补偿
```

## 五、系统扩容追问：QPS 突增先判断流量性质

```
QPS 突增
  → 先判断正常增长还是攻击/爬虫
      ├─ 攻击：限流、验证码、封禁、黑名单
      └─ 正常：扩容和削峰
          → 接入层：CDN/Nginx/Sentinel
          → 应用层：无状态扩容、异步化、本地缓存
          → 存储层：读写分离、分库分表、Redis Cluster
          → 兜底：降级非核心能力，保交易主链路
```

10 万 OPS 会员系统：

| 链路 | 方案 | 结论 |
|------|------|------|
| 读会员信息 | Redis Cluster + Caffeine | 读多写少首选 |
| 写等级变更 | DB 权威 + MQ/Canal 刷缓存 | 保证最终一致 |
| 热点用户 | 本地缓存短 TTL | 避免 Redis 热 key |
| 数据分布 | userId hash | 避免单分片热点 |

## 六、技术选型追问：ShardingJDBC、MyCat、Seata

| 维度 | ShardingJDBC | MyCat | 推荐 |
|------|--------------|-------|------|
| 接入 | 应用内 JDBC | 独立代理 | Java 新项目选 ShardingJDBC |
| 性能 | 无额外网络跳转 | 多一次代理转发 | 延迟敏感选 ShardingJDBC |
| 运维 | 随应用发布 | 独立运维 | 小团队选 ShardingJDBC |
| 多语言 | 不友好 | 统一代理 | 多语言遗留系统选 MyCat/Proxy |

自研 TCC vs Seata：

| 场景 | 推荐 |
|------|------|
| 少量核心交易、强定制 | 自研 TCC，状态可控 |
| 通用分布式事务、多团队 | Seata，减少重复建设 |
| 高并发库存扣减 | 优先本地事务 + 可靠消息，慎用全局事务 |

## 七、线上疑难问题：free 充足但 FullGC

| 可能原因 | 排查方式 | 结论 |
|----------|----------|------|
| `-Xmx` 太小 | `jinfo -flags <pid>` | OS free 不等于 JVM 堆够 |
| 元空间扩容 | GC 日志 `Metadata GC Threshold` | 动态类生成/类加载泄漏 |
| 堆外内存 | NMT / DirectBuffer 监控 | Netty/ByteBuffer 常见 |
| 软引用清理 | heap dump 看 SoftReference | 缓存策略不当 |
| G1 Humongous | GC 日志看 humongous region | 大对象分配导致 FullGC |

排查顺序：

```
jstat -gcutil 确认频率和代际
  → GC 日志确认原因
  → jmap/jcmd 看堆和类加载
  → NMT 看堆外
  → MAT/Arthas 定位对象来源
```

## 八、面试追问

### 1. “这是不是过度设计？”怎么答？

先给当时的规模、SLA 和故障代价，再说明复杂度匹配问题复杂度。如果确实现在看偏重，要说出可简化方案和演进路径，而不是硬说所有设计都正确。

---

### 2. 异步刷盘能不能保证消息不丢？

不能。异步刷盘时 Broker ACK 后消息可能还在内存，Broker 崩溃会丢。交易消息要同步刷盘和主从同步复制，并且生产者处理发送失败，消费者保证幂等。

---

### 3. Confirm 成功后又来了 Cancel 怎么办？

不能按普通 Cancel 解冻。要根据 TCC 事务表判断状态：已 Confirm 的只能做反向补偿，例如加库存或退款；未 Try 的 Cancel 是空回滚；已 Try 未 Confirm 才正常解冻。

---

### 4. ShardingJDBC 会成为瓶颈吗？

可能。它在应用内做 SQL 解析和路由，复杂 SQL、跨库 JOIN、分片键缺失都会放大开销。正确做法是限制 SQL 形态、按分片键访问、压测 P99；极端高吞吐或多语言接入再考虑 Proxy。
