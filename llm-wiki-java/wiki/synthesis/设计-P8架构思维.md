## 一、系统边界追问

**面试官质疑话术**：
> "你做了很多反查上游的逻辑来保证一致性，这种高耦合方案真的好吗？"
> "为什么不让下游提供一个原子性接口，而是让你们自己处理两步操作？"

**回答框架**：

| 步骤 | 内容 |
|------|------|
| 识别问题 | 上游做了下游本该做的事，说明领域边界不清 |
| 分析根因 | 下游接口设计不够原子（两步 → 应该是一步原子接口） |
| 更好方案 | 推动下游提供 `解冻并转账` 原子幂等接口 |
| 过渡方案 | 改造有成本时，在基础设施层加防腐层（ACL）隔离 |

**标准话术**：
> "这个方案确实有高耦合问题，根因是下游接口设计不够原子。更好的方案是推动下游将解冻+转账合并为一个幂等接口，上游只需调一次，也符合领域边界——下游内部状态不应暴露给上游。短期无法推动时，可以在基础设施层加防腐层做隔离，但这只是过渡方案。"

## 二、造轮子防御话术

**场景**：每次说到"我们自研了 XX"，面试官必问。

**五问框架**（必须能全部回答）：

| 问题 | 参考答法 |
|------|---------|
| 为什么造？ | 现有轮子哪里不满足（依赖太重/功能太多/延迟不满足）|
| 造之前评估过哪些方案？ | 列举 2-3 个竞品及放弃原因 |
| 你造的优点是什么？ | 轻量/特定场景性能更好/运维简单 |
| 你造的缺点是什么？ | 缺少 XX 特性，生产事故 XX 案例 |
| 如果要支持 YY 特性，你怎么做？ | 展示你真的理解了底层 |

**Redis Stream 高频追问点**：

```
底层结构：小数据用 Listpack，大数据用 linked list of entries
消费组：XREADGROUP + XACK 标记已消费，XPENDING 查看未确认消息
缺少：无重平衡机制（vs Kafka Consumer Group Rebalance），消费者离线不会自动重分配
```

**结论**：Redis Stream 适合轻量级消息传递，不适合需要高可靠、消费者动态扩缩容的场景。

## 三、库存一致性深追问

### 库存三态模型

```
库存总量 = 可售库存 + 冻结库存 + 已扣减库存
```

| TCC 阶段 | 库存状态变化 | 结论 |
|---------|------------|------|
| Try | 可售 → 冻结（占位，未扣减） | 防止超卖的关键 |
| Confirm | 冻结 → 已扣减（真正扣减） | 最终扣减在此发生 |
| Cancel | 冻结 → 可售（归还） | 超时/失败时必须回滚 |

### Redis 与 DB 库存一致性

**追问**：Redis 扣减成功，但订单创建失败，库存怎么处理？

**旁路验证方案**：

```
正常流程：Redis Lua 原子扣减 → 创建订单 → DB 异步扣减 → 写库存流水

旁路验证（兜底）：
  每次扣减/回补都写库存流水（event_type + 幂等号）
  定时任务：对比 Redis 库存 vs DB 流水聚合值
  不一致时：以 DB 流水为准，修正 Redis（DB 是最终权威）
```

**Confirm 之后 Cancel 的处理（高频陷阱）**：

```
TCC 事务表记录当前阶段：TRIED / CONFIRMED / CANCELLED
Cancel 前查事务表：
  → 状态为 CONFIRMED → 执行反向补偿（不是空操作！）
    反向补偿 ≠ 回滚：向库存服务发"加库存"而非"解冻"
  → 状态为 TRIED → 正常 Cancel（解冻回可售）
  → 无记录 → 空回滚（写 CANCELLED 状态，防悬挂）
```

## 四、分布式组件边界追问

### 雪花算法边界

| 追问 | 边界回答 |
|------|---------|
| 没有时钟回拨也会重复吗？ | **会**：同一毫秒序列号耗尽（4096/ms），超过 4096 并发会等下一毫秒；实现有 bug 时可能重复 |
| workerId 上限是多少？ | 默认 5 bit = 最大 31，超过 32 台机器需要自定义分配策略 |
| 超过 32 台机器怎么办？ | ①重分配 bit 布局（减序列号位、增机器位）；②用 ZK/Redis 动态分配 workerId（Leaf 做法）|
| 时钟回拨怎么解决？ | ①抛异常（强一致，服务短暂不可用）；②等待追上旧时间；③向 ZK 申请新 workerId |
| Redis 自增 workerId 有问题吗？ | Redis 重启从 0 开始，可能和已有机器码冲突 → 需要 Redis 持久化或加起始偏移 |

### MQ 异步刷盘保证

**追问**：RocketMQ 用异步刷盘，能保证消息不丢吗？

| 层级 | 保证手段 | 异步刷盘风险 |
|------|---------|------------|
| 生产者 → Broker | acks 确认 | 消息在 Broker 内存但未落盘，崩溃后丢失 |
| Broker → 磁盘 | 同步/异步刷盘 | 异步刷盘 Broker 崩溃可能丢缓冲区消息 |
| Broker → Consumer | 手动 ACK | 消费失败会重试，需消费幂等 |

**结论**：异步刷盘**不能**保证 100% 不丢。要求不丢必须：`flushDiskType=SYNC_FLUSH`（同步刷盘）+ `brokerRole=SYNC_MASTER`（主从同步复制）。

## 五、设计能力追问

### QPS 突增 N 倍应对框架

```
1. 先判断来源：正常增长 vs 攻击/爬虫？
   → 异常：限流 + 封禁；正常：扩容

2. 接入层：CDN 静态资源缓存、Nginx 限流（limit_req_zone）、Sentinel 熔断降级

3. 应用层：水平扩容（无状态）、异步化（下单立即返回，处理异步）、
          本地 Caffeine 二级缓存（减少 Redis 穿透）

4. 存储层：读写分离（扩展读副本）、分库分表（TPS 超单库上限）、Redis Cluster 扩容

5. 兜底：限流保最核心链路可用，降级非核心功能（推荐/积分）
```

### 10 万 OPS 会员系统设计

```
特征：读多写少（查会员信息频繁，变更低频）

设计决策：
  ├─ 读请求：Redis Cluster 抗（10 万 OPS × 1ms = 单机 Redis 轻松支撑）
  ├─ 热点防护：本地 Caffeine 二级缓存（减少 Redis 单 key 压力）
  ├─ 写请求：会员等级变更直接 DB + 异步 MQ 更新缓存（Canal 监听）
  ├─ 数据分布：按 userId hash 均匀分桶，避免热点
  └─ 扩容边界：Redis Cluster（3主3从）+ 应用无状态横向扩容
```

## 六、技术选型追问

### ShardingJDBC vs MyCat

| 维度 | ShardingJDBC | MyCat | 结论 |
|------|-------------|-------|------|
| 接入方式 | JDBC 驱动，无独立进程 | 独立中间件，改连接地址 | ShardingJDBC 对 Java 应用更友好 |
| 性能 | 无额外网络跳转 | 多一次网络转发 | ShardingJDBC 延迟更低 |
| 运维 | 随应用发布 | 独立运维，有单点风险 | ShardingJDBC 运维更简单 |
| 功能 | 分库分表/读写分离/数据加密 | 功能更全（如全局表广播） | MyCat 功能更丰富 |
| 适用 | Java 应用，中小规模 | 多语言/遗留系统，大规模 | 新项目用 ShardingJDBC |

**选型话术**：

> "我们选 ShardingJDBC，因为纯 Java 项目，无需额外部署中间件，减少了网络跳转，运维随应用一起发布。代价是如果将来要支持非 Java 语言接入，需要切换到 MyCat 或改用数据库代理。"

## 七、线上问题定位追问

### free 显示充足但频繁 FullGC

| 原因 | 排查命令 |
|------|---------|
| JVM 堆上限配置太小（`-Xmx` 设置小） | `jinfo -flags <pid>` 查看 JVM 参数 |
| 元空间（Metaspace）频繁扩容 | GC 日志中查 `Metadata GC Threshold` |
| 堆外内存占用（DirectByteBuffer 等） | NMT（Native Memory Tracking）追踪 |
| 大量软引用在 FullGC 时被清理 | heap dump 分析软引用持有量 |
| G1 大对象进 Humongous Region | 调大 `G1HeapRegionSize`，减少大对象分配 |

**排查顺序**：先看 `jstat -gcutil <pid> 1s` 确认 GC 频率和原因 → 再 `jmap -heap <pid>` 看堆分配 → 针对性排查。

## 八、面试高频追问

**面试官质疑话术应对**：

| 面试官追问 | 正确应对 |
|----------|---------|
| "这不是过度设计吗？" | 描述当时业务量级和未来预期，说明复杂度匹配问题复杂度 |
| "你有没有考虑更简单的方案？" | 主动说出 Plan B，再解释为什么选了当前方案 |
| "如果重新设计，你会怎么改？" | 说出当前方案缺点，提出改进思路，展示持续反思能力 |
| "设计架构时除了技术还考虑什么？" | 团队能力/研发成本/机器资源/运维复杂度/业务演进速度 |

**问**：异步刷盘和同步刷盘怎么选？
**答**：看业务对消息丢失的容忍度。日志类消息可以用异步刷盘（性能好，但宕机丢少量）；支付/订单类消息必须同步刷盘 + 主从同步复制（`SYNC_FLUSH` + `SYNC_MASTER`），确保消息不丢。代价是写入延迟提升约 2-5 倍。

**问**：Confirm 之后的 Cancel 是什么情况？怎么处理？
**答**：极端情况：TCC 二阶段 Confirm 成功后，因某些异常触发了 Cancel（如幂等框架 bug）。此时不能做简单的"解冻归还"，而要做反向补偿——已经 Confirm 的要反向回补（向库存加库存，向账户加余额）。TCC 事务表的状态（TRIED/CONFIRMED）是判断要做哪种操作的依据。

**问**：ShardingJDBC 会不会成为性能瓶颈？
**答**：ShardingJDBC 在应用内做 SQL 解析和路由，虽然无网络跳转，但复杂 SQL（跨库 JOIN、子查询）解析成本高，单次路由增加约 0.5-2ms。对于 P99 要求 < 10ms 的场景，这个开销可以接受。极端高吞吐（TPS > 50000/节点）场景需要做压测验证，必要时换 Proxy 模式。
