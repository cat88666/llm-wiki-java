# 大厂秒杀实践

> 教科书讲"Redis 预扣 + MQ 异步落盘"；大厂实际在用什么？——阿里用自研 MySQL 补丁，小红书用内核级合并秒杀，核心思路是**在数据库层消除热点行竞争，而不是绕开数据库**。

## 为什么大厂不全靠 Redis？

| 教科书方案 | 大厂实际痛点 |
|-----------|------------|
| Redis 预扣 + MQ 落盘 | Redis 宕机 / 网络抖动 → 数据不一致、少卖 |
| 纯 Redis 扣减 | 超卖风险，事务模型弱 |
| 数据库乐观锁 | 热点行更新几百 QPS → CPU 直接打满 |

**大厂结论**：MySQL 最可靠（不存在 Redis 一致性问题），但需要在内核层解决热点行瓶颈。

## 阿里方案：Inventory Hint（热点行合并）

### 背景

淘宝、天猫、猫超、大麦等业务均基于此方案，通过阿里云 RDS 的 Inventory Hint 补丁实现。

### 核心原理

在 UPDATE 语句中加 Hint 标记：

```sql
UPDATE /*+ COMMIT_ON_SUCCESS ROLLBACK_ON_FAIL TARGET_AFFECT_ROW(1) */
  inventory SET quantity = quantity - 1
WHERE sku_id = ? AND quantity > 0;
```

| Hint | 含义 |
|------|------|
| `COMMIT_ON_SUCCESS` | 语句执行成功立即提交事务上下文 |
| `ROLLBACK_ON_FAIL` | 语句执行失败立即回滚 |
| `TARGET_AFFECT_ROW(N)` | 影响行数不是 N 则失败 |

### 三大优化点

**1. 减少行级锁等待（排队→Leader/Follower）**

- MySQL 识别热点 SQL，按主键/唯一键**分组**
- 第一条 SQL（Leader）获取行锁，同组后续（Follower）判断到 Leader 已持锁直接获取，**不等待**
- 双执行单元交替收集+提交，保持流水线满负荷

**2. 减少 B+树遍历（Row Cache）**

- Leader 遍历索引定位热点行后写入 **Row Cache**
- 同组 Follower 直接从 Row Cache 读取修改，不再遍历索引

**3. 减少事务提交次数（组提交）**

- 一组内所有并发操作完成后一次性组提交
- 提交次数从 N 次 → 1 次

### Statement Queue（补充）

阿里还提供 Statement Queue：配置桶数量 + 桶内并发数，减少同一热点行的锁冲突开销。

### 并发量梯度方案

| 并发量级 | 方案 |
|---------|------|
| 几千 QPS | 普通 MySQL + Inventory Hint |
| 数万 QPS | 分桶（1000 库存 → 10×100），需解决均匀分配/碎片/动态扩容 |
| 本地生活电商 | Redis 预扣 + 异步落盘（接受短暂不一致） |
| 百万 QPS 大促 | 热点库 + SQL 合并 + 分布式缓存（近端缓存）+ 分桶 全部叠加 |

> 适合的才是最好的。复杂度和并发量成正比，不要过度设计。

## 小红书方案：合并秒杀（MySQL 内核级）

### 背景

直播带货场景，目标下单吞吐 **1W+/s**。

演进路径：
1. 社区版 MySQL：并发 update 同一行 TPS ≈ 100-200（几乎不可用）
2. 排队秒杀（自研线程排队）：提升 **10 倍** → 约 2000-4000 TPS
3. 合并秒杀（本方案）：再提升 **5 倍** → **1.5W+/s TPS**

### 事务模型

```sql
BEGIN;
  INSERT INTO inventory_log VALUES (...);   -- 流水（并发无冲突）
  UPDATE inventory SET quantity=quantity-1  -- 热点行（瓶颈所在）
    WHERE sku_id=? AND quantity>0;
COMMIT;
```

瓶颈在 update 同一行，合并秒杀消除此瓶颈。

### Leader-Follower 合并原理

```
客户端1 ─┐
客户端2 ─┤──▶ 抢独占锁 ──▶ Leader: 读 InnoDB → 写全局缓存 → 收集 Follower N 毫秒
客户端3 ─┘
              Follower: 读全局缓存 → 完成扣减 → 写全局缓存（等待唤醒）
              Leader 收集完：重获独占锁 → 全局缓存值一次写 InnoDB → 提交
              → 唤醒所有 Follower，各自返回结果
```

**关键设计**：
- **全局缓存**按表维度共享，生命周期与表结构绑定
- 通过 Leader/Follower 状态机控制数据一致性
- Binlog 仍记录每个事务的改动（1000→999, 999→998），Redolog 记录合并值（1000→900），Crash Recovery 以数据组为单位处理

### 流水线优化（行锁极致压榨）

```
Phase 1（收集）: 组1 收集 Follower ──────────────────────┐
Phase 2（提交）:                      组1 提交（持有行锁）─┘
                                                           组2 Phase1 可与组1 Phase2 并行！
```

→ 上一组 commit 时下一组已开始收集，行锁串行时间压缩。

### 对比排队秒杀

| | 社区 MySQL | 排队秒杀 | 合并秒杀 |
|---|---|---|---|
| 核心思路 | 无优化 | 热点线程排队减少锁冲突 | 多事务合并为1次写入 |
| TPS（128 线程） | ~200 | ~4276 | ~23543 |
| 提升倍数 | 基准 | ~21x | ~5.5x（相对排队） |
| 业务 SQL 改动 | 无 | 无（内核实现） | 无（内核实现，可动态开关） |
| 生态兼容 | — | — | Binlog 格式不变，DTS/Canal 无感知 |

## 方案选型决策树

```
需要秒杀/热点扣减
         │
并发 < 1000 QPS？
    Yes → 普通 MySQL，注意热点行检测
    No  → 
         │
是否可接受数据不一致（短期）？
    Yes + 本地生活 → Redis预扣 + 异步落盘（需兜底对账）
    No（电商强一致）→
         │
是否有自研数据库或阿里云 RDS？
    Yes → Inventory Hint / 合并秒杀内核补丁
    No  → 
         │
并发 < 10万 QPS？
    Yes → 分桶库存 + Redis 本地缓存
    No  → 分桶 + 近端缓存 + SQL 合并 + 热点库全组合
```

## 面试话术

**Q：你们的秒杀系统是怎么做的？**

> "我们参考了阿里的做法，核心思路是在数据库层面解决热点行竞争，而不是把数据库绕过去。具体用了 Inventory Hint：通过 COMMIT_ON_SUCCESS 等 Hint 标记热点 SQL，让 MySQL 内核自动将同一行的并发更新分组，Leader 获取行锁、Follower 直接复用、最终一次组提交，把行锁等待、B+树遍历、事务提交次数全部压缩。这样在几千 QPS 下就能稳定抗住，比普通 MySQL 热点行更新性能提升 10 倍以上。"

**Q：不用 Redis 预扣有什么好处？**

> "Redis 预扣的核心问题是一致性——Redis 宕机、网络抖动、Redis 扣了但 DB 写失败，都需要额外的对账补偿逻辑。而基于 MySQL 的方案，天然有 WAL 和事务保障，少卖/超卖的概率极低。代价是需要在 MySQL 内核层做优化，但有了 Inventory Hint 这个补丁，可以在保持 SQL 不变的情况下大幅提升热点行并发能力。"
