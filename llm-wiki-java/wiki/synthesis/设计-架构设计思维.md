# 架构设计思维

> P7 架构认知的核心：架构本质是权衡，没有银弹，只有最合适的解法。设计原则是权衡的工具，演进是权衡的结果，技术债是权衡的代价。

---

## 架构的本质：权衡

**为什么架构就是做权衡？**

架构目标天然互相矛盾：
```
可扩展性    ↔  简洁性（每层抽象都是复杂度）
高性能      ↔  高可用（副本/冗余增加延迟）
安全性      ↔  易用性（鉴权增加开发摩擦）
快速交付    ↔  代码质量（产生技术债）
统一技术栈  ↔  最优工具选型
```

**P7 答题框架**：不说"我们用了 X 技术"，而说"**在 A 和 B 之间我们选择了 X，因为在当时的业务阶段，[具体理由] 更重要，代价是 [Y]，我们通过 [Z] 来弥补**"。

---

## 没有银弹

来源：Brooks《No Silver Bullet》（软件工程圣典）。

**核心命题**：软件的本质复杂性（Essential Complexity）无法被任何单一技术或方法消除，只有附属复杂性（Accidental Complexity）可以被工具改善。

**实践含义**：
- 微服务不是银弹（带来分布式复杂性）
- DDD 不是银弹（带来高建模成本）
- 云原生不是银弹（带来运维复杂性）
- 遇到任何"彻底解决 XX 问题"的技术方案，先问"它的代价是什么"

---

## 好架构的判断标准

**核心结论：没有好坏之分，只有适不适合。适合当前业务阶段和团队的架构就是好架构。**

好架构必须满足的特质：

| 特质 | 含义 |
|------|------|
| **可理解性** | 新人能快速上手，复杂度可控 |
| **可扩展性** | 功能增加不需要大规模改造 |
| **可维护性** | 故障可排查，监控完善 |
| **避免过度设计** | 没有无用的抽象层和配置 |
| **服务业务** | 能随业务需求变化而演进 |

> 注意：**避免过度设计**是高频被忽视的维度。为未来三年做设计 = 过度设计。

## 架构设计原则（SOLID + 架构级）

### 基础原则（对应 SOLID）

| 原则 | 架构含义 |
|------|---------|
| 单一职责（SRP）| 每个服务/模块只做一件事 |
| 开放封闭（OCP）| 通过接口/插件扩展，不改原有代码 |
| 依赖倒置（DIP）| 高层依赖抽象，不依赖具体实现 |
| 接口隔离（ISP）| 接口按需定义，不暴露不需要的方法 |

### 架构级原则

```
分离关注点：Controller / Service / Repository 各守边界
高内聚低耦合：模块内紧密协作，模块间最小依赖
最少知识原则：A 只需知道直接依赖的 B，不需要知道 B 依赖的 C
分层架构：L1 → L2 → L3，下层不依赖上层
```

## 架构是设计出来的还是演进出来的？

**结论：两者缺一不可。**

```
初始设计 ──────────────────────────────→ 演进
     │                                      │
决定系统骨架                         响应业务变化
(模块划分、通信方式、                (拆分/合并服务、
 存储选型、接口契约)                  引入新中间件、
                                     技术债重构)
```

**P7 话术**：面试时说"我们当初设计时考虑了 X，随着业务发展发现了 Y 的问题，因此我们做了 Z 演进，这次演进的核心权衡是..."

## 技术选型的决策框架

**没有最好，只有更合适。**选型时需综合评估：

| 维度 | 评估问题 |
|------|---------|
| **功能满足度** | 当前需求和未来 1~2 年需求都能覆盖吗？ |
| **成熟度** | 社区活跃？Bug 少？大厂生产验证过？ |
| **成本** | 学习成本、使用成本、迁移成本、维护成本 |
| **非功能需求** | 性能、可扩展性、安全性达标吗？ |
| **技术栈匹配** | 和团队现有技术体系是否兼容？ |

> 典型反例：全公司用 Kafka，为了新颖引入 Pulsar → 团队无人有经验 + 运维成本翻倍。

## 技术债务

**定义**：为快速交付而采取的妥协方案所积累的技术负担，本质是以速度换质量。

```
低质量代码（快速上线）
     ↓
代码越来越难改
     ↓
功能迭代越来越慢
     ↓
反而拖累业务
```

**如何管理**：
1. **显式化**：技术债加 TODO + 工单跟踪，不让它隐形
2. **按优先级还债**：高频修改路径的债优先还（改动频率 × 债的严重程度）
3. **规范预防**：代码评审、架构规范、测试覆盖率门禁

## 微服务拆分原则

| 原则 | 说明 |
|------|------|
| **职责单一** | 一个微服务聚焦一件事（用户 ≠ 交易）|
| **业务边界** | 围绕业务域拆分（投保 vs 理赔）|
| **中台化** | 公共能力独立成服务（消息、短信、权限）|
| **系统保障隔离** | 秒杀 vs 日常交易分开；在线 vs 离线任务分开 |
| **技术栈一致** | 不同技术栈不要强行融合 |
| **单向依赖** | 禁止循环依赖，否则拆分不合理 |
| **康威定律** | 微服务边界应与团队边界对齐（多团队维护一个服务 = 冲突来源）|

**康威定律核心**：组织架构决定系统架构，反之亦然。先理清团队职责边界，再定服务边界。

## 单元化架构

**定义**：以"单元"为部署基本单位，每个单元包含完整的服务链路 + 分配给它的数据分片，单元内封闭执行，不跨单元。

```
┌─────────────────┐   ┌─────────────────┐   ┌─────────────────┐
│    上海单元      │   │    北京单元      │   │   新加坡单元     │
│  交易/支付/订单  │   │  交易/支付/订单  │   │  交易/支付/订单  │
│  用户数据(部分)  │   │  用户数据(部分)  │   │  用户数据(部分)  │
└─────────────────┘   └─────────────────┘   └─────────────────┘
              ↑ 全量数据 + 跨单元操作路由到中心 ↑
```

**优势**：低网络延迟、天然异地多活、弹性扩容（加单元）

**代价（高频考点）**：
1. **全链路改造成本极高**，一个节点未单元化则该节点后所有链路都走中心
2. **数据路由复杂**：用户漫游（出差）导致就近原则 vs 数据分片的矛盾
3. **成本极高**：仅大厂部分核心链路采用

> 适用：阿里淘宝、蚂蚁金服等超大规模系统。普通公司不需要，但面试要能说清楚其权衡。

## MVC vs 三层架构

| | MVC | 三层架构 |
|--|-----|---------|
| **本质** | 设计模式 | 架构模式 |
| **关注点** | 用户界面 / 用户交互 | 整个应用程序分层 |
| **三个部分** | Model / View / Controller | 表示层 / 业务逻辑层 / 数据访问层 |
| **层级** | UI 层内部 | 跨越整个系统 |
| **关系** | 可在三层架构的表示层中使用 MVC | 包含 MVC |

**一句话记忆**：MVC 是表示层的内部分工；三层是系统的纵向切割。两者不互斥，常组合使用。

## 介绍项目架构的四个维度

面试时介绍架构，按优先级：

```
1. 业务架构（首选）
   核心模块划分（用户/订单/交易/支付） + 核心业务流程图

2. 技术架构（首选）
   技术栈 + 分层设计 + 服务通信方式（REST/RPC/MQ）
   + 中间件选型（缓存/MQ/搜索/网关）

3. 部署架构（非主动问不提）
   容器化/K8s + 机器规模 + 集群方案

4. 数据架构（非主动问不提）
   存储选型 + 分库分表 + 数仓链路
```

## 亿级商品存储方案

**挑战**：单表上限（2000w行），热点数据访问，大规模读写并发。

```
存储层选型（按规模）：
├── MySQL + 分库分表（主流方案）
│   按 商品ID hash 分片，冷热数据归档
├── TiDB / OceanBase（NewSQL）
│   分布式 + 兼容 MySQL 协议，避免分库分表改造成本
└── 数据缓存（Redis）
    热门商品缓存，命中率 > 95% 则 DB 压力可接受

索引优化：
  商品列表查询（品类/品牌/价格过滤）→ ES 建索引，Canal 同步
  商品详情（按 ID 精确查）→ 分库分表 + Redis

归档策略：
  下架超过 N 天的商品 → 冷存储（OSS/归档库）
  归档后主库只保留元数据

数据压缩：
  商品描述/图片 URL 等文本字段可压缩存储
```

> 参见 [[设计-MySQL大表与查询优化]] 中的分表阈值和 DDL 方案。
