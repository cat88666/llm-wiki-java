# 分布式事务

> 分布式事务的本质矛盾：**跨进程调用不能共享同一个数据库事务**（事务绑定数据库连接，连接不跨进程）。解决策略分两大类：强一致（2PC/XA，牺牲可用性）和最终一致（TCC/消息事务/本地消息表，接受短暂中间态）。

## 问题驱动

**面试高频问题组合**：
- TCC 和 2PC 有什么区别？为什么说 TCC 是最终一致性？
- TCC 的空回滚和悬挂问题如何解决？
- 本地消息表和事务消息（RocketMQ）有什么区别？何时选哪个？
- Seata 的模式有哪些？AT 模式原理是什么？

## 方案全景

```
强一致性
├── 2PC (Two-Phase Commit) — 协调者+参与者，同步阻塞
├── 3PC (Three-Phase Commit) — 解决 2PC 协调者单点阻塞
└── XA — 数据库原生支持的 2PC（性能差，少用）

最终一致性
├── TCC (Try-Confirm-Cancel) — 业务层补偿，性能好
├── 本地消息表 — 可靠消息 + 定时重试
├── 事务消息（RocketMQ）— MQ 半消息机制
└── Saga — 长事务，正向补偿链 + 逆向补偿链
```

## 核心机制

### 2PC vs 3PC

**2PC（两阶段提交）**：
```
阶段1（Prepare）：协调者 → 所有参与者 prepare → 参与者锁资源并返回 vote
阶段2（Commit/Abort）：全部 vote=yes → 协调者发 commit；否则发 rollback
```
- 缺点：**同步阻塞**（参与者等 Commit 消息期间持有锁）；协调者宕机 → 参与者永久阻塞

**3PC 改进**：增加 CanCommit 阶段（先询问是否能提交），超时后参与者自动提交（降低阻塞）；但仍有数据不一致风险（超时自动提交时协调者已 Abort）。

### TCC

```
Try（预留/冻结资源）
    ↓ 全部成功
Confirm（确认/提交）
    ↓ 任意失败
Cancel（回滚/解冻）
```

**电商下单 TCC 示例**：
- Try：库存冻结 `frozen_stock += 1, available_stock -= 1`；订单创建为"待确认"状态
- Confirm：冻结库存扣减 `frozen_stock -= 1`；订单状态改为"确认"
- Cancel：解冻库存 `frozen_stock -= 1, available_stock += 1`；订单废弃

**为什么是最终一致性而非强一致性**：Try → Confirm 之间存在中间状态（库存冻结但订单未确认），其他事务可以看到这个中间状态。

**TCC 三大问题**：

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| 空回滚 | Cancel 先于 Try 到达（网络乱序）| 事务记录表：Cancel 时查询 Try 是否执行，未执行则直接返回成功 |
| 悬挂 | Try 在 Cancel 之后到达 | 事务记录表：Cancel 已执行时，拒绝后续到达的 Try 操作 |
| 幂等 | 网络重试导致 Confirm/Cancel 多次执行 | 事务记录表中标记状态，幂等校验 |

**核心**：引入一张**事务状态记录表**，记录每个 xid 的执行状态，解决上述三个问题。

### 本地消息表 vs 事务消息（RocketMQ）

**本地消息表**：
```
1. DB 操作 + 写消息表（同一本地事务）
2. 定时任务扫描未投递消息 → 发送到 MQ
3. Consumer 消费成功 → 回调更新消息表状态（或 Consumer 直接删除）
```
- 优点：无需引入 MQ 事务特性，实现简单
- 缺点：定时任务有延迟；消息表与业务库耦合；需维护定时任务

**事务消息（RocketMQ 半消息机制）**：
```
1. Producer 发送 Half Message（消息对 Consumer 不可见）
2. Producer 执行本地事务
3a. 本地事务成功 → 发送 Commit → 消息对 Consumer 可见
3b. 本地事务失败 → 发送 Rollback → 消息被删除
3c. 超时 → RocketMQ 回查 Producer 本地事务状态（checkLocalTransaction）
```
- 优点：MQ 框架保证消息和事务的一致性，无需额外定时任务
- 缺点：依赖 RocketMQ（Kafka 不支持事务消息）；Consumer 必须实现幂等

### Seata 主要模式

| 模式 | 原理 | 适用场景 |
|------|------|---------|
| **AT（默认）** | 自动解析 SQL，生成 undo log；阶段1先提交+undo log，阶段2Commit删undo log或Rollback用undo log回滚 | 绝大多数 CRUD 场景，侵入性低 |
| **TCC** | 用户自定义 Try/Confirm/Cancel 三个接口，Seata 协调调用 | 性能要求高，自己控制补偿逻辑 |
| **Saga** | 长事务，正向执行链 + 补偿链；只保证最终一致 | 无法改造为 TCC 的遗留系统，长流程 |
| **XA** | 利用数据库原生 XA 协议（严格强一致）| 强一致要求高，接受性能损失 |

**Seata AT 与 2PC 区别**：AT 阶段1就提交（不持锁等待阶段2），通过 undo log 做补偿；2PC 阶段1不提交，持锁等待协调者指令，锁粒度大。

## 选型决策树

```
是否能接受最终一致？
├── 不能（金融核心账务）→ XA / Seata XA（性能差但强一致）
└── 可以接受
    ├── 涉及多个微服务，且各服务已有 Try/Confirm/Cancel 接口
    │   → TCC（可用 Seata TCC 模式）
    ├── 主要靠消息传递（MQ 已是 RocketMQ）
    │   → 事务消息（Half Message）
    ├── 已有 MQ 且不想改代码
    │   → 本地消息表 + 定时任务
    └── 长事务，无法改造为 TCC
        → Saga
```

## 关键权衡

1. **强一致 vs 最终一致**：金融核心账务用强一致（接受性能差）；电商交易流程用最终一致（用户体验优先）
2. **TCC vs 本地消息表**：TCC 性能好（不依赖 MQ 投递延迟）但开发量大（三个接口）；消息表实现简单但有消息延迟
3. **Seata AT vs 自实现 TCC**：AT 侵入性低、接入快，但有 undo log 额外存储开销且不支持所有 SQL；自实现 TCC 灵活但每个方法都需实现三个接口
4. **2PC 阻塞问题**：2PC 阻塞根因是参与者在等待协调者 Commit/Abort 期间持锁；TCC 不持锁（Try 只冻结，不阻塞其他事务），这是 TCC 性能更好的核心原因

## 跨层依赖分析

- **L5 MySQL**：[[机制-MySQL三种日志]] 的 undo log 是 Seata AT 模式的核心工具；本地事务边界来自 [[机制-Spring事务]]
- **L5 MySQL 锁**：2PC 阶段1持有行锁/表锁导致阻塞，对应 [[机制-InnoDB锁]] 中的锁等待场景
- **L7 MQ**：本地消息表和事务消息都依赖 [[机制-消息队列可靠性]] 的 ACK 机制保证消息至少投递一次
- **L7 Dubbo/SpringCloud**：跨服务调用框架是分布式事务的触发场景，见 [[机制-RPC与Dubbo]]、[[机制-微服务与SpringCloud]]

## 面试回答要点

1. **先分类**：强一致（2PC/XA）vs 最终一致（TCC/消息/Saga），体现体系化认知
2. **TCC 重点讲空回滚和悬挂**：核心是事务状态记录表，面试官爱考
3. **对比 Seata AT 和 2PC**：AT 阶段1就提交（不持锁），2PC 阶段1不提交（持锁等待），这是核心区别
4. **给出选型建议**：根据一致性要求、技术栈、开发成本给出合理推荐
