# 系统设计-游戏架构

> 一套基于 SOFA-Bolt + Netty + Dubbo 的实时多人在线游戏平台，核心工程决策：用户级线程隔离防止玩家间干扰、工厂模式路由支持游戏类型热扩展、任务链+Redis eventKey 实现多桌并发游戏状态机；选型结论取决于对延迟隔离和可扩展性的权衡。

    ↓ WebSocket 长连接
┌───────────────────────────────────┐

│ SOFA-Bolt 转发至游戏服             │
│ 工厂模式按 ServiceTypeEnum 路由    │
└────┬──────────────┬───────────────┘
     │ Bolt RPC      │ Dubbo RPC
┌────▼──────┐  ┌─────▼──────────┐  ┌────────────────┐
│dx-game-hall│  │dx-game-texas   │  │dx-game-race-hall│

└────────────┘  └────────────────┘  └────────────────┘

**技术栈全景**：

| 层级 | 技术选型 | 选型结论 |
|------|----------|---------|
| 客户端→网关 | Netty WebSocket | 浏览器/App 友好，支持百万连接 |
| 网关→游戏服 | SOFA-Bolt（高性能二进制 RPC）| 低延迟，与 Dubbo 生态无缝集成 |
| 服务治理 | Dubbo 3.x + Nacos | 成熟 Java 生态，支持热配置下发 |
| 消息队列 | RocketMQ | 顺序消息支持游戏事件有序消费 |
| 缓存 | Guava（L1）+ Redis（L2）| 两级缓存降低延迟 |
| 分布式锁 | Redisson | watchdog 自动续期，防业务超时 |

## 二、核心设计亮点

### 用户级线程隔离（dx-net-bolt）

**问题**：不同玩家共享线程池时，慢玩家（网络差/操作慢）会拖累所有人。

**方案**：`NetCommandExecutorDispatcher` 按 `userId` 创建独立 `SingleThreadExecutor`，每个玩家的请求串行执行，互不干扰。

```java
// 按 userId 分配独立线程
dispatcher.submitFor(userId, new DispatcherThread(...));

// 心跳用独立线程池（不抢占业务队列）
if (SystemConstant.BOLT_SERVER_HEART_BEAT.equals(requestTypeId)) {
    dispatcher = heartBeatExecutorDispatcher;
}
```

| 维度 | 用户级线程隔离 | 全局共享线程池 | 结论 |
|------|------------|------------|------|
| 隔离性 | 完全隔离，天然串行 | 慢请求影响全局 | 游戏场景用隔离 |
| 资源占用 | 线程数随玩家线性增长 | 固定线程数 | 需设置玩家上限 |
| 锁需求 | 单用户无需加锁 | 共享状态需加锁 | 隔离消除锁竞争 |

**线程数公式**：`cpu 核数 × 8`（可配置），队列深度 2000；超过上限则拒绝新玩家连接（保护系统）。

### 工厂模式服务路由（dx-net-gateway）

**问题**：网关需要将请求分发到不同游戏服，硬编码 if-else 每次新增游戏类型都需改网关。

**方案**：`DxServerManager` 实现 `InitializingBean`，自动扫描所有 `DxServerHandle` 实现类，构建 `ServiceTypeEnum.id → Handler` Map，O(1) 路由。

```java
// Spring 启动时自动注入所有实现类
@Component
public class DxServerManager implements InitializingBean {
    @Autowired
    private List<DxServerHandle> dxServerHandleList;

    private Map<Integer, DxServerHandle> serverHandleMap;

    @Override
    public void afterPropertiesSet() {
        serverHandleMap = dxServerHandleList.stream()
            .collect(Collectors.toMap(k -> k.serviceType().getId(), Function.identity()));
    }

    public DxServerHandle route(int serviceId) {
        return serverHandleMap.get(serviceId);  // O(1) 路由
    }
}
// 新增游戏类型只需新增 DxServerHandle 实现，无需修改网关核心（开闭原则）
```

## 三、游戏状态机与分布式任务调度

**德州扑克一手牌流程**：

```
SendHandCardTask（发手牌，1.5s 延迟）
  ↓ 所有玩家看牌后
UserBetCountdownTask（下注倒计时，1.5s 间隔，超时自动弃牌）
  ↓ 所有人行动后
SendPublicCardTask（发 Flop/Turn/River）
  ↓
ProvisionalSettlementTask（自动结算）
  ↓
NextHandStartTask（开启下一手）
```

**任务设计要点**：

```java
// 任务 key 包含完整上下文，多桌并发互不干扰
String eventKey = "USER_BET_COUNT_DOWN_TABLE_" + tableId
                + "_UID_" + userId
                + "_HAND_" + handCode;

// 存 Redis 延迟队列：服务重启后状态可恢复
redisTemplate.opsForZSet().add("task:queue", eventKey, executeTime);
```

**三个核心能力**：
1. 多桌并发互不干扰（key 包含 tableId+handCode）
2. 服务重启后状态恢复（Redis 持久化）
3. 超时自动推进（防止单玩家导致游戏卡死）

## 四、保险赔率实时计算

**问题**：保险（Insurance）需要实时计算任意手牌组合的胜率，无法预建表（组合爆炸）。

**分阶段策略**：

| 阶段 | 策略 | 组合数 | 结论 |
|------|------|--------|------|
| Pre-Flop | 查表（预建 2 万+组合）| 固定有限 | 性能最优 |
| Flop | 枚举 C(47,2) = 1081 种 | 可接受 | 精度最优 |
| Turn | 蒙特卡洛模拟 | N/A | 速度优先 |
| River | 精确比牌 | O(n) | 最简单 |

**方案选型**（实时枚举 vs 预建表）：

| 方案 | 优点 | 代价 | 结论 |
|------|------|------|------|
| 实时枚举 | 支持所有变体，无存储 | CPU 密集 | 选此，配合手牌级缓存 |
| 预建表 | 查询极快 | 存储空间爆炸（52C2×52C2=数十亿）| 不可行 |

**结果缓存**：同一手牌阶段内胜率不变（公牌固定时），以 (手牌+公牌+对手手牌) 为 key 缓存结果，缓存 TTL = 当前手牌阶段结束。

## 五、多级缓存架构

**场景**：房间列表、牌桌状态、规则配置是高频读取热数据。

```java
// Guava LoadingCache（L1，进程内）→ Redis（L2）→ MySQL（L3）
LoadingCache<String, ConcurrentMap<Long, TableInfoRespDto>> hallRoomTableListCache =
    CacheBuilder.newBuilder()
        .expireAfterAccess(10, TimeUnit.HOURS)  // L1：10 小时 TTL
        .build(new CacheLoader<>() {
            @Override
            public ConcurrentMap<Long, TableInfoRespDto> load(String roomCode) {
                return hallRoomService.getHallRoomTableListByRoomCode(roomCode);
            }
        });
```

| 层级 | 存储 | 延迟 | 容量 | 更新方式 |
|------|------|------|------|---------|
| L1 | Guava（进程内）| 微秒级 | 受 JVM 堆限制 | 过期后重加载 |
| L2 | Redis | 毫秒级 | 大 | MQ 事件驱动更新 |
| L3 | MySQL | 毫秒~秒 | 无限 | 写操作直接更新 |

**一致性保证**：牌桌状态变更时，先更新 MySQL，再通过 MQ 通知各节点删除 L1 缓存（L2 Redis 由应用直接更新），保证最终一致。

## 六、关键权衡对比

| 决策点 | 选项 A | 选项 B | 选型结论 |
|--------|--------|--------|---------|
| 网关通信协议 | SOFA-Bolt（与 Dubbo 生态集成）| Spring WebFlux 响应式 | 选 Bolt：无缝集成现有服务治理，WebFlux 与 Netty 有额外适配成本 |
| 线程隔离 | 用户级 SingleThreadExecutor | 全局固定线程池 | 选隔离：游戏场景一个慢玩家不能影响其他人 |
| 游戏流程引擎 | 任务链 + Redis 调度 | Actor 模型（Akka）| 选任务链：实现简单，依托现有 Redis；Akka 引入新系统复杂度 |
| 保险计算 | 实时枚举 + 手牌级缓存 | 预建全量表 | 选实时：支持多种变体，预建表存储空间不可行 |

## 七、面试高频追问

**问**：你们的长连接网关为什么不用 SpringCloud Gateway？
**答**：SpringCloud Gateway 基于 WebFlux + Netty，适合 HTTP REST 请求的路由和过滤；我们的场景是 WebSocket 长连接（有状态，持续双向通信）+ SOFA-Bolt RPC（游戏服内部通信），两者协议不同。SOFA-Bolt 是 Dubbo 生态的高性能二进制 RPC 协议，与我们现有的 Dubbo 服务治理无缝集成，延迟更低。SpringCloud Gateway 并不原生支持 WebSocket 到 Bolt 的协议转换，引入适配成本高。

**问**：如何保证同一玩家的请求顺序执行？
**答**：按 userId 创建独立 SingleThreadExecutor（`NetCommandExecutorDispatcher`），同一 userId 的所有请求都提交到同一个单线程执行器。单线程天然串行，无需加锁。额外好处：用户间完全隔离，一个用户的慢请求不影响其他用户。代价是线程数随在线玩家数线性增长，需要设置最大连接数上限保护系统。

**问**：德州扑克的游戏状态机如何实现多桌并发？
**答**：任务链 + Redis eventKey 方案。每个任务的 key 包含 `(tableId, handCode, userId)`，多个牌桌的任务 key 不同，在 Redis 延迟队列中独立存储，互不干扰。服务重启后可从 Redis 恢复所有进行中的牌局状态。超时自动推进（定时扫描 Redis ZSet）防止单个玩家超时导致游戏卡死。单机可支撑数百个并发牌桌。

**问**：批量解码为什么能把 Netty 网关吞吐量提升 200%？
**答**：继承 `ByteToMessageDecoder`，一次 `decode()` 循环解码 ByteBuf 中所有完整消息，输出 `List<GameMessage>`，按消息类型分组批量提交业务线程池。原来 10 条消息 = 10 次 EventLoop→业务线程池 task 提交，批量后 1 次提交 10 条，上下文切换减少 10 倍，EventLoop task queue 竞争大幅降低。

**问**：Netty 网关优雅下线怎么实现？
**答**：三步：①从 Nacos 注销实例，不再路由新连接；②停止 accept 新连接（`channel().close()`），已建立 Channel 继续处理；③等待旧连接消息处理完毕，超时（30s）后向客户端发"服务器维护"消息，客户端自动重连其他节点。通过 JVM shutdown hook 触发，K8s `preStop` 也调用此逻辑。

**问**：多级缓存如何保证一致性？
**答**：写操作时：先写 MySQL（持久化），再更新 Redis L2（主动刷新），再通过 MQ 广播给所有节点删除本地 Guava L1 缓存。读操作时：L1 → L2 → L3 依次降级查询，L2/L3 命中后回写上层。可能存在短暂不一致窗口（L1 删除前的并发读），对游戏配置和房间列表这类数据（秒级一致即可）完全可以接受。游戏状态（牌局）不走多级缓存，直接从单点权威状态读取。

## 八、与其他概念的关系

- 接入层长连接管理依赖 [[机制-Netty]]：主从 Reactor 支撑百万级连接，IdleStateHandler 检测僵尸连接，PooledByteBufAllocator 减少内存碎片
- 游戏服务间调用使用 [[机制-RPC与Dubbo]]：Dubbo SPI 扩展、负载均衡策略、服务注册发现
- 服务注册和动态配置依赖 [[机制-Nacos]]：游戏规则参数（保底/赔率/限制）支持热更新
- 游戏事件消息（顺序消费）使用 [[机制-RabbitMQ]]（RocketMQ 顺序消息）：保证游戏事件处理顺序
- 三高设计属于 [[主题-三高架构]]：用户级线程隔离（高可用）+ 多级缓存（高性能）+ 分布式任务（高并发）
- 架构权衡和选型依据参见 [[设计-P7架构思维]]：用户隔离 vs 共享线程池是典型的"隔离性 vs 资源效率"权衡
