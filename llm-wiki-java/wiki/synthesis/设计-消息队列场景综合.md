# 消息队列场景综合

> MQ 的核心价值：解耦 + 异步 + 削峰。工程场景中需要回答三类问题：①用还是不用MQ（vs Spring Event / BlockingQueue）；②如何保证可靠性（不丢不重）；③如何保证顺序性。

---

## Spring Event vs MQ：如何选择

| 维度 | Spring Event | MQ（Kafka/RabbitMQ） |
|------|-------------|---------------------|
| **作用范围** | 同一 JVM 内 | 跨服务、跨进程 |
| **事务支持** | 可绑定事务（`@TransactionalEventListener`） | 需要额外的事务消息方案 |
| **可靠性** | 进程崩溃消息丢失 | 持久化，可靠投递 |
| **延迟** | 极低（方法调用）| 毫秒级 |
| **运维成本** | 零（Spring内置）| 需要部署维护集群 |
| **典型场景** | 模块间解耦（同服务）、事务提交后触发 | 跨服务异步通信、流量削峰 |

**选型原则**：同服务内模块间解耦 → Spring Event；跨服务 or 需要持久化 or 需要削峰 → MQ。

---

## BlockingQueue vs MQ：为何不用前者

`BlockingQueue` 作为消息队列的缺陷：

| 缺陷 | 说明 |
|------|------|
| **不持久化** | JVM 重启消息全丢 |
| **无分布式** | 单机，无法跨进程/服务 |
| **无路由** | 消息只能点对点，无 Topic/Exchange 语义 |
| **无重试** | 消费失败无自动重投 |
| **无监控** | 无消息堆积监控、无死信队列 |
| **无背压** | 生产快消费慢时只能靠 JVM 内存扛 |

**适用场景**：BlockingQueue 只适合同 JVM 内线程间的简单任务传递（如线程池任务队列）。

---

## 拉模式 vs 推模式

| | 拉模式（Pull，Kafka）| 推模式（Push，RabbitMQ）|
|--|---|---|
| **控制权** | 消费者主动拉，自控速率 | Broker 推送，消费者被动接收 |
| **实时性** | 轮询有延迟（可用长轮询弥补）| 近实时 |
| **过载风险** | 消费者不会被压垮 | 消费慢时需要 basicQos 限流 |
| **批量消费** | 天然支持（一次拉多条）| 需要额外配置 |
| **适用场景** | 大吞吐量、消费速率不均匀 | 实时性要求高、消费速率稳定 |

## 消息乱序：四种解决方案

**典型场景**：订单状态变更消息（创建→支付→发货），消息到达顺序不能保证。

### 方案1：顺序消息（发送侧保障）
- Kafka：同一业务 key（如 orderId）发送到同一 Partition
- RocketMQ：发送时选择固定 Queue（`MessageQueueSelector`）
- **要求**：同一 Partition/Queue 只能由一个消费者串行消费

### 方案2：前置状态判断（消费侧保障）
```
消息体中携带 beforeStatus（期望的前置状态）
消费时：查询当前状态 vs beforeStatus
  一致 → 正常处理
  不一致 → 返回失败，MQ 重投（等待前驱消息先到达）
```
**要求**：状态只能单向推进，不能回退。

### 方案3：事件表 + 异步重试（最健壮）
```
1. 收到消息 → 幂等校验 → 存入事件表（state=PENDING）→ 立即返回成功（告诉MQ处理完了）
2. 异步线程处理事件表中的消息（按业务单号+序列号排序）
3. 处理成功 → state=SUCCESS
4. 处理失败 → 定时任务重试，超次数报警人工处理

优化：处理成功后查 Redis 是否有等待的后续消息 → 主动触发处理
```
**优点**：避免 MQ 消息堆积；支持任意顺序重排；失败重试可控。

## 提升 Kafka 单分区消费吞吐量

**约束**：单 Partition 只能被同一 Consumer Group 的一个消费者消费，增加消费者实例无效。

**可行方案**：
```
1. 批量消费：增大 max.poll.records，一次拉更多消息
2. 异步处理：消费后提交到线程池处理，不阻塞 poll 线程
3. 并行写入：消费多条记录后，批量写 DB（减少 IO 次数）
4. 增加 Partition：根本解法，但需要重新分配 Consumer
```

## 为什么不用MQ实现订单超时关闭

MQ 延迟消息做超时关闭的缺陷：

| 问题 | 说明 |
|------|------|
| **延迟不精准** | RabbitMQ TTL + 死信队列受队头阻塞影响；RocketMQ 只有固定级别（1s~2h）|
| **消息丢失** | MQ 可靠性配置不当时延迟消息丢失 → 订单永远不关闭 |
| **重复消费** | MQ 的 at-least-once 可能触发多次关闭逻辑（需幂等）|
| **不可取消** | 用户支付后，已投递的超时消息无法撤回 |

**推荐方案**：Redisson `RDelayedQueue`（基于 ZSet，精度到秒）或 专用任务调度框架（XXL-Job 定时扫表）。

## 本地消息表：下游失败如何处理

**上游的答案：不回滚，只保证最终一致。**

```
本地消息表设计：
  id, biz_id, biz_type, payload, state(PENDING/SUCCESS/FAILED), retry_count, next_retry_time

下游失败处理流程：
  1. 定时任务扫描 state=PENDING 且 next_retry_time < now 的消息
  2. 重新调用下游，成功→state=SUCCESS
  3. 失败→retry_count+1，next_retry_time 按退避策略延后（指数退避）
  4. retry_count 超阈值（如 5次）→ state=FAILED + 报警
  5. 人工处理 FAILED 消息（补偿或核对）
```

**核心设计原则**：上游已提交的业务不应因下游失败而回滚（会破坏上游业务的完整性）；转而保证下游**最终执行成功**。

## 消息表设计：避免丢消息

```sql
CREATE TABLE message_record (
    id          BIGINT PRIMARY KEY AUTO_INCREMENT,
    msg_id      VARCHAR(64) NOT NULL UNIQUE,   -- 消息唯一ID（幂等键）
    biz_id      VARCHAR(64) NOT NULL,           -- 业务单号
    biz_type    VARCHAR(32) NOT NULL,           -- 业务类型
    topic       VARCHAR(64),                    -- 目标 Topic
    payload     TEXT,                           -- 消息体（JSON）
    state       TINYINT DEFAULT 0,              -- 0=待发送 1=已发送 2=失败
    retry_count INT DEFAULT 0,
    next_retry_time DATETIME,
    create_time DATETIME,
    update_time DATETIME,
    INDEX idx_state_retry (state, next_retry_time)
);
```

**双保险**：① 先写消息表（与业务操作同事务）；② 定时任务扫描 state=0 的消息重新投递。确保"消息表有记录"和"MQ收到消息"最终一致。
