# IO模型

> 程序发起 IO 请求后，如何等待数据就绪、如何获取数据——三种模型对"等待方式"的不同选择，决定了并发能力的上限。

## 第一性原理

IO 操作分两个阶段：①等待数据从设备（磁盘/网卡）就绪到内核缓冲区；②把数据从内核缓冲区拷贝到用户空间。不同 IO 模型的差异在于：**线程如何应对这两个阶段的等待**。

## 核心机制

### 三种模型对比

| 模型 | 等待方式 | 线程状态 | Java 支持版本 |
|------|---------|---------|--------------|
| **BIO**（Blocking IO）| 等待 ① + ② 全程阻塞 | 阻塞，啥都不能干 | JDK 1.0+ |
| **NIO**（Non-Blocking IO）| ① 轮询检查；② 阻塞 | 可做其他事，但需要主动轮询 | JDK 1.4 |
| **AIO**（Asynchronous IO）| 两阶段全部异步，完成后回调 | 完全不阻塞 | JDK 1.7 |

### BIO — 同步阻塞

```
线程发起 read() → 阻塞等待 → 数据就绪 → 拷贝到用户空间 → 返回
```
- 一个连接一个线程，10000 连接需要 10000 线程
- 线程切换开销大，连接数受限于线程池大小

### NIO — 同步非阻塞 + Selector

```
线程注册 Channel 到 Selector → Selector 轮询就绪事件 → 就绪后处理
```
- 一个线程（Selector）管理多个 Channel
- **适合**：连接数多、每次 IO 数据量小的场景（聊天服务器）
- Netty 就是基于 NIO 的高性能网络框架

### AIO — 异步非阻塞

```
线程发起 read() → 立即返回 → OS 数据就绪后自动回调
```
- 完全不需要轮询，OS 主动通知
- **适合**：连接数多、每次 IO 耗时长的场景（文件服务器、相册服务器）
- Linux 上 AIO 实现不成熟，Netty 放弃了 AIO 支持

### 同步/异步 vs 阻塞/非阻塞

```
同步：调用方主动等待或检查结果（BIO 等；NIO 轮询）
异步：调用方发起后不管，结果由回调通知（AIO）

阻塞：线程在等待期间不能做其他事（BIO）
非阻塞：线程发起后可以去做其他事（NIO、AIO）
```

## 关键权衡

| | BIO | NIO | AIO |
|--|-----|-----|-----|
| 编程复杂度 | 简单 | 复杂 | 复杂 |
| 并发能力 | 低（线程数限制）| 高（少量线程管多连接）| 高 |
| 适用连接特征 | 少量长连接 | 多短连接（高并发）| 多长连接（重操作）|
| OS 支持 | 全平台 | 全平台 | Linux 支持差 |

**实践中**：绝大多数高并发服务器用 NIO（通过 Netty 封装），AIO 在 Java 生态中使用较少。

## 与其他概念的关系

- 在 L8 系统设计层：高并发服务器的连接模型选择（BIO/NIO/AIO）直接决定单机并发上限
- 支撑了 L7 Netty/RPC：Dubbo、SpringCloud Gateway 底层均基于 Netty（NIO）
- 关联 L2 JVM：NIO 使用堆外内存（DirectByteBuffer），绕过 JVM GC，减少一次内存拷贝

## 应用边界

**BIO**：连接数少、逻辑简单的内部服务；JDBC 连接（连接池复用规避了线程问题）。

**NIO（Netty）**：高并发 TCP 服务器（IM、游戏、API 网关）；业务开发不直接用原生 NIO，用 Netty 封装。

**AIO**：文件批处理、日志归档等 IO 密集型非网络场景（生产中极少见）。
