# 高可用设计

**一句话定义**：高可用（HA）是通过冗余、故障隔离和自动恢复手段，使系统在部分组件失效时仍能持续提供服务，量化指标为 SLA（可用性等级）。

---

## 第一性原理

单点必然失败——硬件故障、软件 Bug、网络抖动、机房断电都是时间问题。高可用的核心洞察是：**故障不可避免，但可以通过冗余让故障不可见**。

代价：冗余意味着多份资源投入，同时引入一致性问题（多副本数据如何同步）和复杂性问题（如何检测故障、如何自动切换）。高可用设计的本质是在**成本、复杂度、可用性**三者之间的取舍。

---

## 核心机制

### SLA（服务等级协议）

SLA 量化可用性承诺，核心指标是年度停机时间：

| SLA | 年停机时间 | 适用场景 |
|-----|----------|---------|
| 99%（2个九）| 87.6 小时/年 | 内部工具 |
| 99.9%（3个九）| 8.76 小时/年 | 一般 ToB 系统 |
| **99.99%（4个九）**| **52.56 分钟/年** | **互联网核心服务** |
| 99.999%（5个九）| 5.26 分钟/年 | 电信/金融核心 |

**MTTR**（平均恢复时间）和 **MTBF**（平均故障间隔时间）是 SLA 的底层指标：`可用性 = MTBF / (MTBF + MTTR)`，提高 HA 要么减少故障频率，要么缩短恢复时间。

### 备份模式

| 模式 | 状态 | 切换时间 | 成本 | 数据一致性 | 适用 |
|------|------|---------|------|-----------|------|
| **冷备** | 离线（不运行）| 分钟~小时 | 低 | 可能丢数据（上次备份后的变更）| 归档、非关键数据 |
| **暖备** | 定期同步，不实时 | 分钟级 | 中 | 轻微数据损失 | 中等重要性服务 |
| **热备** | 实时同步，随时可切 | 秒级（自动）| 高 | 强一致（主从实时复制）| 核心数据库、核心服务 |

热备典型实现：MySQL 主从（异步/半同步复制）、Redis Sentinel（自动故障转移）。

### 异地多活

**同城双活/两地三中心/三地五中心**是从低到高的容灾等级：

```
同城双活：两个机房，距离 < 50km，光纤延迟 < 1ms，数据同步简单
异地多活：多个城市，延迟 5~30ms，需要设计数据分区和流量路由

异地多活架构要素：
┌─────────────────────────────────────────┐
│  DNS/CDN 层：就近路由用户到最近机房         │
│  流量分发：城市粒度（北京用户→北京机房）     │
│  数据层：各机房有完整副本（分布式DB/MQ）     │
│  同步层：异步双向同步，冲突仲裁策略          │
│  监控层：跨机房心跳检测，秒级故障感知        │
└─────────────────────────────────────────┘
```

**挑战**：
- 跨机房写冲突（两个机房同时修改同一数据）→ 业务分区（每个用户固定归属某机房）
- 跨机房延迟（5~30ms）→ 强一致要求的场景（如支付扣款）仍需单机房内事务
- 网络分区（脑裂）→ CAP 定理约束，选 AP 需接受短暂不一致

**单元化架构（异地多活的终极形态）**：

将系统按用户维度切分为若干个**自包含的单元**，每个单元独立部署完整的服务栈（代码+DB+缓存+MQ），用户请求在单元内闭环，**不跨单元调用**：
```
上海单元（杭州/上海用户）   北京单元（北京/东北用户）   新加坡单元（海外用户）
  ├── 交易服务               ├── 交易服务               ├── 交易服务
  ├── 支付服务               ├── 支付服务               ├── 支付服务
  ├── 用户 DB（子集）         ├── 用户 DB（子集）         ├── 用户 DB（子集）
  └── Redis/MQ               └── Redis/MQ               └── Redis/MQ
                  中心（全量数据聚合，兜底路由）
```
优势：消除跨单元网络延迟，天然容灾（一个单元故障不影响其他单元），弹性扩容（新增单元即扩容）。
代价：全链路改造成本极高，数据只是全量的子集（用户路由必须准确），只在超大规模场景（阿里双11量级）值得投入。

### 高可用架构设计要点

```
1. 消除单点（SPOF）
   - 服务层：多实例 + 负载均衡（Nginx/K8s Service）
   - 数据层：主从复制 + 自动故障转移（Sentinel/Cluster）
   - 网关层：多节点 + VIP 漂移（Keepalived + VRRP）

2. 故障隔离（防止级联崩溃）
   - 限流：保护下游资源（见 [[概念-限流与熔断]]）
   - 熔断：快速失败，不阻塞线程池
   - 降级：故障时返回简化响应/缓存数据/默认值
   - 舱壁（Bulkhead）：为不同服务分配独立线程池，互不影响

3. 弹性伸缩
   - 水平扩展（Scale Out）：无状态服务自动扩缩容（K8s HPA）
   - 垂直扩展（Scale Up）：提升单机配置，有上限
   - 预扩容：大促前提前扩容，不依赖自动触发

4. 监控与自愈
   - 秒级指标采集（Prometheus + Grafana）
   - 分布式链路追踪（SkyWalking/Zipkin）
   - 自动重启/摘流（K8s Probe + Readiness/Liveness）
   - 告警分级：P0/P1/P2，配合 On-call 轮值
```

### 压测与全链路压测

**压测目的**：在流量峰值前找到系统瓶颈（不是验证正常功能）。

**压测关键步骤**：
```
1. 定目标：峰值 QPS × 1.5 倍容量（留缓冲）
2. 建模型：正常流量比例（首页:搜索:下单 = 60:30:10）
3. 准脚本：JMeter / Gatling 模拟真实访问路径
4. 执行：阶梯加压（10%→20%→50%→100%→120%）
5. 监控：QPS、RT（P99）、失败率、CPU、内存、GC、DB 连接数
6. 分析：找到第一个瓶颈点 → 优化 → 再压
```

**全链路压测**（生产环境压测）：

普通压测在测试环境误差大（DB 数据量、缓存命中率、网络拓扑都不同）；全链路压测直接在生产环境模拟真实流量。

核心技术：
```
流量染色：压测请求打上 stress_tag 标记，贯穿全链路传递
影子表：压测数据写入 order_shadow、user_shadow 等影子表，不污染真实数据
影子队列：MQ 消息路由到影子 Topic，不影响真实消费
影子缓存：Redis 使用独立 key 前缀（shadow:）隔离
```

**扩容不是线性的**——10 台单机 300 QPS ≠ 集群 3000 QPS：
- DB 是共享瓶颈：所有实例连同一个库，连接数有上限
- 缓存竞争：大量实例同时打 Redis，网络成为瓶颈
- 负载不均：Hash 路由偏斜导致部分节点热点
- 服务依赖：下游接口有独立的 QPS 限制

## 关键权衡

1. **热备成本 vs 冷备成本**：热备资源利用率低（备机平时空转），成本是冷备的 2×；可用性要求 99.99% 以上才值得热备，否则暖备是更经济的选择
2. **异地多活 vs 同城双活**：异地多活能抵御城市级灾难但设计复杂（数据一致性难、延迟高）；大多数公司同城双活已满足需求，异地多活是大厂才会大规模实施的
3. **全链路压测 vs 测试环境压测**：生产压测最准确但有风险（影子表漏配会污染数据）；需要严格的压测平台支撑，中小团队通常用预发环境 + 生产只做小流量验证
4. **自动故障转移 vs 手动切换**：全自动切换（Sentinel 模式）有脑裂风险（哨兵误判触发不必要的主从切换）；金融场景有时选择半自动（告警+人工确认）以避免误操作

## 与其他概念的关系

- **[[概念-限流与熔断]]**：限流+熔断是高可用在流量控制层面的具体实现；高可用架构的"故障隔离"依赖熔断器
- **[[机制-Redis集群与高可用]]**：哨兵模式和 Cluster 模式是存储层高可用的具体实现，包含自动故障转移和脑裂防护
- **[[机制-微服务与SpringCloud]]**：Gateway+LoadBalancer+Sentinel 是应用层高可用的基础设施；Eureka/Nacos 的 AP 设计本身就是服务发现层的高可用取舍
- **[[机制-容器化与Docker]]**：K8s Deployment 副本控制 + HPA 自动扩缩 + Liveness/Readiness Probe 是容器层高可用的实现；Pod 反亲和性规则确保副本分布在不同节点/机房

## 应用边界

**需要高可用设计的场景**：用户直接访问的核心链路（下单、支付、登录）、数据不可丢失的存储层、SLA 要求 ≥ 99.9% 的服务。

**可以简化的场景**：内部工具、数据分析管线（允许小时级停机）、非实时批处理任务（失败可重跑）。

**容灾 vs 备份区别**：容灾是自动故障转移（秒级~分钟级），备份是数据副本恢复（小时~天级）。生产系统两者都需要，不可互相替代。
