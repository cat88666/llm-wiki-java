# 操作系统基础

**一句话定义**：操作系统通过进程/线程抽象分配 CPU 时间，通过虚拟内存抽象管理内存，通过用户态/内核态隔离保证安全，并由 [[概念-IO模型]] 提供高效的资源访问。

---

## 第一性原理

硬件资源（CPU、内存、磁盘、网卡）是有限且危险的——多个程序同时使用可能互相干扰甚至崩溃系统。操作系统的核心角色是**资源的抽象与隔离**：

- **CPU 抽象**：通过时间片轮转，让多个程序"看起来"并发执行。
- **内存抽象**：通过虚拟地址空间，每个进程以为自己独占内存。

---

## 核心机制

### 进程 / 线程 / 协程

| 维度 | 进程 | 线程 | 协程 |
|------|------|------|------|
| 资源分配 | 独立资源（内存、文件句柄）| 共享进程资源 | 共享线程资源 |
| 调度主体 | OS 调度 | OS 调度 | 用户态调度（OS 不感知）|
| 切换开销 | 大（上下文完全切换）| 中（保存/恢复寄存器）| 小（用户态保存少量状态）|
| 隔离性 | 强（崩溃不影响其他进程）| 弱（线程崩溃可影响进程）| 弱 |
| Java 支持 | JVM 进程 | Thread/线程池 | 虚拟线程（JDK 21，Project Loom）|

**为什么需要协程**：多线程的频繁上下文切换和锁竞争消耗 CPU；协程由用户自决调度时机，不阻塞 OS 线程，特别适合 IO 密集型高并发场景（Golang goroutine 是典型实现）。

### 用户态 / 内核态

```
用户态（Ring 3）：应用程序运行空间，只能访问授权资源，不可操作硬件
      ↕ 系统调用（syscall）/ 中断 / 异常
内核态（Ring 0）：OS 内核运行空间，可访问所有硬件资源

切换过程：
  用户程序发起 syscall → CPU 保存寄存器状态 → 切换到内核态
  → 执行内核逻辑 → 切换回用户态 → 恢复寄存器 → 继续执行
```

**性能损耗**：每次系统调用都经历两次上下文切换。频繁的小 IO 调用会导致性能瓶颈，这也是 [[概念-IO模型#零拷贝 (Zero Copy)]] 优化的动机。

### 多级缓存与 MESI 协议

```
CPU 执行速度 >> 内存读写速度（差距 100x~1000x）
解决方案：多级缓存 (L1/L2/L3)
```

**MESI 缓存一致性协议**：
- **M (Modified)**：已修改，与内存不一致。
- **E (Exclusive)**：独占，与内存一致。
- **S (Shared)**：共享，多核有副本。
- **I (Invalid)**：无效，需重新加载。

**MESI 与 JMM**：Java `volatile` 关键字底层通过 MESI 协议 + 内存屏障实现，保证多线程可见性。

### 进程调度算法

| 算法 | 核心思路 | 优点 | 缺点 |
|------|---------|------|------|
| FCFS | 先到先得 | 公平简单 | 长任务阻塞短任务 |
| 时间片轮转 (RR) | 固定时间片轮流分配 | 公平，无饥饿 | 频繁上下文切换 |
| CFS (Linux 默认) | 基于虚拟运行时间+红黑树 | 高公平，低延迟 | 实现复杂 |

### 内存管理：分页与分段

- **分页 (Paging)**：将物理内存划分为固定大小页（4KB），通过页表映射。解决内存碎片。
- **分段 (Segmentation)**：按逻辑段（代码、数据、栈）划分。方便权限管理和共享。
- **TLB (快表)**：硬件缓存热点页表项，加速地址转换。

### 进程间通信 (IPC)

| 方式 | 特点 | 适用场景 |
|------|------|---------|
| 管道 (Pipe) | 半双工，父子进程间 | Shell 命令组合 |
| 共享内存 | 最快（无拷贝），需加锁 | 大量数据高效共享 |
| Socket | 跨主机，通用 | 网络通信，微服务 |

## 与其他概念的关系

- **[[概念-IO模型]]**：操作系统如何高效处理进程发出的 IO 请求。
- **[[机制-线程池]]**：Java 线程对应 OS 内核线程（1:1 模型），其管理受 OS 调度算法影响。
- **[[概念-JMM]]**：Java 内存模型的硬件基础是多级缓存与 MESI 协议。
