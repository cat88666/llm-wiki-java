# IO模型

> 程序发起 IO 请求后，如何等待数据就绪、如何获取数据——从底层的零拷贝机制、Page Cache 缓存，到 OS 的多路复用模型，再到 Java 的 BIO/NIO/AIO 封装。

## 第一性原理

IO 操作本质上是**数据在硬件设备（磁盘/网卡）、内核空间缓冲区与用户空间缓冲区之间的流动**。
它分为两个阶段：
1. **等待阶段**：等待数据从设备就绪到内核缓冲区。
2. **拷贝阶段**：将数据从内核缓冲区拷贝到用户空间（或反之）。

高性能 IO 的核心目标是：**减少 CPU 等待（异步/多路复用）和减少数据拷贝（零拷贝）**。

---

## 核心机制：底层支撑

### Page Cache (内核页缓存)

操作系统在内存中维护 Page Cache，将磁盘数据缓存为内存页（通常 4KB）：

- **写流程**：应用 `write()` → 写入 Page Cache（脏页）→ 后台异步 flush 到磁盘。
    - *权衡*：默认延迟写（write-back）性能好但有崩溃丢数据风险；`fsync()` 可强制同步（如 MySQL WAL）。
- **读流程**：应用 `read()` → 检查 Page Cache → 命中则直接返回；未命中则触发磁盘读取并预读。

**应用**：Redis AOF everysec 策略就是每秒调用一次 `fsync`。

### 零拷贝 (Zero Copy)

传统 IO 需要 4 次上下文切换 + 4 次数据拷贝。零拷贝旨在消除无意义的 CPU 拷贝。

| 方式 | 原理 | CPU拷贝 | 上下文切换 | 适用场景 |
|------|------|------------|-----------|---------|
| 普通 read+write | 全流程拷贝 | 2 | 4 | 通用 |
| mmap+write | 内核/用户共享内存映射 | 1 | 4 | 随机读写（RocketMQ） |
| sendfile | 描述符直传，DMA 直发网卡 | 0 | 2 | 静态文件传输（Nginx） |
| sendfile+DMA SG | DMA 聚合发送 | 0 | 2 | 最优化文件传输（Kafka） |

**Kafka 高吞吐秘密**：利用 `sendfile` 将磁盘文件直发网卡，完全绕过用户态。

## 核心机制：操作系统 IO 模型

### 五种 OS IO 模型

1. **同步阻塞 IO (BIO)**：调用 `read()` 一直阻塞到数据返回。
2. **同步非阻塞 IO (NIO)**：调用 `read()` 若无数据立即返回，需不断轮询。
3. **IO 多路复用**：单线程监听多个 fd，有就绪才处理（Redis/Nginx 核心）。
4. **信号驱动 IO**：就绪后发信号通知。
5. **异步 IO (AIO)**：发起后立即返回，内核完成后回调（真正的异步）。

### IO 多路复用：select/poll/epoll

| 实现 | fd 上限 | 用户态→内核态拷贝 | 遍历就绪 fd | 时间复杂度 |
|------|--------|----------------|------------|----------|
| select | 1024 | 每次调用都拷贝 | 遍历全部 fd | O(n) |
| poll | 无限制 | 每次调用都拷贝 | 遍历全部 fd | O(n) |
| epoll | 无限制 | 注册时拷贝一次 | 直接返回就绪列表 | O(1) |

**epoll 触发模式**：
- **LT (水平触发)**：只要有数据就一直通知（默认，安全）。
- **ET (边缘触发)**：状态变化才通知一次，效率极高但必须非阻塞且一次读完（Netty 可选）。

## 核心机制：Java IO 封装

Java 对上述模型进行了高级封装：

| 模型 | 特点 | 线程状态 | 适用场景 |
|------|-----|---------|---------|
| **BIO** | 一个连接一个线程 | 全程阻塞 | 连接数少、短连接 |
| **NIO** | Selector 管理多 Channel | 仅在事件查询时阻塞 | 高并发、网络连接密集 |
| **AIO** | 异步回调，OS 负责拷贝 | 完全不阻塞 | IO 耗时长、连接数多 |

**同步 vs 异步**：数据拷贝阶段谁来做？应用自己拷（同步）还是内核拷完通知（异步）。
**阻塞 vs 非阻塞**：等待数据阶段线程能不能干别的？

---

## 关键权衡

1. **NIO vs AIO**：Linux 下 AIO 实现不成熟（底层仍是 epoll 模拟），Netty 最终选择了性能更优且更可控的 NIO。
2. **epoll 的局限**：epoll 对**网络 IO** 极佳，但对**磁盘文件 IO** 无效（磁盘文件总处于就绪状态），磁盘密集型场景需用 `io_uring`。
3. **IO 多路复用 vs 多线程**：
    - **IO 密集型**：选多路复用（单线程管万级连接）。
    - **CPU 密集型**：选多线程（充分利用多核计算）。

## 与其他概念的关系

- **[[概念-操作系统进程线程]]**：IO 涉及用户态与内核态的频繁切换；JVM 线程与内核线程的对应关系。
- **[[机制-线程池]]**：BIO 必须配合线程池，但线程数受限于 OS。
- **[[机制-Netty架构]]**：Netty 是 Java NIO 的工业级实现，封装了 epoll 和零拷贝（堆外内存）。

## 应用边界

- **Kafka/RocketMQ**：深度利用零拷贝和 Page Cache 进行消息持久化与发送。
- **Redis**：单线程 + epoll 支撑了万级 QPS。
- **MySQL**：通过 `fsync` 控制脏页刷盘，平衡数据安全与 IO 性能。
