# 操作系统

> 操作系统抽象并隔离 CPU、内存、磁盘、网络等硬件资源，核心关注进程调度、内存管理、内核边界、并发可见性和资源利用率。

## 一、第一性原理

硬件资源有限且不安全，多个程序直接共享硬件会导致互相破坏。操作系统提供三类核心抽象：

| 抽象 | 屏蔽的问题 | Java 后端关注点 |
| --- | --- | --- |
| 进程/线程 | CPU 如何分配给多个执行流 | 线程数、上下文切换、调度延迟 |
| 虚拟内存 | 每个进程如何“独占”地址空间 | JVM 堆、直接内存、Page Cache、OOM |
| 系统调用 | 应用如何安全访问内核和硬件 | IO、网络、文件、锁、时间、进程管理 |

操作系统不是 Java 程序的背景知识，而是性能问题的下界：线程调度、内存映射、网络 IO、磁盘刷盘最终都落到 OS。

## 二、核心机制

### 进程、线程、协程

| 维度 | 进程 | 线程 | 协程/虚拟线程 |
| --- | --- | --- | --- |
| 资源边界 | 独立地址空间、文件句柄等 | 共享进程资源 | 共享线程/进程资源 |
| 调度者 | OS | OS | 用户态运行时/JVM 调度为主 |
| 切换成本 | 高 | 中 | 低 |
| 隔离性 | 强 | 弱 | 弱 |
| Java 对应 | 一个 JVM 进程 | `Thread`、线程池 | Java 21 virtual thread |

Java 平台线程通常映射到 OS 线程。虚拟线程降低阻塞等待成本，但不降低 CPU 计算成本，也不消除数据库、锁竞争和下游限流。

### 用户态与内核态

| 状态 | 权限 | 典型操作 |
| --- | --- | --- |
| 用户态 | 受限，不能直接操作硬件 | Java 业务代码、普通内存读写 |
| 内核态 | 可访问硬件和内核数据结构 | 网络收发、磁盘 IO、线程调度、内存映射 |

用户态进入内核态依赖系统调用、中断或异常。频繁小 IO、频繁锁竞争、频繁线程切换都会放大用户态/内核态切换成本。

### 虚拟内存、分页、TLB

| 机制 | 作用 | Java 后端影响 |
| --- | --- | --- |
| 虚拟地址空间 | 隔离进程内存 | JVM 认为自己拥有连续地址空间 |
| 分页 | 固定大小页映射物理内存 | 缺页、Page Cache、mmap、swap 都基于页 |
| TLB | 缓存页表映射 | TLB miss 会增加内存访问成本 |
| mmap | 文件映射到虚拟地址空间 | RocketMQ、零拷贝、堆外访问 |

分段更多是逻辑分区和权限管理概念，现代通用 OS 的内存管理以分页为主。

### CPU 缓存与 MESI

| 机制 | 解决什么 | Java 关联 |
| --- | --- | --- |
| L1/L2/L3 Cache | 缓解 CPU 与内存速度差 | 局部性、伪共享、缓存行 |
| MESI | 多核缓存一致性 | volatile 可见性的硬件基础之一 |
| 内存屏障 | 限制重排序和刷新顺序 | JMM、volatile、锁实现 |

不要把 MESI 简化成 volatile 的全部实现。Java 可见性由 JMM 语义定义，底层由缓存一致性协议、内存屏障和编译器/CPU 约束共同支撑。

### 调度、IPC 与资源限制

| 机制 | 关键点 | 工程含义 |
| --- | --- | --- |
| CFS 调度 | Linux 依据虚拟运行时间追求公平 | 线程过多会增加调度开销 |
| IPC | 管道、共享内存、Socket、信号 | 微服务常用 Socket，本机高性能可用共享内存 |
| 文件句柄 | 进程可打开 fd 有上限 | 连接数、文件数、epoll fd 都受限制 |
| cgroup | 限制 CPU、内存、IO | 容器内看到的资源可能不同于宿主机 |

## 三、Java 核心使用

| 场景 | OS 机制 | Java 关注点 |
| --- | --- | --- |
| 线程池 | OS 线程调度 | 核心线程数、队列、阻塞比例、上下文切换 |
| 网络 IO | Socket、epoll、系统调用 | NIO、Netty、连接数、fd 限制 |
| 文件 IO | Page Cache、fsync、mmap | 日志刷盘、WAL、零拷贝、DirectBuffer |
| 内存管理 | 虚拟内存、分页、swap | JVM 堆、堆外内存、容器内存限制 |
| 并发可见性 | CPU Cache、MESI、屏障 | JMM、volatile、锁、伪共享 |

Java 应用排查性能问题时，不能只看 JVM 指标。至少要同时看 CPU、Load、内存、磁盘 IO、网络、fd、线程数和 GC。

## 四、核心使用原则

1. **线程数不是越多越好**：CPU 密集型接近核心数；IO 密集型看阻塞比例和下游容量。
2. **Load 高不等于 CPU 高**：不可中断 IO、锁等待、磁盘慢都可能推高 Load。
3. **少做小而频繁的系统调用**：批量读写、缓冲、零拷贝能显著降低切换成本。
4. **容器环境必须看 cgroup**：JVM 参数、线程数、堆大小要按容器限制估算。
5. **避免 swap**：Java 服务一旦频繁 swap，延迟通常会出现长尾抖动。

## 五、使用案例

| 系统 | 依赖的 OS 能力 | 关键点 |
| --- | --- | --- |
| Tomcat | Socket、线程调度 | NIO connector + 工作线程池 |
| Netty | epoll/kqueue、DirectBuffer | 事件循环不能阻塞，堆外内存要可控 |
| Redis | epoll、单线程事件循环 | 网络 IO 多路复用，命令执行串行 |
| MySQL | Page Cache、fsync、文件锁 | WAL、刷盘策略、IO 延迟 |
| Kafka | Page Cache、sendfile | 顺序写、零拷贝、磁盘吞吐 |
| JVM | 虚拟内存、线程、信号 | GC、线程 dump、Native Memory |

## 六、综合对比

| 对比 | 结论 |
| --- | --- |
| 进程 vs 线程 | 进程隔离强，线程共享资源但崩溃影响整个进程 |
| 线程 vs 虚拟线程 | 线程由 OS 调度，虚拟线程由 JVM 调度到平台线程，适合阻塞等待 |
| CPU 利用率 vs Load | CPU 看忙不忙，Load 看运行队列和不可中断等待压力 |
| 分页 vs 分段 | 分页解决物理内存管理，分段偏逻辑组织和权限 |
| 用户态 vs 内核态 | 用户态安全受限，内核态能力强但切换有成本 |
| 堆内存 vs 堆外内存 | 堆受 GC 管理，堆外适合 IO 但更容易泄漏和难排查 |

## 七、生产风险

| 风险 | 表现 | 排查入口 |
| --- | --- | --- |
| CPU 飙高 | 响应慢、线程占满核心 | `top`、`pidstat`、`jstack`、火焰图 |
| Load 高但 CPU 不高 | 请求长尾、磁盘/网络等待 | `iostat`、`vmstat`、线程状态 |
| 线程爆炸 | 上下文切换高、内存上涨 | 线程 dump、线程池指标 |
| fd 耗尽 | 新连接失败、Too many open files | `ulimit -n`、`lsof` |
| 直接内存泄漏 | Direct buffer OOM | Native Memory Tracking、Netty leak detector |
| 容器 OOM | Pod 被 kill | cgroup memory、JVM MaxRAMPercentage |

## 八、与其他概念的关系

- **[[概念-IO模型]]**：IO 模型建立在系统调用、fd、Page Cache、epoll 等 OS 能力上。
- **[[机制-线程池]]**：Java 线程池最终受 OS 线程调度、CPU 核数和上下文切换约束。
- **[[概念-JMM]]**：JMM 的可见性和有序性需要 CPU Cache、MESI、内存屏障支撑。
- **[[概念-计算机网络]]**：网络协议栈在内核中实现，Java Socket/NIO 通过系统调用使用它。

## 九、应用边界

- 操作系统知识不能替代 JVM 调优，二者需要结合看。
- CFS、epoll、cgroup 等细节与 Linux 强相关，跨平台时要确认实现差异。
- CPU、内存、磁盘、网络任何一个达到瓶颈，都可能表现为 Java 应用慢。
