> 计算机网络用分层协议把跨机器通信拆成寻址、传输、加密、应用语义和流量治理问题，Java 后端重点关注 TCP/UDP、HTTP/HTTPS、DNS、代理、认证和故障边界。

## 一、第一性原理

网络通信要解决五个问题：

| 问题 | 典型协议/机制 | Java 后端关注点 |
| --- | --- | --- |
| 对方是谁 | IP、DNS、端口 | 域名解析、服务发现、端口暴露 |
| 怎么到达 | 路由、网关、NAT | 跨机房、跨 VPC、网络策略 |
| 是否可靠 | TCP、重传、滑动窗口 | 超时、重试、连接复用 |
| 是否安全 | TLS、证书、加密套件 | HTTPS、证书过期、双向认证 |
| 如何表达业务 | HTTP、RPC、Cookie、Token | 状态码、Header、序列化、鉴权 |

分层的价值是隔离复杂度：IP 不保证可靠，TCP 在其上补可靠性，HTTP 在 TCP/TLS 之上定义应用语义。

## 二、核心机制

### 网络分层

| 层次 | 解决问题 | 典型协议 |
| --- | --- | --- |
| 应用层 | 业务语义 | HTTP、DNS、SMTP、RPC |
| 传输层 | 端到端传输 | TCP、UDP、QUIC |
| 网络层 | 寻址和路由 | IP、ICMP |
| 链路层 | 局域网传输 | Ethernet、Wi-Fi、ARP |

OSI 七层适合面试表达，工程实践通常按 TCP/IP 四层理解。

### TCP 与 UDP

| 维度 | TCP | UDP |
| --- | --- | --- |
| 连接 | 面向连接 | 无连接 |
| 可靠性 | 序号、ACK、重传、流控、拥塞控制 | 不保证可靠、有序、不重传 |
| 开销 | 握手、状态维护、头部更重 | 轻量 |
| 典型场景 | HTTP/1.1、HTTP/2、MySQL、Redis、RPC | DNS、直播、游戏、QUIC |

TCP 追求可靠有序，UDP 只提供最小传输能力。QUIC 在 UDP 之上实现加密、可靠传输和多路复用。

### TCP 握手、挥手与可靠传输

| 机制 | 目的 | 高频结论 |
| --- | --- | --- |
| 三次握手 | 确认双方收发能力，建立初始序列号 | 两次握手无法确认客户端能收，也难处理历史 SYN |
| 四次挥手 | 关闭全双工连接 | ACK 和 FIN 可能不能合并，因为一方可能还有数据要发 |
| TIME_WAIT | 等待旧报文失效，保证最后 ACK 可重传 | 主动关闭方进入，过多会消耗端口和连接资源 |
| 序列号/ACK | 保证有序和确认 | TCP 是字节流，不保留消息边界 |
| 滑动窗口 | 接收方流控 | 防止接收端被打爆 |
| 拥塞控制 | 网络侧限速 | 慢启动、拥塞避免、快重传、快恢复 |

### 粘包与拆包

TCP 是字节流协议，没有“消息”概念。业务协议必须自己定义边界：

| 方案 | 适用场景 | 典型例子 |
| --- | --- | --- |
| 固定长度 | 协议简单、字段固定 | 二进制定长协议 |
| 分隔符 | 文本协议 | Redis RESP、换行协议 |
| 长度字段 | 通用、可靠 | Dubbo、Netty LengthFieldBasedFrameDecoder |
| TLV | 可扩展协议 | Type-Length-Value 自定义协议 |

### HTTP、HTTPS、QUIC

| 协议 | 核心能力 | 主要问题/边界 |
| --- | --- | --- |
| HTTP/1.1 | Keep-Alive、Header、状态码 | 应用层队头阻塞，请求响应按序 |
| HTTP/2 | 二进制分帧、多路复用、HPACK | 仍受 TCP 队头阻塞影响 |
| HTTP/3 | QUIC、流级多路复用、0-RTT | 实现复杂，中间设备兼容成本 |
| HTTPS | TLS 加密、身份认证、完整性 | 证书、握手、加密套件和续期管理 |

TLS 不是只做加密，还提供身份认证和防篡改。生产中证书过期、域名不匹配、TLS 版本不兼容都可能导致故障。

### DNS、代理、CDN、跨域

| 机制 | 作用 | Java 后端关注点 |
| --- | --- | --- |
| DNS | 域名到 IP | TTL、缓存、解析故障、GeoDNS |
| 正向代理 | 代理客户端访问外部 | 客户端出口、企业代理 |
| 反向代理 | 代理服务端接收流量 | Nginx、网关、负载均衡 |
| CDN | 静态资源边缘缓存 | 回源、缓存失效、命中率 |
| CORS | 浏览器跨域放行机制 | Gateway/服务端响应头配置 |

跨域是浏览器安全策略，不是 TCP/HTTP 协议本身不通。

## 三、Java 核心使用

| 场景 | 常见技术 | 关键点 |
| --- | --- | --- |
| Web 服务 | Servlet、Spring MVC、Tomcat | 连接池、线程池、超时、Header |
| HTTP 客户端 | HttpClient、OkHttp、Feign、RestTemplate | 连接复用、连接/读/写超时、重试 |
| RPC | Dubbo、gRPC、OpenFeign | 长连接、序列化、负载均衡、熔断 |
| 网关 | Spring Cloud Gateway、Nginx | 路由、CORS、TLS、限流、转发头 |
| 登录态 | Cookie、Session、JWT | 安全、跨域、续期、主动失效 |

Java 网络问题排查时，优先确认：域名解析、连接建立、TLS 握手、连接池、超时配置、下游响应时间。

## 四、核心使用原则

1. **所有远程调用必须设置超时**：连接超时、读超时、写超时要分开配置。
2. **重试必须配合幂等**：POST、扣款、下单不能无脑重试。
3. **连接池要有限制**：最大连接数、每路由连接数、空闲回收都要可控。
4. **长连接不等于永不关闭**：要处理 idle timeout、连接探活和服务端主动断开。
5. **DNS 不是永远稳定**：关注 JVM DNS cache、TTL、容器 DNS 和服务发现刷新。
6. **TLS 也会成为故障源**：证书过期、CA 变更、协议版本、SNI 都要纳入巡检。

## 五、使用案例

| 场景 | 链路 | 关键点 |
| --- | --- | --- |
| 浏览器访问 HTTPS 页面 | DNS -> TCP -> TLS -> HTTP -> 服务端处理 -> 响应渲染 | 每一段都可能慢 |
| Spring Cloud 服务调用 | Feign/Gateway -> HTTP -> LB -> 下游服务 | 超时、重试、熔断、连接池 |
| Dubbo/gRPC 调用 | 长连接 + 二进制协议 | 连接保活、序列化、背压 |
| 静态资源访问 | DNS/GeoDNS -> CDN -> 回源 | 缓存命中率、刷新、回源保护 |
| 登录认证 | Cookie/Session/JWT | 续期、注销、跨域、安全属性 |

## 六、综合对比

| 对比 | 结论 |
| --- | --- |
| TCP vs UDP | TCP 可靠有序，UDP 轻量；实时性和自定义可靠传输可选 UDP/QUIC |
| HTTP/1.1 vs HTTP/2 | HTTP/2 解决应用层队头阻塞和连接复用，但仍受 TCP 丢包影响 |
| HTTP/2 vs HTTP/3 | HTTP/3 用 QUIC 解决 TCP 队头阻塞，并优化握手 |
| Cookie vs Session vs Token | Cookie 是载体，Session 服务端有状态，Token 更适合跨端和微服务 |
| 正向代理 vs 反向代理 | 正向代理客户端，反向代理服务端 |
| 连接超时 vs 读超时 | 连接超时是建立连接慢，读超时是对端处理或传输慢 |

## 七、生产风险

| 风险 | 表现 | 处理原则 |
| --- | --- | --- |
| 超时风暴 | 上游大量等待，下游被重试打爆 | 分层超时、熔断、限流、隔离 |
| 连接池耗尽 | 请求排队、线程堆积 | 限制并发、合理池大小、监控 leased/pending |
| TIME_WAIT 过多 | 端口耗尽、连接失败 | 连接复用、排查主动关闭方、调整内核参数需谨慎 |
| DNS 故障 | 某些实例不可达或解析慢 | 缓存策略、重试不同 IP、服务发现兜底 |
| TLS 证书问题 | HTTPS 握手失败 | 证书巡检、自动续期、SNI/CA 校验 |
| 跨域配置错误 | 浏览器报 CORS，服务端实际可达 | 网关统一 CORS，区分预检请求 |
| 粘包拆包 | RPC 解码异常 | 明确协议边界，使用成熟 codec |
