# 计算机网络

**一句话定义**：计算机网络通过分层协议栈（OSI/TCP-IP）屏蔽底层差异，TCP/IP 是互联网的核心协议族，HTTP 是 Web 应用的应用层协议，身份认证（Cookie/Session/Token）和内容分发（CDN/代理）是应用层的常见扩展。

---

## 第一性原理

网络通信的核心问题：两台物理上不相邻的机器如何可靠地交换数据？

答案是**分层**——把复杂问题拆成独立的层，每层只解决一类问题，通过接口向上层提供服务。每一层都是对下层"不可靠性"的补偿：物理层只管传比特，IP 层只管寻址路由，TCP 层补偿 IP 的不可靠性实现有序可靠传输，HTTP 在 TCP 之上定义应用层语义。

---

## 核心机制

### OSI 七层 vs TCP/IP 模型

```
OSI 七层              TCP/IP 四层（实践中使用）
─────────────         ─────────────
7. 应用层             4. 应用层（HTTP/FTP/DNS/SMTP）
6. 表示层          ↗
5. 会话层
─────────────         ─────────────
4. 传输层             3. 传输层（TCP/UDP）
─────────────         ─────────────
3. 网络层             2. 网络层（IP/ICMP/ARP）
─────────────         ─────────────
2. 数据链路层         1. 网络接口层（以太网/Wi-Fi）
1. 物理层          ↗
```

每层只与相邻层交互，数据从应用层逐层封装（加 Header）向下传，到达目标机器后逐层解封装向上传。

### TCP vs UDP

| 维度 | TCP | UDP |
|------|-----|-----|
| 连接 | 面向连接（三次握手+四次挥手）| 无连接 |
| 可靠性 | 可靠（序号、确认、重传）| 不保证（可能丢包、乱序）|
| 速度 | 慢（握手+确认开销）| 快（无额外开销）|
| 头部 | 20~60 字节 | 固定 8 字节 |
| 拥塞控制 | 有（慢启动/拥塞避免）| 无 |
| 适用场景 | Web/邮件/文件传输 | 视频流/游戏/VoIP/DNS |

### TCP 三次握手与四次挥手

**三次握手**（建立连接，验证双方收发能力）：
```
客户端                       服务端
   │── SYN(seq=x) ──────────→│  ← 客户端确认：自己能发
   │← SYN+ACK(seq=y,ack=x+1)─│  ← 服务端确认：双方都能收发
   │── ACK(ack=y+1) ──────────→│  ← 服务端最终确认客户端能收
```
**为什么不能两次**：两次握手时服务端无法确认客户端收到了自己的 SYN，无法防止历史过期连接（旧 SYN 复用导致无效连接）。

**四次挥手**（释放连接）：
```
客户端                       服务端
   │── FIN ────────────────→│  客户端不再发数据
   │← ACK ──────────────────│  服务端确认（但可能还有数据要发）
   │← FIN ──────────────────│  服务端也没数据发了
   │── ACK ────────────────→│  客户端确认，等待 2MSL 后关闭
```
**为什么四次不能三次**：TCP 全双工，服务端的 ACK 和 FIN 不能合并——服务端收到 FIN 后可能还有数据要继续发送，需要等发完再 FIN。

**TIME_WAIT（等待 2MSL）**：防止旧连接的延迟报文影响新连接；确保最后一个 ACK 到达对端（如果 ACK 丢失，服务端会重发 FIN，客户端需要能够重新 ACK）。

### TCP 可靠传输机制

| 机制 | 作用 |
|------|------|
| 序列号 + 排序 | 保证数据有序到达，重组乱序数据 |
| 校验和 | 检测传输中的数据变化 |
| 超时重传（ARQ）| 发送后启动计时器，超时未收到 ACK 则重传 |
| 滑动窗口（流量控制）| 接收方通告可接收的窗口大小，限制发送速率 |
| 拥塞控制 | 感知网络状态，主动降低发送速率 |

**拥塞控制四阶段**：
```
cwnd（拥塞窗口）
  │
  │  慢启动：cwnd 指数增长（每收到1 ACK +1 MSS）
  │          ↓ 达到 ssthresh（慢启动阈值）
  │  拥塞避免：cwnd 线性增长（每 RTT +1 MSS）
  │            ↓ 丢包（超时 or 3重复ACK）
  │
  │  超时：ssthresh = cwnd/2，cwnd 重置为 1，重新慢启动
  │  快重传：收到3个重复ACK → 立即重传丢失包（不等超时）
  │  快恢复：ssthresh = cwnd/2，cwnd = ssthresh，直接进拥塞避免
```

### TCP 粘包 / 拆包

TCP 是**面向字节流**的协议，没有消息边界概念——发送方的多个小包可能被合并发送（粘包），一个大包也可能被分片（拆包）。

**解决方案**（协议层约定边界）：
1. **固定长度**：每个消息固定 N 字节，不足补空格
2. **分隔符**：消息末尾加 `\n` 或特定分隔符
3. **Header + Body**：Header 中记录 Body 长度（HTTP/Dubbo 协议均用此方式）
4. **自定义协议**：TLV（Type-Length-Value）格式

### HTTP 版本演进

| 版本 | 关键特性 | 问题 |
|------|---------|------|
| HTTP/1.0 | 每次请求新建 TCP 连接 | 大量握手开销 |
| HTTP/1.1 | Keep-Alive 连接复用；管道化（可并发发请求）| HTTP 队头阻塞（响应必须按序）|
| **HTTP/2** | **二进制分帧 + 多路复用**（乱序帧，解决 HTTP 队头阻塞）；HPACK 头部压缩；Server Push | TCP 队头阻塞仍存在 |
| **HTTP/3** | **QUIC（基于 UDP）**；流级别独立可靠传输（消除 TCP 队头阻塞）；0-RTT；强制 TLS 1.3 | 实现复杂；中间设备兼容性 |

**HTTP/2 的队头阻塞问题**：HTTP/2 共用单一 TCP 连接，TCP 层某个数据包丢失会阻塞该连接上**所有流**——多路复用反而放大了 TCP 队头阻塞的影响。HTTP/3 通过 QUIC 在 UDP 上实现流级别的独立传输，彻底消除这一问题。

**为什么不升级 TCP 而是用 UDP**：TCP 在操作系统内核实现，升级需要所有中间设备和操作系统同步更新（"协议僵化"问题）；UDP 是通用协议，可以在应用层基于 UDP 实现自定义可靠传输（QUIC）。

### HTTPS 与 TLS 握手

```
HTTP  = TCP + 明文传输（端口 80）
HTTPS = TCP + TLS/SSL 加密（端口 443）

HTTPS 建立连接过程：
1. TCP 三次握手（建立传输层连接）
2. TLS 握手（建立加密通道）：
   - TLS 1.2：2 RTT（4 步：ClientHello → ServerHello+证书 → 客户端验证+密钥交换 → Finished）
   - TLS 1.3：1 RTT（2 步：ClientHello+密钥共享 → ServerHello+证书+Finished）
   - QUIC（HTTP/3）：0-RTT 复连（已有会话票据时可跳过握手）
```

**对称加密 vs 非对称加密**：
- 对称加密（AES/DES）：同一密钥加解密，速度快；密钥传输是安全弱点
- 非对称加密（RSA/ECDHE）：公钥加密、私钥解密，速度慢但安全；用于 TLS 握手阶段交换对称密钥
- 实际做法：非对称加密协商出**会话密钥**，后续通信用对称加密（性能好）

### DNS（域名解析）

```
用户输入 www.taobao.com
  ↓
浏览器缓存 → OS hosts 文件 → 本地 DNS（路由器/ISP）→ 根域名服务器
  → .com 顶级域名服务器 → taobao.com 权威 DNS 服务器 → 返回 IP
```

DNS 是分布式层级数据库，各级缓存结果（TTL 控制缓存时间）减少查询延迟。

### Cookie / Session / Token

| 机制 | 存储位置 | 安全性 | 大小限制 | 跨域 |
|------|---------|--------|---------|------|
| Cookie | 客户端浏览器 | 低（明文可读）| 4KB/domain | 不支持 |
| Session | 服务器内存/Redis | 高 | 无限制 | 不支持（依赖 Cookie 传 ID）|
| Token（JWT）| 客户端（Header/localStorage）| 中（签名防篡改）| 较大（携带声明）| 支持 |

**选型场景**：
- 传统 Web 应用：Session（服务端可控，可强制失效）
- 前后端分离/移动端/微服务：Token（无状态，不依赖 Cookie，跨域友好）
- 分布式集群：避免 Session（需要 Redis 共享）或直接用 Token

### 正向代理 vs 反向代理

| 维度 | 正向代理 | 反向代理 |
|------|---------|---------|
| 代理对象 | 代理客户端 | 代理服务端 |
| 对哪方透明 | 对服务端透明（服务端不知道真实客户端）| 对客户端透明（客户端不知道真实服务器）|
| 典型应用 | VPN/科学上网 | Nginx 负载均衡 / CDN |

**CDN（内容分发网络）**：在全球部署边缘节点，将静态资源（图片/JS/CSS）缓存到离用户最近的节点，减少源服务器压力和网络延迟。用户首次请求时边缘节点回源，后续请求直接命中边缘缓存。

### 跨域（CORS）

**同源策略**：浏览器要求脚本只能访问**协议+域名+端口**三者完全相同的资源，不同则跨域报错。

**解决方案**：
1. **CORS**（推荐）：服务端响应头加 `Access-Control-Allow-Origin`，浏览器放行
2. **JSONP**：利用 `<script>` 标签不受同源限制，仅支持 GET
3. **反向代理**：前端同域代理转发，Nginx 配置 `proxy_pass`

### 浏览器输入 URL 后发生了什么

```
1. URL 解析 + 浏览器缓存检查
2. DNS 解析（域名 → IP）
3. TCP 三次握手（建立连接）
4. TLS 握手（如果是 HTTPS）
5. 发送 HTTP 请求（GET/POST）
6. 服务器处理（Nginx 负载均衡 → Servlet容器 → MVC → Service → DB）
7. 返回 HTTP 响应（状态码 + 响应体）
8. 浏览器缓存响应、解析 HTML
9. 渲染页面（DOM → CSSOM → Render Tree → Layout → Paint）
```

## 关键权衡

1. **TCP 可靠性的代价**：三次握手+四次挥手+重传机制引入额外延迟，每次请求的连接建立最少 1.5RTT，对实时性要求高的场景（游戏/视频）用 UDP 更合适
2. **HTTP/2 多路复用 vs HTTP/1.1 多连接**：HTTP/2 单连接多路复用减少握手开销，但单连接 TCP 队头阻塞影响所有流；HTTP/1.1 通过 6 个并行连接绕开排队，但连接开销大
3. **Session vs Token**：Session 服务端可强制失效（用于高安全场景如银行）；Token 无状态（轻松水平扩展），但无法主动失效（需要短 TTL+黑名单机制）
4. **HTTPS 性能开销**：TLS 握手增加 1~2 RTT 延迟，加解密增加 CPU 开销；TLS 1.3 将握手从 2RTT 降至 1RTT，Session Resumption 实现 0-RTT 复连，现代硬件的加解密开销可忽略不计

## 与其他概念的关系

- **[[概念-IO模型]]**：NIO/AIO 的网络 IO 多路复用（epoll）是 Netty 和高性能 HTTP 服务器的底层，是传输层可靠传输之上的应用层优化
- **[[机制-微服务与SpringCloud]]**：HTTP/HTTPS 是 OpenFeign/Gateway 的传输协议；长连接（Keep-Alive）是 Dubbo/gRPC 等 RPC 框架的基础；CORS 是 Gateway 的必要配置
- **[[概念-高可用设计]]**：DNS 负载均衡（DNS 轮询/GeoDNS）是异地多活的流量入口；CDN 是静态资源高可用的基础设施；反向代理（Nginx）是负载均衡的实现层
