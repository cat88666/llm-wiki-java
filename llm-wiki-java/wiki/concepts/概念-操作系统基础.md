# 操作系统基础

**一句话定义**：操作系统通过进程/线程抽象分配 CPU 时间，通过虚拟内存抽象管理内存，通过用户态/内核态隔离保证安全，并提供 IO 模型让应用高效使用硬件资源。

---

## 第一性原理

硬件资源（CPU、内存、磁盘、网卡）是有限且危险的——多个程序同时使用可能互相干扰甚至崩溃系统。操作系统的核心角色是**资源的抽象与隔离**：

- **CPU 抽象**：通过时间片轮转，让多个程序"看起来"并发执行
- **内存抽象**：通过虚拟地址空间，每个进程以为自己独占内存
- **IO 抽象**：通过系统调用和缓冲机制，统一管理磁盘、网络等硬件

---

## 核心机制

### 进程 / 线程 / 协程

| 维度 | 进程 | 线程 | 协程 |
|------|------|------|------|
| 资源分配 | 独立资源（内存、文件句柄）| 共享进程资源 | 共享线程资源 |
| 调度主体 | OS 调度 | OS 调度 | 用户态调度（OS 不感知）|
| 切换开销 | 大（上下文完全切换）| 中（保存/恢复寄存器）| 小（用户态保存少量状态）|
| 隔离性 | 强（崩溃不影响其他进程）| 弱（线程崩溃可影响进程）| 弱 |
| Java 支持 | JVM 进程 | Thread/线程池 | 虚拟线程（JDK 21，Project Loom）|

**为什么需要协程**：多线程的频繁上下文切换和锁竞争消耗 CPU；协程由用户自决调度时机，不阻塞 OS 线程，特别适合 IO 密集型高并发场景（Golang goroutine 是典型实现）。

### 用户态 / 内核态

```
用户态（Ring 3）：应用程序运行空间，只能访问授权资源，不可操作硬件
      ↕ 系统调用（syscall）/ 中断 / 异常
内核态（Ring 0）：OS 内核运行空间，可访问所有硬件资源

切换过程：
  用户程序发起 read() → CPU 保存寄存器状态 → 切换到内核态
  → 内核执行 IO → 切换回用户态 → 恢复寄存器 → 继续执行
```

**每次系统调用都经历两次上下文切换**（用户→内核→用户），频繁的系统调用（如循环 read 小块数据）会造成显著的性能损耗。

**零拷贝的动机**：普通读写需要 4 次数据拷贝（磁盘→内核缓冲区→用户缓冲区→Socket缓冲区→网卡）+ 4 次上下文切换，其中很多拷贝没有业务价值。

### 零拷贝实现

| 方式 | 原理 | CPU拷贝次数 | 上下文切换 | 适用场景 |
|------|------|------------|-----------|---------|
| 普通 read+write | 全流程拷贝 | 2 | 4 | 通用 |
| mmap+write | 内核/用户共享内存映射 | 1 | 4 | 随机读写 |
| sendfile | 仅传描述符，DMA 直传 | 0（仅 DMA）| 2 | 静态文件传输（Nginx）|
| sendfile+DMA Scatter/Gather | DMA 从多个内存块聚合发送 | 0 | 2 | 最优化文件传输 |

**Kafka 用 sendfile**（Zero Copy）将磁盘文件直接发到网卡，避免经过用户空间，这是 Kafka 高吞吐的关键原因之一。

### IO 模型（五种）

```
同步阻塞 IO（BIO）：调用 read() 一直阻塞到数据返回
  → 简单，但每个连接需要一个线程，线程数成为瓶颈

同步非阻塞 IO（NIO）：调用 read() 若无数据立即返回，轮询检查
  → 不阻塞线程，但 CPU 空转轮询浪费

IO 多路复用（select/poll/epoll）：单线程同时监听多个 fd，有就绪才处理
  → 一个线程管理大量连接，Nginx/Redis/Netty 的基础

信号驱动 IO：注册信号函数，内核数据就绪后通过信号通知
  → 使用较少

异步 IO（AIO）：调用 aio_read() 后立即返回，内核完成数据拷贝后回调
  → 真正异步，但实现复杂，Linux AIO 不成熟，Windows IOCP 较好
```

**select vs poll vs epoll**：

| 实现 | fd 上限 | 用户态→内核态拷贝 | 遍历就绪 fd | 时间复杂度 |
|------|--------|----------------|------------|----------|
| select | 1024 | 每次调用都拷贝 | 遍历全部 fd | O(n) |
| poll | 无限制 | 每次调用都拷贝 | 遍历全部 fd | O(n) |
| epoll | 无限制 | 注册时拷贝一次 | 直接返回就绪列表 | O(1) |

**epoll LT vs ET**：LT（水平触发，默认）就绪未处理下次继续通知；ET（边缘触发）只通知一次，必须立即处理完，效率更高但要求非阻塞 socket。

### IO 多路复用 vs 多线程

| 场景 | 推荐 | 原因 |
|------|------|------|
| IO 密集（大量并发网络连接）| IO 多路复用 | 单线程管理万级连接，无锁竞争 |
| CPU 密集（大量数值计算）| 多线程 | 利用多核并行，IO 多路复用无优势 |

现代高性能服务器（Netty/Nginx/Redis）通常结合两者：少量 IO 线程（epoll 多路复用）+ 业务线程池（多线程处理 CPU 计算）。

### 多级缓存与 MESI 协议

```
CPU 执行速度 >> 内存读写速度（差距 100x~1000x）

解决方案：多级缓存
CPU ←L1 Cache(KB级)← L2 Cache(MB级)← L3 Cache(MB-GB级)← 主内存← 磁盘

每个核心有独立的 L1/L2，共享 L3
数据读取：L1未命中→L2→L3→主内存（逐级查找）
```

**MESI 缓存一致性协议**（多核 CPU 缓存数据一致性）：

每个缓存行有 4 种状态：
- **M（Modified）**：已修改，与主内存不一致，仅存在本核
- **E（Exclusive）**：独占，与主内存一致，仅存在本核
- **S（Shared）**：共享，与主内存一致，多核有副本
- **I（Invalid）**：无效，本核数据不可用

写数据时，该核发信号通知其他核将对应缓存行置为 Invalid，其他核下次读取时强制从主内存重新加载。

**MESI 与 JMM**：Java 内存模型（JMM）的 volatile 关键字底层通过 MESI 协议 + 内存屏障实现，保证多线程可见性（见 [[概念-JMM]]）。

### Page Cache

操作系统在内存中维护 Page Cache，将磁盘数据缓存为内存页（通常 4KB）：

```
写流程：应用 write() → 写入 Page Cache（脏页）→ 后台异步 flush 到磁盘
        ↑ 默认延迟写（write-back），性能好但有崩溃丢数据风险
        ↑ fsync() 强制将脏页同步到磁盘（MySQL WAL 的 fsync 就是这个）

读流程：应用 read() → 检查 Page Cache → 命中则直接返回
                     → 未命中则从磁盘读取，预读额外的页到 Cache
```

**Redis 持久化 AOF everysec** 就是每秒调用一次 fsync，在性能和数据安全之间取得平衡。

### 进程调度算法

| 算法 | 核心思路 | 优点 | 缺点 | 适用 |
|------|---------|------|------|------|
| FCFS | 先到先得 | 公平简单 | 长任务阻塞短任务 | 批处理 |
| SJF | 短任务优先 | 最优平均等待时间 | 需预知时长，长任务饥饿 | 已知时长任务 |
| 时间片轮转（RR）| 固定时间片轮流分配 | 公平，无饥饿 | 频繁上下文切换 | 交互式系统 |
| 优先级调度 | 高优先级优先 | 重要任务快速响应 | 低优先级饥饿 | 实时系统 |
| 多级反馈队列（MLFQ）| 多优先级队列+动态降级 | 自适应短长任务 | 实现复杂 | 通用 OS |
| CFS（Linux 默认）| 基于虚拟运行时间+红黑树 | 高公平，低延迟 | 实时任务需额外配置 | Linux |

### 内存管理：分页与分段

**分页（Paging）**：
- 物理内存划分为固定大小的页框（4KB），虚拟地址通过**页表**映射到物理页
- 缺页中断：访问到未分配物理内存的虚拟页时触发，OS 装入相应页
- **TLB（快表）**：硬件缓存热点页表项，避免每次访问内存都查页表（O(1) vs 多次内存访问）

**分段（Segmentation）**：
- 将程序划分为代码段、数据段、堆栈段等逻辑段，每段独立扩展，互相隔离
- 程序员可感知（段地址），防止代码段和栈段相互覆盖

**现代系统**：分页+分段组合使用，先分段再分页（段页式）。

### 进程间通信（IPC）

| 方式 | 特点 | 适用场景 |
|------|------|---------|
| 管道（Pipe）| 半双工，父子进程间 | Shell 命令组合（`ls \| grep`）|
| 消息队列 | 异步，按消息传递 | 分布式系统（参考 [[机制-消息队列可靠性]]）|
| 共享内存 | 最快（无拷贝），需加锁同步 | 高性能进程间大量数据共享 |
| 信号量 | 同步/互斥控制 | 进程间资源锁 |
| Socket | 跨主机，通用 | 网络通信，微服务间调用 |
| 文件映射（mmap）| 跨进程共享文件内存 | 数据库共享缓冲池（如 PostgreSQL）|

## 关键权衡

1. **多线程 vs 协程**：线程 1MB 栈空间，协程 KB 级；百万并发连接需协程（Java 21 虚拟线程）；CPU 密集型仍需真实线程利用多核
2. **Page Cache 延迟写 vs 同步写**：延迟写（write-back）性能极高但崩溃丢数据；fsync 保证持久性但降低写 QPS；MySQL InnoDB redo log 选 O_DIRECT 绕过 Page Cache 自管缓冲（Buffer Pool）
3. **IO 多路复用的局限**：epoll 对**网络 IO** 效果最好；对**磁盘文件 IO** 效果差（Linux 磁盘 IO 一般总是就绪，epoll 会认为磁盘 fd 永远可读），磁盘 IO 密集场景需用 io_uring（Linux 5.1+）或线程池
4. **MESI 与 Java volatile**：volatile 通过内存屏障 + MESI 协议保证可见性，但 MESI 是 CPU 级别保证，volatile 是 JVM 级别抽象，两者共同作用才能保证 Java 多线程可见性

## 与其他概念的关系

- **[[概念-IO模型]]**：BIO/NIO/AIO 是 Java 对操作系统五种 IO 模型的封装；epoll 是 NIO 和 Netty 的底层基础
- **[[机制-线程池]]**：Java 线程对应 OS 内核线程（1:1 模型），线程切换开销来源于 OS 上下文切换；JDK 21 虚拟线程绑定少量 OS 线程（M:N 模型）
- **[[概念-JMM]]**：Java 内存模型的 happens-before 规则和 volatile 语义，底层依赖 MESI 协议 + CPU 内存屏障实现
- **[[概念-计算机网络]]**：epoll 是高性能网络服务器（Nginx/Redis/Netty）的核心，将 IO 多路复用与 TCP 协议栈结合
