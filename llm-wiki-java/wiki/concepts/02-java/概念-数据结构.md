# 数据结构

> 元素按顺序排列、访问有明确规则的四种基础结构——数组（随机访问）、链表（动态增删）、栈（LIFO）、队列（FIFO）。

## 一、第一性原理

所有线性结构都解决同一个问题：**如何在一组有序元素中高效地存储、访问和修改数据**。四种结构的差异来自对"访问模式"的不同约束——不同约束带来不同的实现代价和适用场景。

## 二、核心机制

### 数组 vs 链表

| 维度 | 数组 | 链表 |
|------|------|------|
| 内存 | 连续 | 不连续（每个节点存指针）|
| 按下标访问 | O(1) | O(n)（必须从头遍历）|
| 按值查找 | O(n)；有序数组 O(log n) | O(n) |
| 插入/删除（中间）| O(n)（需移动元素）| O(1)（改指针，前提是已有节点引用）|
| 空间利用 | 预分配，可能浪费 | 按需分配，但有指针额外开销 |

**链表变体**：
- **单向链表**：只有 next 指针
- **双向链表**：有 prev + next，LinkedList 的底层，支持 O(1) 头尾操作
- **环形链表**：尾节点指向头节点，用于循环缓冲区

### 栈（Stack）— LIFO 先进后出

```
push(x) → 压栈   pop() → 弹栈   peek() → 查看栈顶
```

**实现**：数组（更常用，Cache 友好）或链表  
**Java**：`Deque<Integer> stack = new ArrayDeque<>()`（不推荐 `Stack` 类，有同步开销）

**本质**：函数调用栈、表达式求值、括号匹配、DFS 非递归实现

### 队列（Queue）— FIFO 先进先出

```
offer(x) → 入队   poll() → 出队   peek() → 查看队头
```

**实现**：链表（头出尾入，O(1)）；循环数组（避免假溢出）  
**Java**：`Queue<Integer> q = new LinkedList<>()`

**变体**：
- **双端队列（Deque）**：两端均可入出，`ArrayDeque` 实现
- **循环队列**：数组实现，首尾指针循环移动，解决假溢出
- **优先队列**：按优先级出队，底层是堆（见 [[机制-堆与优先队列]]）

### 四者关系图

```
元素访问模式
├── 任意位置 O(1) → 数组
├── 顺序访问，动态增删 → 链表
├── 只访问"最新加入"的元素 → 栈（LIFO）
└── 只访问"最早加入"的元素 → 队列（FIFO）
```

### Java 集合框架中的线性结构

| 实现类 | 底层 | 线程安全 | 扩容 | 建议 |
|--------|------|---------|------|------|
| `ArrayList` | 数组 | 否 | 1.5x（仅序列化有效元素，transient 数组） | 推荐 |
| `LinkedList` | 双向链表 | 否 | 无需扩容 | 频繁头尾操作时使用 |
| `Vector` | 数组 | 是（方法锁） | 2x | 不推荐，用 CopyOnWriteArrayList |
| `ArrayDeque` | 循环数组 | 否 | 2x | 替代 Stack 和 LinkedList 队列 |

**ArrayList.subList() 陷阱**：返回的不是新 List，而是原 List 的视图（内部类 SubList）。对父 List 做结构性修改（增/删元素）后再访问 subList 会抛 `ConcurrentModificationException`。

### Java 集合框架体系概览

**Collection 体系**（元素容器）：

```
Iterable
  └── Collection
        ├── List（有序、可重复）：ArrayList / LinkedList / Vector
        ├── Set（无序、不可重复）：HashSet / LinkedHashSet / TreeSet
        └── Queue（FIFO）：ArrayDeque / LinkedList / PriorityQueue
```

**Map 体系**（键值映射，不继承 Collection）：

```
Map
  ├── HashMap（无序，O(1) 增删查）
  ├── LinkedHashMap（插入/访问顺序，LRU 基础）
  ├── TreeMap（红黑树，key 有序，O(log n)）
  ├── Hashtable（已废弃，方法锁）
  └── ConcurrentHashMap（并发安全，CAS+桶锁）
```

**Set 去重机制**：
- `HashSet` 基于 `HashMap` key 去重 → 依赖 `hashCode()` + `equals()`
- `TreeSet` 基于 `TreeMap` → 依赖 `compareTo()`，元素有序

**线程安全选型**：

| 需求 | 推荐 | 不推荐 |
|------|------|-------|
| 线程安全 List | `CopyOnWriteArrayList` | `Vector`（全方法锁，粗粒度）|
| 线程安全 Map | `ConcurrentHashMap` | `Hashtable`（同上）|
| 接口语义优先 | 先定接口（List/Set/Map），再选实现 | 直接用实现类声明变量 |

**排序**：对象自身排序实现 `Comparable`（`compareTo`）；外部临时规则用 `Comparator`（`compare`）。

## 三、核心使用原则

1. **数组适合读多写少**：随机访问 O(1) 是核心优势；扩容需复制整个数组（ArrayList 扩容 1.5x）
2. **链表适合写多读少**：频繁在中间插入删除；但现实中 Cache Miss 让链表实际性能常低于数组
3. **栈不要用 `java.util.Stack`**：继承 Vector，所有方法 synchronized，用 `ArrayDeque` 替代
4. **循环队列 vs LinkedList 队列**：ArrayDeque（循环数组）比 LinkedList 队列性能更好（Cache 友好）
5. **遍历时修改**：foreach 内 remove 触发 fail-fast 抛 `ConcurrentModificationException`；用 `iterator.remove()`、`removeIf()` 或 `Stream.filter()` 替代

## 四、与其他概念的关系

- 支撑了 [[机制-堆与优先队列]]：堆用数组实现（父节点 index = (i-1)/2）
- 支撑了 [[概念-图论基础]]：BFS 用队列；DFS 用栈（或递归调用栈）
- `TreeMap/TreeSet` 依赖 [[机制-红黑树]] 保持 key 有序
- `HashSet` 内部复用 [[机制-HashMap]] 的 key 唯一性
- 并发容器详见 [[机制-ConcurrentHashMap]]
- 在 L2 JVM：Java 方法调用栈就是栈结构，每次方法调用 push 一个栈帧
- 在 L3 并发：`BlockingQueue`（队列）是生产者-消费者模式的核心数据结构

## 五、应用边界

**数组（ArrayList）**：随机访问频繁；元素数量相对稳定；基本类型数组性能最优。

**链表（LinkedList）**：频繁头尾插入删除（如队列/双端队列）；不需要随机访问。

**栈**：递归改非递归；括号匹配；撤销/重做功能；单调栈（解决"下一个更大元素"类问题）。

**队列**：BFS；任务调度；消息缓冲；滑动窗口（单调队列）。
