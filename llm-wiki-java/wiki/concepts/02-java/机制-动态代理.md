# 反射与动态代理

> 反射是程序在运行时获取并操作类型元信息的能力；动态代理是在运行期按需生成代理类，对方法调用进行拦截与增强——两者共同构成 Java 框架（IoC、AOP、RPC、ORM）的元编程基石。

## 一、第一性原理

**反射存在理由**：编译期无法确定所有类型——插件系统、框架注入、序列化/反序列化都需要在运行期做类型决策。JVM 加载类时在堆中为每个类维护唯一 `Class` 对象，反射即读取该对象的元数据并发起调用。

**动态代理存在理由**：静态代理要求为每个被代理类编写一个代理类，代码量线性增长。动态代理在运行期按需生成代理类，消除样板代码，使横切关注点（日志、事务、权限）可统一施加。

## 二、核心机制

### 2.1 反射三步

**① 获取 Class 对象**

| 方式 | 特点 |
|------|------|
| `Class.forName("全限定名")` | 触发类加载，适合运行期动态类名 |
| `obj.getClass()` | 从已有实例获取 |
| `SomeClass.class` | 编译期字面量，无加载开销 |

**② 读取结构**

```java
Method[] methods = clazz.getDeclaredMethods();  // 含私有方法
Field field = clazz.getDeclaredField("name");
field.setAccessible(true);                       // 绕过 private 访问控制
```

**③ 发起调用**

```java
method.invoke(obj, args);
// 调用链路：参数装箱 Object[] → 可见性检查 → 方法表遍历 → native 调用 → 结果拆箱
```

**反射慢的四重开销**：无 JIT 内联优化、参数装拆箱、方法表遍历、可见性检查。热路径避免反射，框架层通常通过缓存 `Method`/`Field` 对象 + `MethodHandle`（JDK 7+）缓解。

### 2.2 JDK 动态代理

**前提**：目标类必须实现至少一个接口。

```java
Object proxy = Proxy.newProxyInstance(
    target.getClass().getClassLoader(),
    target.getClass().getInterfaces(),
    (proxyObj, method, args) -> {
        // 前置增强
        Object result = method.invoke(target, args);  // 反射调用目标
        // 后置增强
        return result;
    }
);
```

**字节码层面**：JVM 运行时生成 `$Proxy0` 等类，实现目标接口，所有方法委托给 `InvocationHandler.invoke()`。

### 2.3 CGLIB 动态代理

**前提**：目标类不能是 `final`（通过生成子类实现代理）。

底层使用 ASM 库直接生成字节码，通过 `MethodInterceptor` 拦截方法调用。CGLIB 生成 FastClass 索引，方法调用时通过索引直接调用而非反射，因此调用阶段性能优于 JDK 代理。

## 三、Java 核心使用

### 3.1 Spring AOP 代理选择策略

```text
目标类实现了接口？
├── 是，且未设置 proxyTargetClass=true → JDK 动态代理
└── 否，或 proxyTargetClass=true       → CGLIB
```

- Spring Boot 2.x 起默认 `proxyTargetClass=true`，即默认 CGLIB。
- JDK 17+ 强封装模块后，JDK 代理对跨模块类的反射访问有限制，Spring 6 / Spring Boot 3 进一步倾向 CGLIB。

### 3.2 框架典型用法

| 框架 | 反射/代理用途 |
|------|-------------|
| Spring IoC | 反射实例化 Bean、注入依赖 |
| Spring AOP | 动态代理拦截方法，执行切面逻辑 |
| MyBatis | JDK 代理为 Mapper 接口生成实现 |
| Dubbo | 代理生成远程调用 Stub |
| Jackson/Gson | 反射读写字段进行序列化 |

## 四、核心使用原则

1. **面向接口设计时用 JDK 代理**：语义清晰，无第三方依赖。
2. **无接口或需代理具体类时用 CGLIB**：Spring Boot 默认行为，无需额外配置。
3. **避免在热路径使用裸反射**：缓存 `Method`/`Field` 对象，或用 `MethodHandle`/`LambdaMetafactory` 替代。
4. **不要依赖代理 final 方法**：JDK 代理和 CGLIB 均无法拦截 final 方法（面试高频误区）。
5. **Spring 中 `@Transactional` 自调用失效**：同类内部方法调用不经过代理对象，事务注解不生效——这是代理机制的本质限制，非 Bug。

## 五、综合对比

| 维度 | JDK 动态代理 | CGLIB |
|------|------------|-------|
| 前提条件 | 必须有接口 | 无需接口，生成子类 |
| 底层实现 | `java.lang.reflect.Proxy` | ASM 字节码生成 |
| 方法调用性能 | 反射调用（稍慢） | FastClass 索引直接调用 |
| 类生成速度 | 快 | 慢（ASM 字节码生成） |
| 代理 final 方法 | 不支持 | 不支持（无法 override） |
| 代理 final 类 | 可代理其接口 | 不支持（无法继承） |
| 第三方依赖 | 无（JDK 内置） | 需引入 cglib 或 spring-core 内嵌 |
| Spring 默认 | Spring MVC 早期默认 | Spring Boot 2.x+ 默认 |

## 六、生产风险

| 风险 | 根因 | 应对 |
|------|------|------|
| 反射性能差 | 无 JIT 内联、装拆箱、方法遍历 | 缓存 Method 对象、用 MethodHandle 替代 |
| `setAccessible(true)` 破坏封装 | 绕过 private 访问控制 | JDK 17+ 强封装模块限制跨模块反射 |
| CGLIB 启动慢 | ASM 字节码生成耗时 | 容器启动期一次性开销，运行期无影响 |
| 代理对象类型判断 | `instanceof` 对 CGLIB 子类和 JDK 代理行为不同 | 面向接口编程，用接口类型判断 |
| `@Transactional` 自调用失效 | 内部方法调用不经过代理 | 注入自身或使用 `AopContext.currentProxy()` |
| JDK 17+ `InaccessibleObjectException` | 强封装拒绝跨模块反射 | 添加 `--add-opens` 或迁移到公开 API |

## 七、与其他概念的关系

- 支撑 [[机制-SPI]]：`ServiceLoader` 通过 `Class.forName()` + `newInstance()` 加载实现类
- 支撑 [[机制-Spring]]：Spring IoC 通过反射实例化 Bean，AOP 切面逻辑在 `InvocationHandler.invoke()` / `MethodInterceptor.intercept()` 中执行
- 支撑 [[机制-Java序列化]]：序列化框架用反射读写对象字段

## 八、应用边界

| 技术 | 适合 | 不适合 |
|------|------|--------|
| 反射 | 框架开发（IoC、ORM、RPC）、通用工具（BeanUtils、JSON 库）、SPI 加载 | 高频业务热路径、编译期可确定类型的场景、JDK 17+ 跨模块访问 |
| JDK 代理 | 面向接口的设计、需代理所有接口方法 | 目标类无接口、需代理具体类方法 |
| CGLIB | 无接口的类、Spring Boot 默认场景、需代理非接口方法 | final 类/方法、对启动速度极度敏感的场景 |
