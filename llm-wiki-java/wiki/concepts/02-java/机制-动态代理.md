# 反射与动态代理

> **反射**：程序在运行时获取并操作自身类型信息的能力——给一个类名，就能拿到它的全部结构并调用。  
> **动态代理**：在运行期生成代理类，对目标对象的方法调用进行拦截和增强——不修改原始代码实现横切逻辑。

两者密切关联：动态代理底层依赖反射调用目标方法；框架层（Spring AOP、MyBatis Mapper、Dubbo Stub）同时使用两者。

## 第一性原理

**反射存在理由**：编译期无法确定所有类型（插件系统、框架注入、序列化），必须推迟到运行期做类型决策。JVM 加载类时在堆中为每个类维护唯一的 `Class` 对象——反射即读取这个对象里的元数据并发起调用。

**动态代理存在理由**：静态代理在编译期写死代理类，代码量随被代理类数量线性增长。动态代理运行期按需生成代理类，**消除样板代码，使 AOP 等横切关注点可以统一施加**。

## 核心机制

### 一、反射三步

**① 获取 Class 对象**

| 方式 | 特点 |
|------|------|
| `Class.forName("全限定名")` | 触发类加载，适合运行期动态类名 |
| `obj.getClass()` | 从已有实例获取 |
| `SomeClass.class` | 编译期字面量，无加载开销 |

**② 读取结构**

```java
Method[] methods = clazz.getDeclaredMethods();  // 含私有方法
Field field = clazz.getDeclaredField("name");
field.setAccessible(true);                       // 破坏封装，绕过 private
```

**③ 发起调用**

```java
method.invoke(obj, args);
// 每次调用：参数装箱 Object[] → 可见性检查 → 方法表遍历 → 调用 → 结果拆箱
// 没有 JIT 内联优化 → 慢
```

**反射慢的根本原因**：跳过 JIT 内联优化、参数装拆箱、方法表遍历、可见性检查四重开销。

### 二、JDK 动态代理

**前提**：目标类必须实现至少一个接口。

```java
Object proxy = Proxy.newProxyInstance(
    target.getClass().getClassLoader(),
    target.getClass().getInterfaces(),
    (proxyObj, method, args) -> {
        // 前置处理
        Object result = method.invoke(target, args);  // 反射调用目标
        // 后置处理
        return result;
    }
);
```

**字节码层面**：JVM 运行时生成 `$Proxy0` 等类，实现目标接口，所有方法委托给 `InvocationHandler`。

### 三、CGLIB 动态代理

**前提**：目标类不能是 `final`（通过继承子类实现代理）。

| 维度 | JDK 代理 | CGLIB |
|------|---------|-------|
| 前提 | 必须有接口 | 无需接口，生成子类 |
| 底层 | `java.lang.reflect.Proxy` | ASM 字节码生成 |
| 调用性能 | 经过反射（稍慢） | 直接调用（fast class） |
| 代理 final 方法 | 不支持 | 不支持（无法 override） |
| 类生成速度 | 快 | 慢（ASM 字节码） |

### 四、Spring AOP 的选择策略

```
目标类实现了接口？
├── 是，且未设置 proxyTargetClass=true → JDK 动态代理
└── 否，或 proxyTargetClass=true       → CGLIB
```

JDK 17+ 强封装模块后，JDK 代理对跨模块类的反射访问有限制，Spring 6 默认改为倾向 CGLIB。

## 关键权衡

| 代价 | 根本原因 |
|------|---------|
| 反射性能低 | 无 JIT 内联、装拆箱、方法遍历 |
| 反射破坏封装 | `setAccessible(true)` 绕过 private；JDK 17+ 加固 |
| CGLIB 启动慢 | 类生成时间长（ASM 字节码），容器启动期有明显开销 |
| 代理 final 均失效 | 继承和接口代理都无法 override final 方法 |

## 与其他概念的关系

- 支撑 [[机制-SPI]]：`ServiceLoader` 通过 `Class.forName()` + `newInstance()` 加载实现类
- 支撑 [[机制-AOP织入]]：Spring AOP 织入的实现层；切面逻辑在 `InvocationHandler.invoke()` 中执行
- 支撑 [[机制-IoC容器]]：Spring IoC 通过反射实例化 Bean 并注入依赖
- 支撑 [[机制-Java序列化]]：序列化框架用反射读写字段值

## 应用边界

**反射适合**：框架开发（IoC、ORM、RPC）、动态代理、通用工具（BeanUtils、JSON 库）、SPI 加载。

**反射不适合**：高频业务代码；可编译期确定类型的场景；JDK 17+ 强封装模块内的跨模块访问。

**JDK 代理适合**：面向接口的设计，需要代理所有接口方法。

**CGLIB 适合**：无接口的类；需要代理非接口方法；Spring Boot 默认场景。
