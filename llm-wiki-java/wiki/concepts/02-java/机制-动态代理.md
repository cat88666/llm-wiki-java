# 反射与动态代理

> 反射解决“运行期拿到并操作类型信息”，动态代理解决“运行期给方法调用织入横切逻辑”，二者是 Java 框架化能力的底座。

## 一、运行期元编程的价值

框架要在不侵入业务代码的前提下完成事务、日志、鉴权、RPC stub、ORM 映射，必须依赖运行期元编程。

- 反射：动态读取结构与调用
- 代理：在调用前后织入统一行为

## 二、反射机制与性能代价

### 2.1 反射三步

```java
Class<?> clazz = Class.forName("com.demo.UserService");
Method m = clazz.getDeclaredMethod("query", Long.class);
Object ret = m.invoke(target, 1L);
```

### 2.2 性能代价来源

| 开销点 | 原因 |
| --- | --- |
| 动态分派 | 运行期解析方法 |
| 参数装箱拆箱 | `Object[]` 中转 |
| 可访问性检查 | 安全检查路径 |
| JIT 优化受限 | 难以内联 |

优化方向：缓存 `Method`、使用 `MethodHandle`、减少热路径反射。

## 三、JDK 代理与 CGLIB 链路

### 3.1 JDK 动态代理

前提：目标对象实现接口。

```java
Object proxy = Proxy.newProxyInstance(
    target.getClass().getClassLoader(),
    target.getClass().getInterfaces(),
    (p, method, args) -> method.invoke(target, args)
);
```

### 3.2 CGLIB 代理

前提：目标类非 final。通过生成子类并覆盖方法完成拦截。

### 3.3 核心差异

| 维度 | JDK 代理 | CGLIB |
| --- | --- | --- |
| 代理对象 | 接口 | 具体类 |
| 依赖 | JDK 原生 | 字节码库 |
| 对 final 方法 | 无法拦截 | 也无法拦截 |
| 常见场景 | 接口化服务 | 无接口旧代码 |

## 四、Spring AOP 代理决策

```text
目标是否有接口
  -> 有接口且未强制类代理：JDK
  -> 无接口或强制类代理：CGLIB
```

高频陷阱：`@Transactional` 自调用失效，因为同类内部调用绕过代理对象。

## 五、设计使用原则

1. 对外服务优先接口化，便于代理与测试替换。
2. 不要在极高频链路用裸反射调用。
3. 代理逻辑保持幂等和可观测，避免多层代理副作用叠加。
4. 对 final 类/方法的增强需求，优先重构设计而不是黑魔法。

## 六、方案对比结论

| 方案 | 优势 | 劣势 | 适用 |
| --- | --- | --- | --- |
| JDK 代理 | 简单稳定 | 仅接口 | 规范接口服务 |
| CGLIB | 可代理类 | final 受限 | 遗留类改造 |
| ByteBuddy | 表达力强 | 学习成本高 | 高级框架开发 |

## 七、生产风险与排查

| 风险 | 现象 | 处理 |
| --- | --- | --- |
| 代理失效 | 切面不执行 | 检查调用是否经过代理 |
| 代理链过深 | RT 增加 | 收敛横切层次 |
| 模块封装限制 | 反射访问异常 | 增加 opens/export 或升级框架 |
| 递归代理 | 栈溢出/死循环 | 防重入标记与调用链审计 |

## 八、关系与边界

- 与 [[机制-SPI]]：SPI 决定“加载谁”，代理决定“如何调用”。
- 与 [[机制-Java序列化]]：反序列化框架常依赖反射注入对象状态。
- 与 [[概念-OOP特征]]：代理强化“面向接口编程”。

边界：代理适合横切逻辑，不应用于复杂业务编排。
