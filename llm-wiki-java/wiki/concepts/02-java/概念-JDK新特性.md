# JDK 新特性

> JDK 演进主线是“表达力提升 + 并发模型升级 + 运维可观测增强”，落地关键不在“会不会新语法”，而在“何时迁移与如何控制风险”。

<a id="sec-2"></a>
## 一、版本演进主线

| 代际 | 代表版本 | 核心收益 |
| --- | --- | --- |
| 函数式阶段 | JDK8 | 行为抽象与集合处理范式升级 |
| 语言表达阶段 | JDK11/17 | 更少模板代码、更强类型表达 |
| 并发升级阶段 | JDK21 | 海量并发任务模型简化 |

<a id="sec-3"></a>
## 二、JDK8 关键能力

1. Lambda：行为参数化与函数式接口。
2. Stream：声明式集合处理与惰性执行。
3. Optional：显式空值语义。
4. `java.time`：线程安全时间 API。

<a id="sec-4"></a>
## 三、JDK11-17 语言增强

| 特性 | 价值 | 注意点 |
| --- | --- | --- |
| `var` | 降低局部样板 | 不等于弱类型 |
| Record | DTO 极简化 | 不可变、结构固定 |
| Sealed Class | 子类型边界可控 | 需系统建模清晰 |
| Pattern Matching | 分支语义更直观 | 需配合类型封闭 |

<a id="sec-5"></a>
## 四、JDK21 并发模型升级

### 4.1 虚拟线程机制

```text
虚拟线程（用户态）
  -> 挂载到少量平台线程执行
  -> 阻塞点可卸载
  -> 恢复后继续挂载执行
```

### 4.2 适用与误区

| 场景 | 结论 |
| --- | --- |
| 高并发 IO 请求 | 显著降低线程管理复杂度 |
| 纯 CPU 密集 | 收益有限 |
| synchronized 长阻塞 | 可能发生 pin，收益下降 |

<a id="sec-6"></a>
## 五、版本选型与升级策略

1. 生产优先 LTS（11/17/21）。
2. 升级前先做依赖兼容审计（JVM 参数、字节码版本、代理库）。
3. 采用双环境压测与灰度发布，观察 RT、吞吐、GC、线程指标。
4. 虚拟线程改造优先从边缘 IO 服务开始。

<a id="sec-7"></a>
## 六、技术方案对比

| 维度 | 传统线程池 | 虚拟线程 | 响应式模型 |
| --- | --- | --- | --- |
| 编程模型 | 命令式 | 命令式（近似同步写法） | 函数式链路 |
| 并发规模 | 中等 | 高 | 高 |
| 调试成本 | 中 | 中 | 高 |
| 迁移成本 | 低 | 中 | 高 |

<a id="sec-8"></a>
## 七、生产风险与治理

| 风险 | 触发条件 | 处理 |
| --- | --- | --- |
| 依赖不兼容 | 老库未适配新 JDK | 升级依赖、替换组件 |
| 性能回退 | 参数与 GC 策略不匹配 | 压测对比并调参 |
| 虚拟线程 pin | 锁与阻塞调用混用 | 降低 pin 点，必要时回退平台线程 |
| Optional 滥用 | 链式包装过深 | 边界层使用，核心链路简化 |

<a id="sec-9"></a>
## 八、关系与边界

- 与 [[机制-Lambda]]：JDK8 函数式能力是多数新 API 的基础。
- 与 [[机制-线程池]]：虚拟线程不是线程池完全替代。
- 与 [[概念-JMM]]：并发模型变化不改变内存可见性规则。

边界：JDK 升级本身不直接提升业务性能，收益来自代码与配置协同改造。

<a id="sec-10"></a>
## 九、面试速答口径

| 问题 | 关键答案 |
| --- | --- |
| 为什么优先 LTS | 生态稳定、补丁长期维护、升级风险更可控 |
| 虚拟线程与响应式怎么选 | 要低改造成本选虚拟线程，要端到端事件流选响应式 |
| Record 适合什么场景 | 不可变 DTO、命令对象、返回视图对象 |
