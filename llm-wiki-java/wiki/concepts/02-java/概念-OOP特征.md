# OOP 特征

> 面向对象通过封装隐藏实现、继承复用行为、多态统一接口，使大规模系统通过接口契约而非代码细节耦合，模块可独立演化。

## 一、第一性原理

- **面向过程**：把问题分解为步骤序列，关注"怎么做"
- **面向对象**：把问题分解为对象，关注"谁来做"，对象持有状态和行为

OOP 的根本价值：**让系统通过接口契约而非实现细节耦合，使模块可以独立变化，降低大规模系统的变更成本**。

封装、继承、多态是实现这个目标的三个正交手段。

## 二、核心机制

### 2.1 封装（Encapsulation）

将数据与操作绑定在类中，对外暴露接口、隐藏实现。

```java
class BankAccount {
    private double balance;                      // 外部不可直接访问
    public void deposit(double amount) { ... }   // 通过方法操作
}
```

**工程含义**：调用方只依赖公开接口，内部实现可自由重构。

### 2.2 继承（Inheritance）

子类复用父类的属性和方法，并可扩展或重写。

**Java 单继承限制**：防止菱形问题 -- 两个父类有同名方法时，子类无法确定继承哪个。Java 用接口多实现 + JDK 8 default 方法弥补。

### 2.3 多态（Polymorphism）

同一接口，不同对象有不同实现，调用方无需感知具体类型。

```java
Animal animal = new Dog();   // 父类引用指向子类对象
animal.speak();              // 运行期调用 Dog.speak()
```

**实现机制**：方法重写（Override）+ 虚方法表（vtable）。JVM 在运行期根据对象实际类型查表决定调用哪个方法 -- 即动态派发（Dynamic Dispatch）。

**面试要点**：重载（Overload）是编译期静态派发（参数类型决定）；重写（Override）是运行期动态派发（对象实际类型决定）。

## 三、Java 核心使用

### 3.1 接口 vs 抽象类

| 维度 | 接口（Interface） | 抽象类（Abstract Class） |
|------|---------|----------|
| 继承限制 | 可实现多个 | 只能继承一个 |
| 字段 | 只有 `public static final` 常量 | 可有实例字段 |
| 方法 | 默认抽象；JDK 8+ 支持 `default`/`static` | 可有具体方法 |
| 构造器 | 无 | 有 |
| 语义 | **能力契约**（"我能做某事"） | **部分实现**（共同行为骨架） |

**选型规则**：

| 场景 | 选择 | 例子 |
|------|------|------|
| 定义能力/角色 | 接口 | `Runnable`、`Comparable`、`Serializable` |
| 共享公共实现逻辑 | 抽象类 | `AbstractList`、`AbstractQueuedSynchronizer` |
| 两者都可 | 优先接口 | 更灵活，支持多实现 |

### 3.2 JDK 8 default 方法

```java
interface Collection<E> {
    default Stream<E> stream() {    // 接口演化：不破坏已有实现类
        return StreamSupport.stream(spliterator(), false);
    }
}
```

解决接口演化问题：新增方法不强制所有实现类修改。但引入了接口层面的菱形冲突可能 -- 编译器要求实现类显式解决。

### 3.3 组合模式

```java
// 继承：强耦合，子类暴露在父类实现细节下
class MyList extends ArrayList<String> { ... }

// 组合：松耦合，运行期可替换
class MyList {
    private final List<String> delegate;   // 持有引用
    public void add(String s) { delegate.add(s); }
}
```

## 四、核心使用原则

| 原则 | 说明 |
|------|------|
| 组合优于继承 | 继承是编译期绑定的强耦合；组合是运行期可替换的松耦合 |
| 继承层次不超过 3 层 | 越深越难理解，重构越痛苦 |
| 继承只用于 is-a 关系 | has-a 关系用组合 |
| 接口聚焦单一能力 | 接口隔离原则（ISP），避免胖接口 |
| 封装是底线 | 字段 private，暴露行为方法 |

## 五、综合对比

### 5.1 继承 vs 组合

| 维度 | 继承 | 组合 |
|------|------|------|
| 耦合度 | 强（编译期绑定） | 弱（运行期可替换） |
| 灵活性 | 低（单继承限制） | 高（可持有多个引用） |
| 适用关系 | is-a | has-a |
| 封装保护 | 差（子类可访问 protected） | 好（通过接口交互） |
| 典型应用 | 框架骨架（Template Method） | 策略模式、装饰器模式 |

### 5.2 重载 vs 重写

| 维度 | 重载（Overload） | 重写（Override） |
|------|---------|----------|
| 发生位置 | 同一个类中 | 子类对父类方法 |
| 派发方式 | 编译期静态派发 | 运行期动态派发 |
| 参数 | 参数列表必须不同 | 参数列表必须相同 |
| 返回值 | 无限制 | 协变返回类型（子类型） |
| 访问修饰符 | 无限制 | 不能比父类更严格 |

## 六、生产风险

| 风险 | 触发场景 | 后果 | 防御 |
|------|---------|------|------|
| 继承层次过深 | 超过 3 层类继承 | 理解和重构成本剧增 | 优先组合，控制继承深度 |
| 接口爆炸 | 过度设计，每个行为一个接口 | 类图复杂，增加认知负担 | 接口聚焦核心能力，遵循 ISP |
| 菱形冲突 | 实现多个带 default 方法的接口 | 编译报错 | 实现类显式重写冲突方法 |
| 父类修改影响子类 | 父类改变 protected 方法行为 | 子类行为被意外改变（脆弱基类问题） | 优先组合；继承时遵循里氏替换原则 |
| 虚方法表开销 | 极端性能敏感路径 | 动态派发比直接调用略慢 | 极少成为瓶颈，JIT 会内联热点方法 |

## 七、与其他概念的关系

- 支撑了 [[机制-动态代理]]：代理模式依赖多态，代理类和目标类共同实现同一接口
- 与 [[机制-泛型]] 协作：泛型上下界通配符（`<? extends T>`、`<? super T>`）建立在继承关系之上
- 支撑了 Spring IoC：`ApplicationContext` 通过接口多态管理 Bean，不感知具体实现类
- 多态是策略模式、工厂模式、模板方法模式的基石

## 八、应用边界

| 特征 | 适用 | 不适用 |
|------|------|--------|
| 封装 | 永远应该做，字段 private，方法 public/protected | 无 |
| 继承 | 框架骨架设计（Template Method）、真正的 is-a 关系 | has-a 关系、需要运行期灵活替换的场景 |
| 多态 | 策略模式、工厂模式、插件化架构 | 简单脚本、无需扩展的工具类 |
| 接口 | 定义能力契约、模块边界 API | 只有一个实现且不会扩展（过度设计） |
| 抽象类 | 多个子类共享模板逻辑 | 已有一个父类（单继承限制） |
