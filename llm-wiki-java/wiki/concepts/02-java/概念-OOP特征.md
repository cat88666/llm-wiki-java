# OOP 特征

> OOP 的核心不是“会写类”，而是通过封装、继承、多态构建可演进系统边界，让变更集中在实现层而非调用层。

<a id="sec-2"></a>
## 一、OOP 的工程目标

OOP 解决的是大系统协作问题：

- 调用方依赖抽象契约
- 实现方可独立替换
- 横切能力可统一织入

<a id="sec-3"></a>
## 二、三大特征机制

### 2.1 封装

隐藏状态细节，仅暴露可控行为接口，防止外部破坏不变量。

### 2.2 继承

复用共性实现，但会引入父子耦合与演化约束。

### 2.3 多态

同一调用点面向抽象，运行期由具体实现决定行为，实现“开闭原则”关键路径。

<a id="sec-4"></a>
## 三、接口与抽象类实践

| 维度 | 接口 | 抽象类 |
| --- | --- | --- |
| 语义 | 能力契约 | 公共骨架 |
| 继承关系 | 多实现 | 单继承 |
| 状态承载 | 常量 | 可承载实例状态 |

选择原则：跨模块契约优先接口；共享实现细节再引入抽象类。

<a id="sec-5"></a>
## 四、组合优于继承

```java
class OrderService {
    private final PricingPolicy pricingPolicy; // 组合
}
```

组合可以在运行期替换策略，继承通常在编译期固定行为。

<a id="sec-6"></a>
## 五、设计原则落地

1. 单一职责：类只承担一个变化原因。
2. 开闭原则：新增行为通过扩展完成，不改稳定调用方。
3. 依赖倒置：高层依赖抽象，不依赖具体实现。
4. 接口隔离：小接口替代胖接口。

<a id="sec-7"></a>
## 六、方案对比结论

| 方案 | 优点 | 风险 | 适用 |
| --- | --- | --- | --- |
| 继承 | 复用快 | 强耦合、层级膨胀 | 框架模板骨架 |
| 组合 | 灵活、可替换 | 初期设计成本高 | 业务策略体系 |
| 委托 | 解耦明显 | 调用链更长 | 中间层适配 |

<a id="sec-8"></a>
## 七、生产风险与排查

| 风险 | 现象 | 处理 |
| --- | --- | --- |
| 继承链过深 | 修改牵一发而动全身 | 拆分为组合策略 |
| 接口膨胀 | 实现类大量空方法 | 拆分细粒度接口 |
| 里氏替换失效 | 子类违背父类契约 | 收敛契约，增加测试 |
| 过度抽象 | 代码难读 | 回归业务语义命名 |

<a id="sec-9"></a>
## 八、关系与边界

- 与 [[机制-动态代理]]：代理是对多态调用点的增强。
- 与 [[机制-泛型]]：泛型让抽象契约在编译期更安全。
- 与 [[机制-SPI]]：SPI 是 OOP 抽象在框架扩展层的落地方式。

边界：OOP 不是所有问题的唯一答案，数据流密集场景可能更适合函数式或事件流模型。

<a id="sec-10"></a>
## 九、面试速答口径

| 问题 | 关键答案 |
| --- | --- |
| 组合为什么优于继承 | 降低父子耦合，运行期可替换策略 |
| 接口和抽象类怎么选 | 契约优先接口，共享实现再抽象类 |
| 多态在 JVM 怎么实现 | 虚方法表 + 运行期动态派发 |
