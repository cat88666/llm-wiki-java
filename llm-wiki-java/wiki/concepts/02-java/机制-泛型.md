# 泛型与类型擦除

> 泛型把类型错误前移到编译期，类型擦除保证运行期字节码复用与历史兼容，核心权衡是类型安全提升与运行期类型信息缺失并存。

## 一、泛型出现的根因

JDK5 之前集合以 `Object` 承载，类型错误通常在运行期才暴露。泛型让编译器提前做类型检查，降低线上类型错误。

## 二、类型擦除机制

### 2.1 擦除规则

| 声明 | 擦除后 |
| --- | --- |
| `<T>` | `Object` |
| `<T extends Number>` | `Number` |
| `List<String>` | `List` |

### 2.2 桥方法

编译器生成 bridge method 维持多态语义，避免擦除导致签名不匹配。

### 2.3 签名信息保留

虽然运行期对象不携带完整泛型实参，但类/字段/方法签名仍可通过反射读取（`Signature` 属性）。

## 三、通配符与 PECS

| 写法 | 语义 | 能读 | 能写 |
| --- | --- | --- | --- |
| `? extends T` | 生产者 | `T` | 否 |
| `? super T` | 消费者 | `Object` | `T` |
| `?` | 未知类型 | `Object` | 否 |

PECS 口诀：Producer Extends, Consumer Super。

## 四、泛型实战模式

### 4.1 泛型方法

```java
public static <T> T first(List<T> list) {
    return list.isEmpty() ? null : list.get(0);
}
```

### 4.2 运行时保留类型（TypeToken）

```java
Type t = new TypeToken<List<String>>() {}.getType();
```

### 4.3 无法 `new T()` 的替代

通过 `Class<T>` 或 `Supplier<T>` 注入构造能力。

## 五、编码原则

1. 禁止原始类型 `List`，统一参数化类型。
2. API 入参使用通配符表达读写意图。
3. 对外接口减少复杂嵌套泛型层级，提升可维护性。
4. 泛型边界表达业务约束，不要滥用 `Object`。

## 六、跨语言对比

| 语言 | 泛型实现 | 取舍 |
| --- | --- | --- |
| Java | 类型擦除 | 兼容性强，运行期信息弱 |
| C# | 具化泛型 | 运行期类型强，元数据更丰富 |
| C++ | 模板实例化 | 性能高，代码膨胀明显 |

## 七、生产风险与排查

| 风险 | 触发条件 | 现象 | 处理 |
| --- | --- | --- | --- |
| 堆污染 | 原始类型混用 | 运行期 `ClassCastException` | 开启编译告警并修复 |
| 重载冲突 | 擦除后签名一致 | 编译失败 | 调整方法名或参数结构 |
| 泛型信息丢失 | 运行期反射判型 | 序列化反序列化异常 | 显式 TypeToken 传递 |

## 八、关系与边界

- 与 [[概念-Java类型]]：装箱类型广泛参与泛型容器。
- 与 [[概念-Java集合]]：集合 API 语义大量依赖泛型设计。
- 与 [[机制-动态代理]]：框架反射推断泛型实参时需读取签名信息。

边界：泛型提供编译期类型安全，不提供运行期业务校验。

## 九、面试速答口径

| 问题 | 关键答案 |
| --- | --- |
| Java 泛型为什么擦除 | 兼容旧字节码与运行时模型，避免模板膨胀 |
| 为什么不能 `new T()` | 擦除后无法确定具体类型构造器 |
| `extends` 和 `super` 怎么选 | 读用 `extends`，写用 `super`（PECS） |
