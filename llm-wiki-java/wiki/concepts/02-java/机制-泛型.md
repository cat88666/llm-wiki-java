# 泛型与类型擦除

> 泛型将类型检查从运行期提前到编译期，类型擦除让 JVM 只维护一份字节码——编译期保证安全，运行期保证复用与向后兼容。

## 一、第一性原理

JDK 5 之前集合只能存 `Object`，取出时强制类型转换，类型错误在运行期才暴露（`ClassCastException`）。

泛型解决的根本问题：**把类型错误从运行期提前到编译期**。

Java 选择类型擦除（而非 C++ 模板的代码膨胀或 C# 的运行时具化）的核心权衡：
- 向后兼容 JDK 5 之前的字节码和序列化格式
- 避免为每种类型参数生成独立字节码（无代码膨胀）
- 代价：运行时丢失泛型类型信息

## 二、核心机制

### 2.1 类型擦除规则

| 泛型声明 | 擦除结果 |
|---------|---------|
| `<T>` 无限定 | `Object` |
| `<T extends Comparable>` | `Comparable` |
| `List<String>` / `List<Integer>` | 均为 `List`（同一份字节码） |

编译器在擦除后自动插入强制类型转换（checkcast 指令），保证运行时类型正确。

### 2.2 桥方法（Bridge Method）

泛型类继承时，擦除可能导致方法签名变化。编译器自动生成桥方法维护多态语义：

```java
class Node<T> { void setData(T data) {} }
class IntNode extends Node<Integer> {
    @Override void setData(Integer data) {}
    // 编译器生成桥方法：void setData(Object data) { setData((Integer) data); }
}
```

### 2.3 通配符与上下界

| 写法 | 语义 | 允许操作 | 典型场景 |
|------|------|---------|---------|
| `? extends T` | 上界（Producer） | 只读，不可写入（写入类型不确定） | 方法参数：读取集合元素 |
| `? super T` | 下界（Consumer） | 可写 T 及子类，读出只能当 Object | 方法参数：向集合写入元素 |
| `?` | 无界 | 只读，等价于 `? extends Object` | 仅关心"是否为空"等操作 |

**PECS 原则**：Producer Extends, Consumer Super。`Collections.copy(List<? super T> dest, List<? extends T> src)` 是经典示例。

### 2.4 泛型不支持协变

`List<String>` 不是 `List<Object>` 的子类——这与数组不同（`String[]` 是 `Object[]` 的子类）。数组协变是 Java 早期设计缺陷，运行时靠 `ArrayStoreException` 补救；泛型在编译期直接禁止，更安全。

## 三、Java 核心使用

### 3.1 泛型类 / 方法 / 接口

```java
// 泛型类
class Box<T> { private T value; }

// 泛型方法（类型参数声明在返回值前）
<T> List<T> singletonList(T item) { return List.of(item); }

// 泛型接口
interface Comparable<T> { int compareTo(T o); }
```

### 3.2 运行时获取泛型信息

擦除后泛型信息并未完全丢失——类/字段/方法签名中的泛型参数保留在字节码 `Signature` 属性中，可通过反射获取：

```java
// 获取字段的泛型类型
Field field = clazz.getDeclaredField("list");
ParameterizedType pt = (ParameterizedType) field.getGenericType();
Type actualType = pt.getActualTypeArguments()[0]; // String.class

// 获取父类的泛型参数（框架常用技巧）
Type superType = getClass().getGenericSuperclass();
```

### 3.3 TypeToken 技巧

Gson/Jackson 等框架用匿名子类保留泛型信息：

```java
Type type = new TypeToken<List<String>>(){}.getType();
// 匿名子类的父类泛型参数保留在字节码中，反射可读取
```

## 四、核心使用原则

1. **始终使用参数化类型**：不要使用原始类型 `List`，应使用 `List<String>`——原始类型丧失编译期检查。
2. **遵循 PECS**：方法参数中集合作为数据源用 `extends`，作为数据目标用 `super`。
3. **不能 `new T()`**：擦除后 T 变为 Object，无法确定构造器。需要实例化时传入 `Class<T>` 或 `Supplier<T>`。
4. **不能 `new T[]`**：同理，用 `(T[]) new Object[n]` 或 `Array.newInstance()` 替代。
5. **泛型方法优于泛型通配符**：当需要捕获类型关系时（如返回值类型与参数类型相关），用泛型方法 `<T>` 而非 `?`。

> **面试高频误区**：`List<String>` 和 `List<Integer>` 的 `.getClass()` 返回相同结果（都是 `ArrayList.class`），但这不意味着它们"相同"——编译器在编译期已保证类型安全，擦除仅影响运行时。

## 五、综合对比

| 维度 | Java 泛型 | C++ 模板 | C# 泛型 |
|------|----------|---------|---------|
| 实现方式 | 类型擦除（一份字节码） | 为每种类型生成专属代码 | 运行时具化（reification） |
| 运行时类型信息 | 丢失 | 保留 | 保留 |
| 代码膨胀 | 无 | 有 | 值类型有，引用类型共享 |
| 能力 | 受限（不能 `new T()`） | 强大（支持非类型参数、特化） | 较强（支持 `new T()`） |
| 基本类型支持 | 不支持（必须用包装类） | 支持 | 支持 |
| 向后兼容 | 与 JDK 5 前字节码兼容 | 无此约束 | 无此约束 |

## 六、生产风险

| 风险 | 根因 | 应对 |
|------|------|------|
| `instanceof` 无法检查泛型参数 | `list instanceof List<String>` 编译报错 | 用 `instanceof List<?>` 后手动检查元素类型 |
| 泛型方法重载冲突 | `void f(List<String>)` 和 `void f(List<Integer>)` 擦除后签名相同 | 改为不同方法名或合并为 `<T> void f(List<T>)` |
| 堆污染（Heap Pollution） | 混用原始类型与泛型类型，导致运行时 `ClassCastException` | 消除所有 unchecked warning |
| 泛型数组创建非法 | `new List<String>[10]` 编译报错 | 使用 `List<List<String>>` 替代 |
| 可变参数 + 泛型 | `@SafeVarargs` 场景下潜在堆污染 | 确保方法内不对 varargs 数组赋值 |
| 序列化丢失泛型 | JSON 反序列化 `List<T>` 时 T 信息丢失 | 使用 `TypeReference` / `TypeToken` 传递完整类型 |

## 七、与其他概念的关系

- 依赖 [[机制-动态代理]]（反射部分）：反射可绕过类型擦除，通过 `Field.getGenericType()` 获取泛型参数
- 约束了 [[概念-Java基础类型]]：泛型不支持基本类型 → 集合必须使用包装类 → 带来自动拆装箱性能开销
- 支撑集合框架：`List<E>`、`Map<K,V>`、`Optional<T>` 的类型安全全靠泛型
- 支撑框架设计：Spring `GenericTypeResolver`、Jackson `TypeReference` 均依赖泛型元信息

## 八、应用边界

| 适合 | 不适合 |
|------|--------|
| 容器类（集合、Optional、Future、Result） | 需要运行时精确类型信息（用 `Class<T>` 参数传递） |
| 通用工具 / 基础库 API | 基本类型高性能场景（用专用容器如 `IntStream`、Eclipse Collections） |
| 类型安全的 Builder / Factory 模式 | 需要 `new T()` 的场景（传入 Supplier/Class 替代） |
| 框架级类型推导与注入 | 跨序列化边界（需 TypeToken 辅助） |
