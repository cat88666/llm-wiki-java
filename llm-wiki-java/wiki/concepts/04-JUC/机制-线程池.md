# 线程池

> 线程池是池化技术在并发编程中的应用——预先创建一批线程常驻内存，任务到来时直接复用，避免频繁创建/销毁线程的系统调用开销；Java 的核心实现 `ThreadPoolExecutor` 通过 7 个构造参数控制核心线程、最大线程、任务队列、拒绝策略的行为，用一个 `ctl` 变量同时编码线程池状态和工作线程数。

## 一、第一性原理

线程的创建和销毁是操作系统级别的系统调用（需要分配栈内存、OS 调度等），代价远高于任务执行本身。

线程池的存在理由：**通过复用已创建的线程，把线程的生命周期摊薄到整个应用程序运行期；同时通过有界队列 + 拒绝策略防止任务无限堆积压垮系统**。

两个约束缺一不可：
- 无有界队列 → 任务堆积 OOM
- 无最大线程数限制 → 线程数爆炸，资源耗尽

## 二、核心机制

### 7 个构造参数

```java
new ThreadPoolExecutor(
    int corePoolSize,         // 核心线程数：常驻线程（空闲也不销毁，除非设 allowCoreThreadTimeOut）
    int maximumPoolSize,      // 最大线程数：corePoolSize + 临时线程上限
    long keepAliveTime,       // 临时线程空闲存活时间（超时销毁）
    TimeUnit unit,            // keepAliveTime 的时间单位
    BlockingQueue<Runnable> workQueue,  // 任务等待队列
    ThreadFactory threadFactory,        // 线程创建工厂（可设线程名前缀、守护状态）
    RejectedExecutionHandler handler    // 拒绝策略（队列满 + 达到最大线程数时触发）
)
```

### 任务执行流程（execute 方法）

```
提交任务 execute(task)
  ├── 线程数 < corePoolSize
  │     → 创建新核心线程，直接执行 task（addWorker(task, true)）
  │
  ├── 线程数 >= corePoolSize
  │     → 尝试入队（workQueue.offer(task)）
  │         ├── 入队成功 → 等待空闲 Worker 来 getTask()
  │         └── 队列已满
  │               ├── 线程数 < maximumPoolSize
  │               │     → 创建临时线程执行 task（addWorker(task, false)）
  │               └── 线程数 >= maximumPoolSize
  │                     → 执行拒绝策略 handler.rejectedExecution(task, this)
```

**记忆口诀**：**核心线程 → 队列 → 最大线程 → 拒绝**

⚠️ 常见误解：队列不满时不会创建临时线程；只有队列**满了**，才会在线程数 < maximumPoolSize 时创建临时线程。

### Worker 机制（线程复用实现）

`Worker` 实现了 `Runnable`，每个 Worker 包含：
- `firstTask`：创建时分配的第一个任务
- `Thread thread`：该 Worker 持有的工作线程

**复用循环**：
```java
// Worker.run() 核心逻辑（简化）
void runWorker(Worker w) {
    Runnable task = w.firstTask;
    while (task != null || (task = getTask()) != null) {
        task.run();   // 执行任务
        task = null;
    }
    // getTask() 返回 null → 线程销毁（keepAliveTime 超时或池关闭）
}
```

`getTask()` 调用 `workQueue.poll(keepAliveTime)` 或 `workQueue.take()`，实现超时退出（临时线程）或无限等待（核心线程）。

### ctl 变量设计

`ThreadPoolExecutor` 用一个 `AtomicInteger ctl` 同时编码两个信息：

```
ctl（32位）：高3位 = 线程池状态，低29位 = 工作线程数
状态：RUNNING(-1) / SHUTDOWN(0) / STOP(1) / TIDYING(2) / TERMINATED(3)
```

用一个 CAS 原子变量同时管理状态和线程数，避免两个变量之间的竞争窗口。

## 三、拒绝策略

| 策略 | 行为 | 适用场景 |
|------|------|---------|
| `AbortPolicy`（**默认**）| 抛 `RejectedExecutionException` | 绝大多数场景，让调用方感知过载 |
| `DiscardPolicy` | 静默丢弃新任务（无感知）| 日志、统计等非关键任务 |
| `DiscardOldestPolicy` | 丢弃队列最老任务，重试提交新任务 | 实时性优先（新任务比旧任务重要）|
| `CallerRunsPolicy` | 由提交任务的线程自己执行 | 降速反压，不丢任务但会阻塞生产者线程 |

**自定义策略**：实现 `RejectedExecutionHandler`，常见自定义：
- 记录被拒绝任务的监控报警
- 任务持久化到 DB/MQ 等待后续重试
- 动态扩容线程池再重试

## 四、Executors 的坑

不建议使用 `Executors` 工厂方法的原因：

| 工厂方法 | 坑 | 风险 |
|---------|-----|------|
| `newFixedThreadPool(n)` | `workQueue = new LinkedBlockingQueue<>()`（无界）| 任务堆积 OOM |
| `newCachedThreadPool()` | `maximumPoolSize = Integer.MAX_VALUE` | 线程数爆炸，资源耗尽 |
| `newSingleThreadExecutor()` | 同 newFixedThreadPool，无界队列 | OOM 风险 |
| `newScheduledThreadPool()` | 同 newCachedThreadPool，无界线程 | 线程数爆炸 |

**推荐**：手动构造 `ThreadPoolExecutor`，明确设置有界 `workQueue` 和合理的 `maximumPoolSize`，并给线程命名（便于排查）。

## 五、参数调优

### 核心线程数（corePoolSize）

| 任务类型 | 计算公式 | 说明 |
|---------|---------|------|
| **CPU 密集型** | `CPU 核数 + 1` | +1 防止偶发缺页中断时 CPU 空闲 |
| **IO 密集型** | `CPU 核数 × 2` | 线程大量阻塞在 IO，需更多线程保持 CPU 忙碌 |
| **混合型** | `CPU 核数 × (1 + IO耗时/CPU耗时)` | 根据实际 IO 占比调整 |

**实践建议**：以公式为起点，通过压测观察 CPU 利用率和任务延迟，动态调整。

### 任务队列（workQueue）

| 队列类型 | 特性 | 适用 |
|---------|------|------|
| `ArrayBlockingQueue(n)` | 有界，FIFO，线程安全 | **推荐**，防 OOM |
| `LinkedBlockingQueue(n)` | 有界（传 n）/ 无界（不传 n），FIFO | 传有界大小时推荐 |
| `SynchronousQueue` | 不存储，直接交付（newCachedThreadPool 用）| 需要 maximumPoolSize 较大配合 |
| `PriorityBlockingQueue` | 优先级排序，无界 | 任务有优先级时使用 |

## 六、生产监控与动态线程池

### 关键监控指标

```java
ThreadPoolExecutor pool = (ThreadPoolExecutor) executor;
pool.getActiveCount()      // 当前活跃线程数
pool.getCompletedTaskCount() // 已完成任务数
pool.getQueue().size()     // 队列中等待任务数
pool.getLargestPoolSize()  // 历史最大线程数
pool.getTaskCount()        // 已提交总任务数
```

**报警阈值**：队列积压超过队列容量的 80%、活跃线程数长时间等于 maximumPoolSize → 考虑扩容或排查慢任务。

### 动态线程池

`ThreadPoolExecutor` 提供运行期调整方法：

```java
pool.setCorePoolSize(newCore);       // 动态调整核心线程数
pool.setMaximumPoolSize(newMax);     // 动态调整最大线程数
pool.setKeepAliveTime(time, unit);   // 动态调整存活时间
```

开源方案：Hippo4j（美团）、Dynamic-Thread-Pool（支持配置中心动态下发线程池参数）。

## 七、综合对比

### ThreadPoolExecutor vs ForkJoinPool

| 维度 | ThreadPoolExecutor | ForkJoinPool |
|------|-------------------|-------------|
| 设计目标 | 通用任务执行 | 递归分治任务（fork/join）|
| 队列 | 共享一个任务队列 | 每个线程独立队列 + 工作窃取（Work Stealing）|
| 适用任务 | IO 密集、独立任务 | CPU 密集、可分解的递归任务（如并行排序、并行 Stream）|
| 线程数 | 手动配置 | 默认 CPU 核数-1 |
| Java API | `ExecutorService` | `CompletableFuture.commonPool()`、`parallelStream()` 背后 |

**工作窃取（Work Stealing）**：ForkJoinPool 中空闲线程从其他线程的队列尾部"偷"任务执行，减少资源浪费。

## 八、与其他概念的关系

- 内部用 [[机制-AQS]] 实现线程阻塞等待：`workers` 集合的访问用 `ReentrantLock` 保护，`BlockingQueue` 基于 Condition
- 内部用 [[机制-CAS]] 原子更新状态：`ctl` 变量（线程数 + 池状态）用 `AtomicInteger.CAS` 修改
- 与 [[概念-ThreadLocal]] 交互有内存泄漏风险：线程复用时旧任务的 ThreadLocal 值残留，需任务结束后 `remove()`
- 被 [[机制-CompletableFuture]] 依赖：`supplyAsync` 默认使用 ForkJoinPool；IO 密集场景必须自定义线程池传入

## 九、应用边界

**任何需要多线程异步处理的场景都应该用线程池**，避免手动 `new Thread`（无法控制资源、无法复用）。

**不要用线程池的场景**：
- JDK 21+ 虚拟线程（Project Loom）：每个任务一个虚拟线程，虚拟线程是轻量级的（类似协程），阻塞 IO 不占平台线程，让"每请求一线程"模型重新可行
- 简单的后台定时任务：用 `ScheduledThreadPoolExecutor` 或 Spring `@Scheduled`

**容量规划经验**：
- 高峰 QPS × 平均任务耗时 = 所需并发线程数
- 核心线程 = 正常负载所需并发数，最大线程 = 峰值 1.5-2 倍
- 队列容量 = 核心线程数 × 峰值任务耗时（秒）× 合理等待上限
