# 线程

> 线程是 CPU 调度的最小单位，是程序并发执行的基本手段；Java 线程经历 NEW → RUNNABLE → BLOCKED/WAITING/TIMED_WAITING → TERMINATED 六种状态，通过 JMM 共享内存 + 各类同步机制实现线程间协调。

## 一、线程解决的执行问题

**进程 vs 线程**：

| | 进程 | 线程 |
|--|------|------|
| 是什么 | 资源分配的最小单位 | CPU 调度的最小单位 |
| 拥有 | 独立内存空间（虚拟地址、文件描述符）| 只有栈、程序计数器；共享进程的堆和方法区 |
| 切换代价 | 高（上下文切换需保存/恢复整套地址空间）| 低（只保存/恢复少量寄存器）|
| 通信 | IPC（管道/信号/共享内存）| 直接共享内存（需同步）|
| 崩溃影响 | 进程崩溃不影响其他进程 | 线程崩溃可能导致整个进程崩溃 |

**并发 vs 并行**：
- **并发（Concurrency）**：同一时间段内多个任务交替执行（单核时间片轮转），宏观上"同时进行"
- **并行（Parallelism）**：同一时刻多个任务真正同时执行（多核 CPU 各核独立运行）

**线程的存在理由**：单线程程序等待 IO 时 CPU 空闲。线程让一个进程可以同时做多件事，充分利用多核 CPU，提升响应速度和吞吐量。代价是引入了共享内存下的并发安全问题（详见 [[概念-JMM]]）。

## 二、Java 线程状态流转

Java 线程有 **6 种状态**（`Thread.State` 枚举）：

```
NEW ──(start())──→ RUNNABLE ──┬──(等CPU时间片)──→ RUNNING(OS层面)
                               ├──(yield()/时间片到期)──→ RUNNABLE
                               ├──(sleep(ms))──────────→ TIMED_WAITING
                               ├──(wait(ms))───────────→ TIMED_WAITING
                               ├──(join(ms))───────────→ TIMED_WAITING
                               ├──(parkNanos/Until())──→ TIMED_WAITING
                               ├──(wait())─────────────→ WAITING（必须notify/notifyAll唤醒）
                               ├──(join())─────────────→ WAITING
                               ├──(park())─────────────→ WAITING
                               ├──(竞争synchronized失败)→ BLOCKED（等待锁释放）
                               └──(执行结束)────────────→ TERMINATED
```

**各状态说明**：

| 状态 | 含义 | 触发方法 | 唤醒条件 |
|------|------|---------|---------|
| NEW | 已创建，未启动 | `new Thread()` | 调用 `start()` |
| RUNNABLE | 可运行（含正在运行和就绪）| `start()` 后 | — |
| BLOCKED | 阻塞，等待 **synchronized 锁** | 竞争 `synchronized` 失败 | 锁释放后 JVM 自动唤醒 |
| WAITING | 无限期等待，必须显式唤醒 | `wait()` / `join()` / `park()` | `notify()` / `join线程终止` / `unpark()` |
| TIMED_WAITING | 超时等待，到时自动恢复 | `sleep(ms)` / `wait(ms)` / `join(ms)` / `parkNanos()` | 超时或显式唤醒 |
| TERMINATED | 已结束 | 正常执行完 / 未捕获异常 | — |

**BLOCKED vs WAITING 区别**：
- BLOCKED：只与 `synchronized` 相关，等待操作系统分配监视器锁，锁释放后 JVM 自动竞争
- WAITING/TIMED_WAITING：主动放弃 CPU，必须靠 `notify/unpark/join结束` 等事件唤醒；`Lock` 接口等待进入的是 WAITING，不是 BLOCKED

## 三、线程创建与任务封装

**5 种方式对比**：

| 方式 | 返回值 | 异常 | 推荐度 | 说明 |
|------|--------|------|--------|------|
| 继承 `Thread` | 无 | 不可抛 checked | ❌ | 扩展性差，单继承限制 |
| 实现 `Runnable` | 无 | 不可抛 checked | ✅ | 轻量，可与线程池配合 |
| 实现 `Callable<T>` | ✅ 有 | ✅ 可抛 | ✅ | 需配合 FutureTask 或线程池 |
| `FutureTask<T>` | ✅ 有 | ✅ 可抛 | ✅ | 包装 Callable，本身实现 Runnable |
| **线程池**（推荐）| 可选 | 可选 | ⭐⭐⭐ | 复用线程、资源可控，生产首选 |

**Callable + FutureTask 用法**：

```java
Callable<String> task = () -> {
    // 可抛 checked exception，有返回值
    return "结果";
};
FutureTask<String> ft = new FutureTask<>(task);
new Thread(ft).start();
String result = ft.get();  // 阻塞等待结果；可传超时参数
```

`FutureTask` 状态机：NEW → COMPLETING → NORMAL（正常完成）/ EXCEPTIONAL（异常）/ CANCELLED。

## 四、Thread 与 Object 方法辨析

| 方法 | 定义在 | 是否释放锁 | 状态变化 | 说明 |
|------|--------|-----------|---------|------|
| `start()` | Thread | — | NEW → RUNNABLE | 启动新线程，调用 OS 创建线程 |
| `run()` | Thread/Runnable | — | 无变化 | 直接调用等于普通方法，不创建新线程 |
| `sleep(ms)` | Thread（静态）| ❌ 不释放锁 | → TIMED_WAITING | 当前线程暂停，时间到自动恢复；不释放持有的锁 |
| `wait()` | Object | ✅ 释放锁 | → WAITING | 必须在 synchronized 块内调用；释放锁后进等待队列 |
| `notify()` | Object | — | 唤醒一个 WAITING 线程 | 随机唤醒，唤醒后需重新竞争锁 |
| `notifyAll()` | Object | — | 唤醒所有 WAITING 线程 | 推荐用 notifyAll 避免漏唤醒 |
| `join()` | Thread | — | 调用方 → WAITING | 等待目标线程执行完毕；底层用 wait() |
| `yield()` | Thread（静态）| ❌ 不释放锁 | RUNNING → RUNNABLE | 让出 CPU 时间片，但不保证其他线程立即获得 |

**`sleep()` vs `wait()` 关键区别**：sleep 不释放锁、定义在 Thread、可在任意地方调用；wait 释放锁、定义在 Object、必须在 synchronized 块中调用。

## 五、线程协作通信方式

| 机制 | 适用场景 | 核心 API |
|------|---------|---------|
| **volatile** | 简单的状态标志可见性 | `volatile boolean flag` |
| **synchronized + wait/notify** | 经典生产者-消费者、条件等待 | `synchronized(lock) { lock.wait(); lock.notifyAll(); }` |
| **ReentrantLock + Condition** | 多条件队列（如读写分离通知）| `condition.await(); condition.signal()` |
| **LockSupport.park/unpark** | 底层线程挂起/唤醒（AQS内部使用）| `LockSupport.park(thread); LockSupport.unpark(thread)` |
| **Thread.join()** | 等待另一个线程执行完再继续 | `t.join()` |
| **interrupt()** | 协作式中断（通知线程应停止）| `t.interrupt()` |

**wait/notify 使用规范**（避免虚假唤醒）：

```java
synchronized (lock) {
    while (!condition) {  // 必须用 while，不能用 if（防虚假唤醒）
        lock.wait();
    }
    // 条件满足，处理业务
}
```

## 六、协作式中断机制

Java 的中断是**协作式**的，不是强制终止——调用 `interrupt()` 只是设置一个中断标志位，线程自己决定如何响应。

| 方法 | 静态/实例 | 作用 | 是否清除标志位 |
|------|---------|------|--------------|
| `t.interrupt()` | 实例 | 设置目标线程中断标志为 true；若线程在 wait/sleep/join 中则抛 InterruptedException | — |
| `t.isInterrupted()` | 实例 | 查询目标线程的中断标志 | ❌ 不清除 |
| `Thread.interrupted()` | 静态 | 查询**当前线程**的中断标志 | ✅ 清除（重置为 false）|

**InterruptedException 处理规范**：

```java
// 错误：吞掉中断，线程无法感知
try { Thread.sleep(1000); } catch (InterruptedException e) { /* 忽略 */ }

// 正确1：重新设置中断标志，让上层知道
try { Thread.sleep(1000); } catch (InterruptedException e) { Thread.currentThread().interrupt(); }

// 正确2：直接向上抛出
void method() throws InterruptedException { Thread.sleep(1000); }
```

**中断作为停止线程的标准方式**：

```java
while (!Thread.currentThread().isInterrupted()) {
    // 执行任务
}
// 或在 catch InterruptedException 后 break/return
```

## 七、死锁、活锁与饥饿

**死锁四个必要条件**（缺一则无死锁）：

| 条件 | 说明 |
|------|------|
| 资源互斥 | 资源同一时刻只能被一个线程持有 |
| 占有并等待 | 线程持有资源同时等待其他资源 |
| 不可剥夺 | 已持有的资源不能被强制释放 |
| 循环等待 | 线程间形成资源依赖环：A→B→C→A |

**避免死锁的手段**：

```
1. 加锁顺序固定（资源排序）：所有线程按相同顺序申请锁，打破循环等待
2. tryLock 超时：ReentrantLock.tryLock(timeout) 获取不到则放弃，避免永久等待
3. 锁粒度最小化：持有锁的时间尽量短，减少交叉等待概率
4. 死锁检测：ThreadMXBean.findDeadlockedThreads() + jstack / Arthas 分析
```

**活锁 vs 饥饿**：
- **活锁**：线程不断重试但都互相谦让，无法推进（没有阻塞但也没有进展）
- **饥饿**：某线程长期得不到 CPU/锁（高优先级线程持续抢占，低优先级永远等待）

## 八、线程问题排查路径

### 常用工具

| 工具 | 用途 | 典型命令 |
|------|------|---------|
| `jstack` | 生成线程堆栈快照，分析 BLOCKED/死锁 | `jstack <pid> > dump.txt` |
| `jconsole` | 图形化监控线程状态、死锁检测 | GUI 工具 |
| `jvisualvm` | 图形化 CPU/内存/线程分析 | GUI 工具 |
| **Arthas** | 线上不停服诊断，推荐生产使用 | `thread -n 5`（最忙线程）/ `thread -b`（死锁）|

### 排查思路

**场景1：CPU 飙高**
```
1. top 找到高CPU进程 → top -Hp <pid> 找高CPU线程 → printf "%x\n" <tid> 转16进制
2. jstack <pid> | grep <16进制tid> → 查看线程在做什么
   → RUNNABLE + 死循环/密集计算：优化算法或加限速
   → RUNNABLE + IO等待：IO密集，增大线程数或使用异步IO
3. Arthas: thread -n 5 直接展示最忙5个线程堆栈
```

**场景2：死锁**
```
1. jstack <pid> 输出末尾会有 "Found X deadlock" 报告
2. Arthas: thread -b 直接显示死锁信息
3. 分析死锁链：线程A持有lock1等待lock2，线程B持有lock2等待lock1
4. 修复：统一加锁顺序 或 改用tryLock超时
```

**场景3：线程池耗尽 / 大量 BLOCKED**
```
jstack <pid> | grep "java.lang.Thread.State" | sort | uniq -c
→ 大量 BLOCKED：锁竞争激烈，缩小锁粒度/减少锁持有时间
→ 大量 WAITING：生产者-消费者不匹配，或忘记唤醒 → 检查 notify/notifyAll
→ 大量 TIMED_WAITING：超时设置不合理，或线程池队列积压
```

**故障排查四原则**：
1. BLOCKED 只与 `synchronized` 相关，与 `Lock` 接口无关
2. WAITING/TIMED_WAITING 涉及线程间协作（wait/park/join）
3. 线程卡死 → 检查 BLOCKED/WAITING → 分析锁/条件竞争
4. CPU 飙升 → 检查 RUNNABLE 线程 → 是否死循环 / 算法复杂度过高

## 九、线程使用权衡

| 权衡点 | 结论 |
|--------|------|
| 线程开销 | 每个线程默认栈 512KB~1MB；线程切换有上下文开销；高并发场景优先用线程池 |
| sleep vs wait | sleep 不释放锁适合纯等待；wait 释放锁适合条件等待（生产者-消费者）|
| notify vs notifyAll | notify 随机唤醒一个（有漏唤醒风险）；notifyAll 唤醒所有但性能低；生产推荐 Condition 精确唤醒 |
| 协作式中断 vs 强制停止 | Java 只支持协作式中断（interrupt()），无法强制终止（stop() 已废弃）；设计上需线程自行检查 isInterrupted() |
| 线程数设置 | CPU 密集型：核心数 + 1；IO 密集型：核心数 × 2（详见 [[机制-线程池]]）|

## 十、线程与并发组件关系

- **[[概念-JMM]]**：线程间通过主内存/工作内存通信，JMM 定义了可见性/有序性/原子性规则，是所有线程安全问题的底层规范
- **[[机制-Synchronized]]**：基于对象监视器（Monitor）的管程实现，解决线程互斥与 wait/notify 协调
- **[[机制-AQS]]**：ReentrantLock/CountDownLatch/Semaphore 等 JUC 工具的骨架，底层是 CLH 队列 + CAS + park/unpark
- **[[机制-CAS]]**：无锁算法的基础，AtomicInteger 等原子类的实现原理
- **[[机制-线程池]]**：生产环境管理线程的标准方式，复用线程、限制资源消耗
- **[[概念-ThreadLocal]]**：线程本地存储，每个线程独享变量副本，避免共享带来的并发问题

## 十一、线程使用边界

**直接创建线程（new Thread）适用**：
- 简单一次性后台任务，生命周期明确
- 学习/测试场景

**Callable + FutureTask 适用**：
- 需要获取任务结果，或需要感知任务异常
- 与线程池配合（`executor.submit(callable)` 返回 `Future`）

**线程池（生产首选）**：
- 所有并发任务，复用线程、控制资源、支持监控告警

**不适合直接使用线程的场景**：
- 高并发场景（成千上万请求）：线程太多 → OOM；用线程池 + 队列
- 需要调度（定时/延迟）任务：用 `ScheduledThreadPoolExecutor`
- 响应式/异步编排：用 `CompletableFuture`（见 [[机制-CompletableFuture]]）
