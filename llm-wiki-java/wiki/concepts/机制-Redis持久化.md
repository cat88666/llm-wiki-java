# Redis 持久化

> Redis 提供两种持久化机制：RDB（定期全量快照）和 AOF（追加写命令日志）；两者各有权衡，Redis 4.0+ 引入混合持久化结合两者优点，是当前生产推荐方案。

## 第一性原理

Redis 是内存数据库，进程崩溃或机器断电会导致数据全部丢失。持久化的本质是把内存数据定期/实时写入磁盘，重启时从磁盘恢复。核心矛盾是**数据安全性（丢失数据少）vs 性能开销（写磁盘的代价）**：全量快照开销大但恢复快，增量日志开销小但文件大且恢复慢。

## 核心机制

### RDB（Redis Database Backup）— 全量快照

**原理**：将某一时刻内存的完整数据以二进制压缩格式存储为 `.rdb` 文件。

**触发方式**：
- `BGSAVE`（主要方式）：`fork()` 子进程负责写快照，父进程继续处理命令
- `SAVE`：阻塞主线程，生产禁用
- 配置触发：`save 900 1`（900 秒内至少 1 次写操作则触发 BGSAVE）
- 主从复制：新从节点全量同步时触发

**fork + COW（Copy-On-Write）**：
```
父进程 fork → 子进程获得相同页表（共享物理内存，页面只读标记）
父进程收到写请求 → 触发 COW，操作系统复制该页给父进程修改
子进程继续读原始数据 → 生成一致性快照
```

**优点**：
- 文件紧凑（二进制压缩），恢复速度快
- fork 后主进程不阻塞（除 fork 瞬间）

**缺点**：
- 最多丢失上次快照到故障点之间的数据（分钟级丢失）
- 大内存实例 fork 慢（复制页表），可能瞬间卡顿

### AOF（Append Only File）— 写命令日志

**原理**：每次写操作后将命令以文本格式追加到 `.aof` 文件，恢复时重放所有命令。

**AOF 三种写回策略**（`appendfsync` 配置）：

| 策略 | 触发时机 | 数据安全 | 性能 | 适用场景 |
|------|---------|---------|------|---------|
| **Always** | 每条命令写后立即 fsync | 最高（最多丢1条）| 最低 | 金融、不能丢数据 |
| **Everysec**（默认）| 每秒 fsync 一次 | 高（最多丢1秒）| 中 | 大多数场景 |
| **No** | 由 OS 决定 fsync 时机 | 低 | 最高 | 对数据丢失不敏感 |

**AOF 重写**：随时间 AOF 文件膨胀（同一 key 被写了 1000 次，只需保留最终值）
```
BGREWRITEAOF：fork 子进程，读当前内存数据生成等效的精简命令集
（不是重放旧 AOF，是重新生成）
重写期间新命令写入旧 AOF + AOF 重写缓冲区，子进程完成后合并追加
```

**优点**：数据安全性高，可读性强（文本格式便于人工修复）

**缺点**：文件比 RDB 大；重放速度比 RDB 恢复慢；Everysec 下最多丢 1 秒数据

### 混合持久化（Redis 4.0+，推荐）

```
AOF 重写时，前半段写 RDB 格式（快照），后半段写增量 AOF 命令
┌──────────────────────────────────────────────────┐
│  RDB 快照（某时刻全量，二进制压缩，恢复快）           │
│  AOF 增量（快照后的写命令，文本格式，保证最新数据）    │
└──────────────────────────────────────────────────┘
```

恢复时：先加载 RDB 快照（快），再回放后半段 AOF（少量），兼顾恢复速度和数据安全。

**生产推荐配置**：
```
appendonly yes          # 开启 AOF
appendfsync everysec    # 每秒 fsync，平衡安全与性能
aof-use-rdb-preamble yes  # 开启混合持久化（4.0+）
```

## 关键权衡

| 维度 | RDB | AOF (Everysec) | 混合 |
|------|-----|----------------|------|
| 数据丢失 | 分钟级 | ≤1 秒 | ≤1 秒 |
| 恢复速度 | 快 | 慢（重放命令）| 快（RDB+少量AOF）|
| 文件大小 | 小（压缩）| 大（所有命令）| 中 |
| 对性能影响 | fork 瞬间卡顿 | 每秒 fsync 有抖动 | 同 AOF |
| 适用场景 | 可接受分钟级丢失 | 对丢失敏感 | 生产推荐 |

**核心权衡**：`appendfsync always` 让 Redis 降级为磁盘速度，与关系型数据库相当但失去了内存速度优势；`everysec` 是工程上的合理折中。

## 与其他概念的关系

- 类比 [[机制-MySQL三种日志]]：AOF ≈ binlog（追加写操作日志），RDB ≈ 数据库备份文件；两阶段提交思想在混合持久化中也有体现
- 被 [[机制-Redis集群与高可用]] 依赖：主从全量同步使用 RDB，主从增量同步相当于 AOF 传播
- 与 [[概念-Redis数据类型与底层结构]] 关联：持久化时不需要了解底层编码，但重写时读内存数据生成命令需要理解数据结构

## 应用边界

**什么时候只用 RDB**：缓存场景，数据来源有其他系统，Redis 只是加速层，丢数据可以从 DB 重建

**什么时候只用 AOF**：数据只在 Redis 中（Redis 是主存储），对数据安全性要求高

**生产建议**：开启混合持久化（`aof-use-rdb-preamble yes`）+ Everysec 写回策略

**不要做的事**：
- 不要在主节点用 `SAVE`（阻塞主线程）
- 不要关闭 AOF 重写（文件会无限增长）
- 大内存实例（>10GB）fork 可能耗时数秒，规划好 maxmemory
