# 分布式系统理论

> CAP 理论揭示了分布式系统在一致性/可用性/分区容错性三者中最多只能同时保证两项；BASE 理论是对 CAP 的工程化延伸，以最终一致性换取高可用；一致性哈希解决节点动态变化时数据迁移最小化问题；Raft 是工程上最主流的共识算法，支撑 etcd/TiKV 等核心基础设施。

## 一、第一性原理

分布式系统的本质困境来自**网络**：节点间通信可能延迟或中断（分区），而业务要求系统随时可用且数据正确。CAP 证明了这三件事不可能同时完美实现，而 BASE 则给出了工程师的务实答案——**不追求强一致，转而以软状态 + 重试推进到最终一致**。

网络分区是分布式系统的自然事实（机房间网络抖动、数据中心故障），P 不可放弃。因此 CAP 的真正选择是 CP 还是 AP——是宁可服务不可用，还是允许读到旧数据。

## 二、CAP 理论

### 三个属性

| 属性 | 定义 | 实现代价 |
|------|------|---------|
| **C（Consistency）** | 每次读都能获取到最新写入的数据，或者返回错误 | 网络分区期间必须阻塞写，牺牲可用性 |
| **A（Availability）** | 每个请求都会收到非错误响应（但可能是旧数据）| 可能读到过期数据，牺牲一致性 |
| **P（Partition Tolerance）** | 网络分区（节点间消息丢失/延迟）时系统仍能运行 | 分布式系统必须支持，不可放弃 |

### CP vs AP 选型

| 选型 | 特征 | 代表系统 |
|------|------|---------|
| **CP** | 网络分区时宁可返回错误或阻塞，不返回不一致数据 | ZooKeeper（顺序一致性）、HBase、etcd |
| **AP** | 网络分区时返回旧数据保证可用，事后再同步 | Eureka、Cassandra、Nacos（注册中心模式）|

> ZooKeeper 保证的是**顺序一致性**（所有节点写入顺序一致，半数节点写成功才确认），而非线性一致性（实时最新值）。想要线性一致性：调用 `sync()` 强制 Follower 与 Leader 同步后再读。

### 一致性模型层次

```
强一致性 ─┬─ 线性一致性（最强，实时最新值，Paxos/Raft 提供）
          └─ 顺序一致性（全局操作顺序一致，不要求实时，ZK 提供）
弱一致性 ─┬─ 因果一致性（有因果关系的操作保序）
          └─ 会话一致性（同一会话内读到自己的写）
最终一致性（最弱，经过有限时间后所有副本收敛）
```

## 三、BASE 理论

BASE 是对 ACID（传统数据库强一致）的反命题，是 CAP 中 AP 的工程化落地：

| 属性 | 含义 | 实现手段 |
|------|------|---------|
| **Basically Available**（基本可用）| 出现故障时允许局部降级，保证核心功能可用 | 返回降级页/限流/熔断，而不是直接 500 |
| **Soft State**（软状态）| 允许数据存在中间过渡状态（INIT→PROCESSING→SUCCESS）| 状态机设计，记录操作流水 |
| **Eventual Consistency**（最终一致性）| 经过有限时间后，所有副本收敛到相同状态 | 消息队列 + 重试 + 对账补偿 |

**核心口诀**：**中间状态（软状态）+ 重试（最终一致性）+ 降级（基本可用）**

**与 ACID 的对比**：

| | ACID | BASE |
|--|------|------|
| 一致性 | 强一致，事务提交即一致 | 最终一致，允许中间态 |
| 可用性 | 分区时可能不可用（CP）| 分区时基本可用（AP）|
| 典型场景 | 传统单机数据库（MySQL 单节点）| 大规模分布式系统 |

**软状态陷阱**：PROCESSING 状态必须有超时兜底机制（如 30min 后自动判为失败），防止永久卡在中间状态。

### 服务雪崩、限流、熔断与降级

**服务雪崩**：A 调用 B，B 调用 C——C 不可用时，B 的请求堆积耗尽线程池，进而 A 也不可用，故障沿调用链级联扩散。

```
正常：A → B → C（可用）
雪崩：A → B → C（宕机）→ B 线程堆积 → A 超时 → 全链路不可用
```

**解决方案三层防护**：

| 手段 | 触发时机 | 作用 |
|------|---------|------|
| **服务限流** | 请求量超过阈值 | 拒绝多余请求，保护下游不被压垮 |
| **服务熔断** | 下游失败率超阈值 | 上游快速失败，不再调用故障节点 |
| **服务降级** | 系统整体负载过高 | 主动关闭/限流非核心功能，保障核心链路 |

**服务熔断（Circuit Breaker）**：下游服务不可用时，上游触发"断路器打开"，直接快速失败，不再将请求发到故障节点。渐进式恢复策略：

```
失败率 10% → 减少 50% 流量到该节点
失败率 50% → 减少 80% 流量
熔断打开 → 半开探测（少量流量试探）→ 成功则逐步恢复：30% → 50% → 100%
```

**服务降级（Service Degradation）**：系统压力过载时，主动通过配置开关（配置中心动态修改）关闭非核心业务，返回兜底值（FailOver 快速失败）。例如：大促期间关闭"猜你喜欢"、"历史浏览记录"等非核心接口。

**熔断 vs 降级的区别**：

| 维度 | 熔断 | 降级 |
|------|------|------|
| 触发原因 | **下游服务故障**（失败率/超时率超阈值）| **系统整体负载过高**（CPU/内存/QPS 超限）|
| 方向 | 防御性：保护调用方不被拖垮 | 主动性：主动放弃非核心功能 |
| 操作对象 | 关闭对某个下游的调用 | 关闭/限流本服务的某些功能 |
| 相同点 | 均防止系统崩溃，均导致部分功能暂时不可用 | |

## 四、Raft 共识算法

**解决问题**：多个节点如何就同一个值达成一致（即使部分节点故障）。CAP 理论证明了 CP 是可能的，共识算法是实现 CP 的工程路径。

**Paxos** 是理论基础（Leslie Lamport 1989），证明了在异步网络中共识是可解的，但实现极复杂。**Raft**（Diego Ongaro 2014）为"易于理解"而设计，工程上替代 Paxos。

### Leader 选举

节点三种角色：**Leader**（处理所有读写）/ **Follower**（被动）/ **Candidate**（选举中）

```
Follower 超过随机超时（150-300ms）未收到 Leader 心跳
    → 升为 Candidate，发起选举
    → 获得多数票（N/2 + 1）→ 成为新 Leader
    → 立即发送心跳抑制其他选举
```

**任期（Term）**：单调递增的逻辑时钟，每次选举产生新任期，节点收到更高 Term 的消息自动降为 Follower（防脑裂）。随机超时（150-300ms）是防止多个节点同时发起选举造成选票分裂的关键设计。

### 日志复制

```
Client Write → Leader 追加日志（uncommitted）
             → 并行复制给所有 Follower
             → 多数 Follower 确认 → Leader commit（已提交）
             → 通知 Follower commit → 响应 Client
```

日志一旦多数节点确认即 committed，后续 Leader 选举必须保证新 Leader 拥有所有 committed 日志。

### 安全性

只有拥有最新 committed 日志的节点才能赢得选举（投票前检查 log index + term），保证已提交的日志永不丢失。

**工程应用**：etcd（K8s 状态存储）、TiKV（TiDB 底层）、CockroachDB、Consul、Nacos CP 模式（JRaft）均基于 Raft。

## 五、一致性哈希

**解决问题**：普通取模哈希（`hash(key) % n`）在节点数 n 变化时，几乎所有数据都要重新映射（代价 O(n)）。一致性哈希让节点增删时只有 O(1/n) 的数据需要迁移。

### 核心机制

```
1. 构建哈希环（0 ~ 2^32-1 首尾相连）
2. 节点映射：hash(node_name) % 2^32 → 节点落在环上某位置
3. 数据映射：hash(key) % 2^32 → 数据落在环上某位置
4. 顺时针查找：数据沿顺时针方向找到的第一个节点，即为存储节点
```

**增删节点的影响**：
- 新增节点：只有该节点到其前驱节点之间的数据需要迁移，其余数据不变
- 删除节点：只有该节点上的数据需要迁移到其顺时针后继节点

### 虚拟节点（解决 Hash 倾斜）

当物理节点数少时，节点可能分布不均（热点）。将一个物理节点映射为多个虚拟节点（如 `node1#1, node1#2, ..., node1#150`），使数据分布更均匀。

```
物理节点  →  虚拟节点（分散在环上）  ←  数据（映射到最近虚拟节点，再路由到物理节点）
```

通常每个物理节点配置 100-200 个虚拟节点，兼顾分布均匀性和路由表大小。

**应用场景**：分库分表扩容迁移（最小化数据搬运）、分布式缓存（Redis Cluster/Memcached）、Nginx 一致性哈希负载均衡。

## 六、3PC 与拜占庭问题

### 3PC：为什么实际不用

**3PC 相对于 2PC 的改进**：增加 CanCommit 阶段，引入超时机制，参与者不会永久阻塞：

```
2PC：PreCommit → DoCommit（两阶段，阻塞等待）
3PC：CanCommit → PreCommit → DoCommit（三阶段，超时自动提交）
```

**3PC 的致命缺陷（网络分区下一致性更差）**：
> DoCommit 阶段协调者发送 Abort，但由于网络问题参与者没收到，参与者等待超时后**自行 commit**。而其他收到 Abort 的参与者执行了回滚 → **数据不一致**！

**工程选择**：业界直接用 TCC/Saga/本地消息表替代 2PC/3PC，彻底规避阻塞和不一致问题。

### 拜占庭将军问题（BFT）

**问题本质**：分布式系统中存在**恶意节点**（不仅会失联，还会发送虚假消息）时如何达成共识。

普通共识算法（Raft/Paxos）假设节点只会宕机，不会主动撒谎；BFT 算法假设节点可能发送矛盾/虚假消息。

**数学结论**：需要 n ≥ 3f+1 个总节点，才能容忍 f 个叛徒。消息复杂度 O(n²)，节点增多时开销急剧增大。

| 场景 | 协议 | 原因 |
|------|------|------|
| 区块链（比特币）| PoW（工作量证明）| 通过算力成本使攻击不经济 |
| 联盟链（Hyperledger Fabric）| PBFT（实用拜占庭容错）| 已知参与方，节点数可控 |
| 传统分布式（etcd/Kafka/ZK）| Raft/Paxos | 假设无恶意节点，效率高 |

**为什么互联网分布式系统不用 BFT**：内部集群节点可信（不会恶意撒谎），用 Raft 即可；BFT 仅用于需要跨组织信任边界的场景。

## 七、分布式数据库与 ID 生成

### NewSQL 选型

**背景**：MySQL 分库分表的天花板——维护成本高、跨分片查询难、无法透明扩容。

| | MySQL + 分库分表 | TiDB | OceanBase |
|---|---|---|---|
| **来源** | 自研中间件（ShardingJDBC 等）| PingCAP 开源 | 蚂蚁金服自研 |
| **底层存储** | B+树（InnoDB）| LSM 树（TiKV + RocksDB）| B+ 树变种 |
| **共识协议** | 无（应用层保证）| Raft（Region 级）| Paxos |
| **OLAP 支持** | 需要独立数仓 | TiFlash 列存（HTAP）| 有限支持 |
| **扩容方式** | 手动 Resharding + 数据迁移 | 自动 Region 调度 | Zone 扩容 |
| **适用量级** | < 亿级 | 亿级+ | 金融级（如支付宝）|
| **事务强度** | ACID（单分片内）| Snapshot Isolation | 强 ACID |

**选型决策**：亿级以下 MySQL + 分库分表仍是主流；亿级以上写入密集选 TiDB；金融核心链路选 OceanBase。TiDB 最小 3 节点，小业务无法用。

### Leaf 分布式 ID 生成器（美团开源）

**两种模式**：

**① 号段（Segment）模式**：
```
DB 表：biz_tag | max_id | step | update_time
应用：一次取一个号段（如 step=1000），缓存使用，用完再取
双 Buffer：当前号段消耗到 10% 时，异步预取下一号段，避免取号时的停顿
```

**② Snowflake 模式（ZK 辅助）**：
- 节点从 ZK 获取 workerId，启动时校验与历史时间戳，有时钟回拨则拒绝启动
- 每个 Leaf 节点区间内自增，减少 ZK 访问频率

**ID 生成方案全景**：

| 方案 | 全局唯一 | 趋势递增 | 无单点 | 适用场景 |
|------|---------|---------|--------|---------|
| 数据库自增 | ✅ | ✅ | ❌ | 小规模 |
| 雪花算法 | ✅（正常时钟）| ✅ | ✅ | 中大规模 |
| Leaf-Segment | ✅ | ✅（单机内）| ❌（DB 单点）| 低成本可靠 |
| Leaf-Snowflake | ✅ | ✅ | ✅ | 高吞吐分布式 |
| UUIDv4 | ✅ | ❌ | ✅ | 不需要排序的场景 |

**雪花算法时钟回拨 4 种解法**：
1. 直接抛异常（强烈推荐），让运维修复时钟
2. 等待（回拨 < 5ms 时 sleep 等待）
3. 启动时校验，拒绝启动（Leaf-Snowflake 方案）
4. 使用逻辑时钟替代物理时钟（百度 UidGenerator）

## 八、关键权衡

| 权衡点 | 结论 |
|--------|------|
| CP vs AP | 金融/锁/配置选 CP；电商展示/社交 feed 选 AP+最终一致 |
| ZooKeeper 的 CP | 保证顺序一致性（非线性一致性）；`sync()` 可强制同步 |
| BASE 软状态陷阱 | PROCESSING 状态必须有超时兜底，防止永久卡在中间状态 |
| 虚拟节点数 | 越多分布越均匀，但路由表越大；通常 100-200 个/物理节点 |
| 3PC 实际不用 | 网络分区时比 2PC 一致性更差，工程用 TCC/BASE 替代 |
| 分布式数据库 vs 分库分表 | 亿级以下优先分库分表（成本低）；亿级以上 TiDB |
| 熔断 vs 降级 | 熔断由**下游故障**触发（防御性），降级由**系统负载**触发（主动性）；两者均导致部分功能暂不可用 |

## 九、与其他概念的关系

- **[[机制-Zookeeper]]**：ZAB 是 ZooKeeper 实现 CP 的底层协议，通过 Leader 广播 + 半数确认保证顺序一致性
- **[[机制-SpringCloud]]**：Eureka（AP）vs ZooKeeper（CP）的注册中心选型正是 CAP 权衡的直接体现；Nacos 通过 AP+CP 双模式兼顾两种场景
- **[[概念-分布式事务]]**：2PC 追求强一致性（CP），TCC/本地消息表/事务消息是 BASE 最终一致性的工程实现
- **[[概念-MySQL]]**：一致性哈希是分库分表扩容迁移的核心算法；雪花算法是分布式 ID 方案

## 十、应用边界

**选 CP（强一致）的场景**：
- 分布式锁（必须同一时刻只有一个持有者）
- 金融交易、账户余额
- 配置中心、服务注册（ZooKeeper/Nacos CP 模式）

**选 AP（最终一致）的场景**：
- 电商库存展示（允许短暂超卖后补偿）
- 社交 feed 流（允许短暂不一致）
- DNS 解析（TTL 缓存期间允许旧数据）

**一致性哈希适用**：节点会动态增删，且希望最小化数据迁移的分布式存储/缓存场景。节点数稳定时，普通取模更简单高效。

**Raft/共识算法适用**：需要强一致的元数据存储（配置中心/分布式锁/K8s etcd）；不适合超高频写入（协调开销大）。
