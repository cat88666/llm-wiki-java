# 分布式系统理论

**一句话定义**：CAP 理论揭示了分布式系统在一致性/可用性/分区容错性三者中最多只能同时保证两项；BASE 理论是对 CAP 的工程化延伸，以最终一致性换取高可用；一致性哈希则解决了节点动态变化时的数据迁移最小化问题。

---

## 第一性原理

分布式系统的本质困境来自**网络**：节点间通信可能延迟或中断（分区），而业务要求系统随时可用且数据正确。CAP 证明了这三件事不可能同时完美实现，而 BASE 则给出了工程师的务实答案——**不追求强一致，转而以软状态 + 重试推进到最终一致**。

---

## 核心机制

### CAP 理论

| 属性 | 定义 | 实现代价 |
|------|------|---------|
| **C（Consistency）** | 每次读都能获取到最新写入的数据，或者返回错误 | 网络分区期间必须阻塞写，牺牲可用性 |
| **A（Availability）** | 每个请求都会收到非错误响应（但可能是旧数据）| 可能读到过期数据，牺牲一致性 |
| **P（Partition Tolerance）** | 网络分区（节点间消息丢失/延迟）时系统仍能运行 | 分布式系统必须支持，不可放弃 |

**关键推论**：分布式系统中网络分区是必然的自然事实，P 不可放弃。因此实际选择只有两种：

- **CP（牺牲可用性）**：网络分区时，宁可返回错误或阻塞，不返回不一致数据。代表：ZooKeeper（顺序一致性）、HBase
- **AP（牺牲强一致性）**：网络分区时，返回旧数据保证可用，事后再同步。代表：Eureka、Cassandra

> 注意：ZooKeeper 保证的是**顺序一致性**（所有节点写入顺序一致，半数节点写成功才确认），而非线性一致性（实时最新值）。

**一致性模型层次**：
```
强一致性 ─┬─ 线性一致性（最强，实时最新值）
          └─ 顺序一致性（全局操作顺序一致，不要求实时）
弱一致性 ─┬─ 因果一致性
          └─ 会话一致性
最终一致性（最弱，经过一段时间后收敛）
```

---

### BASE 理论

BASE 是对 ACID（传统数据库强一致）的反命题，是 CAP 中 AP 的工程化落地：

| 属性 | 含义 | 实现手段 |
|------|------|---------|
| **Basically Available**（基本可用）| 出现故障时允许局部降级，保证核心功能可用 | 返回降级页/限流/熔断，而不是直接 500 |
| **Soft State**（软状态）| 允许数据存在中间过渡状态（INIT→PROCESSING→SUCCESS）| 状态机设计，记录操作流水 |
| **Eventual Consistency**（最终一致性）| 经过有限时间后，所有副本收敛到相同状态 | 消息队列 + 重试 + 对账补偿 |

**核心口诀**：**中间状态（软状态）+ 重试（最终一致性）+ 降级（基本可用）**

**与 ACID 的对比**：
```
ACID：原子性 + 强一致性 + 隔离性 + 持久性 → 传统单机数据库
BASE：基本可用 + 软状态 + 最终一致性      → 大规模分布式系统
```

### Paxos 与 Raft —— 共识算法

**解决问题**：多个节点如何就同一个值达成一致（即使部分节点故障）。CAP 理论证明了 CP 是可能的，共识算法是实现 CP 的工程路径。

**Paxos**：分布式一致性的理论基础（Leslie Lamport 1989）。证明了在异步网络中共识是可解的，但实现极复杂，工程上难以正确实现。

**Raft**（工程实践主流，Diego Ongaro 2014）：为"易于理解"而设计，将一致性问题分解为三个相对独立的子问题：

**① Leader 选举**：
- 节点三种角色：**Leader**（处理所有读写）/ **Follower**（被动）/ **Candidate**（选举中）
- Follower 超过随机超时（150-300ms）未收到 Leader 心跳 → 升为 Candidate，发起选举
- 获得多数票（`N/2 + 1`）→ 成为新 Leader，立即发送心跳抑制其他选举
- **任期（Term）**：单调递增的逻辑时钟，每次选举产生新任期，节点收到更高 Term 的消息自动降为 Follower（防脑裂）

**② 日志复制**：
```
Client Write → Leader 追加日志（uncommitted）
             → 并行复制给所有 Follower
             → 多数 Follower 确认 → Leader commit（已提交）
             → 通知 Follower commit → 响应 Client
```
- 日志一旦多数节点确认即 committed，后续 Leader 选举必须保证新 Leader 拥有所有 committed 日志

**③ 安全性**：只有拥有最新 committed 日志的节点才能赢得选举（投票前检查 log index + term），保证已提交的日志永不丢失。

**Raft vs Paxos**：Raft 功能上等价于 Multi-Paxos，但设计更模块化，限制更多（如强 Leader），工程实现更容易验证正确性。

**工程应用**：etcd（K8s 状态存储）、TiKV（TiDB 底层）、CockroachDB、Consul 均基于 Raft。

### 一致性哈希

**解决问题**：普通取模哈希（`hash(key) % n`）在节点数 n 变化时，几乎所有数据都要重新映射（代价 O(n)）。一致性哈希让节点增删时只有 O(1/n) 的数据需要迁移。

**核心机制**：

```
1. 构建哈希环（0 ~ 2^32-1 首尾相连）
2. 节点映射：hash(node_name) % 2^32 → 节点落在环上某位置
3. 数据映射：hash(key) % 2^32 → 数据落在环上某位置
4. 顺时针查找：数据沿顺时针方向找到的第一个节点，即为存储节点
```

**增删节点的影响**：
- 新增节点：只有该节点到其前驱节点之间的数据需要迁移，其余数据不变
- 删除节点：只有该节点上的数据需要迁移到其顺时针后继节点

**虚拟节点（解决 Hash 倾斜）**：当节点数少时，节点可能分布不均（热点）。将一个物理节点映射为多个虚拟节点（如 `node1#1, node1#2, ..., node1#150`），使数据分布更均匀，同时减少节点变更时的数据迁移量。

```
物理节点  →  虚拟节点（分散在环上）  ←  数据（映射到最近虚拟节点，再路由到物理节点）
```

**应用场景**：
- 分库分表：分片键哈希后找到对应分表，扩容时只迁移部分数据
- 分布式缓存（如 Memcached/Redis Cluster）：键路由到缓存节点
- Nginx upstream 的一致性哈希负载均衡

### 三阶段提交（3PC）—— 为什么实际不用

**3PC 相对于 2PC 的改进**：在 2PC 基础上增加 CanCommit 阶段，引入**超时机制**，参与者不会永久阻塞。

```
2PC：PreCommit → DoCommit（两阶段）
3PC：CanCommit → PreCommit → DoCommit（三阶段）
```

3PC 核心改进：
1. **减少阻塞**：参与者等待超时后可自行 commit（不会永久阻塞）
2. **新协调者恢复**：新协调者可从参与者状态推断出正确操作（PreCommit 状态说明可以提交）

**3PC 的致命缺陷（网络分区下一致性更差）**：
> DoCommit 阶段协调者发送 Abort，但由于网络问题参与者没收到，参与者等待超时后**自行 commit**。而其他收到 Abort 的参与者执行了回滚 → **数据不一致**！

**为什么实际不用 3PC**：
- 正常情况：3PC 比 2PC 多一次网络往返，性能更差
- 极端情况：网络分区时 3PC 反而可能产生数据不一致（比 2PC 更糟）
- 工程选择：业界直接用 TCC/Saga/本地消息表替代 2PC/3PC，彻底规避阻塞问题

### 拜占庭将军问题（BFT）

**问题本质**：分布式系统中存在**恶意节点**（不仅会失联，还会发送虚假消息）时如何达成共识。

- 普通共识算法（Raft/Paxos）假设：节点只会宕机，不会主动撒谎
- 拜占庭容错算法（BFT）假设：节点可能发送矛盾/虚假消息

**解决方案核心思想**：超过 2/3 多数投票（而非普通共识的 1/2+1）。

> 数学证明：如果叛徒数量 f，则需要总节点数 n ≥ 3f+1，才能保证忠诚将军达成正确共识。

**BFT 的工程代价**：消息复杂度 O(n²)，性能远低于 Raft；节点增多时开销急剧增大。

**实际应用场景**：
| 场景 | 协议 |
|------|------|
| 区块链（比特币）| PoW（工作量证明）— 通过算力成本使攻击不经济 |
| 联盟链（Hyperledger Fabric）| PBFT（实用拜占庭容错）— 已知参与方，节点数可控 |
| 传统分布式（etcd/Kafka/ZK）| Raft/Paxos — 假设无恶意节点，效率高 |

**为什么互联网分布式系统不用 BFT**：内部集群节点可信（不会恶意撒谎），宕机/网络分区用 Raft 即可；BFT 仅用于需要跨组织信任边界的场景（区块链、联盟链）。

### 分布式数据库选型（NewSQL）

**背景**：MySQL 分库分表的天花板——维护成本高、跨分片查询难、无法透明扩容。

| | MySQL + 分库分表 | TiDB | OceanBase |
|---|---|---|---|
| **来源** | 自研中间件（ShardingJDBC等）| PingCAP 开源 | 蚂蚁金服自研 |
| **底层存储** | B+树（InnoDB）| LSM 树（TiKV + RocksDB）| B+ 树变种 |
| **共识协议** | 无（应用层保证）| Raft（Region级）| Paxos |
| **OLAP 支持** | 需要独立数仓 | TiFlash 列存（HTAP）| 有限支持 |
| **扩容方式** | 手动 Resharding + 数据迁移 | 自动 Region 调度 | Zone 扩容 |
| **适用量级** | < 亿级（分库分表可抗）| 亿级+ | 金融级（如支付宝） |
| **事务强度** | ACID（单分片内）| Snapshot Isolation | 强 ACID |

**真实 Trade-off**：
- **亿级以下**：MySQL + 分库分表仍是主流（团队熟悉、成本低）
- **亿级以上写入密集**：TiDB（LSM 树写入快，TiFlash 支持实时分析）
- **金融核心链路**：OceanBase（蚂蚁双十一主数据库，一致性保证更强）
- **禁用场景**：TiDB 不支持单机部署（最小 3 节点），小业务无法用

**分布式数据库的本质代价（CAP 视角）**：
- 跨节点事务需要 2PC 协调（牺牲延迟）
- 无法保证 ID 连续自增（只能保证单调递增）
- 复杂 SQL 中的跨分区聚合性能差（需要 Scatter-Gather）

### Leaf 分布式 ID 生成器（美团开源）

**解决问题**：雪花算法依赖机器时钟，存在时钟回拨问题；数据库单点自增 ID 吞吐有限。

**两种模式**：

**① 号段（Segment）模式**：
```
DB 表：biz_tag | max_id | step | update_time
应用：一次取一个号段（如 1000 个），缓存使用，用完再取
双 Buffer：当前号段消耗到 10% 时，异步预取下一号段，避免取号时的停顿
```
- 优点：减少 DB 压力（原来每次请求 DB 变为每 1000 次请求 1 次）；DB 宕机后号段缓存还能用一段时间
- 缺点：ID 不严格随机（同实例内单调，但不同实例之间会有间隙）；DB 宕机时间过长则服务失败

**② Snowflake 模式（ZK 辅助）**：
```
时钟回拨解决：启动时从 ZK 上报时间戳，校验与历史时间戳 + 集群平均时间戳，有异常则拒绝启动
工作机制：每个 Leaf 节点从 ZK 获取区间（避免每次请求 ZK），区间内自增
```

**Leaf vs 原生雪花算法对比**：

| | 雪花算法（原版）| Leaf-Snowflake |
|---|---|---|
| 机器 ID 配置 | 硬编码 | ZK 动态分配 |
| 时钟回拨处理 | 无 | 启动校验 + 拒绝启动 |
| ZK 依赖 | 无 | 依赖 ZK（启动时注册）|
| ID 趋势 | 趋势递增 | 趋势递增 |

**ID 生成方案全景**：

| 方案 | 全局唯一 | 趋势递增 | 无单点 | 适用场景 |
|------|---------|---------|--------|---------|
| 数据库自增 | ✅ | ✅ | ❌ | 小规模 |
| 雪花算法 | ✅（正常时钟）| ✅ | ✅ | 中大规模 |
| Leaf-Segment | ✅ | ✅（单机内）| ❌（DB 单点）| 低成本可靠 |
| Leaf-Snowflake | ✅ | ✅ | ✅ | 高吞吐分布式 |
| UUIDv4 | ✅ | ❌ | ✅ | 不需要排序的场景 |

## 关键权衡

1. **CP vs AP 的选择**：钱财/强一致性场景（支付、分布式锁）选 CP；高并发/可用性优先场景（购票展示、社交 feed）选 AP+最终一致性
2. **ZooKeeper 是 CP 但不是强一致**：ZK 保证顺序一致性（写入顺序），不保证线性一致性（立即读到最新值）；ZK 的 `sync()` 命令可强制同步，但有性能代价
3. **BASE 软状态的陷阱**：软状态需要精心设计状态机；PROCESSING 状态必须有超时兜底机制，防止永久卡在中间状态
4. **一致性哈希的虚拟节点数**：虚拟节点越多，数据分布越均匀，但内存消耗和路由表也越大；通常每个物理节点配置 100-200 个虚拟节点
5. **3PC 实际不用**：设计上比 2PC 更好，但网络分区时一致性更差，工程上直接用 TCC/BASE 替代
6. **分布式数据库 vs 分库分表**：亿级以下优先分库分表（成本低），亿级以上且需要实时分析时考虑 TiDB

## 与其他概念的关系

- **[[机制-ZAB协议与Zookeeper]]**：ZAB 是 ZooKeeper 实现 CP 的底层协议，通过 Leader 广播 + 半数确认保证顺序一致性
- **[[机制-微服务与SpringCloud]]**：Eureka（AP）vs ZooKeeper（CP）的注册中心选型正是 CAP 权衡的直接体现
- **[[设计-分布式事务]]**：2PC 追求强一致性（CP），TCC/本地消息表/事务消息是 BASE 最终一致性的工程实现
- **[[概念-分库分表]]**：一致性哈希是分库分表扩容迁移的核心算法；雪花算法是分布式 ID 方案

## 应用边界

**选 CP（强一致）的场景**：
- 分布式锁（必须同一时刻只有一个持有者）
- 金融交易、账户余额
- 配置中心、服务注册（ZooKeeper）

**选 AP（最终一致）的场景**：
- 电商库存展示（允许短暂超卖后补偿）
- 社交 feed 流（允许短暂不一致）
- DNS 解析（TTL 缓存期间允许旧数据）

**一致性哈希适用**：节点会动态增删，且希望最小化数据迁移的分布式存储/缓存场景。节点数稳定时，普通取模更简单高效。
