# 限流与熔断

**一句话定义**：限流是在流量入口控制请求速率以保护下游资源，熔断是在调用链路检测到故障后自动隔断以防止级联崩溃——两者都是在系统过载场景下的主动降级手段。

---

## 第一性原理

任何系统的处理能力都有上限（CPU、内存、线程池、DB 连接数）。当请求量超过上限时，系统面临两种结局：

1. **排队等待**：延迟急剧升高，最终全部请求超时
2. **主动拒绝**：快速失败部分请求，保证其余请求正常响应

限流选择第 2 条路——**宁愿让少数请求快速失败，也不让所有请求慢速失败**。

熔断解决的是依赖调用问题：当下游服务故障时，上游服务的调用线程被阻塞，线程池耗尽导致级联崩溃（雪崩效应）。熔断器主动断开调用，让上游快速返回，等待下游恢复。

## 核心机制

### 限流算法对比

#### 漏桶算法

```
请求 → [漏桶（固定容量）] → 固定速率流出 → 下游
         ↑
     桶满则丢弃
```

- **特点**：输出速率严格固定，平滑流量；不支持突发流量（即使桶空着也只能固定速率出水）
- **适合**：对下游有严格速率要求的场景（如调用付费 API 有 QPS 限制）

#### 令牌桶算法

```
定时器每秒放 N 个令牌 → [令牌桶（最多 burst 个）]
请求到达 → 取 1 个令牌 → 处理
         → 无令牌 → 等待或拒绝
```

- **特点**：支持突发流量（桶中积累的令牌允许短时间高速处理）；平均速率受令牌生成速率限制
- **实现**：Guava `RateLimiter`（令牌桶实现）
- **适合**：允许短暂突发但限制平均速率的场景（Web API、消息消费）

```java
RateLimiter limiter = RateLimiter.create(100.0); // 100 QPS
if (limiter.tryAcquire()) {
    // 处理请求
} else {
    // 限流拒绝
}
```

#### 漏桶 vs 令牌桶

| 维度 | 漏桶 | 令牌桶 |
|------|------|--------|
| 突发流量 | 不支持（严格固定速率）| 支持（消耗积累的令牌）|
| 输出稳定性 | 极稳定 | 有抖动（突发时高速出）|
| 适用场景 | 保护下游有严格速率限制 | 应对正常业务峰值 |
| Java 实现 | 自定义 | Guava RateLimiter |

#### 滑动窗口限流

```
固定窗口（易出现边界突刺）：
|----1秒----|----1秒----|
        100  100        每秒100个，但窗口边界处可能0.1秒内200个

滑动窗口（每次请求检查过去1秒）：
|--过去1s的计数--|当前请求时刻
计数 = 时间切片内请求数之和，超过阈值则拒绝
```

- **实现**：把 1s 窗口切成 10 个 100ms 的切片，维护每个切片的计数，滑动时删除过期切片
- **优势**：消除固定窗口边界的突刺问题；Redis ZSet 实现（score=时间戳，count=成员数）
- **代价**：内存开销比固定窗口大（需存储每个切片的计数）

#### 自适应限流

不依赖预设阈值，而是根据系统实时指标动态调整限流阈值：

**Sentinel 自适应限流触发条件**（满足任一）：
- CPU 使用率 > 80%
- 系统 Load（1分钟平均）> CPU 核数 × 2.5
- 平均响应时间 > 阈值
- 并发线程数 > 阈值
- QPS > 阈值

优势：无需提前压测确定阈值；劣势：触发可能有滞后，初期配置较复杂。

### 单机限流 vs 集群限流

**两层都需要**：

```
客户端请求 → [集群限流：全局 QPS 配额] → 负载均衡 → 节点1 [单机限流：本机保护]
                                                    → 节点2 [单机限流：本机保护]
                                                    → 节点3 [单机限流：本机保护]
```

| 维度 | 单机限流 | 集群限流 |
|------|---------|---------|
| 保护对象 | 单个节点不被压垮 | 全局资源（DB、下游服务）|
| 实现 | Guava/Sentinel 本地计数 | Redis 计数器 + Lua 原子操作 |
| 精度 | 高（本地计算快）| 中（网络延迟）|
| 缺点 | 节点间不均衡时全局配额失控 | 引入 Redis 额外依赖 |

### 熔断器三态

```
正常状态
   ↓ 错误率/超时率超过阈值
[关闭(Closed)] ─────────────────────→ [打开(Open)]
   ↑                                      │
   │  半开→成功请求占比达到阈值              │ 超时（如5秒）后
   │                                      ↓
   └────────────────────────── [半开(Half-Open)]
                                   放入少量探测请求
```

| 状态 | 行为 | 转换条件 |
|------|------|---------|
| 关闭（Closed）| 正常放行请求 | 错误率 > 阈值 → 开启 |
| 打开（Open）| 快速失败，不调用下游 | 超时后 → 半开 |
| 半开（Half-Open）| 放入少量探测请求 | 成功率高 → 关闭；仍失败 → 打开 |

**工具选型**：
- Hystrix（Netflix，已停止维护）：线程隔离池，功能完善
- Sentinel（Alibaba，活跃）：基于滑动窗口，支持多种规则（流控/降级/热点/系统）
- Resilience4j：轻量级，适合非 Spring Cloud 项目

### 预热（Warmup）

**问题**：服务刚启动时，JVM 未 JIT 编译、缓存未加载，此时突然大流量会导致处理慢 → 超时 → 缓存击穿/雪崩。

**预热策略**：
1. **启动预加载**：`@PostConstruct` 或 `ApplicationRunner` 启动时加载热点数据到缓存
2. **定时任务预热**：大促前定时刷新缓存，确保活动数据提前到位
3. **手动触发**：运维脚本在流量切入前预热关键接口
4. **限流预热**：Guava RateLimiter 的 `warmupPeriod` 参数，允许初始低速率逐步升至目标速率

## 关键权衡

1. **令牌桶 vs 漏桶选择**：大多数业务选令牌桶（允许正常业务峰值），只有对下游有严格速率合约时才用漏桶
2. **限流阈值设定**：太低则正常流量被拒绝（用户体验差）；太高则无法保护系统——需要压测确定系统上限再设为 70%~80%
3. **熔断后降级策略**：熔断触发后必须有降级逻辑（返回缓存数据/默认值/友好提示），否则熔断等于报错
4. **集群限流一致性代价**：Redis 网络延迟约 0.1~1ms，高频限流检查（每个请求都查 Redis）会增加整体延迟；可以批量申请令牌（一次申请 10 个配额，用完再申请）

## 与其他概念的关系

- **[[机制-线程池]]**：线程池本身是限流手段之一（队列满触发拒绝策略），但粒度粗；专用限流器（Sentinel）提供更精细的算法控制
- **[[机制-微服务与SpringCloud]]**：Sentinel 是 Spring Cloud Alibaba 的流量防护组件；熔断是 Gateway/Feign 调用链路的标准配置；熔断器三态模型来自 Hystrix，被 Sentinel 继承
- **[[概念-高可用设计]]**：限流+熔断+降级是高可用三板斧；高可用设计的"故障隔离"依赖熔断，"容量保障"依赖限流
- **[[机制-Redis分布式锁]]**：集群限流中 Redis Lua 脚本实现原子计数，与分布式锁使用相同的 Redis 原子操作能力

## 应用边界

**限流适用**：
- API 网关层（保护所有下游服务）
- 秒杀/抢购接口（防止超卖和系统被打垮）
- 第三方 API 调用（避免触发对方速率限制）
- 消息消费（控制消费速率，保护下游 DB）

**熔断适用**：
- 微服务间 RPC 调用
- 调用不稳定的第三方服务（支付、短信）
- 数据库连接池满时的快速失败

**不适用**：
- 内部计算密集型函数（限流意义不大，瓶颈在 CPU）
- 实时性要求极高的场景（集群限流的 Redis 网络延迟不可接受时，退化为单机限流）
