# 分布式事务

> 分布式事务的本质矛盾：**跨进程调用不能共享同一个数据库事务**（事务绑定数据库连接，连接不跨进程）。解决策略分两大类：强一致（2PC/XA，牺牲可用性）和最终一致（TCC/消息事务/本地消息表，接受短暂中间态）。

## 一、第一性原理

单机事务中，所有操作共享同一数据库连接，事务的 ACID 由数据库引擎保证。分布式系统中，不同服务有独立的数据库连接，无法共享一个 Connection——这是分布式事务问题的根本来源，不是设计缺陷，是网络边界的必然结果。

解决路径只有两条：
1. **强迫所有操作在同一个两阶段提交中完成**（2PC/XA）——正确但慢，阻塞时间长
2. **允许中间状态存在，靠补偿推进到最终一致**（TCC/消息事务/Saga）——快但复杂，需要业务侧配合

## 二、方案全景

```
强一致性（CP）
├── 2PC (Two-Phase Commit) — 协调者+参与者，同步阻塞
├── 3PC (Three-Phase Commit) — 解决 2PC 阻塞，但仍有不一致风险
└── XA — 数据库原生支持的 2PC（ACID 强一致，性能差，少用）

最终一致性（AP + BASE）
├── TCC (Try-Confirm-Cancel) — 业务层三接口，性能好，侵入性高
├── 本地消息表 — 业务DB + 消息表同一事务，定时任务重试
├── 事务消息（RocketMQ）— MQ 半消息机制，框架层保证
└── Saga — 长事务，正向链 + 逆向补偿链，无隔离性
```

## 三、2PC vs 3PC

### 2PC（两阶段提交）

```
阶段1（Prepare）：协调者 → 所有参与者 prepare → 参与者锁资源并返回 vote
阶段2（Commit/Abort）：全部 vote=yes → 协调者发 commit；否则发 rollback
```

**核心缺陷**：
- **同步阻塞**：参与者等 Commit 消息期间持有锁，其他事务被阻塞
- **协调者单点**：协调者宕机 → 参与者永久持锁阻塞
- **数据不一致**：协调者发 Commit 后宕机，部分参与者收到 Commit 部分未收到 → 数据不一致

### 3PC 改进与致命缺陷

**改进**：增加 CanCommit 阶段（先询问是否能提交），引入超时机制，参与者超时后自动 commit（降低阻塞）。

**致命缺陷**：DoCommit 阶段协调者发送 Abort，参与者因网络问题未收到，等待超时后**自行 commit**，而其他收到 Abort 的参与者执行了回滚 → **数据不一致**！比 2PC 更糟。

**结论**：业界不用 3PC，直接用 TCC/BASE 替代。

## 四、TCC 模式

```
Try（预留/冻结资源）
    ↓ 全部成功
Confirm（确认/提交）
    ↓ 任意失败
Cancel（回滚/解冻）
```

**电商下单 TCC 示例**：
- Try：库存冻结 `frozen_stock += 1, available_stock -= 1`；订单创建为"待确认"状态
- Confirm：冻结库存扣减 `frozen_stock -= 1`；订单状态改为"确认"
- Cancel：解冻库存 `frozen_stock -= 1, available_stock += 1`；订单废弃

**为什么是最终一致性**：Try → Confirm 之间存在中间状态（库存冻结但订单未确认），其他事务可以看到这个中间状态。

### TCC 三大问题

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| **空回滚** | Cancel 先于 Try 到达（网络乱序）| 事务记录表：Cancel 时查询 Try 是否执行，未执行则直接返回成功 |
| **悬挂** | Try 在 Cancel 之后到达 | 事务记录表：Cancel 已执行时，拒绝后续到达的 Try 操作 |
| **幂等** | 网络重试导致 Confirm/Cancel 多次执行 | 事务记录表中标记状态，幂等校验 |

**核心解法**：引入一张**分布式事务记录表**（`tx_id, state, ...`），记录每个 xid 的执行状态，三个问题统一解决。

## 五、本地消息表 vs 事务消息

### 本地消息表

```
1. DB 操作 + 写消息表（同一本地事务）
2. 定时任务扫描未投递消息 → 发送到 MQ
3. Consumer 消费成功 → 回调更新消息表状态
```

- **优点**：无需引入 MQ 事务特性，实现简单；DB 事务保证消息表和业务数据一致
- **缺点**：定时任务有延迟（通常 1~5 分钟）；消息表与业务库耦合；定时任务维护成本

**下游失败时上游如何处理**：原则是**不回滚上游已提交业务，只保证下游最终一致**。流程：消息表 PENDING → 定时任务重试 → 超阈值进入 FAILED + 报警 → 人工补偿。

### 事务消息（RocketMQ 半消息机制）

```
1. Producer 发送 Half Message（消息对 Consumer 不可见，存入 Half Topic）
2. Producer 执行本地事务（转账/扣款/下单）
3a. 本地事务成功 → 发送 Commit → 消息对 Consumer 可见
3b. 本地事务失败 → 发送 Rollback → 消息被删除
3c. 超时 → RocketMQ 回查 Producer 本地事务状态（checkLocalTransaction）
```

- **优点**：MQ 框架保证消息和事务的一致性，无需额外定时任务；实时性好
- **缺点**：依赖 RocketMQ（Kafka 不支持事务消息）；Consumer 必须实现幂等

### 两者对比

| 维度 | 本地消息表 | RocketMQ 事务消息 |
|------|-----------|-----------------|
| 实时性 | 分钟级延迟 | 秒级 |
| 开发复杂度 | 低（增加消息表 + 定时任务）| 中（需实现回查接口）|
| MQ 依赖 | 弱（任意 MQ 均可）| 强（仅 RocketMQ）|
| 适用场景 | 已有定时任务体系，消息量小 | 高实时性，已用 RocketMQ |

## 六、Seata 主要模式

| 模式 | 原理 | 适用场景 |
|------|------|---------|
| **AT（默认）** | 自动解析 SQL，生成 undo log；阶段1先提交 + undo log，阶段2 Commit 删 undo log 或 Rollback 用 undo log 回滚 | 绝大多数 CRUD 场景，侵入性低 |
| **TCC** | 用户自定义 Try/Confirm/Cancel 三个接口，Seata 协调调用 | 性能要求高，自己控制补偿逻辑 |
| **Saga** | 长事务，正向执行链 + 补偿链；只保证最终一致 | 无法改造为 TCC 的遗留系统，长流程 |
| **XA** | 利用数据库原生 XA 协议（严格强一致）| 强一致要求高，接受性能损失 |

**Seata AT 与 2PC 区别**：AT 阶段1就提交（不持锁等待阶段2），通过 undo log 做补偿；2PC 阶段1不提交，持锁等待协调者指令，锁粒度大。这是 AT 性能更好的核心原因。

## 七、选型决策树

```
是否能接受最终一致？
├── 不能（金融核心账务）→ XA / Seata XA（性能差但强一致）
└── 可以接受
    ├── 已用 Seata，各服务是关系型 DB
    │   → Seata AT（无侵入，加注解即可）
    ├── 需要极致性能，可接受侵入性开发
    │   → TCC（Seata TCC 模式 or 自实现）
    ├── 已用 RocketMQ，需要秒级一致性
    │   → 事务消息（Half Message）
    ├── 任意 MQ，对延迟要求不高
    │   → 本地消息表 + 定时任务
    └── 长事务，无法改造为 TCC（外部接口、遗留系统）
        → Saga
```

## 八、关键权衡

| 权衡点 | 结论 |
|--------|------|
| 强一致 vs 最终一致 | 金融核心账务用强一致（接受性能差）；电商交易流程用最终一致（用户体验优先）|
| TCC vs 本地消息表 | TCC 性能好（不依赖 MQ 投递延迟）但开发量大（三个接口）；消息表实现简单但有消息延迟 |
| Seata AT vs 自实现 TCC | AT 侵入性低、接入快，但有 undo log 额外存储开销且不支持所有 SQL；自实现 TCC 灵活但每个方法都需实现三个接口 |
| 2PC 阻塞根因 | 参与者在等待协调者 Commit/Abort 期间持锁；TCC 不持锁（Try 只冻结，不阻塞其他事务），这是 TCC 性能更好的核心原因 |

## 九、与其他概念的关系

- **[[概念-MySQL]]**：undo log 是 Seata AT 模式回滚的核心工具；本地事务边界来自 [[机制-Spring]] AOP
- **[[机制-RabbitMQ]]**：本地消息表和事务消息都依赖 MQ 的 ACK 机制保证消息至少投递一次
- **[[概念-Redis]]**：TCC 的分布式锁（幂等控制）可基于 Redis SETNX 实现
- **[[机制-Seata]]**：Seata 是分布式事务的框架化实现，详细机制见 Seata 页

## 十、面试要点

1. **先分类**：强一致（2PC/XA）vs 最终一致（TCC/消息/Saga），体现体系化认知
2. **TCC 重点讲空回滚和悬挂**：核心是事务状态记录表，面试官爱考
3. **对比 Seata AT 和 2PC**：AT 阶段1就提交（不持锁），2PC 阶段1不提交（持锁等待），这是核心区别
4. **给出选型建议**：根据一致性要求、技术栈、开发成本给出合理推荐（非答"看情况"）
