## 一、三种锁的选型

| | 乐观锁（CAS/version）| 悲观锁（SELECT FOR UPDATE）| 分布式锁（Redis/ZK）|
|--|---|---|---|
| **冲突假设** | 冲突少（乐观）| 冲突多（悲观）| 跨服务互斥 |
| **实现代价** | 低（版本号字段）| 中（DB行锁）| 高（Redis/ZK依赖）|
| **适用范围** | 单服务单DB | 单服务单DB，高争用 | 跨服务/跨进程 |
| **失败处理** | 重试（ABA问题）| 等待/超时 | 等待/超时/续期 |
| **性能** | 高（无阻塞）| 低（阻塞等待）| 中（网络开销）|

**选型规则**：
```
读多写少，冲突概率低？        → 乐观锁（version字段 + UPDATE WHERE version=old）
写多写多，单DB，高争用？      → 悲观锁（SELECT FOR UPDATE）
跨服务 / 跨进程竞争资源？     → 分布式锁（Redis SETNX + Lua）
同一服务高并发但要降低锁竞争？→ 分布式锁粒度细化 or 库存拆分
```

## 二、高并发库存扣减

**场景**：大促期间1万个库存，10万QPS 并发扣减。

### 方案1：数据库乐观锁

```sql
-- version 字段做乐观锁
UPDATE stock SET count = count-1, version = version+1
WHERE id = #{id} AND version = #{oldVersion} AND count > 0;
-- 返回0表示被抢占，需重试
```

**问题**：高并发下大量重试 → CPU 自旋浪费；DB QPS 压力大。

### 方案2：Redis 预扣 + DB 异步落盘（推荐）

```
库存初始化：SET stock:{itemId} 10000
扣减：DECRBY stock:{itemId} 1
  → 返回值 ≥ 0：扣减成功，异步发MQ通知DB落盘
  → 返回值 < 0：库存不足，INCRBY 回滚

DB落盘：消费MQ，批量更新DB（削峰）
```

**优点**：Redis 单命令原子，吞吐量高；DB 不承受高并发。

### 方案3：库存拆分（热点行消除）

```
1个热点行 → N个子库存行（如 N=100）
扣减时：随机/轮询选一个子库存扣减
查询总量：SUM(sub_stock) 或定期汇总

适合：DB直接扣减 + 极致高并发（如秒杀）
```

## 三、分布式锁的常见陷阱

**"一锁二查三更新"还出现重复数据**，可能原因：

```
1. 锁粒度不够细
   锁 key=userId，但不同用户可以并发触发同一资源竞争
   → 锁 key 应包含资源 ID（如 itemId），而非仅 userId

2. 锁超时时间设置过短
   业务执行超过锁过期时间 → 锁自动释放 → 其他线程抢到锁
   → 使用 Redisson watchdog 自动续期

3. 未覆盖所有入口
   有多个写入路径（如直接API + MQ消费）但只在其中一个加锁
   → 幂等设计在 DB 层（唯一索引）兜底

4. 锁内查询与更新之间时间差
   "查"和"更新"之间有其他事务修改了数据
   → 更新时加版本号校验（乐观锁）兜底
```

**加了分布式锁影响并发怎么办**：
```
1. 细化锁粒度（按 itemId 而非全局锁）
2. 减少锁内执行时间（不在锁内做外部调用）
3. 锁内批量操作（拿到锁后一次处理多个请求）
4. 无锁化设计（Redis 原子操作 / DB 乐观锁）
```

## 四、分布式 Session 共享

**场景**：用户登录在 Server A，请求打到 Server B 后 Session 找不到。

**方案演进**：

```
方案1：粘性会话（Sticky Session）
  Nginx ip_hash，同一 IP 的请求固定打到同一服务器
  缺点：服务器宕机后 Session 丢失；无法水平扩展

方案2：Redis 集中 Session（推荐）
  Spring Session + Redis：Session 存 Redis，所有节点共享
  优点：无状态服务，任意扩容
  缺点：增加 Redis 依赖

方案3：JWT（无状态）
  用户信息编码在 Token 中，服务端不存 Session
  优点：完全无状态，天然分布式
  缺点：Token 大；注销/踢下线需额外机制（黑名单）
```

## 五、两个独立网站 SSO

**场景**：A.com 和 B.com 是不同域名，如何实现 A 登录后 B 也自动登录。

**方案：统一认证中心（CAS / OAuth2 + OIDC）**：

```
1. 用户访问 B.com → B 发现未登录 → 重定向到 SSO 认证中心
2. SSO 判断用户是否已登录（检查 SSO Session）
   ├── 未登录：显示登录页 → 登录成功 → SSO 创建全局 Session
   └── 已登录（A 之前登录过）→ 直接下一步
3. SSO 生成 Ticket（一次性令牌）→ 重定向回 B.com（携带 Ticket）
4. B.com 后端拿 Ticket 向 SSO 验证 → 获取用户信息 → 创建 B 的本地 Session
```

**要点**：SSO 的全局 Session 存在（用 Cookie 绑定 SSO 域），是"已登录"判断的依据。

## 六、订单号生成设计

**P7 视角**：订单号不只是唯一 ID，还需要包含**业务语义**（可读性、可排查问题）。

```
推荐格式（18~20位）：
时间戳（年月日时分秒，10位）+ 机器ID（2位）+ 序列号（4位）+ 业务类型（2位）

如：20260508123012 + 07 + 0042 + 01
   = 2026050812301207004201

实现方案：
1. 雪花算法变体（去掉毫秒时间戳，换成秒级可读时间）
2. 保证：同一毫秒内序列号递增；时钟回拨处理（等待/报错）
3. 机器 ID 来源：ZK 分配 or K8s Pod 编号
```

**vs 纯 UUID**：UUID 无序导致 B+树索引频繁分裂，写入性能差；订单号有序写入性能好 + 可读。

## 七、跨库 JOIN 的四种解法

分库分表后无法直接 JOIN 不同分片的表：

| 方案 | 原理 | 适用 |
|------|------|------|
| **冗余字段** | JOIN 的字段冗余到本表 | JOIN 少、字段变化少 |
| **应用层 JOIN** | 分别查两个库，代码中合并 | 数据量不大，可接受多次查询 |
| **数据同步到同库** | Canal binlog 同步，在查询库合并 | 读多写少，有延迟可接受 |
| **全局检索（ES）** | 将需要联查的字段同步到 ES | 复杂条件查询 + 全文检索 |

## 八、数据平滑迁移方案

**场景**：不停服情况下，将旧表数据迁移到新库/新表结构。

```
三步走：双写 → 数据追平 → 切流

1. 双写阶段
   新写入：同时写老库和新库（老库为主）
   读取：仍然读老库

2. 历史数据追平
   离线任务读老库全量数据 → 写入新库（增量部分靠双写保证）
   数据校验：对比两边记录数、关键字段 MD5

3. 读切流（灰度）
   1% → 10% → 50% → 100% 流量切到新库
   监控 RT、错误率、数据一致性

4. 停写老库
   确认新库数据完整且稳定后，关闭双写，老库只读保留一段时间

回滚策略：任何阶段异常，立即切回老库（老库一直保持最新数据）
```

## 九、集群定时任务并发

**@Scheduled 在多实例下多个实例同时触发**：

```
方案1：分布式锁（轻量，无框架依赖）
  @Scheduled(cron="0 0 * * * ?")
  public void job() {
      if (!redisLock.tryLock("job_xxx", 30, TimeUnit.SECONDS)) return;
      try { doWork(); } finally { redisLock.unlock(); }
  }
  问题：锁超时时间难估计（业务执行超时则锁失效）

方案2：数据库唯一键（可靠但稍重）
  INSERT INTO job_lock(job_name, exec_time) VALUES('job_xxx', now())
  ON DUPLICATE KEY UPDATE — 失败则跳过

方案3：任务调度框架（推荐生产）
  XXL-Job：DB悲观锁保证只有一个执行器触发，支持故障转移
  适合：任务较多、需要监控/告警/补偿的场景
```

## 十、百万级 Excel 导入

```
核心挑战：读取速度 vs DB写入速度 vs 内存占用

推荐方案（EasyExcel + 分批写入）：
1. 用 EasyExcel 流式读取（SAX模式），避免全量加载
2. 每读取 1000 行 → 批量 INSERT（JDBC addBatch + executeBatch）
3. 开启并行：多线程分段读取 + 线程池写入（注意线程安全）
4. 进度反馈：异步任务 + 进度存 Redis

注意：
- 关闭自动提交，手动事务批量提交
- 导入前关闭外键检查，导入后重建索引（大量数据时）
- 唯一键冲突：INSERT IGNORE or REPLACE INTO
```
