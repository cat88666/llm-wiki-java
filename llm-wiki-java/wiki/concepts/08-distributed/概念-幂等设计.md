# 幂等设计

> 幂等是指同一个操作执行多次与执行一次的结果相同；业务幂等的核心是"推进到最终状态后，重复请求不再重复执行"，高并发场景下通过"一锁二判三更新"实现。

## 一、第一性原理

网络不可靠（超时重试）、消息队列至少一次语义（重复投递）、用户重复点击——这些场景都会导致同一个业务操作被触发多次。幂等设计的目标是：**无论触发几次，系统的最终状态唯一且正确**，而不是依赖调用方"只发一次"。

幂等是系统可靠性的防御性设计，与业务无关——任何需要网络通信的操作都可能被重复触发，必须从系统设计层面保证安全。

## 二、两种幂等的区分

| 类型 | 定义 | 典型场景 |
|------|------|---------|
| **请求幂等** | 相同参数的请求，结果相同 | GET 查询接口（天然幂等）|
| **业务幂等** | 同一次业务（同一幂等号），在拿到最终状态前需继续推进；拿到最终状态后重复请求直接返回已有结果 | 支付、下单、MQ 消费 |

**业务幂等的关键**：需要有一个**幂等号**（如订单号、消息 ID、requestId），作为判断"是否已处理"的唯一凭据，通常由上游调用方生成后传入，服务端不负责生成。

**幂等号设计原则**：
- 全局唯一（UUID/雪花算法）
- 由请求发起方生成（服务端生成无法在重试时复用）
- 幂等号与业务操作绑定（同一幂等号只执行一次业务逻辑）

## 三、一锁二判三更新

```
1. 加锁（互斥）  →  2. 幂等判断  →  3. 执行更新  →  释放锁
```

### 第一步：加锁

使用 Redis 分布式锁（`tryLock`，非阻塞），以幂等号为 key：
- 锁的作用：防止并发请求同时进入下一步判断（解决高并发场景的瞬间重复请求）
- 选 Redis 而非数据库悲观锁：`tryLock` 非阻塞，性能更高，不占 DB 连接

```java
String lockKey = "idempotent:" + idempotentId;
boolean locked = redisLock.tryLock(lockKey, 30, TimeUnit.SECONDS);
if (!locked) throw new ConcurrentException("请求处理中，请稍后重试");
```

### 第二步：幂等判断

- 基于**操作流水表**（推荐）：以幂等号为唯一索引，查询是否已有终态记录
- 基于**状态机**：查询当前业务状态，若已是终态（SUCCESS/FAIL）则直接返回
- 数据库唯一性约束是**最后一道防线**，不是主要手段

### 第三步：更新

执行业务操作，将结果持久化，写入操作流水记录终态。

```java
try {
    // 二判：查流水表
    BizFlow flow = bizFlowMapper.queryByIdempotentId(idempotentId);
    if (flow != null && flow.isTerminal()) {
        return flow.getResult();  // 已处理，直接返回缓存结果
    }
    // 三更新：执行业务 + 写流水
    Result result = bizService.process(...);
    bizFlowMapper.insert(idempotentId, result, "SUCCESS");
    return result;
} finally {
    redisLock.unlock(lockKey);
}
```

## 四、唯一性约束：兜底而非主力

数据库唯一性约束方案（低并发场景）：

```sql
-- 幂等号作唯一索引
CREATE UNIQUE INDEX idx_idempotent_id ON biz_flow(idempotent_id);

-- 捕获 DuplicateKeyException 作为幂等依据
try {
    bizFlowMapper.insert(idempotentId, result);
} catch (DuplicateKeyException e) {
    return bizFlowMapper.getResult(idempotentId);  // 已处理，返回原结果
}
```

**为什么不推荐作为主要手段**：
1. **依赖 insert**：update 操作无法利用此机制（无法用唯一索引判断 update 是否重复执行）
2. **依赖异常控制流**：通过 catch 异常控制业务逻辑，破坏代码可读性，且绑定了特定 ORM/数据库
3. **压力全给数据库**：高并发时并发冲突全部由数据库吸收，无法在应用层提前过滤

**推荐用法**：唯一性约束作为兜底防线（第三道防线），前面的分布式锁 + 流水表作为主要手段（一、二道防线）。万一锁失效，数据库层能阻止脏数据。

## 五、MQ 消费幂等

消息队列"至少一次"（At-least-once）语义意味着消息可能重复投递，消费者必须实现幂等：

```
消费者接收消息
  ↓
以 messageId 为幂等号查消费记录表
  ├── 已消费 → 直接 ACK（跳过业务逻辑）
  └── 未消费 → 执行业务逻辑 + 写入消费记录 + ACK
```

**注意**：消费记录写入与业务操作必须在同一事务内（或用本地消息表机制），否则消费记录写成功但业务失败时，下次消费会被误判为已处理。

```java
@RabbitListener(queues = "order.queue")
@Transactional
public void onMessage(OrderMessage msg) {
    if (consumeRecordMapper.exists(msg.getMessageId())) {
        return;  // 幂等：已消费，跳过
    }
    orderService.process(msg);  // 业务逻辑
    consumeRecordMapper.insert(msg.getMessageId());  // 同一事务写消费记录
}
```

## 六、关键权衡

| 权衡点 | 结论 |
|--------|------|
| 加锁粒度 | 锁的 key 必须是幂等号级别（如订单 ID），不能是全局锁，否则吞吐量极低 |
| 锁的超时设置 | 锁超时时间需大于正常业务处理时间；太短会导致锁提前释放引发并发；考虑使用 Redisson watchdog 自动续期 |
| 软状态的必要性 | 支付中间状态（PROCESSING）允许重试继续推进，但必须有超时兜底，防止永远卡在 PROCESSING |
| 分布式锁 vs 数据库唯一约束 | 并发低时唯一约束更简单高效；并发高时必须加分布式锁保护，数据库约束作兜底 |

## 七、与其他概念的关系

- **[[概念-Redis]]**：幂等设计"一锁"步骤的标准实现，SETNX + 过期时间 + Lua 防误删；Redis 保存幂等标记也是轻量级幂等方案
- **[[机制-RabbitMQ]]**：MQ 消费幂等是可靠消费的必要条件；消息队列"至少一次"语义要求消费端实现幂等
- **[[系统设计-电商系统]]**：秒杀下单的全链路幂等设计：前端防重 + Redis 预扣减幂等 + MQ 消费幂等 + DB 唯一约束兜底
- **[[概念-分布式事务]]**：TCC 的 Confirm/Cancel 必须实现幂等（网络异常可能重复调用）；分布式事务记录表本质上就是幂等控制表

## 八、应用边界

**必须设计幂等的场景**：
- 支付/转账（金融操作，错误代价大）
- MQ 消费处理（至少一次语义）
- 对外 API（客户端超时重试）
- 定时任务补偿（可能与正常流程重叠执行）

**不需要特别设计幂等的场景**：
- 纯查询接口（天然幂等）
- 写操作已有数据库唯一约束且并发低（直接依赖约束即可）
- 内存操作不跨进程（如同 JVM 内的 Map 操作，ABA 问题另行处理）
