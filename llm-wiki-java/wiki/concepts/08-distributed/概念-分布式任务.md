# 分布式任务调度

> 定时任务调度的核心是：**在指定时间点可靠地触发一次且仅一次**；分布式场景下还需解决集群中多实例的协调问题（防止重复执行）和大数据量的并发处理（分片）。

## 一、单机定时任务的分布式缺陷

单机定时任务（Timer/ScheduledExecutorService）依赖 JVM 内存，存在三个根本缺陷：
1. **单点**：机器重启任务丢失，无高可用
2. **无协调**：多实例部署时同一任务被多台机器重复执行
3. **无可视化**：任务执行结果不可追踪

分布式任务调度框架（XXL-Job/Elastic-Job）的本质：用**集中式调度中心**替代分散的 JVM 定时器，通过**分布式锁**保证任务只触发一次，通过**分片**让任务可以水平扩展。

## 二、底层数据结构对比

| 数据结构 | 代表实现 | 特点 | 适用 |
|---------|---------|------|------|
| 小顶堆 | Timer、DelayQueue、ScheduledExecutorService | O(log n) 插入，O(1) 取最小 | 小规模内存定时任务 |
| 时间轮 | Netty HashedWheelTimer、XXL-Job、Kafka | O(1) 插入和触发 | 大量定时任务（如超时检测）|
| 链表 | 简单实现 | O(n) 插入 | 极少量任务 |

## 三、Timer 的核心缺陷

```
Timer（单线程 + TaskQueue）
  问题1：任务A执行时间长 → 阻塞任务B（单线程串行）
  问题2：任务A抛异常未捕获 → Timer线程终止，所有任务停止
  问题3：基于JVM内存，重启后任务丢失

替代方案：ScheduledExecutorService（多线程）+ 持久化框架
```

## 四、时间轮算法

```
时间轮（60 槽，每槽 1 秒）：
  槽位 0 → 1 → 2 → ... → 59 → 0（循环）
  每秒移动一格，执行该槽任务链表

问题：只支持 60s 以内的任务
解决方案1 - round 标识：
  200s 后执行 → 放到槽位 200%60=20，round=3
  每次扫到此槽且 round>0 则 round--，round=0 时执行
  缺点：需遍历所有任务找到 round=0 的

解决方案2 - 分层时间轮（推荐）：
  秒级轮 + 分钟级轮 + 小时级轮
  200s 任务 → 先放分钟轮（第 3 分钟槽）
            → 到期降级到秒轮（第 20 秒槽）→ 触发
```

分层时间轮应用：Netty HashedWheelTimer、Kafka（消息过期清理）、XXL-Job（7.28 版本替换 Quartz）。

## 五、XXL-Job 核心机制

**架构**：调度中心（Admin）+ 执行器（Executor）

**唯一触发保证（防重复调度）**：
```sql
-- 调度中心 JobScheduleHelper 中的关键逻辑
SELECT * FROM xxl_job_lock WHERE lock_name = 'schedule_lock' FOR UPDATE
```
通过 `SELECT FOR UPDATE` 数据库悲观锁，确保集群中只有一个调度中心实例执行调度逻辑。

**分片任务（shardIndex + shardTotal）**：
```
任务被触发时，调度中心分配分片参数：
  shardIndex：当前执行器的分片序号（从0开始）
  shardTotal：总分片数

执行器根据分片参数过滤数据：
  SELECT * FROM orders WHERE id % shardTotal = shardIndex
  → 多个执行器并行处理不同数据段
```
适合：批量处理数百万订单、用户数据扫描等大数据量任务。

## 六、定时扫表的三个缺陷

| 缺陷 | 解法 |
|------|------|
| 数据量大扫表慢 | ① 加索引（状态字段低区分度可用，未完成数据占比小）；② 线程池并发扫 + 按 ID/bizId 分段隔离 |
| 集中扫表压主库 | ① 扫备库（延迟可接受）；② 分库分表减少单库压力 |
| 定时精度不足 | 改用**延迟消息**（MQ）+ 同步转异步（先同步试一次，失败再异步捞） |

## 七、退避策略

任务失败重试时，不能立即无限重试（会压垮下游）：

| 策略 | 延迟计算 | 适用 |
|------|---------|------|
| 固定退避 | 每次等待相同时间（如 5s）| 简单场景 |
| 指数退避 | `delay = base × 2^n`（100ms→200ms→400ms）| 大多数重试场景 |
| 指数随机退避 | 指数 + 随机因子，避免惊群效应 | 多客户端并发重试 |

RabbitMQ/RocketMQ 消费失败重投默认使用指数退避。

## 八、框架选型对比

| 特性 | XXL-Job | Elastic-Job | PowerJob |
|------|---------|-------------|---------|
| 分片 | 静态分片（shardIndex/Total）| 静态分片 | MapReduce 动态分片 |
| 唯一触发 | DB 悲观锁 | Zookeeper | 无锁化设计 |
| 可视化 | 有 | 有 | 有 |
| 工作流/DAG | 不支持 | 不支持 | 支持 |
| 复杂度 | 低（推荐入门）| 中 | 高 |

**MapReduce 动态分片（PowerJob）** 对比 XXL-Job 静态分片的优势：不需要预设分片数，根据数据量动态生成子任务，新增机器立即生效，扩展性更强。

## 九、分布式调度方案取舍

1. **DB 锁 vs ZK 锁 vs 无锁**：XXL-Job 用 DB 悲观锁简单可靠但性能一般；Elastic-Job 用 ZK 性能更好但引入 ZK 依赖；PowerJob 无锁化性能最佳但实现复杂
2. **定时扫表 vs 延迟消息**：扫表简单但有延迟和库压力；延迟消息实时性好但引入 MQ 依赖，需处理消息可靠性
3. **静态分片 vs 动态分片**：静态分片（XXL-Job）配置简单，适合数据量相对固定；动态分片适合数据驱动、弹性扩缩场景
4. **时间轮 vs 小顶堆**：小顶堆插入 O(logN)，时间轮 O(1)；任务量大时时间轮性能优，任务量少时差异不大

## 十、分布式任务的系统位置

- 依赖 [[机制-线程池]]：分片任务的并发执行依赖线程池；定时任务框架内部用 ScheduledExecutorService 驱动调度
- 关联 [[概念-幂等设计]]：分布式任务可能因网络重试被重复触发，执行逻辑必须幂等
- 关联 [[机制-Zookeeper]]：Elastic-Job 用 ZK 做分布式协调，保证分片分配的一致性
- 替代 [[机制-RabbitMQ]] 延迟消息的部分场景：定时扫表和延迟消息在"延迟触发"场景互为替代方案

## 十一、分布式任务适用边界

**适合定时任务**：批量数据处理（报表生成、账单结算）、数据定期清理、轮询类任务。

**适合延迟消息替代定时任务**：对实时性要求高（秒级）、数据量极大、不想在 DB 中存大量定时记录的场景。

**选型建议**：
- 中小团队、快速上手 → XXL-Job
- 需要弹性分片、DAG 工作流 → PowerJob
- 已有 ZK 基础设施 → Elastic-Job
