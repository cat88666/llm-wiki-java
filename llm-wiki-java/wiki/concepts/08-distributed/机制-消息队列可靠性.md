# 消息队列可靠性

> 消息队列的核心价值：异步解耦 + 削峰填谷。可靠性保证需从三段入手：生产者确认（消息不丢进 MQ）→ 消息持久化（MQ 不丢）→ 消费者 ACK（消费不丢）。工程上还要回答：什么时候该用 MQ、如何处理乱序、如何做本地消息表。

## 第一性原理

消息队列解决的问题：**让生产者和消费者在时间和空间上解耦**。但解耦带来的代价是消息可能在任何一个环节丢失，且消费者可能重复收到消息。工程上的主流方案不是追求理论上的“恰好一次”，而是实现**至少一次 + 业务幂等**。

## 核心机制

### RabbitMQ 整体架构

```
Producer → Exchange（交换机）→ Queue（队列）→ Consumer
                ↑
            Binding（路由规则）
```

### Exchange 类型

| 类型 | 路由规则 | 场景 |
|------|---------|------|
| Direct | 精确匹配 RoutingKey | 点对点路由 |
| Fanout | 广播到所有绑定队列 | 发布/订阅 |
| Topic | 通配符匹配 | 复杂路由 |
| Headers | 匹配消息 headers | 少用 |

### 消息可靠性三段保证

```
【生产者端】Publisher Confirm + Publisher Returns
    ↓
【MQ 侧】队列/交换机/消息持久化
    ↓
【消费者端】手动 ACK / NACK
```

### 生产者确认

- Publisher Confirm：消息到达 Exchange 后 ACK/NACK
- Publisher Returns：消息无法路由到 Queue 时返回给生产者
- 失败时要记录日志、报警、重试

### 消息持久化

- Queue `durable=true`
- 消息 `deliveryMode=2`
- 但极端情况下仍可能丢失，因此要有业务层兜底

### 消费者确认

- 关闭自动 ACK
- 处理成功后手动 `basicAck`
- 失败后 `basicNack` / 重投

### 消费者限流

```java
channel.basicQos(10);
```

用于防止消费端被大量未 ACK 消息打垮。

### 防重复消费（幂等）

重复消费根因：ACK 丢失或超时，MQ 重新投递。

通用解法：
1. 消息体带唯一业务 ID
2. 消费时一锁 / 唯一索引保护
3. 二判：查是否已处理
4. 三更新：处理并标记成功

### 死信队列（DLQ）与延迟消息

消息成为死信的条件：TTL 过期 / 队列满 / Reject 且不重投。

| 方案 | 原理 | 优缺点 |
|------|------|--------|
| 死信队列 + TTL | 消息过期进入 DLQ | 简单，但有队头阻塞 |
| delayed_message 插件 | 延迟到期后投递 | 更灵活，但不适合超大量级超长延迟 |

### 高可用模式

| 模式 | 数据分布 | 高可用性 |
|------|---------|---------|
| 普通集群 | 消息只在一个节点 | 节点宕机时消息不可读 |
| 镜像集群 | 消息和元数据同步到多个节点 | 任一节点宕机仍可读 |
| Quorum Queue | 基于 Raft | 推荐方案 |

## 场景综合

### Spring Event vs MQ

| 维度 | Spring Event | MQ |
|------|-------------|----|
| 作用范围 | 同 JVM | 跨服务/跨进程 |
| 可靠性 | 进程崩溃即丢失 | 可持久化 |
| 延迟 | 极低 | 毫秒级 |
| 运维成本 | 低 | 高 |
| 典型场景 | 同服务模块解耦 | 跨服务异步通信、削峰 |

**原则**：同服务内解耦优先 Spring Event；跨服务、持久化、削峰才用 MQ。

### BlockingQueue vs MQ

`BlockingQueue` 只适合同 JVM 内线程协作，不适合分布式消息：
- 不持久化
- 不能跨进程
- 没有路由能力
- 没有重试和死信
- 没有监控和堆积治理

### 拉模式 vs 推模式

| | 拉模式（Kafka）| 推模式（RabbitMQ）|
|--|---|---|
| 控制权 | 消费者主动拉 | Broker 主动推 |
| 实时性 | 略低 | 更高 |
| 过载风险 | 不易压垮消费者 | 需配合限流 |
| 批量消费 | 天然适合 | 需额外控制 |

### 消息乱序：四种解法

1. **顺序消息**：同一业务 key 进同一 Partition / Queue
2. **前置状态判断**：消息携带 `beforeStatus`，消费时校验状态机
3. **事件表 + 异步重试**：先落事件表，再异步按序处理
4. **业务容忍乱序**：通过最终状态覆盖或幂等推进规避顺序依赖

### 提升 Kafka 单分区消费吞吐

约束：单 Partition 在同一 Consumer Group 下只能被一个消费者消费。

可行方案：
- 增大 `max.poll.records`
- poll 线程只拉取，业务放线程池异步处理
- 批量写 DB
- 根本解法是增加 Partition

### 为什么不建议用 MQ 实现订单超时关闭

主要问题：
- 延迟不够精准
- 消息可靠性要求高
- 重复消费需幂等
- 已投递消息难取消

更推荐：
- Redisson `RDelayedQueue`
- 调度框架扫表
- 专用延迟任务系统

## 本地消息表

### 下游失败时，上游如何处理

原则：**不回滚上游已提交业务，只保证下游最终一致。**

流程：
1. 上游业务提交成功
2. 消息表写入 `PENDING`
3. 定时任务扫描待发送消息
4. 下游失败则重试
5. 超阈值进入 `FAILED` + 报警
6. 人工补偿

### 消息表设计

```sql
CREATE TABLE message_record (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    msg_id VARCHAR(64) NOT NULL UNIQUE,
    biz_id VARCHAR(64) NOT NULL,
    biz_type VARCHAR(32) NOT NULL,
    topic VARCHAR(64),
    payload TEXT,
    state TINYINT DEFAULT 0,
    retry_count INT DEFAULT 0,
    next_retry_time DATETIME,
    create_time DATETIME,
    update_time DATETIME,
    INDEX idx_state_retry (state, next_retry_time)
);
```

双保险：
1. 业务操作和消息表写入同事务
2. 定时任务扫描未发送消息重投

## 关键权衡

1. **持久化 vs 性能**：持久化提高可靠性，也增加 I/O
2. **至少一次 vs 恰好一次**：工程上通常选择至少一次 + 幂等
3. **RabbitMQ / Kafka / RocketMQ**：不同产品侧重点不同，但可靠性三段思想通用
4. **MQ vs 本地调用**：不是所有异步都值得引入 MQ，要看跨进程、削峰、可靠性诉求

## 应用边界

**适合 RabbitMQ**：复杂路由、精细 ACK、企业内部业务流。

**适合 Kafka**：高吞吐、日志流、可回放、流计算生态。

**适合 RocketMQ**：事务消息、延迟消息、金融电商链路。

**不要滥用 MQ**：
- 同 JVM 内部解耦
- 实时强一致写链路
- 简单定时任务
