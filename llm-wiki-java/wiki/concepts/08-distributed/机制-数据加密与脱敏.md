# 数据加密与脱敏

> 通过对称加密（AES）、非对称加密（RSA）的混合使用，以及脱敏策略，在数据传输和存储两个层面保护敏感信息；接口签名防止请求篡改和重放攻击。

## 一、第一性原理

敏感数据的三个暴露面：
1. **传输中**（In-transit）：网络抓包 → 用 HTTPS/WSS + 应用层加密双重保护
2. **存储中**（At-rest）：DB 拖库 → 字段级加密 + 不可逆脱敏
3. **使用中**（In-use）：日志/监控泄露、内存持留 → 脱敏展示 + 内存清零

单靠 HTTPS 不够：HTTPS 只保护传输，DB 被拖库后数据仍裸露；日志收集不当也会泄露明文。纵深防御要求每一层都有独立的保护措施。

## 二、对称加密 AES-256-GCM

```java
// AES-256-GCM：认证加密（AEAD），同时保证保密性 + 完整性
SecretKey key = new SecretKeySpec(keyBytes, "AES");  // 256-bit key
byte[] iv = new byte[12];  // 12字节 IV，每次加密必须随机生成！
new SecureRandom().nextBytes(iv);

Cipher cipher = Cipher.getInstance("AES/GCM/NoPadding");
cipher.init(Cipher.ENCRYPT_MODE, key, new GCMParameterSpec(128, iv));
byte[] ciphertext = cipher.doFinal(plaintext);
// 存储格式：Base64(iv + ciphertext)，解密时先取前12字节为IV
```

**为什么用 GCM 而不是 CBC**：
- AES-CBC 只保证保密性，不保证完整性（Padding Oracle 攻击）
- AES-GCM 输出包含 Authentication Tag（16字节），解密时自动验证数据未被篡改
- GCM 无需 Padding，性能更好

**IV（初始化向量）铁律**：同一密钥下每次加密必须使用不同的随机 IV；IV 复用会导致 GCM 完全失去安全性（可还原明文差异）。

## 三、非对称加密 RSA 密钥交换

```java
// RSA 只用于加密对称密钥（AES Key），不用于加密大量数据（慢 100x）
// 混合加密流程：
// 1. 接收方发布 RSA 公钥
// 2. 发送方生成随机 AES 会话密钥（256-bit）
// 3. 用 RSA 公钥加密 AES 密钥 → 发送
// 4. 接收方用 RSA 私钥解密得到 AES 密钥
// 5. 后续数据用 AES 密钥加密

PublicKey rsaPub = ...;
Cipher rsaCipher = Cipher.getInstance("RSA/ECB/OAEPWithSHA-256AndMGF1Padding");
rsaCipher.init(Cipher.ENCRYPT_MODE, rsaPub);
byte[] encryptedAesKey = rsaCipher.doFinal(aesKeyBytes);
```

**RSA-OAEP vs PKCS1v15**：OAEP 是现代标准，PKCS1v15 有已知漏洞（Bleichenbacher 攻击），生产必须用 OAEP。

## 四、数字签名

```
加密：用对方公钥加密 → 只有对方私钥能解密（保密性）
签名：用自己私钥签名 → 任何人用公钥验证（不可否认性 + 完整性）
```

```java
// 接口请求签名防篡改
Signature sig = Signature.getInstance("SHA256withRSA");
sig.initSign(privateKey);
sig.update(rawData);
byte[] signature = sig.sign();

// 接收方验证
sig.initVerify(publicKey);
sig.update(rawData);
boolean valid = sig.verify(signature);
```

## 五、请求防重放

```java
// 请求头携带：timestamp（Unix ms）+ nonce（UUID）+ sign
// 防重放逻辑：
// 1. 检查 |currentTime - timestamp| ≤ 5分钟（时间窗口）
// 2. 检查 nonce 是否在 Redis 中存在（已见过则拒绝）
//    SETNX nonce:${nonce} 1 EX 300  → 5分钟 TTL 与时间窗口一致
// 3. 验证签名：HMAC-SHA256(timestamp + nonce + sorted_params, secret_key)

String toSign = timestamp + "\n" + nonce + "\n" + sortedParams;
Mac hmac = Mac.getInstance("HmacSHA256");
hmac.init(new SecretKeySpec(secretKey, "HmacSHA256"));
String signature = Base64.encode(hmac.doFinal(toSign.getBytes()));
```

**三层防御**：时间窗口（防旧请求重放）+ Nonce 去重（防时间窗口内重放）+ HMAC 签名（防参数篡改）。

## 六、数据脱敏策略

| 数据类型 | 脱敏规则 | 示例 |
|---------|---------|------|
| 手机号 | 保留前3后4，中间替换 | `138****8000` |
| 身份证 | 保留前4后4，中间替换 | `1101**********2310` |
| 银行卡 | 保留后4位 | `****1234` |
| 邮箱 | 保留前2字符+域名 | `zh****@gmail.com` |
| 姓名 | 保留首字，其余替换 | `张**` |
| 密码 | 永远不展示（`******`）| — |

**可逆脱敏（AES 加密存储）**：用于需要业务使用原始值的字段（如手机号需发短信）。
**不可逆脱敏（Hash 索引）**：用于仅需比较的场景（如查询某手机号是否注册）。存储 `SHA256(phone + salt)`，查询时也 Hash 后比较。

## 七、字段级加密 + Hash 索引

```sql
-- 手机号明文不存 DB，存加密值 + 查询用 Hash
CREATE TABLE user (
    id          BIGINT,
    phone_enc   VARCHAR(512),   -- AES-256-GCM 加密后 Base64
    phone_hash  CHAR(64),       -- SHA256(phone + system_salt)，用于精确查询
    INDEX idx_phone_hash (phone_hash)
);
-- 查询：WHERE phone_hash = SHA256(inputPhone + salt)
-- 展示：解密 phone_enc，脱敏后返回
```

**设计要点**：
- Hash 字段只支持等值查询（LIKE、范围查询不可用）
- salt 每个系统独立，防彩虹表攻击
- AES 密钥存 KMS/HSM，不能硬编码在代码中

## 八、私钥内存安全

```java
// 使用完立即清零，防止 GC 前被 Heap Dump 读取
byte[] keyBytes = loadKey();
try {
    // ... 使用 keyBytes 执行加密
} finally {
    Arrays.fill(keyBytes, (byte) 0);  // 强制清零，GC 不能保证及时回收
}
```

**为什么 GC 不够**：JVM GC 不保证立即回收，堆内存中的密钥可能在 heap dump 中暴露。`Arrays.fill` 主动清零是纵深防御的最后一道防线。

## 九、关键权衡

| 维度 | 说明 |
|------|------|
| 性能开销 | AES-GCM 在现代 CPU（AES-NI 指令集）下约 1-4 GB/s，对业务延迟影响极小；RSA 慢（~1ms/次），只用于密钥交换 |
| 可搜索性 | 加密字段无法 LIKE 查询，需要配套 Hash 索引或 Token 化方案 |
| 密钥管理 | 密钥比数据更重要；密钥泄露等于加密失效；生产用 KMS/HSM 管理 |
| 可逆 vs 不可逆 | 脱敏展示用不可逆（Hash/替换）；业务需要原始值用可逆（AES）|

## 十、与其他概念的关系

- 区别于 [[概念-网络安全]]：网络安全页侧重 Web 攻防（XSS/CSRF/注入），本页侧重数据保护的实现机制（加密算法 + 脱敏）
- 依赖 [[概念-Redis]]：Nonce 去重存 Redis，防止重放攻击（SETNX + TTL = 时间窗口内唯一）
- 依赖 [[概念-MySQL]]：Hash 索引字段设计需要理解 B+树索引限制（无法 LIKE，只能等值查询）

## 十一、应用边界

**必须全链路加密的场景**：
- 支付/金融系统（账号、卡号、金额、交易凭证）
- 用户 PII 数据（手机、身份证、地址）
- 区块链私钥（绝对不可明文存储）

**脱敏而非加密的场景**：
- 日志输出中的敏感字段（日志级别无法加密，只能脱敏）
- 前端展示（用户看自己的手机号时展示脱敏值）
- 数据分析/报表（用脱敏数据，避免分析团队接触原始 PII）

**不需要应用层加密的场景**：
- 纯内网服务间通信（mTLS 已足够）
- 非敏感数据（公开商品名称、ID）
