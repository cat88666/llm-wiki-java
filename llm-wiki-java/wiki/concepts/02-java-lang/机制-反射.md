# 反射机制

> 程序在**运行时**获取并操作自身类型信息的能力——给一个类名，就能拿到它的全部结构并调用。

## 第一性原理

Java 编译器在生成字节码时，把类的完整结构（字段、方法、构造器、修饰符）写入 `.class` 文件。JVM 加载类时，在堆中为每个类维护唯一的 `Class` 对象作为运行时镜像。反射的本质是：**读取这个 `Class` 对象里存的元数据，再通过它发起调用**。

存在理由：编译期无法确定所有类型（插件系统、框架注入、序列化）时，必须推迟到运行期做类型决策。

## 核心机制

1. **获取 Class 对象**（三种途径）
   - `Class.forName("全限定名")` — 触发类加载
   - `obj.getClass()` — 运行时实例拿
   - `SomeClass.class` — 编译期字面量

2. **读取结构**
   - `getMethods()` / `getDeclaredMethods()` — 后者含私有方法
   - `getFields()` / `getConstructors()` 同理

3. **发起调用** `method.invoke(obj, args)`
   - 参数装箱成 `Object[]`，调用后再拆箱
   - 每次 invoke 都要做**可见性检查 + 方法表遍历**
   - 没有 JIT 内联优化

## 关键权衡

| 代价 | 原因 |
|------|------|
| **性能低** | 跳过 JIT 优化；参数装/拆箱；方法遍历；可见性检查 |
| **破坏封装** | `setAccessible(true)` 可绕过 private |
| **可读性差** | 调用链在编译期不可见，IDE 无法追踪 |
| **安全风险** | 可修改 final 字段（JDK 17+ 对此加固）|

反射不是银弹：性能敏感路径（高频调用）应避免，可用 MethodHandle 或代码生成替代。

## 与其他概念的关系

- 支撑了 [[机制-动态代理]]：JDK 动态代理的 `InvocationHandler.invoke()` 底层是反射调用
- 支撑了 [[机制-SPI]]：ServiceLoader 通过 `Class.forName()` + `newInstance()` 加载实现类
- 支撑了 [[机制-Java序列化]]：序列化框架用反射读写字段值
- 支撑了 L6 框架层：Spring IoC 通过反射实例化 Bean 并注入依赖

## 应用边界

**适合用**：框架开发（IoC、ORM、RPC）、动态代理、通用工具（BeanUtils、JSON 库）、SPI 加载。

**不适合用**：高频业务代码；可以编译期确定类型的场景；安全敏感模块（JDK 17+ 模块系统对反射访问有限制）。
