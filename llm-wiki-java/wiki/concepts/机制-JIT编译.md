# JIT编译

> JVM 默认混合执行模式（Mixed Mode）：解释器逐行翻译字节码执行，JIT（即时编译器）识别"热点代码"将其编译为本地机器码缓存复用——用"先跑再编译"换取运行期的更优化。

## 第一性原理

纯解释执行：一边翻译一边运行，每次执行都要重复翻译，效率低。纯预编译（AOT）：编译时不知运行时的实际调用热点，优化信息不足。JIT 的存在理由：**在程序运行后，用实际执行频率数据指导编译，只对真正"热"的代码做激进优化，代价换来最优化收益**。

## 核心机制

### Java 的执行模式

```
源代码 .java
  ↓ javac 编译
字节码 .class（平台无关中间语言）
  ↓ JVM 解释执行（慢，但立即可用）
  ↓ JIT 编译热点代码（快，但需要预热）
本地机器码（直接运行，最快）
```

**Java 既不是纯编译型，也不是纯解释型，是混合执行（Hybrid）**。

### 热点代码识别

HotSpot JVM 用**基于计数器的热点探测**：
- **方法调用计数器**：方法被调用次数超过阈值（默认 Client 1500 次，Server 10000 次）→ 触发 JIT
- **回边计数器**：循环体执行次数超过阈值 → 触发 OSR（栈上替换，对正在运行的方法进行 JIT）

### JIT 的主要优化手段

| 优化 | 原理 | 效果 |
|------|------|------|
| **方法内联** | 把被调用方法的代码复制到调用处，消除方法调用开销 | 最常见优化，是其他优化的前提 |
| **逃逸分析** | 判断对象是否会被方法外访问（是否"逃逸"）| 是栈上分配、锁消除的前提 |
| **栈上分配** | 无逃逸对象在栈上分配，随栈帧消失自动回收，不触发 GC | 减少 GC 压力 |
| **锁消除** | 无逃逸对象的 synchronized 块，JIT 直接消除锁 | 减少同步开销 |
| **标量替换** | 无逃逸对象分解为各个字段存于寄存器/栈，不创建对象 | 进一步减少 GC 对象数 |

### 逃逸状态

| 状态 | 说明 | 可做的优化 |
|------|------|----------|
| **全局逃逸** | 赋给静态字段 / 作为方法返回值 | 方法内联、循环优化 |
| **参数逃逸** | 作为参数传给其他方法 | + 锁消除 |
| **无逃逸** | 仅在当前方法内存活 | + 栈上分配、标量替换 |

### JIT vs AOT

| 维度 | JIT | AOT（Ahead-of-Time）|
|------|-----|---------------------|
| 编译时机 | 运行时（热点触发）| 程序启动前（预编译）|
| 启动速度 | 慢（需要预热）| 快（直接运行机器码）|
| 峰值性能 | 高（利用运行期 profile 做激进优化）| 次之 |
| 适用场景 | 长期运行的服务 | Serverless、短生命周期容器（GraalVM Native Image）|

## 关键权衡

1. **预热问题（Warmup）**：JIT 优化在热点触发前，所有请求经解释执行，**应用刚启动时响应最慢**。高并发发布时会出现大量超时，需要"流量预热"（先放少量流量让 JIT 完成优化）
2. **逃逸分析不成熟**：分析本身有开销；不是所有对象都能被分析为无逃逸；在极端情况下分析开销 > 优化收益
3. **JIT 编译的机器码也占内存**：存于方法区的 CodeCache，`-XX:ReservedCodeCacheSize` 控制大小，满了则退化回解释执行

## 与其他概念的关系

- JIT 编译后的机器码缓存在 [[机制-JVM内存模型]] 的方法区（CodeCache）
- 逃逸分析让对象在栈上分配，直接减少了 [[机制-GC算法与垃圾收集器]] 的工作量
- 锁消除优化与 L3 并发编程强关联：`synchronized` 在无逃逸时会被 JIT 直接消除

## 应用边界

**需要关注 JIT 预热的场景**：
- 蓝绿/滚动发布：刚重启的实例 JIT 未完成，QPS 不能立即拉满
- 解决方案：摘流量预热（先放 10% 流量），或使用阿里 Dragonwell JwarmUp（将上次 JIT 编译信息落盘，下次启动提前编译）

**不需要 JIT 的场景**：Serverless 函数、Lambda 冷启动 → 改用 GraalVM Native Image（AOT 预编译成本地可执行文件，启动毫秒级）
