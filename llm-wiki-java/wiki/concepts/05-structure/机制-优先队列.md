# 优先队列

> 堆是满足"父节点优先于子节点"约束的完全二叉树，用数组存储；优先队列是堆的抽象接口——每次出队的是优先级最高（或最低）的元素，插入和删除均为 O(log n)，读堆顶为 O(1)。

## 一、第一性原理

很多场景需要持续维护一个集合中的"最值"（最大/最小），且集合是动态变化的（边插入边取出）。排序每次重算代价是 O(n log n)；堆维护"偏序"而非全序，只保证堆顶是最值，代价仅 O(log n)。

## 二、核心机制

### 结构定义

**完全二叉树**：除最后一层外每层填满，最后一层从左往右排列。

- **大顶堆（Max Heap）**：任意节点 ≥ 所有子节点 → 堆顶是最大值
- **小顶堆（Min Heap）**：任意节点 ≤ 所有子节点 → 堆顶是最小值

### 数组映射公式

完全二叉树可用数组紧凑存储，无需指针：

```
设节点下标为 i（从 0 开始）：
  parent[i]  = (i - 1) / 2
  left[i]    = 2 * i + 1
  right[i]   = 2 * i + 2
```

### 核心操作

| 操作 | 描述 | 时间复杂度 |
|------|------|-----------|
| 插入 | 末尾追加 + 上浮（Heapify Up） | O(log n) |
| 删除堆顶 | 堆顶与末尾交换 + 下沉（Heapify Down） | O(log n) |
| 查看堆顶 | 直接读 arr[0] | O(1) |
| 建堆 | 从最后一个非叶节点倒序下沉 | O(n) |

### Top K 问题——用小顶堆找最大 K 个

**关键反直觉**：找最大 K 个 → 用小顶堆（维护 K 个候选）

```
算法：维护容量为 K 的小顶堆
  遍历每个新元素 x：
    若堆未满 → 直接入堆
    若 x > 堆顶（当前最小候选）→ 弹出堆顶，x 入堆
    否则 → 丢弃 x
  结果：堆中 K 个元素即为最大 K 个
```

**为何不用大顶堆**：大顶堆需要存全部数据才能持续弹出最大值，内存 O(n)；小顶堆内存仅 O(k)，每步 O(log k)，适合海量数据流。

### TP99 计算

用大顶堆 + 小顶堆维护 99:1 分割点：
- 大顶堆存前 99% 请求时间，小顶堆存后 1%
- 大顶堆堆顶即为实时 TP99，无需等全部数据到齐

## 三、核心使用原则

1. **堆 ≠ 全局排序**：只保证堆顶是最值，内部无序；若需有序遍历，用 `TreeMap`
2. **不支持随机访问**：无法 O(1) 查找任意值，只能 O(1) 读堆顶
3. **Java PriorityQueue 非线程安全**：并发场景用 `PriorityBlockingQueue`
4. **建堆 O(n) vs 逐个插入 O(n log n)**：批量建堆更高效，适合离线处理

## 四、与其他概念的关系

- 是 [[概念-Java集合]] 中优先队列的底层实现
- 区别于 [[机制-红黑树]]：`TreeMap` 底层红黑树支持有序遍历；堆只保证堆顶极值
- 支撑了 L3 并发：`PriorityBlockingQueue` 是线程安全的堆，用于有优先级的任务队列
- 支撑了 L7 分布式：Dijkstra 最短路径算法用小顶堆选当前最短路径节点

## 五、应用边界

**小顶堆**：Top K 最大值（内存效率高）；定时器（取最近触发的任务）；Dijkstra 最短路。

**大顶堆**：Top K 最小值；堆排序（从大到小）；实时获取最大值的场景。

**Java 实现**：`PriorityQueue<Integer> minHeap = new PriorityQueue<>()` （默认小顶堆）；大顶堆：`new PriorityQueue<>(Comparator.reverseOrder())`。

**不适合**：需要有序遍历（用 `TreeSet`/`TreeMap`）；需要随机查找（用 `HashMap`）。
