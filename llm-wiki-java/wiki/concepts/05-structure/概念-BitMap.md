# BitMap

> **BitMap**：用 1 bit 标记整数是否存在的紧凑数组，空间压缩比 32x（int → 1 bit）。  
> **布隆过滤器**：BitMap + k 个哈希函数，概率性判断元素"一定不存在 / 可能存在"。  
> **布谷鸟过滤器**：存指纹 + 双候选桶，解决布隆过滤器无法删除的核心缺陷。

## 一、第一性原理

存储 N 个整数最朴素的方法是数组，每个 int 占 4 字节。若只需知道某整数"有没有"，实际只需 1 bit。**BitMap：当数据语义退化为 boolean，bit 数组将空间压缩 32 倍**。

精确集合（HashSet）判断存在性内存代价与元素数量线性相关。**布隆过滤器洞察：很多场景只需知道"一定不存在"，允许小概率误判即可用极少 bit 替代大量内存**。

## 二、核心机制

### 一、BitMap

**结构**：下标 = 整数值，值 = 0/1（不存在/存在）

```
存储 {1, 4, 6}：
index: 0 1 2 3 4 5 6 7
value: 0 1 0 0 1 0 1 0
```

**空间对比**（存储 10 亿个 int 的去重集合）：

| 结构 | 内存 |
|------|------|
| HashSet | 10亿 × ~40 字节 ≈ **40 GB** |
| BitMap | 10亿 bit = **125 MB** |

**Java 实现**：`java.util.BitSet`

```java
BitSet bs = new BitSet(100);
bs.set(1);    // 标记存在
bs.get(4);    // 查询
bs.clear(4);  // 移除（BitMap 支持删除，布隆过滤器不行）
```

**Redis 支持**：`SETBIT key offset 1` / `GETBIT key offset`，适合用户签到、UV 统计。

### 二、布隆过滤器

**数据结构**：长度为 m 的 bit 数组 + k 个独立哈希函数

**写入**：对元素 x，用 k 个哈希函数各算出一个 [0, m-1] 索引，对应 bit 置 1

**查询**：k 个索引对应 bit 全为 1 → 可能存在；有一个为 0 → **一定不存在**

```
元素 "Hollis" → hash1→ bit[3]=1, hash2→ bit[17]=1, hash3→ bit[42]=1
查询 "Hero"   → bit[3]=1, bit[17]=1, bit[99]=0 → 一定不存在 ✓
查询 "Villain" → bit[3]=1, bit[17]=1, bit[42]=1 → 可能存在（哈希碰撞误判）
```

**误判率（False Positive Rate）**由 m 和 k 共同决定：
- 最优 k = (m/n) × ln2（n 为元素数）
- 实际使用时传入预期容量和目标误判率，框架自动计算 m 和 k

**Java 实现**（Guava）：

```java
BloomFilter<String> bf = BloomFilter.create(
    Funnels.stringFunnel(Charsets.UTF_8), 1_000_000, 0.01);  // 容量100w，误判率1%
bf.put("Hollis");
bf.mightContain("Hero");    // false → 一定不存在
bf.mightContain("Hollis");  // true  → 可能存在
```

### 三、布隆过滤器缺陷与解决

**核心缺陷**：一个 bit 可能被多个元素哈希共享，清 0 会误伤其他元素 → **无法删除**。

| 解决方案 | 思路 | 代价 |
|---------|------|------|
| 定期重建 | 基于最新数据重建后原子切换 | 重建期间旧数据可能误判 |
| 计数布隆过滤器 | 每位改为 3-4 bit 计数器，删除时减一 | 空间增 300-400%，计数可能溢出（>15）|
| **布谷鸟过滤器** | 全新设计，存指纹 + 双候选桶 | 支持删除，插入极端情况可能失败 |

### 四、布谷鸟过滤器

**核心设计**：不用 bit，改存每个元素的**指纹**（短哈希，如 8 bit）；每个元素有**两个候选桶**，用 XOR 关联：

```
i1 = hash(x) % N
i2 = i1 XOR hash(fingerprint(x)) % N
```

XOR 性质：已知 i1 和指纹 f 可算出 i2，反之亦然 → 删除时可以精确定位。

**插入**：i1 或 i2 有空槽 → 存入；都满 → 随机踢出已有指纹（被踢者找其另一候选桶，名称由此来）；踢出链过长 → 插入失败，需扩容。

**删除**：计算 i1 和 i2，找到对应指纹槽位，清空。误删概率极低（指纹精确匹配）。

**布隆 vs 布谷鸟对比**：

| 特性 | 布隆过滤器 | 布谷鸟过滤器 |
|------|-----------|------------|
| 误判率 | 有 | 更低 |
| 删除 | ❌ | ✅ |
| 空间效率 | bit 级别，极小 | 略大（存指纹）|
| 插入失败 | 不会（误判率升高）| 极端情况会失败 |
| Redis 支持 | ✅ RedisBloom 模块 | ✅ Redis 8.0 原生 |

## 三、核心使用原则

1. **BitMap 值域限制**：值域越大空间越大（与元素数无关）；64 位随机 ID 不适合 BitMap
2. **布隆容量要留余量**：超过预期容量后误判率急剧劣化，建议设置 1.5-2 倍
3. **Redis 集群中的布隆**：需用 hash tag `{key}` 保证同一 key 落同一节点，否则 bit 数组被分片后查询结果不一致
4. **布谷鸟 vs 布隆的选择**：需要删除 → 布谷鸟；只查询且数据量极大 → 布隆更省空间

## 四、与其他概念的关系

- **[[概念-缓存三大问题]]**：布隆过滤器是解决缓存穿透的标准方案——将存在的 key 加入过滤器，请求先经过过滤，"一定不存在"的直接拦截
- **[[机制-Redis持久化]]**：Redis 存储的布隆/布谷鸟过滤器需要考虑 RDB/AOF 持久化，防止重启后过滤器数据丢失
- **[[概念-Java集合]]**：BitMap 是最紧凑的线性 boolean 数组

## 五、应用边界

**BitMap**：海量整数去重（10 亿整数去重 125MB）；用户签到/在线状态；大数据排序（整数值域有限）。

**布隆过滤器**：缓存穿透防护（黑名单 URL/不存在商品 ID）；海量数据去重（爬虫 URL）；分布式缓存 key 预判。

**布谷鸟过滤器**：需要频繁删除的场景替代布隆过滤器。

**不适合**：需要精确"一定存在"判断；元素数量远超预期（改用精确集合）。
