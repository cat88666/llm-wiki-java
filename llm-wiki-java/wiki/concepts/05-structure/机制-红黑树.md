# 红黑树

> 一种自平衡二叉查找树——通过节点着色（红/黑）和 5 条约束规则，保证最长路径 ≤ 最短路径 × 2，从而在 O(log n) 内完成增删查。

## 一、第一性原理

普通 BST（二叉搜索树）在最坏情况（顺序插入）退化为链表，时间复杂度降至 O(n)。AVL 树要求严格平衡（左右子树高度差 ≤ 1），维护代价过高。红黑树的存在理由：**用"近似平衡"换取更低的维护成本，让插入/删除在 O(log n) 内完成，同时比 AVL 树少做旋转**。

## 二、核心机制

### 5 条红黑规则

1. **节点颜色**：每个节点是红色或黑色
2. **根节点**：根节点必须是黑色
3. **红色约束**：红色节点的子节点必须是黑色（不能有连续两个红节点）
4. **叶节点**：所有 NIL 叶节点（空节点）是黑色
5. **黑高一致**：从任一节点到其所有叶节点的路径，黑色节点数相同

**推论**：规则 3 + 5 共同保证最长路径（红黑交替）≤ 最短路径（全黑）× 2，因此树高 ≤ 2log(n+1)。

### 插入与删除的自平衡

新节点着红色插入后，若违反规则，通过以下操作修复：

| 场景 | 操作 |
|------|------|
| 叔叔节点是红色 | 父+叔变黑，祖父变红，上移递归 |
| 叔叔黑，新节点在内侧 | 先旋转（内→外），再进入下一场景 |
| 叔叔黑，新节点在外侧 | 父变黑+祖父变红+单旋祖父 |

删除操作类似，删除黑色节点后需补充黑高。

### 与 AVL 树对比

| 维度 | 红黑树 | AVL 树 |
|------|--------|--------|
| 平衡程度 | 近似（高度差可达 2×）| 严格（高度差 ≤ 1）|
| 查询性能 | 略低 | 略高 |
| 插入/删除旋转次数 | 最多 3 次 | 可能需要多次 |
| 适合场景 | 写多读少 | 读多写少 |

## 三、核心使用原则

1. **不追求完全平衡**：牺牲部分查询常数，换取写操作更低的旋转开销
2. **额外内存**：每个节点仅多 1 bit（颜色），比 AVL 的高度字段更省
3. **实现复杂**：5条规则 + 多种插入/删除场景，正确实现难度较高

## 四、与其他概念的关系

- 比较于 [[机制-B+树]]：红黑树是二叉树，适合内存；B+树是多叉树，适合磁盘IO
- 支撑了 L4 集合框架：`HashMap`（JDK 8+）桶链表长度 ≥ 8 时转为红黑树；`TreeMap`/`TreeSet` 底层是红黑树
- 区别于 [[概念-Java集合]]：红黑树是非线性结构，查找利用有序性

## 五、应用边界

**适合**：
- 关联数组（`TreeMap`）：需要有序遍历或范围查询
- `HashMap` 内部桶结构：应对哈希冲突退化
- 操作系统进程调度（CFS 调度器用红黑树按虚拟运行时间排序）

**不适合**：
- 磁盘存储索引：树高 O(log n) 对磁盘 IO 仍过高，应用 B+树
- 只需查最大/最小值：用堆更高效
