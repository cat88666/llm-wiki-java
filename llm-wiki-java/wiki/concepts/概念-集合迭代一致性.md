# 集合迭代一致性

> 集合迭代一致性描述遍历过程中集合被修改时，迭代器选择抛异常、读快照，还是提供弱一致视图。

## 第一性原理

遍历要求“当前元素”和“下一个元素”关系稳定，而结构性修改会改变这种关系。集合必须选择一种策略：立即暴露风险（fail-fast）、隔离修改影响（fail-safe / 快照），或在并发下接受弱一致性。

## 核心机制

1. **fail-fast**：普通集合迭代器保存 `expectedModCount`，集合保存 `modCount`；结构修改绕过迭代器时，两者不一致就抛 `ConcurrentModificationException`。
2. **foreach 本质**：增强 for 依赖 `Iterator`，遍历中直接调用集合的 `add/remove` 容易触发 fail-fast。
3. **正确删除方式**：使用 `Iterator.remove()`、倒序索引遍历、`removeIf`，或用 Stream/filter 生成新结果。
4. **fail-safe / 快照**：Copy-On-Write 容器遍历的是创建迭代器时的数组快照，不感知后续修改。
5. **弱一致遍历**：`ConcurrentHashMap` 遍历不抛并发修改异常，但不承诺看到全量最新数据。

## 关键权衡

- **fail-fast 是错误检测，不是并发控制**：它只能尽早暴露不安全修改，不能保证线程安全。
- **快照读避免异常但可能过期**：遍历稳定性来自拷贝或弱一致语义，读到的可能不是最新状态。
- **subList 是视图不是拷贝**：子列表和父列表共享底层数据，父列表结构性修改可能让子列表操作抛异常。
- **修复方式要匹配语义**：想原地删除用迭代器/removeIf；想保留原集合用 Stream/filter 新建结果。

## 与其他概念的关系

- 依赖 [[概念-Java集合框架]]：不同集合实现选择不同迭代策略。
- 支撑 [[机制-CopyOnWrite]]：快照迭代是 Copy-On-Write 容器的核心表现。
- 关联 [[机制-ConcurrentHashMap]]：ConcurrentHashMap 采用弱一致迭代，避免遍历与写入互相阻塞。

## 应用边界

**适合关注**：
- 遍历中删除、过滤或追加元素
- 多线程同时读写集合
- `subList` 结果参与后续修改

**不应误用**：
- 不要把 `ConcurrentModificationException` 等同于“一定发生多线程并发”
- 不要把 fail-safe 理解为强一致；它通常只是避免异常
