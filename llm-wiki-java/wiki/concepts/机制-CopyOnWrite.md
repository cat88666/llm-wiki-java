# Copy-On-Write

> Copy-On-Write 是一种读写分离策略：读共享当前快照，写入时复制新副本并在修改完成后替换引用。

## 第一性原理

当读远多于写时，让每次读都加锁会浪费大量并发能力。Copy-On-Write 反过来让读无锁、写承担复制成本，用空间和写放大换取读路径稳定、简单和高并发。

## 核心机制

1. **读共享快照**：读操作直接访问当前数组引用，不加锁。
2. **写前复制**：`add/remove/set` 先复制底层数组，在新数组上修改。
3. **写操作加锁**：写路径需要加锁，避免多个线程同时复制并覆盖彼此结果。
4. **引用替换发布**：新数组构建完成后替换原数组引用，后续读看到新快照。
5. **迭代器快照**：迭代器持有创建时的数组引用，遍历期间不受后续写入影响。

## 关键权衡

- **读性能高，写成本高**：每次写都复制数组，写多时内存和 CPU 开销明显。
- **最终一致而非强一致**：读线程可能读到旧快照。
- **迭代稳定但不支持迭代器修改**：遍历不会抛 fail-fast 异常，但迭代器不适合做原地删除。
- **不同于同步容器**：`Vector` 读写互斥；Copy-On-Write 让读写操作面向不同数组。

## 与其他概念的关系

- 支撑 [[概念-集合迭代一致性]]：Copy-On-Write 是 fail-safe / 快照遍历的典型实现。
- 依赖 [[机制-synchronized]]：写操作需要锁保护复制和引用替换。
- 关联 [[概念-JMM]]：引用替换后的可见性决定其他线程何时看到新快照。

## 应用边界

**适合**：
- 白名单、黑名单、配置快照、监听器列表等读多写少场景
- 遍历稳定性比实时性更重要的场景

**不适合**：
- 高频写入、大集合写入
- 需要读到最新写入的强一致场景
- 大对象数组，复制成本过高
