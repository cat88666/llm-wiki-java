# 引用类型

> Java 提供四种强度递减的引用类型——强/软/弱/虚，让应用代码能参与 GC 决策：强引用永不回收，软引用在 OOM 前回收，弱引用每次 GC 必回收，虚引用不影响生命周期仅用于回收通知。

## 第一性原理

强引用语义下，应用层无法主动告诉 GC"这个对象低优先级，内存紧张时可以回收"。引用类型的存在理由：**在"内存占用"和"访问性能"之间给应用层一个可调节的旋钮**——软引用是缓存的理想载体，弱引用是防内存泄漏的工具。

## 核心机制

### 四种引用对比

| 引用类型 | 创建方式 | GC 回收时机 | 获取对象 | 典型用途 |
|---------|---------|------------|---------|---------|
| **强引用** | `Object o = new Object()` | 永不（可达时）| 直接访问 | 所有普通对象 |
| **软引用** | `new SoftReference<>(obj)` | **内存不足（OOM 前）**| `.get()`（可能返回 null）| 内存敏感缓存 |
| **弱引用** | `new WeakReference<>(obj)` | **每次 GC 必回收**（只有弱引用时）| `.get()`（可能返回 null）| 防内存泄漏映射 |
| **虚引用** | `new PhantomReference<>(obj, queue)` | GC 时（不影响生命周期）| 始终返回 null | 回收后的资源清理通知 |

**强度排序**：强 > 软 > 弱 > 虚

### Java 内置使用案例

- **软引用**：Guava Cache 的 `softValues()`，内存充足时保留缓存，内存紧张时自动释放
- **弱引用**：`WeakHashMap`（Key 是弱引用，Key 不被强引用时 Entry 自动清除）；`ThreadLocal` 中 `ThreadLocalMap` 的 Key 是弱引用（防止 ThreadLocal 对象本身的内存泄漏）
- **虚引用**：`DirectByteBuffer` 使用虚引用 + `ReferenceQueue` 在堆外内存对象被 GC 后清理本地内存

### `ReferenceQueue`

软引用、弱引用、虚引用均可配合 `ReferenceQueue` 使用：对象被 GC 后，对应的 Reference 对象会被放入队列，应用可轮询队列做后续处理（如统计、资源清理）。

## 关键权衡

1. **软引用缓存的边界问题**：JVM 并未规定"内存不足多少时回收软引用"，不同 JVM 实现策略不同，不能将软引用缓存当作完全可控的 LRU
2. **ThreadLocal 内存泄漏**：Key 是弱引用（ThreadLocal 对象会被回收），但 Value 是强引用。若线程池线程复用，Value 不会被回收，产生泄漏 → **必须在 `finally` 中 `remove()`**
3. **弱引用不保证即时回收**：GC 不一定在每次 Minor GC 时都扫描弱引用，取决于 JVM 实现

## 与其他概念的关系

- 软/弱/虚引用是对 [[机制-GC算法与垃圾收集器]] 中可达性分析规则的补充——弱引用对象"弱可达"，GC 将其视为死亡
- 虚引用依赖 [[机制-JVM内存模型]] 中的堆外内存（DirectByteBuffer）清理场景
- ThreadLocal 弱引用设计与 L3 并发：`ThreadLocal` 的 Key 用弱引用，是并发编程中防止 ClassLoader 泄漏的关键

## 应用边界

**软引用适合**：本地内存缓存（内存充足时用，不足时自动释放，比手动管理更优雅）

**弱引用适合**：
- 需要追踪对象但不想阻止其被 GC 的场景（`WeakHashMap`）
- 线程本地存储的 Key（ThreadLocal 的设计选择）

**虚引用适合**：底层资源清理（直接内存、文件句柄），替代 `finalize()`（后者已废弃，性能差）

**不要用**：软/弱引用替代合理的缓存淘汰策略（Caffeine/Guava 的时间/容量淘汰更可控）
