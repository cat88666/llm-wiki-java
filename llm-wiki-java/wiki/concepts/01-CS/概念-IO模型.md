# IO模型

> IO 模型描述应用程序、操作系统内核与硬件设备之间如何协作完成数据读写，核心关注就绪通知、数据拷贝和线程调度成本。

## 一、第一性原理

一次 IO 通常分两段：

1. **等待就绪**：磁盘、网卡或内核缓冲区是否已有可读/可写数据。
2. **数据拷贝**：数据在内核空间和用户空间之间移动。

所以判断一个 IO 模型，本质看三个问题：

- 等待阶段线程是否阻塞。
- 拷贝阶段由应用线程做，还是由内核完成后通知。
- 一个线程能否管理多个连接。

Java 后端优化 IO 的核心目标是：减少阻塞线程、减少系统调用、减少内核态/用户态切换、减少数据拷贝。

## 二、核心机制

### 同步、异步、阻塞、非阻塞

| 维度 | 关注点 | 典型含义 |
| --- | --- | --- |
| 阻塞 / 非阻塞 | 等待数据就绪时，线程是否被挂起 | `read()` 没数据时阻塞等待，或立即返回 |
| 同步 / 异步 | 数据拷贝由谁完成 | 应用线程自己读写，或内核完成后回调/通知 |

常见误区：NIO 的 `Non-blocking` 主要解决“等待阶段不阻塞”，不等于完整异步。Java NIO 读写数据时，应用线程仍要参与拷贝。

### 五类 OS IO 模型

| 模型 | 等待阶段 | 拷贝阶段 | Java/工程视角 |
| --- | --- | --- | --- |
| 阻塞 IO | 阻塞 | 应用线程拷贝 | 简单，但连接多时线程成本高 |
| 非阻塞 IO | 立即返回 | 应用线程拷贝 | 需要轮询，单独使用会浪费 CPU |
| IO 多路复用 | 阻塞在 selector/epoll | 应用线程拷贝 | 一个线程监听多个 fd，是 Java NIO/Netty 的基础 |
| 信号驱动 IO | 信号通知就绪 | 应用线程拷贝 | Java 后端少直接使用 |
| 异步 IO | 不阻塞 | 内核完成拷贝后通知 | 理论模型清晰，Linux/Java 生态中使用面有限 |

### select、poll、epoll

| 实现 | fd 限制 | 就绪发现方式 | 复杂度 | 结论 |
| --- | --- | --- | --- | --- |
| select | 通常 1024 | 每次复制 fd 集合并线性扫描 | O(n) | 老模型，规模受限 |
| poll | 无固定 1024 限制 | 链表保存 fd，仍线性扫描 | O(n) | 解决数量限制，不解决扫描成本 |
| epoll | 无固定 1024 限制 | 注册一次，事件就绪回调到就绪队列 | 近似 O(1) | Linux 高并发网络 IO 主流方案 |

epoll 只说明“哪些 fd 就绪”，不负责业务处理。应用仍要读取、解码、执行业务逻辑、写回响应。

epoll 触发模式：

| 模式 | 含义 | 优点 | 风险 |
| --- | --- | --- | --- |
| LT 水平触发 | 只要 fd 仍可读/可写，就会持续通知 | 编程简单，不容易漏事件 | 通知次数更多 |
| ET 边缘触发 | 只有状态变化时通知一次 | 通知少，吞吐更高 | 必须非阻塞并一次读/写到 `EAGAIN`，否则可能漏读 |

Netty 原生 epoll transport 可使用 ET，但前提是框架和业务代码正确处理非阻塞读写。

### Reactor 与 Proactor

| 模型 | 谁等待就绪 | 谁执行 IO 拷贝 | 典型对应 |
| --- | --- | --- | --- |
| Reactor | 事件分发器等待 fd 就绪 | 应用线程读写 | Java NIO、Netty、Redis、Nginx |
| Proactor | 内核负责异步 IO 完成 | 内核完成后通知应用 | Windows IOCP、Java AIO 的理想模型 |

Java 服务端主流是 Reactor，而不是 Proactor。原因是 Linux 下网络 AIO 生态长期不如 epoll/NIO 成熟，Netty 选择 Reactor 模型更可控。

### Page Cache 与零拷贝

| 机制 | 解决什么 | 关键点 | 典型场景 |
| --- | --- | --- | --- |
| Page Cache | 减少真实磁盘 IO | 读命中直接返回；写先进入脏页，后续刷盘 | MySQL、Kafka、RocketMQ、Redis AOF |
| mmap | 减少用户态拷贝 | 文件映射到进程地址空间 | RocketMQ CommitLog |
| sendfile | 文件直接从内核缓冲区发到 socket | 少一次用户态拷贝，少上下文切换 | Kafka、Nginx 静态文件 |
| DirectBuffer | 减少 JVM 堆内拷贝 | 堆外内存适合网络 IO，但分配释放更贵 | Netty、Java NIO |

零拷贝不是“没有拷贝”，而是减少 CPU 参与的无效拷贝，尤其减少内核空间和用户空间之间的来回搬运。

## 三、Java 核心使用

| Java 模型 | 核心 API | 线程模型 | 适用场景 | 不适合 |
| --- | --- | --- | --- | --- |
| BIO | `java.io`、`Socket`、`ServerSocket` | 通常一连接一线程 | 连接少、逻辑简单、阻塞可接受 | 长连接、高并发 |
| NIO | `Channel`、`Buffer`、`Selector` | 少量 selector 线程管理大量连接 | 网关、RPC、IM、MQ、Netty | 编程复杂，业务阻塞会拖垮事件循环 |
| AIO | `AsynchronousSocketChannel` | 回调/ Future | Windows IOCP 场景更合适 | Linux Java 服务端主流场景 |

Java NIO 核心流程：

1. `Channel` 设置为 non-blocking。
2. `Channel` 注册到 `Selector`，声明关注 `OP_ACCEPT`、`OP_READ`、`OP_WRITE`。
3. `Selector.select()` 阻塞等待就绪事件。
4. 遍历 `selectedKeys`，按事件类型 accept/read/write。
5. 用 `Buffer` 承载数据，读写前后注意 `flip()`、`clear()`、`compact()`。

NIO 难点不在 API 名字，而在半包/粘包、写缓冲、状态机、连接生命周期和异常关闭。

Java 后端主流选择：

- 普通业务系统：通常通过 Tomcat/Jetty/Undertow 屏蔽底层 IO。
- 高并发网络框架：优先 Netty NIO + epoll/kqueue 原生传输。
- 文件传输：优先利用 `FileChannel.transferTo()`、`sendfile`、对象存储直传等能力。
- Java 21 后，普通阻塞式业务可考虑虚拟线程；高性能网络框架仍通常使用 NIO/Netty。

### Netty 线程模型

| 组件 | 职责 | 注意点 |
| --- | --- | --- |
| BossGroup | 接收新连接 | 通常线程数较少 |
| WorkerGroup | 处理连接上的读写事件 | 不要执行阻塞业务 |
| EventLoop | 一个线程绑定多个 Channel，串行处理事件 | 同一 Channel 内避免并发锁竞争 |
| ChannelPipeline | 组织编解码、协议处理、业务 handler | 阻塞 handler 要切到业务线程池 |

Netty 的关键不是“用了 NIO 所以快”，而是把连接事件、buffer 管理、编解码、线程模型和背压控制工程化封装。

## 四、核心使用原则

1. **不要在事件循环线程做阻塞操作**：Netty handler 中不要直接查数据库、调用慢接口、执行大计算。
2. **连接多但单连接低活跃**：适合 NIO/多路复用，例如网关、长连接、IM。
3. **请求少但逻辑重**：BIO 或普通 servlet 线程池未必差，瓶颈常在 DB/业务逻辑。
4. **磁盘文件传输**：优先考虑零拷贝，不要把大文件完整读入 JVM 堆。
5. **刷盘语义要单独设计**：`write()` 成功不等于落盘成功，强一致场景要关注 `fsync()`、WAL、复制确认。
6. **区分网络 IO 和文件 IO**：epoll 适合网络 fd 就绪通知；普通磁盘文件通常一直 ready，性能重点在 Page Cache、顺序写、刷盘和存储硬件。

## 五、虚拟线程的影响

Java 21 虚拟线程降低了阻塞式编程的线程成本，让“一请求一虚拟线程”的写法重新具备吸引力。

| 维度 | 虚拟线程 | NIO/Netty |
| --- | --- | --- |
| 编程模型 | 接近同步阻塞代码 | 事件驱动、回调/ pipeline |
| 适合场景 | 普通业务服务、外部 RPC/DB 等等待型任务 | 网关、长连接、协议框架、极致网络吞吐 |
| 解决问题 | 降低阻塞等待占用平台线程的成本 | 少量线程管理大量连接和网络事件 |
| 不能解决 | 数据库慢、锁竞争、CPU 计算重、下游限流 | 业务阻塞、背压失控、内存膨胀 |

结论：虚拟线程会弱化普通业务对异步化的需求，但不替代 Netty 这类高性能网络框架。

## 六、使用案例

| 场景 | 推荐模型/机制 | 原因 |
| --- | --- | --- |
| Netty RPC 服务 | NIO + epoll + 业务线程池 | 事件循环负责网络，业务线程池隔离耗时逻辑 |
| API 网关 | NIO/事件驱动 | 大量连接和转发，线程不能随连接数线性增长 |
| Kafka 文件发送 | Page Cache + sendfile | 数据从磁盘页缓存直发 socket，减少用户态拷贝 |
| RocketMQ CommitLog | Page Cache + mmap | 顺序写日志，利用内存映射提高吞吐 |
| Redis 网络事件 | epoll + 单线程事件循环 | IO 多路复用降低线程切换，命令执行保持串行 |
| Tomcat 普通 Web 应用 | NIO connector + 工作线程池 | 网络 IO 和 servlet 业务处理分离 |

## 七、综合对比

| 方案 | 优点 | 代价 | 面试/设计结论 |
| --- | --- | --- | --- |
| BIO | 模型简单、调试容易 | 线程数随连接数增长，内存和上下文切换成本高 | 小规模可以用，高并发长连接不适合 |
| NIO | 少量线程管理大量连接 | 编码复杂，需要处理半包、粘包、状态机、背压 | Java 高并发网络编程主流基础 |
| AIO | API 表达完整异步 | Linux 下生态和性能优势不突出 | Java 服务端很少作为首选 |
| 多路复用 | 提高连接管理效率 | 不提升业务 CPU 处理能力 | 解决“等 IO”的问题，不解决“业务慢”的问题 |
| 零拷贝 | 降低 CPU 拷贝和上下文切换 | 适用边界明确，通常偏文件/网络传输 | 大文件和日志型系统收益明显 |

## 八、生产风险

| 风险 | 表现 | 处理原则 |
| --- | --- | --- |
| 事件循环阻塞 | 少量慢请求拖慢大量连接 | 慢业务切线程池，监控 event loop 延迟 |
| 写缓冲堆积 | 下游慢、客户端慢导致内存上涨 | 做高低水位、限流、断开慢连接、背压 |
| DirectBuffer 泄漏 | `OutOfMemoryError: Direct buffer memory` | 池化、引用计数释放、限制堆外内存 |
| 半包/粘包 | 协议解析错乱 | 明确帧格式：长度字段、分隔符、固定长度或协议编解码器 |
| 刷盘误判 | `write()` 成功但宕机丢数据 | 明确 `fsync`、WAL、主从复制确认语义 |

## 九、与其他概念的关系

- **[[概念-操作系统]]**：IO 涉及系统调用、用户态/内核态切换、Page Cache 和线程调度。
- **[[机制-线程池]]**：BIO 依赖线程池兜底；NIO 也需要业务线程池隔离阻塞任务。
- **[[机制-Netty]]**：Netty 是 Java NIO 的工业级封装，处理 selector、buffer、pipeline、线程模型和背压。
- **[[概念-计算机网络]]**：网络协议解决数据如何传输，IO 模型解决应用如何高效收发。

## 十、应用边界

- IO 模型不是性能银弹。数据库慢、锁竞争、序列化重、业务 CPU 高，都不是单靠 NIO 能解决的。
- epoll 优化网络 fd 就绪通知，对普通磁盘文件意义有限；磁盘性能要看 Page Cache、顺序写、刷盘策略和存储硬件。
- 零拷贝适合文件到 socket、堆外 buffer 等特定路径，不适合需要复杂业务加工的数据。
