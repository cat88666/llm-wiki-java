# OOP三大特征

> 封装隐藏实现、继承复用行为、多态统一接口——三者共同使复杂系统可以分层抽象和独立演化。

## 第一性原理

面向过程：把问题分解为步骤，关注"怎么做"。  
面向对象：把问题分解为对象，关注"谁来做"，对象持有状态和行为。

OOP 的存在理由：**让大规模系统通过接口契约而非代码细节耦合，模块可以独立变化**。

## 核心机制

### 封装（Encapsulation）

把数据和操作数据的方法绑定在类中，隐藏内部实现细节。

```java
class BankAccount {
    private double balance;  // 外部不能直接访问
    public void deposit(double amount) { ... }  // 通过方法操作
}
```

**意义**：调用方只依赖接口，不依赖实现 → 内部重构不影响外部。

### 继承（Inheritance）

子类复用父类的属性和方法，并可扩展或重写。

**Java 不支持多继承**（只能继承一个类）：防止"菱形问题"——两个父类有同名方法时，子类不知道继承哪个。Java 用**接口（多实现）+ 默认方法**来弥补。

**组合优于继承**（经典原则）：
- 继承是强耦合（子类暴露在父类实现细节下）
- 组合更灵活（持有其他类的引用，运行期可替换）
- 继承适合"is-a"关系；组合适合"has-a"关系

### 多态（Polymorphism）

同一个接口，不同对象有不同实现，调用方不关心具体类型。

```java
Animal animal = new Dog();  // 父类引用指向子类对象
animal.speak();             // 运行期调用 Dog.speak()，而非 Animal.speak()
```

**实现机制**：方法重写（Override）+ 虚方法表（vtable）。JVM 在运行期根据对象实际类型查表决定调用哪个方法——这是动态派发（Dynamic Dispatch）。

### 接口 vs 抽象类

| | 接口（Interface）| 抽象类（Abstract Class）|
|--|---------|----------|
| 继承限制 | 可实现多个 | 只能继承一个 |
| 字段 | 只有常量（`public static final`）| 可有实例字段 |
| 方法 | 默认抽象；JDK 8+ 支持 `default`/`static` | 可有具体方法 |
| 语义 | **能力契约**（实现方说"我能做某事"）| **部分实现**（定义共同行为骨架）|

**选择规则**：
- 定义能力/角色 → 接口（`Runnable`、`Comparable`、`Serializable`）
- 有公共实现逻辑需要共享 → 抽象类（`AbstractList`）
- 两者都能用时 → 优先接口（更灵活，支持多实现）

## 关键权衡

1. **继承层次不要超过 3 层**：越深越难理解，重构越痛苦
2. **多态需要设计代价**：虚方法表查找比直接调用略慢（极少成为瓶颈）
3. **接口爆炸**：过度设计接口导致类图复杂；接口应聚焦单一能力（接口隔离原则）

## 与其他概念的关系

- 支撑了 [[机制-动态代理]]：代理模式依赖多态，代理类和目标类共同实现同一接口
- 与 [[机制-泛型类型擦除]] 协作：泛型的上下界通配符（`<? extends T>`）建立在继承关系上
- 支撑了 L6 Spring IoC：`ApplicationContext` 通过接口多态管理 Bean，不感知具体实现类

## 应用边界

**封装**：永远应该做，字段 private，方法 public/protected。

**继承**：谨慎用，优先考虑组合；适合框架设计中定义骨架（Template Method 模式）。

**多态**：是面向对象的核心价值，策略模式、工厂模式、模板方法均依赖多态。
