# 网络安全

> Web/分布式系统的安全防护，通过加密、鉴权、访问控制等机制保护数据的**机密性（Confidentiality）、完整性（Integrity）和可用性（Availability）**（CIA 三角）。

## 第一性原理

所有 Web 安全漏洞都源于两个根本问题：**不可信的输入**（攻击者构造恶意数据）和**不可信的调用方**（攻击者伪造身份或权限）。防御的本质是在信任边界处对输入做过滤/编码，对调用方做身份和权限验证。

---

## 核心机制

### 密码学基础

**MD5 不是加密算法**，是哈希（消息摘要）算法：单向、不可逆、存在碰撞漏洞。不能用于密码存储（彩虹表秒破），应使用 SHA-256 或加盐的 bcrypt/Argon2。

| 操作 | 目的 | 使用密钥 | 典型算法 |
|------|------|---------|---------|
| **加密/解密** | 保护数据机密性（只有持有密钥者能读取）| ✅ | AES（对称）/RSA（非对称）|
| **加签/验签** | 保证数据完整性 + 来源可信（私钥签名，公钥验证）| ✅ 私钥 | RSA+SHA-256 / ECDSA |
| **哈希** | 生成摘要（不可逆，用于校验/存储）| ❌ | MD5（弱）/ SHA-256 / SM3 |

**加签流程**：发送方用私钥对消息哈希签名 → 接收方用公钥验签，确认消息未被篡改且来自合法发送方。

### 注入攻击：SQL 注入

**攻击原理**：通过在用户输入中拼接 SQL 语句控制数据库行为。
```sql
-- 恶意输入: ' OR '1'='1
SELECT * FROM users WHERE name='' OR '1'='1'  -- 查出全表
```

**为什么预编译（PreparedStatement）能防御**：占位符 `?` 先编译 SQL 结构，参数后传入时被视为纯数据而非 SQL 代码，无论输入什么内容都不会改变 SQL 结构。

```java
// 安全：参数不参与 SQL 解析
PreparedStatement ps = conn.prepareStatement("SELECT * FROM users WHERE name=?");
ps.setString(1, userInput);  // ' OR '1'='1 被当作字符串字面量

// MyBatis: #{} 预编译（安全），${} 字符串拼接（不安全）
```

**防御层次**：MyBatis `#{}` / JPA / ORM 框架（应用层）→ 最小权限原则（数据库层）→ WAF 规则过滤（网络层）。

### 跨站攻击：XSS 与 CSRF

**XSS（跨站脚本）**：攻击者将恶意脚本注入到页面，用户浏览时脚本在受害者浏览器执行（窃取 Cookie、劫持操作）。
- 防御：**输出编码**（`<` → `&lt;`）+ Content-Security-Policy（CSP）响应头 + HttpOnly Cookie（脚本无法读取）

**CSRF（跨站请求伪造）**：利用用户已登录的 Cookie，诱导用户访问恶意页面触发伪造请求（如转账）。
- 防御核心：**服务端无法区分来源**，因此需加入攻击者无法获取的凭证：
  1. **CSRF Token**：随机 Token 写入页面表单，请求时携带，服务端校验
  2. **SameSite Cookie**：`SameSite=Strict/Lax` 阻止第三方站点携带 Cookie
  3. **Referer 校验**：检查请求来源域名（可被绕过，作为辅助）

| | XSS | CSRF |
|--|-----|------|
| 攻击目标 | 注入脚本在用户浏览器执行 | 利用用户身份发送伪造请求 |
| 利用的漏洞 | 缺少输出编码 | 仅靠 Cookie 做身份凭证 |
| 核心防御 | 输出编码 + CSP | CSRF Token / SameSite |

### 越权访问

**垂直越权**：低权限用户访问高权限资源（普通用户调管理员接口）。
- 防御：**RBAC 角色模型**（Role-Based Access Control）+ 接口级鉴权（Spring Security / Sa-Token），拒绝最小权限之外的所有操作。

**水平越权**：同权限用户访问他人数据（用户 A 查看用户 B 的订单）。
- 防御：**从 Session/Token 取当前 userId，与数据库记录的 owner_id 比对**，不信任请求参数中的 userId。
- 补充：对外展示的资源 ID 用 **Sqids/HashId 混淆**（`123` → `abc45xyz`），让 ID 不可猜测，阻断水平枚举攻击。

```java
// 错误：信任请求参数中的 userId
Order order = orderService.getById(request.getUserId(), orderId);

// 正确：从 Session 获取当前用户，防水平越权
Long currentUserId = SecurityContextHolder.getContext().getUserId();
Order order = orderService.getById(orderId);
if (!order.getUserId().equals(currentUserId)) throw new ForbiddenException();
```

### 中间人攻击（MITM）

攻击者在通信双方之间截获并篡改数据，常见形式：
- **Wi-Fi 欺骗**：伪造热点，拦截明文 HTTP 流量
- **SSL 劫持**：伪造证书，降级或替换 HTTPS 握手
- **DNS 劫持/污染**：将域名解析到恶意 IP

**防御**：
- HTTPS（TLS 证书验证服务端身份）
- **HSTS**（HTTP Strict Transport Security）：强制浏览器只用 HTTPS 访问
- 证书固定（Certificate Pinning）：客户端内置指定证书指纹，拒绝其他证书
- DNSSEC / DNS over HTTPS：验证 DNS 响应完整性

### 数据安全：撞库/拖库/洗库

| 攻击阶段 | 描述 | 防御 |
|---------|------|------|
| **撞库** | 用其他网站泄露的账号密码，在目标网站批量尝试登录 | 登录频率限制 + IP 封禁 + 二次验证（MFA）|
| **拖库** | 攻击者获取数据库访问权限，导出全量数据 | 数据库最小权限 + 敏感数据加密存储（bcrypt 密码）|
| **洗库** | 将拖走的数据变现（出售/撞库利用）| 字段脱敏（手机号 138****1234）+ 访问审计日志 |

**密码存储最佳实践**：bcrypt/Argon2 加盐哈希，绝不存明文或 MD5；数据库里的密码哈希即便泄露也难以还原。

---

### DDoS（分布式拒绝服务）

攻击者通过大量"肉鸡"（被控设备）同时发包，耗尽目标带宽/连接数/CPU，使服务不可用。

**防御层次**：
1. **网络层**：ACL/黑名单过滤恶意 IP；增加带宽冗余
2. **传输层**：SYN Cookie 防 SYN Flood；连接数限制
3. **应用层**：限流（令牌桶）；WAF 过滤异常请求特征
4. **调度层**：BGP Anycast + CDN 分散流量；云服务商 DDoS 清洗中心

---

### 国密算法

中国国家密码局标准，替代 RSA/AES/SHA-256 的国产体系：

| 算法 | 类型 | 等价国际标准 | 用途 |
|------|------|------------|------|
| **SM4** | 对称加密（128 位分组）| AES-128 | 数据加密、TLS 国密版 |
| **SM2** | 非对称（ECC 256 位）| RSA/ECDSA | 数字签名、密钥交换 |
| **SM3** | 哈希（256 位输出）| SHA-256 | 消息摘要、数字签名前哈希 |
| **SM9** | 标识密码（IBE）| 无直接等价 | 以邮箱/手机号为公钥，无需证书 |

国密要求：金融、政务系统通常必须使用国密算法；SM2 比 RSA 密钥更短、计算更快。

## 关键权衡

1. **加密算法强度 vs 性能**：非对称加密（SM2/RSA）计算代价是对称加密（SM4/AES）的 100 倍以上，实际方案是非对称加密用于密钥交换，对称加密用于数据传输（TLS 工作原理）
2. **CSRF Token vs 无状态**：CSRF Token 需要服务端维护状态或客户端存储，JWT 方案通过 `Authorization` 请求头传递（而非 Cookie）天然绕开 CSRF 问题
3. **安全 vs 用户体验**：过于严格的验证（频繁 MFA、过短 Session 过期）降低用户体验；需根据业务敏感度分级
4. **防御深度**：不能依赖单一防线（应用层过滤），需要多层防御（WAF + 参数校验 + 最小权限 + 审计日志）

## 与其他概念的关系

- **依赖 [[概念-幂等设计]]**：防重放攻击（同一请求发两次）需要 nonce + 幂等 Token 机制，与业务幂等思路相通
- **依赖 [[机制-Redis分布式锁]]**：限流（令牌桶）、防刷（IP 计数）通常用 Redis + Lua 脚本实现，避免并发计数不准
- **相关 [[机制-微服务与SpringCloud]]**：微服务网关（Spring Cloud Gateway）是做统一鉴权、限流、WAF 规则的核心位置

## 应用边界

**必须做的场景**：凡有用户输入的接口（防注入）；凡存储用户密码/敏感数据（防拖库）；凡涉及资金/权限变更（防越权/CSRF）；政务/金融系统（国密合规）。

**不过度设计的场景**：内部调试接口/全内网服务可适当放宽；纯读取的公开 API 不必强 CSRF Token。
