# 类加载机制

> JVM 通过类加载器将 `.class` 字节码加载进内存并初始化，双亲委派模型保证核心类（如 `java.lang.Object`）的唯一性与安全性——每次加载请求先委派给父加载器，父加载器找不到才由子加载器自己加载。

## 第一性原理

Java 是动态加载的语言——类只在第一次被使用时才加载，而非程序启动时全部加载。核心问题：如何保证 `java.lang.Object` 在所有类加载器环境下都是同一个类？双亲委派的存在理由：**优先级层次关系保证基础类由最顶层的启动类加载器加载，防止用户伪造核心类（如自定义 `java.lang.Object`）**。

## 核心机制

### 类的生命周期（7 个阶段）

```
加载（Loading）
  → 验证（Verification）
  → 准备（Preparation）    ← 链接
  → 解析（Resolution）     ←
  → 初始化（Initialization）
  → 使用（Using）
  → 卸载（Unloading）
```

| 阶段 | 做什么 |
|------|--------|
| 加载 | 通过类名找到 .class 文件，读入字节数组，创建 `Class` 对象 |
| 验证 | 字节码合法性检查（文件格式、元数据、字节码语义）|
| 准备 | 为类的**静态变量**分配内存，赋**默认值**（int = 0，引用 = null）|
| 解析 | 将符号引用（类名字符串）替换为直接引用（内存地址）|
| 初始化 | 执行 `<clinit>`（静态代码块 + 静态变量赋值），**真正赋业务初值**|
| 卸载 | 该类所有实例被 GC + 该类的 ClassLoader 被 GC → FullGC 时从方法区回收 |

### 三种默认类加载器（JDK 8）

```
Bootstrap ClassLoader（启动类加载器）   ← C++ 实现，Java 中表示为 null
    加载：JRE/lib/rt.jar（Object, String 等）
    ↑ 父
Application ClassLoader（应用类加载器）
    加载：Classpath 下的类（自己写的代码）
    ↑ 父
Extension ClassLoader（扩展类加载器）
    加载：JRE/lib/ext/ 下的类
```

> JDK 9 后模块化，Extension ClassLoader 改名为 Platform ClassLoader。

### 双亲委派模型的执行逻辑

```java
// ClassLoader.loadClass 核心逻辑
protected Class<?> loadClass(String name, boolean resolve) {
    // 1. 先查缓存（是否已加载）
    Class<?> c = findLoadedClass(name);
    if (c == null) {
        // 2. 委派给父加载器
        if (parent != null)
            c = parent.loadClass(name, false);
        else
            c = bootstrapClassLoader.loadClass(name);
        // 3. 父加载器找不到，自己加载
        if (c == null)
            c = findClass(name);
    }
    return c;
}
```

**扩展点**：
- 重写 `findClass()`：自定义加载逻辑，**不破坏**双亲委派
- 重写 `loadClass()`：**破坏**双亲委派

### 破坏双亲委派的三种场景

| 场景 | 原因 | 实现方式 |
|------|------|---------|
| **SPI（如 JDBC）** | Bootstrap 加载器加载 `java.sql.Driver` 接口，但实现类在 Classpath 由 App ClassLoader 加载，父加载器无法"向下"找子类 | 线程上下文类加载器（`Thread.currentThread().getContextClassLoader()`）|
| **Tomcat** | 多个 webapp 需要隔离，各自有独立的 WebApp ClassLoader | 每个 webapp 自己的 ClassLoader 优先加载，隔离不同版本的类库 |
| **热部署（OSGi）** | 模块化，需要不同模块加载不同版本类 | 网状加载，而非树形委派 |

## 关键权衡

1. **`findClass` vs `loadClass`**：自定义类加载器通常只重写 `findClass`（保留双亲委派）；需要隔离时才重写 `loadClass`（打破委派）
2. **类隔离的代价**：破坏双亲委派可以实现隔离（不同 webapp 不同版本），但同一个 Class 由不同 ClassLoader 加载后是**不同的类**，类型转换会抛 `ClassCastException`
3. **类卸载条件苛刻**：JDK 自带类永不卸载；用户类需 ClassLoader + 所有实例均被 GC，FullGC 才能卸载

## 与其他概念的关系

- 加载后的类元数据存入 [[机制-JVM内存模型]] 中的方法区（元空间）
- 依赖 [[机制-反射]]：`Class.forName(name)` 就是触发类加载并返回 Class 对象
- 与 [[机制-SPI]] 强关联：SPI 用线程上下文类加载器加载实现类，是最典型的破坏双亲委派场景

## 应用边界

**需要自定义类加载器的场景**：
- 热部署（重新加载修改后的类）
- 加密 class 文件（自定义解密逻辑）
- 多版本类库隔离（如 Tomcat）

**元空间 OOM 的排查**：频繁创建类（动态代理、JSP 编译）而未卸载，通常是 ClassLoader 泄漏——ClassLoader 被 GC Root 引用导致无法回收。
