# B+树

> 为磁盘存储设计的多路平衡查找树——每个节点存多个键值，以极低的树高（通常 3-4 层）将磁盘 IO 次数控制在 O(log_m n) 内；B+树是 B 树的变体，所有数据仅存叶节点，叶节点形成链表，天然支持范围查询。

## 一、第一性原理

二叉树每次磁盘 IO 只能缩小一半搜索范围，在亿级数据时树高超过 30 层，意味着 30 次磁盘 IO。B 树的存在理由：**通过增大每个节点的分叉数（m 路），将树高压缩到 O(log_m n)，每层对应一次磁盘 IO**。

## 二、核心机制

### m 阶 B 树的约束

- 每个节点最多 m 个子节点、m-1 个键
- 非根非叶节点至少 ⌈m/2⌉ 个子节点（防止过于稀疏）
- 所有叶节点在同一层（高度一致）
- **数据存在所有节点上**（叶 + 非叶）

### B+ 树的两处改动

| 特性 | B 树 | B+ 树 |
|------|------|-------|
| 数据存储 | 叶节点 + 非叶节点 | **仅叶节点** |
| 非叶节点作用 | 存键值 + 数据 | **仅存键值**（路由） |
| 叶节点连接 | 独立，无指针 | **双向链表**，支持顺序扫描 |
| 范围查询 | 需要回溯 | 直接链表遍历 |
| 单点查询 | 可在非叶结束（快） | 必须到叶节点 |

### 为什么 MySQL InnoDB 选 B+ 树

1. **非叶节点不存数据**，单页能放更多路由键，树更矮，IO 次数更少
2. **叶节点链表**支持 `BETWEEN`、`ORDER BY`、范围扫描
3. 全表顺序扫描：只需遍历叶节点链表，不需要遍历整棵树

典型 InnoDB B+ 树：3 层即可支撑千万级数据（根节点常驻内存，只需 2 次磁盘 IO）。

## 三、核心使用原则

1. **B 树单点查询可能更快**：数据在非叶节点即可命中，而 B+ 树必须走到叶层
2. **B+ 树范围查询远优于 B 树**：链表遍历 vs 反复回溯
3. **B 树适合随机访问**：MongoDB 使用 B 树，文档数据库以点查为主
4. **节点分裂代价**：插入触发分裂时，需要更新父节点直至根，写性能低于哈希索引

## 四、与其他概念的关系

- 比较于 [[机制-红黑树]]：红黑树是二叉树，适合内存有序结构；B+树是多叉树，专为磁盘 IO 优化
- 支撑了 L5 存储层：MySQL InnoDB 索引（主键索引/二级索引）底层均为 B+ 树
- 支撑了 L5 存储层：B+ 树叶节点链表是 InnoDB 支持范围查询、覆盖索引的物理基础

## 五、应用边界

**B+ 树适合**：关系型数据库索引（MySQL InnoDB）；文件系统；需要范围查询或排序的场景。

**B 树适合**：文档数据库（MongoDB）；以单点随机查找为主的场景。

**不适合用 B/B+ 树**：内存中的有序结构（用红黑树/跳表，无磁盘 IO 成本）；等值哈希查询（哈希索引 O(1) 更快）。
