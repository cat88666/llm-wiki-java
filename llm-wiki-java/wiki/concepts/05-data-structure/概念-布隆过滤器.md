# 布隆过滤器

**一句话定义**：布隆过滤器是基于 BitMap + 多个哈希函数的概率性数据结构，能以极小的空间代价回答"元素一定不存在 / 可能存在"；布谷鸟过滤器在此基础上增加了指纹存储和双候选桶，解决了布隆过滤器无法删除的核心缺陷。

---

## 第一性原理

精确集合（HashSet）判断元素是否存在需要存储全量数据，内存代价与元素数量线性相关。布隆过滤器的核心洞察是：**很多场景只需要知道"一定不存在"，不需要知道"一定存在"**，只要能将误判率控制在可接受范围内，就可以用极少的 bit 替代大量内存。

---

## 核心机制

### 布隆过滤器原理

**数据结构**：一个长度为 m 的 bit 数组 + k 个独立哈希函数

**写入**：对元素 x，用 k 个哈希函数各计算出一个 [0, m-1] 的索引，将对应 bit 设置为 1

**查询**：对 x，计算 k 个索引，若所有对应 bit 都为 1 → 可能存在；只要有一个为 0 → 一定不存在

```
元素 "Hollis" → hash1→ bit[3]=1, hash2→ bit[17]=1, hash3→ bit[42]=1
查询 "Hero"   → bit[3]=1, bit[17]=1, bit[99]=0 → 一定不存在 ✓
查询 "Villain" → bit[3]=1, bit[17]=1, bit[42]=1 → 可能存在（哈希冲突的误判）
```

**误判率（False Positive Rate）**由 m（bit 数组大小）和 k（哈希函数数）决定：
- m 越大（bit 数组越稀疏），碰撞越少，误判率越低
- k 最优值 = (m/n) × ln2，其中 n 是元素数量
- 实际使用时传入**预期容量**和**目标误判率**，框架自动计算 m 和 k

**Java 实现**（Guava/Redis Redisson 均支持）：
```java
// Guava：容量100w，误判率0.01
BloomFilter<String> bf = BloomFilter.create(
    Funnels.stringFunnel(Charsets.UTF_8), 1_000_000, 0.01);
bf.put("Hollis");
bf.mightContain("Hollis");  // true（可能存在）
bf.mightContain("Hero");    // false（一定不存在）
```

---

### 布隆过滤器的缺陷

1. **无法删除**：一个 bit 位可能被多个元素的哈希共享，设置为 0 会误伤其他元素
2. **误判率随元素数量增加而升高**：超过预期容量后，bit 数组趋于饱和，误判率急剧上升
3. **不支持计数**：无法知道某个 key 被插入了多少次

**解决删除问题的三种方案**：

| 方案 | 思路 | 代价 |
|------|------|------|
| 定期重建 | 定时基于最新数据重建过滤器并原子切换 | 重建期间旧数据可能误判 |
| 计数布隆过滤器 | 每个 bit 位改为 3-4 bit 计数器，删除时减一 | 空间增加 300-400%，计数可能溢出（>15 次）|
| **布谷鸟过滤器** | 全新设计，存指纹而非 bit | 支持删除，误判率更低，但插入在极端情况下可能失败 |

---

### 布谷鸟过滤器

**核心设计**：不用 bit 数组，改为存储每个元素的**指纹**（短哈希，如 8 bit）；每个元素有**两个候选桶**，使用 XOR 关联两者：

```
i1 = hash(x) % N            // 与元素相关
i2 = i1 XOR hash(fingerprint(x)) % N  // 与指纹相关，XOR 保证 i1 和 i2 可互相推导
```

**XOR 的关键性质**：知道 i1 和指纹 f 就能算出 i2，反之亦然。这是删除安全性的基础。

**插入流程**：
1. 若 i1 或 i2 有空槽 → 存入指纹
2. 若两者都满 → 随机选一个桶，踢出已有指纹，被踢出者再找其另一个候选桶（名称来源：像布谷鸟把别的蛋踢出巢）
3. 踢出链过长（超过最大重试次数）→ 插入失败，需扩容

**删除流程**：计算 i1 和 i2，找到存有对应指纹的桶，将该槽位清空。因为精确匹配指纹值，误删概率极低。

**布隆 vs 布谷鸟对比**：

| 特性 | 布隆过滤器 | 布谷鸟过滤器 |
|------|-----------|------------|
| 查询误判 | 有（可能存在实为不存在）| 有（更低）|
| 删除 | ❌ 不支持 | ✅ 支持 |
| 空间效率 | 高（bit 级别）| 略低（存指纹）|
| 插入失败 | 不会（但误判率升高）| 极端情况会失败 |
| 实现复杂度 | 简单 | 较复杂 |
| Redis 支持 | ✅ RedisBloom 模块 | ✅ Redis 8.0 原生支持 |

## 关键权衡

1. **预期容量要留余量**：超过容量后误判率快速劣化，建议设置为实际容量的 1.5-2 倍
2. **误判率设置**：0.01（1%）是常见选择；降低到 0.001 需要更多 bit，内存增加约 50%
3. **布谷鸟 vs 布隆的选择**：需要删除操作时选布谷鸟；只需查询且数据量极大时布隆更省空间
4. **Redis 集群中的布隆过滤器**：需要同一 key 的所有操作落到同一节点（使用 hash tag `{key}`），否则 bit 数组被分片后查询结果不一致

---

## 与其他概念的关系

- **[[概念-BitMap]]**：布隆过滤器在 BitMap 基础上增加了多个哈希函数；BitMap 是确定性映射（1 bit = 1 个整数），布隆过滤器是概率性映射
- **[[概念-缓存三大问题]]**：布隆过滤器是解决缓存穿透的标准方案：将已存在的 key 加入过滤器，请求到来时先过滤器判断，一定不存在的直接拦截，不穿透到 DB
- **[[机制-Redis持久化]]**：Redis 的 RedisBloom 模块和布谷鸟过滤器存在 Redis 中，需要考虑持久化（RDB/AOF）保证重启后过滤器数据不丢失

---

## 应用边界

**适合场景**：
- 缓存穿透防护（最典型）：黑名单 URL/恶意 IP/不存在的商品 ID
- 海量数据去重判断（爬虫 URL 去重、用户行为去重）
- 分布式缓存中判断 key 是否存在（减少跨节点查询）

**不适合场景**：
- 需要精确判断"一定存在"（布隆只保证"一定不存在"）
- 元素数量远超预期（建议改用精确集合或增大容量重建）
- 需要频繁删除（用布谷鸟过滤器替代）
