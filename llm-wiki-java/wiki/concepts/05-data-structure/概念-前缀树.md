# 前缀树

> 一种多叉树，每条从根到节点的路径对应一个字符串前缀——以共享公共前缀节省存储，将字符串检索从 O(n×m) 降至 O(m)（m 为查询字符串长度）。

## 一、第一性原理

字符串集合的朴素存储（哈希表/数组）无法利用字符串间的公共前缀。Trie 的存在理由：**把字符串"竖着"展开存进树，公共前缀只存一份，同时把查找复杂度锁定在字符串自身长度 O(m)，与集合大小无关**。

## 二、核心机制

### 结构规则

- **根节点不含字符**
- 每个节点存一个字符，每条边表示一个字符的转移
- 从根到某节点的路径拼接 = 该节点对应的字符串
- 每个节点的所有子节点字符不同（即同层不重复）
- 节点通常有 `isEnd` 标志，标记该路径是否构成一个完整单词

### 时间复杂度

| 操作 | 复杂度 | 说明 |
|------|--------|------|
| 插入 | O(m) | m = 字符串长度，逐字符建节点 |
| 查找 | O(m) | 逐字符沿路径走，O(1)/步 |
| 前缀匹配 | O(m) | 查到前缀节点后，遍历子树 |

vs 哈希表：单次查找哈希也是 O(m)，但哈希表无法做前缀匹配。

### 空间压缩

大量字符串有公共前缀时，Trie 存储远优于独立存储：
- 存 `abc, abd, abe`：哈希表 3×3 = 9 字符；Trie 共享 `ab`，仅需 5 节点

## 三、核心使用原则

1. **空间换时间**：节点需要存 26（字母）或更多子节点指针，稀疏时浪费；可用哈希表或压缩 Trie（Patricia Trie）优化
2. **只适合字符串**：非字符串场景不适用
3. **实现比哈希表复杂**：需要管理树节点和子节点数组

## 四、与其他概念的关系

- 区别于 [[概念-数据结构]] 中的哈希表：哈希表精确匹配 O(1)，但不支持前缀查询；Trie 支持前缀查询
- 与 [[概念-BitMap]] 类似的思路：都是通过结构化编码压缩存储空间
- 支撑了 L8 工程实践：搜索引擎关键词补全；IP 路由表最长前缀匹配；违禁词过滤（AC 自动机基于 Trie）

## 五、应用边界

**适合**：
- 搜索自动补全（输入前缀，返回候选词）
- 违禁词/敏感词过滤（AC 自动机）
- IP 路由表（最长前缀匹配）
- 字典序排序（先序遍历 Trie 即得字典序）
- 词频统计（内存有限时比哈希表省空间）

**不适合**：
- 精确等值查找（哈希表 O(1) 更快）
- 数值类数据（Trie 专为字符串设计）
- 字符集超大时（如 Unicode），每节点子指针数组过大
