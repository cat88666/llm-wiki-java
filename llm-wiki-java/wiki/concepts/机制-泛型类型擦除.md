# 泛型与类型擦除

> 泛型让编译器在编译期做类型检查；类型擦除让 JVM 只维护一份字节码——安全在编译期，复用在运行期。

## 第一性原理

泛型之前，集合只能存 `Object`，取出时必须强制类型转换，错误在运行期才暴露。泛型的存在理由：**把类型错误从运行期提前到编译期，同时不增加运行时开销**。

Java 选择"编译期类型擦除"而非 C++ 的"为每种类型实例化一份代码"，理由是：避免代码膨胀，兼容 JDK 5 之前的字节码。

## 核心机制

### 泛型的编译处理

```java
List<String> list = new ArrayList<>();  // 编译器保留类型信息
list.add(123);  // 编译期报错 ← 泛型的核心价值
```

编译后字节码等价于：
```java
List list = new ArrayList();   // 类型参数被擦除
```

### 类型擦除规则

1. `List<String>` 和 `List<Integer>` 擦除后都是 `List`（同一份字节码）
2. 有上界限定 `<T extends Foo>` → 擦除为 `Foo`；无限定 → 擦除为 `Object`
3. 编译器在需要的地方**自动插入强制类型转换**

### 通配符上下界

| 写法 | 语义 | 允许操作 |
|------|------|---------|
| `? extends T` | 上界（Producer）| 只能读，不能写（写入类型不确定）|
| `? super T` | 下界（Consumer）| 可以写 T 及子类，读出只能当 Object |

记忆口诀：**PECS — Producer Extends, Consumer Super**

### 泛型 vs C++ 模板

| | Java 泛型 | C++ 模板 |
|--|-----------|---------|
| 实现方式 | 类型擦除（一份字节码）| 为每种类型生成专属代码 |
| 运行时类型 | 丢失 | 保留 |
| 代码膨胀 | 无 | 有 |
| 能力 | 受限（不能 `new T()`）| 强大 |

## 关键权衡

类型擦除带来的限制（设计代价）：

1. **运行时无法获取泛型参数**：`list instanceof List<String>` 非法；只能用 `TypeToken` 等技巧绕过
2. **不能 `new T()`**：实例化泛型类型时需传入 `Class<T>` 或工厂
3. **泛型不支持协变**：`List<String>` 不是 `List<Object>` 的子类（与数组不同）
4. **泛型方法不能重载**：`void f(List<String>)` 和 `void f(List<Integer>)` 擦除后冲突
5. **泛型异常不能多次 catch**

## 与其他概念的关系

- 依赖 [[机制-反射]]：反射可以绕过类型擦除（`Field.getGenericType()` 获取泛型参数）
- 约束了 [[概念-包装类与自动拆装箱]]：集合不能存基本类型，必须用包装类——这是类型擦除 + 泛型非协变共同导致的
- 支撑了 L4 集合框架：`List<E>`, `Map<K,V>` 的类型安全全靠泛型

## 应用边界

**适合用**：容器类（集合、Optional、Future）；通用工具类；类型安全的 API 设计。

**不适合用**：需要运行时类型信息时（用 `Class<T>` 传参替代）；基本类型（必须用包装类）。
