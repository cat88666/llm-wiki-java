# 线程池

> 线程池是池化技术在并发编程中的应用——预先创建一批线程常驻内存，任务到来时直接复用，避免频繁创建/销毁线程的开销；Java 中的核心实现是 ThreadPoolExecutor，通过 7 个构造参数控制核心线程、最大线程、任务队列、拒绝策略的行为。

## 第一性原理

线程的创建和销毁是操作系统级别的系统调用，代价远高于任务执行本身。在高并发场景下，频繁创建销毁线程会耗费大量 CPU 时间。线程池的存在理由：**通过复用已创建的线程，把线程的生命周期摊薄到整个应用程序运行期，同时通过有界队列 + 拒绝策略防止系统被压垮**。

## 核心机制

### 7 个构造参数

```java
new ThreadPoolExecutor(
    corePoolSize,      // 核心线程数：常驻线程（即使空闲也不销毁）
    maximumPoolSize,   // 最大线程数：核心线程 + 临时线程上限
    keepAliveTime,     // 临时线程空闲存活时间
    unit,              // 时间单位
    workQueue,         // 任务等待队列（BlockingQueue<Runnable>）
    threadFactory,     // 线程创建工厂（可设置线程名、优先级、守护状态）
    handler            // 拒绝策略
)
```

### 任务执行流程（execute 方法）

```
提交任务
  ├── 当前线程数 < corePoolSize
  │     → 创建新核心线程直接执行（addWorker）
  ├── 当前线程数 >= corePoolSize
  │     → 尝试入队（workQueue.offer）
  │         ├── 入队成功 → 等待空闲线程来取
  │         └── 队列已满
  │               ├── 当前线程数 < maximumPoolSize
  │               │     → 创建临时线程执行（addWorker）
  │               └── 已达最大线程数
  │                     → 执行拒绝策略（handler.rejectedExecution）
```

**记忆口诀**：核心线程 → 队列 → 最大线程 → 拒绝

### Worker 机制

每个 `Worker` 实现了 `Runnable`，包含：
- `firstTask`：初次分配的任务
- `Thread thread`：执行该 Worker 的线程

Worker 执行完 `firstTask` 后，会循环从 `workQueue` 取新任务（`getTask()`），直到队列为空或线程池关闭——这就是"线程复用"的实现。

### 4 种拒绝策略

| 策略 | 行为 | 适用场景 |
|------|------|---------|
| `AbortPolicy`（默认）| 抛出 `RejectedExecutionException` | 绝大多数场景，让调用方感知过载 |
| `DiscardPolicy` | 静默丢弃新任务 | 日志、统计等非关键任务 |
| `DiscardOldestPolicy` | 丢弃队列最老任务，重试提交 | 实时性优先（新任务比旧任务重要）|
| `CallerRunsPolicy` | 由提交任务的线程自己执行 | 降速反压，不丢任务但会阻塞生产者 |

### 为什么不建议用 Executors 工厂方法

| 工厂方法 | 坑 |
|---------|-----|
| `Executors.newFixedThreadPool` | 使用无界 `LinkedBlockingQueue`，任务堆积可能 OOM |
| `Executors.newCachedThreadPool` | maximumPoolSize = Integer.MAX_VALUE，可能创建大量线程耗尽资源 |
| `Executors.newSingleThreadExecutor` | 同上，无界队列 OOM 风险 |

**推荐**：手动构造 `ThreadPoolExecutor`，明确设置有界队列和合理的最大线程数。

## 关键权衡

1. **corePoolSize 如何设置**：
   - CPU 密集型：`corePoolSize = CPU 核数 + 1`（+1 防止偶发缺页中断）
   - IO 密集型：`corePoolSize = CPU 核数 × 2`（线程大量阻塞在 IO）
2. **队列选型**：有界队列（`ArrayBlockingQueue`）防 OOM；无界队列（`LinkedBlockingQueue`）防拒绝但有 OOM 风险
3. **线程池复用 ThreadLocal 的风险**：旧任务设置的 ThreadLocal 值会被下一个任务看到，详见 [[概念-ThreadLocal]]

## 与其他概念的关系

- 内部用 [[机制-AQS]] 实现线程的阻塞等待（`workers` 管理依赖 ReentrantLock）
- 内部用 [[机制-CAS]] 原子更新线程计数（`ctl` 变量同时编码线程数和线程池状态）
- 与 [[概念-ThreadLocal]] 交互时有内存泄漏风险，需任务结束后手动 `remove()`

## 应用边界

**用线程池**：所有需要多线程异步处理的场景，避免手动 new Thread。

**参数调优方向**：
- 监控 `getActiveCount()`、`getQueue().size()`、`getRejectedTaskCount()` 来动态调整
- 考虑动态线程池（运行期调整 `corePoolSize`/`maximumPoolSize`）

**不要用线程池**：轻量级异步任务可考虑 `CompletableFuture` + 虚拟线程（JDK 21+）
