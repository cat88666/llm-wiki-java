# JIT编译

> JVM 默认混合执行模式（Mixed Mode）：解释器逐行翻译字节码执行，JIT（即时编译器）识别"热点代码"将其编译为本地机器码缓存复用——用"先跑再编译"换取运行期更优化，峰值性能可超静态编译（AOT）。

## 一、为什么 JIT 优于纯解释和纯 AOT

| 执行方式 | 实现 | 启动速度 | 峰值性能 | 核心缺陷 |
|---------|------|---------|---------|---------|
| **纯解释执行** | 每次执行都重复翻译字节码 | 快 | 低 | 无缓存，重复翻译开销 |
| **纯 AOT（静态编译）** | 启动前预编译为机器码 | 极快 | 中 | 编译时缺乏运行期 profile，优化保守 |
| **JIT 混合执行** | 先解释，热点触发后编译缓存 | 慢（需预热）| **最高** | 预热期延迟，CodeCache 内存开销 |

**JIT 的核心价值**：用运行后积累的实际调用频率数据指导编译，只对真正"热"的代码做激进优化——编译成本聚焦到 20% 的代码带来 80% 的性能提升。

```
源代码 .java
  ↓ javac 编译
字节码 .class（平台无关中间语言）
  ↓ JVM 解释执行（慢，但立即可用）
  ↓ JIT 编译热点代码（触发阈值后编译为机器码缓存）
本地机器码（直接运行，最快）
```

Java 既不是纯编译型，也不是纯解释型，是**混合执行（Hybrid）**。

## 二、热点探测与分层编译（Level 0-4）

### 热点探测：两个计数器

| 计数器 | 作用 | 阈值（Server 模式）| 触发行为 |
|--------|------|------------------|---------|
| **方法调用计数器** | 统计方法被调用次数 | 默认 10000 | 触发标准 JIT 编译 |
| **回边计数器** | 统计循环体执行次数 | 与 CompileThreshold 关联 | 触发 OSR（栈上替换）——对**正在执行中**的方法进行 JIT 编译 |

`-XX:CompileThreshold=N`：调整方法调用计数器阈值（非分层模式下生效）。

### 分层编译 5 级（JDK8+ 默认开启，-XX:+TieredCompilation）

| 层级 | 编译器 | 特征 |
|------|--------|------|
| **Level 0** | 解释器 | 纯解释执行，收集 profiling 数据 |
| **Level 1** | C1 | 快速编译，无 profiling |
| **Level 2** | C1 + 有限 profiling | C1 编译 + 计数器/采样逻辑 |
| **Level 3** | C1 + 完整 profiling | 最常见中间态，为 C2 收集全量数据 |
| **Level 4** | C2 | 深度优化编译，利用 profiling 激进优化 |

典型晋升路径：**L0 → L3 → L4**（解释 → C1 带 profiling → C2 深度优化）。

**C1 vs C2 分工**：C1 编译快、优化保守，目标是快速降低解释开销；C2 编译慢但优化激进（逃逸分析、内联、循环展开），目标是长期服务的峰值吞吐。

## 三、逃逸分析与六类优化手段

### 逃逸状态三级

| 逃逸状态 | 说明 | 可追加的优化 |
|---------|------|------------|
| **全局逃逸** | 赋给静态字段 / 作为方法返回值 | 方法内联、循环优化 |
| **参数逃逸** | 作为参数传给其他方法 | + 锁消除 |
| **无逃逸** | 仅在当前方法内存活 | + 栈上分配、标量替换（优化最大化）|

### 六类优化手段

| 优化 | 前提 | 原理 | 效果 |
|------|------|------|------|
| **方法内联** | 热点方法字节码 < 35B（热点 < 325B）| 把被调用方法的代码复制到调用处，消除调用开销 | 最常见优化，是其他优化的前提 |
| **逃逸分析** | C2 编译（`-XX:+DoEscapeAnalysis`，JDK8+ 默认）| 判断对象是否会被方法外访问 | 是栈上分配、锁消除的前提 |
| **栈上分配** | 无逃逸对象 | 对象在栈帧中分配，随栈帧消失自动回收 | 不触发 GC，减少 GC 压力 |
| **锁消除** | 无逃逸对象 | `synchronized` 块 JIT 直接消除锁 | 减少同步开销 |
| **标量替换** | 无逃逸对象 | 对象分解为各字段存于寄存器/栈，不创建对象实例 | 进一步减少 GC 对象数 |
| **循环展开** | 循环次数可预测 | 减少循环控制指令开销 | 提升循环密集代码性能 |

**反直觉**：方法体超过 35 字节（热点方法超过 325 字节）后不会被内联；内联失败同时导致逃逸分析放弃——**写小方法**是对 JIT 最友好的编码习惯。

## 四、调控参数与编译状态查看

### 关键 JVM 参数

| 参数 | 作用 | 默认值 |
|------|------|--------|
| `-XX:+TieredCompilation` | 开启分层编译 | JDK8+ 默认开启 |
| `-XX:CompileThreshold=N` | 方法调用计数器阈值（非分层模式）| Client 1500 / Server 10000 |
| `-XX:ReservedCodeCacheSize=N` | CodeCache 大小（存放 JIT 编译后的机器码）| 240MB（JDK8 分层编译）|
| `-XX:+DoEscapeAnalysis` | 开启逃逸分析 | JDK8+ 默认开启 |
| `-XX:+EliminateLocks` | 开启锁消除 | 默认开启 |
| `-XX:+EliminateAllocations` | 开启标量替换 | 默认开启 |
| `-XX:MaxInlineSize=N` | 普通方法内联字节码大小阈值 | 35 字节 |
| `-XX:FreqInlineSize=N` | 热点方法内联字节码大小阈值 | 325 字节 |
| `-XX:CICompilerCount=N` | JIT 编译器线程数 | CPU 核数相关 |
| `-XX:+PrintCompilation` | 打印 JIT 编译日志 | 关闭 |

### 查看编译状态

```bash
java -version
# 输出含 "mixed mode" → 混合执行（默认）
# "compiled mode" → 仅 JIT；"interpreted mode" → 仅解释

java -XX:+PrintCompilation -jar app.jar
# 格式：<时间戳ms> <编译ID> <编译级别> <方法签名>

jstat -compiler <pid>
# 输出：Compiled Failed Invalid Time FailedType FailedMethod
```

**使用原则**：
- 绝大多数场景下默认分层编译配置已是最优，避免手动设置 CompileThreshold
- 监控 CodeCache 使用率，满了会退化回解释执行，性能断崖式下降
- 短生命周期对象尽量控制在方法内，不要不必要地赋给字段或作为返回值

## 五、发布预热与 AOT 选型

### 发布预热（JIT 三种方案）

JIT 优化在热点触发前全靠解释执行，**应用刚启动时响应最慢**——高并发发布时会出现大量超时。

| 方案 | 原理 | 适用场景 |
|------|------|---------|
| **流量预热** | 先放 10% 流量让 JIT 完成优化，再逐步拉满 | 蓝绿/滚动发布 |
| **Dragonwell JwarmUp** | 将上次 JIT 编译信息落盘，下次启动提前编译 | 阿里系 Java 服务 |
| **模拟请求预热** | 启动后发模拟流量触发热点编译 | 内部服务 |

### 何时改用 AOT（GraalVM Native Image）

```
长期运行后端服务（Spring Boot、中间件）  → JIT（预热后峰值最优）
计算密集型任务（循环/内联收益大）        → JIT
Serverless / Lambda 冷启动              → GraalVM Native Image（毫秒级启动）
CLI 工具 / 短命进程                      → AOT（JIT 还没完成编译进程就结束了）
启动时间硬性要求 < 100ms                 → AOT 是唯一选择
```

## 六、JIT vs AOT / C1 vs C2

### JIT vs AOT

| 维度 | JIT | AOT（GraalVM Native Image）|
|------|-----|---------------------------|
| 编译时机 | 运行时（热点触发）| 程序启动前（预编译）|
| 启动速度 | 慢（需要预热）| 快（直接运行机器码）|
| 峰值性能 | **高**（运行期 profile 激进优化）| 次之（缺乏运行时 profile）|
| 内存占用 | 较高（CodeCache + profiling 数据）| 较低 |
| 动态特性支持 | 完整（反射、动态代理正常工作）| 受限（需提前配置反射/代理元数据）|
| 适用场景 | 长期运行的服务 | Serverless、CLI 工具、短生命周期容器 |
| 代表实现 | HotSpot C1/C2 | GraalVM Native Image |

### C1 vs C2

| 维度 | C1（Client Compiler）| C2（Server Compiler）|
|------|---------------------|---------------------|
| 编译速度 | 快 | 慢（约为 C1 的 2-3 倍）|
| 优化深度 | 保守（基本优化）| 激进（逃逸分析、标量替换、循环展开）|
| 生成代码质量 | 中等 | 高 |
| 定位 | 快速编译降低解释开销 | 深度优化提升峰值吞吐 |

## 七、生产风险

| 风险 | 表现 | 原因 | 应对 |
|------|------|------|------|
| **预热期超时** | 刚发布后大量请求 RT 飙高、超时 | JIT 未完成编译，全靠解释执行 | 流量预热 / JwarmUp / 模拟请求 |
| **CodeCache 满** | 性能突然下降，`CodeCache is full` 告警 | JIT 机器码超过 ReservedCodeCacheSize，退回解释执行 | 调大 `-XX:ReservedCodeCacheSize`，监控 CodeCache 使用率 |
| **C2 编译线程 CPU 飙高** | 启动期 CPU 异常高 | 大量方法同时触发 C2 编译 | 适当调整 `-XX:CICompilerCount` |
| **逃逸分析失效** | 预期栈上分配的对象仍在堆上，GC 压力大 | 对象实际发生了逃逸，或方法体过大导致分析放弃 | 检查是否存在不必要的对象逃逸；拆分大方法 |
| **反优化（Deoptimization）** | 已编译机器码被丢弃，回退到解释执行 | 类层次变化（如新类加载）导致之前的假设失效 | 关注 `-XX:+TraceDeoptimization` 日志 |

## 八、JIT 在 JVM 体系的位置

```
JIT 编译后的机器码
  → 缓存在方法区 CodeCache（[[概念-JVM]] 五大区域之一）

逃逸分析 → 栈上分配
  → 直接减少堆内对象数量 → 减轻 GC 压力（[[机制-垃圾收集器]]）

锁消除
  → 依赖 synchronized 语义（L3 并发），无逃逸时由 C2 消除

C1 profiling 数据 → 指导 C2 激进优化
  → JVM 峰值性能的核心链路
```

**JIT 不解决的问题**：
- 冷启动延迟——需要 AOT（GraalVM Native Image）或 JwarmUp
- 可预测的超低延迟（< 1ms）——JIT 编译本身有 CPU 突发开销，极致延迟场景需配合 ZGC
- AOT 场景下反射/动态代理受限——需要 GraalVM 元数据配置文件（reflect-config.json）
