# JIT编译

> JVM 默认混合执行模式（Mixed Mode）：解释器逐行翻译字节码执行，JIT（即时编译器）识别"热点代码"将其编译为本地机器码缓存复用——用"先跑再编译"换取运行期的更优化。

## 一、第一性原理

纯解释执行：一边翻译一边运行，每次执行都要重复翻译，效率低。纯预编译（AOT）：编译时不知运行时的实际调用热点，优化信息不足。JIT 的存在理由：**在程序运行后，用实际执行频率数据指导编译，只对真正"热"的代码做激进优化，代价换来最优化收益**。

核心取舍：编译本身消耗 CPU 和内存，因此只有热点代码值得编译——"二八定律"在运行时的体现。

## 二、核心机制

### 2.1 Java 的执行模式

```
源代码 .java
  ↓ javac 编译
字节码 .class（平台无关中间语言）
  ↓ JVM 解释执行（慢，但立即可用）
  ↓ JIT 编译热点代码（快，但需要预热）
本地机器码（直接运行，最快）
```

**Java 既不是纯编译型，也不是纯解释型，是混合执行（Hybrid）**。

### 2.2 热点代码识别

HotSpot JVM 用**基于计数器的热点探测**：

| 计数器 | 作用 | 阈值（默认） | 触发行为 |
|--------|------|-------------|---------|
| **方法调用计数器** | 统计方法被调用次数 | Client 1500 / Server 10000 | 触发标准 JIT 编译 |
| **回边计数器** | 统计循环体执行次数 | 与 CompileThreshold 关联 | 触发 OSR（栈上替换，对正在运行的方法进行 JIT） |

关键参数：`-XX:CompileThreshold=N` 可调整方法调用计数器阈值。

### 2.3 C1/C2 分层编译（Tiered Compilation）

JDK 8+ 默认开启分层编译（`-XX:+TieredCompilation`），将编译分为 5 个层级：

| 层级 | 编译器 | 说明 |
|------|--------|------|
| **Level 0** | 解释器 | 纯解释执行，收集 profiling 数据 |
| **Level 1** | C1（Client Compiler） | 简单编译，无 profiling |
| **Level 2** | C1 + profiling | C1 编译并植入计数器/采样逻辑 |
| **Level 3** | C1 + full profiling | C1 编译 + 完整 profiling（最常见中间态） |
| **Level 4** | C2（Server Compiler） | 深度优化编译，利用 profiling 做激进优化 |

典型晋升路径：**L0 → L3 → L4**（解释 → C1 带 profiling → C2 深度优化）。

C1 特点：编译速度快，优化保守，适合快速启动；C2 特点：编译速度慢，优化激进（方法内联、逃逸分析、循环展开），适合长期运行服务的峰值性能。

### 2.4 JIT 的主要优化手段

| 优化 | 原理 | 效果 |
|------|------|------|
| **方法内联** | 把被调用方法的代码复制到调用处，消除方法调用开销 | 最常见优化，是其他优化的前提 |
| **逃逸分析** | 判断对象是否会被方法外访问（是否"逃逸"）| 是栈上分配、锁消除的前提 |
| **栈上分配** | 无逃逸对象在栈上分配，随栈帧消失自动回收，不触发 GC | 减少 GC 压力 |
| **锁消除** | 无逃逸对象的 synchronized 块，JIT 直接消除锁 | 减少同步开销 |
| **标量替换** | 无逃逸对象分解为各个字段存于寄存器/栈，不创建对象 | 进一步减少 GC 对象数 |
| **循环展开** | 减少循环控制指令开销 | 提升循环密集代码性能 |

### 2.5 逃逸状态三级

| 状态 | 说明 | 可做的优化 |
|------|------|----------|
| **全局逃逸** | 赋给静态字段 / 作为方法返回值 | 方法内联、循环优化 |
| **参数逃逸** | 作为参数传给其他方法 | + 锁消除 |
| **无逃逸** | 仅在当前方法内存活 | + 栈上分配、标量替换 |

## 三、Java 核心使用

### 3.1 关键 JVM 参数

| 参数 | 作用 | 默认值 |
|------|------|--------|
| `-XX:+TieredCompilation` | 开启分层编译 | JDK 8+ 默认开启 |
| `-XX:CompileThreshold=N` | 方法调用计数器阈值（非分层模式下生效） | Client 1500 / Server 10000 |
| `-XX:+PrintCompilation` | 打印 JIT 编译日志 | 关闭 |
| `-XX:ReservedCodeCacheSize=N` | CodeCache 大小（存放 JIT 编译后的机器码） | 240MB（JDK 8 分层编译） |
| `-XX:+DoEscapeAnalysis` | 开启逃逸分析 | JDK 8+ 默认开启 |
| `-XX:+EliminateLocks` | 开启锁消除 | 默认开启 |
| `-XX:+EliminateAllocations` | 开启标量替换 | 默认开启 |
| `-XX:MaxInlineSize=N` | 方法内联的字节码大小阈值 | 35 字节 |
| `-XX:FreqInlineSize=N` | 热点方法内联的字节码大小阈值 | 325 字节 |

### 3.2 查看 JIT 编译状态

```bash
# 查看当前 JVM 编译模式
java -version
# 输出含 "mixed mode" 表示混合执行

# 运行时打印 JIT 编译事件
java -XX:+PrintCompilation -jar app.jar

# 使用 jstat 查看编译统计
jstat -compiler <pid>
```

## 四、核心使用原则

1. **不要过早干预 JIT**：绝大多数场景下默认分层编译配置已是最优，避免手动设置 CompileThreshold
2. **关注 CodeCache**：监控 CodeCache 使用率，满了会导致退化回解释执行，性能断崖式下降
3. **写内联友好的代码**：小方法（< 35 字节）更容易被内联，避免超大方法体
4. **逃逸分析友好**：短生命周期对象尽量控制在方法内，不要不必要地将对象赋给字段或作为返回值

## 五、使用案例

### 5.1 发布预热方案

JIT 优化在热点触发前，所有请求经解释执行，**应用刚启动时响应最慢**。高并发发布时会出现大量超时。

| 方案 | 原理 | 适用场景 |
|------|------|---------|
| **流量预热** | 先放 10% 流量让 JIT 完成优化，再逐步拉满 | 蓝绿/滚动发布 |
| **Dragonwell JwarmUp** | 将上次 JIT 编译信息落盘，下次启动提前编译 | 阿里系 Java 服务 |
| **模拟请求预热** | 启动后发模拟流量触发热点编译 | 内部服务 |

### 5.2 Serverless 场景改用 AOT

Serverless 函数、Lambda 冷启动 → 改用 GraalVM Native Image（AOT 预编译成本地可执行文件，启动毫秒级），牺牲峰值性能换取极速启动。

## 六、综合对比

### 6.1 JIT vs AOT

| 维度 | JIT | AOT（Ahead-of-Time） |
|------|-----|---------------------|
| 编译时机 | 运行时（热点触发） | 程序启动前（预编译） |
| 启动速度 | 慢（需要预热） | 快（直接运行机器码） |
| 峰值性能 | **高**（利用运行期 profile 做激进优化） | 次之（缺乏运行时 profile 信息） |
| 内存占用 | 较高（需要 CodeCache + profiling 数据） | 较低 |
| 动态特性支持 | 完整（反射、动态代理正常工作） | 受限（需提前配置反射/代理元数据） |
| 适用场景 | 长期运行的服务 | Serverless、CLI 工具、短生命周期容器 |
| 代表实现 | HotSpot C1/C2、GraalVM JIT | GraalVM Native Image、jaotc（已废弃） |

### 6.2 C1 vs C2

| 维度 | C1（Client Compiler） | C2（Server Compiler） |
|------|----------------------|----------------------|
| 编译速度 | 快 | 慢（约为 C1 的 2-3 倍耗时） |
| 优化深度 | 保守（基本优化） | 激进（逃逸分析、标量替换、循环展开） |
| 生成代码质量 | 中等 | 高 |
| 定位 | 快速编译降低解释开销 | 深度优化提升峰值吞吐 |

## 七、生产风险

| 风险 | 表现 | 原因 | 应对 |
|------|------|------|------|
| **预热期超时** | 刚发布后大量请求 RT 飙高、超时 | JIT 未完成编译，全靠解释执行 | 流量预热 / JwarmUp / 模拟请求 |
| **CodeCache 满** | 性能突然下降，`CodeCache is full` 告警 | JIT 编译的机器码超过 ReservedCodeCacheSize | 调大 `-XX:ReservedCodeCacheSize`，监控 CodeCache 使用率 |
| **C2 编译线程 CPU 飙高** | 启动期 CPU 使用率异常高 | 大量方法同时触发 C2 编译 | 适当调整 `-XX:CICompilerCount` |
| **逃逸分析失效** | 预期栈上分配的对象仍在堆上，GC 压力大 | 对象实际发生了逃逸，或方法体过大导致分析放弃 | 检查代码是否存在不必要的对象逃逸 |
| **反优化（Deoptimization）** | 已编译的机器码被丢弃，回退到解释执行 | 类层次变化（如新类加载）导致之前的假设失效 | 关注 `-XX:+TraceDeoptimization` 日志 |

## 八、与其他概念的关系

- JIT 编译后的机器码缓存在 [[机制-JVM内存模型]] 的方法区（CodeCache）
- 逃逸分析让对象在栈上分配，直接减少了 [[机制-垃圾收集器]] 的工作量
- 锁消除优化与 L3 并发编程强关联：`synchronized` 在无逃逸时会被 JIT 直接消除
- 分层编译中 C1 收集的 profiling 数据指导 C2 做激进优化，这是 HotSpot 性能的核心链路

## 九、应用边界

**适合依赖 JIT 的场景**：
- 长期运行的后端服务（Spring Boot 应用、中间件）——JIT 有足够时间完成预热，峰值性能最优
- 计算密集型任务——循环展开、方法内联、逃逸分析带来显著收益

**不适合依赖 JIT 的场景**：
- Serverless 函数 / Lambda 冷启动 → 改用 GraalVM Native Image（AOT）
- CLI 工具 / 短命进程 → JIT 编译还没完成进程就结束了，AOT 更合适
- 对启动时间有硬性要求的场景（< 100ms） → AOT 是唯一选择
