## 一、第一性原理

JVM 解决三个根本问题：

1. **平台无关**：将 `.java` 编译为字节码 `.class`，由 JVM 在不同 OS 上统一解释/编译执行——"Write Once, Run Anywhere"
2. **自动内存管理**：开发者不需要手动 malloc/free，GC 自动识别和回收不可达对象
3. **运行期优化**：JIT 在程序运行后，基于实际热点数据做激进优化，峰值性能可超越静态编译

这三件事的代价是引入了 GC 停顿、预热延迟和内存开销，JVM 调优的本质就是在这些代价之间做取舍。

## 三、生命周期与类加载

### 3.1 JVM 实例生命周期

| 阶段 | 说明 |
| --- | --- |
| 诞生 | 执行 `java` 命令，创建一个 JVM 实例 |
| 运行 | `main()` 线程启动；只要还有非守护线程存活，JVM 就继续运行 |
| 消亡 | 所有非守护线程结束，或显式执行 `System.exit()` |

JVM 不是“类库”，而是一个运行进程；调优和排查时要从进程、线程、堆、栈、类加载和 GC 多个层面一起看。

### 3.2 对象生命周期

对象从创建到回收可以粗略分为：

1. **创建阶段**：分配内存，执行构造器，包括父类构造器，初始化实例变量。
2. **可见/不可见阶段**：仍可通过引用访问；离开作用域后局部变量失效。
3. **不可达阶段**：从 GC Roots 不可达。
4. **可收集阶段**：等待 GC 回收。
5. **终结阶段**：可能执行 `finalize()`，且最多只执行一次。
6. **释放阶段**：内存空间被 GC 回收。

现代 Java 不建议依赖 `finalize()` 做资源释放，推荐使用 `try-with-resources`、`Cleaner` 或显式关闭。

### 3.3 类生命周期

类生命周期 5 大阶段：

```text
加载 -> 链接（验证/准备/解析）-> 初始化 -> 使用 -> 卸载
```

| 阶段 | 做什么 |
| --- | --- |
| 加载 Load | 读取 `.class` 文件，将类元信息放入方法区，并在堆中创建 `java.lang.Class` 对象作为访问入口 |
| 验证 Verify | 校验字节码格式、语义、符号引用合法性，保证不会破坏 JVM 安全 |
| 准备 Prepare | 为静态变量分配内存并赋默认值；`final static` 编译期常量可能直接赋真实值 |
| 解析 Resolve | 将符号引用替换为直接引用 |
| 初始化 Initialize | 执行静态变量赋值和 `static {}`，父类先于子类初始化 |
| 使用 Use | 正常反射、调用静态方法、创建实例等 |
| 卸载 Unload | 类实例、类加载器、Class 对象都不可达后才可能卸载 |

### 3.4 类初始化触发与被动引用

常见主动触发初始化：

1. `new` 创建实例。
2. 读取或修改静态变量。
3. 调用静态方法。
4. `Class.forName()` 反射加载。
5. 初始化子类前先初始化父类。
6. JVM 启动时加载 main 类。
7. 访问非编译期常量的 `final static` 字段。

不会触发子类初始化的典型被动引用：

- 通过子类访问父类静态字段，只初始化父类。
- 定义某个类的数组。
- 访问编译期常量。

### 3.5 类加载器与双亲委派

| 加载器 | 负责加载 |
| --- | --- |
| Bootstrap ClassLoader | JDK 核心类库 |
| Extension ClassLoader | JRE 扩展包，JDK 9 后模块化体系已有变化 |
| Application ClassLoader | classpath 下的应用类 |
| 自定义 ClassLoader | 继承 `ClassLoader`，通常重写 `findClass()` |

类加载机制：

- **全盘负责**：一个类由哪个加载器加载，它引用的其他类通常也由该加载器负责加载，除非显式指定。
- **双亲委派**：先委派父加载器，父加载器加载失败后，子加载器再尝试加载。
- **缓存机制**：类加载后会缓存，类更新通常需要重启 JVM 或使用可卸载的自定义类加载器。

双亲委派的价值：保证核心 API 不被篡改，避免用户自定义 `java.lang.String` 替换 JDK 类；同时避免重复加载，保证类唯一性。

## 四、Java 核心使用

### 4.1 JVM 内存区域（高频考点）

- **5 大区域**：堆（共享，GC 主战场）、方法区（共享，元数据）、虚拟机栈（私有，方法帧）、本地方法栈（私有，native）、程序计数器（私有，唯一不会 OOM）
- **堆分代**：Eden:S0:S1 = 8:1:1；新生代:老年代 = 1:2（默认）
- **方法区演变**：JDK7 永久代（堆内）→ JDK8 元空间（本地内存，无固定上限）
- **OOM 定位**：heap space → 堆溢出；Metaspace → 类加载器泄漏；StackOverflow → 递归过深

| 区域 | 线程归属 | 作用 | 常见异常 |
| --- | --- | --- | --- |
| 程序计数器 PC | 线程私有 | 记录当前线程执行的字节码指令地址；执行 Native 方法时为空 | 唯一不会 OOM |
| 虚拟机栈 | 线程私有 | 每次方法调用创建栈帧，包含局部变量表、操作数栈、动态链接、返回地址 | `StackOverflowError` / `OutOfMemoryError` |
| 本地方法栈 | 线程私有 | 为 Native 方法提供栈帧 | `StackOverflowError` / `OutOfMemoryError` |
| 堆 | 线程共享 | 存放对象实例，GC 主战场 | `OutOfMemoryError: Java heap space` |
| 方法区/元空间 | 线程共享 | 类元数据、常量池、静态变量、JIT 代码等 | `OutOfMemoryError: Metaspace` |

虚拟机栈的栈帧包含：

- 局部变量表：基本类型、对象引用、`returnAddress`；`long/double` 占 2 个 slot。
- 操作数栈：存放运算中间结果和方法调用参数。
- 动态链接：指向运行时常量池，为符号引用转直接引用提供支撑。
- 方法返回地址：正常返回时回到下一条指令；异常退出时走异常表。

堆分代细节：

| 区域 | 默认比例 | 特点 |
| --- | --- | --- |
| Eden | 新生代 8/10 | 大多数新对象先分配在 Eden |
| S0/S1 | 各新生代 1/10 | Minor GC 后存活对象在 Survivor 间复制 |
| Old | 约堆 2/3 | 长期存活对象、大对象、晋升对象 |

对象年龄达到阈值（常见最大 15）或 Survivor 放不下时，会进入老年代。

### 4.2 GC Roots 与 GC 算法（高频考点）

JVM 主流采用可达性分析，而不是引用计数。引用计数实现简单，但无法处理循环引用。

GC Roots 常见来源：

- 虚拟机栈中局部变量引用的对象。
- 方法区中类静态变量引用的对象。
- 常量池引用的对象。
- 本地方法栈 JNI 引用的对象。
- 正在运行的线程对象、系统类加载器等 JVM 内部引用。

| 算法 | 特点 | 适用 |
|------|------|------|
| 标记-清除 | 快但产生碎片 | CMS 老年代 |
| 标记-复制 | 无碎片但浪费一半空间 | 新生代（Survivor 来回） |
| 标记-整理 | 无碎片无浪费但慢 | G1/ZGC 老年代 |

- **可达性分析**：从 GC Roots 遍历引用链，不可达 = 垃圾；引用计数法无法解决循环引用

Minor GC 时对象移动：

1. 扫描 GC Roots，找到存活对象。
2. 将 Eden + From Survivor 的存活对象复制到 To Survivor 或老年代。
3. 记录旧地址到新地址的关系，更新引用指向。
4. 原区域不逐个清除，只重置分配指针，后续对象覆盖即可。

### 4.3 垃圾收集器选型（高频考点）

| 收集器 | 特点 | 适用 |
|--------|------|------|
| CMS（已淘汰） | 老年代+标记-清除，有碎片，STW 不可预测，JDK14 删除 | — |
| **G1（主流）** | 整堆+Region 化，`-XX:MaxGCPauseMillis` 控制 STW 上限 | JDK9+ 默认，通用场景 |
| **ZGC（低延迟）** | 亚毫秒级停顿，染色指针+读屏障 | TB 级堆，JDK15+ |
| ParallelGC | 吞吐量优先 | 老项目维持 |

**选型口诀**：通用 → G1；超低延迟（< 1ms）→ ZGC；老项目维持 → ParallelGC。

### 4.4 双亲委派（高频考点）

- **流程**：收到加载请求 → 查缓存 → 委派父加载器 → 父找不到才自己加载
- **作用**：保证 `Object` 等核心类唯一性，防止用户伪造 `java.lang.Object`
- **破坏方式**：重写 `loadClass()`（破坏）vs 重写 `findClass()`（不破坏）
- **三大破坏场景**：SPI（线程上下文 ClassLoader）、Tomcat（WebApp 隔离）、OSGi（模块化）

### 4.5 JIT 与预热（中频考点）

- **混合执行**：字节码先解释执行，热点代码 JIT 编译为机器码缓存
- **热点阈值**：Server 模式默认 10000 次方法调用触发
- **逃逸分析**：无逃逸对象 → 栈上分配（不触发 GC）+ 锁消除 + 标量替换
- **预热问题**：应用刚启动 JIT 未完成，解释执行慢 → 发布时用流量预热

### 4.6 引用类型（中频考点）

| 引用类型 | GC 行为 | 典型用途 |
|---------|---------|---------|
| 强引用 | 不回收 | 普通变量引用 |
| 软引用 | OOM 前回收 | 缓存（Guava Cache softValues） |
| 弱引用 | 每次 GC 回收 | `WeakHashMap`、`ThreadLocal` Key |
| 虚引用 | 不影响生命周期 | 配合 ReferenceQueue 做回收后资源清理 |

**ThreadLocal 陷阱**：Key 弱引用 Value 强引用 → 线程池复用时 Value 泄漏 → 必须 `finally` 中 `remove()`。

## 五、JVM 调优与排查

### 5.1 JVM 问题排查流程

生产排查不要一上来就调参数，先采集证据：

1. `jstat` 看 GC 次数、耗时、Eden/Survivor/Old 使用变化，判断是否频繁 Young GC/Full GC。
2. `jmap` 看堆使用情况，必要时 dump 堆。
3. `jstack` 看线程状态，定位 BLOCKED、WAITING、死锁、高 CPU 线程栈。
4. `jinfo` 查看 JVM 参数是否符合预期。
5. 使用 VisualVM、MAT、JProfiler、Arthas 等工具分析 dump、线程和调用栈。
6. 根据现象提出假设：对象进入老年代过快、大对象直接晋升、内存泄漏、线程阻塞、元空间增长等。
7. 小步调整参数或代码，并通过压测/监控验证。

### 5.2 OOM 排查

生产建议开启 OOM dump：

```bash
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/data/dump
```

OOM 排查思路：

1. 用 MAT/VisualVM 打开 dump。
2. 找 retained size 最大的对象。
3. 看 GC Roots 引用链，判断为什么对象不能释放。
4. 结合线程栈和业务流量定位具体代码。
5. 如果是缓存类容器无限增长，增加容量上限、TTL、LRU/LFU 淘汰或改用成熟缓存框架。

典型内存泄漏：JVM 级 `HashMap` 缓存只放不清，老数据长期占用老年代，导致 Full GC 频繁，最终 OOM。

### 5.3 jstat 估算调优

常用命令：

```bash
jstat -gc <pid> 1000 10
```

可以估算：

| 指标 | 方法 |
| --- | --- |
| 年轻代对象增长速率 | 观察 EU（Eden used）在间隔内的增长 |
| Young GC 触发频率 | 根据 Eden 大小和增长速率估算多久打满 |
| Young GC 平均耗时 | `YGCT / YGC` |
| 每次 Young GC 后存活对象 | 观察 Survivor 和 Old 在 GC 后的增长 |
| 老年代增长速率 | 多次 Young GC 后 Old 使用量增量 |
| Full GC 平均耗时 | `FGCT / FGC` |

核心目标：尽量让短命对象在 Young GC 中被回收，减少对象晋升到老年代，从而降低 Full GC 频率。

如果 Full GC 频繁但没有 OOM，说明 Full GC 回收了不少对象。此时要判断这些对象是否“存活时间不长但过早进入老年代”。常见优化是增大年轻代或 Survivor，使对象有机会在年轻代被回收。

### 5.4 常见参数

```bash
java \
  -Xms2048M \
  -Xmx2048M \
  -Xmn1024M \
  -Xss512K \
  -XX:MetaspaceSize=256M \
  -XX:MaxMetaspaceSize=256M \
  -XX:+HeapDumpOnOutOfMemoryError \
  -XX:HeapDumpPath=/data/dump \
  -XX:+PrintGCDetails \
  -XX:+PrintGCDateStamps \
  -Xloggc:/data/logs/gc.log \
  -jar app.jar
```

| 参数 | 含义 |
| --- | --- |
| `-Xms` | 初始堆大小 |
| `-Xmx` | 最大堆大小 |
| `-Xmn` | 新生代大小 |
| `-Xss` | 每个线程栈大小 |
| `-XX:NewRatio` | 新生代和老年代比例，默认 2 表示老年代:新生代 = 2:1 |
| `-XX:SurvivorRatio` | Eden 与单个 Survivor 的比例，默认 8 |
| `-XX:MetaspaceSize` | 触发元空间 GC 的初始阈值 |
| `-XX:MaxMetaspaceSize` | 元空间最大值 |

元空间调优注意：`MetaspaceSize` 太小会导致应用启动期频繁 Full GC。生产中常将 `MetaspaceSize` 和 `MaxMetaspaceSize` 设为相同或接近的合理值，例如 256M，减少动态调整成本。

### 5.5 调优结论

JVM 调优的主线不是“背参数”，而是围绕对象生命周期做容量匹配：

- 给堆足够空间，避免 Eden 过小导致频繁 Young GC。
- 合理设置新生代，让短命对象尽量在年轻代被回收。
- 控制大对象和长生命周期对象，避免快速进入老年代。
- 通过 GC 日志和 `jstat` 观察，而不是凭感觉调参。
- 优先修代码层面的对象创建、缓存泄漏、阻塞和慢 SQL，再调 JVM 参数。

## 六、综合对比

### 6.1 概念依赖关系

```
JVM内存模型（堆分代结构）
    ↓ 决定
GC算法选择（新生代复制，老年代整理/清除）
    ↓ 实现为
垃圾收集器（CMS/G1/ZGC）
         ↑ 受影响
引用类型（软/弱/虚引用改变可达性分析结果）

类加载机制（双亲委派加载类到方法区）
    ↓ Class 对象
反射（运行期读取类结构）
    ↓
动态代理 / SPI（L1）

JIT编译（热点代码编译机器码）
    ↓ 逃逸分析
栈上分配（减少 GC 对象数量）
```

### 6.2 与上下层的关系

| 层 | 依赖 JVM 体系的方式 |
|----|-------------------|
| L1 语言基础 | Lambda 的 `invokedynamic` 依赖 JVM 指令集；反射依赖 Class 对象（由类加载产生）|
| L3 并发 | JMM（Java Memory Model）是 JVM 内存模型的并发视角；`volatile`/`synchronized` 语义由 JVM 保证 |
| L4 集合框架 | `HashMap` 扩容创建大量对象，触发 Young GC；理解 GC 才能分析集合框架性能 |
| L5 存储 | 数据库连接池（JDBC）依赖 SPI 机制（破坏双亲委派）加载驱动 |

## 七、应用边界

**JVM 体系是 Java 后端工程师的底层基石**——性能调优（GC 选型、内存配置）、故障排查（OOM、CPU 飙高）、面试深度追问都依赖对 JVM 的理解。

**待深化方向**：
- JMM（Java Memory Model）：happens-before 规则，`volatile` 的内存语义 → 与 L3 并发关联
- 字符串常量池：String.intern() 的行为变化（JDK6 vs JDK7+）
- 对象内存布局：对象头（Mark Word + Klass Pointer）+ 实例数据 + 对齐填充
- Safe Point：GC STW 的触发点，所有线程需要到达 Safe Point 才能开始 GC

## 八、面试速答

| 问题 | 速答 |
| --- | --- |
| JVM 内存区域有哪些 | 堆、方法区/元空间、虚拟机栈、本地方法栈、程序计数器 |
| 唯一不会 OOM 的区域 | 程序计数器 |
| 类生命周期 | 加载、链接（验证/准备/解析）、初始化、使用、卸载 |
| 双亲委派作用 | 防止核心类被篡改，避免重复加载，保证类唯一性 |
| GC Roots 有哪些 | 栈中引用、静态变量、常量池引用、JNI 引用、活动线程等 |
| Full GC 频繁怎么排查 | `jstat` 看频率和耗时，`jmap`/dump 看对象，`jstack` 看线程，判断晋升过快或泄漏 |
| JVM 调优核心目标 | 让短命对象在年轻代回收，减少老年代增长和 Full GC |
| 元空间为什么要设置 | 默认受本地内存限制，动态扩容会触发 Full GC，生产建议设置合理上限 |
