# 类加载机制

> JVM 通过类加载器将 `.class` 字节码加载进内存并初始化，双亲委派模型保证核心类（如 `java.lang.Object`）的唯一性与安全性——每次加载请求先委派给父加载器，父加载器找不到才由子加载器自己加载。

## 一、第一性原理

Java 是动态加载的语言——类只在第一次被使用时才加载，而非程序启动时全部加载。核心问题：如何保证 `java.lang.Object` 在所有类加载器环境下都是同一个类？

双亲委派的存在理由：**优先级层次关系保证基础类由最顶层的启动类加载器加载，防止用户伪造核心类（如自定义 `java.lang.Object`）**。

## 二、核心机制

### 类的生命周期（7 个阶段）

```
加载（Loading）
  → 验证（Verification）
  → 准备（Preparation）    ← 链接
  → 解析（Resolution）     ←
  → 初始化（Initialization）
  → 使用（Using）
  → 卸载（Unloading）
```

| 阶段 | 做什么 | 关键细节 |
|------|--------|---------|
| 加载 | 通过类名找到 .class 文件，读入字节数组，创建 `Class` 对象 | 触发时机：首次主动使用（new、静态方法调用等） |
| 验证 | 字节码合法性检查（文件格式、元数据、字节码语义） | 防止恶意字节码破坏 JVM |
| 准备 | 为类的**静态变量**分配内存，赋**默认值**（int = 0，引用 = null） | 注意：不是赋业务值，`static int x = 10` 此时 x = 0 |
| 解析 | 将符号引用（类名字符串）替换为直接引用（内存地址） | 可延迟到运行时（lazy resolution） |
| 初始化 | 执行 `<clinit>`（静态代码块 + 静态变量赋值），**真正赋业务初值** | `static int x = 10` 此时 x = 10 |
| 使用 | 创建实例、调用方法、访问字段 | 正常业务阶段 |
| 卸载 | 该类所有实例被 GC + 该类的 ClassLoader 被 GC → FullGC 时从方法区回收 | JDK 自带类永不卸载 |

### 三种默认类加载器（JDK 8）

```
Bootstrap ClassLoader（启动类加载器）   ← C++ 实现，Java 中表示为 null
    加载：JRE/lib/rt.jar（Object, String 等）
    ↓ 子
Extension ClassLoader（扩展类加载器）
    加载：JRE/lib/ext/ 下的类
    ↓ 子
Application ClassLoader（应用类加载器）
    加载：Classpath 下的类（自己写的代码）
```

> JDK 9 后模块化，Extension ClassLoader 改名为 Platform ClassLoader。

### 双亲委派模型的执行逻辑

加载请求自下而上委派，加载动作自上而下尝试：

1. Application ClassLoader 收到请求 → 不加载，委派给 Extension
2. Extension ClassLoader 收到请求 → 不加载，委派给 Bootstrap
3. Bootstrap 尝试加载 → 找到则返回，找不到则回退给 Extension
4. Extension 尝试加载 → 找到则返回，找不到则回退给 Application
5. Application 尝试加载 → 找到则返回，找不到则抛 ClassNotFoundException

## 三、Java 核心使用

### loadClass 核心代码

```java
// ClassLoader.loadClass 核心逻辑
protected Class<?> loadClass(String name, boolean resolve) {
    // 1. 先查缓存（是否已加载）
    Class<?> c = findLoadedClass(name);
    if (c == null) {
        // 2. 委派给父加载器
        if (parent != null)
            c = parent.loadClass(name, false);
        else
            c = bootstrapClassLoader.loadClass(name);
        // 3. 父加载器找不到，自己加载
        if (c == null)
            c = findClass(name);
    }
    return c;
}
```

### 扩展点选择

| 扩展方式 | 行为 | 是否破坏双亲委派 |
|---------|------|-----------------|
| 重写 `findClass()` | 自定义加载逻辑（如从网络/加密文件读取字节码） | 不破坏 |
| 重写 `loadClass()` | 改变委派顺序或跳过委派 | 破坏 |

### 线程上下文类加载器

`Thread.currentThread().getContextClassLoader()` 提供了一种"反向委派"通道：父加载器加载的代码可以通过当前线程获取子加载器，从而加载子加载器路径上的类。这是 SPI 机制的基础。

## 四、核心使用原则

| 原则 | 说明 |
|------|------|
| 优先重写 findClass | 自定义加载器通常只重写 `findClass`，保留双亲委派的安全性 |
| 仅在隔离需求时重写 loadClass | 需要类隔离（如多版本共存）时才打破委派 |
| 注意类型一致性 | 不同 ClassLoader 加载的同名类是**不同类型**，不可强转 |
| 类卸载需全链路回收 | 类卸载要求：ClassLoader 被 GC + 所有实例被 GC + Class 对象无引用 |

## 五、使用案例

### 破坏双亲委派的三种典型场景

| 场景 | 原因 | 实现方式 |
|------|------|---------|
| **SPI（如 JDBC）** | Bootstrap 加载器加载 `java.sql.Driver` 接口，但实现类在 Classpath 由 App ClassLoader 加载，父加载器无法"向下"找子类 | 线程上下文类加载器（`Thread.currentThread().getContextClassLoader()`） |
| **Tomcat** | 多个 webapp 需要隔离，各自有独立的 WebApp ClassLoader | 每个 webapp 自己的 ClassLoader 优先加载，隔离不同版本的类库 |
| **热部署（OSGi）** | 模块化，需要不同模块加载不同版本类 | 网状加载，而非树形委派 |

## 六、综合对比

### JDK 8 vs JDK 9+ 类加载器变化

| 对比维度 | JDK 8 | JDK 9+ |
|---------|-------|--------|
| 第二层加载器 | Extension ClassLoader | Platform ClassLoader |
| 加载路径 | JRE/lib/ext/ 目录 | 模块系统中的平台模块 |
| Bootstrap 加载范围 | rt.jar 整体加载 | 按模块按需加载（java.base 等） |
| 类路径可见性 | Classpath 全局可见 | 模块间需 exports/opens 声明 |
| 向后兼容 | - | unnamed module 兼容旧代码 |

## 七、生产风险

| 风险 | 现象 | 根因 | 应对 |
|------|------|------|------|
| ClassCastException | 同名类之间转换报错 | 不同 ClassLoader 加载同一个 .class 文件 → JVM 视为不同类型 | 统一类加载器；避免跨 ClassLoader 传递对象 |
| 元空间 OOM | `java.lang.OutOfMemoryError: Metaspace` | 频繁创建类（动态代理、JSP 编译、Groovy 脚本）而 ClassLoader 未被 GC | 排查 ClassLoader 是否被 GC Root 引用导致无法回收；限制 `-XX:MaxMetaspaceSize` |
| 类加载死锁 | 应用启动卡死或线程 BLOCKED | 两个类加载器互相触发对方的 loadClass，形成锁循环 | JDK 7+ 已改为并行加载（`registerAsParallelCapable`）；自定义加载器注意注册 |
| 热部署类泄漏 | 每次热部署后 Metaspace 持续增长 | 旧 ClassLoader 未被回收（被 ThreadLocal、静态字段等 GC Root 引用） | 热部署前清理 ThreadLocal；避免 static 字段持有跨 ClassLoader 对象 |
| SPI 加载失败 | `ServiceConfigurationError` 或接口实现找不到 | 线程上下文类加载器未正确设置 | 确认 `Thread.setContextClassLoader` 指向正确加载器 |

## 八、与其他概念的关系

- 加载后的类元数据存入 [[机制-JVM内存模型]] 中的方法区（元空间）
- 依赖 [[机制-动态代理]]：`Class.forName(name)` 就是触发类加载并返回 Class 对象
- 与 [[机制-SPI]] 强关联：SPI 用线程上下文类加载器加载实现类，是最典型的破坏双亲委派场景

## 九、应用边界

| 场景 | 是否需要自定义类加载器 | 说明 |
|------|----------------------|------|
| 热部署 | 需要 | 重新加载修改后的类，需新建 ClassLoader 替换旧实例 |
| 加密 class 文件 | 需要 | 自定义 findClass 中加入解密逻辑 |
| 多版本类库隔离 | 需要 | 如 Tomcat 多 webapp 各自加载不同版本依赖 |
| 普通业务开发 | 不需要 | 默认三层加载器已满足需求 |
| 单体 Spring Boot 应用 | 不需要 | DevTools 热重启已内置处理，无需手动干预 |
