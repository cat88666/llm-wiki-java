# 对象池技术

> 预先创建一批高代价对象并循环复用，避免频繁 new/GC 回收；本质是以**固定内存空间**换取 **Young GC 频率下降 + Full GC 消除**，是高并发系统实现 0 FGC 的核心手段之一。

## 一、第一性原理

JVM 垃圾回收的触发与**对象分配速率**强相关：每秒分配越多短命对象，Young GC 越频繁；大量短命对象在 GC 时晋升 Old 区则导致 Full GC（Stop-The-World）。

**颠覆思路**：不让对象死，而是让它"休眠"。用完的对象归还池中、下次直接借用；JVM 看不到这些对象的死亡，自然无需回收。在高并发游戏/交易系统中，配合 G1 GC 实现 **0 Full GC** 的核心手段之一。

## 二、核心机制

### 通用对象池模型

```
Pool
├── idle queue（空闲队列）        ← 归还的对象放这里
├── borrow()：从 idle queue 取    ← 有则直接返回，无则按策略创建/等待/报错
└── return(obj)：用完归还         ← reset 对象状态后放回 idle queue
```

### Netty PooledByteBufAllocator 分层结构

每次网络收发都需分配 ByteBuf；高并发下频繁 `new byte[]` 是 GC 杀手。

```
PooledByteBufAllocator 分层结构（类 jemalloc）：
  Chunk（4MB）→ Run（页的集合）→ Subpage（< 8KB 小对象）
  └── 每个 EventLoop 有独立的 PoolArena，避免跨线程竞争
```

**为什么用堆外内存（Direct）**：直接内存不经 JVM GC 管理，零拷贝路径（DMA → 网卡，无需在堆内再拷贝一次）。归还的 ByteBuf 将底层内存块标记为可用，下次申请直接复用同一块物理内存。

## 三、Java 核心使用

### Apache Commons Pool2 关键参数

| 参数 | 含义 |
|------|------|
| maxTotal | 池中对象上限（超出则 borrow 阻塞/报错）|
| maxIdle | 最大空闲数（超出的归还对象直接销毁）|
| minIdle | 最小保活数（evictor 线程维持此数量）|
| maxWaitMillis | borrow 等待超时（-1 无限等待）|
| testOnBorrow | 借出前校验可用性（数据库连接常用）|

### HikariCP 无锁设计（ConcurrentBag）

连接建立代价极高（TCP + DB 认证 + 会话初始化，通常 10-100ms）。HikariCP 用 **ConcurrentBag 无锁设计**，三级获取策略：

```
线程 borrow 连接：
  ① 优先从 ThreadLocal 取自己上次归还的连接（最快，零竞争）
  ② 次选 CAS 从共享队列抢（轻量竞争）
  ③ 最后等待（超过 connectionTimeout 抛异常）
```

### 自定义对象池（游戏/交易系统）

```java
public class SimpleObjectPool<T> {
    private final BlockingQueue<T> pool = new ArrayBlockingQueue<>(MAX_SIZE);
    private final Supplier<T> factory;

    public T borrow() {
        T obj = pool.poll();
        return obj != null ? obj : factory.get();  // 池空则新建
    }

    public void returnObj(T obj) {
        reset(obj);          // 重置状态，防止数据污染（关键！）
        if (!pool.offer(obj)) { /* 池满则丢弃，让GC回收 */ }
    }
}
```

## 四、核心使用原则

| 原则 | 说明 |
|------|------|
| 归还必重置 | `return()` 时必须彻底 reset 对象状态，防止数据泄漏 |
| 严格上限 | maxTotal 依据系统资源上限设定，阻塞 + 超时保护系统不 OOM |
| 保活适度 | minIdle 不宜过大，evictor 保活有网络开销，数据库有连接数上限 |
| 池大小经验 | HikariCP：`maximumPoolSize = CPU核数 × 2`（IO 密集型场景） |
| 冷启动预热 | `minimumIdle = maximumPoolSize`，预创建满池避免冷启动 |
| 超时兜底 | `connectionTimeout = 30s`，超时报错而非无限等待 |

## 五、使用案例

### 与 G1 GC 的协同效果

| 维度 | 无对象池（高并发场景） | 引入对象池后 |
|------|----------------------|-------------|
| 分配行为 | 每秒百万次 `new ByteBuffer(64KB)` | 核心路径复用已有对象，分配率接近 0 |
| Young GC | 每 100ms 一次（STW 5ms） | 几乎消失 |
| 大对象问题 | 大对象直接进 G1 Humongous Region，碎片化 | Humongous Region 稳定 |
| Full GC | Humongous 碎片 → Mixed GC → Full GC（秒级停顿） | **0 Full GC** |

## 六、综合对比

### HikariCP vs Druid

| 对比维度 | HikariCP | Druid |
|---------|----------|-------|
| 设计哲学 | 极致性能，代码精简（~130KB） | 功能全面，监控内置 |
| 核心数据结构 | ConcurrentBag（无锁） | ReentrantLock + Condition |
| 获取连接速度 | ThreadLocal 快速路径，极低延迟 | 锁竞争下略慢 |
| 监控能力 | 需配合外部工具（Micrometer 等） | 内置 StatFilter + Web 监控页 |
| SQL 防火墙 | 无 | 内置 WallFilter，可防 SQL 注入 |
| Spring Boot 默认 | 2.0+ 默认连接池 | 需手动引入 |
| 适用场景 | 追求极致性能、低延迟 | 需要监控、审计、SQL 防火墙 |

## 七、生产风险

| 风险 | 现象 | 根因 | 应对 |
|------|------|------|------|
| 连接泄漏 | 活跃连接持续增长，最终池满超时 | borrow 后未 return（异常路径遗漏 finally） | 开启 leakDetectionThreshold（HikariCP）；Druid 开启 removeAbandoned |
| 状态污染 | 请求 A 的数据出现在请求 B 的响应中 | 归还时未彻底 reset 对象状态 | return 时强制清理所有可变字段；池化对象设计为可重置 |
| 池满阻塞 | 大量线程阻塞在 borrow，P99 飙升 | maxTotal 过小或连接未及时归还 | 合理设置 maxTotal + connectionTimeout；监控活跃连接数告警 |
| 保活风暴 | evictor 频繁发送心跳，数据库压力大 | minIdle 设置过大 + testWhileIdle 间隔过短 | minIdle 不超过业务常态并发量；适当拉长保活间隔 |
| 堆外内存泄漏 | 进程 RSS 持续增长，堆内无异常 | Netty ByteBuf 未 release（引用计数未归零） | 开启 `-Dio.netty.leakDetection.level=PARANOID` 检测 |
| ThreadLocal 碎片 | 实际连接利用率低，部分线程持有连接但空闲 | 线程数远多于池大小，各线程 ThreadLocal 缓存分散 | 控制线程数与池大小比例；监控连接利用率 |

## 八、与其他概念的关系

- **依赖 [[机制-垃圾收集器]]**：对象池的价值边界由 GC 算法决定；G1 的 Region 模型和 Humongous 对象处理是对象池收益最大的场景
- **依赖 [[机制-JVM内存模型]]**：Netty 池化堆外内存（Direct Memory）不受 JVM GC 管理，是零拷贝的关键；池化堆内对象减少 Old 区压力
- **应用于 [[机制-Netty]]**：PooledByteBufAllocator 是 Netty 默认分配器（4.0+），ByteBuf 池化是 Netty 能支撑数万并发连接的关键
- **类比 [[机制-线程池]]**：线程池是线程对象的池化；二者本质相同（高代价资源的预创建+复用），连接池是数据库连接的池化

## 九、应用边界

| 场景 | 是否适合池化 | 原因 |
|------|-------------|------|
| 数据库连接 | 适合 | 创建代价高（TCP + DB 认证 + 会话初始化，10-100ms） |
| 网络连接 / ByteBuffer | 适合 | 高并发下频繁分配是 GC 杀手 |
| 复杂初始化对象 | 适合 | 构造成本高、使用频繁 |
| 请求级对象（百万次/秒） | 适合 | 已出现频繁 Young GC 或 Full GC |
| 含不可 reset 状态的对象 | 不适合 | 如含用户私密数据的不可变对象，无法安全复用 |
| Integer/String 等 | 不适合 | 创建代价低，JVM 已有缓存机制 |
| 简单 CRUD 应用 | 不适合 | GC 压力不是系统瓶颈，池化增加复杂度无收益 |
