# 对象池技术

> 预先创建一批高代价对象并循环复用，避免频繁 new/GC 回收；本质是以**固定内存空间**换取 **Young GC 频率下降 + Full GC 消除**，是高并发系统实现 0 FGC 的核心手段之一。

## 第一性原理

JVM 垃圾回收的触发与**对象分配速率**强相关：每秒分配越多短命对象，Young GC 越频繁；大量短命对象在 GC 时晋升 Old 区则导致 Full GC（Stop-The-World）。

**颠覆思路**：不让对象死，而是让它"休眠"。用完的对象归还池中、下次直接借用；JVM 看不到这些对象的死亡，自然无需回收。在高并发游戏/交易系统中，配合 G1 GC 实现 **0 Full GC** 的核心手段之一。

## 核心机制

### 通用对象池模型

```
Pool
├── idle queue（空闲队列）        ← 归还的对象放这里
├── borrow()：从 idle queue 取    ← 有则直接返回，无则按策略创建/等待/报错
└── return(obj)：用完归还         ← reset 对象状态后放回 idle queue
```

**关键参数（Apache Commons Pool2）**：

| 参数 | 含义 |
|------|------|
| maxTotal | 池中对象上限（超出则 borrow 阻塞/报错）|
| maxIdle | 最大空闲数（超出的归还对象直接销毁）|
| minIdle | 最小保活数（evictor 线程维持此数量）|
| maxWaitMillis | borrow 等待超时（-1 无限等待）|
| testOnBorrow | 借出前校验可用性（数据库连接常用）|

### 数据库连接池（HikariCP）

连接建立代价极高（TCP + DB 认证 + 会话初始化，通常 10-100ms）。HikariCP 用 **ConcurrentBag 无锁设计**：

```
线程 borrow 连接：
  ① 优先从 ThreadLocal 取自己上次归还的连接（最快，零竞争）
  ② 次选 CAS 从共享队列抢（轻量竞争）
  ③ 最后等待（超过 connectionTimeout 抛异常）
```

关键配置经验：
```
maximumPoolSize = CPU核数 × 2   （IO密集型场景）
minimumIdle     = maximumPoolSize（预创建满池，避免冷启动）
connectionTimeout = 30s          （超时报错而非无限等待）
```

### Netty ByteBuf 池（PooledByteBufAllocator）

每次网络收发都需分配 ByteBuf；高并发下频繁 `new byte[]` 是 GC 杀手。

```
PooledByteBufAllocator 分层结构（类 jemalloc）：
  Chunk（4MB）→ Run（页的集合）→ Subpage（< 8KB 小对象）
  └── 每个 EventLoop 有独立的 PoolArena，避免跨线程竞争
```

**为什么用堆外内存（Direct）**：直接内存不经 JVM GC 管理，零拷贝路径（DMA → 网卡，无需在堆内再拷贝一次）。归还的 ByteBuf 将底层内存块标记为可用，下次申请直接复用同一块物理内存。

### 自定义对象池（游戏/交易系统）

```java
public class SimpleObjectPool<T> {
    private final BlockingQueue<T> pool = new ArrayBlockingQueue<>(MAX_SIZE);
    private final Supplier<T> factory;

    public T borrow() {
        T obj = pool.poll();
        return obj != null ? obj : factory.get();  // 池空则新建
    }

    public void returnObj(T obj) {
        reset(obj);          // 重置状态，防止数据污染（关键！）
        if (!pool.offer(obj)) { /* 池满则丢弃，让GC回收 */ }
    }
}
```

### 与 G1 GC 的协同效果

```
无对象池（高并发场景）：
  每秒百万次 new ByteBuffer(64KB)
    → Young GC 每 100ms 一次（Stop-The-World 5ms）
    → 大对象直接进 G1 Humongous Region
    → Humongous Region 碎片 → Mixed GC → Full GC（秒级停顿）

引入对象池后：
  核心路径复用已有对象 → 分配率接近 0
    → Young GC 几乎消失
    → Humongous Region 稳定 → 0 Full GC
```

## 关键权衡

1. **状态重置（最大风险）**：归还时必须彻底 reset，否则上一个请求的用户数据泄漏到下一个请求（安全漏洞 + 业务错乱）
2. **池满策略**：严格上限（阻塞 + 超时）保护系统不 OOM，但高峰等待会放大 P99；宽松上限（超出即新建）避免等待，但极端流量仍有 OOM 风险——依据系统资源上限设定 maxTotal
3. **minIdle 保活代价**：保活线程（evictor）定期 testWhileIdle 检测连接存活，对数据库连接有网络开销；minIdle 过大浪费数据库连接资源（数据库有连接数上限）
4. **ThreadLocal 快速路径 vs 负载不均**：HikariCP/Netty 用线程本地 cache 减少竞争，但线程数远多于池大小时，各线程持有的本地连接造成空闲碎片，实际 maxTotal 利用率低
5. **对象大小 vs 池化收益**：大对象（> 1MB）池化收益高但固定内存占用高；小对象（< 1KB）是否池化要看分配频率，不能盲目池化

## 与其他概念的关系

- **依赖 [[机制-GC算法与垃圾收集器]]**：对象池的价值边界由 GC 算法决定；G1 的 Region 模型和 Humongous 对象处理是对象池收益最大的场景
- **依赖 [[机制-JVM内存模型]]**：Netty 池化堆外内存（Direct Memory）不受 JVM GC 管理，是零拷贝的关键；池化堆内对象减少 Old 区压力
- **应用于 [[机制-Netty]]**：PooledByteBufAllocator 是 Netty 默认分配器（4.0+），ByteBuf 池化是 Netty 能支撑数万并发连接的关键
- **类比 [[机制-线程池]]**：线程池是线程对象的池化；二者本质相同（高代价资源的预创建+复用），连接池是数据库连接的池化

## 应用边界

**适合对象池**：
- 创建代价高（数据库连接、网络连接、ByteBuffer、复杂初始化对象）
- 生命周期短但使用频繁（请求级对象，百万次/秒分配）
- 已出现频繁 Young GC 或 Full GC，Profiler 显示大量同类对象分配

**不适合**：
- 对象含有不可 reset 的状态（如含用户私密数据的不可变对象）
- 创建代价低（Integer/String 等 JVM 已有缓存机制）
- 简单 CRUD 应用，GC 压力不是系统瓶颈
