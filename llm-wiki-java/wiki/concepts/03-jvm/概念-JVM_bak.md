## 一、JVM 的职责与代价

JVM 的价值可以压缩成三句话：

1. **平台无关**：`.java` 编译为 `.class`，同一字节码由不同 OS 上的 JVM 解释或 JIT 编译执行。
2. **自动内存管理**：对象生命周期由可达性分析 + GC 决定，开发者不手动 free。
3. **运行期优化**：基于真实热点路径做编译和优化，峰值性能通常高于纯解释执行。

代价同样明确：

| 价值 | 对应代价 |
| --- | --- |
| 自动回收 | GC 停顿（STW）不可完全消除 |
| 运行期优化 | 冷启动期有预热成本 |
| 抽象屏蔽硬件 | 内存与执行行为可控性低于 C/C++ |

JVM 调优本质就是把这三类代价压在业务可接受区间内。

## 二、内存版图与对象流转

### 2.1 运行时区域（先分清谁私有谁共享）

| 区域 | 线程归属 | 主要内容 | 常见异常 |
| --- | --- | --- | --- |
| 堆 | 共享 | 对象实例、数组 | `Java heap space` |
| 方法区 / 元空间 | 共享 | 类元信息、运行时常量池、静态变量、JIT 代码缓存 | `Metaspace` |
| 虚拟机栈 | 私有 | 栈帧（局部变量、操作数栈、动态链接、返回地址） | `StackOverflowError` |
| 本地方法栈 | 私有 | Native 方法栈帧 | `StackOverflowError` |
| 程序计数器 | 私有 | 当前线程执行位置 | 无 OOM |

反直觉点：程序计数器是规范中唯一明确不会 OOM 的区域。

### 2.2 堆分代为什么能提升吞吐

```
Java Heap
├── Young
│   ├── Eden
│   ├── Survivor From
│   └── Survivor To
└── Old
```

- 对象“朝生夕灭”是主流分布，所以把短命对象集中在 Young，用复制算法高频快速回收。
- Survivor 需要 From/To 双区轮转，不是冗余设计，而是复制算法落地条件。
- 默认比例通常是 `Eden:S0:S1 = 8:1:1`。

### 2.3 对象创建到晋升的完整链路

```
new 指令
  -> 检查类是否已加载
  -> 分配内存（指针碰撞 / 空闲列表）
  -> 零值初始化
  -> 写对象头（哈希/分代年龄/锁标志等）
  -> 执行 <init>
```

晋升到老年代的高频触发条件：

| 条件 | 结论 |
| --- | --- |
| 年龄达到阈值 | 默认 `MaxTenuringThreshold=15` 后晋升 |
| 动态年龄判断 | Survivor 占用过半时，年龄较大的批量晋升 |
| 大对象直入老年代 | 由 `PretenureSizeThreshold` 控制 |

### 2.4 TLAB：对象分配不加锁的关键

- 每个线程在 Eden 预留私有小块内存（TLAB）。
- 常规对象分配先走 TLAB，避开全局锁竞争。
- TLAB 用尽才回到全局分配路径。

### 2.5 PermGen 到 Metaspace：为什么一定要改

| 版本 | 实现 | 主要问题 |
| --- | --- | --- |
| JDK7- | PermGen（堆内） | 容量固定，易 `PermGen OOM` |
| JDK8+ | Metaspace（本地内存） | 不设上限可能吃光机器内存 |

生产建议：永远显式设置 `MaxMetaspaceSize`，把“无限扩张”变成“可预警、可回收、可降级”。

### 2.6 堆外内存（Direct Memory）

| 维度 | 说明 |
| --- | --- |
| 优势 | 降低用户态与内核态拷贝开销，适合高吞吐网络 IO |
| 风险 | 不在堆里，GC 图上不直观；泄漏更隐蔽 |
| 参数 | `-XX:MaxDirectMemorySize` |

## 三、GC 从判活到回收

### 3.1 判活：为什么 HotSpot 选可达性分析

- 引用计数无法解决循环引用。
- 可达性分析从 GC Roots 出发，无法到达的对象才是垃圾。

常见 GC Roots：

| 类型 | 示例 |
| --- | --- |
| 栈引用 | 方法局部变量、参数 |
| 静态引用 | `static` 字段 |
| 常量池引用 | 字符串常量等 |
| JNI 引用 | Native 侧持有对象 |
| 活跃线程 / 锁对象 | 运行线程、监视器对象 |

### 3.2 三色标记与漏标修复

- 白：未访问
- 灰：已访问，子节点未扫完
- 黑：已访问且子节点已扫完

并发标记期间，引用关系会变化：

| 收集器 | 漏标修复思路 |
| --- | --- |
| CMS | 增量更新 |
| G1 | SATB（初始快照） |

### 3.3 三种回收算法与落地位置

| 算法 | 优点 | 代价 | 常见落地 |
| --- | --- | --- | --- |
| 标记-清除 | 简单、快 | 碎片多 | CMS 老年代 |
| 复制 | 无碎片 | 空间折损 | Young 区 |
| 标记-整理 | 无碎片且空间利用高 | 移动对象成本高 | G1/ZGC 相关阶段 |

### 3.4 引用强度与可回收性

| 类型 | 回收时机 | 典型用途 |
| --- | --- | --- |
| 强引用 | 可达即不回收 | 业务主对象 |
| 软引用 | OOM 前尽量回收 | 内存敏感缓存 |
| 弱引用 | 下一次 GC 即可回收 | `WeakHashMap`、`ThreadLocal` key |
| 虚引用 | 仅做回收通知 | 资源清理钩子 |

反直觉点：`ThreadLocalMap` 的 key 是弱引用，但 value 常是强引用；线程池复用时不 `remove()` 会形成长期泄漏。

### 3.5 收集器选型（先看延迟目标再看堆规模）

| 收集器 | 停顿目标 | 吞吐 | 适用场景 |
| --- | --- | --- | --- |
| ParallelGC | 允许更长停顿 | 高 | 批处理、吞吐优先 |
| G1 | 可控停顿（`MaxGCPauseMillis`） | 高 | 通用服务默认首选 |
| ZGC | 亚毫秒级停顿 | 中高 | 大堆 + 严苛低延迟 |
| CMS（旧） | 低停顿 | 中 | 仅历史系统维护 |

## 四、类加载与类隔离

### 4.1 生命周期 7 阶段

```
加载 -> 验证 -> 准备 -> 解析 -> 初始化 -> 使用 -> 卸载
```

高频考点：`static int x = 10` 在“准备”阶段是默认值 `0`，在“初始化”阶段才是 `10`。

### 4.2 三层加载器与委派链

```
Bootstrap
  -> Platform(8时代是 Extension)
    -> Application
```

双亲委派默认流程：

1. `findLoadedClass` 查本地缓存
2. 委派父加载器尝试加载
3. 父失败后才 `findClass`

核心价值：防止核心类被业务侧篡改，避免同一类重复加载。

### 4.3 什么时候会“破坏”双亲委派

| 场景 | 为什么必须破坏 | 典型手段 |
| --- | --- | --- |
| SPI（JDBC 等） | 父加载器要用子路径实现类 | 线程上下文类加载器 |
| Tomcat 多应用隔离 | 同名依赖需多版本并存 | WebAppClassLoader 优先本地 |
| 热部署 / OSGi | 模块独立升级 | 自定义网状加载关系 |

## 五、JIT 与执行优化

### 5.1 解释执行到编译执行

- 冷代码先解释执行，避免“先编译再运行”的启动开销。
- 热点方法触发 C1/C2 编译后，进入机器码路径提升吞吐。

### 5.2 逃逸分析给 GC 减压

| 优化 | 条件 | 收益 |
| --- | --- | --- |
| 栈上分配 | 对象不逃逸方法边界 | 减少堆分配与 GC 压力 |
| 锁消除 | 锁对象线程私有 | 减少同步开销 |
| 标量替换 | 对象可拆为字段 | 减少对象创建 |

### 5.3 预热不是玄学

- 冷启动 RT 波动通常来自热点尚未编译。
- 发布后导入小流量预热，比“一上来吃满流量”更稳定。

## 六、调优与排障闭环

### 6.1 五步闭环（先证据后参数）

```
基线 -> 异常识别 -> 工具采样 -> 代码/参数修复 -> 回归验证
```

### 6.2 基线参考（4C8G / 堆4G / G1）

| 指标 | 经验正常值 | 告警阈值 |
| --- | --- | --- |
| YoungGC 频率 | ~100 次/分钟 | >500 次/分钟 |
| YoungGC 时长 | 10~30ms | >100ms |
| FullGC 频率 | 周级到小时级（视业务峰值） | >1 次/小时 |
| FullGC 时长 | 400~700ms | >1s |
| 堆占用 | <50% | >80% |

### 6.3 工具链（命令先后顺序）

```bash
jstat -gc <pid> 1000 10
jmap -histo:live <pid>
jstack <pid>
jinfo -flags <pid>
```

定位逻辑：

1. `jstat` 看频率和趋势，不看单点。
2. `jmap` 看“大对象是谁”，不是只看对象数量。
3. `jstack` 把高 CPU 线程映射回业务栈。
4. 参数核对后才改 JVM 参数。

### 6.4 四类常见故障的标准打法

| 症状 | 先查什么 | 常见根因 | 优先动作 |
| --- | --- | --- | --- |
| FullGC 频繁 | `jstat` FGC/FGCT | 晋升过快、长生命周期对象、泄漏 | 调整新生代 + dump 分析 retained size |
| YoungGC 过密 | Eden 增长斜率 | Eden 太小、对象 churn 高 | 增大 Young，减少短命对象创建 |
| STW 过长 | GC 日志停顿分段 | 回收目标过大、参数不匹配 | 先调 G1 目标停顿，再评估 ZGC |
| OOM | dump + GC Root 链 | 泄漏、容量规划错误、线程/堆外失控 | 修代码，再补参数上限 |

### 6.5 常用参数速查

```bash
-Xms4g -Xmx4g
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200
-XX:MaxMetaspaceSize=512m
-XX:MaxDirectMemorySize=256m
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/data/dump
-Xlog:gc*:file=/var/log/gc.log:time,uptime:filecount=5,filesize=20m
```

## 七、关键对比与选型结论

### 7.1 堆 vs 栈

| 维度 | 堆 | 栈 |
| --- | --- | --- |
| 归属 | 线程共享 | 线程私有 |
| 存储 | 对象实例 | 栈帧与局部变量 |
| 回收 | GC | 方法返回自动弹栈 |
| 性能 | 分配慢、容量大 | 分配快、容量小 |

### 7.2 PermGen vs Metaspace

| 维度 | PermGen（JDK7-） | Metaspace（JDK8+） |
| --- | --- | --- |
| 存储位置 | 堆内 | 本地内存 |
| 容量 | 固定上限 | 默认可扩展 |
| 典型问题 | `PermGen OOM` | 不限上限导致机器内存吃满 |

### 7.3 G1 vs ZGC 快速结论

| 维度 | G1 | ZGC |
| --- | --- | --- |
| 默认性 | JDK9+ 通用默认路线 | 低延迟专项路线 |
| 停顿目标 | 百毫秒级可控 | 亚毫秒级 |
| 资源代价 | 中等 | 更高并发开销 |
| 适用 | 绝大多数服务 | 低延迟且堆大的核心链路 |

## 八、生产高频故障模式

| 风险 | 触发条件 | 直接表现 | 应对 |
| --- | --- | --- | --- |
| 堆泄漏 | 缓存无上限、静态集合持有对象 | `Java heap space` | dump 看 GC Root，做容量与淘汰策略 |
| 元空间泄漏 | 动态代理/热部署类加载器不释放 | `Metaspace` 持续增长 | 排查 ClassLoader 引用链，限制上限 |
| 直接内存泄漏 | NIO/Netty 未释放 | RSS 高于 `-Xmx` 很多 | 限制 `MaxDirectMemorySize`，审计释放逻辑 |
| 线程爆炸 | 无边界线程池、阻塞堆积 | `unable to create new native thread` | 线程池限流、队列治理、降级 |
| 类隔离失败 | 同名类跨加载器传递 | `ClassCastException` | 统一边界类加载策略 |

## 九、面试速答与边界

### 9.1 高频问答（速查）

| 问题 | 口径 |
| --- | --- |
| JVM 内存区域有哪些 | 堆、方法区/元空间、虚拟机栈、本地方法栈、程序计数器 |
| 哪个区域不会 OOM | 程序计数器 |
| 双亲委派解决什么 | 核心类安全与类唯一性，避免重复加载 |
| FullGC 频繁先做什么 | `jstat` 看趋势，`jmap` 找大对象，dump 看引用链，再决定参数 |
| G1 和 ZGC 怎么选 | 默认 G1；低延迟强约束且大堆用 ZGC |

### 9.2 JVM 解决不了的问题

- 跨 JVM 状态一致性（要靠 Redis/DB/MQ）。
- 严格硬实时（GC 仍会有停顿）。
- 对象精确内存布局控制（Java 不提供这一级控制能力）。

> 面试收口口径：JVM 不是“调参数工具”，而是“运行时系统”；高质量回答要覆盖机制、代价、证据链和决策边界。
