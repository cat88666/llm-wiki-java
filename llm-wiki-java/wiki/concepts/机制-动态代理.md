# 动态代理

> 在运行期生成代理类，对目标对象的方法调用进行拦截和增强——不修改原始代码实现横切逻辑。

## 第一性原理

代理模式的本质是**在调用方与被调用方之间插入一层**，用于做前置/后置处理。静态代理在编译期写死代理类，代码量随被代理类数量线性增长。动态代理的存在理由：**运行期按需生成代理类，消除样板代码，使 AOP 等横切关注点可以统一施加**。

## 核心机制

### JDK 动态代理

- **前提**：目标类必须实现至少一个接口
- **入口**：`Proxy.newProxyInstance(ClassLoader, Class<?>[], InvocationHandler)`
- **调用链**：代理对象方法调用 → `InvocationHandler.invoke(proxy, method, args)` → `method.invoke(target, args)`（底层反射）
- **字节码**：JVM 运行时生成 `$Proxy0` 等类，实现目标接口，所有方法委托给 InvocationHandler

### CGLIB 动态代理

- **前提**：目标类不能是 final（通过继承实现代理）
- **底层**：ASM 字节码库，直接生成目标类的子类字节码
- **性能**：方法调用比 JDK 代理快（无反射），但类生成比 JDK 代理慢
- **限制**：final 方法不能被代理（无法 override）

### Spring AOP 的选择策略

```
目标类实现了接口？
├── 是 → 默认用 JDK 动态代理
└── 否 → 改用 CGLIB
     （也可通过 proxyTargetClass=true 强制全用 CGLIB）
```

### 与静态代理的对比

| | 静态代理 | 动态代理 |
|--|---------|---------|
| 确定时机 | 编译期 | 运行期 |
| 代码量 | 随被代理类增长 | 固定（InvocationHandler） |
| 灵活性 | 低 | 高 |

## 关键权衡

| 选择 | 适用场景 | 代价 |
|------|---------|------|
| JDK 代理 | 面向接口设计 | 必须有接口；invoke 经过反射 |
| CGLIB | 无接口 / 追求运行时方法调用性能 | 生成类耗时更长；final 类不可用 |

## 与其他概念的关系

- 依赖 [[机制-反射]]：JDK 代理的 `invoke()` 最终通过反射调用目标方法
- 支撑了 L6 Spring AOP：AOP 的"切面织入"依赖动态代理实现
- 支撑了 L6 MyBatis：Mapper 接口的实现类由 CGLIB/JDK 代理动态生成
- 支撑了 L7 Dubbo：RPC 的客户端桩（Stub）是动态代理生成的

## 应用边界

**适合用**：AOP（日志、事务、权限、限流）、RPC 客户端桩、ORM 懒加载、过滤器/拦截器。

**不适合用**：代理 final 类（用 JDK 代理或重设计）；极高频方法调用且对延迟极敏感时（考虑 MethodHandle 或编译期字节码织入 AspectJ）。
