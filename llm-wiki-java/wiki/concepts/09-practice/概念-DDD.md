# DDD 领域驱动设计

> 以**业务领域**为核心的软件设计方法论：先对业务进行领域建模，再考虑持久化存储；让领域对象自包含业务行为，避免业务逻辑散落在 Service 层（贫血模型的典型问题）。

## 第一性原理

传统"表驱动开发"路径：来了需求 → 先设计数据库表 → 生成 POJO → 在 Service 层写所有业务逻辑 → POJO 退化为纯数据容器（贫血模型）。这条路导致 Service 臃肿、逻辑分散、难以维护。

DDD 的颠覆：**代码的核心是业务知识，而不是数据库表**。软件开发人员与业务人员共同建立统一语言（Ubiquitous Language），将业务概念映射到代码中的领域对象，由领域对象自己完成行为（充血模型）。

---

## 核心机制

### 核心概念

**实体（Entity）**：具有唯一标识（ID）的领域对象，有自己的生命周期，可变。
```java
// 订单是实体（有唯一 orderId，状态可变）
Order order = new Order(orderId, userId);
order.pay(payAmount);    // 充血模型：行为封装在实体内
```

**值对象（Value Object）**：无唯一标识，通过属性集合描述，**不可变**（修改=创建新对象）。
```java
// 地址是值对象（没有 ID，只是一组字段）
Address address = new Address("浙江省", "杭州市", "西湖区", "文三路 1 号");
// 修改地址 = 创建新地址对象，而非修改原对象
```

**聚合（Aggregate）与聚合根（Aggregate Root）**：
- 聚合 = 将多个相关实体/值对象组合为一个整体，对外暴露统一边界
- 聚合根 = 聚合的入口实体，控制内部对象的生命周期，**外部只能通过聚合根访问内部对象**

```
订单聚合
└── Order（聚合根）← 所有操作入口
    ├── OrderItem（实体，依赖 Order 存在）
    └── DeliveryAddress（值对象）
```

**领域服务（Domain Service）**：跨多个实体/值对象的业务逻辑（单个实体无法完成的行为），如"转账"涉及转出账户和转入账户。

**领域事件（Domain Event）**：领域内状态变化时对外发出的通知（`OrderCreated`/`OrderPaid`/`OrderShipped`）。
- **领域事件 ≠ MQ 消息**：领域事件通常在**单个微服务内部传播**，用于服务内部模块解耦；MQ 消息用于跨服务传递
- 通过发布/订阅实现模块松耦合（改变订单状态 → 发布事件 → 库存模块/积分模块各自处理）

### 充血模型 vs 贫血模型

| 维度 | 充血模型（DDD 推荐）| 贫血模型（传统 MVC）|
|------|----------------|----------------|
| **行为归属** | 业务行为封装在领域对象内（Order.pay()）| 所有行为在 Service 层（OrderService.pay()）|
| **数据与行为** | 高内聚，对象自包含状态和行为 | 分离，POJO 只有 getter/setter |
| **代码维护** | 业务意图清晰，易于理解和扩展 | Service 臃肿，业务逻辑分散 |
| **测试** | 单元测试粒度细（直接测领域对象）| 需 mock 大量依赖 |
| **学习曲线** | 高，需深入理解业务 | 低，CRUD 快速上手 |
| **适用场景** | 复杂业务逻辑，需长期维护 | 简单 CRUD，小项目 |

### 分层架构（四层）

```
┌────────────────────────────────┐
│  用户接口层 (User Interface)     │ ← Controller、DTO、VO、接口适配
├────────────────────────────────┤
│  应用层 (Application)           │ ← 用例编排（Application Service）、事务、权限
├────────────────────────────────┤
│  领域层 (Domain)                │ ← 实体、值对象、聚合、领域服务、领域事件
├────────────────────────────────┤
│  基础设施层 (Infrastructure)    │ ← 持久化(Repository实现)、MQ、Redis、HTTP Client
└────────────────────────────────┘
```

**依赖方向严格单向**：上层依赖下层，禁止下层依赖上层。领域层不依赖基础设施层（通过 Repository 接口依赖倒置）。

**变体架构**：
- **洋葱架构（Onion）**：领域模型在最内层，基础设施在最外层，依赖由外到内
- **六边形架构（Hexagonal / Ports & Adapters）**：业务核心在中心，外部系统通过"端口+适配器"接入；天然支持多种输入（HTTP/MQ/CLI）和输出（DB/Redis/第三方 API）

三种架构本质相同：**以领域模型为中心，外部依赖单向指向内部**。

### 统一语言（Ubiquitous Language）

业务人员和开发人员共用同一套术语：
- 不说"users 表的 status 字段"，说"用户的账号状态"
- 不说"order 表 insert 一条记录"，说"下单（create order）"
- 代码中的类名/方法名与业务术语一致（`Order.cancel()`、`InventoryService.deduct()`）

### 限界上下文（Bounded Context）

每个领域概念在特定上下文（Bounded Context）中有明确含义，不同上下文中同一词含义可以不同：
- 商品（Commodity）在商品中心 = 商品详情，在库存中心 = 库存 SKU，在订单中心 = 订单行
- **限界上下文是微服务拆分的理论依据**：一个 Bounded Context 对应一个微服务（或几个关系紧密的微服务）

### CQRS（命令查询职责分离）

**核心思想**：将写操作（Command）和读操作（Query）的模型完全分离：

```
写路径：Command → 领域模型（聚合根校验业务规则）→ 事件/状态变更 → 写库
读路径：Query → 专用查询模型（可直接 SQL/ES/缓存）→ 读库（反范式，为查询优化）
```

**为什么分离**：DDD 的领域模型天然为"写"设计（封装业务规则、维护不变量），但读操作通常需要跨聚合、复杂 JOIN、不同的视图模型——强行用领域模型处理读既性能差又逻辑混乱。

**适用场景**：读写比例极不对称（读远多于写）；读需要聚合多个聚合根数据；需要独立扩展读写性能。

### Event Sourcing（事件溯源）

**核心思想**：不存储当前状态，而是**存储所有状态变更事件序列**；当前状态通过重放（Replay）事件得到。

```
传统：数据库存储 Order{status=SHIPPED, amount=100}
事件溯源：存储事件流
  [OrderCreated{amount=100}, OrderPaid{}, OrderShipped{}]
  当前状态 = 重放以上所有事件
```

**优点**：
- 完整审计日志（每次变更都有记录，满足金融合规要求）
- 支持时间旅行（可以查看任意时刻的历史状态）
- 事件即是领域事件，天然与 CQRS 配合（事件驱动读模型更新）

**缺点**：
- 查询复杂（当前状态需要重放，通常需要 Snapshot 定期快照）
- 存储量大（所有历史事件都保存）
- 事件 Schema 演化困难（历史事件不可修改）

**与 CQRS 的关系**：Event Sourcing 经常与 CQRS 组合（写端存事件，事件投影到读端的查询模型），但二者可以独立使用。

## 关键权衡

1. **DDD 复杂度高 vs 长期价值**：DDD 引入了大量概念（聚合根/值对象/领域事件），初期开发成本更高；但对于业务复杂、长期迭代的系统，代码结构清晰、维护成本低
2. **充血 vs 贫血**：充血模型代码表达力强、业务意图清晰，但要求团队有深入的领域知识；贫血模型简单易理解，适合业务逻辑简单的场景
3. **领域层纯粹性 vs 技术妥协**：领域层理论上不应有 Spring 注解、JPA 注解（破坏纯粹性），但实际项目中往往会有务实的妥协
4. **Bounded Context 粒度**：拆太细导致分布式事务增加；拆太粗上下文内聚性不足。以"一个团队能完整负责"为粒度参考

## 与其他概念的关系

- **支撑 [[机制-微服务与SpringCloud]]**：限界上下文（Bounded Context）是微服务边界划分的理论基础；DDD 的单向依赖原则与微服务的单向依赖原则一致
- **应用 [[机制-设计模式]]**：充血模型天然实现了面向对象的封装性；聚合根模式是外观模式（Facade）的领域层应用；领域事件是观察者模式
- **相关 [[概念-高可用设计]]**：单元化架构中每个单元的边界划分，与 DDD 的 Bounded Context 概念相通

## 应用边界

**适合 DDD**：业务逻辑复杂（金融、电商核心链路、保险）；需要长期演进和维护（3 年以上）；多团队协作（统一语言减少沟通成本）；需要在架构上隔离业务变化。

**不适合 DDD**：纯 CRUD 的管理后台；技术驱动而非业务驱动的系统（如数据管道）；小团队快速验证 MVP 阶段；团队没有 DDD 经验且项目周期短。
