# Redis 集群与高可用

> Redis 高可用有三种模式：主从复制（手动故障转移）、哨兵模式（自动故障转移）、Cluster 集群（分片 + 自动故障转移）；三者能力依次递进，应按数据量和可用性要求选择。

## 第一性原理

单机 Redis 的两个痛点：
1. **单点故障**：主节点挂了，服务不可用
2. **容量上限**：单机内存有限（通常 64-256GB），超出无法横向扩展

高可用的解决思路：**冗余**（主从复制解决单点）+ **自动化**（哨兵/Cluster 解决手动切换）+ **分片**（Cluster 解决容量）。

## 核心机制

### 主从复制（Replication）

```
主节点（Master）: 读写
   │  复制
   ▼
从节点（Slave）: 只读
从节点（Slave）: 只读
```

**全量同步**（新从节点首次接入）：
```
从节点 → 发 PSYNC 命令 → 主节点 BGSAVE → 生成 RDB 文件 → 传输给从节点
从节点加载 RDB → 同步完成
BGSAVE 期间主节点新写命令 → 存入复制缓冲区 → 加载 RDB 后继续发送
```

**增量同步**（断线重连）：
- 主节点维护 `replication buffer`（复制积压缓冲区，环形缓冲区）
- 从节点重连后，发送上次复制的 offset
- 主节点判断 offset 在缓冲区内 → 增量同步；超出缓冲区 → 全量同步

**局限**：主节点故障时需要**手动**执行 `SLAVEOF NO ONE` 提升从节点为主节点。

### 哨兵模式（Sentinel）

在主从基础上，增加哨兵进程（Sentinel）实现**自动故障转移**：

```
        Sentinel-1
       /
Master - Sentinel-2    ← 3 个哨兵形成仲裁集群
       \
        Sentinel-3
        
Slave-1, Slave-2
```

**故障检测与切换流程**：
```
1. 哨兵每秒向 Master 发 PING
2. 超时无响应 → 该哨兵标记 Master 为"主观下线"（SDOWN）
3. 超过 quorum 数量的哨兵都认为下线 → "客观下线"（ODOWN）
4. 哨兵之间选出 Leader（Raft 选举）
5. Leader 选择最优从节点提升为新 Master（选最新 offset 的从节点）
6. 通知其他从节点改为复制新 Master
7. 通知客户端（通过发布订阅），客户端重新获取 Master 地址
```

**局限**：
- 仍是单主，写能力受限于单机
- 主从切换过程有短暂不可用（秒级）
- 容量仍受限于单机内存

### Redis Cluster（分片集群）

**解决容量问题**：将数据分散到多个主节点（分片）。

**哈希槽（Hash Slot）分配**：
```
总槽位：16384 个 slot（固定值）
key → CRC16(key) % 16384 → 决定落在哪个 slot → 由哪个主节点负责

例：3 主节点平均分配
节点 A: slot 0 - 5460
节点 B: slot 5461 - 10922
节点 C: slot 10923 - 16383
```

**为什么是 16384 个槽**：
- 节点数不超过 1000 时，16384 槽足够分配
- 心跳包中 16384 bit 的 bitmap 只占 2KB，不会太大

**每个主节点有若干从节点**（提供高可用）：
```
Master-A  ← Slave-A1, Slave-A2
Master-B  ← Slave-B1
Master-C  ← Slave-C1
```

**客户端 MOVED 重定向**：
```
客户端 → 向节点 A 请求 key → 节点 A 判断 key 属于槽 9000（节点 B 负责）
→ 返回 MOVED 9000 {nodeB-ip:port}
→ 客户端重新向节点 B 发请求
```

智能客户端（如 Lettuce/Jedis Cluster）会缓存槽位映射，避免频繁重定向。

### 三种模式对比

| 维度 | 主从 | 哨兵 | Cluster |
|------|------|------|---------|
| 故障转移 | 手动 | 自动 | 自动 |
| 写扩展性 | 单主 | 单主 | 多主分片 |
| 数据容量 | 单机上限 | 单机上限 | 横向扩展 |
| 复杂度 | 低 | 中 | 高 |
| 适用场景 | 开发/测试 | 中小型生产 | 大数据量生产 |

### 脑裂问题

**场景**：网络分区导致哨兵和从节点认为 Master 宕机，选出新 Master；但旧 Master 实际还活着，仍在接受写请求 → 两个 Master 同时写，数据分裂。

**Redis 解法**（配置项）：
```
min-replicas-to-write 1      # 至少 1 个从节点连接才允许写
min-replicas-max-lag 10      # 从节点延迟超过 10s 则拒绝写
```

旧 Master 在网络分区后发现从节点都断了（超过 lag 阈值）→ 拒绝写请求 → 脑裂期间不接受新数据 → 网络恢复后重新成为从节点，数据丢失最小化。

## 关键权衡

1. **哨兵 vs Cluster**：哨兵运维简单，数据量 <100GB 时足够；Cluster 支持横向扩展，但多节点事务（MULTI/EXEC）、跨 slot 操作受限
2. **槽位数 16384 的固定性**：槽位数固定意味着节点扩容时只是重新分配槽（数据迁移），不改变哈希算法；一致性哈希方案下节点增减只影响相邻节点，但实现更复杂
3. **主从复制的异步性**：主节点写成功即返回，异步复制给从节点；主节点宕机时从节点可能落后 → 切换后数据丢失（最多丢失复制延迟内的数据）

## 与其他概念的关系

- 依赖 [[机制-Redis持久化]]：主从全量同步使用 RDB；增量同步使用类 AOF 的复制缓冲区
- 解决 [[概念-缓存三大问题]] 的雪崩场景：Cluster 多主节点让单节点故障不影响整个缓存层
- 影响 [[机制-Redis分布式锁]]：Cluster 模式下，主从切换可能导致锁信息丢失（新 Master 未同步到锁）；RedLock 是针对此问题的解法

## 应用边界

**如何选型**：
- 数据量 <10GB，高可用要求一般 → 主从 + 哨兵
- 数据量 >100GB 或需要写扩展 → Redis Cluster
- 开发测试环境 → 单机或简单主从

**Cluster 使用注意**：
- `MGET`/`MSET` 涉及多 key 时，所有 key 必须在同一 slot（用 hashtag `{user:1}:name` 和 `{user:1}:age` 保证同一 slot）
- 事务（`MULTI/EXEC`）只能在单节点执行，跨 slot 事务需要应用层处理
- Lua 脚本中的所有 key 必须在同一 slot
