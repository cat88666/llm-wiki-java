# 分库分表与全局ID

> 分库分表是应对单表数据量过大（通常超过 2000 万行，或单库 QPS 超过上限）时的水平拆分方案。它以引入分片键限制、跨表查询困难、运维复杂为代价，换取线性可扩展的存储和读写能力。

## 问题驱动

**面试高频问题组合**：
- 分库分表的分片键如何选择？分表算法是什么？
- 雪花算法如何保证唯一性？时钟回拨问题如何解决？
- 没有分片键时如何查询？如何解决数据倾斜？
- 分表后如何扩容？业务不中断的情况下如何迁移？

## 何时需要分库分表

| 瓶颈类型 | 典型现象 | 解决方案 |
|---------|---------|---------|
| 单表行数过大 | 索引树层数增加，查询慢（> 2000 万行经验值）| 分表（物理分散数据）|
| 单库连接数/QPS 达上限 | 数据库 CPU 100%，连接超时 | 分库（分散连接和读写压力）|
| 数据量大但访问热点不均 | 冷热分离即可 | 归档（历史数据移到历史表）|

**优先考虑替代方案**：归档（热冷分离）、读写分离（读从库）、分布式数据库（TiDB/OceanBase），分库分表是最后手段。

## 核心机制

### 分片键选择原则

1. **高散列性**：分片键的值分布均匀，避免数据倾斜（热点商家/热点用户）
2. **高关联性**：业务查询最常用的字段（如订单表用 user_id，买家查询无需跨分片）
3. **不可变性**：分片键一旦确定不应修改（修改意味着跨分片迁移数据）
4. **不用自增 ID 作为分片键**：全局自增 ID 在多分片中无法保证递增，且分片不均匀

### 分表算法

**Hash 取模**（最常用）：
```
shard_index = hash(user_id) % shard_count  // shard_count 通常取 2^n

为什么先 hash 再取模？
防止 user_id 本身分布不均（如 user_id 连续但有商业含义的规律）
hash 函数（MurmurHash/SHA 等）打散分布，使取模结果更均匀
```

**基因法**（订单号携带分片信息）：
```
user_id = 12345678
user_id % 64 = 10  // 分片 index
订单号 = 时间戳 + 随机数 + "10"  // 最后 2 位是分片基因

查询订单：直接从订单号提取分片 index，不需要 JOIN user_id
适合：买家用 user_id 分表，订单号携带分片基因，卖家维度另建卖家订单表（双写）
```

### 全局唯一 ID：雪花算法

```
64 bit = 1(符号位) + 41(时间戳ms) + 10(机器码/workerID) + 12(序列号)
                      ↑支持69年        ↑最多1024台机器         ↑同毫秒最多4096个ID

何时可能重复：
1. 时钟回拨（NTP 同步导致时间倒退）→ 同毫秒重复
2. workerID 相同的两台机器（容器重启/编号重用）→ 同ms同序列号重复
```

**时钟回拨解决方案**：
1. **抛异常**：拒绝生成 ID，等待时钟追上（简单但影响可用性）
2. **等待**：sleep 到上次时间戳后再生成（适合回拨时间很短）
3. **切换机器码**：回拨期间临时使用备用机器码段，避免等待
4. **预生成 ID**：提前批量生成 ID 缓存到内存（Leaf 等方案）

**workerID 分配**（防容器重启冲突）：使用 Redis 自增（`INCR worker_id`），服务启动时从 Redis 申请唯一 ID。

### 跨分片查询问题

| 查询类型 | 解决方案 |
|---------|---------|
| 按分片键精确查询 | 路由到单个分片，无问题 |
| 非分片键单条查询 | Hint 强制指定分片；或建立二级索引表（id→分片映射）|
| 列表分页查询（无分片键）| 查所有分片后合并（ShardingJDBC 默认）；性能差，加全局索引优化 |
| 卖家维度查询（买家分表）| **双写**：买家维度分表 + 卖家维度冗余表（同步写入）|
| 历史数据扫表 | XXL-Job 分批按分片 id 遍历，每个 Task 处理一个分片 |

### 扩容与迁移

**核心难点**：扩容时分片数从 64 → 128，原 `hash % 64` 路由失效。

**业务不中断迁移方案（双写迁移）**：
```
阶段1：开启双写
  新写入同时写老分片和新分片
  
阶段2：存量数据迁移
  后台任务读取老分片数据写入新分片
  
阶段3：切读
  读请求切换到新分片（对比老分片数据验证一致性）
  
阶段4：停写老分片，清理老数据
```

**防止存量迁移和双写之间丢数据**：迁移时用数据的 `updated_at` 时间戳兜底，迁移完成后再次扫描迁移期间更新的数据。

## 关键权衡

1. **分库 vs 分表**：单库连接数有上限（通常 2000-4000），瓶颈是连接数和写 QPS 时分库；单表数据量大但库级 QPS 可接受时只需分表
2. **2^n 分片数设计**：分片数取 2 的幂次，扩容时从 64 → 128，一半数据原地不动，另一半迁移到新节点（减少迁移量）
3. **ShardingJDBC vs MyCat**：ShardingJDBC 客户端分片，无中间件单点；MyCat 服务端代理，对应用完全透明但多一跳延迟
4. **分片键不能是随机 ID**：UUID 完全随机，会破坏 B+树局部性（[[机制-InnoDB索引]] 中页分裂频繁），推荐单调递增的雪花 ID 作为主键

## 跨层依赖分析

- **L5 MySQL**：[[机制-InnoDB索引]] 的 B+树页分裂是分片键选择的约束（主键推荐递增）；[[机制-MVCC]] 跨分片事务无法用 MVCC，需上层分布式事务解决
- **L5 Redis**：雪花算法的 workerID 分配用 Redis 自增；热点行库存预扣减用 Redis（见 [[概念-Redis]]）
- **L8 分布式事务**：分库分表后跨库事务问题，参见 [[设计-分布式事务]]

## 面试回答要点

1. **先说判断标准**（何时需要分库分表，先考虑归档/读写分离）
2. **分片键选择**：高散列性 + 高关联性 + 不可变，举电商下单场景（user_id）
3. **雪花算法结构**（41+10+12），必提时钟回拨解决方案
4. **基因法**：主动说出跨维度查询的解法，体现工程经验
5. **扩容方案**：双写迁移 + 切读验证，体现大局观
