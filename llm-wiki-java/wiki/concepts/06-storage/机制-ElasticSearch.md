# 倒排索引与ElasticSearch

> ElasticSearch 是基于倒排索引的分布式全文搜索引擎：将文本分词后建立"词条→文档列表"的映射，以亚秒级速度在海量文本中定位匹配文档；以牺牲 ACID 事务和实时一致性为代价，换取极致搜索性能。

## 一、为什么需要倒排索引（B+树的边界）

关系数据库的 B+树索引解决的是"按字段精确/范围查找记录"——文档是键，词条是值。但全文搜索需要反过来问：**哪些文档包含这个词？**

| 数据结构 | 擅长 | 不擅长 |
|---------|------|-------|
| B+树（InnoDB）| 精确匹配 + 范围查询（`WHERE age BETWEEN 20 AND 30`）| 全文搜索（某个词出现在哪些文档）|
| 倒排索引（ES）| "词条→文档集合"的逆向映射，亚秒级全文定位 | 精确事务、复杂 JOIN、强一致读 |

倒排索引（Inverted Index）以词条为键、以文档集合为值——把"正向"的文档→词条关系，翻转为"倒向"的词条→文档关系。

## 二、倒排索引建立过程

```
原始文档（JSON）
    ↓ 分词（Analyzer：standard/ik_max_word/ik_smart 等）
词条列表（Terms）
    ↓ 倒排表压缩（Variable Byte Encoding 等）
倒排索引 = Term → [DocID, Position, Frequency, ...]
```

**示例**：

| 词条 | 文档 ID |
|------|---------|
| Java | 1, 2, 3 |
| 深入 | 1, 2 |
| 虚拟机 | 2 |

搜索"Java 虚拟机"→ 取两个倒排列表的**交集** → 文档 2。

**分词器选型**：中文必须用 IK 分词（`ik_max_word` 细粒度 / `ik_smart` 粗粒度）；standard 分词器对中文逐字切分，搜索效果差。

## 三、ES 为什么快（5个维度）

| 维度 | 机制 |
|------|------|
| **倒排索引** | 词条直接定位文档，无需全表扫描 |
| **分布式分片** | 索引拆成多个 Shard，查询在分片上并行执行后归并 |
| **内存缓存** | 热数据（Segment + OS Page Cache）驻内存，减少磁盘 I/O |
| **预处理** | 写入时预计算分词、评分因子（IDF/TF），查询时不再重算 |
| **近实时异步** | 写入 translog → `refresh` 到内存 Segment（默认 1s）→ 可搜索；异步 flush 到磁盘 |

**经验法则**：每 GB 堆内存不超过 20 个 Shard；Shard 越多写入吞吐越高，但查询需要协调的分片越多反而更慢。

## 四、集群角色与节点分工

| 角色 | 职责 | 注意 |
|------|------|------|
| **Master** | 管理集群状态、索引元数据、分片分配 | 一个集群只有一个活跃 Master，选举防脑裂 |
| **Master-eligible** | 候选节点，必须配奇数（3/5）防脑裂 | 不承担数据节点职责 |
| **Data** | 存储分片（主+副本），执行 CRUD/搜索/聚合 | 可细分 data_hot/data_warm/data_cold |
| **Coordinating** | 接收客户端请求，路由到数据节点，合并结果返回 | 所有节点默认也是 Coordinating |
| **Ingest** | 写入前对文档做预处理（pipeline）| 可选，替代部分 Logstash 功能 |

## 五、并发控制与事务边界

### 乐观锁并发控制

ES 只支持乐观锁，不支持悲观锁。

```
旧：_version（递增整数，6.7 后废弃）
新：if_seq_no + if_primary_term（推荐）
    - seq_no：文档每次变更自动递增的序列号
    - primary_term：主分片切换时递增，配合 seq_no 防止脑裂后旧数据写入
```

更新时携带 `if_seq_no` / `if_primary_term`，与当前值不匹配则返回 409 冲突错误，由业务重试。

### 事务边界：ES 不支持跨文档 ACID

**ES 只保证单个文档操作的原子性**，不支持跨文档事务。

**根本原因**：数据分散在多个分片（不同节点），跨文档 2PC 实现成本极高且严重影响搜索性能；分片局部事务只能保证单分片内一致。

**结论**：ES 适合搜索、日志分析；**不适合**金融交易等强一致性场景——用 [[概念-MySQL]]。

## 六、深度分页：from+size / scroll / search_after

`from + size` 的总和默认不超过 10000（`index.max_result_window`）。深分页时 ES 需要在每个分片上扫描并排序 `from + size` 条记录后丢弃前 `from` 条，随页码增加性能急剧下降。

| 方案 | 原理 | 适用场景 | 缺点 |
|------|------|---------|------|
| `from + size` | 跳过前 N 条记录 | 小数据集，浅分页（< 1000 条）| 深分页性能崩溃，上限 10000 |
| `scroll` | 初始化快照 + scroll_id 游标 | 数据导出/全量遍历 | 不适合实时请求；维护游标消耗内存 |
| `search_after` | 基于上一页最后一条的排序值继续查询 | **生产环境深分页推荐** | 不支持随机跳页；需稳定排序字段（须含全局唯一值如 `_id`）|

## 七、ES 与数据库双写同步

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| 双写（同一事务）| 代码中同时写 DB 和 ES，纳入本地事务 | 简单，实时性高 | 外调拖长事务，超时场景不一致 |
| MQ 异步 + 本地消息表 | 写 DB 后发消息，Consumer 更新 ES | 解耦，异步，最终一致 | 引入 MQ 中间件，存在延迟 |
| 定时扫表 | 定时任务轮询 DB 变更，批量同步 ES | 无侵入 | 实时性差，给 DB 带来额外压力 |
| **binlog 监听（推荐）** | Canal 等框架监听 MySQL binlog，变更推送 ES | 完全无侵入，业务解耦，毫秒级延迟 | 需引入 Canal 等框架 |

**消费幂等**：任一方案的 ES Consumer 必须幂等——ES 文档更新天然是幂等的（同 doc id 写入即覆盖），可用 `if_seq_no` 做乐观锁防止旧数据覆盖新数据。

## 八、Hot-Warm-Cold 与 ILM 生命周期

利用 `data_hot` / `data_warm` / `data_cold` / `data_frozen` 节点角色 + ILM（索引生命周期管理），将热数据放 SSD 快速响应、冷数据迁移到大容量低速磁盘降低成本：

```
Hot 节点（SSD）：接收新写入，高频读写
  → 7天后迁移到 Warm 节点（普通磁盘）：只读，聚合为主
  → 30天后迁移到 Cold 节点（大容量冷存储）：偶尔查询
  → 90天后 Frozen 或删除（极低频访问，解冻才能查询）
```

**ILM 配置**：在 `PUT _ilm/policy/{policy_name}` 中定义各阶段的转换条件（时间/大小）和动作（Rollover/Shrink/Freeze）。

**典型场景**：日志系统（ELK Stack）中 7 天内的日志存 Hot 节点，30 天前的归到 Cold，90 天后自动删除——自动化运维，存储成本降低 80% 以上。

## 九、核心权衡 / 适用边界 / 与 MySQL 的关系

### 核心权衡

| 权衡点 | 说明 |
|--------|------|
| **搜索 vs 事务** | ES 放弃了 ACID，不能替代 RDBMS 存储核心业务数据 |
| **近实时 vs 强实时** | 写入 ES 后默认 1 秒 refresh 才能被搜索，不适合写后立即强一致读 |
| **分片数量 vs 查询性能** | 分片越多写入吞吐越高，但查询需协调的分片越多反而更慢；每 GB 堆内存不超过 20 个分片 |
| **scroll vs search_after** | scroll 保证快照一致性但消耗服务端内存；search_after 无状态但不支持随机跳页 |
| **text vs keyword** | text 分词支持全文搜索但不能精确聚合/排序；keyword 不分词支持精确匹配/排序/聚合；两者经常对同一字段同时配置（`fields` 多字段映射）|

### 适用边界

**适合 ES**：全文搜索（商品、文章、日志）；复杂多条件过滤+评分排序；日志分析（ELK Stack）；大数据量聚合统计；近实时监控。

**不适合 ES**：需要 ACID 事务；需要写后立即强一致读；频繁更新单条记录（ES 更新实质是删旧建新，代价较高）；超大量级精确深分页随机跳页（search_after 不支持跳页）。

### ES 在存储体系的位置

```
MySQL（来源数据，ACID 保障）
  ↓ Canal 监听 binlog（见 [[概念-MySQL]] 十一、Canal）
ES（搜索索引，性能保障）
  → 全文检索、复杂过滤、评分排序

生产范式：MySQL 存源数据 + ES 提供搜索
ES 依赖 [[机制-RabbitMQ]]：MQ 异步同步方案直接依赖消息队列的可靠投递
```
