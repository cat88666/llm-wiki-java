# 缓存三大问题

> 缓存系统面临三大经典问题：穿透（请求不存在的 key，绕过缓存打数据库）、击穿（热点 key 过期瞬间大量请求打数据库）、雪崩（大量 key 同时过期导致数据库瞬间压力暴增）；每种问题成因不同，解决方案不同，且相互独立。

## 第一性原理

缓存的价值是**用内存速度挡住数据库压力**。三大问题本质上都是"缓存挡不住请求，请求打到数据库"，只是原因不同：
- **穿透**：key 根本不存在，缓存永远 miss
- **击穿**：key 存在但在关键时刻失效，并发涌入
- **雪崩**：大批 key 同时失效，全部 miss

理解了根因，解法自然推导出来。

## 核心机制

### 缓存穿透

**场景**：请求一个数据库中不存在的 key（如恶意攻击用不存在的用户 ID）→ 缓存 miss → 查数据库 → 也 miss → 下次继续 miss → 缓存永远无效

**解法 1：布隆过滤器（Bloom Filter）**

```
请求 → 先查布隆过滤器 → 不存在 → 直接返回（不查 DB）
                      → 可能存在 → 查缓存 → 查 DB
```

布隆过滤器用多个哈希函数将 key 映射到 bit 数组（参见 [[概念-BitMap]]）：
- 判断"不存在"永远准确（0 就是不存在）
- 判断"存在"有误判率（多个 bit 都是 1，但可能是其他 key 设置的）
- **不支持删除**（bit 被清 0 会误判其他 key）

**解法 2：缓存空值**

数据库 miss 时，在缓存中存入空值（`NULL` 或特殊标记），设置短 TTL（如 5 分钟）：
- 简单实现，适合随机不存在的 key
- 缺点：恶意攻击用不同的不存在 key → 缓存大量空值，浪费内存

### 缓存击穿

**场景**：某个热点 key 过期，大量并发请求同时 miss，全部打到数据库

**解法 1：互斥锁（Mutex）**

```
缓存 miss → 尝试获取分布式锁
  → 获取成功 → 查数据库 → 回填缓存 → 释放锁
  → 获取失败 → 等待/重试 → 再查缓存（此时已被第一个线程填好）
```

保证同一时刻只有一个请求重建缓存，其他请求等待。缺点：等待期间响应变慢。

**解法 2：热点 key 不设 TTL（逻辑过期）**

热点数据永不过期，通过后台定时任务异步更新缓存。缺点：可能读到短暂过期的旧数据。

**解法 3：提前预热 + 主动续期**

监控 key 的 TTL，在过期前主动续期或重建缓存，不等到过期后再处理。

### 缓存雪崩

**场景 1**：大量 key 设置了相同 TTL，同时过期 → 同时 miss → 大量请求打 DB

**解法**：**随机化 TTL**（在基础 TTL 上加随机偏移）
```python
ttl = base_ttl + random.randint(0, 300)  # ±5分钟随机
```

**场景 2**：Redis 服务宕机 → 所有请求打 DB

**解法**：
- **Redis Cluster 集群**：主节点故障时从节点自动接管（参见 [[机制-Redis集群与高可用]]）
- **熔断降级**：DB 压力超阈值时拒绝部分请求，返回兜底数据，保护数据库不过载
- **多级缓存**：本地缓存（Caffeine）+ Redis，Redis 宕机时本地缓存兜底

### 缓存与数据库一致性

| 策略 | 操作顺序 | 分析 |
|------|---------|------|
| 先更新 DB，再删缓存 | UPDATE DB → DELETE cache | 推荐。短暂不一致（DB 新，缓存旧）；删除失败可重试 |
| 先删缓存，再更新 DB | DELETE cache → UPDATE DB | 有脏读风险：删后有线程查缓存 miss → 读旧 DB → 回填旧值 |
| 延迟双删 | DELETE cache → UPDATE DB → sleep → DELETE cache | 防止上述脏读，但"sleep 多久"难以确定 |
| Binlog 异步删缓存 | 监听 binlog（Canal）→ 异步删 cache | 最终一致，解耦，推荐大规模场景 |

**最终一致性方案（推荐）**：
```
应用 → UPDATE MySQL → Canal 监听 binlog → 发消息 → 消费者删除 Redis cache
```

### 内存淘汰策略（8 种）

当 Redis 内存用满时（达到 `maxmemory`），需要淘汰部分 key：

| 策略 | 范围 | 淘汰规则 |
|------|------|---------|
| `noeviction` | 全部 | 不淘汰，写操作直接报错 |
| `allkeys-lru` | 全部 key | LRU（最近最少使用）|
| `allkeys-lfu` | 全部 key | LFU（最近最少频率，4.0+）|
| `allkeys-random` | 全部 key | 随机淘汰 |
| `volatile-lru` | 有 TTL 的 key | LRU |
| `volatile-lfu` | 有 TTL 的 key | LFU（4.0+）|
| `volatile-random` | 有 TTL 的 key | 随机淘汰 |
| `volatile-ttl` | 有 TTL 的 key | TTL 越短越先淘汰 |

**生产选择**：
- 缓存场景 → `allkeys-lru` 或 `allkeys-lfu`（LFU 对热点数据保护更好）
- Redis 中有不能丢的数据 → `volatile-lru`（不淘汰无 TTL 的 key）

## 关键权衡

1. **布隆过滤器 vs 缓存空值**：布隆过滤器节省内存但有误判率，适合 key 空间巨大（亿级）；缓存空值实现简单但占内存，适合偶发的不存在 key
2. **互斥锁 vs 逻辑过期**：互斥锁数据强一致但性能低（等锁）；逻辑过期性能高但有短暂不一致窗口
3. **强一致 vs 最终一致**：先删缓存 + 等待更新是强一致的努力但仍有窗口；Binlog 异步是最终一致，适合大规模分布式场景

## 应用边界

**面试标准答法**（三大问题定义+解法）：
- 穿透：key 不存在 → 布隆过滤器 / 缓存空值
- 击穿：热点 key 过期 → 互斥锁 / 永不过期 + 异步更新
- 雪崩：大量 key 同时过期 → 随机 TTL；Redis 宕机 → 集群 + 熔断降级

**不要混淆**：
- 穿透 ≠ 击穿：穿透是请求根本不存在的数据；击穿是热点数据的缓存失效
- 雪崩 ≠ 穿透：雪崩是批量失效；穿透是单个不存在 key 持续 miss
