# 消息队列可靠性

> 消息队列的核心价值：异步解耦 + 削峰填谷。可靠性保证需从三段入手：生产者确认（消息不丢进 MQ）→ 消息持久化（MQ 不丢）→ 消费者 ACK（消费不丢）。

## 第一性原理

消息队列解决的问题：**让生产者和消费者在时间和空间上解耦**。但解耦带来的代价是消息可能在任何一个环节丢失，且消费者可能重复收到消息。可靠性设计就是在"最多一次"、"至少一次"、"恰好一次"三种语义中选择并实现"至少一次 + 业务幂等"。

## 核心机制

### RabbitMQ 整体架构

```
Producer → Exchange（交换机）→ Queue（队列）→ Consumer
                ↑
            Binding（路由规则）
```

**Exchange 类型**（路由策略）：

| 类型 | 路由规则 | 场景 |
|------|---------|------|
| Direct | 精确匹配 RoutingKey | 点对点路由 |
| Fanout | 广播到所有绑定队列 | 发布/订阅（如日志广播）|
| Topic | 通配符匹配（`*` 单词，`#` 多词）| 复杂路由（如订单.*.成功）|
| Headers | 匹配消息 headers | 少用 |

**VHost**：虚拟主机，实现资源隔离（类似 DB 的 database 概念）。

### 消息可靠性三段保证

```
【生产者端】Publisher Confirm（到 Exchange）+ Publisher Returns（Exchange → Queue）
    ↓
【MQ 侧】队列/交换机/消息持久化（durable=true，deliveryMode=2）
    ↓
【消费者端】手动 ACK / NACK（处理成功后确认，失败后重投）
```

**生产者确认（Publisher Confirm）**：
- 消息到达 Exchange → MQ 发送 ACK；Exchange 处理失败 → 发送 NACK
- Publisher Returns：消息无法路由到队列时，MQ 将消息返回给生产者
- 失败时：记录日志、报警、重试

**消息持久化**：
- Queue 声明 `durable=true`：重启后队列元数据不丢
- 消息 `deliveryMode=2`：消息写入磁盘
- 注意：持久化是异步的，极端情况（内存暂存→磁盘写入之前宕机）仍可能丢消息；要 100% 不丢需引入本地消息表

**消费者确认（Consumer ACK）**：
- 关闭自动 ACK，处理成功后手动 `channel.basicAck()`
- 失败时 `channel.basicNack()`，MQ 重新投递

### 消费者限流

```java
channel.basicQos(10);  // 最多 10 条未 ACK 的消息同时推送
```

高峰期消息堆积时，防止消费者被大量消息打挂。配合手动 ACK 使用。

### 防重复消费（幂等）

消费者 ACK 因网络延迟丢失时，MQ 会重投消息。解决：

```
生产者：消息体中携带唯一 messageId
消费者：一锁（分布式锁）+ 二判（查 DB/Redis 是否已处理）+ 三更新（标记已处理）
```

### 死信队列（DLQ）

消息成为"死信"的条件：TTL 过期 / 队列满拒绝 / 消费者 Reject 且不重投。

死信进入 DLX（死信交换机）→ 路由到死信队列 → 可二次处理/报警/分析。

**延迟消息实现**：

| 方案 | 原理 | 优缺点 |
|------|------|--------|
| 死信队列 + TTL | 给消息设置 TTL，过期进入 DLQ | 简单，但有队头阻塞（只检查队首是否过期）|
| rabbitmq_delayed_message_exchange 插件 | 延迟消息存 Mnesia DB，定时投递 | 无阻塞，但最大支持 ~49 天；不适合数百万量级延迟消息 |

### 高可用模式

| 模式 | 数据分布 | 高可用性 |
|------|---------|---------|
| 普通集群 | 消息只在一个节点，元数据同步到全集群 | 节点宕机时消息不可读（该节点数据丢失）|
| **镜像集群** | 消息 + 元数据同步到所有节点 | 任意节点宕机，其他节点有完整数据 |
| Quorum Queue（3.8+）| 基于 Raft 的仲裁队列，取代镜像集群 | 推荐的高可用方案 |

## 关键权衡

1. **持久化 vs 性能**：`deliveryMode=2` 会增加磁盘 I/O；高吞吐场景可接受少量丢失时可用非持久化
2. **死信队列 vs 插件延迟消息**：DLQ 有队头阻塞缺陷；插件不支持超大量级延迟消息；超复杂延迟场景考虑 RocketMQ（原生支持任意时延）
3. **幂等性由消费者负责**：MQ 本身只保证"至少一次"投递，业务层必须实现幂等
4. **镜像集群写性能**：写消息需同步所有镜像节点，写 QPS 受节点数影响

## 与其他概念的关系

- 消息持久化思路与 [[机制-Redis持久化]]（RDB/AOF）类似：都是内存暂存 + 异步落盘
- 消费者幂等性解法（一锁二判三更新）与 [[概念-缓存三大问题]] 中的缓存击穿双重检测相通
- 延迟消息在 L8 工程实践中用于订单超时关闭、消息重试等场景

## 应用边界

**适合 RabbitMQ**：复杂路由（Topic/Direct Exchange）；需要精细 ACK 控制；Java 生态集成（Spring AMQP）。

**不适合**：超大规模吞吐量（Kafka 更适合）；精确一次语义（MQ 基本只做至少一次，需业务幂等兜底）；超长延迟（> 49 天，插件不支持）。
