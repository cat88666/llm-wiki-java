# 三高体系：高并发 · 高可用 · 高性能

> P8 核心差距：P7 分类掌握三高；P8 能说清**三者如何相互制约**。

## 三者制约关系

```
高并发 → 引入异步/分片/分布式 → 牺牲强一致（最终一致）
高可用 → 引入副本/冗余/多活  → 牺牲延迟（性能下降）
高性能 → 引入缓存/本地计算   → 牺牲一致性（脏数据风险）
```

**核心矛盾**：三高互相增压，提升任意一高通常需要在另外两个维度付出代价。设计时需要明确**业务容忍边界**，而非追求三者极值。

---

## 高并发（Concurrency）

> 核心问题：如何让系统在大量并发请求下不崩溃？

### 限流

**目的**：保护系统不被超量请求打垮，核心是"拒绝"或"延迟"超出容量的请求。

| 算法 | 原理 | 突发流量 | 适用场景 |
|------|------|---------|---------|
| 计数器（固定窗口） | 时间窗口内计数，满则拒绝 | 临界突刺（窗口边界） | 简单场景 |
| 滑动窗口 | 时间窗连续滑动，精准控制 | 平滑应对 | API 限流 |
| 漏桶 | 以固定速率"漏出"请求 | 无法应对 | 平滑输出（MQ 消费）|
| 令牌桶 | 固定速率生产令牌，可积累 | 可突发（桶内令牌数） | 接口限流（Guava RateLimiter）|
| 自适应限流 | 结合 CPU/Load/RT/QPS 动态调整阈值 | 自动降配 | Sentinel 系统保护 |

**单机 vs 集群限流**：
- 单机：Guava RateLimiter，保护单个节点不过载
- 集群：Redis + Sentinel，保护全局总量（但需同时保留单机限流，防止流量倾斜打爆单机）
- **两者都需要**：集群限 1000 不代表每台能抗 100，负载不均时可能单机被打爆

### 熔断

**三态状态机**：关闭（正常）→ 开启（全拒绝）→ 半开启（允许少量探测）

```
请求失败率/慢响应超阈值 → 开启熔断
半开启状态通过 → 恢复关闭
半开启状态失败 → 重新开启
```

**工具**：Sentinel（推荐）> resilience4j > Hystrix（已停更）

**为什么需要熔断**：重试 → 下游雪上加霜 → 服务雪崩。熔断使上游快速失败，给下游喘息时间。

### 锁策略

**高并发下悲观锁 > 乐观锁**（反直觉）：

乐观锁"先干活后加锁"——在并发冲突频繁时，浪费大量计算资源组装模型，最后发现版本号变了，全部白费。

悲观锁"先加锁再干活"——fail-fast，加锁失败立即返回失败，不浪费后续计算。

> 结论：高并发写场景优先用分布式锁（悲观思想），而非乐观锁重试。

### 预热

- **缓存预热**：大促前将热点数据提前加载到缓存，避免冷启动时 cache miss 穿透打爆 DB
- **JIT 预热**：JVM JIT 编译需要"热身"，刚发布时 RT 偏高，全链路压测需排除 JIT 影响
- **连接预热**：线程池、DB 连接池冷启动时没有满连接，需要脉冲式逐步施压

## 高可用（Availability）

> 核心问题：如何让系统在部分组件失败时仍然可以服务？

### 可用性指标（SLA）

| 可用性 | 年宕机时间 | 典型叫法 |
|--------|----------|---------|
| 99% | 87.6h | 2 个九 |
| 99.9% | 8.76h | 3 个九 |
| 99.99% | 52.6min | 4 个九 |
| 99.999% | 5.26min | 5 个九 |

**SLA 拆解**：整体可用性 = 各组件可用性的乘积。10 个 99.9% 的组件串联 → 99%。

### 备份策略

| 类型 | 描述 | 恢复时间 | 成本 |
|------|------|---------|------|
| 冷备 | 停机后备份到离线介质 | 长（小时级）| 低 |
| 暖备 | 定期同步，需手动切换 | 中 | 中 |
| 热备 | 实时同步，自动切换（如 MySQL 主从）| 短（秒级）| 高 |

### 异地多活 vs 同城容灾

| | 同城容灾 | 异地多活 |
|---|---|---|
| 目标 | 机房级灾难恢复 | 地域级灾难恢复 |
| 数据同步延迟 | 低（同城高速专线）| 高（跨地域 WAN）|
| 成本 | 中 | 高 |
| 适用 | 99.99% 可用性要求 | 99.999%+ 或监管合规 |

异地多活难点：
1. **数据同步与冲突解决**（分布式系统 CAP 的 P 不可避免）
2. **流量路由**（DNS 切换、单元化架构）
3. **本地写远端读**（主地域写，异地只读，减少跨地域同步问题）

### 压测

**压测五步骤**：确定目标 → 制定计划 → 准备脚本和数据 → 执行施压 → 观察指标。

**监控指标**：QPS、RT、失败率、CPU 利用率、Load、内存、GC 次数/时长、网络 IO、报警。

**全链路压测 vs 单链路压测**：
- 单链路：关注自身，结果偏乐观，忽略资源竞争
- 全链路（阿里双11来源）：从 Nginx → 应用 → 中间件 → DB 全链路验证，暴露真实瓶颈
- 关键：**流量染色**（full_chain_stress_tag）+ **影子表**（order__shadow，压测数据写影子表）

**高可用架构设计要点**：
1. 集群 + 负载均衡（消除单点）
2. 限流降级熔断（防止雪崩）
3. 多活/异地容灾（地域级容灾）
4. 监控 + 自动化（快速发现和恢复）
5. 容量规划（提前知道系统水位）

## 高性能（Performance）

> 核心问题：如何让系统在相同资源下处理更多请求、响应更快？

### 性能指标

| 指标 | 含义 | 典型值 |
|------|------|--------|
| QPS | 每秒查询数 | 核心接口 ≥1000 |
| RT | 响应时间（平均） | 核心接口 ≤100ms |
| P90/P95/P99 | 90%/95%/99% 请求的最大 RT | P99 ≤500ms 是常见 SLA |

> P99 比平均 RT 更重要：平均 RT=50ms 可能掩盖了 1% 的请求耗时 5s（超时！）

**单机 QPS 估算**：`QPS = 线程数 × 1000 / RT(ms)`，例：200 线程，RT=100ms → 单机 2000 QPS

### 服务端性能优化方案

1. **缓存**：本地缓存（Caffeine）→ 分布式缓存（Redis）→ 减少 DB 访问
2. **异步**：CompletableFuture 并行编排、@Async 异步处理，减少串行等待
3. **批量**：批量插入（减少 DB 连接次数）、批量 RPC（减少网络开销）
4. **连接池复用**：DB 连接池、HTTP 连接池（避免频繁建立 TCP）
5. **NIO/Netty**：非阻塞 I/O，单线程处理多连接，避免线程切换开销
6. **锁优化**：减少锁粒度（ConcurrentHashMap 分段锁）、锁分离（ReadWriteLock）、锁消除（JIT）
7. **SQL 优化**：索引、深分页优化、批量操作代替循环单条
8. **压缩传输**：HTTP Gzip，减少网络带宽

### 线程池调优

**核心参数**：`corePoolSize`、`maximumPoolSize`、`workQueue`、`handler`

**初始值公式**：
- CPU 密集型：`corePoolSize = N+1`（N=CPU 核数）
- IO 密集型：`corePoolSize = 2N ~ 4N`

**调整信号**：
- 提升线程数：队列长 + 任务拒绝多 + CPU 利用率不高
- 降低线程数：上下文切换频繁 + CPU 利用率高 + 活跃线程少

### 读写分离与主从延迟

**读写分离**：写主库，读从库，以牺牲**实时一致性**换取**读吞吐提升**。

**主从延迟处理**：

| 方案 | 适用 | 代价 |
|------|------|------|
| 强制读主库 | 强一致性业务（支付、下单后查询）| 主库压力增加 |
| 二次读取（从→主）| 弱一致业务容错 | 主库偶发压力 |
| sleep/等位点方案 | 教科书方案 | 延迟高，生产少用 |
| **读请求分类** | **推荐！** 历史数据走从库 | 业务改造 |

> 实际场景：事务上下文内的读必须强制走主库（同一事务 insert 后立即 select）。

### 布隆过滤器 & 布谷鸟过滤器

**布隆过滤器**：多个 hash 函数 → 位数组，空间效率极高，但：
- **假阳性**（说存在 → 可能不存在），无假阴性（说不存在 → 一定不存在）
- **不支持删除**（解决：计数布隆过滤器 3-4 位/位 or 定期重建）
- 误判率 = f(位数组大小, hash 函数数量)，典型设置：误判率 1% 对应每元素约 9.6 bit

**布谷鸟过滤器**（Cuckoo Filter）：
- 支持删除 ✅，误判率更低，但更复杂
- 原理：两张 hash 表 + 指纹存储，冲突时"踢出"对方（布谷鸟习性）
- 负载率高（>90%）时插入性能下降

**选型**：
- 不需要删除 → 布隆过滤器（简单）
- 需要删除 → 布谷鸟过滤器（或定期重建布隆过滤器）

## 三高联动分析

### 场景：秒杀系统

| 维度 | 措施 | 代价 |
|------|------|------|
| 高并发 | 令牌桶限流、分桶库存 | 多桶管理复杂度 |
| 高可用 | 熔断降级（库存不足时降级展示售罄）| 少量用户无法下单 |
| 高性能 | Redis 预扣/Inventory Hint | 一致性风险或需内核支持 |

核心权衡：Redis 预扣（高性能）vs MySQL 可靠性（高可用）——见 [[系统设计-电商系统]] 中的大厂实践补充。

### 场景：读多写少（商品详情）

| 维度 | 措施 | 代价 |
|------|------|------|
| 高并发 | 多级缓存拦截大部分读请求 | 缓存一致性问题 |
| 高可用 | 缓存击穿/雪崩防护（布隆过滤器+互斥锁）| 首次加载延迟 |
| 高性能 | 读写分离（强制主库仅用于写后立即读）| 主库偶发压力 |

### 场景：高 TPS 写入（日志/流水）

| 维度 | 措施 | 代价 |
|------|------|------|
| 高并发 | 批量写入、MQ 削峰 | 实时性降低 |
| 高可用 | WAL + 异步持久化 | 崩溃时可能少量数据丢失 |
| 高性能 | LSM 树（写优化）| 读性能下降，需 Compaction |
