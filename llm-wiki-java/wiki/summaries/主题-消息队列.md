# 消息队列体系

消息队列的核心价值：**解耦、异步、削峰**。本页提供消息队列三大主流方案的对比视图和选型指南。

---

## 三大框架全景对比

| 维度 | Kafka | RabbitMQ | RocketMQ |
|------|-------|----------|---------|
| 设计定位 | 高吞吐流平台 | 灵活路由任务队列 | 金融级可靠消息 |
| 消息模型 | 发布订阅 + offset | AMQP（Exchange → Queue）| 发布订阅 + Tag 过滤 |
| 吞吐量 | 百万级 TPS | 万级 TPS | 十万级 TPS |
| 延迟 | ms 级（批处理） | μs~ms 级 | ms 级 |
| 消费后删除 | 否（保留，按 offset 消费）| 是（ACK 后删除）| 是 |
| 消息回放 | 支持（调整 offset）| 不支持 | 不支持（有限支持）|
| 延迟消息 | 不直接支持 | 支持（死信/插件）| 支持（固定级别）|
| 事务消息 | 支持（Exactly-Once）| 支持（AMQP 事务）| 支持（半消息机制）|
| 顺序消息 | Partition 内有序 | 不保证 | Queue 内有序 |
| 主要语言生态 | Java/Scala（大数据）| Erlang（多语言友好）| Java（阿里系）|
| 典型用户 | LinkedIn、大数据平台 | 金融、微服务 | 阿里、电商 |

## 消息可靠性三道防线（通用模式）

```
① 生产者侧：确认机制
   Kafka:      ACK=all（ISR 全部确认）
   RabbitMQ:   Publisher Confirm + Publisher Returns
   RocketMQ:   同步/异步发送确认

② 存储侧：持久化
   Kafka:      多副本（replication-factor）
   RabbitMQ:   队列 durable + 消息 deliveryMode=2
   RocketMQ:   主从同步刷盘

③ 消费者侧：ACK 确认
   Kafka:      手动提交 offset
   RabbitMQ:   手动 ACK（basicAck）
   RocketMQ:   消费成功后返回 CONSUME_SUCCESS
```

**100% 不丢的终极方案**：本地消息表（事务消息表 + 定时轮询重投），所有框架通用。

---

## 消息幂等（防重复消费）

重复消费根因：网络延迟导致 ACK 超时 → MQ 重投。

通用解法：**"一锁、二判、三更新"**
1. 消息携带唯一业务 ID（messageId）
2. 消费时加分布式锁或数据库唯一键
3. 查询已消费记录（二判）
4. 处理并更新消费状态（三更新）

---

## 消息积压处理

```
场景：消费者宕机或消费速度 < 生产速度 → 大量消息积压

处理方案：
1. 临时扩容消费者实例（Kafka 受 Partition 数限制）
2. 提高消费者消费速率（批量消费、异步处理）
3. 跳过积压（有业务允许时），从最新 offset 开始消费
4. 消费限流（RabbitMQ basicQos）防止消费者被打垮
```

## 选型决策树

```
需要消息回放/流处理/大数据集成？
  → Kafka

对消息顺序性要求严格（如订单状态机）？
  需要金融级可靠 + 延迟消息？
    → RocketMQ
  灵活路由 + 延迟消息？
    → RabbitMQ

已有 Spring Cloud Alibaba / Dubbo 生态？
  → RocketMQ 或 Kafka（阿里系）

通用企业应用、消息量中等（< 万级 TPS）？
  → RabbitMQ

日志采集、用户行为追踪、实时计算管道？
  → Kafka
```

## 面试追问

**问**：支付和游戏日志场景，RocketMQ 和 Kafka 各取哪个？
**答**：

| 场景 | 选择 | 原因 |
|------|------|------|
| 支付事务消息（精确一次语义）| RocketMQ | 半消息机制 + 回查保障本地事务与消息强一致 |
| 延迟/定时消息 | RocketMQ | 内置延迟级别，Kafka 原生不支持 |
| 消费失败重试 + 死信队列 | RocketMQ | 内置重试机制（默认 16 次），Kafka 需自实现 |
| 游戏日志 / 用户行为大吞吐 | Kafka | 百万级 TPS，顺序磁盘写，大数据生态集成 |

**问**：消息堆积了怎么处理？
**答**：**止血**：扩 Consumer 实例（Kafka 受 Partition 数限制，超过 Partition 数的 Consumer 实例无效）；若 Partition 不够，新建扩容 Topic，写转发消费者把积压消息搬过去再扩消费者并行消费；降级非核心逻辑（如排行榜更新延迟处理）。**根因**：慢 SQL / 外部依赖阻塞；`consumeMessageBatchMaxSize` 调大批量消费；提前设延迟告警阈值（消费时延 > 5min）。
