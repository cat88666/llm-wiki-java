# 源码体系 — Java 后端核心框架源码地图

> 本页不重复概念定义，专注**源码入口 → 关键调用链 → 面试必答结论**三层结构，覆盖 Java 核心类库、并发包、Spring 全家桶、MyBatis、Tomcat 五大域。

---

## 一、Java 核心类库

### 1.1 String

```
内部存储（JDK 8）：private final char[] value
内部存储（JDK 9+）：private final byte[] value + byte coder（Latin1/UTF-16 自适应）
```

| 关键源码路径 | 结论 |
|------------|------|
| `String.intern()` | StringTable（哈希表）去重；JDK 7 把 StringTable 移到堆，避免 PermGen OOM |
| `String.concat()` | new char[] 拷贝，O(n)，循环内禁用 |
| `StringBuilder.append()` | 扩容：`newCapacity = (value.length << 1) + 2`（2倍+2） |
| JDK 9 字符串拼接优化 | `+` 编译为 `invokedynamic` 调用 `StringConcatFactory`，比 JDK 8 的 StringBuilder 更灵活 |

**不可变性关键点**：`value` 是 `private final`，没有 setter，`substring` 等方法都返回新对象。

### 1.2 Integer / 包装类

```java
// Integer.valueOf 源码片段
public static Integer valueOf(int i) {
    if (i >= IntegerCache.low && i <= IntegerCache.high)  // [-128, 127]
        return IntegerCache.cache[i + (-IntegerCache.low)];
    return new Integer(i);
}
```

- 缓存范围：`-128 ~ 127`，`Integer.high` 可通过 `-XX:AutoBoxCacheMax=<n>` 调整
- `==` 比较的是对象引用；缓存范围内 `==` 为 true，超出范围为 false → **永远用 equals**

### 1.3 ArrayList

```
关键字段
  Object[] elementData       ← 底层数组，实际存储
  int size                   ← 已存元素数

扩容调用链（add 触发）
  add() → ensureCapacityInternal() → grow()
  grow(): newCapacity = oldCapacity + (oldCapacity >> 1)  // 1.5 倍
          Arrays.copyOf(elementData, newCapacity)         // 数组拷贝

序列化：elementData 是 transient，writeObject 只序列化 [0, size) 范围
subList：返回 SubList 视图（共享底层数组），修改互相可见，不可序列化
```

### 1.4 LinkedList

```
节点结构
  static class Node<E> {
      E item;
      Node<E> next;
      Node<E> prev;
  }

关键字段：first（头）、last（尾）、size
addFirst/addLast：O(1)；get(index)：O(n) 从头或尾二分查找方向再遍历
```

**面试结论**：ArrayList 随机访问 O(1)，中间插入 O(n)；LinkedList 头尾操作 O(1)，随机访问 O(n)；实际因缓存局部性 ArrayList 性能通常更好。

### 1.5 HashMap（JDK 8）

```
关键字段
  Node<K,V>[] table           ← 哈希桶数组，延迟初始化
  int size                    ← 键值对数量
  int threshold               ← 扩容阈值 = capacity × loadFactor (0.75)
  int modCount                ← 修改次数（fail-fast）

put 调用链
  put() → putVal() → hash(key)
    ├── table == null → resize()（初始化 16）
    ├── 桶位空 → 直接放
    ├── key 相同 → 覆盖 value
    ├── 桶位是 TreeNode → 红黑树 put
    └── 链表尾插 → 长度 ≥ 8 且 table.length ≥ 64 → treeifyBin()

hash 扰动（高低位异或）
  static final int hash(Object key) {
      int h;
      return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
  }
  → 降低低位 hash 碰撞概率，容量小时高位参与寻址

resize 扩容
  newCap = oldCap << 1（2 倍）
  高低位拆分：(e.hash & oldCap) == 0 → 留在原位；== 1 → 移到 index + oldCap
  → 不需要重新 hash，O(n) 完成迁移
```

详见 → [[机制-HashMap]]

### 1.6 LinkedHashMap

```
继承 HashMap，Node 扩展为双向链表节点
  static class Entry<K,V> extends HashMap.Node<K,V> {
      Entry<K,V> before, after;
  }

accessOrder=false（默认）：按插入顺序
accessOrder=true           ：按访问顺序（LRU 基础）

afterNodeAccess()：访问时将节点移到链表尾
afterNodeInsertion()：插入时调用 removeEldestEntry()，子类可覆盖实现 LRU 淘汰
```

**LRU 实现**：继承 LinkedHashMap，构造传 `accessOrder=true`，覆盖 `removeEldestEntry()` 当 `size > capacity` 时返回 `true`。

### 1.7 ConcurrentHashMap（JDK 8）

```
关键字段
  volatile Node<K,V>[] table
  volatile Node<K,V>[] nextTable   ← 扩容时的新数组
  volatile long baseCount
  volatile int sizeCtl             ← -1:初始化中; -N:N-1个线程参与扩容; 正数:下次扩容阈值

put 调键路径
  putVal()
    ├── table 未初始化 → initTable()（CAS sizeCtl=-1 争初始化权）
    ├── 桶位为空 → CAS 直接写入（无锁）
    ├── hash == MOVED → helpTransfer()（协助扩容）
    └── 桶位非空 → synchronized(桶头节点) → 链表/红黑树 put

size 统计（LongAdder 思想）
  baseCount + CounterCell[] 分段累加，避免单点竞争
```

**JDK 7 vs JDK 8**：JDK 7 用 Segment（继承 ReentrantLock）分段锁，最多 16 个 Segment；JDK 8 改为 CAS+桶头节点 synchronized，并发度 = 数组长度，大幅提升。

详见 → [[机制-HashMap]]

## 二、Java 并发包

### 2.1 synchronized 底层（JVM 层）

```
对象头 Mark Word（64位 JVM，8字节）
  ┌──────────────────────────────────────────────┐
  │ 无锁：hashCode(31) | age(4) | biased(1)=0 | tag=01 │
  │ 偏向：threadId(54) | epoch(2) | age(4) | 1 | 01    │
  │ 轻量：ptr_to_lock_record(62) | 00                   │
  │ 重量：ptr_to_heavyweight_monitor(62) | 10            │
  │ GC  ：(62) | 11                                     │
  └──────────────────────────────────────────────┘

ObjectMonitor（C++ 实现）
  _owner     → 持锁线程（Thread*）
  _count     → 重入计数
  _EntryList → BLOCKED 线程（FIFO 双向链表）
  _WaitSet   → wait() 后的线程集合

monitorenter 字节码：CAS 尝试将 Mark Word 指向当前线程栈帧的 Lock Record
monitorexit  字节码：释放 Lock Record，唤醒 EntryList 头节点
```

### 2.2 AQS 核心源码

```java
// ReentrantLock.NonfairSync.tryAcquire 非公平尝试
protected final boolean tryAcquire(int acquires) {
    return nonfairTryAcquire(acquires);
}
final boolean nonfairTryAcquire(int acquires) {
    final Thread current = Thread.currentThread();
    int c = getState();
    if (c == 0) {                                       // 锁空闲
        if (compareAndSetState(0, acquires)) {          // CAS 直接抢
            setExclusiveOwnerThread(current);
            return true;
        }
    } else if (current == getExclusiveOwnerThread()) {  // 重入
        setState(c + acquires);
        return true;
    }
    return false;
}

// 公平锁额外的 hasQueuedPredecessors() 检查
// → 同步队列中是否有前驱节点，有则乖乖排队
```

```
acquire 完整路径
  lock() → acquire(1)
    → tryAcquire()  成功 → 直接返回
    → 失败 → addWaiter(Node.EXCLUSIVE)  入队
    → acquireQueued()
        → shouldParkAfterFailedAcquire()  设前驱 SIGNAL
        → parkAndCheckInterrupt()  LockSupport.park(this)
    
release() → tryRelease()
    → state -= 1；归零 → unparkSuccessor(head)  唤醒下一节点
```

### 2.3 ThreadPoolExecutor.execute 流程

```java
public void execute(Runnable command) {
    int c = ctl.get();
    if (workerCountOf(c) < corePoolSize) {        // ①核心线程未满
        if (addWorker(command, true)) return;
    }
    if (isRunning(c) && workQueue.offer(command)) { // ②入队
        // double-check
        if (!isRunning(recheck = ctl.get()) && remove(command))
            reject(command);
        else if (workerCountOf(recheck) == 0)
            addWorker(null, false);
    } else if (!addWorker(command, false))          // ③创建非核心线程
        reject(command);                            // ④拒绝
}
```

```
Worker 内部类
  extends AbstractQueuedSynchronizer   ← 自身就是一个独占锁（不可重入）
  implements Runnable
  
  run() → runWorker(this)
    → 循环 getTask()（从 workQueue 取任务）
    → task.run()
    → getTask() 返回 null（超时/停止）→ 退出

线程复用关键：Worker 构造传入 firstTask，之后 runWorker 循环 getTask()
非核心线程回收：getTask() 中 poll(keepAliveTime) 超时返回 null，Worker 退出
```

详见 → [[机制-线程池]]

### 2.4 ThreadLocal 内存模型

```
Thread
  └── ThreadLocalMap threadLocals        ← 每个线程独立的 Map

ThreadLocalMap
  └── Entry[] table                      ← 开放寻址散列表
        Entry extends WeakReference<ThreadLocal<?>>
          key   = WeakReference(ThreadLocal)  ← 弱引用 key
          value = 实际值                      ← 强引用 value

关键操作
  set()：threadLocalHashCode & (len-1) 定位，线性探测解冲突
  get()：定位后调用 expungeStaleEntry() 顺带清理 key=null 的 entry
  remove()：必须显式调用，否则 value 强引用泄漏（线程池中线程复用）

内存泄漏路径
  线程池线程长生命周期 → ThreadLocalMap 不销毁 → key GC（弱引用断）
  → value 失去引用但未清理 → 泄漏
```

详见 → [[概念-ThreadLocal]]

## 三、Redis / Redisson 分布式锁源码

> 本节专注 **Lua 脚本逐行解析 + Pub/Sub 等待原理 + watchdog 实现 + 公平锁/读写锁结构**，补充概念页未覆盖的源码细节。

### 3.1 基础实现：原子命令 + Lua 解锁

**加锁**（一条原子命令，无需 Lua）

```
SET lock:key {uuid} PX 10000 NX
  PX 10000 → 过期时间 10000ms（毫秒级精度）
  NX       → Not eXists，key 不存在才 SET（CAS 语义）
  value    → UUID：标识锁持有者，防止释放他人锁
```

**解锁 Lua（GET + DEL 必须原子）**

```lua
-- KEYS[1] = lock key
-- ARGV[1] = 持有者 UUID
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
-- 原子性保证：Lua 脚本在 Redis 单线程中整体执行，中间不会插入其他命令
```

**为什么 GET 后不能直接 DEL**：
```
线程A GET → 确认是自己的锁 → (此时 TTL 到期，线程B 加锁成功) → 线程A DEL → 误删线程B 的锁
```

---

### 3.2 Redisson 可重入锁：tryAcquireAsync Lua

Redisson 用 **Hash 替代 String** 存锁，支持可重入：

```
Hash key:   lock:order:123
Hash field: {uuid}:{threadId}   ← 标识到具体线程（跨节点唯一）
Hash value: 重入次数（整数）
```

**加锁 Lua（tryLockInnerAsync，简化版）**

```lua
-- KEYS[1] = lock key
-- ARGV[1] = TTL（毫秒）
-- ARGV[2] = {uuid}:{threadId}

-- 分支①：锁不存在 → 新建，重入次数 = 1
if (redis.call('exists', KEYS[1]) == 0) then
    redis.call('hset', KEYS[1], ARGV[2], 1);
    redis.call('pexpire', KEYS[1], ARGV[1]);
    return nil;  -- nil 表示加锁成功
end;

-- 分支②：锁存在且是自己的 → 重入，次数 +1
if (redis.call('hexists', KEYS[1], ARGV[2]) == 1) then
    redis.call('hincrby', KEYS[1], ARGV[2], 1);
    redis.call('pexpire', KEYS[1], ARGV[1]);
    return nil;
end;

-- 分支③：锁被其他线程持有 → 返回剩余 TTL（调用方用于等待）
return redis.call('pttl', KEYS[1]);
```

**解锁 Lua（unlockInnerAsync，简化版）**

```lua
-- KEYS[1] = lock key
-- KEYS[2] = channel（redisson_lock__channel:{lockName}）
-- ARGV[1] = UNLOCK_MESSAGE（解锁通知内容，固定值 0L）
-- ARGV[2] = TTL（用于还有重入时续期）
-- ARGV[3] = {uuid}:{threadId}

-- 不是自己的锁 → 直接返回（幂等，不报错）
if (redis.call('hexists', KEYS[1], ARGV[3]) == 0) then
    return nil;
end;

-- 重入次数 -1
local counter = redis.call('hincrby', KEYS[1], ARGV[3], -1);

if (counter > 0) then
    -- 还有重入层，续期后返回
    redis.call('pexpire', KEYS[1], ARGV[2]);
    return 0;
else
    -- 完全释放：删除 key，向 channel 发布解锁消息
    redis.call('del', KEYS[1]);
    redis.call('publish', KEYS[2], ARGV[1]);  -- ← 触发等待者唤醒
    return 1;
end;
```

**关键**：`publish` 是通知其他等待线程"锁已释放，可以来抢了"，这是 Pub/Sub 等待机制的触发点。

---

### 3.3 等待机制：Pub/Sub 而非自旋

获取锁失败后，Redisson **不自旋重试**，而是订阅 Redis 频道等通知，避免无效 CPU 消耗：

```
lock() 调用链（RedissonLock）

lock()
  → tryAcquire(-1, -1, null, threadId)
      → tryAcquireAsync()  ← 执行加锁 Lua
        成功（返回 nil）→ 注册 watchdog → 返回
        失败（返回 TTL） ↓

  → subscribe(threadId)
      → LockPubSub.subscribe("redisson_lock__channel:{lockName}")
      → 返回 CompletableFuture<RedissonLockEntry>

  → 进入等待循环：
      while (true) {
          ttl = tryAcquire(...)           // 再尝试一次（防止在订阅期间锁已释放）
          if (ttl == null) return true    // 成功
          if (ttl >= 0)
              entry.getLatch().tryAcquire(ttl, MILLISECONDS)  // Semaphore 限时等待
          else
              entry.getLatch().acquire()                       // 永久等待
      }
      // 收到 publish 解锁消息 → LockPubSub 回调 entry.getLatch().release()
      //                        → Semaphore 被唤醒 → 重新进入循环竞争

  → unsubscribe()   // 加锁成功后取消订阅
```

**Semaphore 限时等待的作用**：以锁剩余 TTL 为超时时间，防止因 `publish` 丢失（网络异常）而永久挂起。

### 3.4 Watchdog 续期：scheduleExpirationRenewal

```java
// RedissonBaseLock.scheduleExpirationRenewal（简化）

private void renewExpiration() {
    ExpirationEntry ee = EXPIRATION_RENEWAL_MAP.get(getEntryName());
    if (ee == null) return;  // unlock() 已调用，停止续期

    Timeout task = commandExecutor.newTimeout(new TimerTask() {
        @Override
        public void run(Timeout timeout) {
            ExpirationEntry ent = EXPIRATION_RENEWAL_MAP.get(getEntryName());
            if (ent == null) return;

            Long threadId = ent.getFirstThreadId();
            if (threadId == null) return;

            // 执行续期 Lua
            RFuture<Boolean> future = renewExpirationAsync(threadId);
            future.onComplete((res, e) -> {
                if (e != null) { return; }  // 网络异常则停止续期
                if (res) {
                    renewExpiration();       // 续期成功 → 递归调度下一次
                }
                // res == false：锁已不存在（异常释放），停止续期
            });
        }
    }, internalLockLeaseTime / 3, TimeUnit.MILLISECONDS);  // 每 TTL/3 触发
    ee.setTimeout(task);
}
```

**续期 Lua（renewExpirationAsync）**：

```lua
-- KEYS[1] = lock key
-- ARGV[1] = TTL（毫秒）
-- ARGV[2] = {uuid}:{threadId}
if (redis.call('hexists', KEYS[1], ARGV[2]) == 1) then
    redis.call('pexpire', KEYS[1], ARGV[1]);
    return 1;
end;
return 0;
-- 返回 0：锁不存在或持有者不是自己 → 停止续期
```

**关键设计**：
- `EXPIRATION_RENEWAL_MAP` 是 JVM 级的全局 Map（`ConcurrentHashMap`），key 为 `{ip}:{port}:{lockName}`
- `unlock()` 时调用 `cancelExpirationRenewal(threadId)`，从 Map 中移除 → watchdog 自然停止
- JVM 崩溃时 JVM 线程全死 → Map 消失 → 没有线程再续期 → TTL 自然到期释放锁

### 3.5 手动指定 TTL vs Watchdog 对比

| 场景 | 代码 | Watchdog | 风险 |
|------|------|---------|------|
| 自动续期（推荐） | `lock.lock()` | 启动，每 10s 续期 30s | 业务死循环时锁永不释放 |
| 手动 TTL | `lock.lock(10, SECONDS)` | **不启动** | TTL 估短则业务未完成锁过期 |
| 带超时等待 | `lock.tryLock(3, 10, SECONDS)` | **不启动** | 同上 |

`lock.lock()` 不传 leaseTime 时，Redisson 内部传 `-1`，`tryAcquireAsync` 中检测到 `-1` 才启动 watchdog；手动传 leaseTime 时跳过 watchdog 注册。

### 3.6 RedissonFairLock（公平锁）

标准 Redisson 锁是非公平的（unlock 后所有等待者竞争）。公平锁用 **List（有序等待队列） + ZSet（超时管理）** 实现 FIFO：

```
Redis 数据结构
  Hash:  lock:order:123                        ← 锁本体（同普通锁）
  List:  redisson_lock_queue:{lock:order:123}  ← 等待线程的 FIFO 队列（lpush/rpop）
  ZSet:  redisson_lock_timeout:{lock:order:123}← 等待线程的超时时间（score = 过期时间戳）

加锁 Lua 核心逻辑
  1. 清理 ZSet 中已超时的等待者（score < now）
  2. 锁不存在 且 队列为空（或队头是自己）→ 加锁
  3. 是自己（重入） → hincrby
  4. 否则 → 将自己加入 List 尾 + ZSet（score = now + waitTime）
     返回等待时间（供调用方 Semaphore 限时等待）

解锁 Lua 核心逻辑
  1. 重入次数 -1；归零则 del 锁
  2. rpop List 队头节点，用 pexpire 给它 10s 时间（优先权窗口）
  3. publish 解锁消息 → 唤醒队头线程

效果：队头线程被唤醒后在 10s 内优先获取锁（其他线程看到 List 非空则继续等待）
```

### 3.7 RedissonReadWriteLock（读写锁）

```
Hash 结构（lock:rw:123）
  mode          → "read" 或 "write"（当前锁模式）
  {uuid:tid}    → 写锁重入次数（mode=write 时）
  {uuid:tid}:read → 读锁重入次数（mode=read 时）

加读锁 Lua
  if mode == "write" → 返回 pttl（等待写锁释放）
  if mode == "read" OR 不存在：
      hset mode read
      hincrby {uuid:tid}:read 1
      pexpire TTL
      zadd redisson_rwlock:{key} score=now+TTL field={uuid:tid}  ← 追踪读锁持有者
      return nil

加写锁 Lua
  if key 不存在 → 正常加写锁（hset mode write + hincrby 1）
  if mode == "read" → 返回 pttl（等待所有读锁释放）
  if mode == "write" AND hexists == 1 → 重入

解读锁 Lua
  hincrby {uuid:tid}:read -1
  计数归零 → hdel 该 field
  如果 Hash 中除 mode 外无其他 field → del 锁 + publish

关键限制
  读锁升写锁：禁止（多个读锁互相等待写锁 → 死锁）
  写锁降读锁：允许（先加读锁，再释放写锁）
```

### 3.8 RedLock（了解其争议）

```
算法步骤（N 个独立 Redis 实例，通常 N=5）
  1. 记录当前时间 t1
  2. 依次向所有 N 个节点发送加锁请求（SET NX PX，每节点设超时 ~5ms）
  3. 成功节点数 ≥ N/2+1（3/5）且 elapsed < lockTTL → 加锁成功
  4. 有效剩余时间 = lockTTL - elapsed - 时钟漂移
  5. 加锁失败（超时/成功节点不足）→ 向所有节点发解锁 Lua

争议（Martin Kleppmann 2016）
  时钟漂移 + GC STW 可能导致锁提前过期，两个客户端同时持锁
  ↓
  解法：Fencing Token（每次加锁返回单调递增 token，存储层拒绝旧 token 的写）

工程结论：互联网业务用单节点 Redisson 即可；金融强一致场景考虑 ZooKeeper
```

## 四、Spring 框架源码

### 3.1 IoC 容器启动流程

```
ApplicationContext.refresh() 13 步（AbstractApplicationContext）

1.  prepareRefresh()              准备环境变量
2.  obtainFreshBeanFactory()      创建/刷新 BeanFactory，加载 BeanDefinition
3.  prepareBeanFactory()          注册内置 BeanPostProcessor、环境 Bean
4.  postProcessBeanFactory()      子类扩展点（留给 Web 容器）
5.  invokeBeanFactoryPostProcessors()  执行 BeanFactoryPostProcessor
                                   ↑ @Configuration、@ComponentScan、@Bean 在此解析
6.  registerBeanPostProcessors()  注册 BeanPostProcessor（不执行）
7.  initMessageSource()           国际化
8.  initApplicationEventMulticaster()  初始化事件广播器
9.  onRefresh()                   子类扩展（Web: 创建 Servlet 容器）
10. registerListeners()           注册监听器
11. finishBeanFactoryInitialization()  实例化所有非懒加载单例 Bean
                                   ↑ 三级缓存解循环依赖在此发生
12. finishRefresh()               发布 ContextRefreshedEvent
13. resetCommonCaches()           清理反射缓存
```

**Bean 生命周期关键节点**（在步骤 11 内）：

```
instantiateBean()           → 构造函数/工厂方法实例化（未注入）
  ↓ 放入 singletonFactories（三级缓存）
populateBean()              → @Autowired 依赖注入
  ↓
initializeBean()
  ├── setBeanName/setBeanFactory（Aware 接口）
  ├── postProcessBeforeInitialization（@PostConstruct 在此执行）
  ├── afterPropertiesSet()（InitializingBean）
  ├── init-method
  └── postProcessAfterInitialization（AOP 代理在此生成）
  ↓ 移入 singletonObjects（一级缓存）
```

**三级缓存结构**：

```
singletonObjects      一级：完整 Bean（初始化完成）
earlySingletonObjects 二级：早期 Bean（已实例化未初始化，可能已被代理）
singletonFactories    三级：ObjectFactory（生成早期 Bean 的工厂 lambda）

循环依赖 A→B→A 解法
  A 实例化 → 存三级缓存 ObjectFactory
  → 注入 B → B 实例化 → B 注入 A
  → getBean(A) → 三级缓存 ObjectFactory.getObject() → 生成 A 的早期引用（或 AOP 代理）
  → 存二级缓存 → B 完成初始化
  → A 完成初始化 → 移入一级缓存
```

### 3.2 AOP 源码

```
代理创建（AbstractAutoProxyCreator.postProcessAfterInitialization）
  → wrapIfNecessary()
    → getAdvicesAndAdvisorsForBean()  找匹配的 Advisor
    → createProxy()
        ├── 目标类实现了接口 → JdkDynamicAopProxy（java.lang.reflect.Proxy）
        └── 目标类无接口 → CglibAopProxy（ASM 字节码生成子类）

JdkDynamicAopProxy.invoke()

  → proceed()：责任链模式逐个执行 Advice

  this.methodA() 调用 this.methodB()
  → 走的是原始对象（非代理）→ methodB 上的 AOP 不生效

### 3.3 @Transactional 源码

```
TransactionInterceptor.invoke()（MethodInterceptor）
  → invokeWithinTransaction()
    → TransactionAttributeSource 解析 @Transactional 注解（传播、隔离、回滚规则）
    → PlatformTransactionManager.getTransaction()   开启/加入/挂起事务
    → proceed()  执行目标方法
    ├── 无异常 → commitTransactionAfterReturning()
    └── 异常 → completeTransactionAfterThrowing()
                  ├── rollbackOn() 判断异常类型
                  └── 回滚 or 提交（noRollbackFor）

传播机制关键源码（AbstractPlatformTransactionManager）
  REQUIRED：已有事务 → 加入（doBegin 复用连接）
  REQUIRES_NEW：已有事务 → suspend() 挂起 → 新开事务
  NESTED：已有事务 → 设置 Savepoint（JDBC SavepointManager）
```

**失效场景**：方法非 public / 同类调用（AOP 不走代理）/ 异常被 catch 吞 / 多线程（事务绑定 ThreadLocal）/ rollbackFor 未配置。

## 五、MyBatis 源码

### 5.1 工作原理全链路

```
SqlSessionFactory（由 SqlSessionFactoryBuilder 解析 mybatis-config.xml 构建）
  └── Configuration（全局配置：MappedStatement、TypeHandler、Plugin 链）

SqlSession.selectList("namespace.id", param)
  → DefaultSqlSession → Executor（SimpleExecutor / CachingExecutor）
  → MappedStatement（SQL + 结果映射）
  → StatementHandler
      ├── PreparedStatementHandler.prepare()    → connection.prepareStatement(sql)
      ├── ParameterHandler.setParameters()      → TypeHandler 设置占位符参数
      └── ResultSetHandler.handleResultSets()   → TypeHandler + ResultMapping 映射结果集
```

**#{ } vs ${ }**：`#{}` → PreparedStatement 占位符（预编译，防 SQL 注入）；`${}` → 字符串直接拼接（用于表名/列名动态替换，有注入风险）。

### 5.2 Mapper 接口代理

```
MapperProxyFactory.newInstance()
  → JDK 动态代理 Proxy.newProxyInstance(MapperProxy)

MapperProxy.invoke()
  → 非 Object 方法 → MapperMethod.execute()
  → 根据 SqlCommand.type（SELECT/INSERT/UPDATE/DELETE）分发
  → 调用 sqlSession 对应方法
```

### 5.3 Plugin / Interceptor 拦截器

```
四个可拦截点（Invocation.target 类型）
  Executor        → query / update / commit / rollback
  StatementHandler → prepare / parameterize / query / update
  ParameterHandler → setParameters
  ResultSetHandler → handleResultSets

Plugin.wrap(target, interceptor)
  → 若 target 类型在 @Intercepts 声明中 → JDK 动态代理包装（责任链）
  → 多个插件按注册顺序嵌套代理，执行时逆序剥开

PageHelper 原理
  → 实现 Executor.query 拦截器
  → 在 SQL 前追加 COUNT(*) 查询
  → 修改 SQL 追加 LIMIT offset, pageSize（方言适配）
```

### 5.4 缓存机制

```
一级缓存（SqlSession 级）
  SimpleExecutor → BaseExecutor.localCache（PerpetualCache = HashMap）
  key = CacheKey(statementId + rowBounds + sql + params)
  同一 SqlSession 相同查询命中；执行 update/commit/rollback/close 后清空
  多个 SqlSession 之间不共享 → 并发修改时脏读问题

二级缓存（Mapper 级，需显式开启）
  在 Mapper.xml 加 <cache/>
  CachingExecutor 装饰 SimpleExecutor（装饰器模式）
  事务提交后才写入二级缓存（防脏读）
  多 SqlSession 共享，key 同一级；实现类可选 Ehcache / Redis
```

### 6.1 架构层次

Server
  └── Service
        ├── Connector（接收 HTTP 请求）

        └── Container（处理请求）

### 6.2 请求处理流程

```

  → NioEndpoint.Acceptor 接收连接 → 注册到 Poller（Selector）
  → Poller 检测到 READ 事件 → 封装 SocketProcessor 提交线程池
  → 线程池 Worker → Http11Processor 解析 HTTP 协议

**为何线程数可设 200 而不是 N+1**：Tomcat 处理的是 I/O 密集型请求（等待 DB、网络），线程大部分时间阻塞等待，CPU 不是瓶颈，适合大线程池；N+1 只适合 CPU 密集型任务。

## 七、JVM 类加载源码

### 6.1 ClassLoader.loadClass 核心逻辑

```java
protected Class<?> loadClass(String name, boolean resolve) {
    synchronized (getClassLoadingLock(name)) {
        Class<?> c = findLoadedClass(name);    // ① 检查已加载
        if (c == null) {
            try {
                if (parent != null)
                    c = parent.loadClass(name, false);   // ② 委派父加载器
                else
                    c = findBootstrapClassOrNull(name);  // ③ Bootstrap
            } catch (ClassNotFoundException e) {}
            if (c == null)
                c = findClass(name);           // ④ 自己加载（子类覆盖）
        }
        if (resolve) resolveClass(c);          // ⑤ 连接（验证+准备+解析）
        return c;
    }
}
```

**破坏双亲委派**：覆盖 `loadClass()`（不调 super）而非 `findClass()`。Tomcat / OSGi / JDBC SPI 均如此。

### 6.2 SPI 与类加载器

```
JDBC 驱动加载
  DriverManager（Bootstrap 加载）→ ServiceLoader.load(Driver.class)
  → Thread.currentThread().getContextClassLoader()  ← 取线程上下文加载器
  → 实际是 AppClassLoader，能加载 classpath 下的驱动 jar
  → 绕过双亲委派（父加载器委托子加载器）
```

## 九、高频面试源码考点速查

| 问题 | 源码关键词 |
|------|----------|
| HashMap 扩容为何 2 倍 | `newCap = oldCap << 1`；index = hash & (cap-1) 位运算要求 cap 为 2 的幂 |
| ConcurrentHashMap size 怎么统计 | `baseCount + ∑CounterCell`（LongAdder 思想，分段累加减竞争） |
| Spring Bean 为何默认单例 | `singletonObjects` HashMap 缓存；`scope=prototype` 则每次 `getBean()` 新实例化 |
| @Transactional 自调用失效原因 | `this.method()` 绕过 Spring 代理，直接调原始对象，TransactionInterceptor 不执行 |
| AOP 代理选择规则 | `DefaultAopProxyFactory.createAopProxy()`：`targetClass` 有接口且非 `proxyTargetClass` → JDK；否则 → CGLIB |
| MyBatis 一级缓存失效 | 执行 DML / 手动 `clearLocalCache()` / `flushCache=true` / 不同 `SqlSession` |
| Tomcat 为何破坏双亲委派 | WebappClassLoader 覆盖 `loadClass()` 优先自己找，实现多应用类隔离 |
| ThreadLocal 泄漏根因 | Entry 的 key 是 `WeakReference<ThreadLocal>`，key GC 后 value 成僵尸（强引用留着） |
| ArrayList fail-fast 原理 | `modCount` 每次结构修改递增；Iterator.next() 检查 `modCount != expectedModCount` 则抛 CME |
| 自增主键为何不连续 | AUTO-INC 锁（`mode=1`）：批量 INSERT 预分配 ID，部分回滚后 ID 空洞 |
| Redisson 解锁为何要 publish | `unlockInnerAsync` Lua 最后 `publish channel UNLOCK_MESSAGE`，触发 `LockPubSub` 回调 `Semaphore.release()`，唤醒等待线程 |
| Redisson 等待为何不自旋 | `subscribe(channel)` 收到 Pub/Sub 通知才唤醒，`entry.getLatch().tryAcquire(ttl)` 限时等待，避免 CPU 空转 |
| watchdog 何时不启动 | `lock.lock(leaseTime, unit)` 手动传 leaseTime 时 `internalLockLeaseTime != -1`，跳过 `scheduleExpirationRenewal()` |
| Redisson 重入用 Hash 而非 String | String 存单一值无法区分线程，Hash field=`{uuid:threadId}` value=重入次数，`hincrby` 原子加减 |
| RedissonFairLock 排队原理 | List 维护 FIFO 等待队列，解锁 Lua `rpop` 队头 + `pexpire` 给 10s 优先窗口，其他线程检测 List 非空继续等 |
| RedissonReadWriteLock 读升写为何死锁 | 多个读锁同时持有，每个都等写锁，写锁等所有读锁释放 → 循环等待 |
| RedLock 争议关键 | 时钟漂移 + GC STW 可使锁提前过期，两客户端同时持锁；Fencing Token 才是真正的安全兜底 |
