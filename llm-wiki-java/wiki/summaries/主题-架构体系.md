## 二、Spring Cloud 组件全景

### 2.1 Netflix（旧）vs Alibaba（现主流）

| 功能 | Netflix（旧/停维）| Alibaba（推荐）| 说明 |
| --- | --- | --- | --- |
| 服务注册发现 | Eureka（纯 AP）| **Nacos** | Nacos 同时支持 AP/CP 切换，见 [[机制-Nacos]] |
| 配置中心 | Config + Bus（依赖 Git）| **Nacos Config** | Nacos 支持动态推送，无需 Bus |
| 负载均衡 | Ribbon（已停维）| **Spring Cloud LoadBalancer** | 内置轮询/随机策略 |
| 服务调用 | Feign | **OpenFeign** | 声明式 HTTP 客户端，注解驱动 |
| 熔断降级 | Hystrix（停维）| **Sentinel** | Sentinel 支持 QPS/并发线程数/热点参数多种规则 |
| 网关 | Zuul（Servlet 阻塞）| **Spring Cloud Gateway** | 基于 WebFlux 异步非阻塞 |
| 链路追踪 | Sleuth + Zipkin | **SkyWalking** | SkyWalking 无侵入字节码增强，见 [[概念-可观测性]] |

### 2.2 Sentinel vs Hystrix

| 特性 | Hystrix | Sentinel |
| --- | --- | --- |
| 隔离策略 | 线程池 / 信号量 | 信号量（无额外线程开销）|
| 熔断策略 | 异常比例 | 慢调用比例 / 异常比例 / 异常数 |
| 限流规则 | 有限支持 | QPS、并发线程数、热点参数 |
| 控制台 | Hystrix Dashboard | Sentinel Dashboard（实时规则推送）|
| 维护状态 | 停止维护 | 活跃维护 |

**Sentinel 熔断状态机**：关闭 → 打开（快速失败）→ 半开（探测请求）→ 关闭/打开（根据探测结果）

### 2.3 Spring Cloud Gateway 核心

```
请求 → Route 匹配（Predicate 断言）
       → Pre Filter 链（鉴权/限流/日志）
       → 代理转发下游服务
       → Post Filter 链（响应处理）
       → 返回客户端
```

**Gateway vs Nginx**：Gateway 是应用层网关，与 Spring 生态深度集成，支持动态路由（配合 Nacos）；Nginx 是基础设施层网关，性能更高，适合静态路由和静态资源。

## 三、高可用设计

### 3.1 SLA 核心指标

`可用性 A = MTBF / (MTBF + MTTR)`

- **MTBF**（Mean Time Between Failures）：平均无故障时间
- **MTTR**（Mean Time To Repair）：平均恢复时间

| 可用性 | 年停机时间 | 俗称 |
| --- | --- | --- |
| 99% | 87.6 小时 | 2 个 9 |
| 99.9% | 8.76 小时 | 3 个 9 |
| 99.99% | 52.6 分钟 | 4 个 9 |
| 99.999% | 5.26 分钟 | 5 个 9 |

**提升可用性**：缩短 MTTR（快速恢复）比延长 MTBF（减少故障）更容易实现，因此自动化故障转移 + 快速回滚比"不出问题"更重要。

### 3.2 冗余与容灾手段

| 手段 | 作用 |
| --- | --- |
| **冗余** | 多实例消除单点，关键组件主从/主主/集群 |
| **限流** | 令牌桶（允许突发）/ 漏桶（平滑输出）；详见 [[主题-三高架构]] |
| **熔断** | 快速失败防雪崩，Sentinel 三态熔断器 |
| **降级** | 关闭非核心功能，返回默认值/缓存数据/友好提示 |
| **超时+重试** | 所有远程调用必须设超时（防线程池耗尽）；重试必须幂等 + 指数退避 + Jitter（防惊群）|
| **健康检查** | 主动探测 + 被动根据请求结果判断，自动摘除异常实例 |

### 3.3 异地多活

| 方案 | 说明 | 延迟 | 能力 |
| --- | --- | --- | --- |
| **同城双活** | 两个数据中心同城，<1ms 延迟 | 极低 | 可强一致，应对机房级故障 |
| **两地三中心** | 同城双活 + 异地灾备（仅备份，不提供服务）| 中 | 异地灾备 RPO 较大 |
| **异地多活** | 多地域同时提供服务，用户就近访问 | 高（跨地域 10~100ms）| 最强容灾，核心挑战是数据同步 |

**异地多活数据分区策略**（关键）：
- 按用户 ID 分区：每个用户的写请求路由到**主地域**（避免跨地域写冲突）
- 读请求可就近路由到本地副本（异步复制，允许短暂延迟）
- 写请求若涉及跨地域（如用户主地域与当前地域不同），通过路由层转发到主地域

**无法完全异地多活的场景**：全局库存扣减、金融强一致账务——这些场景只能牺牲部分可用性（CP 选择），或用异步对账+补偿。

## 四、DDD 领域驱动设计

### 4.1 战略设计

**核心概念**：

| 概念 | 说明 |
| --- | --- |
| **领域（Domain）** | 业务问题空间，如电商领域 |
| **子域（Subdomain）** | 核心子域（竞争优势，如推荐算法）/ 支撑子域（必要但非核心，如物流）/ 通用子域（如认证）|
| **限界上下文（Bounded Context）** | 模型的边界，同一概念在不同上下文含义不同（"用户"在订单 vs 会员上下文）|
| **上下文映射（Context Map）** | 限界上下文间关系：防腐层（ACL）/ 共享内核 / 客户-供应商 |

**防腐层（Anti-Corruption Layer）**：将外部概念翻译成本地领域概念，防止外部模型污染内部模型。调用第三方服务或遗留系统时必加。

### 4.2 战术设计

| 概念 | 说明 |
| --- | --- |
| **实体（Entity）** | 有唯一标识，通过 ID 区分（如订单、用户）；状态可变 |
| **值对象（Value Object）** | 无唯一标识，通过属性值区分，**不可变**（如金额、地址）|
| **聚合（Aggregate）** | 一组相关实体+值对象的集合，有一个聚合根；聚合是事务边界 |
| **聚合根（Aggregate Root）** | 聚合的入口，外部只能通过聚合根访问；聚合内部对象不可跨聚合直接引用 |
| **领域服务（Domain Service）** | 不属于任何实体的领域逻辑（如转账：跨两个账户聚合）|
| **领域事件（Domain Event）** | 领域内发生的重要事情，过去时命名（OrderCreated）；异步解耦跨聚合操作 |
| **仓储（Repository）** | 聚合的持久化接口，领域层定义接口，基础设施层实现 |
| **应用服务（Application Service）** | 协调领域对象完成用例，**不包含业务逻辑**，只编排 |

### 4.3 DDD 分层架构

```
┌─────────────────────────────────┐
│  用户接口层（User Interface）    │  Controller、DTO、Assembler（DTO↔领域对象转换）
├─────────────────────────────────┤
│  应用层（Application）          │  ApplicationService、Command/Query、事务边界
├─────────────────────────────────┤
│  领域层（Domain）               │  Entity、ValueObject、Aggregate、DomainService、DomainEvent
├─────────────────────────────────┤
│  基础设施层（Infrastructure）   │  Repository 实现、MQ Client、外部服务 Client
└─────────────────────────────────┘
```

**依赖方向**：外层依赖内层，基础设施层通过依赖倒置（接口）对领域层提供实现。

### 4.4 CQRS（命令查询职责分离）

- **Command 侧**（写）：走领域模型，保证业务规则和一致性，可能有额外校验和事件发布
- **Query 侧**（读）：走优化的查询模型（直接查 DB / ES），不走领域模型，简单高效

**适用场景**：读写比例悬殊（如10:1以上）、读模型和写模型差异大、需要 Event Sourcing。

**CQRS + Event Sourcing**：不存储当前状态，只存储所有**领域事件**序列，当前状态通过重放事件得到。优点：完整审计日志、支持时间旅行（replay 到任意时刻）；缺点：查询复杂（需投影）、存储量大。

### 4.5 微服务与 DDD 的关系

**一个限界上下文 ≈ 一个微服务**（不是绝对，初期可以合并在一个服务中，随业务增长再拆）。

DDD 提供了微服务拆分的理论依据：**按业务边界拆分，而不是按技术层次拆分**。错误拆法：把 Controller 层、Service 层、DAO 层拆成三个服务。

## 五、SOLID 原则与架构模式

### 5.1 SOLID 原则

| 原则 | 全称 | 核心要点 |
| --- | --- | --- |
| **S** 单一职责 | Single Responsibility | 一个类只有一个变化原因；不是"只有一个方法" |
| **O** 开闭原则 | Open/Closed | 对扩展开放，对修改关闭；用接口/抽象层隔离变化 |
| **L** 里氏替换 | Liskov Substitution | 子类可以无感知替换父类；不能收窄前置条件、放宽后置条件 |
| **I** 接口隔离 | Interface Segregation | 接口应小而专一，不强迫客户端依赖不需要的方法 |
| **D** 依赖倒置 | Dependency Inversion | 高层模块不依赖低层实现，都依赖抽象；IOC 容器是典型实践 |

### 5.2 事件驱动架构（EDA）

- 生产者发布事件 → 消息队列 → 消费者订阅事件
- 天然支持**异步**和**横向扩展**；消费者无需感知生产者
- 缺点：调试困难（事件链路难以追踪）、最终一致性难以推理
- 实践：[[机制-Kafka]] / [[机制-RocketMQ]] 作为事件总线；Spring `ApplicationEvent` 用于本地进程内事件

### 5.3 设计模式在架构层的应用

详见 [[主题-设计模式]]。架构层高频模式：

| 模式 | 架构场景 |
| --- | --- |
| 策略模式 | 支付渠道选择、规则引擎、消除大量 if-else |
| 观察者模式 | 领域事件发布（DomainEvent）、MQ 消费 |
| 责任链模式 | API 网关 Filter、Servlet Filter、Netty Pipeline |
| 模板方法 | `JdbcTemplate`、`RestTemplate`、框架扩展点 |
| 门面模式 | SLF4J（日志门面）、聚合多个下游服务的应用服务层 |
| 状态模式 | 订单状态机、工作流引擎（消除复杂条件判断）|

## 六、高并发补充：秒杀系统

秒杀核心挑战：瞬间高并发（百倍平时 QPS）、库存超卖、数据一致性。

### 6.1 分层拦截策略

```
前端层      → 按钮置灰防重复点击；随机丢弃（只放行10%）
CDN/Nginx   → 静态页面缓存；limit_req 接入层限流
网关层      → 用户维度限流（同一用户每秒最多1次）
应用层      → 本地缓存库存（减少 Redis 访问）；MQ 队列排队异步处理
Redis 层    → 原子扣减库存（DECR 或 Lua）；库存≤0 直接返回失败
数据库层    → 异步写订单；乐观锁防超卖（WHERE count > 0）
```

### 6.2 库存扣减方案对比

| 方案 | 原理 | 优点 | 缺点 |
| --- | --- | --- | --- |
| Redis DECR | `DECR stock`，返回值 < 0 判超卖 | 极快，O(1) | 需异步同步到 DB |
| Redis Lua | `if stock>0 then DECR end`（原子）| 安全，无超卖 | 同上 |
| DB 乐观锁 | `UPDATE ... WHERE count > 0` | 强一致 | 高并发下失败率高，需重试 |
| DB 悲观锁 | `SELECT ... FOR UPDATE` | 强一致 | 并发低，容易超时 |

**推荐方案**：Redis Lua 扣减（快速拦截）+ 异步消息 → 数据库最终写入。

### 6.3 防刷策略

用户 Token 验证 + IP 限流 + 验证码（高频操作降速）+ 黑名单（Redis 集合存封禁用户）。

## 七、CI/CD 与云原生实践

### 7.1 标准 CI/CD 流水线

```
代码提交（Git Push）
  → 代码检查（SonarQube，门禁：覆盖率/Bug数）
  → 单元测试（失败则阻断）
  → 构建 Docker 镜像
  → 推送镜像仓库（Harbor/ACR）
  → 部署测试环境（自动）
  → 集成测试 / E2E 测试
  → 部署生产环境（人工审批）
```

### 7.2 K8s 发布策略

| 策略 | 原理 | 优点 | 缺点 |
| --- | --- | --- | --- |
| **滚动更新** | 逐步替换旧 Pod（maxSurge/maxUnavailable 控制节奏）| 零停机，资源占用低 | 新旧版本短暂共存，需向后兼容 |
| **蓝绿部署** | 同时维护蓝（旧）绿（新）两套环境，一键切流量 | 回滚瞬间完成 | 资源成本翻倍 |
| **金丝雀发布** | 逐步将流量切到新版本（5% → 20% → 100%）| 灰度验证，风险可控 | 需要流量路由支持（Ingress/网关权重）|

### 7.3 Docker 镜像优化

- 小基础镜像：`alpine`（~5MB）vs `ubuntu`（~70MB）
- **多阶段构建**：编译阶段用 JDK，运行阶段只用 JRE，最终镜像不含编译工具
- 合并 `RUN` 指令减少镜像层数；`.dockerignore` 排除无关文件
