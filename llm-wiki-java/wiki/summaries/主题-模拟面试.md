## 二、基础技术问答

### AQS 核心设计？公平锁和非公平锁的区别？

AQS 核心：`volatile int state` + CLH 变种双向等待队列。获取资源 CAS 改 state，失败则封装 Node 入队 park；释放时 CAS 还原 state 并 unpark 队头。  
公平锁：`tryAcquire` 先 `hasQueuedPredecessors()`，有等待者直接入队。  
非公平锁：直接 CAS 抢锁，抢不到再入队。非公平吞吐高；公平不饥饿但切换多。

### volatile 能替代 synchronized 吗？

**可以**：单写多读的状态标志位（`boolean running`），volatile 保证可见性和有序性。  
**不行**：复合操作如 `count++`（读-改-写三步，volatile 不保证整体原子性），用 `AtomicInteger` 或 `synchronized`。

### synchronized 和 ReentrantLock 怎么选？

默认 `synchronized`——JVM 偏向锁/轻量级锁优化，无竞争性能好，自动释放。  
选 ReentrantLock：tryLock 超时、公平锁、多个 Condition、可中断获取、JDK21 虚拟线程（synchronized 会 pin 载体线程，ReentrantLock 不会）。

### 线程池核心参数怎么设置？

CPU 密集型：`core = CPU核数 + 1`；I/O 密集型：`core = CPU核数 * 2`。实际压测调整。  
**不要用 Executors 工厂方法**：`newCachedThreadPool` 高峰期创建 2000+ 线程 OOM。必须 `new ThreadPoolExecutor`，有界队列 + CallerRunsPolicy。

### InnoDB MVCC 为什么不能完全消除幻读？

MVCC 通过 undo log 版本链 + ReadView 实现读不阻写。普通 SELECT（快照读）不幻读，但同一事务内先快照读再 `UPDATE` 或 `SELECT FOR UPDATE`（当前读），当前读看到新插入行，后续快照读 ReadView 可能更新看到这些行。真正消除幻读需要 **Next-Key Lock**（Gap Lock + Record Lock）锁住"将来可能插入的位置"。

### 分片键 merchant_id 跨分片聚合查询怎么做？

1. 定时任务预聚合写入独立汇总表（不分片），平台级查询走汇总表
2. 明细数据同步到 ClickHouse，大宽表全量分析走 OLAP，不压 MySQL

### 慢 SQL 排查流程？

1. **定位**：慢查询日志（`slow_query_log=ON, long_query_time=1`）或 APM
2. **分析**：`EXPLAIN` 看 `type`（ALL 最差）、`key`（命中索引）、`rows`、`Extra`（Using filesort/Using temporary 是警告）
3. **常见原因**：函数包裹列、隐式类型转换、`LIKE '%前缀'`、SELECT* 回表、深分页
4. **优化顺序**：加/调索引 → 改 SQL → 分表 → 缓存/OLAP 分流

### Redis 为什么快？

1. 纯内存，ns 级读写
2. 单线程，无上下文切换和锁竞争
3. epoll IO 多路复用，一个线程处理大量连接
4. 高效数据结构（SDS/ziplist/skiplist）
5. Redis 6.0+ 多线程 I/O，命令执行仍单线程

瓶颈不在 CPU 而在网络 I/O，单线程足够。

### 缓存穿透、击穿、雪崩？

| 问题 | 定义 | 解法 |
|------|------|------|
| 穿透 | 查询不存在的 key | 布隆过滤器 / 空值缓存（短 TTL） |
| 击穿 | 热点 key 过期瞬间大量打 DB | setnx 互斥锁 / 永不过期+异步刷新 |
| 雪崩 | 大量 key 同时过期 / Redis 宕机 | TTL 加随机值 / 限流降级 / 多级缓存 |

实战：游戏平台 0 点刷新活动，所有缓存统一 TTL 24h 同时过期雪崩。解法：`TTL = 24h + random(0, 3600)` + 本地 Caffeine 兜底。

### Redis 和 MySQL 数据一致性怎么保证？

**先更新 DB，再删缓存**（Cache Aside）。不用"先删后更新"，并发场景会产生脏数据。  
两层保障：①写路径删缓存；②Canal 监听 binlog 异步删缓存（防写路径删除失败残留）。  
强一致场景（如余额）不走缓存，直接读 DB。

### @Transactional 失效场景？

1. 方法非 public
2. 自调用（`this.method()`，不走代理）
3. 异常被 catch 吞掉
4. rollbackFor 不匹配（默认只回滚 RuntimeException）
5. 多线程（事务绑定 ThreadLocal，新线程拿不到）

踩坑：`this.sendNotification()` 有 `REQUIRES_NEW`，自调用没走代理，通知失败把外层事务一起回滚，订单状态未更新但钱已扣。改法：`@Lazy` 注入自身 `self.sendNotification()`。

### Spring 三级缓存为什么要三级？

- **一级**（singletonObjects）：完整 Bean
- **二级**（earlySingletonObjects）：半成品 Bean（已实例化未注入）
- **三级**（singletonFactories）：ObjectFactory，决定返回原始对象还是代理

只有二级不够：AOP 场景 B 依赖 A，拿到的 A 必须是代理后的 A，三级的 ObjectFactory 可以在第一次被获取时创建代理。构造器注入循环依赖用 `@Lazy`；SpringBoot 2.6+ 默认禁止循环依赖。

### Nacos 注册中心选 AP 还是 CP？配置中心呢？

**注册中心选 AP**：容忍短暂数据不一致，不能因网络分区导致发现不可用，宁可读到稍旧的服务列表。  
**配置中心选 CP**：配置必须强一致，读到旧配置可能导致服务行为异常。  
Nacos 实现：AP 用 Distro 协议，CP 用 Raft。

### RocketMQ 和 Kafka 选型？

| 场景 | 选择 |
|------|------|
| 支付事务消息（精确一次语义）| RocketMQ |
| 延迟/定时消息 | RocketMQ |
| 消费失败重试/死信队列 | RocketMQ（内置）|
| 游戏日志大吞吐 | Kafka |

支付中台用 RocketMQ，游戏日志埋点用 Kafka。

### 消息堆积怎么处理？

**止血**：扩 Consumer 实例（不超队列数）；队列不够则新建 Topic + 转发消费者把积压消息转过去再扩消费者；降级非核心逻辑。  
**根因**：查慢 SQL/阻塞调用；`consumeMessageBatchMaxSize` 调大批量消费；设延迟告警阈值提前预警。

### TCC 和本地消息表怎么选？

| | TCC | 本地消息表 |
|---|-----|---------|
| 一致性 | 准实时（秒级）| 最终一致（分钟级）|
| 代码侵入 | 高 | 低 |
| 适用 | 资金类 | 通知/异步类 |

支付转账用 TCC，通知/积分/报表用本地消息表。

## 三、系统设计 — 高并发抢购（Gacha）

### 日活 500 万 10 万 QPS 抢购设计

```
Client → API Gateway → Gacha Service → Redis Lua Script → MQ → DB 异步落盘
```

1. **库存预热**：启动时 `SET item:rare:stock 100` 加载进 Redis
2. **Redis Lua 原子扣减**（天然原子，无超卖）：
   ```lua
   local stock = redis.call('GET', KEYS[1])
   if tonumber(stock) <= 0 then return -1 end
   return redis.call('DECR', KEYS[1])
   ```
3. **异步落盘**：抽奖成功发 MQ，消费者写 MySQL，高峰期 DB 不成瓶颈
4. **概率算法**：内存权重表，`ThreadLocalRandom` 随机数命中区间，μs 级无网络 IO
5. **限流**：Gateway 层令牌桶（单用户）+ Sentinel 全局流控

### Redis 挂了 Lua 扣减成功但 MQ 未发怎么处理？

本地消息表：同一 DB 事务写抽奖记录 + 写 `pending_mq`；定时任务扫描补发；消费者幂等（`order_id` 唯一索引）。Redis 库存是快速通道，DB 是最终凭证。Redis 重启后从 DB 聚合已发放数量重新加载。

### 10 万 QPS 单 key 热点怎么解决？

**库存分桶**：100 库存拆成 `item:rare:stock:0~9`，`userId % 10` 分散请求。某桶扣完了 round-robin 尝试下一个桶，全桶空才返回售罄。本地 Caffeine 标记"售罄"10 秒内不再请求该桶，减少无效 Redis 调用。

### 概率如何保证公平？"1000 次没中"怎么验证？

权重区间映射 + `ThreadLocalRandom`。每次抽奖记录（用户ID/随机数种子/结果）写日志，运营后台可回放验证。**保底机制**：累计未中次数存 Redis，达阈值（如 200 次）必中（软保底）。定期统计整体中奖率做卡方检验。日本法规（ガチャ規制）要求概率公示且可举证。

### 全球部署架构调整？

1. **就近接入**：每区域独立 Gacha Service + Redis + MySQL，GeoDNS 路由
2. **库存**：全球共享限量则主区域 Redis 扣减（接受 50-100ms 跨区延迟）；区域独立库存各自管理（简单，推荐）
3. **合规**：日本概率公示、欧洲 GDPR 数据可删除导出、比利时禁止 loot box 需地域屏蔽

## 四、算法 & 编码

### LRU 缓存实现（O(1) get/put）

`HashMap<K, Node<K,V>>` + 双向链表。HashMap 保证 O(1) 查找，链表维护访问顺序（头部最新，尾部最旧），dummy head/tail 简化边界。

```java
class LRUCache {
    private Map<Integer, Node> map = new HashMap<>();
    private Node head = new Node(), tail = new Node();
    private int capacity;

    public LRUCache(int capacity) {
        this.capacity = capacity;
        head.next = tail;
        tail.prev = head;
    }

    public int get(int key) {
        Node node = map.get(key);
        if (node == null) return -1;
        moveToHead(node);
        return node.value;
    }

    public void put(int key, int value) {
        Node node = map.get(key);
        if (node != null) {
            node.value = value;
            moveToHead(node);
        } else {
            node = new Node(key, value);
            map.put(key, node);
            addToHead(node);
            if (map.size() > capacity) {
                Node removed = tail.prev;
                removeNode(removed);
                map.remove(removed.key);
            }
        }
    }
}
```

### LRU 线程安全怎么做？

1. 全方法 `synchronized`：简单，读写互斥，并发差
2. 读写锁：get 也需要 moveToHead（写链表），退化成写锁，和方案 1 无区别
3. **分段锁 + 近似 LRU**（Caffeine）：ConcurrentHashMap + Ring Buffer 记录访问事件，异步单线程维护淘汰顺序

生产场景直接用 Caffeine，不要自己实现。

## 附 C：高频追问陷阱题集

### Seata TC 本身挂了怎么办？

TC 集群部署（DB/Redis 存储事务记录），单节点挂了其他节点接管。全挂了所有全局事务超时，try 锁定资源等 TC 恢复后 cancel。TC 是 Seata 已知单点风险，生产必须高可用。

### 为什么不去掉 Seata 用可靠消息方案？

可靠消息是最终一致，资金场景需要准实时一致。转账走 MQ 用户几秒内看到"余额未变"，体验和客服投诉不可接受。核心支付链路保留 TCC，非核心（通知/积分/报表）走 MQ。

### 为什么选 G1 不选 ZGC？

当时（2019-2023）ZGC 在 JDK 11 是 Experimental，生产不敢用。现在（JDK 17+）会考虑 ZGC——STW < 1ms，无需调 IHOP。代价是内存占用大（着色指针），4G 堆可能不够，需要 8G+。

### 对象池化后 reset 漏字段怎么办？

`RoomContext` 实现 `Resettable` 接口，写单元测试通过反射遍历所有字段，检查 `reset()` 后是否等于默认值。新增字段时 CI 跑这个测试，漏了就挂。

### Redis bigkey 问题遇到过吗？

排行榜 ZSet 200 万成员，`DEL` 导致主线程阻塞 2 秒。解法：`UNLINK`（Redis 4.0+ 异步删除）；写入时按 `rank:0~9` 拆分，每个 key 20 万成员。监控：`redis-cli --bigkeys` 或 `MEMORY USAGE key`。

### Redis Cluster hash slot 迁移期间客户端怎么处理？

- `MOVED`：key 已迁移完，更新本地 slot 路由表，直接请求新节点
- `ASK`：key 迁移中，临时向新节点发 `ASKING` 再发请求（一次性，不更新路由表）

Redisson/Lettuce 已内置自动处理。

### 如何评估性能瓶颈？

分层排查：接入层（Nginx access log 耗时分布）→ 应用层（Skywalking/Arthas 方法级 trace）→ 数据层（慢查询 + Redis slowlog）→ 基础设施（top/vmstat/iostat）。  
80% 问题在数据层，15% 在应用层（串行调用/大对象），5% 在基础设施。

4. 搭 CI/CD + 日志 + 监控 + 告警（第一天就要有，不是上线前补）
5. 最后写业务代码
