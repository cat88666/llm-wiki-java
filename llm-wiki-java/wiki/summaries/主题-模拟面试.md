## 二、行为 & 管理 & 反问

### 技术决策分歧怎么处理？

网关重构：一半人 SCG，一半人 Netty。让两派各写 1 天 PoC 压测说话。Netty 延迟低 30% 但维护成本 3 倍。决策：游戏内网关 Netty（性能第一），运营后台 SCG（迭代第一）。数据驱动，不拍板。

### 怎么培养初级开发？

渐进式任务（配置改→接口→模块→系统设计）+ Code Review 解释"为什么"建立工程直觉 + 参与线上问题排查（旁边兜底）+ 明确半年目标定期 1:1。核心：给方向、给机会、给反馈，不替他做决定。

### 印象最深的线上事故？

支付某银行渠道回调超时激增，2000 笔订单卡在"支付中"：
1. **止血（5min）**：回滚该渠道到上一版本
2. **定位（10min）**：HTTP Client 连接池从 200 改成了 20，高峰期耗尽
3. **修复**：修正参数，5% 灰度验证后全量
4. **补单**：向银行查单，批量更新 2000 笔
5. **复盘**：CI 增加关键配置参数变更的额外审批

15 分钟恢复，零资损，30 分钟补单完毕。

## 四、项目经验专项追问

### 4.1 技术职责模型

面试官问“你在项目里主要负责什么”时，不要只回答“写接口”。可以按职责层级回答：

1. 业务线研发规划与整体系统架构设计。
2. 主导关键系统模块设计与 Code Review。
3. 技术难点攻关，解决系统级复杂问题。
4. 建立研发规范、工程体系、发布流程和质量门禁。
5. 推动系统复用、扩展性、安全性、性能和低耦合设计。
6. 培训团队、梯队建设、推动技术分享。
7. 跨团队沟通协作与资源协调。
8. 对系统质量、稳定性、交付结果负责。

**答题模板**：

> 我不是只负责某个接口，而是负责从方案设计、核心链路开发、性能压测、上线灰度、监控告警到事故复盘的完整闭环。比如在 IM/游戏/支付项目中，我重点负责网关接入、消息可靠性、分布式事务和高可用治理，最后用 QPS、P99、故障恢复时间这些指标证明结果。

### 4.2 架构师职责模型

架构师项目表达要覆盖“设计、落地、影响力”：

| 维度 | 表达重点 |
| --- | --- |
| 高性能 | 压测、P99、线程池隔离、缓存、异步化、批量写 |
| 高可用 | 多副本、限流熔断、降级、故障演练、自动告警 |
| 分布式 | 注册发现、RPC、消息队列、分布式事务、最终一致性 |
| 工程体系 | CI/CD、Code Review、发布灰度、回滚、规范 |
| 团队影响 | 技术方案评审、培训、复盘、跨团队推进 |

**追问**：你怎么证明自己是架构师，而不是高级开发？

**参考回答**：

> 高级开发更多负责模块交付，架构师要对系统质量和演进负责。我会先定义目标和约束，例如容量、SLA、成本、团队能力；再拆核心链路和风险点；最后通过压测、灰度、监控、复盘闭环落地。我的产出不是一张图，而是可运行、可扩展、可观测、可回滚的系统。

### 4.3 微服务拆分原则

1. 单一职责、高内聚低耦合，避免一个服务承担多类业务。
2. 服务粒度适中，避免接口级拆分导致调用链爆炸。
3. 以业务模型为核心，按用户、订单、商品、支付、营销等领域拆分。
4. 演进式拆分，先粗后细，避免一开始过度微服务化。
5. 避免环形依赖和双向依赖，服务依赖关系应尽量是 DAG。

**项目口径**：

> 我拆服务时不会只按表拆，而是按业务闭环拆。比如交易链路中用户、订单、支付、营销、资金是不同边界；支付和资金必须保证事务边界清晰；营销不能反向强依赖交易，否则发布和故障都会互相拖累。

## 五、IM 系统项目深挖

### 5.1 项目一句话

IM（Instant Messaging）不是简单聊天，而是一个**低延迟、高可靠的实时消息分发系统**。工业级 IM 要处理千万/亿级用户、多端同步、离线消息、群聊、直播互动、未读数和历史消息存储。

典型分层：

| 分层 | 职责 |
| --- | --- |
| 客户端 | 与服务端建立长连接，收发消息，维护本地消息状态 |
| 网关/接入层 | 维护连接、鉴权、路由、推送、心跳 |
| 服务层 | 聊天、离线消息、群聊、红包、直播互动等业务逻辑 |
| 存储层 | 消息内容、账号、关系链、会话、未读数、索引 |

### 5.2 消息可靠投递：不丢消息

核心目标不是 Exactly Once，而是更现实的 **At Least Once + 幂等消费**：

1. 客户端生成全局唯一 `msgId`。
2. 发送超时或失败自动重试。
3. 服务端按 `msgId` 去重。
4. 服务端先可靠落库或写入 MQ，再向下游投递。
5. 接收端收到消息后 ACK。
6. 未 ACK 消息进入重推或离线补偿。

**面试回答**：

> 我们把“不丢消息”拆成三段：发送端重试、服务端可靠存储、接收端 ACK。由于网络不可靠，系统不追求天然 Exactly Once，而是用唯一 msgId、幂等写、ACK 和补偿任务实现最终可靠。

### 5.3 离线消息高性能设计

设计原则：

- 上线拉取，不对离线用户实时推送。
- Redis 扛热点，DB 做兜底。
- 最近 N 天/最近 N 条放缓存，历史消息走冷数据查询。

Redis 结构可以用 ZSet：

```text
Key: offline_msg_{uid}
Score: mid
Value: msg
```

常用操作：

- `ZADD` 添加离线消息。
- `ZREVRANGE` 拉取最近消息。
- `ZREMRANGEBYSCORE` 删除已确认消息。

单 Key 过大时，按周/月拆 Key。

### 5.4 海量历史消息存储

消息存储建议拆成“内容表 + 信箱索引表”：

```text
im_msg_content(mid, content, type, sender_id, recipient_id, create_time)
im_user_msg_box(owner_uid, other_uid, mid, box_type, create_time)
```

| 表 | 分片键 | 作用 |
| --- | --- | --- |
| 消息内容表 | `mid` | 存大字段和消息正文 |
| 用户信箱索引表 | `owner_uid` | 快速查询某个用户的收件箱/发件箱 |

好处：

- 支持双方独立删除。
- 查询会话列表和历史消息时只扫小索引表。
- 避免大字段频繁参与排序和分页。

### 5.5 群聊：读扩散 vs 写扩散

| 模型 | 做法 | 优点 | 缺点 | 适合场景 |
| --- | --- | --- | --- | --- |
| 读扩散 | 群消息只存一份，用户读同一份数据 | 写入快 | 已读/未读精细化难 | 大群、直播 |
| 写扩散 | 给每个用户写一条索引 | 未读/已读精确 | 万人群写入压力巨大 | 小群、协作 |

结论：

- 大群/直播：读扩散。
- 小群/协作：写扩散。
- 混合方案：普通群写扩散，超大群切读扩散。

### 5.6 未读数一致性

单聊未读数通常包括：

- 用户总未读数。
- 会话未读数。

可以用 Redis + Lua 保证总未读和会话未读原子更新。

群聊未读数可以用 Hash：

```text
HINCRBY msg:noreadcount:{gid} {uid} 1
```

但万人群不能每条消息都对所有用户写 Redis，容易造成写放大。优化方向：

- 客户端主维护，服务端兜底。
- 内存累计。
- 定时批量刷 Redis。
- 大群减少精确未读，改成弱提示。

### 5.7 百万在线直播间

直播互动和普通 IM 的目标不同：直播更重吞吐和稳定性，允许非核心消息丢弃。

核心优化：**绕过 Redis 路由查询，改成 MQ 广播 + Netty 本地连接缓存**。

```text
直播消息 -> MQ 广播 -> 所有 Netty 节点
                         |
                         v
                  只推送本机在线用户
```

每台 Netty 维护本机连接的直播用户列表，不再每条消息都查 Redis 判断用户在哪台机器。

### 5.8 稳定性优先

直播间每秒几十条消息用户已经难以完整阅读，系统不应为所有消息强可靠牺牲稳定性。

常见策略：

- 用户发言频率限流。
- 热点房间限流。
- 礼物/红包/系统通知强可靠，普通弹幕可降级。
- 非核心消息在过载时直接丢弃。
- 消息优先级队列，核心消息优先投递。

**总结口径**：

> 单聊强调强可靠，靠 ACK、离线补偿和幂等；群聊强调写放大控制，大群读扩散，小群写扩散；直播强调吞吐和稳定性，核心消息可靠，非核心消息可限流丢弃。

## 六、典型项目答题卡

### 6.1 游戏系统项目

**技术栈**：MySQL、Redis、Netty、Protobuf、Spring、Tomcat、Nginx、Dubbo、RabbitMQ/Kafka、ZooKeeper、Canal、监控告警。

**服务拆分**：

- 网关服务
- API 服务
- 游戏逻辑服务
- 大厅业务服务
- 任务服务
- 回放记录服务
- 比赛组队服务
- 集群管控服务
- 离线消息通知服务
- MySQL 分库扩容与数据迁移模块

**可讲亮点**：

1. Netty 自研网关，以 C/S 模式接 App，转发到下游 Dubbo 服务。
2. Netty3 扩展支持批量 decode，降低包解析开销。
3. 后台配置支持下游服务优雅下线和流量切换。
4. 按机器规格配置不同负载权重。
5. 游戏逻辑分层：Room 负责排队组队，Table 负责牌局管理，Game 抽象接口支持不同游戏实现动态加载。
6. 优化 GC 参数并使用对象池，将 FGC 卡顿优化到 0 FGC。
7. 拆分不同线程池，隔离不同类型请求。
8. 结算事务拆分为最终一致，采用并发与异步提交；RocksDB 做 MQ 和 MySQL 落地兜底，P99 从 800ms 降到 100ms。
9. 集群管控和比赛组队服务基于 ZooKeeper 实现主备部署。

**面试高频追问**：

| 问题 | 回答方向 |
| --- | --- |
| 为什么自研 Netty 网关 | 长连接、低延迟、协议可控、批量解码和连接状态管理 |
| 对象池为什么能降低 GC | 高频临时对象复用，减少年轻代分配和晋升压力 |
| 结算为什么不能一个大事务 | 大事务锁等待长、RT 高、失败影响面大；拆成事件驱动最终一致 |
| RocksDB 兜底做什么 | 本地持久化 pending 事件，防止 MQ/DB 短暂异常导致结算丢失 |

### 6.2 支付系统项目

支付系统项目可以重点讲四类能力：

| 方向 | 表达重点 |
| --- | --- |
| 支付链路 | 下单、支付、回调、查单、补单、关单 |
| 资金安全 | 幂等、防重、签名验签、金额校验、状态机 |
| 分布式事务 | TCC、本地消息表、事务消息、最大努力通知 |
| 对账清结算 | 渠道账单、内部流水、差异处理、日切、补偿 |

**项目难点口径**：

> 支付系统最难的是异常链路，而不是正常支付。比如渠道回调超时、重复回调、支付成功但本地更新失败、用户取消但渠道成功扣款，都要靠状态机、幂等号、查单补偿和对账闭环保证最终资金一致。

### 6.3 撮合系统项目

撮合系统核心模块：

| 模块 | 职责 |
| --- | --- |
| 用户 | 提交委托价格和数量，生成订单 |
| 网关 | 收集订单，派发给撮合引擎 |
| 撮合引擎 | 按价格优先、时间优先撮合订单，生成成交记录 |
| 撮合分区 | 不同交易对/业务分区独立撮合，互不影响 |
| 数据库 | 持久化订单、成交、流水 |

核心算法：

1. 买卖队列保存无法立即成交的订单。
2. 买队列按价格从高到低，同价格按时间从早到晚。
3. 卖队列按价格从低到高，同价格按时间从早到晚。
4. 新买单到来时，匹配卖队列头部，若卖价 <= 买价则成交，否则进入买队列。
5. 新卖单到来时，匹配买队列头部，若买价 >= 卖价则成交，否则进入卖队列。

**内存撮合取舍**：

| 方案 | 优点 | 风险 | 补偿 |
| --- | --- | --- | --- |
| 数据库撮合 | 持久化强 | I/O 多，TPS 低 | 适合低频交易 |
| 内存撮合 | 延迟低、吞吐高 | 进程故障数据易丢 | WAL、快照、多机热备 |

多机热备本质是用一主多备接管故障主节点，保证撮合服务连续性。金融场景不能只追求性能，必须同时保证订单顺序、成交一致性和故障恢复。

### 6.4 区块链 / OTC 项目

项目类型：数字资产交易、OTC、钱包、区块链基础设施。

**技术栈口径**：

| 币种 | 技术实现 |
| --- | --- |
| BTC | bitcoinj，UTXO 模型 |
| ETH / ETC / LTC | web3j，EVM 体系 |
| USDT | OmniCore + ZeroMQ 区块广播 |
| EOS | EOSIO |

核心能力：

- 多链钱包。
- 区块监听。
- 充值/提现。
- 交易确认。
- 对账系统。
- 风控与资金安全。

安全设计：

- 私钥、公钥、地址体系。
- SHA256、RIPEMD160、ECC。
- 热钱包/冷钱包隔离。
- 多签名。
- 地址隔离。
- 区块回滚防护。
- 多确认机制。

高频问题：

| 问题 | 回答方向 |
| --- | --- |
| 什么是 UTXO | 未花费交易输出，BTC 用输入引用旧 UTXO，输出生成新 UTXO |
| 如何防止双花 | PoW 共识、多确认、最长链/最重链、交易状态确认 |
| Merkle Tree 作用 | 快速验证交易是否在区块中，支持 SPV |
| 区块链如何保证不可篡改 | 哈希链 + 共识 + 修改历史成本极高 |
| 链上/链下如何取舍 | 链上可信但慢且贵；链下高性能但需平台信用和对账 |

**架构师级总结**：

> 区块链不是简单数据库，而是密码学、分布式系统、博弈论和金融模型的组合。本质是用技术和经济激励构建去中心化信任。
