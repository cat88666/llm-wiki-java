### 0.4 完整网络链路

**棋牌游戏平台（一面被追问"gateway 之外"）**：

```
客户端 App（TCP 长连接 / WebSocket）
  ↓
运营商网络 → BGP 多线接入（电信/联通/移动）
  ↓
SLB / LVS（四层负载均衡，分发到网关集群）
  ↓
自研 Netty 网关（七层，有状态）
  ├── 认证鉴权（Token / Session 验证）
  ├── Protobuf 协议解析 & 批量 decode
  ├── 心跳检测 & 连接管理（空闲断连/重连）
  └── 路由转发（按游戏类型 / 房间路由）
        ↓
   ZooKeeper 服务注册发现
        ↓
   Dubbo RPC（内网）
        ↓
   游戏逻辑服 / 大厅服 / 任务服 / 比赛服 / 回放服 / 集群管控服
        ↓
   MySQL / Redis / RocketMQ / RocksDB
```

**口径**：

> Gateway 是我们自研的 Netty 接入层，它之上是 SLB 做四层负载均衡，用 BGP 多线接入保证跨运营商延迟；它之下通过 ZooKeeper 服务发现，用 Dubbo RPC 调用各业务服务。Netty 网关本身是有状态的，维护用户连接和在线状态，不能替换成无状态 HTTP 网关。

**20 个实例不奇怪的解释**：

> 服务拆成了 8 个微服务（网关/API/逻辑/大厅/任务/比赛/回放/集群管控），每个生产至少 2 实例保证高可用，加上主备和隔离部署约 20 个是正常规模。棋牌游戏是长连接，日活几千但在线用户同时维持连接，峰值在线连接数是日活的 30-50%，连接维护本身有资源开销。

### 0.6 DDD 基础口径

**二面诚实表述**（不再过度包装）：

> 我对 DDD 有 Tactical Design（战术设计）的实践，在支付和游戏项目中用了聚合根划分事务边界。Strategic Design（上下文映射/防腐层/事件风暴）实践较少，不会声称精通。

**DDD 核心概念**（被追问时必须答出）：

| 概念 | 一句话解释 | 项目举例 |
|------|-----------|---------|
| 聚合根（Aggregate Root） | 一组相关对象的一致性边界，外部只能通过聚合根操作内部 | 支付订单是聚合根，支付记录/退款记录是其内部实体 |
| 实体（Entity） | 有唯一标识、生命周期、可变的对象 | 支付渠道记录、退款单 |
| 值对象（Value Object） | 无标识、不可变、描述属性 | 金额（value + currency 的组合） |
| 领域服务（Domain Service） | 跨多个聚合根的业务逻辑，不属于任何单一聚合根 | 转账：涉及付款账户和收款账户两个聚合根 |
| 应用服务（Application Service） | 编排领域服务/聚合根，处理事务边界和外部适配 | PaymentAppService 调 OrderDomain + PaymentDomain |
| 仓储（Repository） | 聚合根的持久化接口，隔离领域层与数据层 | OrderRepository.save(order) |
| 有界上下文（Bounded Context） | 同一术语在不同上下文含义不同，需要显式边界 | "用户"在支付上下文是账户，在游戏上下文是玩家 |

**实际答法**：

> 以支付项目为例：PaymentOrder 是聚合根，内含 PaymentRecord（支付记录）和 RefundRecord（退款记录）这两个实体，外部服务不能直接操作这两张表，只能通过 PaymentOrder 的领域方法修改状态，这保证了订单状态机的一致性。金额用值对象封装，防止直接传裸数字导致单位错误。

### 0.7 事务机制（InnoDB 层）

**一面被说"只停留在使用层面"，需要讲出 InnoDB 如何实现 ACID**：

**原子性（A）← `undo log`**

- 每次数据修改前，写 undo log 记录旧值
- 事务回滚时，按 undo log 链逆向还原数据
- MVCC 也复用 undo log 提供历史版本

**持久性（D）← `redo log`（WAL）**

- 修改数据先写 redo log（磁盘顺序写），再修改 Buffer Pool（内存）
- crash recovery 时：扫 redo log，把未刷盘的脏页重做
- **两阶段提交**（redo log 与 binlog 一致性保证）：
  ```
  redo log prepare → binlog flush → redo log commit
  崩溃在 binlog 前 → 回滚（redo log 未 commit）
  崩溃在 commit 前 → 重做提交（以 binlog 为准）
  ```

**隔离性（I）← `MVCC + 锁`**

- 每行隐含 `DB_TRX_ID`（最近修改的事务 ID）和 `DB_ROLL_PTR`（指向 undo log 版本链）
- **ReadView 四字段**（每次快照读创建一次）：

  | 字段 | 含义 |
  |------|------|
  | `creator_trx_id` | 创建此 ReadView 的事务 ID |
  | `trx_ids` | 创建时所有活跃事务 ID 列表 |
  | `up_limit_id` | trx_ids 中最小值（小于此值的修改对我可见） |
  | `low_limit_id` | 下一个将分配的事务 ID（≥ 此值的修改对我不可见） |

- **锁类型**：行锁（Record Lock）/ 间隙锁（Gap Lock）/ Next-Key Lock（行锁 + 间隙锁，防幻读）

**一致性（C）**：前三者共同保证，InnoDB 不直接管业务约束（那是应用层的事）

**二面口径**：

> 事务底层不只是 @Transactional，InnoDB 靠 redo log 保证 Durability——WAL 顺序写，crash recovery 时重做；靠 undo log 保证 Atomicity——回滚时按版本链还原；靠 MVCC 和锁保证 Isolation——MVCC 的核心是 ReadView 快照读，通过 creator_trx_id 和 trx_ids 活跃列表判断哪个版本对当前事务可见。

### 0.8 场景题：百万数据导出

**一面评分 70 分，缺少：具体线程使用方式、落地方案**

**1. 接口设计**（立即返回，异步处理）

```
POST /export/orders   → { taskId: "xxx", status: "PENDING" }
GET  /export/tasks/{taskId} → { status: "DONE", fileUrl: "..." }
```

**2. 线程池设计**（与业务线程池隔离，防止导出任务影响正常请求）

```java
ThreadPoolExecutor exportPool = new ThreadPoolExecutor(
    4,                                    // 核心线程：IO 密集型，CPU*2
    20,                                   // 最大线程：限制导出任务并行数
    60, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(100),       // 有界队列，超出时触发拒绝策略
    new ThreadFactoryBuilder().setNameFormat("export-%d").build(),
    new ThreadPoolExecutor.CallerRunsPolicy()  // 降级为调用方线程，天然背压
);
```

**3. 分批游标查询**（禁用 OFFSET 大翻页）

```sql
-- ❌ 错误：OFFSET 100W 需要扫描 100 万行再丢弃
SELECT * FROM orders LIMIT 1000 OFFSET 1000000

-- ✅ 正确：游标翻页，每次只扫描 1000 行
SELECT * FROM orders WHERE id > {lastId} AND {业务条件} ORDER BY id LIMIT 1000
-- lastId 初始为 0，每批取到后更新为本批最后一行 id
```

**4. 流式写文件**（防 OOM）

```java
// Excel：POI SXSSFWorkbook（滑动行窗口，内存只保留最近 100 行）
SXSSFWorkbook wb = new SXSSFWorkbook(100);
// CSV：BufferedWriter（更轻量，百万级优先选）
BufferedWriter bw = new BufferedWriter(new OutputStreamWriter(os, StandardCharsets.UTF_8));
// 每批写入后 flush，不要攒到最后一次性 flush
```

**5. 任务状态机 & 通知**

```
PENDING → RUNNING → DONE（上传 OSS，存 fileUrl，通知用户）
                  → FAILED（记录异常，允许重试）
                  → TIMEOUT（超过 30min 未完成，告警）
通知方式：WebSocket 推送 / 邮件 / 用户前端轮询
```

**6. 边界防护**

- 文件临时写 /tmp，上传 OSS 后删除本地文件，防磁盘打满
- 单文件超过 50MB 自动分卷压缩
- 同一用户同时只允许 1 个进行中的导出任务（防重复提交）

**二面口径**：

> 百万导出三个核心问题：① 不能同步阻塞请求，要异步任务 + taskId 轮询；② 不能 OFFSET 大翻页，要游标翻页防慢查询；③ 不能全量加载内存，要流式写 SXSSFWorkbook 或 CSV。线程池独立隔离，有界队列 + CallerRunsPolicy 做背压；任务有状态机，超时和失败都能被监控和重试。

### 2.1 游戏平台中台化建设

**一句话**：主导从项目组模式向平台化转型，为千万级用户提供高性能网关及通用逻辑服务，实现 0 Full GC、P99 从 800ms 降至 100ms。

**技术栈**：MySQL、Redis、Netty、Protobuf、Spring、Tomcat、Nginx、Dubbo、RabbitMQ/Kafka、ZooKeeper、Canal、监控告警。

**服务拆分**：
- 网关服务 / API 服务 / 游戏逻辑服务 / 大厅业务服务
- 任务服务 / 回放记录服务 / 比赛组队服务 / 集群管控服务
- 离线消息通知服务 / MySQL 分库扩容与数据迁移模块

**核心贡献**：

1. **自研 Netty 网关**：C/S 模式接 App，转发下游 Dubbo 服务；Netty3 扩展支持批量 decode，降低包解析开销；后台配置支持下游服务优雅下线和流量切换；按机器规格配置不同负载权重。
2. **游戏逻辑分层**（DIP 模式）：Room 负责排队组队，Table 负责牌局管理，Game 抽象接口支持不同游戏实现动态加载，大幅提升业务迭代效率。
3. **0 Full GC**：优化 GC 参数结合对象池技术，复用高频临时对象，减少年轻代分配和晋升压力；拆分不同线程池隔离不同类型请求。
4. **P99 800ms → 100ms**：结算事务拆分为最终一致，并发与异步提交；RocksDB 做 MQ 和 MySQL 落地兜底，确保极端网络环境下数据最终一致性。
5. **高可用**：集群管控和比赛组队服务基于 ZooKeeper 实现主备部署。

**面试高频追问**：

| 问题 | 回答方向 |
| --- | --- |
| 为什么自研 Netty 网关 | 长连接、低延迟、协议可控、批量解码和连接状态管理；SCG 不支持二进制协议 |
| 对象池为什么能降低 GC | 高频临时对象复用，减少年轻代分配和晋升压力，避免 Full GC |
| 结算为什么不能一个大事务 | 大事务锁等待长、RT 高、失败影响面大；拆成事件驱动最终一致更健壮 |
| RocksDB 兜底做什么 | 本地持久化 pending 事件，防止 MQ/DB 短暂异常导致结算数据丢失 |

### 2.3 IM 系统

**一句话**：IM 不是简单聊天，而是**低延迟、高可靠的实时消息分发系统**，要处理千万/亿级用户、多端同步、离线消息、群聊、直播互动、未读数和历史消息存储。

**典型分层**：

| 分层 | 职责 |
| --- | --- |
| 客户端 | 与服务端建立长连接，收发消息，维护本地消息状态 |
| 网关/接入层 | 维护连接、鉴权、路由、推送、心跳 |
| 服务层 | 聊天、离线消息、群聊、红包、直播互动等业务逻辑 |
| 存储层 | 消息内容、账号、关系链、会话、未读数、索引 |

**消息可靠投递**（At Least Once + 幂等消费）：

1. 客户端生成全局唯一 `msgId`，超时或失败自动重试。
2. 服务端按 `msgId` 去重；先可靠落库/写 MQ，再向下游投递。
3. 接收端 ACK；未 ACK 消息进入重推或离线补偿。

> 系统不追求天然 Exactly Once，而是用唯一 msgId、幂等写、ACK 和补偿任务实现最终可靠。

**离线消息高性能设计**：上线拉取（不推离线用户），Redis ZSet 扛热点，DB 做兜底；单 Key 过大时按周/月拆 Key。

```text
Key: offline_msg_{uid}  Score: mid  Value: msg
ZADD 添加 | ZREVRANGE 拉取最近 | ZREMRANGEBYSCORE 删除已确认
```

**海量历史消息**：内容表（按 mid 分片）+ 信箱索引表（按 owner_uid 分片）分离，支持双方独立删除，查询只扫小索引表。

```text
im_msg_content(mid, content, type, sender_id, recipient_id, create_time)
im_user_msg_box(owner_uid, other_uid, mid, box_type, create_time)
```

**群聊：读扩散 vs 写扩散**：

| 模型 | 做法 | 优点 | 缺点 | 适合场景 |
| --- | --- | --- | --- | --- |
| 读扩散 | 群消息只存一份 | 写入快 | 已读/未读精细化难 | 大群、直播 |
| 写扩散 | 给每个用户写一条索引 | 未读/已读精确 | 万人群写入压力巨大 | 小群、协作 |

混合方案：普通群写扩散，超大群切读扩散。

**未读数一致性**：Redis + Lua 原子更新总未读和会话未读；万人群改为客户端主维护 + 内存累计 + 定时批刷，大群用弱提示替代精确计数。

```text
HINCRBY msg:noreadcount:{gid} {uid} 1
```

**百万在线直播间**：绕过 Redis 路由查询，改成 MQ 广播 + Netty 本地连接缓存。每台 Netty 维护本机连接的直播用户列表，只推本机在线用户。礼物/红包/系统通知强可靠；普通弹幕过载时直接丢弃；用户发言频率 + 热点房间双层限流。

> 单聊强调强可靠（ACK + 离线补偿 + 幂等）；群聊强调写放大控制（大群读扩散，小群写扩散）；直播强调吞吐和稳定性（核心消息可靠，非核心消息可限流丢弃）。

### 2.4 撮合系统（区块链交易所）

**核心模块**：

| 模块 | 职责 |
| --- | --- |
| 用户 | 提交委托价格和数量，生成订单 |
| 网关 | 收集订单，派发给撮合引擎 |
| 撮合引擎 | 按价格优先、时间优先撮合，生成成交记录 |
| 撮合分区 | 不同交易对/业务分区独立撮合，互不影响 |
| 数据库 | 持久化订单、成交、流水 |

**核心算法**：买队列（价高→价低，同价时间优先）/ 卖队列（价低→价高，同价时间优先）。新买单匹配卖队列头部，卖价 ≤ 买价则成交，否则进买队列；新卖单反之。

**内存撮合取舍**：

| 方案 | 优点 | 风险 | 补偿 |
| --- | --- | --- | --- |
| 数据库撮合 | 持久化强 | I/O 多，TPS 低 | 适合低频交易 |
| 内存撮合 | 延迟低、吞吐高 | 进程故障数据易丢 | WAL + 快照 + 多机热备 |

金融场景不能只追求性能，必须同时保证订单顺序、成交一致性和故障恢复。

### 2.5 区块链 / OTC 项目（CoinsOTC）

**技术栈**：

| 币种 | 技术实现 |
| --- | --- |
| BTC | bitcoinj，UTXO 模型 |
| ETH / ETC / LTC | web3j，EVM 体系 |
| USDT | OmniCore + ZeroMQ 区块广播 |
| EOS | EOSIO |

**核心能力**：多链钱包、区块监听、充值/提现、交易确认、对账系统、风控与资金安全。

**技术亮点（CoinsOTC 特有）**：

| 亮点 | 实现方式 | 效果 |
|------|---------|------|
| 低代码分布式锁注解方案 | 基于 Redisson 自研 `@DistributedLock` 注解，AOP 拦截，SpEL 动态 key | 多微服务账户资金并发一致性，业务代码量减少 ~80% |
| Server 层队列合并 DB 操作 | 交易高峰期将多笔 DB 写合并批量提交，引入 100ms 微延迟换吞吐 | 系统吞吐提升 30%+，DB 连接压力显著下降 |
| 多链集成 | bitcoinj（BTC/UTXO 模型）+ web3j（ETH/EVM 体系）| 主流代币开户、充提、冷热钱包划转全链路打通 |

**追问口径（Redisson 注解方案）**：
> 传统做法每处加锁都要写 `redissonClient.getLock(key).lock()` 样板代码，且 key 构造分散。封装 `@DistributedLock(key="#userId + ':order'")`，AOP 解析 SpEL 拿到运行时 key，统一处理加锁/释放/超时/异常日志，业务代码只剩一行注解，锁策略在注解参数层统一收口。

**安全设计**：私钥/公钥/地址体系、SHA256/RIPEMD160/ECC、热钱包/冷钱包隔离、多签名、地址隔离、区块回滚防护、多确认机制。

**高频问题**：

| 问题 | 回答方向 |
| --- | --- |
| 什么是 UTXO | 未花费交易输出，BTC 用输入引用旧 UTXO，输出生成新 UTXO |
| 如何防止双花 | PoW 共识、多确认、最长链/最重链、交易状态确认 |
| Merkle Tree 作用 | 快速验证交易是否在区块中，支持 SPV |
| 区块链如何保证不可篡改 | 哈希链 + 共识 + 修改历史成本极高 |
| 链上/链下如何取舍 | 链上可信但慢且贵；链下高性能但需平台信用和对账 |

> 区块链本质是用技术和经济激励构建去中心化信任（密码学 + 分布式系统 + 博弈论 + 金融模型的组合）。

### 2.6 CloudStack 云计算平台网络改造

**背景**：CloudStack 开源云计算平台，中国多运营商（电信/移动/联通）混合环境下，原生网络架构不能做多线路优化，需底层源码级改造。

**核心贡献**：

- **多运营商网络适配**：重写 CloudStack 底层网络架构，支持电信/移动/联通多线路优先级路由，解决跨运营商延迟和访问不通问题。
- **硬件插件开发**：适配并集成深信服等硬件设备，自研设备插件替代虚拟路由（VR），实现高性能硬件防火墙、NAT 及负载均衡集成，避免纯软件 VR 的性能瓶颈。

**面试口径**：

> 难点在于 CloudStack 的插件体系（Network Element / Provider 模型）需要深入阅读源码，结合运营商网络特性做定制化适配，不是配置层面的改动。硬件插件开发需实现 CloudStack 的 NetworkElement 接口替代原生虚拟路由器。

**高频追问**：

| 问题 | 回答方向 |
|------|---------|
| 为什么不用 SDN | 当时环境已有硬件设备投资，适配硬件比引入 SDN 改造更经济可行 |
| 多运营商路由怎么做 | 基于源 IP 地址段判断运营商归属，路由策略指定对应出口网关 |
| VR 替代后怎么保证高可用 | 硬件设备主备模式 + 健康检测，故障秒级切换 |

### 3.3 微服务拆分原则

1. 单一职责、高内聚低耦合，避免一个服务承担多类业务。
2. 服务粒度适中，避免接口级拆分导致调用链爆炸。
3. 以业务模型为核心，按用户、订单、商品、支付、营销等领域拆分。
4. 演进式拆分，先粗后细，避免一开始过度微服务化。
5. 避免环形依赖和双向依赖，服务依赖关系应尽量是 DAG。

**项目口径**：

> 我拆服务时不会只按表拆，而是按业务闭环拆。比如交易链路中用户、订单、支付、营销、资金是不同边界；支付和资金必须保证事务边界清晰；营销不能反向强依赖交易，否则发布和故障都会互相拖累。

## 四、软技能

### 4.1 技术决策分歧怎么处理？

网关重构：一半人 SCG，一半人 Netty。让两派各写 1 天 PoC 压测说话。Netty 延迟低 30% 但维护成本 3 倍。决策：游戏内网关 Netty（性能第一），运营后台 SCG（迭代第一）。数据驱动，不拍板。

### 4.2 怎么培养初级开发？

渐进式任务（配置改→接口→模块→系统设计）+ Code Review 解释"为什么"建立工程直觉 + 参与线上问题排查（旁边兜底）+ 明确半年目标定期 1:1。核心：给方向、给机会、给反馈，不替他做决定。

### 4.3 印象最深的线上事故？

支付某银行渠道回调超时激增，2000 笔订单卡在"支付中"：

1. **止血（5min）**：回滚该渠道到上一版本
2. **定位（10min）**：HTTP Client 连接池从 200 改成了 20，高峰期耗尽
3. **修复**：修正参数，5% 灰度验证后全量
4. **补单**：向银行查单，批量更新 2000 笔
5. **复盘**：CI 增加关键配置参数变更的额外审批

15 分钟恢复，零资损，30 分钟补单完毕。

### 4.4 推荐反问问题

- 你们分布式事务怎么解决的，有没有遇到 TCC 悬挂/空回滚？
- JVM 可观测体系（JVM metrics + Skywalking 联动）是怎么搭的？
- oncall 机制和事故响应 SLA 是多少？
- P7 到 P8 晋升主要看什么？
