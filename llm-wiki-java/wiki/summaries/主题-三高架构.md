## 一、三者制约关系

```
高并发 → 引入异步/分片/分布式 → 牺牲强一致（最终一致）
高可用 → 引入副本/冗余/多活  → 牺牲延迟（性能下降）
高性能 → 引入缓存/本地计算   → 牺牲一致性（脏数据风险）
```

**核心矛盾**：三高互相增压，提升任意一高通常需要在另外两个维度付出代价。设计时需要明确**业务容忍边界**，而非追求三者极值。

### CAP 原则与 BASE 理论

**CAP**：分布式系统不能同时满足一致性（C）、可用性（A）、分区容错性（P）。网络分区不可避免，P 必选，因此只能在 CP 和 AP 之间取舍：

| 选择 | 牺牲 | 典型系统 |
|------|------|---------|
| CP | 可用性（分区时拒绝写/降级）| ZooKeeper、Etcd、MySQL 主从（同步复制）|
| AP | 强一致性（分区时允许读旧数据）| Eureka、Redis Cluster、DynamoDB |

**BASE**（CAP 的工程妥协）：系统不必保证强一致，只需做到：
- **BA（Basically Available）**：允许部分降级（响应变慢、非核心功能不可用）
- **S（Soft State）**：允许数据同步中间态（主从复制延迟期间）
- **E（Eventually Consistent）**：不要求强一致，允许经过一段时间后达到一致

> CAP 是理论上限，BASE 是工程常态；大多数微服务业务选 AP + 最终一致。

## 二、高并发：限流 · 熔断 · 预热

> 核心问题：如何让系统在大量并发请求下不崩溃？

### 设计思路

**高并发本质**：用有限资源承载尽可能多的请求，同时保证系统可用性、稳定性和一致性。

**架构师高并发思维（性能优先级）**：
```
1. CPU Cache 命中率  → 数据局部性，顺序访问比随机快 100×（数组 > 链表）
2. 顺序 IO           → 磁盘顺序写比随机写快 100×（Kafka 顺序写、LSM 树核心）
3. 无锁              → 锁竞争是并发性能杀手（CAS / Disruptor RingBuffer）
4. 少线程            → 过多线程上下文切换 > 并发收益；IO 密集型改用异步非阻塞
5. 事件驱动          → 单线程 EventLoop 处理千级并发连接（Netty / Nginx 模型）
```

**并发编程三要素**（多线程正确性的基础）：

| 要素 | 含义 | 保证手段 |
|------|------|---------|
| 原子性 | 操作不可分割，多步骤同时成功或同时失败 | `synchronized`、`Lock`、`AtomicXxx`、数据库事务 |
| 有序性 | 程序执行顺序与代码顺序一致（防止指令重排）| `volatile`（禁止重排）、`synchronized`、happens-before 规则 |
| 可见性 | 一个线程对共享变量的修改，另一个线程能立即看到 | `volatile`（刷回主内存）、`synchronized`（加锁时强制同步）|

**三板斧**：
```
缓存          → 减少 DB/RPC 次数（本地缓存 > Redis > DB）
削峰填谷（异步）→ 用 MQ 把流量洪峰转为平缓的消费流
限流          → 主动拒绝超量请求，保护容量上限
```

**CPU 硬件并发基础**（高并发性能的物理边界）：

| 概念 | 关键数据 | 高并发意义 |
|------|---------|----------|
| Cache 层级（L1/L2/L3）| L1 <1ns → L3 ~20ns → 内存 ~100ns | 热点数据必须命中 L1/L2，跨层代价翻倍 |
| Cache Line（64 byte）| CPU 每次加载 64 字节 | 数组遍历远快于链表（空间局部性）；对象字段紧凑布局能减少 Cache miss |
| False Sharing（伪共享）| 不同线程变量落在同一 Cache Line | 线程间写操作触发 MESI 失效广播；JDK8 `@Contended` 填充解决 |
| MESI 协议 | Modified/Exclusive/Shared/Invalid | `volatile` 写触发 MESI 失效广播，代价远高于普通写 |
| NUMA 架构 | 多 CPU Socket，各有本地内存 | 跨 NUMA 访问代价 3~10×；Kafka/Netty 用 CPU Affinity 绑核缓解 |
| 内存屏障 | StoreLoad/LoadLoad/StoreStore | `volatile`/`synchronized` 底层依赖内存屏障禁止指令重排 |

**扩展策略**：

| 策略 | 做法 | 优势 | 限制 |
|------|------|------|------|
| 垂直扩展 | 升级 CPU、内存、SSD | 改动小，见效快 | 成本高，有上限 |
| 水平扩展 | 加机器，无状态服务横向扩容 | 可无限扩展，稳定 | 需无状态化改造，共享瓶颈（DB/缓存）仍存在 |

任何系统的处理能力都有上限，瓶颈可能是 CPU、内存、线程池、DB 连接数、Redis 网络带宽或下游服务配额。当请求量超过容量上限时，系统只有两种选择：

1. **排队等待**：延迟快速升高，最后大量请求一起超时。
2. **主动拒绝**：让一部分请求快速失败，保住核心请求和系统存活。

限流选择第二条路：宁愿少量请求快速失败，也不让所有请求慢速失败。熔断处理的是依赖故障：当下游异常时，上游不继续阻塞线程等待，而是快速失败或降级，防止级联雪崩。

### 限流

**目的**：保护系统不被超量请求打垮，核心是"拒绝"或"延迟"超出容量的请求。

| 算法 | 原理 | 突发流量 | 适用场景 |
|------|------|---------|---------|
| 计数器（固定窗口） | 时间窗口内计数，满则拒绝 | 临界突刺（窗口边界） | 简单场景 |
| 滑动窗口 | 时间窗连续滑动，精准控制 | 平滑应对 | API 限流 |
| 漏桶 | 以固定速率"漏出"请求 | 无法应对 | 平滑输出（MQ 消费）|
| 令牌桶 | 固定速率生产令牌，可积累 | 可突发（桶内令牌数） | 接口限流（Guava RateLimiter）|
| 自适应限流 | 结合 CPU/Load/RT/QPS 动态调整阈值 | 自动降配 | Sentinel 系统保护 |

**漏桶 vs 令牌桶**：

```
漏桶：请求进入固定容量桶，以固定速率流出，桶满丢弃
令牌桶：按固定速率生产令牌，请求拿到令牌才执行，令牌可积累
```

漏桶输出更稳定，适合调用有严格速率限制的下游；令牌桶允许短时突发，更适合大多数 Web API。Guava `RateLimiter` 是典型令牌桶实现。

**滑动窗口**解决固定窗口的边界突刺问题。固定窗口在秒级边界可能出现 0.1 秒内打入两倍配额；滑动窗口把 1 秒切成多个小片，统计"过去 1 秒"的请求数。Redis ZSet 可用时间戳做 score 实现滑动窗口，但内存和网络开销都高于本地计数。

**自适应限流**不依赖固定阈值，而是根据系统指标动态收缩流量。Sentinel 常用触发信号包括 CPU 使用率、系统 Load、平均 RT、并发线程数和 QPS。优点是能跟随系统状态变化，缺点是触发有滞后，仍需要压测给出初始安全水位。

**单机 vs 集群限流**：
- 单机：Guava RateLimiter / Sentinel 本地规则，保护单个节点不过载
- 集群：Redis 计数器 + Lua / Sentinel 集群规则，保护 DB、第三方接口等全局资源
- **两者都需要**：集群限 1000 不代表每台能抗 100，负载不均时可能单机被打爆

集群限流每个请求都访问 Redis 会增加 0.1~1ms 级延迟，高频接口可用"批量申请令牌"的方式减少网络开销。

**Redis 分布式限流实现（两种方案）**：

```java
// 方案一：ZSet 滑动窗口（精确，内存消耗略高）
String key = "rate:limit:" + userId;
long now = System.currentTimeMillis();
redisTemplate.executePipelined(conn -> {
    conn.zRemRangeByScore(key.getBytes(), 0, now - 1000); // 清 1 秒前的记录
    conn.zCard(key.getBytes());                            // 当前窗口计数
    conn.zAdd(key.getBytes(), now, UUID.randomUUID().toString().getBytes());
    conn.expire(key.getBytes(), 2);
    return null;
});
// 若 zCard 返回值 >= limit → 拒绝

// 方案二：Lua 令牌桶（原子性，生产推荐）
String lua =
    "local tokens = tonumber(redis.call('get', KEYS[1]) or ARGV[1])\n" +
    "if tokens >= 1 then\n" +
    "  redis.call('set', KEYS[1], tokens - 1, 'EX', 60)\n" +
    "  return 1\n" +   -- 允许
    "end\n" +
    "return 0";        -- 拒绝
Long result = redisTemplate.execute(new DefaultRedisScript<>(lua, Long.class),
    List.of("rate:token:" + userId), String.valueOf(capacity));
```

**选型结论**：优先 **Sentinel**（规则热推送 + 集群模式 + 零侵入）；无 Sentinel 时用 **Lua 令牌桶**（原子性）；单机限流用 **Guava RateLimiter**。

### 熔断

**三态状态机**：关闭（正常）→ 开启（全拒绝）→ 半开启（允许少量探测）

```
请求失败率/慢响应超阈值 → 开启熔断
半开启状态通过 → 恢复关闭
半开启状态失败 → 重新开启
```

| 状态 | 行为 | 转换条件 |
|------|------|----------|
| Closed | 正常放行请求 | 错误率、慢调用比例或超时率超过阈值 |
| Open | 快速失败，不调用下游 | 经过熔断窗口后进入 Half-Open |
| Half-Open | 放入少量探测请求 | 探测成功则关闭，失败则重新打开 |

**工具**：Sentinel（推荐）> resilience4j > Hystrix（已停更）

**为什么需要熔断**：重试 → 下游雪上加霜 → 服务雪崩。熔断使上游快速失败，给下游喘息时间。

熔断必须配合降级策略，否则只是把慢失败变成快失败。常见降级包括返回缓存数据、默认值、兜底文案、排队提示或关闭非核心功能。

### 锁策略

**高并发下悲观锁 > 乐观锁**（反直觉）：

乐观锁"先干活后加锁"——在并发冲突频繁时，浪费大量计算资源组装模型，最后发现版本号变了，全部白费。

悲观锁"先加锁再干活"——fail-fast，加锁失败立即返回失败，不浪费后续计算。

> 结论：高并发写场景优先用分布式锁（悲观思想），而非乐观锁重试。

**分布式锁实现选型**：

| 方案 | 原理 | 适用 | 注意 |
|------|------|------|------|
| Redis `SET NX + Lua + TTL` | 原子性 set-if-not-exist + 过期自动释放 | 高性能，低延迟 | 必须做锁续期（Redisson WatchDog）；防误删（value=唯一标识）|
| ZooKeeper 临时有序节点 | 最小序号节点持有锁，节点断开自动释放 | 强一致，自动续期 | 性能低于 Redis，ZK 本身是 CP 系统 |
| Etcd Lease+KeepAlive | 租约机制，客户端心跳续期 | K8s 生态，强一致 | 复杂度高 |

**两个关键设计**：
- **锁续期**：业务执行时间不确定，固定 TTL 可能业务未完锁已过期；Redisson 的 WatchDog 每 TTL/3 自动续期
- **防误删**：释放锁前必须先判断 value 是否是自己的（Lua 脚本保原子性），防止 A 的锁超期后 B 加锁，A 把 B 的锁删掉

### 预热

- **缓存预热**：大促前将热点数据提前加载到缓存，避免冷启动时 cache miss 穿透打爆 DB
- **JIT 预热**：JVM JIT 编译需要"热身"，刚发布时 RT 偏高，全链路压测需排除 JIT 影响
- **连接预热**：线程池、DB 连接池冷启动时没有满连接，需要脉冲式逐步施压
- **接口预热**：发布后先小流量调用关键接口，再逐步放量
- **限流预热**：Guava RateLimiter 的 `warmupPeriod` 让初始速率逐步升至目标速率

### 权衡与选型

1. **令牌桶 vs 漏桶**：大多数业务选令牌桶；只有下游有严格速率合约时才更适合漏桶。
2. **阈值太低 vs 太高**：太低误伤正常用户，太高保护不了系统；通常用压测上限的 70%~80% 作为初始保护线。
3. **集群精确 vs 本地快速**：Redis 集群限流更准确，但有网络开销和单点依赖；本地限流快但无法保护全局资源。
4. **熔断 vs 可用体验**：熔断能阻断雪崩，但必须设计降级结果，否则用户只看到更快的失败。

## 三、高可用：副本 · 容灾 · 压测

> 核心问题：如何让系统在部分组件失败时仍然可以服务？

### 设计思路

单点必然失败：硬件故障、软件 Bug、网络抖动、机房断电都只是时间问题。高可用的核心不是"避免故障"，而是通过**冗余、隔离、快速恢复**让局部故障对用户尽量不可见。

高可用的代价也很明确：冗余带来资源成本，多副本带来一致性问题，自动切换带来误判和脑裂风险。设计时要在**成本、复杂度、可用性**之间取舍。

### 可用性指标（SLA）

| 可用性 | 年宕机时间 | 典型叫法 |
|--------|----------|---------|
| 99% | 87.6h | 2 个九 |
| 99.9% | 8.76h | 3 个九 |
| 99.99% | 52.6min | 4 个九 |
| 99.999% | 5.26min | 5 个九 |

**SLA 拆解**：整体可用性 = 各组件可用性的乘积。10 个 99.9% 的组件串联 → 99%。

底层指标是 **MTTR**（平均恢复时间）和 **MTBF**（平均故障间隔时间）：

```
可用性 = MTBF / (MTBF + MTTR)
```

提升可用性要么减少故障频率，要么缩短恢复时间。工程上通常更容易通过监控、自愈、故障转移把 MTTR 降下来。

### 备份策略

| 类型 | 状态 | 恢复时间 | 成本 | 数据一致性 | 适用场景 |
|------|------|----------|------|------------|----------|
| 冷备 | 离线，不运行 | 分钟到小时 | 低 | 可能丢失上次备份后的变更 | 归档、非核心系统 |
| 暖备 | 定期同步，通常需手动切换 | 分钟级 | 中 | 可能有少量延迟 | 中等重要性服务 |
| 热备 | 实时同步，可自动切换 | 秒级 | 高 | 取决于复制模式 | 核心数据库、核心服务 |

热备常见实现包括 MySQL 主从、Redis Sentinel、Redis Cluster、多副本服务实例。可用性要求不到 99.99% 时，暖备往往更经济；金融、支付、交易链路才更常见热备或半自动切换。

### 异地多活 vs 同城容灾

| | 同城容灾 | 异地多活 |
|---|---|---|
| 目标 | 机房级灾难恢复 | 地域级灾难恢复 |
| 数据同步延迟 | 低（同城高速专线）| 高（跨地域 WAN）|
| 成本 | 中 | 高 |
| 适用 | 99.99% 可用性要求 | 99.999%+ 或监管合规 |

异地多活难点：
1. **数据同步与冲突解决**（分布式系统 CAP 的 P 不可避免）
2. **流量路由**（DNS 切换、单元化架构）
3. **本地写远端读**（主地域写，异地只读，减少跨地域同步问题）

**单元化架构**是异地多活的高级形态。系统按用户、商户、区域等维度切分为多个自包含单元，每个单元内部包含完整的服务、DB、缓存、MQ，请求尽量在单元内闭环，不跨单元调用：

```
上海单元（华东用户）       北京单元（华北用户）       新加坡单元（海外用户）
  ├── 交易服务              ├── 交易服务              ├── 交易服务
  ├── 支付服务              ├── 支付服务              ├── 支付服务
  ├── 用户 DB 子集          ├── 用户 DB 子集          ├── 用户 DB 子集
  └── Redis / MQ            └── Redis / MQ            └── Redis / MQ
                    中心单元：全量数据聚合 + 兜底路由
```

优势是跨地域延迟低、故障隔离强、扩容粒度清晰；代价是全链路改造成本极高，用户路由和数据归属必须准确，只适合超大规模核心链路。

### 负载均衡策略

| 策略 | 原理 | 适用场景 |
|------|------|---------|
| 随机 | 随机选一个节点 | 节点处理能力均等，请求开销差异小 |
| 轮询（Round Robin）| 依次分发，可加权 | 通用首选，节点性能相近 |
| 最小活跃数 | 选当前并发连接最少的节点 | 请求耗时差异大（长短任务混合）|
| 一致性哈希 | 相同 key 路由到固定节点 | 热点用户路由、缓存亲和性、有状态服务 |

> 一致性哈希配合虚拟节点可均匀分散真实节点的负载，避免单点热点。

### 高可用架构要点

1. **消除单点**：服务层多实例 + 负载均衡；数据层主从复制 + 自动故障转移；入口层多节点 + VIP 漂移或 DNS/GSLB。
2. **故障隔离**：限流保护下游，熔断快速失败，降级返回简化响应或缓存数据，舱壁模式为不同依赖分配独立线程池。
3. **弹性伸缩**：无状态服务水平扩展，K8s HPA 自动扩缩容；大促前仍要预扩容，不能完全依赖自动触发。
4. **监控与自愈**：Prometheus/Grafana 采集指标，链路追踪定位瓶颈，Liveness/Readiness Probe 做自动重启和摘流，告警按 P0/P1/P2 分级。
5. **容量规划**：用压测结果和增长模型确定安全水位，提前知道扩容触发点和共享瓶颈。

### 压测

**压测五步骤**：确定目标 → 制定计划 → 准备脚本和数据 → 执行施压 → 观察指标。

**监控指标**：QPS、RT、失败率、CPU 利用率、Load、内存、GC 次数/时长、网络 IO、报警。

**全链路压测 vs 单链路压测**：
- 单链路：关注自身，结果偏乐观，忽略资源竞争
- 全链路（阿里双11来源）：从 Nginx → 应用 → 中间件 → DB 全链路验证，暴露真实瓶颈
- 关键：**流量染色**（full_chain_stress_tag）+ **影子表**（order__shadow，压测数据写影子表）+ **影子队列**（压测消息进 shadow topic）+ **影子缓存**（Redis key 加 shadow 前缀）

生产压测要避免污染线上用户数据：压测请求必须全链路透传标记，写入影子表、影子 Topic 和影子缓存；漏配任何一个环节都可能把压测数据写进真实业务。

**扩容不是线性的**：10 台单机 300 QPS 不等于集群 3000 QPS。DB 连接数、Redis 网络带宽、负载不均、下游服务限额都会成为共享瓶颈。

## 四、高性能：缓存 · 线程池 · 过滤器

> 核心问题：如何让系统在相同资源下处理更多请求、响应更快？

### 性能指标

| 指标 | 含义 | 典型值 |
|------|------|--------|
| QPS | 每秒查询数 | 核心接口 ≥1000 |
| RT | 响应时间（平均） | 核心接口 ≤100ms |
| P90/P95/P99 | 90%/95%/99% 请求的最大 RT | P99 ≤500ms 是常见 SLA |

> P99 比平均 RT 更重要：平均 RT=50ms 可能掩盖了 1% 的请求耗时 5s（超时！）

**单机 QPS 估算**：`QPS = 线程数 × 1000 / RT(ms)`，例：200 线程，RT=100ms → 单机 2000 QPS

### 服务端性能优化方案

1. **缓存**：本地缓存（Caffeine）→ 分布式缓存（Redis）→ 减少 DB 访问
2. **异步**：CompletableFuture 并行编排、@Async 异步处理，减少串行等待
3. **批量**：批量插入（减少 DB 连接次数）、批量 RPC（减少网络开销）
4. **连接池复用**：DB 连接池、HTTP 连接池（避免频繁建立 TCP）
5. **NIO/Netty**：非阻塞 I/O，单线程处理多连接，避免线程切换开销
6. **锁优化**：减少锁粒度（ConcurrentHashMap 分段锁）、锁分离（ReadWriteLock）、锁消除（JIT）
7. **SQL 优化**：索引、深分页优化、批量操作代替循环单条
8. **压缩传输**：HTTP Gzip，减少网络带宽

### 缓存设计：四种模式与热 Key 防护

**四种缓存模式对比**：

| 模式 | 读流程 | 写流程 | 适用场景 | 代价 |
|------|--------|--------|---------|------|
| **Cache Aside（旁路）** | 缓存→miss→查 DB→回写缓存 | 先写 DB，再**删**缓存 | 99% 的读多写少业务 | 删缓存失败时存在短暂不一致 |
| **Write Through（穿透写）** | 读缓存 | 同步写缓存 + DB | 写延迟不敏感、缓存命中率高 | 每次写仍落 DB，不降低写压力 |
| **Write Behind（异步写）** | 读缓存 | 只写缓存，异步批量刷 DB | 高写吞吐：游戏积分、点赞计数 | 宕机丢未刷盘数据 |
| **Refresh Ahead（预加载）** | 读缓存（始终命中）| 检测 TTL 快到期时后台刷新 | 极低延迟、热点稳定 | 可能预加载用不到的数据 |

**Cache Aside "先删还是后删缓存"**（高频陷阱）：

```
错误：先删缓存 → 并发读在 DB 写入前回填旧值 → 长期脏缓存
正确：先写 DB → 再删缓存 → 即使删失败，下次读 miss 时自动修复
生产兜底：删缓存失败加入 MQ 重试队列，保证最终删除（延迟双删方案）
```

**热 Key 防护策略**：热 key 所有请求打单一 Redis 节点，可能打爆单节点（单节点网络带宽约 1 Gbps）。

| 方案 | 原理 | 适用 | 代价 |
|------|------|------|------|
| 本地缓存（Caffeine）拦截 | L1 命中后不走 Redis | 热点稳定、可接受秒级一致性 | 多实例各自缓存，内存翻倍 |
| Key 加随机后缀拆分 | `hot_key_{0~N}` 分散到 N 个 slot | 数据可复制（配置/商品详情）| 写时需更新所有副本 |
| Redis Cluster 读从节点 | 请求分散到 Slave | 天然多副本 | Cluster Slave 默认不读，需开启 `READONLY` |
| 随机 TTL | 相似 key 加 ±N 随机秒 | 缓存雪崩防护（非单 key 热点）| 实现简单 |

> 生产推荐组合：**Caffeine 本地缓存 + 随机 TTL**，能防住大多数热 key 和雪崩场景；极端热 key（如双十一活动页）再加 Key 拆分。

### 线程池调优

**执行优先级（5步状态机）**：
```
1. 线程数 < corePoolSize    → 创建新核心线程执行（即使有空闲线程）
2. 线程数 = corePoolSize    → 放入 workQueue 排队
3. workQueue 已满           → 创建非核心线程（直到 maximumPoolSize）
4. 线程数 = maximumPoolSize → 触发 RejectedExecutionHandler（拒绝策略）
5. 线程空闲 > keepAliveTime → 回收非核心线程（线程数降回 corePoolSize）
```

**核心参数**：`corePoolSize`、`maximumPoolSize`、`workQueue`、`handler`

**初始值公式**：
- CPU 密集型：`corePoolSize = N+1`（N=CPU 核数）
- IO 密集型：`corePoolSize = 2N ~ 4N`

**调整信号**：
- 提升线程数：队列长 + 任务拒绝多 + CPU 利用率不高
- 降低线程数：上下文切换频繁 + CPU 利用率高 + 活跃线程少

**拒绝策略选型**：

| 策略 | 行为 | 适用场景 |
|------|------|---------|
| `AbortPolicy`（默认）| 抛出 `RejectedExecutionException` | 调用方能感知并重试；需监控拒绝率 |
| `CallerRunsPolicy` | 由提交线程本身执行该任务 | **推荐**：生产者速率自动降速，不丢任务 |
| `DiscardPolicy` | 静默丢弃新任务 | 日志采集、埋点等允许丢失的场景 |
| `DiscardOldestPolicy` | 丢弃队列头部最旧任务 | 需要优先处理最新任务（实时性 > 完整性）|

> 生产最佳实践：核心业务用 `CallerRunsPolicy`（自然反压，不丢不炸）；对拒绝需要告警的场景，自定义 `RejectedExecutionHandler`，在 `rejectedExecution` 方法内打 metrics + 抛异常。

### 读写分离与主从延迟

**读写分离**：写主库，读从库，以牺牲**实时一致性**换取**读吞吐提升**。

**主从延迟处理**：

| 方案 | 适用 | 代价 |
|------|------|------|
| 强制读主库 | 强一致性业务（支付、下单后查询）| 主库压力增加 |
| 二次读取（从→主）| 弱一致业务容错 | 主库偶发压力 |
| sleep/等位点方案 | 教科书方案 | 延迟高，生产少用 |
| **读请求分类** | **推荐！** 历史数据走从库 | 业务改造 |

> 实际场景：事务上下文内的读必须强制走主库（同一事务 insert 后立即 select）。

### 布隆过滤器 & 布谷鸟过滤器

**布隆过滤器**：多个 hash 函数 → 位数组，空间效率极高，但：
- **假阳性**（说存在 → 可能不存在），无假阴性（说不存在 → 一定不存在）
- **不支持删除**（解决：计数布隆过滤器 3-4 位/位 or 定期重建）
- 误判率 = f(位数组大小, hash 函数数量)，典型设置：误判率 1% 对应每元素约 9.6 bit

**布谷鸟过滤器**（Cuckoo Filter）：
- 支持删除 ✅，误判率更低，但更复杂
- 原理：两张 hash 表 + 指纹存储，冲突时"踢出"对方（布谷鸟习性）
- 负载率高（>90%）时插入性能下降

**选型**：
- 不需要删除 → 布隆过滤器（简单）
- 需要删除 → 布谷鸟过滤器（或定期重建布隆过滤器）

### 分库分表

单库到达瓶颈（写入 TPS 高、表数据量亿级）时，通过拆分降低单节点压力：

| 方案 | 做法 | 解决问题 | 核心难点 |
|------|------|---------|---------|
| 垂直分库 | 按业务把不同表拆到不同库（用户库/订单库/商品库）| 业务隔离，减少跨业务竞争 | 跨库 JOIN |
| 水平分库 | 同一张表按 ID/用户 ID 分散到多个库 | 写入 TPS 分散 | 分布式事务、跨库聚合 |
| 水平分表 | 同一库内按 ID 哈希拆为多张表 | 单表数据量过大导致的索引膨胀 | 路由逻辑、历史数据迁移 |

**核心难点**：跨库 JOIN（改为应用层 join）、分布式事务（TCC 或最终一致）、全局唯一 ID（雪花算法/号段模式）。

### MQ 削峰与消息可靠性

**核心纪律**：
- **至少一次（At-Least-Once）**= 消息不丢，但可能重复 → 消费端必须**幂等**
- **最多一次（At-Most-Once）**= 消息可能丢，但不重复 → 可接受少量丢失的场景

**四大可靠性问题**：

| 问题 | 说明 | 解决方案 |
|------|------|---------|
| 消息丢失 | 生产者→Broker 或 Broker→消费者链路断 | 生产者 confirm 机制（RabbitMQ ack / Kafka ack=-1）；Broker 持久化；消费者 ack 后再删 |
| 消息重复 | 网络超时后生产者重试 | 消费端幂等（数据库唯一键/Redis SETNX 去重）|
| 消息顺序 | 分区/队列并发消费乱序 | 同业务数据路由到同一分区（Kafka partition by key）|
| 消费失败 | 消费逻辑抛异常 | 死信队列（DLQ）存放失败消息 + 定时重试 |

### 高性能网络（Reactor / NIO / 零拷贝）

**Reactor 模式**（Netty / Nginx / Redis 底层共同选择）：
```
单 Reactor 单线程   → 所有 IO + 业务处理在同一线程（Redis 6.0 前的 IO 模型）
单 Reactor 多线程   → IO 在 Reactor 线程，业务分给 Worker 线程池
主从 Reactor 多线程 → MainReactor 只接受连接，SubReactor 处理读写（Netty 默认）
```

**IO 多路复用进化**：

| 模型 | 原理 | 瓶颈 |
|------|------|------|
| select | FD 集合拷贝到内核，O(n) 遍历 | 最多 1024 FD |
| poll | 链表无上限，仍 O(n) 遍历 | 大量连接时性能差 |
| epoll | 红黑树注册 + 就绪队列，O(1) 唤醒 | Linux 专属，Nginx/Netty 首选 |

**零拷贝**（减少内核与用户态之间的数据拷贝次数）：

| 方式 | CPU 拷贝次数 | 典型场景 |
|------|------------|---------|
| 传统 read + write | 4 次 | 通用 |
| mmap + write | 3 次 | 内存映射文件 |
| sendfile | 0 次 CPU 拷贝 | Kafka 消费 / Nginx 静态文件下发 |

**Netty 高性能三件套**：
- `EventLoop`：单线程绑定多 Channel，避免线程切换开销
- `ChannelPipeline`：责任链处理 IO 事件，编解码与业务逻辑解耦
- `ByteBuf 内存池`：池化 DirectBuffer，减少 GC 压力

**Kafka 底层高性能**（顺序写 + 零拷贝 + 批处理三者叠加）：
```
Producer  → 批量写入 PageCache（顺序写，避免随机磁盘 IO）
Broker    → 数据持久化到 log segment（append-only 顺序文件）
Consumer  → sendfile 零拷贝直接从 PageCache 传到 Socket
批处理    → batch.size + linger.ms 聚合消息，减少网络往返次数
```

### 性能瓶颈根因分析

**分析顺序**（由浅入深）：
```
1. 队列积压  → 线程池打满/MQ 堆积，先看 activeCount、queueSize
2. 线程池    → coreSize 不够 or 任务阻塞（IO 密集未调优）
3. 数据库    → 慢 SQL、缺索引、连接池耗尽（show processlist）
4. 缓存      → 缓存命中率低、大 key、热 key 集中打一台 Redis
5. 下游服务  → RPC 超时、依赖方限流（看 RT 分布和超时率）
6. 网络      → 带宽打满、TCP 队列溢出、连接数不够
```

**三大根因**：

| 根因 | 特征 | 工具 |
|------|------|------|
| CPU 打满 | 计算密集 or 锁竞争 | `top -H`、`jstack` 找热点线程 |
| IO 慢 | DB/Redis/RPC 响应慢，线程大量等待 | 链路追踪（SkyWalking/Zipkin）、慢查询日志 |
| 内存不足 | GC 频繁、Full GC 时间长 | `jstat -gcutil`、`jmap`、GC 日志；改 G1/ZGC |

## 五、三高联动场景分析

### 场景：秒杀系统

| 维度 | 措施 | 代价 |
|------|------|------|
| 高并发 | 令牌桶限流、分桶库存 | 多桶管理复杂度 |
| 高可用 | 熔断降级（库存不足时降级展示售罄）| 少量用户无法下单 |
| 高性能 | Redis 预扣/Inventory Hint | 一致性风险或需内核支持 |

核心权衡：Redis 预扣（高性能）vs MySQL 可靠性（高可用）——见 [[系统设计-电商系统]] 中的大厂实践补充。

### 高并发系统典型架构

```
Client → CDN（静态资源缓存）
       → Nginx（动静分离 + 限流 + Gzip）
       → 网关（鉴权 + 路由 + 全局限流/熔断）
       → Service（无状态，水平扩容）
            ├── 本地缓存（Caffeine）
            ├── Redis（分布式缓存 + 分布式锁）
            ├── MQ（削峰填谷：RocketMQ/Kafka）
            ├── DB（分库分表 + 读写分离）
            └── Search（ES 全文检索）
```

**核心是多级缓存 + 异步化 + 限流降级 + 可水平扩展**。每一层都应有限流和降级能力，任意层的失败不应级联击穿下一层。

### 高并发架构模式

| 模式 | 核心思想 | 代表框架 / 场景 |
|------|---------|--------------|
| Actor Model | 每个 Actor 有独立邮箱和状态，消息驱动，天然线程隔离 | Akka、Erlang |
| Event-Driven（事件驱动）| 系统通过事件交互，生产者/消费者解耦，事件可持久化重放 | Kafka + 微服务 |
| Backpressure（背压）| 消费者向生产者反馈处理能力上限，防止无限堆积 OOM | Reactor `Flux.limitRate()`、Kafka `poll` 主动拉取 |
| CQRS | 写模型强一致，读模型异步投影（可独立扩展和缓存）| EventStore、Axon |
| Event Sourcing | 以事件日志为状态唯一来源，可重放调试、审计 | 金融账务链路 |
| Pipeline 架构 | 数据流经一系列独立处理节点，节点可独立扩展 | Netty ChannelPipeline、Flink 算子链 |

> **Backpressure 是高并发异步系统的核心约束**：不考虑背压的异步系统，流量洪峰时会无限堆积内存直到 OOM。Kafka Consumer 主动 `poll`（而非 Broker Push）是背压的天然实现；RxJava/Reactor 的 Flowable/Flux 提供编程式背压协议（Reactive Streams 规范）。

### 场景：读多写少（商品详情）

| 维度 | 措施 | 代价 |
|------|------|------|
| 高并发 | 多级缓存拦截大部分读请求 | 缓存一致性问题 |
| 高可用 | 缓存击穿/雪崩防护（布隆过滤器+互斥锁）| 首次加载延迟 |
| 高性能 | 读写分离（强制主库仅用于写后立即读）| 主库偶发压力 |

### 场景：高 TPS 写入（日志/流水）

| 维度 | 措施 | 代价 |
|------|------|------|
| 高并发 | 批量写入、MQ 削峰 | 实时性降低 |
| 高可用 | WAL + 异步持久化 | 崩溃时可能少量数据丢失 |
| 高性能 | LSM 树（写优化）| 读性能下降，需 Compaction |

## 六、架构经验沉淀

### 分布式系统架构

电商系统可按业务域拆成商品线、交易线、资金线、用户线、支付线、营销活动线等服务，使用 Dubbo 做 RPC 治理，Nacos 做注册中心和配置中心，Gateway 做统一入口：

| 领域 | 典型做法 |
|------|----------|
| 服务拆分 | 商品、交易、资金、用户、支付、营销活动按业务边界拆分，服务之间通过 Dubbo 调用 |
| Dubbo 治理 | 关闭启动时检查、服务分组、注册分组、配置分组、多版本、延迟暴露、滑动升级、降级、重试 |
| 注册与配置 | Nacos 作为注册中心和配置中心，支持热更新配置和新旧代码开关 |
| 网关入口 | Spring Cloud Gateway 统一路由断言、鉴权中心、白名单校验 |
| 大批量更新 | MongoDB 承载消息中心、广告中心等大量终端 ID 信息更新 |
| 可观测性 | ELK + Kafka 做日志统计，SkyWalking 做链路追踪，钉钉/短信做实时告警 |
| 分布式事务 | 交易中心支付链路用 TCC 保证支付、风控、优惠券一致；支付回调用可靠消息最终一致；支付反馈用最大努力通知 |
| 分布式锁 | Redisson + SpringBoot Starter 封装注解锁，解决跨实例并发互斥 |

面试表达重点：不是“用了哪些组件”，而是能说明每个组件解决的治理问题，例如 Dubbo 解决服务治理，Nacos 解决注册配置，Gateway 解决统一入口，TCC/可靠消息解决跨服务一致性。

### 高并发系统架构

| 场景 | 方案 |
|------|------|
| 长连接/WS 压力 | SLB + Redis + 一致性 Hash 将用户稳定路由到 Netty + SpringBoot Starter 封装的 WS 节点，分散并发连接 |
| Web 容器并发 | Tomcat 使用 `Http11Nio2Protocol` 非阻塞协议，启用压缩，优化最大连接数和线程池 |
| 数据库吞吐 | 用户中心、交易中心使用 MySQL + ShardingJDBC + SLB 做分库分表、读写分离 |
| 热点读 | 会员端小程序接口使用本地缓存 + Redis 多级缓存保存热点数据 |
| 异步削峰 | 个人中心、交易中心使用 RocketMQ 处理低耦合异步业务 |
| 缓存风险 | 热点数据预处理，双重检查 + 互斥锁降低缓存击穿/雪崩风险 |

### 高可用架构

| 层级 | 高可用手段 |
|------|------------|
| 应用层 | 各业务模块 K8s 多副本部署，依靠探针、自愈、滚动发布保证服务可用 |
| 流量治理 | 会员端小程序接口使用 Sentinel 做限流、熔断、降级和监控 |

| 监控告警 | SkyWalking 自动检测链路异常，核心接口异常和数据对比异常通过钉钉/短信告警 |

### 高性能架构

| 方向 | 实践 |
|------|------|
