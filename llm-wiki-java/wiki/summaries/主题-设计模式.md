## 一、设计七大原则

软件会变化——需求变化、团队扩张、技术升级。设计模式的核心价值不是"把代码写复杂"，而是**将变化隔离、固定不变**：

- **封装变化点**（OCP）：找到哪里会变，把变化封装在一个地方
- **面向接口编程**（DIP）：依赖抽象，不依赖具体实现
- **组合优于继承**（CARP）：继承是硬编码关系，组合更灵活

**SOLID（5大）**：

| 原则 | 缩写 | 核心思想 | 违反表现 |
|------|------|---------|---------|
| 单一职责 | SRP | 一个类只有一个变化原因 | 一个类既做 IO 又做业务逻辑 |
| 开放封闭 | OCP | 对扩展开放，对修改封闭 | 每次新增功能都改原有代码 |
| 里氏替换 | LSP | 子类必须能无缝替换父类 | 重写父类方法改变了语义 |
| 依赖倒置 | DIP | 高层依赖抽象，不依赖具体类 | Service 直接 new 具体的 DAO 实现 |
| 接口隔离 | ISP | 接口应小而专注，不应强迫实现不需要的方法 | 一个接口包含十几个不相关方法 |

**额外 2 条**：

- **迪米特法则（LoD）**：对象应对其他对象有最少的了解（只与直接朋友通信）
- **合成/聚合复用原则（CARP）**：优先用组合/聚合代替继承

## 二、创建型模式

### 单例模式（5种实现）

| 实现方式 | 线程安全 | 延迟加载 | 防反序列化破坏 | 推荐度 |
|---------|---------|---------|-------------|--------|
| 饿汉 | ✅（类加载时初始化）| ❌ | ❌ | 一般 |
| 懒汉（无同步）| ❌ | ✅ | ❌ | 不推荐 |
| 懒汉（synchronized）| ✅ | ✅ | ❌ | 性能差 |
| 双重检验锁（DCL）| ✅（需 volatile）| ✅ | ❌ | 常用 |
| 静态内部类 | ✅（类加载保证）| ✅ | ❌ | 推荐 |
| **枚举** | **✅** | **✅** | **✅** | **最佳** |

**DCL 为什么需要 volatile**：`instance = new Singleton()` 分三步（分配内存→初始化对象→指针指向内存），JIT/CPU 可能将步骤2和3重排序，导致另一个线程拿到未初始化完成的对象。volatile 禁止指令重排序。

**枚举为什么最好**：
1. JVM 类加载机制保证线程安全（`static` 字段初始化由 ClassLoader 同步）
2. Java 规范明确枚举反序列化不会创建新对象（特殊处理，不走 `readObject()`）
3. 反射无法创建枚举实例（`Constructor.newInstance()` 对枚举抛异常）

```java
public enum Singleton {
    INSTANCE;
    public void doSomething() { ... }
}
```

### 工厂模式（三种）

| 模式 | 核心思路 | 符合OCP | 复杂度 | 适用 |
|------|---------|---------|--------|------|
| 简单工厂 | 一个工厂创建所有产品（switch/if-else）| ❌（新增产品要改工厂）| 低 | 产品种类少且稳定 |
| 工厂方法 | 每种产品对应一个工厂类 | ✅ | 中（类成对增加）| 产品种类多，扩展频繁 |
| 抽象工厂 | 一个工厂创建一族相关产品 | 部分（新增产品族OK，新增产品等级需改接口）| 高 | 多产品族（如跨平台 UI 组件）|

**Spring 中的工厂模式**：`BeanFactory` 是工厂方法模式的体现；`FactoryBean` 允许自定义对象创建逻辑。

### 其他创建型模式

| 模式 | 核心思路 | Java/Spring 典型 |
|------|---------|----------------|
| 建造者 | 复杂对象分步构建，链式写法 | `StringBuilder`、Lombok `@Builder`、`UriComponentsBuilder` |
| 原型 | 通过克隆快速创建对象，减少初始化开销 | `Object.clone()`、Spring `@Scope("prototype")` |

## 三、结构型模式

### 代理模式

代理对象控制对真实对象的访问，在访问前后可以插入额外逻辑（鉴权/缓存/日志）。

**应用场景**：
- **远程代理**：Dubbo、Feign 客户端代理（本地调用透明化为远程调用）
- **动态代理**：Spring AOP（JDK Proxy + CGLIB），在不修改原代码的情况下添加切面
- **缓存代理**：查询前先查缓存，缓存未命中再查真实数据源
- **保护代理**：权限控制，代理验证调用方权限后再转发

（详见 [[机制-动态代理]] 的 JDK/CGLIB 实现细节）

### 适配器模式（Adapter）

**让原本不兼容的接口协作**，分类适配器（继承）和对象适配器（组合，更常用）。

**Java/Spring 中的应用**：
- `HandlerAdapter`：SpringMVC 统一调用不同类型 Controller（`@Controller`/`HttpRequestHandler`/`Servlet`）
- `AdvisorAdapter`：统一适配各类 Advisor 为 `MethodInterceptor`
- `InputStreamReader`：字节流适配为字符流

### 装饰器模式（Decorator）

**包装后增强原对象功能，不修改原类**，与代理区别在于：装饰器由调用方主动组合，代理由框架透明注入。

**Java 中的应用**：
- `BufferedInputStream(new FileInputStream(...))` — IO 流层层包装
- `HttpServletRequestWrapper` — 增强 Request 功能（如修改参数）
- Spring 的 `BeanDefinitionDecorator`

### 其他结构型模式

| 模式 | 核心思路 | 典型场景 |
|------|---------|---------|
| 桥接 | 分离两个独立变化的维度，各自扩展 | 消息类型（短信/邮件）× 发送方式（立即/延迟）|
| 组合 | 树形结构中叶节点与容器节点统一接口 | 文件目录树、组织架构树、菜单树 |
| 外观 | 对外暴露简化的统一接口，隐藏子系统复杂度 | `Facade` 层、SDK 封装、Tomcat `RequestFacade` |

---

### 享元模式（Flyweight）

**共享大量细粒度对象以节省内存**，将对象状态分为内部状态（共享）和外部状态（不共享）。

**Java 中的应用**：
- `String` 常量池：相同字面量的字符串共享同一对象
- `Integer.valueOf()` 缓存 -128~127：`Integer.valueOf(100) == Integer.valueOf(100)` 为 true
- 线程池中的线程复用（本质是享元）

## 四、行为型模式

### 观察者模式（Observer）

**一对多依赖**：当主题（Subject）状态变化时，自动通知所有观察者（Observer）。

```
Subject（被观察者）          Observer（观察者）
  - observers: List<Observer>   + update()
  + addObserver()
  + notifyObservers()
  + setState()  → 调用 notifyObservers()
```

**Java/Spring 中的应用**：
- Java 内置 `java.util.Observer`（已废弃）
- Spring `ApplicationEvent` + `@EventListener`（Spring 事件机制）
- GUI 的 `ActionListener`
- MQ 消息队列（生产者发布 → 消费者订阅，是观察者的分布式变体）

### 模板方法模式（Template Method）

**父类定义算法骨架，子类实现可变步骤**，是复用公共流程的标准做法。

```java
abstract class AbstractService {
    public final void process() {    // 模板方法（final，不允许子类重写）
        validate();
        doProcess();                 // 留给子类实现
        log();
    }
    protected abstract void doProcess();
    private void validate() { ... }
    private void log() { ... }
}
```

**Spring 中的应用**：
- `JdbcTemplate`、`RedisTemplate`、`RestTemplate`（Spring Template系列）
- `AbstractBeanFactory.getBean()`（Bean获取的模板流程）
- AOP 的 `MethodInterceptor`

### 策略模式（Strategy）

**将算法族封装成独立的策略类，运行时可替换**，消灭 if-else 分支。

```java
// 策略接口
interface OrderStrategy { void process(Order order); }

// 工厂 + 策略 组合（Spring 中常见写法）
@Service
class OrderStrategyFactory {
    @Autowired private Map<String, OrderStrategy> strategies;  // Spring 自动注入所有实现
    public OrderStrategy get(String type) { return strategies.get(type); }
}
```

**与 if-else 相比**：可读性更好；新增策略不修改工厂代码（OCP）；策略可测试、可复用；结合模板方法提取公共逻辑。

### 责任链模式（Chain of Responsibility）

**将请求沿着处理器链传递，直到有处理器消费它**，解耦请求发送者与多个接收者。

```
Request → Handler1 → Handler2 → Handler3 → ...
           │          │          │
        处理/跳过  处理/跳过  处理/跳过
```

**Java/Spring 中的应用**：
- Spring `FilterChain`（Servlet 过滤器链）
- Spring Security `SecurityFilterChain`（认证授权过滤链）
- MyBatis `Plugin` 拦截器链
- Netty `ChannelPipeline`（ChannelHandler 链）
- 工作流/审批流（订单审批多级审核）

### 其他行为型模式

| 模式 | 核心思路 | 典型场景 |
|------|---------|---------|
| 命令 | 把请求封装成对象，支持撤销/重做/队列 | 操作日志回放、事务回滚、任务队列 |
| 迭代器 | 统一遍历方式，屏蔽集合内部结构 | Java `Iterator`、`for-each` 语法糖 |
| 中介者 | 多对象通过中介统一协调，避免直接引用 | `DispatcherServlet` 调度多组件、聊天室服务端 |
| 备忘录 | 保存对象快照，支持恢复 | 编辑器 undo、游戏存档 |
| 状态 | 对象内部状态切换时，行为随状态变 | 订单状态机（待支付→已支付→已发货）|
| 访问者 | 操作多变、结构稳定时，将操作抽离 | AST 编译器、XML/JSON 解析器 |
| 解释器 | 定义语法规则，递归解释表达式 | 小型 DSL、规则引擎、Spring EL |

## 五、模式组合使用（实战）

**策略 + 工厂 + 模板方法**（Spring 中最常见的组合）：

```
1. 策略接口定义算法边界（doProcess()）
2. 抽象基类（模板方法）提取公共逻辑（validate/log/notify）
3. 工厂（Spring Map 注入）根据类型选策略
4. 调用方只与工厂和接口交互，无 if-else
```

## 六、选型原则与适用边界

**选择设计模式的判断依据**：

| 需求 | 模式 |
|------|------|
| 对象创建复杂或需要统一管理 | 工厂/单例 |
| 不修改原代码的前提下扩展行为 | 代理/装饰器 |
| 多种算法可互换 | 策略 |
| 有共同流程但步骤可定制 | 模板方法 |
| 一个状态变化影响多个对象 | 观察者 |
| 请求需要多个对象依次处理 | 责任链 |
| 大量重复细粒度对象 | 享元 |

**过度设计的风险**：
1. 简单的 if-else（3个分支）不需要策略模式；只有一个实现的接口不需要工厂；引入设计模式的前提是**变化真实存在且预期增加**
2. **单例的并发陷阱**：懒汉无同步版不是线程安全的；DCL 不加 volatile 在 JDK 5 之前可能失败；优先选枚举或静态内部类
3. **观察者的解耦 vs 可追踪性**：观察者模式增加了代码解耦性，但调试困难（通知链难以追踪）；Spring 的同步事件 vs 异步事件（`@Async`）是需要显式选择的
4. **责任链的短路问题**：责任链中每个节点需要决定是否继续传递；设计时要明确是"第一个能处理的处理"（短路）还是"所有节点都处理"（管道）

**不需要设计模式的情况**：业务逻辑简单稳定（扩展点不存在）、只有1~2个实现（抽象过早）、团队对该模式不熟悉（引入理解成本大于维护成本）。

## 七、Spring 设计模式全景

| 模式 | Spring 中的体现 |
|------|----------------|
| 工厂方法 | `BeanFactory` / `ApplicationContext` / `FactoryBean` |
| 单例 | Bean 默认 Scope，容器维护唯一实例 |
| 代理 | AOP 核心（JDK 动态代理 + CGLIB），`@Transactional` / `@Async` |
| 装饰器 | `HttpServletRequestWrapper`、`BeanDefinitionDecorator` |
| 适配器 | `HandlerAdapter`（适配不同 Controller 类型）、`AdvisorAdapter` |
| 策略 | `XXXStrategy` / `XXXResolver` 系列（如 `HandlerExceptionResolver`）|
| 观察者 | `ApplicationEvent` + `@EventListener` / `ApplicationListener` |
| 模板方法 | `JdbcTemplate`、`RedisTemplate`、`AbstractBeanFactory.refresh()` |
| 责任链 | `BeanPostProcessor` 链、`FilterChain`、`SecurityFilterChain` |
| 中介者 | `DispatcherServlet` — 统一调度 HandlerMapping / HandlerAdapter / ViewResolver |

## 八、设计模式在动态代理与 Spring 体系的落地

- **[[机制-动态代理]]**：代理模式在 Java 中的最重要实现；JDK 动态代理实现接口代理，CGLIB 实现类代理；Spring AOP 选择哪种代理取决于目标类是否实现接口
- **[[机制-Spring]]**：IoC 是工厂模式的高级实现（BeanFactory）；AOP 是代理模式 + 装饰器模式的组合；`@Transactional` 把代理模式和模板方法式事务流程结合起来；Spring 事件机制是观察者模式的工程化封装
