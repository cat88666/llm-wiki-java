## 一、4C8G 性能指标基准

| 指标 | 正常范围 | 需关注 | 不可接受 |
|------|---------|--------|---------|
| **CPU 利用率（单核）** | < 70% | 70%~90% | ≥ 100% |
| **系统 Load** | < 2（4核机器理论上限=4）| > 3 | > 4 |
| **磁盘利用率** | < 80% | ≥ 80% | ≥ 100% |
| **内存利用率** | < 80% | ≥ 80% | > 100% |
| **JVM 堆内存占用** | < 80%（4~6G 之间）| ≥ 80% | OOM 风险 |
| **YoungGC 次数** | < 1次/分钟 | > 1次/分钟 | 10次/分钟以上 |
| **YoungGC 时长** | < 50ms | > 200ms | ≥ 1s |
| **FullGC 次数** | < 1次/周 | 1次/天 | 1次/小时 |
| **FullGC 时长** | < 1s | > 2s | ≥ 5s |

> **Load 含义**：单位时间内（1分钟/5分钟/15分钟）处于可运行状态或不可中断等待状态的平均进程数。4核机器 Load=4 表示满负荷，超过则有进程排队。

## 二、机器数量预估：公式 + QPS 估算

### 基础公式

```
单线程吞吐 = 1000ms ÷ RT(ms)
单机吞吐   = 单线程吞吐 × 线程数（Tomcat 默认 200）
理论机器数 = 目标QPS ÷ 单机吞吐
实际机器数 = 理论机器数 × 2~3（留冗余、考虑非理想因素）
```

### 例题：3000 QPS，RT=200ms

```
单线程：1000ms ÷ 200ms = 5 req/s
单机（200线程）：5 × 200 = 1000 QPS（理论）
理论机器数：3000 ÷ 1000 = 3台
实际预估：3 × 2~3 = 6~9台（取 6 台加上弹性扩容策略）
```

**注意**：以上公式是 CPU 密集型简化模型。真实场景需压测，CPU/内存/连接数都会制约吞吐量上限。

### 如何预估系统 QPS

```
1. 峰值流量估算
   DAU（日活）× 平均请求数/天 × 峰值系数（通常取 2~5×）
   例：DAU=100万，人均5次/天，峰值系数=3
   → 100万 × 5 ÷ 86400 × 3 ≈ 174 QPS

2. 接口拆分
   登录接口 QPS ≠ 核心业务 QPS，按接口比例估算

3. 压测验证
   上线前单机压测，得到单机最大 QPS，机器数 = 峰值QPS ÷ 单机QPS × 1.5
```

## 三、4C8G×16 vs 8C16G×8 选型

**总核心数相同（64C），总内存相同（128G），如何选？**

| 维度 | 4C8G × 16 | 8C16G × 8 |
|------|-----------|-----------|
| **高并发服务** | 更多实例 = 更好的负载均衡 | 单实例并发处理能力更强 |
| **高可用** | 挂一台影响 1/16 | 挂一台影响 1/8 |
| **CPU 密集** | 每实例 CPU 少，线程切换开销低 | 每实例 CPU 多，更适合多线程运算 |
| **内存密集（大 JVM 堆）** | 8G/台，堆只能配 6G | 16G/台，堆可配 12G，适合大对象应用 |
| **运维成本** | 实例多，运维复杂 | 实例少，运维简单 |

**结论**：
- Web 应用 / 微服务：选 **4C8G×16**（更高容错，天然水平扩展）
- 数据处理 / 大内存需求：选 **8C16G×8**（JVM 堆可更大，GC压力更小）

## 四、压测 600 线上 300 扛不住：排查清单

| 原因 | 典型表现 | 排查方向 |
|------|---------|---------|
| **数据倾斜** | 特定请求特别慢，平均 RT 抬升 | 热点数据（大V/热门商品）; 索引失效于真实数据分布 |
| **定时任务干扰** | 特定时间点后性能下降 | 检查 cron 任务是否在业务高峰期运行 |
| **未全链路压测** | 下游服务（支付/风控）成为瓶颈 | 压测时是否只压了入口，未压下游 |
| **缓存未预热** | 刚发布时 RT 高，之后恢复 | 检查缓存命中率；上线前预热 |
| **JIT 未预热** | 启动后前几分钟性能差 | `-XX:+TieredCompilation`；灰度发布给JIT时间 |
| **环境差异** | 压测机器配置 > 生产 | 核对 CPU/内存/JVM参数/连接池配置是否一致 |
| **共享中间件竞争** | 生产 Redis/DB 被其他服务共享 | 中间件资源隔离；监控 Redis 连接数/DB 连接数 |
| **监控 Agent 开销** | SkyWalking/ELK 在生产额外运行 | 压测时是否也启用了 Agent |

## 五、启动期 CPU/Load/RT 飙高：三大根因

**CPU/Load/RT 在启动后短暂飙高，之后恢复正常**：

```
根因1：JIT 预热
  JVM 解释执行 → JIT 编译热点代码（大量 CPU 占用）→ 编译完成后性能提升
  解法：灰度流量（先给少量流量），等 JIT 稳定后再全量

根因2：缓存未预热
  本地缓存/Redis 为空 → 大量 DB 查询 → DB 压力骤增
  解法：启动后执行 cache warm-up（预加载热点数据）

根因3：MQ 消息积压
  MQ 消费者启动后追赶历史积压消息 → 资源占用
  解法：优雅上线（先限速消费积压，再开放业务流量）
```

## 六、堆外内存泄漏：进程 RES 持续增长排查

**表现**：`jmap -heap` 显示堆使用正常，`top` 显示进程 RES 持续增长。

**根因**：堆外内存泄漏。

```
可能来源：
1. Netty DirectBuffer：-XX:MaxDirectMemorySize 未设置上限
2. JNI 调用的 native 内存（如 OpenSSL、SIGAR）
3. Metaspace 增长（类加载不断增多，如动态代理、Groovy脚本）

排查步骤（NMT 追踪）：
1. 启动加 -XX:NativeMemoryTracking=detail
2. jcmd <pid> VM.native_memory summary     → 定位增长区域
3. 若 Internal 段持续增长 → 可能是 JNI/Unsafe
4. 若 Class 段增长 → 类加载泄漏，检查动态生成类的地方
5. jcmd <pid> VM.native_memory summary.diff → 前后对比增量
```

## 七、频繁 Full GC 的五种根因

**有内存却频繁 FGC 的几种情况**：

```
根因1：代码显式调用 System.gc()（GC日志会显示 System.gc()）
  → 排查代码；或 -XX:+DisableExplicitGC 禁用

根因2：内存碎片（CMS收集器特有）
  老年代空间够，但碎片化严重，无法分配大对象
  → 换 G1/ZGC；或定期做 CMS Full GC 整理

根因3：堆外内存不足，触发 GC 尝试释放
  DirectByteBuffer 清理依赖 GC 触发
  → 设置 -XX:MaxDirectMemorySize；手动触发 System.gc() 或用 -XX:+ExplicitGCInvokesConcurrent

根因4：大对象直接进老年代
  超过 -XX:PretenureSizeThreshold 的对象不走 Eden
  老年代不断被大对象填满
  → 评估大对象分配，减少大对象；升级 G1（G1 大对象走 Humongous Region）

根因5：老年代实际不够大
  堆配了 4G，但老年代比例设置不当（默认老年代=堆×2/3）
  → 检查 -XX:NewRatio；或直接用 G1 动态调整
```

## 八、低延迟金融场景：GC 收集器选型

**金融场景要求**：低延迟（RT < 10ms），停顿时间敏感，吞吐量其次。

```
JDK ≤ 8：CMS（-XX:+UseConcMarkSweepGC）
  → STW 时间较短，但有内存碎片、Concurrent Mode Failure 风险

JDK 9~14：G1（-XX:+UseG1GC）
  → 默认收集器，-XX:MaxGCPauseMillis=100 控制停顿上限
  → 大堆（>6G）效果更好，适合金融中等规模应用

JDK 15+：ZGC（-XX:+UseZGC）
  → STW < 1ms（几乎不感知），最适合低延迟金融系统
  → 吞吐量略低于 G1（约 10%~15%）
  → 银行核心系统推荐选项
```

## 九、日志瓶颈与多线程读文件优化

### 日志成为瓶颈

**根因**：同步日志 Logger 在高并发下，磁盘 IO 阻塞主线程。

```
1. 异步日志（首选）
   Logback: AsyncAppender 包装 RollingFileAppender
   Log4j2: AsyncLogger（LMAX Disruptor 无锁队列，性能最好）

2. 降低日志级别
   生产环境只保留 WARN/ERROR，关掉 DEBUG/INFO

3. 减少日志量
   - 循环内不打日志
   - 频繁调用的路径抽样日志（每 N 次打一次）
   - 使用结构化日志 + ELK 收集，避免本地大量写盘

4. 日志格式优化
   避免在日志中做复杂计算/toString（即使 level 不输出也会执行）
   → 用 lambda: log.debug("value={}", () -> expensiveCalc());
```

### 多线程读文件：IO vs CPU

**1个线程 vs 10个线程读1000文件**：

```
IO 密集型（普通机械盘/网络文件）：
  多线程好。线程等待 IO 时 CPU 可切换到其他线程继续工作
  推荐：线程数 = CPU核数 × (1 + IO等待时间/CPU时间)

CPU 密集型（SSD读取 + 复杂解析）：
  线程数 > CPU核数反而更慢（线程切换开销）
  推荐：线程数 ≈ CPU核数

实际建议：
  - 先测量：htop 看 CPU 使用率，CPU < 50% → IO瓶颈 → 加线程
  - 1000个小文件：线程池 10~20 个线程并发读取
  - 考虑 NIO（Files.readAllBytes 的 NIO 版本）减少系统调用开销
```
