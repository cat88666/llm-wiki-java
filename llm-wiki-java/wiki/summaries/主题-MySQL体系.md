# MySQL 体系（L5 存储层）

## ACID 实现速查

| 特性 | 实现机制 |
|------|---------|
| **原子性（A）** | undo log 回滚所有修改 |
| **一致性（C）** | A+I+D 共同保障 + 数据库约束 |
| **隔离性（I）** | MVCC（快照读）+ 锁（当前读）|
| **持久性（D）** | redo log + 两阶段提交 |

---

## 事务隔离级别速查

| 隔离级别 | 实现 | 解决 |
|---------|------|------|
| Read Uncommitted | 直接读最新 | 无 |
| Read Committed | 每次 SELECT 生成新 ReadView | 脏读 |
| Repeatable Read（默认）| 首次生成 ReadView 复用 + Next-Key Lock | 脏读 + 不可重复读 + 幻读（大部分）|
| Serializable | 全加锁串行 | 全解决 |

## 高频考点汇总

### 索引模型

- **B+ 树选型**：叶子节点双向链表（范围查询）+ 非叶子节点不存数据（树高低）+ IO 预读友好
- **聚簇索引 vs 二级索引**：聚簇索引叶子存整行，二级索引叶子存主键；查二级索引需要回表
- **覆盖索引**：查询字段全在索引中，直接返回，EXPLAIN Extra 显示 `Using index`
- **索引下推**（ICP，5.6+）：存储引擎层先过滤再回表，EXPLAIN Extra 显示 `Using index condition`
- **最左前缀**：联合索引按最左列排序，跳过最左列就不能走该索引
- **主键推荐自增整数**：UUID 随机写导致页频繁分裂，写性能差

### MVCC

- **三要素**：undo log 版本链 + 行记录隐式字段（db_trx_id / db_roll_ptr）+ ReadView
- **ReadView 4个字段**：trx_ids（活跃事务列表）/ up_limit_id（低水位）/ low_limit_id（高水位）/ creator_trx_id
- **可见性规则**：trx_id < up_limit_id → 可见；trx_id >= low_limit_id → 不可见；中间 → 查 trx_ids
- **RC vs RR 差异**：RC 每次 SELECT 新建 ReadView；RR 事务首次 SELECT 建好之后复用

### 三种日志

- **undo log**：记录修改前旧值，用于回滚（原子性）和 MVCC 版本链
- **redo log**：记录物理修改（"第X页第Y字节改为Z"），用于崩溃恢复（持久性），循环写
- **binlog**：逻辑日志，记录所有 DDL/DML，用于主从复制和备份，Server 层，所有引擎
- **两阶段提交**：prepare（redo log prepare + XID）→ write binlog → commit（redo log commit），保证主备一致

### 锁机制

- **S 锁 / X 锁**：共享读锁（S 间兼容，S-X 冲突）/ 排他写锁（X 与一切冲突）
- **意向锁（IS/IX）**：表级锁，加行锁前自动获取，让表锁申请 O(1) 判断冲突
- **Record Lock**：锁索引记录本身；**Gap Lock**：锁索引间隙（防 INSERT）；**Next-Key Lock**：Record + Gap（左开右闭，RR 加锁基本单位）
- **行锁锁的是索引**：无索引退化为全表锁（扫描全聚簇索引）
- **加锁两个优化**：唯一索引等值命中 → Record Lock；等值查询不满足最右值 → Gap Lock

### SQL 优化

- **EXPLAIN 重点**：type（ALL = 全表扫描必须优化）、key（走的索引）、rows（预估扫描行）、Extra
- **索引失效场景**：索引列用函数 / 类型隐式转换 / LIKE 前导% / OR 一侧无索引 / 违反最左前缀
- **深分页**：LIMIT 10000000,10 扫描 1000 万行；用游标分页（`WHERE id > last_id`）或子查询定位主键
- **慢 SQL 调优步骤**：发现（慢查询日志）→ EXPLAIN → 定位（索引 / 数据量 / 锁）→ 优化

## 层间依赖

- L5 MySQL 依赖 **L4 数据结构**：InnoDB 索引是 B+ 树的工程实现，B+ 树特性决定了索引设计规则
- L5 MySQL 依赖 **L3 并发**：InnoDB 锁机制中的乐观锁 / 悲观锁思想与 L3 CAS / synchronized 一脉相承；MVCC 是数据库层的乐观并发控制
- L5 MySQL 被 **L6 框架**使用：MyBatis / Spring 事务管理器建立在 ACID 和隔离级别之上
- L5 MySQL 被 **L7 分布式**使用：分库分表（Sharding）、分布式事务（2PC / Saga）都以单机 MySQL 为基础
