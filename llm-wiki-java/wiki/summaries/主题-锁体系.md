# 锁体系 — Java / MySQL / 分布式全景

> 锁是多个执行单元（线程/进程/节点）竞争共享资源时的协调机制。本页从**分类维度 → Java锁 → MySQL锁 → 分布式锁 → 选型决策树**逐层展开。

---

## 一、锁的分类维度全景

```
按竞争策略
  ├── 乐观锁：不加锁，提交时校验版本/CAS
  └── 悲观锁：先加锁，占有后操作

按公平性
  ├── 公平锁：FIFO 队列，先等先得
  └── 非公平锁：允许插队，吞吐更高

按可重入性
  ├── 可重入锁：同线程可多次获取（计数器）
  └── 不可重入锁：同线程再次获取会自己死锁

按读写性
  ├── 排他锁（写锁）：X Lock，独占
  ├── 共享锁（读锁）：S Lock，并发读
  └── 乐观读锁（StampedLock）：无锁读，冲突时升级

按粒度
  ├── 全局锁：锁整个实例
  ├── 表级锁：锁单张表
  └── 行级锁：锁单行/间隙/记录

按实现层次
  ├── JVM 层：synchronized / ReentrantLock（线程间）
  ├── OS 层：Mutex（操作系统原语）
  └── 分布式层：Redis锁 / ZK锁 / DB锁（进程/节点间）

按自旋方式
  ├── 自旋锁：CAS 忙等，不切换内核（短临界区）
  └── 阻塞锁：park 线程，OS 内核调度（长临界区）
```

## 二、Java 锁体系

### 2.1 锁分类全表

| 锁 | 实现 | 可重入 | 公平 | 读写分离 | 适用场景 |
|----|------|--------|------|---------|---------|
| `synchronized` | JVM ObjectMonitor | ✅ | ❌（非公平） | ❌ | 简单互斥，JVM 持续优化 |
| `ReentrantLock` | AQS | ✅ | 可选 | ❌ | 需中断/超时/多条件 |
| `ReentrantReadWriteLock` | AQS | ✅ | 可选 | ✅ | 读多写少 |
| `StampedLock` | 无 AQS，自定义 | ❌ | ❌ | ✅ + 乐观读 | 读极多、延迟敏感 |
| `Semaphore` | AQS 共享模式 | — | 可选 | — | 限流/资源池并发数控制 |
| `CountDownLatch` | AQS 共享 | — | — | — | 等待 N 个任务完成（一次性） |
| `CyclicBarrier` | ReentrantLock | — | — | — | 多线程到达同一屏障点后一起走（可复用） |
| `LockSupport` | Unsafe.park/unpark | — | — | — | AQS 的底层线程挂起工具 |
| `AtomicXxx` / CAS | Unsafe.compareAndSwap | — | — | — | 无锁乐观更新，低冲突场景 |

### 2.2 synchronized 锁升级链

```
无锁
  │  首次进入 synchronized（JDK 15 前）
  ▼
偏向锁（Biased Lock）
  │  只记录 ThreadID，同一线程零 CAS 开销
  │  另一线程竞争 → 撤销偏向（需 STW）
  ▼
轻量级锁（Lightweight Lock）
  │  Mark Word 复制到栈帧 Lock Record，CAS 竞争
  │  多次 CAS 失败（默认自旋 10 次）→ 升级
  ▼
重量级锁（Heavyweight Lock）
  │  Mark Word 指向 ObjectMonitor
  │  竞争者进入 EntryList → OS 挂起（用户态→内核态）
```

**关键**：锁只升不降；JDK 15 默认废弃偏向锁（撤销 STW 代价太高）；JDK 21 虚拟线程下重量级锁代价降低（park 的是虚拟线程，不切换 OS 线程）。

详见 → [[机制-Synchronized]]

### 2.3 ReentrantLock / AQS

```
AQS 骨架
  ├── volatile int state（CAS 修改）
  └── CLH 双向 FIFO 同步队列
        ├── 公平锁：先检查队列有无前驱，有则入队
        └── 非公平锁：直接 CAS 抢 state，失败才入队

ReentrantLock 重入原理：state 累加（state=2 表示重入 2 次），unlock() 逐次减，归零释放

Condition 条件队列：await() → 释放锁 → 进条件队列；signal() → 移出条件队列 → 进同步队列
```

详见 → [[机制-AQS]]

### 2.4 ReentrantReadWriteLock

```
state（32位）拆为两半
  ├── 高 16 位：读锁持有数（共享）
  └── 低 16 位：写锁持有数（独占）

规则
  ├── 读-读：兼容，并发读
  ├── 读-写：互斥，写等读完
  ├── 写-读：互斥，读等写完
  └── 写-写：互斥

锁降级：持有写锁时先获取读锁，再释放写锁（允许）
锁升级：持有读锁时直接获取写锁 → 死锁（不允许，多个读锁互相等待写锁）
```

**适用**：读多写少（缓存、配置热更新）；写操作频繁时读写锁开销反而大于 ReentrantLock。

### 2.5 StampedLock（JDK 8+）

```
三种模式
  ├── 乐观读（tryOptimisticRead）：不加任何锁，返回邮戳 stamp
  │     validate(stamp) 检查是否被写锁打断，失败则升级为读锁
  ├── 悲观读（readLock）：阻塞写，类似 ReadWriteLock
  └── 写锁（writeLock）：独占

代码模板
  long stamp = lock.tryOptimisticRead();
  // 读数据
  if (!lock.validate(stamp)) {
      stamp = lock.readLock();
      try { /* 重读 */ } finally { lock.unlockRead(stamp); }
  }
```

**优势**：乐观读完全无锁，适合读极多的 CPU 密集场景（如坐标计算）。
**劣势**：不支持重入；不支持 Condition；API 复杂，容易写错。

### 2.6 CAS —— 乐观锁的硬件基础

```
compare-and-swap
  期望值 expected == 内存当前值 → 写入新值，返回 true
  不等 → 不写，返回 false（调用者自旋重试）

CPU 指令：LOCK CMPXCHG（x86）
ABA 问题：A→B→A 时 CAS 看到的还是 A → 用 AtomicStampedReference 加版本号解决
```

详见 → [[机制-CAS]]

## 三、MySQL 锁体系

### 3.1 粒度层次全图

```
全局锁（FLUSH TABLES WITH READ LOCK）
  └── 锁整个数据库实例，全库只读
        用途：逻辑备份（mysqldump 非 InnoDB 场景）

表级锁
  ├── 表锁（LOCK TABLES t READ/WRITE）
  │     读锁：其他会话可读，不可写
  │     写锁：独占，其他会话不可读写
  │     MyISAM 的唯一锁；InnoDB 尽量不用
  │
  ├── MDL 锁（元数据锁，隐式）
  │     DML/DQL → 自动加 MDL 读锁（共享）
  │     DDL（ALTER TABLE）→ 自动加 MDL 写锁（独占）
  │     长事务持有 MDL 读锁 + DDL 等待 → 引发锁链阻塞
  │
  ├── 意向锁 IS / IX（表级，自动）
  │     加行 S 锁前先加 IS；加行 X 锁前先加 IX
  │     让表锁申请 O(1) 判断冲突，无需逐行扫描
  │
  └── AUTO-INC 锁
        自增主键 INSERT 时临时持有，语句结束即释放
        innodb_autoinc_lock_mode=2（高并发推荐）改为轻量 mutex

行级锁（InnoDB 独有，锁的是索引记录）
  ├── Record Lock（记录锁）：锁单条索引记录
  ├── Gap Lock（间隙锁）：锁索引记录之间的间隙，防止幻读插入
  │     只在 RR 隔离级别生效；RC 下无 Gap Lock
  └── Next-Key Lock（临键锁）= Record Lock + Gap Lock
        RR 下的加锁基本单位：(prev, current]
        优化：唯一索引等值命中 → 退化为 Record Lock
              等值查询向右遍历不满足 → 退化为 Gap Lock
```

### 3.2 加锁方式：悲观锁 vs 乐观锁

**悲观锁（当前读）**

```sql
-- X 锁（写锁），阻止其他读写
SELECT * FROM orders WHERE id=1 FOR UPDATE;

-- S 锁（读锁），允许并发读，阻止写
SELECT * FROM orders WHERE id=1 LOCK IN SHARE MODE;

-- DML 自动加 X 锁
UPDATE orders SET status=2 WHERE id=1;
```

**乐观锁（版本号机制）**

```sql
-- 表加 version 字段
UPDATE orders
SET status=2, version=version+1
WHERE id=1 AND version=#{oldVersion};
-- 返回 affected=0 → 版本冲突，重试或返回失败
```

乐观锁适合：读多写少、冲突概率低；冲突频繁时重试风暴反而更慢。

### 3.3 S/X 锁兼容矩阵

| | S（读锁） | X（写锁） |
|--|--------|--------|
| **S** | ✅ 兼容 | ❌ 不兼容 |
| **X** | ❌ 不兼容 | ❌ 不兼容 |

### 3.4 死锁与排查

```
典型场景：事务A锁id=1等id=2；事务B锁id=2等id=1

InnoDB 死锁检测（等待图）：自动选 rollback_segment 小的事务回滚（victim）
set innodb_deadlock_detect=ON（默认开）

排查命令
  SHOW ENGINE INNODB STATUS;          -- 查看最近死锁信息
  SELECT * FROM information_schema.innodb_trx;        -- 活跃事务
  SELECT * FROM information_schema.innodb_lock_waits; -- 锁等待链
```

详见 → [[概念-MySQL]]、[[概念-MySQL]]

## 四、分布式锁体系

### 4.1 三种实现对比

| 维度 | Redis 分布式锁 | ZooKeeper 分布式锁 | 数据库分布式锁 |
|------|---------------|-------------------|--------------|
| **原理** | SETNX+TTL 原子占坑 | 临时顺序节点+Watch | 唯一键 INSERT 或 SELECT FOR UPDATE |
| **CAP** | AP（主从切换丢锁风险） | CP（ZAB强一致） | 视DB配置 |
| **性能** | 最高（内存，<1ms） | 中等（ZAB协议涉及网络RTT） | 最低（磁盘IO） |
| **可重入** | Redisson Hash 计数 | 同节点路径检查 | 需额外字段 |
| **自动释放** | TTL 到期 | 临时节点（会话断开自动删） | 需 TTL 列或手动 |
| **工具** | Redisson | Curator | — |
| **适用** | 高并发业务锁 | 强一致要求场景 | 低频、降级兜底 |

### 4.2 Redis 分布式锁

**基础实现**

```lua
-- 加锁：原子，NX 保证互斥，EX 保证 TTL
SET lock:order:123 {uuid} EX 10 NX

-- 解锁：Lua 原子：判断 + 删除（防误删他人锁）
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
end
```

**Redisson 工程级实现**

```
SET 时 TTL = 30s
  └── watchdog 每 10s（TTL/3）续期 → 线程活着锁不过期
  └── 线程死亡 → watchdog 停 → TTL 倒计时 → 自然过期

可重入：用 Hash 存 {uuid:threadId} → value 为重入次数（HINCRBY）

API
  RLock lock = redisson.getLock("order:lock:123");
  lock.lock();       // 自动续期
  lock.unlock();     // 重入次数-1，归零删 key
```

详见 → [[概念-Redis]]

### 4.3 ZooKeeper 分布式锁

```
争锁流程
  1. 在 /locks/order_lock/ 下创建临时顺序节点，如 /locks/order_lock/seq-0000000003
  2. 获取 /locks/order_lock/ 所有子节点，排序
  3. 序号最小的节点 = 持有锁
  4. 非最小节点监听（Watch）前一个节点
  5. 前一个节点删除时，Watch 触发，重新判断自己是否最小

自动释放
  临时节点：ZK 会话断开（JVM 崩溃）→ 临时节点自动删除 → 下一个节点 Watch 触发
```

**优势**：无需 TTL，不依赖时钟，真正的 CP 强一致（ZAB 协议保证）。
**劣势**：每次加锁解锁涉及网络 RTT + ZAB 同步，TPS 上限约数千；临时节点过多有内存压力。

详见 → [[机制-Zookeeper]]

### 4.4 数据库分布式锁

```sql
-- 方案A：唯一键 INSERT
INSERT INTO dist_lock(lock_name, owner, expire_time)
VALUES('order:123', 'uuid', NOW() + INTERVAL 30 SECOND);
-- 成功 → 持锁；主键冲突 → 锁被占

-- 方案B：SELECT FOR UPDATE（行级悲观锁）
SELECT 1 FROM dist_lock WHERE lock_name='order:123' FOR UPDATE;

-- 释放
DELETE FROM dist_lock WHERE lock_name='order:123' AND owner='uuid';
```

**劣势**：依赖 DB 负载；需定期扫描过期锁（TTL 不自动生效）；性能最低。
**适用**：已有 DB 基础设施、简单场景的降级兜底方案。

## 五、乐观锁 vs 悲观锁 深度对比

| 维度 | 乐观锁 | 悲观锁 |
|------|--------|--------|
| **假设** | 冲突是小概率事件 | 冲突是常态 |
| **加锁时机** | 提交时校验（不加锁） | 操作前加锁 |
| **实现** | CAS / 版本号 / 时间戳 | synchronized / ReentrantLock / SELECT FOR UPDATE |
| **冲突处理** | 重试或返回失败 | 等待锁释放 |
| **吞吐量** | 读多写少时高 | 写多冲突频时更稳定 |
| **死锁风险** | 无（不持锁） | 有（持锁等待） |
| **饥饿风险** | 高冲突时重试风暴 | 公平锁可避免 |
| **典型场景** | DB 版本号更新、CAS Atomic、乐观锁库存 | synchronized、MySQL FOR UPDATE、Redisson |

## 六、读写锁 vs 乐观读 对比

| | ReentrantReadWriteLock | StampedLock 乐观读 |
|--|------------------------|-------------------|
| **读-读** | 共享，并发 | 无锁，完全并发 |
| **读-写** | 互斥 | 写时读可能脏读，validate 检验 |
| **可重入** | ✅ | ❌ |
| **锁升级** | 读→写：死锁 | 乐观读→悲观读：安全 |
| **适用** | 读多写少，数据一致性强 | 读极多，短临界区，CPU 密集 |

## 七、选型决策树

```
资源竞争的范围？
  │
  ├─ 同一个 JVM 内的线程竞争
  │    │
  │    ├─ 简单互斥，无需高级特性？
  │    │    └─ synchronized（JVM 自动优化）
  │    │
  │    ├─ 需要：可中断 / 超时等待 / 多条件队列 / 公平锁？
  │    │    └─ ReentrantLock
  │    │
  │    ├─ 读多写少（读 >> 写）？
  │    │    ├─ 读操作较长 → ReentrantReadWriteLock
  │    │    └─ 读极短、CPU 密集 → StampedLock 乐观读
  │    │
  │    └─ 无锁、低冲突、原子计数？
  │         └─ AtomicInteger / LongAdder / CAS
  │
  └─ 多个 JVM 进程 / 微服务节点竞争（分布式锁）
       │
       ├─ 高性能 + 允许极低概率重复（AP）？
       │    └─ Redis Redisson（99% 场景首选）
       │
       ├─ 强一致、不允许重复（CP）？
       │    └─ ZooKeeper Curator（ZAB 协议）
       │
       └─ 无额外中间件、降级兜底？
            └─ DB 唯一键 INSERT / SELECT FOR UPDATE

MySQL 数据库场景
  │
  ├─ 冲突概率低，读多写少？
  │    └─ 乐观锁（version 字段 + CAS UPDATE）
  │
  ├─ 冲突概率高，强一致要求？
  │    └─ 悲观锁（SELECT FOR UPDATE）
  │
  ├─ 防幻读，范围查询？
  │    └─ RR 隔离 + Next-Key Lock
  │
  └─ 读写并发，快照读？
       └─ MVCC（undo log + ReadView）
```

## 八、高频面试考点速查

| 问题 | 答案要点 |
|------|---------|
| AQS 核心设计 | `volatile int state` + CLH 变种双向队列；获取资源 CAS 改 state，失败则封装 Node 入队 park；释放时 unpark 队头 |
| volatile 能替代 synchronized 吗 | 单写多读状态标志可以（保证可见性 + 有序性）；复合操作 `count++` 不行（不保证整体原子性），需用 `AtomicInteger` |
| 线程池参数：CPU/IO 密集型 | CPU 密集型：`core = CPU核数 + 1`；I/O 密集型：`core = CPU核数 × 2`；**禁用** `Executors` 工厂方法（无界队列 OOM），必须 `new ThreadPoolExecutor` + 有界队列 |
| synchronized vs ReentrantLock | 见 2.1 表；核心差异：可中断/公平锁/多条件 |
| 偏向锁为什么废弃 | 撤销需要 STW，多线程高竞争频繁撤销比轻量级锁更慢 |
| 公平锁一定好吗 | 非公平锁吞吐更高；公平锁防饥饿但增加上下文切换 |
| CAS ABA 怎么解决 | AtomicStampedReference 加版本号（stamp 递增） |
| 为什么用 Lua 解锁 | GET + DEL 非原子；两步之间锁可能过期被他人占，直接 DEL 误删 |
| Redisson watchdog 原理 | 每 TTL/3（默认10s）续期到 30s；JVM 死则 watchdog 停，锁自然过期 |
| 为什么行锁锁的是索引 | InnoDB 数据按主键聚簇，行锁本质是对 B+ 树索引 key 加锁；无索引退化为全表扫描加锁 |
| RR 下幻读如何防 | Next-Key Lock = Record Lock + Gap Lock，锁间隙防止插入 |
| ZK 锁 vs Redis 锁 | ZK CP 强一致（ZAB），Redis AP 高性能；金融强一致选 ZK，互联网高并发选 Redis |
| 死锁如何排查 | SHOW ENGINE INNODB STATUS 查看死锁信息；jstack 查看 Java 线程死锁 |
| 乐观锁重试风暴 | 高冲突时大量事务反复重试，CPU 飙升；解法：退避 + 限重试次数 + 改悲观锁 |
| 锁降级 | 写锁 → 读锁：允许（ReadWriteLock）；读锁 → 写锁：禁止（多读互相等待，死锁）|
