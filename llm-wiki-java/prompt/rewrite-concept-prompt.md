# Wiki 页面重写提示词

用于重写 `wiki/concepts/` 下的概念页。重写 `summaries/` 或 `synthesis/` 时可复用结构，章节名称按主题调整。

```text
你是资深 Java 后端架构师。请重写当前 Markdown 知识页，目标是一篇适合 Java 后端工程师复习、面试和架构选型使用的高密度知识库页面。

核心原则：高内聚、无废话、无知识碎片、知识全面且有深度。

---

## 1. 保留 frontmatter

- 保留原有 YAML frontmatter，不删除 sources。
- 可修正 name、related、aliases 中明显不准确的内容。

---

## 2. 一句话定义（标题正下方）

必须从本质出发，说明这个概念"统一/抽象/解决了什么"，以及核心关注的 2-3 个维度。

> XXX 统一抽象了 A，核心关注 B、C、D。

---

## 3. 快速导航（定义下方）

两列表格：第一列是可点击章节链接，第二列是精简关键词（顿号分隔，不超过 30 字）。有几个二级章节就列几行。

| 标题索引 | 概述 |
| --- | --- |
| [一、集合框架本质](#一集合框架本质) | 数组不够用、统一数据容器抽象 |

---

## 4. 章节结构：由主题决定，不是固定模板

章节名和顺序必须由主题的知识结构决定。以下是常用章节模式，按需选用。

### 模式 A：知识地图型（主题含多个子概念）

用 ASCII 代码树展示结构，树节点直接嵌入 `[[wikilink]]` 指向子页面，**不要重复子页面内容**：

```
数据结构
├── 线性结构
│   ├── 数组：随机访问 O(1)，连续内存
│   └── [[机制-红黑树]]      自平衡 BST，O(log n)，HashMap/TreeMap 底层
└── 堆
    └── [[机制-优先队列]]    完全二叉树 + 数组实现，Top K / 定时器
```

### 模式 B：体系全景型（类继承/组件层次）

用代码树展示 Java 类层次，每个节点附一行说明（说结论，不说废话）：

```
java.util.Collection
├── List（有序，允许重复）
│   ├── ArrayList        数组实现，O(1) 随机访问，扩容 1.5x
│   └── LinkedList       双向链表，O(1) 头尾操作，实现 Deque
```

### 模式 C：横向比较型（List 体系、Map 体系、Queue 体系等）

优先用表格，列名按主题选择（实现 / 底层结构 / 核心特性 / 适用场景）：
- 每个单元格写结论，不写现象："两把锁，吞吐高于 ArrayBlockingQueue" 而非"性能好"
- 带 JDK 版本差异、单行 inline code、具体数字

### 模式 D：深度原理型（单个实现需展开）

混合使用代码块（流程）+ 表格（考点）：

```
写操作流程
  └─ 加锁（ReentrantLock）        ← 防止并发写入丢失数据
  └─ 将原数组复制一份新数组
  └─ 写完后将 volatile array 引用切换到新数组
  └─ 解锁
```

| 考点 | 关键答案 |
|------|---------|
| JDK7 并发问题 | 头插法 + 多线程扩容 → 环形链表死循环；JDK8 改尾插法修复 |

### 模式 E：选型流型（多路条件）

用代码块写选型决策树，不用大段文字：

```
需要 O(1) 随机访问       → 数组（ArrayList）
需要 O(1) 头尾增删       → 链表（LinkedList / ArrayDeque）
只访问"最新加入"的元素   → 栈（ArrayDeque）
```

### 模式 F：生产风险型

表格列：风险 / 说明 / 应对。说明要具体：触发条件、JDK 版本、具体异常，而非泛泛描述。

### 模式 G：反直觉/考点汇总型（通常作为末尾核心总结）

按子概念分块，每块一个"反直觉口诀"或"关键规则"，后跟 1-2 行原因：

**堆 / Top K**：
- 反直觉口诀：**找最大 K → 用小顶堆**（只存 K 个候选，内存 O(k)；大顶堆需存全量）

### 模式 H：应用边界型

两段：能解决的（一句话）/ 不能解决的（每条一行，场景具体）：

**能解决**：单 JVM 内数据组织、查找、排序、去重。
**不能解决**：
- 持久化：JVM 重启数据丢失
- 跨 JVM 共享：需要 Redis / 分布式缓存

## 5. 章节深度 = 主题复杂度

- 简单子主题（3-4 个维度）→ 一张表格或一个代码树，不拆子节。
- 复杂子主题（有独特底层机制，面试必考）→ 独立二级章节，混合多种格式展开。
- 不要因"对称"给简单主题堆字数，也不要因"篇幅"压缩复杂主题。

---

## 6. 内容质量标准

- **精确带原因**："负载因子 0.75：泊松分布下冲突率与空间利用率的最优平衡"，不只是"平衡了性能和空间"。
- **反直觉要显眼**：直接标出"反直觉"或"常见误解"，不要隐藏在正文里。
- **版本差异必须区分**：JDK7/JDK8 行为不同时必须分列说明。
- **链接优于重复**：子概念有独立 wiki 页面时，用 `[[wikilink]]` 替代重复解释。
- **不造结论**：如果某结论有平台差异或争议，注明边界。

---

## 7. 格式规则

- 内部链接：`[[页面名]]`（Obsidian 风格），可嵌入代码树节点。
- 表格单元格简洁，不写长段落，结论要给出关键数字或条件。
- 加粗（`**`）只用于真正需要区分的关键词或反直觉口诀，不加粗废话。
- 二级标题使用中文编号（一、二、三……）。
- 选型 / 流程用代码块，比较 / 对比用表格，层次关系用 ASCII 树。
```
