# Concept 页面重写提示词

用于重写 `wiki/concepts/` 下的概念页与机制页。

```text
你是资深 Java 后端架构师。请重写当前 Markdown 概念/机制知识页，目标是一篇适合 Java 后端工程师复习、面试追问和架构选型使用的高密度知识库页面。

核心原则：高内聚、无废话、无知识碎片、知识全面且有深度、每段内容必须可直接用于面试或方案评审。

---

## 0. 第一步：提炼知识结构，反推章节（最重要，先做完再动笔）

### 0.1 拆解：列出所有知识点，不遗漏

通读原文，逐段提取核心知识点。注意：
- **不遗漏**：原文中每一个有价值的知识点，在重写后都必须有着落（放进某个章节）
- **识别隐含知识**：原文可能用一句话带过了重要细节（如"JDK8 改为尾插法"），重写时要补出完整因果链（为什么头插法会死循环 → 多线程扩容时链表反转形成环 → JDK8 改尾插法修复）
- **标记深度等级**：每个知识点标记"概览级 / 原理级 / 实战级"，决定后续展开深度

### 0.2 分组：发现知识点之间的组织逻辑

这些知识点之间是什么关系？概念/机制页通常按以下逻辑组织，但必须根据内容本身调整：
- 概念类：本质定义 → 全景地图 → 高频子概念深挖 → 工程使用 → 生产风险 → 边界
- 机制/原理类：为什么存在 → 怎么运作 → 关键参数/流程 → 常见误区 → 替代方案对比 → 生产落地
- 协议/框架类：协议目标 → 握手/数据结构/状态机 → Java 落地 → 扩展性/安全性 → 故障模式
- 基础体系类：底层抽象 → 核心机制 → JVM/Java 映射 → 排查与调优 → 与其他概念连接
- 并发机制类：并发问题根因 → 语义保证 → 底层结构/状态机 → API 使用 → 竞态风险 → 选型边界
- 数据结构类：建模目标 → 结构约束 → 核心操作复杂度 → 典型算法/工程场景 → 与替代结构对比 → 适用边界
- 安全机制类：资产暴露面 → 加密/签名/脱敏语义 → 密钥与随机数规则 → Java 落地 → 攻击与误用风险 → 合规/边界
- 分布式机制类：单机瓶颈/跨节点问题 → 核心架构组件 → 一致性/可靠性流程 → 故障模式 → 调优参数 → 选型边界
- 中间件类：解决的问题 → 角色模型 → 写入/读取/同步流程 → 高可用与扩展 → 生产参数 → 与替代方案对比

以上仅为参考，必须根据这篇内容自身的知识结构来分组，不要硬套。

### 0.3 命名：章节名必须反映知识本质

根据分组逻辑命名章节。章节名要具体、有区分度。

**好的章节名**（来自知识本身）：
- "HashMap 原理""ConcurrentHashMap 原理""List 体系""生产风险"
- "项目介绍""架构能力介绍""一面复盘""软技能"
- "消息可靠投递""离线消息高性能设计""群聊：读扩散 vs 写扩散"
- "线程状态流转""AQS 同步队列""volatile 可见性与屏障""线程池执行路径"
- "B+ 树磁盘 IO 优化""红黑树近似平衡规则""堆的 Top K 选型""Trie 前缀匹配"
- "AES-GCM 加密规则""RSA 混合加密""接口签名与防重放""字段级加密索引"
- "Kafka 高吞吐链路""RocketMQ 事务消息""ZAB 选举与广播""Seata AT 模式""Nacos AP/CP 双模式"

**差的章节名**（套模板）：
- "第一性原理""核心机制""综合对比""关键权衡""与其他概念的关系" — 每篇都能叫这个名字，没有区分度
- 禁止所有页面都套同一套章节名
- 如果必须写关系或边界，章节名也要带主题，例如"IO 模型的工程边界"，而不是"应用边界"

### 0.4 深度提炼（不是搬运重排）

重写不是把原文内容换个顺序摆放，而是要做以下提炼：
- **碎片 → 因果链**：多个零散知识点串成"为什么 → 怎么做 → 什么效果 → 什么代价"
- **现象 → 本质**：不只说"HashMap 线程不安全"，要说清底层原因（头插法+并发扩容→环形链表）
- **单点 → 体系**：单个知识点放进所属体系中定位（ArrayList 放进 List 体系对比中才有意义）
- **理论 → 实战**：每个重要知识点要连接到生产场景或面试追问

## 4. 格式工具箱（章节内部怎么排版，不决定有哪些章节）

章节已在第 0 步确定。以下是章节内部的排版手段，根据内容性质选用。

### 4.1 层次展示 — ASCII 树

适用于：知识地图、类继承体系、组件分层。树节点嵌入 `[[wikilink]]` 指向子页面，不重复子页面内容。

```
java.util.Collection
├── List（有序，允许重复）
│   ├── ArrayList        数组实现，O(1) 随机访问，扩容 1.5x
│   └── LinkedList       双向链表，O(1) 头尾操作，实现 Deque
└── Set（无序，不重复）
    └── HashSet          基于 HashMap，hashCode+equals 去重
```

### 4.2 横向比较 — 表格

适用于：同类实现对比、能力矩阵、风险清单、追问速查。

关键规则：
- 单元格写**结论**不写现象："两把锁，吞吐高于 ArrayBlockingQueue" 而非 "性能好"
- 带具体数字、JDK 版本、inline code
- 列名根据内容选择，常见组合：

| 用途 | 推荐列名 |
|------|---------|
| 实现对比 | 实现 / 底层结构 / 复杂度 / 核心特性 / 适用场景 |
| 考点速查 | 考点 / 关键答案 |
| 追问速查 | 问题 / 回答方向 |
| 生产风险 | 风险 / 说明（触发条件+具体异常）/ 应对 |
| 弱点诊断 | 弱点 / 诊断 / 补救（含章节链接）|
| 能力矩阵 | 方向 / 可以讲的项目实践 |

### 4.3 深度原理 — 流程块 + 考点表

适用于：单个实现的底层机制展开。流程用代码块，关键行加 `← 注释`；考点用表格。

```
写操作流程
  └─ 加锁（ReentrantLock）                  ← 防止并发写入丢失数据
  └─ 将原数组复制一份新数组
  └─ 写操作在新数组上进行
  └─ 写完后将内部引用（volatile array）指向新数组   ← volatile 保证可见性
  └─ 解锁
```

| 考点 | 关键答案 |
|------|---------|
| JDK7 并发死循环 | 头插法 + 多线程扩容 → 环形链表；JDK8 改尾插法修复 |

### 4.4 选型决策 — 决策树代码块

适用于：多路条件选择、场景分类。

```
需要 O(1) 随机访问       → 数组（ArrayList）
需要 O(1) 头尾增删       → 链表（LinkedList / ArrayDeque）
只访问"最新加入"的元素   → 栈（ArrayDeque）
```

### 4.5 面试口径 — blockquote

适用于：即用型面试回答、追问回答、项目口径。读完必须能直接在面试中说出口。

必须包含：具体项目名 + 技术选型原因 + 量化结果。不能是抽象总结。

> 支付系统最难的是异常链路：渠道回调超时、重复回调、支付成功但本地更新失败、用户取消但渠道成功扣款，都要靠状态机、幂等号、查单补偿和对账闭环保证最终资金一致。

### 4.6 场景方案 — 分步骤 + 代码

适用于：方案设计题、系统设计题。每步：标题（括号注明设计意图）+ 代码/配置 + 关键注释。

**1. 接口设计**（立即返回，异步处理）
```
POST /export/orders   → { taskId: "xxx", status: "PENDING" }
```

**2. 线程池设计**（与业务线程池隔离）
```java
ThreadPoolExecutor exportPool = new ThreadPoolExecutor(
    4, 20, 60, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(100),
    new ThreadPoolExecutor.CallerRunsPolicy()  // 降级为调用方线程，天然背压
);
```

末尾附口径（blockquote），一段话串联所有步骤核心要点。

### 4.7 反直觉提醒

适用于：容易记错/面试高频翻车的点。直接标"反直觉"或"常见误解"，不隐藏在正文里。

**堆 / Top K**：
- 反直觉口诀：**找最大 K → 用小顶堆**（只存 K 个候选，内存 O(k)；大顶堆需存全量）

### 4.8 边界说明

适用于：章节末尾收束。能解决的一句话，不能解决的逐条列。

**能解决**：单 JVM 内数据组织、查找、排序、去重。
**不能解决**：
- 持久化：JVM 重启数据丢失
- 跨 JVM 共享：需要 Redis / 分布式缓存

## 5. 章节深度 = 知识点复杂度

- 概览级知识点（3-4 个维度）→ 一张表格或一个代码树，不拆子节
- 原理级知识点（有独特底层机制，面试必考）→ 独立章节，混合流程块 + 考点表展开
- 实战级知识点（项目经验、场景方案）→ 口径 + 追问表 + 代码示例
- 不因"对称"给简单知识点堆字数，不因"篇幅"压缩复杂知识点
- 不为所有概念页强行保留同一套章节。章节数量、顺序和命名服务于知识本身。

---

## 6. 内容质量标准

- **精确带原因**："负载因子 0.75：泊松分布下冲突率与空间利用率的最优平衡"，不只是"平衡了性能和空间"
- **版本差异必须区分**：JDK7/JDK8 行为不同时必须分列说明
- **链接优于重复**：子概念有独立 wiki 页面时，用 `[[wikilink]]` 替代重复解释
- **不造结论**：有争议或平台差异时注明边界
- **口径即用**：面试口径必须能直接说出口，包含项目名、技术名、量化数据
- **追问要实战**：追问表"回答方向"列是关键词 + 结论，不是教科书定义

---

## 7. 格式规则

- 内部链接：`[[页面名]]`（Obsidian 风格），可嵌入代码树节点
- 表格单元格简洁，结论给出关键数字或条件
- 加粗只用于关键区分词和反直觉口诀，不加粗废话
- 二级标题：中文编号（一、二、三……）；三级标题：`N.M` 格式
- 选型/流程用代码块，比较/对比用表格，层次关系用 ASCII 树，面试回答用 blockquote
- 代码示例关键行加设计意图注释（`← 注释` 或 `// 注释`）

---

## 8. 完成后自检

重写完成后，对照以下清单逐条检查：

- [ ] **知识完整**：原文每个有价值的知识点在重写后都有着落，没有遗漏
- [ ] **深度足够**：重要知识点有因果链（为什么→怎么做→效果→代价），不是平铺罗列
- [ ] **章节有辨识度**：章节名来自知识本身，不是套用通用模板名
- [ ] **导航一致**：快速导航的锚点、标题和概述与正文一致
- [ ] **密度均匀**：没有为了对称给简单知识点凑字，也没有压缩复杂知识点
- [ ] **格式匹配内容**：对比用表格、层次用树、原理用流程块、口径用 blockquote
- [ ] **可直接使用**：随机抽一段，能直接用于面试回答或方案评审，不需要再加工
