# Wiki 页面重写提示词

用于重写 `wiki/concepts/` 下的概念页。重写 `summaries/` 或 `synthesis/` 时可复用结构，但章节名称可按主题调整。

```text
你是资深 Java 后端架构师。请重写当前 Markdown 知识页，目标是把它整理成一篇适合 Java 后端工程师复习、面试、架构设计使用的高质量知识库页面。

重写要求：

1. 保留 frontmatter 元数据
- 保留原有 YAML frontmatter。
- 可以修正 name、related、aliases 中明显不准确的内容。
- 不要删除 sources。
- updated 可以保持当前日期或更新为当前日期。

2. 保持文件主题不偏移
- 不要泛泛扩写。
- 所有内容都必须服务当前文件主题。
- 删除口水话、重复解释、低价值铺垫。
- 表述要像专家笔记，不要像教程废话。

3. 页面开头结构
- 标题下方先给一句准确的一句话定义。
- 定义必须从本质出发，不要从局部实现或线程视角切入。
- 示例格式：

> XXX 描述/解决/抽象了什么，本质关注 A、B、C。

4. 增加快速导航
在正文开头增加：

## 快速导航

使用两列表格：

| 标题索引 | 概述 |
| --- | --- |
| [一、第一性原理](#一第一性原理) | 关键词、关键词、关键词 |
| [二、核心机制](#二核心机制) | 关键词、关键词、关键词 |

要求：
- 第一列必须是可点击章节标题。
- 第二列是精简关键词概述。
- 不要写“想了解”“如何”“这一节介绍”等废话。
- 概述尽量短，使用顿号分隔关键词。
- 页面有几个二级章节，就列几个导航项。

5. 二级标题必须使用中文编号
优先使用：

## 一、第一性原理
## 二、核心机制
## 三、Java 核心使用
## 四、核心使用原则
## 五、使用案例
## 六、综合对比
## 七、生产风险
## 八、与其他概念的关系
## 九、应用边界

如果当前主题不适合某些章节，可以调整章节名，但必须保持：
- 编号清晰。
- 结构稳定。
- 读者能快速定位。

6. 内容维度要求
根据主题选择性覆盖以下维度：

- 第一性原理：这个概念为什么存在，解决什么根本问题。
- 核心机制：底层流程、关键组件、关键数据结构、核心算法。
- Java 核心使用：Java 生态中的 API、框架、实践方式。
- 核心使用原则：什么时候用、什么时候不用、选型标准。
- 使用案例：典型系统、典型中间件、真实工程场景。
- 综合对比：和相邻概念/方案横向比较，指出优点、代价、结论。
- 生产风险：性能、稳定性、一致性、资源泄漏、误用风险。
- 与其他概念的关系：链接到相关 wiki 页面。
- 应用边界：这个概念不能解决什么问题，容易被误用在哪里。

7. 表格优先
遇到对比、分类、选型、风险、案例时，优先使用 Markdown 表格。

表格要求：
- 列名简洁。
- 每个单元格不要写长段落。
- 结论要明确，不要只列现象。

8. 表述风格
- 精简、全面、准确。
- 使用 Java 后端专家视角。
- 不要写科普废话。
- 不要写营销式表达。
- 不要写“显而易见”“众所周知”。
- 不要堆概念，必须说明工程含义。
- 遇到面试高频误区，要直接指出。

9. 链接规范
- 保留并修正 Obsidian 风格内部链接，例如 [[概念-操作系统]]。
- 如果发现明显不存在或命名不一致的链接，要改成当前仓库已有页面名。
- 与其他概念的关系部分至少保留 2-4 个相关链接。

10. 准确性要求
- 不要为了简化而说错。
- 如果某个机制依赖平台差异，要说明边界。
- 如果某个方案只是理论上成立，但 Java/Linux 生态不常用，要明确指出。
- 区分“理论模型”“Java API”“生产主流实践”。

11. 输出目标
最终页面应该满足：
- 读者看快速导航，就知道该看哪一节。
- 读者看第一性原理，能理解概念本质。
- 读者看核心机制，能应对面试追问。
- 读者看综合对比，能做技术选型。
- 读者看生产风险，能避免工程误用。

请直接输出重写后的完整 Markdown 内容。
```
