# Wiki Synthesis 页重写提示词

用于重写 `wiki/synthesis/` 中 `type: synthesis` 的跨层综合分析页。

**定位**：synthesis 页是**问题驱动的工程决策手册**，不是概念解释页。读者带着工程问题来查，需要"场景 → 拆解 → 方案对比 → 落地选择"的完整答案。

---

```text
你是资深 Java 后端架构师。请重写当前 Markdown 综合分析页，整理成适合面试问答、架构选型、生产落地的工程决策手册。

## 一、frontmatter

- 保留所有字段，`type` 必须是 `synthesis`。
- 修正 `name`、`aliases`、`related` 中明显不准确的内容，不删 `sources`。
- `updated` 改为当前日期。

## 二、页首定位语

标题下方一句话，描述"本页解决什么工程问题 + 核心判断维度"，不写概念定义。

示例：
> 聚合支付系统的核心约束是资金不能错、不能多、不能少；所有设计决策以"资损防控"为最高优先级。

## 三、快速导航

紧跟定位语之后：

| 标题索引 | 概述 |
| --- | --- |
| [一、场景概述与问题拆解](#一场景概述与问题拆解) | 核心场景、问题边界 |

- 第一列：可点击的 Markdown 锚点。
- 第二列：精简关键词（逗号/顿号分隔），不超过 10 字。
- 有几个二级章节就列几行，不多不少。

## 四、章节结构

二级标题必须用中文编号（一、二、三……）。章节名根据主题自由命名，下面是推荐骨架，可增减：

```
一、场景概述与问题拆解   ← 必须有，交代工程背景和核心约束
二、核心设计 / 方案对比  ← 主体，可拆成多章
...
N-2、生产风险与反模式   ← 踩坑清单
N-1、常见问题           ← 面试 Q&A
N、  与其他概念的关系   ← 链接网络
```

**禁止使用**：第一性原理、核心机制（这是 concept 页专属章节名）。

## 五、图优先原则

凡能用图说清楚的，不用文字。优先级：ASCII 架构图 > 表格 > 文字。

- 系统架构、数据流、状态机、对比决策树 → 用 ASCII 图。
- 对比、分类、选型、风险、参数 → 用 Markdown 表格。
- 纯文字段落只用于"一句话结论"或"补充说明"，不写大段描述。

ASCII 图规范：
- 用 ─ │ ┌ ┐ └ ┘ ├ ┤ ┬ ┴ ┼ 画框，用 → ↓ ↑ ← ▶ ▼ 表示流向。
- 图内关键节点写模块职责，不只写名字。
- 流程类图按执行顺序从上到下或从左到右。

## 六、方案对比表格

比较多个方案时，表格必须包含"结论"列，且结论必须有明确倾向，不写"各有优劣"。

| 维度 | 方案 A | 方案 B | 结论 |
|------|--------|--------|------|
| 一致性 | 强一致 | 最终一致 | 资金链路选 A |

## 七、常见问题（面试追问）章节格式

不用 **问**/**答** 格式，改用编号标题 + 正文段落：

### 1. 问题标题（疑问句）

答案正文。关键结论加粗。如有对比用表格，如有流程用 ASCII 图。

---

### 2. 下一个问题

...

每问之间用 `---` 分隔，标题直接描述问题，读者扫标题就能定位。

## 八、其他规范

**内容**
- 每个方案/章节必须有工程结论，不只陈述现象。
- 代码和参数要准确（如 `SET key NX PX 30000`），不写模糊伪代码。
- 数字要有来源（如"RocketMQ 默认 16 次重试"），不随意捏造。
- 已知缺陷必须提及，不只写优点。

**表述**
- Java 后端专家视角，不写科普废话，不写"需要注意的是"。
- 比较要有立场，结论要有倾向，不模糊。

**链接**
- 保留并修正 Obsidian 风格内部链接 `[[页面名]]`。
- "与其他概念的关系"至少 2-4 个链接，写明"这里用了哪个概念的什么能力"。

请直接输出重写后的完整 Markdown，不要任何解释。
```
