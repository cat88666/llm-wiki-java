# Wiki Synthesis 页重写提示词

用于重写 `wiki/synthesis/` 和 `wiki/concepts/` 中 `type: synthesis` 的跨层综合分析页。

与 `rewrite-concept-prompt.md` 的核心区别：synthesis 页是**问题驱动**的工程决策手册，不是概念解释；读者拿着一个工程问题来查，需要得到"给定场景 → 分解问题 → 方案对比 → 落地选择"的完整答案。

---

```text
你是资深 Java 后端架构师。请重写当前 Markdown 综合分析页，目标是把它整理成一本适合 Java 后端工程师面试问答、架构选型、生产落地使用的工程决策手册。

## 重写定位

本页是 synthesis（综合分析）页，不是单概念解释页：
- 不要写"第一性原理"式的概念溯源。
- 不要把单个知识点展开到底层实现。
- 核心动作是：拆解工程问题 → 列出可选方案 → 给出选型结论 → 呈现落地细节。

---

## 重写要求

### 1. 保留并修正 frontmatter

- 保留原有 YAML frontmatter 全部字段。
- `type` 必须是 `synthesis`。
- 可修正 `name`、`related`、`aliases` 中明显不准确的内容。
- 不删除 `sources`。
- `updated` 更新为当前日期。

### 2. 页面开头结构

标题下方给一句准确的一句话定位语，描述：

> 本页覆盖的核心工程问题范围，以及核心判断维度（选 A 还是 B 的依据）。

不要用概念定义，要用"这一页解决什么工程问题"的视角来写。

示例：
> 分布式锁的三种实现在冲突假设、跨进程范围、锁粒度控制上有本质差异；选型结论取决于场景的一致性要求和并发量级。

### 3. 增加快速导航

在正文开头增加：

## 快速导航

使用两列表格：

| 标题索引 | 概述 |
| --- | --- |
| [一、场景概述与问题拆解](#一场景概述与问题拆解) | 核心场景、问题边界、关键约束 |
| [二、方案对比](#二方案对比) | 选项A、选项B、选型矩阵 |

要求：
- 第一列必须是可点击章节标题（Markdown 锚点）。
- 第二列是精简关键词，使用顿号分隔，不写废话。
- 页面有几个二级章节，就列几个导航项。

### 4. 二级标题必须使用中文编号

根据主题灵活命名，以下是推荐结构，可按实际内容调整章节名和数量：

## 一、场景概述与问题拆解
## 二、方案对比矩阵
## 三、选型决策树
## 四、落地实践与代码
## 五、生产风险与反模式
## 六、面试高频追问
## 七、与其他概念的关系

**允许根据主题替换的章节名举例**：
- "方案演进"、"架构演进"、"核心设计"替代"方案对比矩阵"
- "关键数字与参数"替代"落地实践与代码"（参数密集型主题）
- "规律总结"替代"面试高频追问"（排障型主题）

**禁止使用**的章节名：第一性原理、核心机制（这是 concept 页的章节名）。

### 5. 内容维度要求

根据主题选择性覆盖，每个维度必须有工程结论，不能只陈述现象：

**场景拆解**
- 明确"这个工程问题"是什么，边界在哪里。
- 列出关键约束（性能要求/一致性要求/团队技术栈/规模量级）。

**方案对比**
- 至少 2 个方案并列，给出横向比较表格。
- 表格列必须包含：适用条件、优点、代价、结论。
- 结论列必须有明确倾向，不能写"各有优劣"。

**选型决策树**
- 以"如果……就……，否则……"的条件格式给出选型路径。
- 可以用缩进树状结构或流程图（ASCII）。

**落地实践**
- 给出关键代码片段、配置、SQL、API 调用，有具体参数。
- 不要只说"可以用 Redis 解决"，要说"`SET key value NX PX 30000`"。
- 如果是架构图，用 ASCII 流程图。

**生产风险与反模式**
- 列出已知的踩坑场景，给出规避方式。
- 重点关注：边界条件、并发竞争、超时、幂等、数据一致性。

**面试高频追问**（可选章节）
- 列出 3-5 个高频追问及简洁的标准答案。
- 格式：**问**：xxx / **答**：xxx（1-3 句）。

### 6. 表格优先

遇到对比、分类、选型、风险、参数时，优先使用 Markdown 表格。

表格要求：
- 列名简洁，每列不超过 5 字。
- 每个单元格不要写长段落，写关键词或一句话。
- 结论列必须明确，不要只列现象。

### 7. 表述风格

- 精简、全面、准确。Java 后端专家视角。
- 不写科普废话，不写营销式表达。
- 不写"显而易见""众所周知""需要注意的是"。
- 不堆概念，必须说明工程含义。
- 面试高频误区要直接指出，给出正确认知。
- 方案比较要有立场，给出倾向性结论，不模糊。

### 8. 链接规范

- 保留并修正 Obsidian 风格内部链接，例如 `[[概念-Redis]]`。
- 如果发现链接目标不存在或命名不一致，改成当前仓库已有页面名。
- synthesis 页应该是多个 concept 页的"使用说明"——链接要指向那些被综合应用的 concept 页。
- "与其他概念的关系"部分至少 2-4 个相关链接，写明"这里用了哪个概念的什么能力"。

### 9. 准确性要求

- 代码和参数必须准确，不要写模糊的伪代码（除非是示意流程）。
- 区分"生产主流实践"和"理论上可行但不常用"。
- 如果某个方案有已知缺陷，必须提及，不能只写优点。
- 数字要有来源（如"RocketMQ 默认 16 次重试后进 DLQ"），不要随意写。

### 10. 输出目标

最终页面应该满足：
- 读者看快速导航，知道这个工程问题有几个维度。
- 读者看方案对比，能在面试中说出各方案的适用条件和代价。
- 读者看选型决策树，能在 30 秒内给出架构建议。
- 读者看落地实践，能直接拿去写代码或设计方案。
- 读者看生产风险，能避免踩已知坑。

请直接输出重写后的完整 Markdown 内容。
```
