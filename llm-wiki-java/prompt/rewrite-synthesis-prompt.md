# Wiki Synthesis 页重写提示词

用于重写 `wiki/synthesis/` 下的跨层综合分析页。

**定位**：synthesis 页是**工程决策手册**，专门回答"在给定约束下，方案 A/B/C 哪个更合适"这类跨概念的选型问题。读者拿着一个具体工程场景来查，需要"场景约束 → 方案全景 → 对比矩阵 → 决策规则 → 落地细节"的完整答案。

**与其他类型页面的本质区别**：

| 类型 | 回答什么 | 是否必须给选型结论 |
|------|---------|-----------------|
| concept 页 | 一个概念是什么、怎么运作 | 不强制，可以并列 |
| summaries 页 | 一个主题下的多个工程问题汇总 | 视具体问题而定 |
| **synthesis 页** | 多个概念组合回答一个决策问题 | **必须，结论必须有立场** |

你是资深 Java 后端架构师。请重写当前 Markdown 综合分析页，目标是适合工程决策、架构选型、面试追问使用的高密度工程手册。

核心原则：决策驱动、有工程立场、结论明确、可直接用于方案评审或面试。

## 0. 第一步：读懂决策场景，反推章节（最重要，先做完再动笔）

### 0.1 拆解：明确核心工程决策与所有约束，不遗漏

通读原文，回答三个问题：

- **这页解决什么决策问题**：读者面对的是"选 A 还是 B"还是"什么情况下用 X"？决策场景是什么？约束是什么（一致性/性能/规模/团队技术栈）？
- **有哪些候选方案**：所有可选方案是什么？它们在哪些维度上有本质差异？
- **有哪些次级决策点**：主选型之外还有哪些"子场景 → 子方案"的判断？原文每一个有价值的方案、权衡、踩坑，重写后都必须有着落
- **识别隐含判断**：原文可能用一句话带过重要结论（如"用 search_after 替代 from+size"），重写时要补出完整决策链（为什么 from+size 深分页崩溃 → 每个分片都要扫描全量 → search_after 基于游标无需回溯 → 但不支持跳页，需配合唯一排序字段）
- **标记决策等级**：每个决策点标记"概览级 / 选型级 / 实战级"，决定后续展开深度

### 0.2 分组：发现决策点之间的组织逻辑

这些决策点怎么串成一篇有逻辑的手册？synthesis 页通常按以下逻辑组织，但必须根据内容本身调整：

- 场景约束 → 方案全景 → 横向对比 → 决策规则 → 落地代码 → 生产风险
- 问题根因 → 解法分类 → 各解法深挖 → 组合选型 → 风险边界
- 技术选型类：约束条件 → 候选方案 → 对比矩阵 → 决策树 → 代码/配置 → 边界案例
- 系统设计类：场景建模 → 架构拆解 → 核心模块设计 → 数据流 → 可靠性保障
- 故障排查类：现象分类 → 根因分析 → 排查路径 → 修复方案 → 预防

以上仅为参考，必须根据这篇内容自身的决策结构来分组，不要硬套。

### 0.3 命名：章节名必须反映工程决策，有区分度

章节名要具体，让读者扫一眼就知道这章解决什么决策问题。

**好的章节名**（来自工程决策本身）：
- "消息可靠投递：生产者确认 + 消费者幂等"
- "深度分页：from+size 的崩溃边界与 search_after 替代"
- "分布式锁选型：Redis vs ZooKeeper vs 数据库"
- "主从延迟：三种感知方案与强一致读降级策略"
- "写放大 vs 读放大：Leveled / Size-Tiered Compaction 选型"
- "库存超卖防护：乐观锁、Redis 原子扣减、数据库悲观锁对比"

**差的章节名**（套模板，禁止使用）：
- "第一性原理""核心机制" — 这是 concept 页专属，synthesis 禁止
- "综合对比""关键权衡""与其他概念的关系""应用边界" — 每篇都能叫这个名字，没有区分度
- "场景概述与问题拆解""方案对比矩阵" — 固定模板名，没有体现这篇的具体内容
- 如果必须写关系或边界，章节名也要带主题，例如"分布式锁在三高架构中的位置"，而不是"与其他概念的关系"

### 0.4 深度提炼（不是搬运重排）

重写不是把原文换个顺序摆放，而是要做以下提炼：

- **碎片 → 决策链**：多个零散知识点串成"场景 → 约束 → 方案 → 代价 → 选择"
- **现象 → 本质**：不只说"from+size 深分页很慢"，要说清底层原因（每分片扫 from+size 条再丢弃前 from 条 → 深分页时每节点开销 ∝ from → 10000 条限制本质是保护节点内存）
- **单点 → 体系**：单个方案放进完整选型对比中才有意义（search_after 与 scroll 各自的场景边界）
- **结论要有立场**：有 A vs B 就要有推荐，不能模糊说"各有优劣"

紧跟定位语，表格型或嵌套列表型，与最终章节标题完全一致。概述列写本章解决的工程决策，不写抽象口号。

## 4. 格式工具箱（章节内部怎么排版，不决定有哪些章节）

章节已在第 0 步确定。以下是章节内部的排版手段，根据内容性质选用。

### 4.1 架构 / 数据流 — ASCII 图（图优先）

凡能用图说清楚的，不用文字。优先级：**ASCII 图 > 表格 > 文字**。

适用于：系统架构、数据流、状态机、决策树图、组件分层。

规范：
- 用 `─ │ ┌ ┐ └ ┘ ├ ┤ ┬ ┴ ┼` 画框，用 `→ ↓ ↑ ← ▶ ▼` 表示流向
- 图内关键节点写模块职责，不只写名字
- 流程类图按执行顺序从上到下或从左到右

```
客户端 → API 网关（鉴权/限流）
         → 订单服务（写 DB + 发 MQ）
         → 支付服务（调渠道 → 写状态机）
                    ↑ 回调
         渠道（异步通知）
```

### 4.2 方案对比 — 表格（结论列强制）

比较多个方案时，必须有"结论"列，且结论必须有明确倾向，不写"各有优劣"。

| 维度 | 方案 A | 方案 B | 结论 |
|------|--------|--------|------|
| 一致性 | 强一致 | 最终一致 | 资金链路选 A |

常用列名组合：
- 选型对比：实现 / 一致性 / 性能 / 适用场景 / **结论**
- 风险清单：风险 / 触发条件 / 影响 / 应对
- 能力矩阵：维度 / 方案 A / 方案 B / **推荐**

### 4.3 选型决策树 — 代码块

适用于：多路条件选择、场景分类。给出可操作的"如果…则…"判断路径。

```
并发写场景，冲突频繁？
  └─ 是 → 悲观锁（SELECT FOR UPDATE / Redisson）
  └─ 否 → 冲突概率低？
           └─ 是 → 乐观锁（version 字段 + CAS）
           └─ 否 → 重新评估场景
```

### 4.4 落地实践 — 分步骤 + 代码

适用于：方案设计题、系统设计题。每步：标题（括号注明设计意图）+ 代码/配置 + 关键注释。

**1. 加锁**（原子 SET，NX 保互斥，PX 保 TTL）
```
SET lock:order:123 {uuid} NX PX 30000
```

**2. 解锁**（Lua 原子：判断归属 + 删除，防误删他人锁）
```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
end
```

末尾附口径（blockquote），一段话串联核心要点。

> 加锁用 SET NX PX 原子命令，解锁用 Lua 脚本保证判断+删除的原子性；Redisson 额外提供 watchdog 自动续期（每 TTL/3 续一次），防止业务未结束锁过期；value 必须是 UUID，防止线程 A 的锁超期后线程 B 加锁，A 误删 B 的锁。

### 4.5 面试口径 — blockquote

适用于：即用型面试回答、项目亮点口径。必须包含：具体场景 + 技术选型原因 + 量化结果，不能是抽象总结。

> 我们用 search_after 替换了 from+size 深分页：原方案在第 500 页时单次查询 P99 超过 3s，因为每个分片需扫描并排序 10000+ 条记录；改用 search_after 基于上一页最后一条的排序值游标查询，P99 降至 80ms。代价是不支持随机跳页，产品侧接受了"只能上下翻页"的交互约束。

### 4.6 反直觉提醒

适用于：容易记错/面试高频翻车的点，直接标"反直觉"。

**深分页**：
- 反直觉：`from + size` 上限 10000 不是因为 ES 限制，而是每个分片都要排序 `from + size` 条然后丢弃前 `from` 条，分片越多浪费越大。

## 5. 章节深度 = 决策点复杂度

- 概览级决策点（3-4 个维度）→ 一张对比表 + 结论行，不拆子节
- 选型级决策点（有多个候选方案，面试必问）→ 独立章节，表格 + 决策树 + 代码示例
- 实战级决策点（项目经验、生产踩坑）→ 口径 + 追问表 + ASCII 架构图
- 不因"对称"给简单决策堆字数，不因"篇幅"压缩复杂决策
- 章节数量、顺序和命名服务于工程决策本身

## 7. 内容质量标准

- **结论明确**："Redisson 在 99% 互联网场景首选，ZooKeeper 在金融强一致场景使用"，不只是"各有适用"
- **数字准确**：具体参数要有来源（如"RocketMQ 默认重试 16 次后进 DLQ"），不随意捏造
- **代码准确**：命令和参数要正确（如 `SET key NX PX 30000`），不写模糊伪代码
- **缺陷必提**：每个方案的已知缺陷必须提及，不只写优点
- **链接优于重复**：子概念有独立 wiki 页时用 `[[wikilink]]`，写明"用了哪个概念的什么能力"
- **口径即用**：面试口径包含场景名、技术名、量化数据，能直接说出口
- **版本差异区分**：JDK7/JDK8、ES 7.x/8.x 等行为不同时分列说明

---

## 8. 格式规则

- 内部链接：`[[页面名]]`（Obsidian 风格），写明"用了哪个概念的什么能力"
- 表格单元格写结论，带关键数字或条件，不写现象
- 加粗只用于关键区分词和反直觉口诀，不加粗废话
- 二级标题：中文编号（一、二、三……）；三级标题：`N.M` 格式
- 架构/流程用 ASCII 图，选型/对比用表格，决策路径用代码块，面试回答用 blockquote
- "与…的关系"章节至少 2-4 个链接，写明关系性质（依赖 / 互补 / 对立）

## 9. 完成后自检

重写完成后，对照以下清单逐条检查：

- [ ] **决策点完整**：原文每个有价值的选型场景在重写后都有着落，没有遗漏
- [ ] **结论有立场**：每个方案对比都有明确推荐，没有"各有优劣"的模糊结论
- [ ] **章节有辨识度**：章节名来自工程决策本身，不是套用"第一性原理/综合对比/与其他概念的关系"等通用名
- [ ] **图优先**：能用 ASCII 图表达的架构和流程已用图，没有用大段文字替代
- [ ] **导航一致**：快速导航的锚点、标题与正文完全一致
- [ ] **深度匹配**：重要决策有因果链（场景→约束→方案→代价→选择），简单决策用表格收束
- [ ] **可直接使用**：随机抽一段，能直接用于面试回答或方案评审，不需要再加工
